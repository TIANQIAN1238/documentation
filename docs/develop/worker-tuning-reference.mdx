本页面提供 Worker 配置选项及其在 Temporal SDK 中的默认值的快速参考。
将本指南与综合 [Worker性能](/develop/worker-performance) 文档一起使用以获得详细的调整指导。

Worker 性能受到三个主要资源的限制：

| 资源 | 描述 |
|----------|-------------|
| **计算** | CPU 密集型操作、并发任务执行 |
| **记忆** | Workflow 缓存、线程池 |
| **IO** | 网络调用 Temporal 服务，轮询 |

<a id="how-a-worker-works"></a>
## Worker 的工作原理 {#how-a-worker-works}

Worker 轮询 Temporal Cloud 中的 [Task Queue](/task-queue) 或自托管 Temporal 服务、执行任务并响应结果。

```
┌─────────────────┐     Poll for Tasks       ┌──────────────────┐
│   - Worker      │ ◄─────────────────────── │ Temporal Service │
│   - Workflows   │                          │                  │
│   - Activities  │ ───────────────────────► │                  │
└─────────────────┘   Respond with results   └──────────────────┘
```

多个 Worker 可以轮询同一个 Task Queue，从而提供水平可扩展性。

<a id="how-worker-failure-recovery-works"></a>
### Worker 故障恢复的工作原理 {#how-worker-failure-recovery-works}

当 Worker 崩溃或遇到主机中断时：

1、Workflow任务超时
2.另一个可用的Worker接任务
3.新的Worker重放Event History以重建状态
4. 从中断处继续执行

有关 Worker 架构的更多详细信息，请参阅 [什么是 Temporal Worker？](/workers)

<a id="compute-settings"></a>
## 计算设置 {#compute-settings}

计算设置控制 Worker 可以同时执行的任务数。

<a id="compute-configuration-options"></a>
### 计算配置选项 {#compute-configuration-options}

| 环境 | 描述 |
|---------|-------------|
| `MaxConcurrentWorkflowTaskExecutionSize` | 最大并发任务数 Workflow |
| `MaxConcurrentActivityTaskExecutionSize` | 最大并发任务数 Activity |
| `MaxConcurrentLocalActivityTaskExecutionSize` | 最大并发本地活动数 |
| `MaxWorkflowThreadCount` / `WorkflowThreadPoolSize` | Workflow Execution 的线程池 |

<a id="compute-defaults-by-sdk"></a>
### 通过 SDK 计算默认值 {#compute-defaults-by-sdk}

| 软件开发工具包 | 最大并发WorkflowTaskExecutionSize | 最大并发ActivityTaskExecutionSize | MaxConcurrentLocalActivityTaskExecutionSize | 最大Workflow线程计数 |
|-----|----------------------------------------|----------------------------------------|---------------------------------------------|------------------------|
| **去** | 1,000 | 1,000 | 1,000 | - |
| **Java** | 200 | 200 | 200 | 600 |
| **打字稿** | 40 | 100 | 100 | 1（重用V8Context） |
| **Python** | 100 | 100 | 100 | - |
| **。网** | 100 | 100 | 100 | - |

<a id="resource-based-slot-suppliers"></a>
### 基于资源的老虎机供应商 {#resource-based-slot-suppliers}

您可以使用基于资源的插槽供应商，而不是固定的插槽数量，这些供应商会根据 CPU 和内存利用率自动调整可用的任务插槽。
有关实现详细信息，请参阅 [老虎机供应商](/develop/worker-performance#slot-suppliers)。

<a id="memory-settings"></a>
## 内存设置 {#memory-settings}

内存设置控制 Workflow 缓存大小和线程池分配。

<a id="memory-configuration-options"></a>
### 内存配置选项 {#memory-configuration-options}

| 环境 | 描述 |
|---------|-------------|
| `MaxCachedWorkflows` / `StickyWorkflowCacheSize` | 要保留在缓存中的 Workflow 数量 |
| `MaxWorkflowThreadCount` | 线程池大小 |
| `reuseV8Context`（打字稿） | 为 Workflow 重用 V8 上下文 |

<a id="memory-defaults-by-sdk"></a>
### SDK 默认内存 {#memory-defaults-by-sdk}

| 软件开发工具包 | MaxCachedWorkflows / StickyWorkflowCacheSize |
|-----|----------------------------------------------|
| **去** | 10,000 |
| **Java** | 600 |
| **打字稿** | 动态（例如，4 GiB RAM 为 2000） |
| **Python** | 1,000 |
| **。网** | 10,000 |

有关缓存调整指南，请参阅 [Workflow 缓存调整](/develop/worker-performance#workflow-cache-tuning)。

##IO设置

IO 设置控制轮询器的数量和 Task Queue 交互的速率限制。

<a id="io-settings"></a>
### IO 配置选项 {#io-settings}

| 环境 | 描述 |
|---------|-------------|
| `MaxConcurrentWorkflowTaskPollers` | 并发 Workflow 轮询器数量 |
| `MaxConcurrentActivityTaskPollers` | 并发 Activity 轮询器数量 |
| `Namespace APS` | Namespace 的每秒操作数限制 |
| `TaskQueueActivitiesPerSecond` | Activity 按 Task Queue 的速率限制 |

<a id="io-configuration-options"></a>
### SDK默认IO {#io-configuration-options}

| 软件开发工具包 | MaxConcurrentWorkflowTaskPollers | MaxConcurrentActivityTaskPollers | Namespace APS | 任务队列每秒活动数 |
|-----|----------------------------------|----------------------------------|---------------|------------------------------|
| **去** | 2 | 2 | 400 | 无限 |
| **Java** | 5 | 5 | - | - |
| **打字稿** | 10 | 10 | - | - |
| **Python** | 5 | 5 | - | - |
| **。网** | 5 | 5 | - | - |

<a id="io-defaults-by-sdk"></a>
### 轮询器自动缩放 {#io-defaults-by-sdk}

使用轮询器自动缩放可根据工作负载自动调整并发轮询的数量。
有关配置详细信息，请参阅 [配置轮询器选项](/develop/worker-performance#configuring-poller-options)。

<a id="io-related-metrics"></a>
### Task Queue 指标 {#io-related-metrics}

| 公制 | 描述 |
|--------|-------------|
| [`poll_success_sync_count`](/cloud/metrics/reference#temporal_cloud_v0_poll_success_sync_count) | 同步匹配率（立即分配给 Worker 的任务） |
| [`approximate_backlog_count`](/cloud/metrics/openmetrics/metrics-reference#temporal_cloud_v1_approximate_backlog_count) | Task Queue 中任务的大约数量 |

Task Queue 统计数据也可通过 `DescribeTaskQueue` API 获得：
- `ApproximateBacklogCount`
- `ApproximateBacklogAge`
- `TasksAddRate`
- `TasksDispatchRate`
- `BacklogIncreaseRate`

有关 Task Queue 指标的更多信息，请参阅 [可用的 Task Queue 信息](/develop/worker-performance#task-queue-metrics)。

<a id="task-queue-metrics"></a>
### 失败指标 {#task-queue-metrics}

| 公制 | 描述 |
|--------|-------------|
| [`long_request_failure`](/references/sdk-metrics#long_request_failure) | 长时间运行的操作失败（轮询、历史检索） |
| [`request_failure`](/references/sdk-metrics#request_failure) | 标准操作失败（任务完成响应） |

常见故障代码：
- `RESOURCE_EXHAUSTED` - 超出速率限制
- `DEADLINE_EXCEEDED` - 操作超时
- `NOT_FOUND` - 找不到资源

<a id="failure-metrics"></a>
## Worker 调音技巧 {#failure-metrics}

1. **生产前进行规模测试**：在实际负载下验证您的配置。
2. **基础设施很重要**：Worker 不会在真空中运行。考虑网络延迟、数据库性能和外部服务依赖性。
3. **调整和观察**：在进行额外调整之前进行增量更改并监控指标。
4. **识别瓶颈**：使用 [约束理论](https://en.wikipedia.org/wiki/Theory_of_constraints)。改善非瓶颈资源不会提高整体吞吐量。
