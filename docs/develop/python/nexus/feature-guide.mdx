## 启动启用了 Nexus 的 Temporal 开发 Server {#run-the-temporal-nexus-development-server}

前置条件：

- [安装最新的 Temporal CLI](https://learn.temporal.io/getting_started/python/dev_environment/#set-up-a-local-temporal-service-for-development-with-temporal-cli)（推荐 `v1.3.0` 或更高版本）
- [安装最新的 Temporal Python SDK](https://learn.temporal.io/getting_started/python/dev_environment/#add-temporal-python-sdk-dependencies)（`v1.14.1` 或更高版本）

使用 Temporal Nexus 的第一步是启动启用了 Nexus 的 Temporal Server。

```
temporal server start-dev
```

此命令会自动启动带有 Web UI 的 Temporal 开发 Server，并创建 `default` Namespace。它使用内存数据库，因此不要在实际使用场景中使用。

现在应可以通过 [http://localhost:8233](http://localhost:8233) 访问 Temporal Web UI，Temporal Server 也可以在 `localhost:7233` 上接受客户端连接。

## 创建 caller 和 handler Namespace {#create-caller-handler-namespaces}

在设置 Nexus Endpoint 之前，为 caller 和 handler 分别创建独立的 Namespace。

```
temporal operator namespace create --namespace my-target-namespace
temporal operator namespace create --namespace my-caller-namespace
```

在此示例中，`my-target-namespace` 将包含 Nexus Operation handler，你将在 `my-caller-namespace` 中使用 Workflow 来调用该 Operation handler。我们使用不同的 Namespace 来演示跨 Namespace 的 Nexus 调用。

## 创建 Nexus Endpoint 将请求从 caller 路由到 handler {#create-nexus-endpoint}

建立 caller 和 handler Namespace 后，下一步是创建 Nexus Endpoint 来路由请求。

```
temporal operator nexus endpoint create \
  --name my-nexus-endpoint-name \
  --target-namespace my-target-namespace \
  --target-task-queue my-handler-task-queue
```

也可以使用 Web UI 创建 Namespace 和 Nexus Endpoint。

## 定义 Nexus Service 契约 {#define-nexus-service-contract}

为 Nexus Service 定义清晰的契约对于顺畅通信至关重要。

在此示例中，有一个描述 Service 和 Operation 名称以及输入/输出类型的 service 包，供 caller Workflow 使用 Nexus Endpoint。

每个 [Temporal SDK 都包含并使用默认的 Data Converter](https://docs.temporal.io/dataconversion)。默认 Data Converter 按以下顺序编码 Payload：Null、字节数组、Protobuf JSON 和 JSON。在多语言环境中（即使用多种语言和 SDK 开发 Temporal 解决方案），Protobuf 和 JSON 是常见的选择。此示例使用序列化为 JSON 的 Python dataclass。

[hello_nexus/service.py](https://github.com/temporalio/samples-python/blob/main/hello_nexus/service.py)

```python
from dataclasses import dataclass

import nexusrpc


@dataclass
class MyInput:
    name: str


@dataclass
class MyOutput:
    message: str


@nexusrpc.service
class MyNexusService:
    my_sync_operation: nexusrpc.Operation[MyInput, MyOutput]
    my_workflow_run_operation: nexusrpc.Operation[MyInput, MyOutput]
```

## 开发 Nexus Service handler 和 Operation handler {#develop-nexus-service-operation-handlers}

Nexus Operation handler 通常与其所抽象的底层 Temporal 原语定义在同一 Worker 中。Operation handler 可以决定给定的 Nexus Operation 是同步还是异步的。它们可以使用 Temporal SDK Client 调用底层 Temporal 原语（如 Query、Signal 或 Update），或运行其他可靠代码。Handler 应当可靠，因为[熔断器](/nexus/operations#circuit-breaking)在连续发生 5 次可重试错误（例如：Worker 超时）后会触发，阻止所有来自 caller 到该 Endpoint 的 Operation。

`nexusrpc.handler` 和 `temporalio.nexus` 模块提供了帮助创建 Nexus Operation 的工具：

- `nexusrpc.handler.sync_operation` - 创建同步 Operation handler
- `nexus.workflow_run_operation` - 创建启动 Workflow 的异步 Operation handler

### 开发同步 Nexus Operation handler

`@nexusrpc.handler.sync_operation` 装饰器用于暴露简单的 RPC handler。

[hello_nexus/handler/service_handler.py](https://github.com/temporalio/samples-python/blob/main/hello_nexus/handler/service_handler.py)

```python
import nexusrpc

@nexusrpc.handler.service_handler(service=MyNexusService)
class MyNexusServiceHandler:
    @nexusrpc.handler.sync_operation
    async def my_sync_operation(
        self, ctx: nexusrpc.handler.StartOperationContext, input: MyInput
    ) -> MyOutput:
        return MyOutput(message=f"Hello {input.name} from sync operation!")
```


同步 Operation handler 必须快速返回（少于 `10s`）。实现也可以进行其他调用，但 handler 应当可靠，以避免触发[熔断器](/nexus/operations#circuit-breaking)。

### 在同步 handler 中使用 Temporal Client 处理 Signal、Query 和 Update

在同步 handler 中使用 Temporal Client 来对 Workflow 进行 Signal、Query 或 Update 是一种常见模式。你也可以使用 Signal-With-Start 或 Update-With-Start 来确保 Workflow 已启动并向其发送 Signal 或 Update。所有调用必须在 [Nexus 请求超时](/cloud/limits#nexus-operation-request-timeout)内完成。Update 应是短暂的，以保持在此截止时间内。

使用 `nexus.client()` 获取 Worker 初始化时使用的 Client。[nexus_sync_operations](https://github.com/temporalio/samples-python/blob/main/nexus_sync_operations) 示例展示了如何创建使用同步 Operation 发送 Update 和 Query 的 Nexus Service：

[nexus_sync_operations/handler/service_handler.py](https://github.com/temporalio/samples-python/blob/main/nexus_sync_operations/handler/service_handler.py)

```python
from temporalio import nexus

@nexusrpc.handler.service_handler(service=GreetingService)
class GreetingServiceHandler:

    @property
    def greeting_workflow_handle(self) -> WorkflowHandle[GreetingWorkflow, str]:
        return nexus.client().get_workflow_handle_for(
            GreetingWorkflow.run, self.workflow_id
        )

    ...
```

除 `nexus.client()` 外，还可以使用 `nexus.info()` 访问当前执行的 Nexus Operation 的相关信息，包括其 Task Queue。


### 开发用于启动 Workflow 的异步 Nexus Operation handler

使用 `@nexus.workflow_run_operation` 装饰器，这是将 Workflow 作为 Operation 暴露的最简便方式。

[hello_nexus/handler/service_handler.py](https://github.com/temporalio/samples-python/blob/main/hello_nexus/handler/service_handler.py)

```python
import nexusrpc
from temporalio import nexus

@nexusrpc.handler.service_handler(service=MyNexusService)
class MyNexusServiceHandler:
    @nexus.workflow_run_operation
    async def my_workflow_run_operation(
        self, ctx: nexus.WorkflowRunOperationContext, input: MyInput
    ) -> nexus.WorkflowHandle[MyOutput]:
        return await ctx.start_workflow(
            WorkflowStartedByNexusOperation.run,
            input,
            id=str(uuid.uuid4()),
        )
```

Workflow ID 通常应是具有业务意义的 ID，用于对 Workflow 启动进行去重。一般情况下，该 ID 应作为 Nexus Service 契约的一部分通过 Operation 输入传递。

:::tip RESOURCES

使用 Use-Existing 的冲突策略[将多个 Nexus caller 附加到 handler Workflow](/nexus/operations#attaching-multiple-nexus-callers)。

:::

#### 将 Nexus Operation 输入映射到多个 Workflow 参数

Nexus Operation 只能接受一个输入参数。如果想让 Nexus Operation 启动一个接受多个参数的 Workflow，请使用 `ctx.start_workflow` 方法。

<!--SNIPSTART samples-python-nexus-handler-multiargs-->
[nexus_multiple_args/handler/service_handler.py](https://github.com/temporalio/samples-python/blob/main/nexus_multiple_args/handler/service_handler.py)
```py
@nexusrpc.handler.service_handler(service=MyNexusService)
class MyNexusServiceHandler:
    """
    Service handler that demonstrates multiple argument handling in Nexus operations.
    """

    # This is a nexus operation that is backed by a Temporal workflow.
    # The key feature here is that it demonstrates how to map a single input object
    # (HelloInput) to a workflow that takes multiple individual arguments.
    @nexus.workflow_run_operation
    async def hello(
        self, ctx: nexus.WorkflowRunOperationContext, input: HelloInput
    ) -> nexus.WorkflowHandle[HelloOutput]:
        """
        Start a workflow with multiple arguments unpacked from the input object.
        """
        return await ctx.start_workflow(
            HelloHandlerWorkflow.run,
            args=[
                input.name,  # First argument: name
                input.language,  # Second argument: language
            ],
            id=str(uuid.uuid4()),
        )


```

### 在 Worker 中注册 Nexus Service handler

开发好异步 Nexus Operation handler 以启动 Workflow 后，下一步是在 Worker 中注册 Nexus Service handler。此时可以向 service handler 的 `__init__` 方法传递任何需要的参数。

[hello_nexus/handler/worker.py](https://github.com/temporalio/samples-python/blob/main/hello_nexus/handler/worker.py)

```python
async def main():
    client = await Client.connect("localhost:7233", namespace=NAMESPACE)
    worker = Worker(
        client,
        task_queue=TASK_QUEUE,
        workflows=[WorkflowStartedByNexusOperation],
        nexus_service_handlers=[MyNexusServiceHandler()],
    )
    await worker.run()
```

## 开发使用 Nexus Service 的 caller Workflow {#develop-caller-workflow-nexus-service}

要从 caller Workflow 执行 Nexus Operation，请导入必要的 Service 定义和 Operation 输入/输出类型：

[hello_nexus/caller/workflows.py](https://github.com/temporalio/samples-python/blob/main/hello_nexus/caller/workflows.py)

```python
from temporalio import workflow

with workflow.unsafe.imports_passed_through():
    from hello_nexus.service import MyInput, MyNexusService, MyOutput

@workflow.defn
class CallerWorkflow:
    @workflow.run
    async def run(self, name: str) -> tuple[MyOutput, MyOutput]:
        nexus_client = workflow.create_nexus_client(
            service=MyNexusService,
            endpoint=NEXUS_ENDPOINT,
        )
        # Start the nexus operation and wait for the result in one go, using execute_operation.
        wf_result = await nexus_client.execute_operation(
            MyNexusService.my_workflow_run_operation,
            MyInput(name),
        )
        # Alternatively, you can use start_operation to obtain the operation handle and
        # then `await` the handle to obtain the result.
        sync_operation_handle = await nexus_client.start_operation(
            MyNexusService.my_sync_operation,
            MyInput(name),
        )
        sync_result = await sync_operation_handle
        return sync_result, wf_result
```

### 在 Worker 中注册 caller Workflow 并启动 caller Workflow

开发好 caller Workflow 后，下一步是将其注册到 Worker。

最后，必须使用 `client.start_workflow()` 或 `client.execute_workflow()` 启动 caller Workflow。

这些步骤与普通 Workflow 相同。Python 示例将它们合并在一个应用程序中。参考 [hello_nexus/caller/app.py](https://github.com/temporalio/samples-python/blob/main/hello_nexus/caller/app.py)。

## Nexus Operation 中的异常 {#exceptions-in-nexus-operations}

Temporal 提供了关于 [Nexus Operation 中错误](https://docs.temporal.io/references/failures#errors-in-nexus-operations)的一般指导。在 Python 中，有三个 Nexus 特定的异常类：

- [`nexusrpc.OperationError`](https://nexus-rpc.github.io/sdk-python/nexusrpc.OperationError.html)：这是你在 Nexus Operation 中应抛出的异常类型，表示 Operation 根据其自身应用逻辑失败，不应重试。
- [`nexusrpc.HandlerError`](https://nexus-rpc.github.io/sdk-python/nexusrpc.HandlerError.html)：可以在 Nexus Operation 中抛出此异常类型，并指定特定的 [HandlerErrorType](https://nexus-rpc.github.io/sdk-python/nexusrpc.HandlerErrorType.html)。错误将根据类型标记为可重试或不可重试，遵循 [Nexus 规范](https://github.com/nexus-rpc/api/blob/main/SPEC.md#predefined-handler-errors)。不可重试的 handler 错误类型为 `BAD_REQUEST`、`UNAUTHENTICATED`、`UNAUTHORIZED`、`NOT_FOUND`、`NOT_IMPLEMENTED`；可重试的类型为 `RESOURCE_EXHAUSTED`、`INTERNAL`、`UNAVAILABLE`、`UPSTREAM_TIMEOUT`。
- [`temporalio.exceptions.NexusOperationError`](https://python.temporal.io/temporalio.exceptions.NexusOperationError.html)：这是在 Nexus Operation 因任何原因失败时在 Workflow 内部抛出的错误。使用异常的 `__cause__` 属性访问原因链。

## 取消 Nexus Operation {#canceling-a-nexus-operation}

要从 Workflow 内部取消 Nexus Operation，请在 Operation handle 上调用 `handle.cancel()`。在 Nexus 中，只有异步 Operation 可以被取消，因为取消是通过 Operation 令牌发送的。支撑该 Operation 的 Workflow 或其他资源可以选择忽略取消请求。如果被忽略，Operation 可能进入终止状态。

当 Nexus Operation 启动时，caller 可以指定不同的取消类型，控制 caller 对取消的反应：

- `ABANDON` - 不请求取消 Operation。
- `TRY_CANCEL` - 发起取消请求并立即向 caller 报告取消。注意，如果 caller 在传递完成之前退出，此类型不保证取消被送达 Operation handler。
- `WAIT_REQUESTED` - 请求取消 Operation 并等待请求已被接收的确认。不等待实际取消完成。
- `WAIT_COMPLETED` - 等待 Operation 完成。Operation 可能以取消状态完成，也可能不是。

默认值为 `WAIT_COMPLETED`。用户可以在启动或执行 Operation 时为 `cancellation_type` 设置不同选项。

caller Workflow 完成后，caller 的 Nexus Machinery 不会再尝试取消仍在运行的 Operation。在某些使用场景中，保持 Operation 运行是可以接受的。为确保取消被送达，请在退出 Workflow 前等待所有待处理的 Operation 完成。

参考示例，请参阅 [Nexus 取消示例](https://github.com/temporalio/samples-python/tree/main/nexus_cancel)。

## 在 Temporal Cloud 上进行跨 Namespace 的 Nexus 调用 {#nexus-calls-across-namespaces-temporal-cloud}

本节假设你已熟悉如何将 Worker 连接到 Temporal Cloud。本节将使用 `tcld` CLI 创建 Namespace 和 Nexus Endpoint，并使用 mTLS 客户端证书将 caller 和 handler Worker 安全地连接到各自的 Temporal Cloud Namespace。

### 安装最新版 `tcld` CLI 并生成证书

要安装最新版 `tcld` CLI，在 macOS 上运行以下命令：

```
brew install temporalio/brew/tcld
```

如果还没有证书，可以使用以下命令生成用于 mTLS Worker 身份验证的证书：

```
tcld gen ca --org $YOUR_ORG_NAME --validity-period 1y --ca-cert ca.pem --ca-key ca.key
```

这些证书有效期为一年。

### 创建 caller 和 handler Namespace

在部署到 Temporal Cloud 之前，请确保已为 caller 和 handler 创建了合适的 Namespace。如果已有这些 Namespace，则无需执行此步骤。

```
tcld login

tcld namespace create \
	--namespace <your-caller-namespace> \
    --cloud-provider aws \
	--region us-west-2 \
	--ca-certificate-file 'path/to/your/ca.pem' \
	--retention-days 1

tcld namespace create \
	--namespace <your-target-namespace> \
    --cloud-provider aws \
	--region us-west-2 \
	--ca-certificate-file 'path/to/your/ca.pem' \
	--retention-days 1
```

或者，可以通过 UI 创建 Namespace：[https://cloud.temporal.io/Namespaces](https://cloud.temporal.io/Namespaces)。

### 创建 Nexus Endpoint 将请求从 caller 路由到 handler

要创建 Nexus Endpoint，必须拥有 Developer 或更高级别的账户角色，并对 `--target-namespace` 拥有 NamespaceAdmin 权限。

```
tcld nexus endpoint create \
  --name <my-nexus-endpoint-name> \
  --target-task-queue my-handler-task-queue \
  --target-namespace <my-target-namespace.account> \
  --allow-namespace <my-caller-namespace.account> \
  --description-file hello_nexus/endpoint_description.md
```

`--allow-namespace` 用于构建 Endpoint 允许列表，指定哪些 caller Namespace 可以使用该 Nexus Endpoint，如运行时访问控制所述。

或者，可以通过 UI 创建 Nexus Endpoint：[https://cloud.temporal.io/nexus](https://cloud.temporal.io/nexus)。

### Temporal CLI

使用 `workflow describe` 命令显示 caller Workflow 中待处理的 Nexus Operation 以及 handler Workflow 上附加的任何回调：

```
temporal workflow describe -w <ID>
```

Nexus 事件包含在 caller 的 Workflow 历史中：

```
temporal workflow show -w <ID>
```

对于**异步 Nexus Operation**，以下内容记录在 caller 的历史中：

- `NexusOperationScheduled`
- `NexusOperationStarted`
- `NexusOperationCompleted`

对于**同步 Nexus Operation**，以下内容记录在 caller 的历史中：

- `NexusOperationScheduled`
- `NexusOperationCompleted`

:::note

对于同步 Operation，`NexusOperationStarted` 不会记录在 caller 的历史中。

:::
