Workflow 可以像有状态 Web 服务一样接收消息：查询、Signal 和 Update。
Workflow 实现通过可以对传入消息和返回值做出反应的处理程序方法定义这些端点。
Temporal CLIents 使用消息来读取 Workflow 状态并控制其执行。
有关此主题的一般概述，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing)。
本页介绍 Temporal Python SDK 的这些功能。

<a id="writing-message-handlers"></a>
## 编写消息处理程序 {#writing-message-handlers}
:::info
下面的代码是传递 [样本](https://github.com/temporalio/samples-python/tree/main/message_passing/introduction) 的工作消息的一部分。
:::

编写消息处理程序时请遵循以下准则：

- 消息处理程序被定义为 Workflow 类上的方法，使用以下三个装饰器之一：[`@Workflow.Query`](https://python.temporal.io/temporalio.workflow.html#query)、[`@Workflow.Signal`](https://python.temporal.io/temporalio.workflow.html#signal) 和 [`@Workflow.Update`](https://python.temporal.io/temporalio.workflow.html#update)。
- 处理程序和主 Workflow 函数的参数和返回值必须是 [可序列化](/dataconversion)。
- 优先选择 [数据类](https://docs.python.org/3/library/dataclasses.html) 而非多个输入参数。数据类参数允许您在不更改调用签名的情况下添加字段。请记住，如果新字段没有默认值，则序列化和反序列化可能会失败并显示默认的 Data Converter。

<a id="queries"></a>
### Query 处理程序 {#queries}
[Query](/sending-messages#sending-queries) 是从 Workflow Execution 检索状态的同步操作：

```python
class Language(IntEnum):
    Chinese = 1
    English = 2
    French = 3

@dataclass
class GetLanguagesInput:
    include_unsupported: bool

@workflow.defn
class GreetingWorkflow:
    def __init__(self) -> None:
        self.greetings = {
            Language.CHINESE: "你好，世界",
            Language.ENGLISH: "Hello, world",
        }

    @workflow.query
    def get_languages(self, input: GetLanguagesInput) -> list[Language]:
        # 👉 A Query handler returns a value: it can inspect but must not mutate the Workflow state.
        if input.include_unsupported:
            return list(Language)
        else:
            return list(self.greetings)
```

- Query 装饰器可以接受参数。
  请参阅 API 文档：[`@Workflow.Query`](https://python.temporal.io/temporalio.workflow.html#query)。

- Query 处理程序使用 `def`，而不是 `async def`。
  您无法执行异步操作，例如在 Query 处理程序中执行 Activity。

<a id="signals"></a>
### Signal 处理程序 {#signals}
[Signal](/sending-messages#sending-signals) 是发送到正在运行的 Workflow Execution 的异步消息，用于更改其状态并控制其流程：

```python
@dataclass
class ApproveInput:
    name: str

@workflow.defn
class GreetingWorkflow:
    ...
    @workflow.signal
    def approve(self, input: ApproveInput) -> None:
        # 👉 A Signal handler mutates the Workflow state but cannot return a value.
        self.approved_for_release = True
        self.approver_name = input.name
```

- Signal 装饰器可以接受参数。
  请参阅 API 文档：[`@Workflow.Signal`](https://python.temporal.io/temporalio.workflow.html#signal)。

- 处理程序不应返回值。
  服务器立即发送响应，无需等待 Workflow 处理 Signal。

- Signal（和 Update）处理程序可以是 `async def`。
  这允许您使用活动、Child Workflow、持久 [`asyncio.sleep`](https://docs.python.org/3/library/asyncio-task.html#asyncio.sleep)、Timer、[`Workflow.wait_condition`](https://python.temporal.io/temporalio.workflow.html#wait_condition) 条件等。
  有关安全使用异步 Signal 和 Update 处理程序的指南，请参阅 [异步处理程序](#async-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing)。

<a id="updates"></a>
### Update 处理程序和验证器 {#updates}
[Update](/sending-messages#sending-updates) 是发送到正在运行的 Workflow Execution 的可跟踪同步请求。
它可以更改 Workflow 状态、控制其流程并返回结果。
发送方必须等待，直到 Worker 接受或拒绝 Update。
如果出现问题，发送者可能会进一步等待接收返回值或异常：

```python
class Language(IntEnum):
    Chinese = 1
    English = 2
    French = 3

@workflow.defn
class GreetingWorkflow:
    ...
    @workflow.update
    def set_language(self, language: Language) -> Language:
        # 👉 An Update handler can mutate the Workflow state and return a value.
        previous_language, self.language = self.language, language
        return previous_language

    @set_language.validator
    def validate_language(self, language: Language) -> None:
        if language not in self.greetings:
            # 👉 In an Update validator you raise any exception to reject the Update.
            raise ValueError(f"{language.name} is not supported")
```

- Update 装饰器可以采用参数（例如 `name`、`dynamic` 和 `unfinished_policy`），如 [`Workflow.Update`](https://python.temporal.io/temporalio.workflow.html#update) 的 API 参考文档中所述。

- 关于验证器：
  - 在将 Update 写入历史记录之前，使用验证器拒绝该 Update。
    验证器始终是可选的。
    如果您不需要拒绝 Update，则可以跳过它们。
  - SDK 自动提供一个名为 `@<Update-handler-name>.validator` 的验证器装饰器。
    验证器必须接受与处理程序相同的参数类型并返回 `None`。

- 使用验证器接受和拒绝 Update：
  - 要拒绝 Update，请在验证器中引发任何类型的异常。
  - 如果没有验证器，则始终接受 Update。
- 验证器和 Event History：
  - 无论是自动接受还是编程接受，`WorkflowExecutionUpdateAccepted` 事件都会写入历史记录。
  - 当验证器引发错误时，Update 会被拒绝，并且 `WorkflowExecutionUpdateAccepted` _不会_ 添加到 Event History 中。
    调用者收到“Update 失败”错误。

- 使用 [`Workflow.current_Update_info`](https://python.temporal.io/temporalio.workflow.html#current_update_info) 获取有关当前 Update 的信息。
  这包括 Update ID，它在使用 Continue-As-New 时可用于重复数据删除：请参阅 [确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing)。
- Update（和 Signal）处理程序可以是 `async def`，让它们使用活动、Child Workflow、持久 [`asyncio.sleep`](https://docs.python.org/3/library/asyncio-task.html#asyncio.sleep) Timer、[`Workflow.wait_condition`](https://python.temporal.io/temporalio.workflow.html#wait_condition) 条件等。
  请参阅 [异步处理程序](#async-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing) 了解安全使用指南。

<a id="send-messages"></a>
## 发送消息 {#send-messages}
要发送查询、Signal 或 Update，您可以调用 [Workflow手柄](https://python.temporal.io/temporalio.client.WorkflowHandle.html) 对象上的方法：

- 使用 [开始_Workflow](https://python.temporal.io/temporalio.client.Client.html#start_workflow) 启动 Workflow 并返回其句柄。

- 使用 [get_Workflow_handle_for](https://python.temporal.io/temporalio.client.Client.html#get_workflow_handle_for) 通过 Workflow Id 检索类型化的 Workflow 句柄。

例如：

```python
client = await Client.connect("localhost:7233")
workflow_handle = await client.start_workflow(
    GreetingWorkflow.run, id="greeting-workflow-1234", task_queue="my-task-queue"
)
```

要检查发送消息时所需的参数类型以及查询和 Update 的返回类型，请参阅 Workflow Definition 中相应的处理程序方法。

:::warning Using Continue-as-New and Updates

- Temporal_不_支持 Update 处理程序中的 Continue-As-New 功能。
- 使用 Continue-As-New_之前_完成所有处理程序。
- 从您的主要 Workflow Definition 方法中使用 Continue-As-New，就像完成或失败 Workflow Execution 一样。

:::

<a id="send-query"></a>
### 发送 Query {#send-query}
使用 [`WorkflowHandle.Query`](https://python.temporal.io/temporalio.client.WorkflowHandle.html#query) 将 Query 发送到 Workflow Execution：

```python
supported_languages = await workflow_handle.query(
    GreetingWorkflow.get_languages, GetLanguagesInput(supported_only=True)
)
```

- 发送 Query 不会将事件添加到 Workflow 的 Event History。

- 您可以在 Namespace 的 Workflow 保留期内向已关闭的 Workflow Execution 发送查询。
  这包括已完成、失败或超时的 Workflow。
  Querying 终止的 Workflow 不安全，因此不受支持。

- Worker 必须在线并轮询 Task Queue 才能处理 Query。

<a id="send-signal"></a>
### 发送 Signal {#send-signal}
您可以从 Temporal CLIent 或另一个 Workflow Execution 向 Workflow Execution 发送 Signal。
但是，您只能将 Signal 发送到尚未关闭的 Workflow Execution。

<a id="send-signal-from-client"></a>
#### 从客户端发送 Signal {#send-signal-from-client}
使用 [`WorkflowHandle.Signal`](https://python.temporal.io/temporalio.client.WorkflowHandle.html#signal) 发送 Signal：

```python
await workflow_handle.signal(GreetingWorkflow.approve, ApproveInput(name="me"))
```

- 当服务器接受 Signal 时调用返回；它并不等待 Signal 被传送到 Workflow Execution。

- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在 Workflow 的 Event History 中。

<a id="send-signal-from-workflow"></a>
#### 从 Workflow 发送 Signal {#send-signal-from-workflow}
Workflow 可以将 Signal 发送到另一个 Workflow，称为_外部 Signal_。
您需要一个用于外部 Workflow 的 Workflow 手柄。
使用 [`get_external_Workflow_handle_for`](https://python.temporal.io/temporalio.workflow.html#get_external_workflow_handle_for)：

<div style={{backgroundColor: '#e0e0e0',padding: '0px 15px',borderRadius: '5px',border: '1px solid #cccccc',display: 'inline-block'}}><a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/signal_your_workflow/signal_external_wf_dacx.py"><tt>See full sample</tt></a></div>

```python
# ...
@workflow.defn
class WorkflowB:
    @workflow.run
    async def run(self) -> None:
        handle = workflow.get_external_workflow_handle_for(WorkflowA.run, "workflow-a")
        await handle.signal(WorkflowA.your_signal, "signal argument")
```

当发送外部 Signal 时：

- [Signal外部Workflow执行已启动](/references/events#signalexternalworkflowexecutioninitiated) 事件出现在发件人的 Event History 中。
- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在收件人的 Event History 中。

<a id="signal-with-start"></a>
#### Signal-带启动 {#signal-with-start}
Signal-With-Start 允许客户端将 Signal 发送到 Workflow Execution，如果尚未运行则启动执行。
要使用 Signal-With-Start，请调用 [`start_Workflow`](https://python.temporal.io/temporalio.client.Client.html#start_workflow) 方法并传递带有 Signal 名称的 `start_Signal` 参数：

<div style={{backgroundColor: '#e0e0e0',padding: '0px 15px',borderRadius: '5px',border: '1px solid #cccccc',display: 'inline-block'}}><a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/signal_your_workflow/signal_with_start_dacx.py"><tt>See full sample</tt></a></div>

```python
from temporalio.client import Client
# ...
async def main():
    client = await Client.connect("localhost:7233")
    await client.start_workflow(
        GreetingWorkflow.run,
        id="your-signal-with-start-workflow",
        task_queue="signal-tq",
        start_signal="submit_greeting",
        start_signal_args=["User Signal with Start"],
    )
```

<a id="send-update-from-client"></a>
### 发送 Update {#send-update-from-client}
Update 是同步、阻塞调用，可以更改 Workflow 状态、控制其流程并返回结果。

发送 Update 的客户端必须等待，直到服务器将 Update 传递给 Worker。
Worker 必须可用且响应迅速。
如果服务器收到请求后需要立即响应，请改用 Signal。
另请注意，您无法将 Update 发送到其他 Workflow Execution。

- 当Worker确认Update通过验证时，`WorkflowExecutionUpdateAccepted`被添加到Event History。
- 当Worker确认Update已完成时，`WorkflowExecutionUpdateCompleted`被添加到Event History。

要将 Update 发送到 Workflow Execution，您可以：

- 呼叫 [`execute_Update`](https://python.temporal.io/temporalio.client.WorkflowHandle.html#execute_update) 并等待 Update 完成。
  此代码获取 Update 结果：

  ```python
  previous_language = await workflow_handle.execute_update(
      GreetingWorkflow.set_language, Language.Chinese
  )
  ```

- 一旦 Update 被接受，请发送 [`start_Update`](https://python.temporal.io/temporalio.client.WorkflowHandle.html#start_update) 以收到 [`UpdateHandle`](https://python.temporal.io/temporalio.client.WorkflowUpdateHandle.html)。

- 稍后使用此 `UpdateHandle` 获取结果。
  - `async def` Update 处理程序通常执行长时间运行的异步操作，例如执行 Activity。
  - `start_Update` 仅等待直到 Worker 接受或拒绝 Update，而不等待所有异步操作完成。

例如：

  ```python
  # Wait until the update is accepted
  update_handle = await workflow_handle.start_update(
      HelloWorldWorkflow.set_greeting,
      HelloWorldInput("World"),
      wait_for_stage=client.WorkflowUpdateStage.ACCEPTED,
  )
  # Wait until the update is completed
  update_result = await update_handle.result()
  ```

有关更多详细信息，请参阅“异步处理程序”部分。

要获取 Update 句柄，您可以：

- 使用 [`start_Update`](https://python.temporal.io/temporalio.client.WorkflowHandle.html#start_update) 启动 Update 并返回句柄，如前面的示例所示。
- 使用 [`get_Update_handle_for`](https://python.temporal.io/temporalio.client.WorkflowHandle.html#get_update_handle_for) 使用 Update ID 获取正在进行的 Update 的句柄。

<a id="update-with-start"></a>
#### Update-带启动 {#update-with-start}
:::tip

对于开源服务器用户，建议使用Temporal Server版本[Temporal Server 版本 1.28](https://github.com/temporalio/temporal/releases/tag/v1.28.0)。

:::

[Update-带启动](/sending-messages#update-with-start) 让您
[发送 Update](/develop/python/workflows/message-passing#send-update-from-client) 检查是否存在具有该 ID 的已运行的 Workflow：

- 如果 Workflow 存在，则处理 Update。
- 如果 Workflow 不存在，则使用给定 ID 启动新的 Workflow Execution，并在主 Workflow 方法开始执行之前处理 Update。

使用[`execute_Update_with_start_Workflow`](https://python.temporal.io/temporalio.client.Client.html#start_update_with_start_workflow)启动Update，一次性等待结果。

或者，使用 [`start_Update_with_start_Workflow`](https://python.temporal.io/temporalio.client.Client.html#start_update_with_start_workflow) 启动 Update 并接收 [`WorkflowUpdateHandle`](https://python.temporal.io/temporalio.client.WorkflowUpdateHandle.html)，然后使用 `await Update_handle.result()` 从 Update 检索结果。

一旦到达所请求的 Update 等待阶段或请求超时，这些调用就会返回。

您需要提供 [`WithStartWorkflowOperation`](https://python.temporal.io/temporalio.client.WithStartWorkflowOperation.html) 来定义必要时启动的 Workflow 及其参数。
创建 `WithStartWorkflowOperation` 时必须指定 [WorkflowId冲突政策](/workflow-execution/workflowid-runid#workflow-id-conflict-policy)。
请注意，`WithStartWorkflowOperation` 只能使用一次。

以下是取自 [惰性初始化](https://github.com/temporalio/samples-python/blob/main/message_passing/update_with_start/lazy_initialization/starter.py) 样本的示例：

```python
start_op = WithStartWorkflowOperation(
    ShoppingCartWorkflow.run,
    id=cart_id,
    id_conflict_policy=common.WorkflowIDConflictPolicy.USE_EXISTING,
    task_queue="my-task-queue",
)
try:
    price = Decimal(
        await temporal_client.execute_update_with_start_workflow(
            ShoppingCartWorkflow.add_item,
            ShoppingCartItem(sku=item_id, quantity=quantity),
            start_workflow_operation=start_op,
        )
    )
except WorkflowUpdateFailedError:
    price = None

workflow_handle = await start_op.workflow_handle()

return price, workflow_handle
```

<a id="async-handlers"></a>
### 使用异步处理程序 {#async-handlers}
Signal 和 Update 处理程序可以是 `async def` 以及 `def`。
使用 `async def` 允许您将 `await` 与活动、Child Workflow、[`asyncio.sleep`](https://docs.python.org/3/library/asyncio-task.html#asyncio.sleep) Timer、[`Workflow.wait_condition`](https://python.temporal.io/temporalio.workflow.html#wait_condition) 条件等一起使用。
这扩展了处理程序可以完成的操作的可能性，但这也意味着处理程序执行和主 Workflow 方法都是同时运行的，并在 `await` 调用时在它们之间进行切换。
为了安全使用 `async def` 处理程序，了解可能出现问题的情况至关重要。
有关异步 Signal 和 Update 处理程序、[安全消息处理程序](/encyclopedia/workflow-message-passing) 示例以及下面的 [控制处理程序并发](https://github.com/temporalio/samples-python/tree/main/message_passing/safe_message_handlers) 和 [等待消息处理程序完成](#control-handler-concurrency) 部分的安全使用指南，请参阅 [Workflow消息传递](#wait-for-message-handlers)。

以下代码执行 Activity 对远程服务进行网络调用。
它修改了本页前面的 Update 处理程序，将其转换为 `async def`：

```python
@activity.defn
async def call_greeting_service(to_language: Language) -> Optional[str]:
    await asyncio.sleep(0.2)  # Pretend that we are calling a remote service.
    greetings = {
        Language.Arabic: "مرحبا بالعالم",
        Language.Chinese: "你好，世界",
        Language.English: "Hello, world",
        Language.French: "Bonjour, monde",
        Language.Hindi: "नमस्ते दुनिया",
        Language.Spanish: "Hola mundo",
    }
    return greetings.get(to_language)


@workflow.defn
class GreetingWorkflow:
    def __init__(self) -> None:
        self.lock = asyncio.Lock()
        ...
    ...
    @workflow.update
    async def set_language(self, language: Language) -> Language:
        if language not in self.greetings:
            # 👉 Use a lock here to ensure that multiple calls to set_language are processed in order.
            async with self.lock:
                greeting = await workflow.execute_activity(
                    call_greeting_service,
                    language,
                    start_to_close_timeout=timedelta(seconds=10),
                )
                if greeting is None:
                    # 👉 An update validator cannot be async, so cannot be used to check that the remote
                    # call_greeting_service supports the requested language. Raising ApplicationError
                    # will fail the Update, but the WorkflowExecutionUpdateAccepted event will still be
                    # added to history.
                    raise ApplicationError(
                        f"Greeting service does not support {language.name}"
                    )
                self.greetings[language] = greeting
        previous_language, self.language = self.language, language
        return previous_language
```

更新代码以使用 `async def` 后，您的 Update 处理程序可以使用 Schedule 和 Activity 并等待结果。
尽管 `async def` Signal 处理程序也可以执行 Activity，但使用 Update 处理程序允许客户端在 Activity 完成后接收结果或错误。
这使您的客户端可以跟踪 Update 的活动、Child Workflow 等执行的异步工作的进度。

<a id="add-wait-conditions-to-block"></a>
### 添加阻塞等待条件 {#add-wait-conditions-to-block}

有时，`async def` Signal 或 Update 处理程序需要满足某些条件才能继续。
您可以使用 [`Workflow.wait_condition`](https://python.temporal.io/temporalio.workflow.html#wait_condition) 来阻止代码继续执行，直到条件成立为止。
您可以通过传递返回 `True` 或 `False` 的函数来指定条件，并且可以选择设置超时。
这是一个重要的功能，可以帮助您控制处理程序逻辑。

以下是 `Workflow.wait_condition` 的三个重要用例：

- 等待 Signal 或 Update 到达。
- 在处理程序中等待，直到适合继续。
- 在主 Workflow 中等待，直到所有活动处理程序完成。

<a id="wait-for-a-signal-or-update-to-arrive"></a>
#### 等待 Signal 或 Update 到达 {#wait-for-a-signal-or-update-to-arrive}

通常使用 `Workflow.condition` 来等待客户端发送特定的 Signal 或 Update：

```python
@workflow.defn
class GreetingWorkflow:
    def __init__(self) -> None:
        self.approved_for_release = False
        self.approver_name: Optional[str] = None

    @workflow.signal
    def approve(self, input: ApproveInput) -> None:
        self.approved_for_release = True
        self.approver_name = input.name

    @workflow.run
    async def run(self) -> str:
        await workflow.wait_condition(lambda: self.approved_for_release)
        ...
        return self.greetings[self.language]
```

<a id="wait-in-message-handler"></a>
#### 在处理程序中使用等待条件 {#wait-in-message-handler}
通常使用 Workflow 等待条件来等待处理程序启动。
您还可以在处理程序中的其他任何位置使用等待条件来等待特定条件变为 `True`。
这允许您编写在多个点暂停的处理程序，每次都等待所需的条件变为 `True`。

考虑在 Update 处理程序执行之前运行的 `ready_for_Update_to_execute` 方法。
`Workflow.wait_condition` 方法会等待，直到满足您的条件：

```python
@workflow.update
async def my_update(self, update_input: UpdateInput) -> str:
    await workflow.wait_condition(
        lambda: self.ready_for_update_to_execute(update_input)
    )
```

您还可以在处理程序中的任何其他位置使用等待条件来等待特定条件变为真。
这允许您编写在多个点暂停的处理程序，每次都等待所需的条件变为真。

<a id="wait-for-message-handlers"></a>
#### 确保您的处理程序在 Workflow 完成 {#wait-for-message-handlers} 之前完成
Workflow 等待条件可以确保您的处理程序在 Workflow 完成之前完成。
当您的 Workflow 使用 `async def` Signal 或 Update 处理程序时，您的主 Workflow 方法可以在处理程序仍在等待异步任务（例如 Activity 结果）时返回 或 Continue-As-New 。
Workflow 完成可能会在处理程序完成关键工作之前中断处理程序，并在尝试检索 Update 结果时导致客户端错误。
使用[`Workflow.wait_condition`](https://python.temporal.io/temporalio.workflow.html#wait_condition)和[`all_handlers_finished`](https://python.temporal.io/temporalio.workflow.html#all_handlers_finished)来解决这个问题，让你的Workflow顺利结束：

```python
@workflow.defn
class MyWorkflow:
    @workflow.run
    async def run(self) -> str:
        ...
        await workflow.wait_condition(workflow.all_handlers_finished)
        return "workflow-result"
```

默认情况下，当您允许 Workflow Execution 完成未完成的处理程序执行时，您的 Worker 将记录一条警告。
您可以通过将 `unfinished_policy` 参数传递给 [`@Workflow.Signal`](https://python.temporal.io/temporalio.workflow.html#signal) / [`Workflow.Update`](https://python.temporal.io/temporalio.workflow.html#update) 装饰器来在每个处理程序的基础上消除这些警告：

```python
@workflow.update(unfinished_policy=workflow.HandlerUnfinishedPolicy.ABANDON)
async def my_update(self) -> None:
    ...
```

请参阅 [在 Workflow 完成之前完成处理程序](/handling-messages#finishing-message-handlers) 了解更多信息。

<a id="use-workflowinit-to-operate-on-workflow-input-before-any-handler-executes"></a>
### 在执行任何处理程序之前，使用 `@Workflow.init` 对 Workflow 输入进行操作 {#use-workflowinit-to-operate-on-workflow-input-before-any-handler-executes}

通常，您的 Workflow `__init__` 方法不会有任何参数。
但是，如果您在 `__init__` 方法上使用 `@Workflow.init` 装饰器，则可以为其指定与 `@Workflow.run` 方法相同的 [Workflow参数](/develop/python/workflows/basics#workflow-parameters)。
然后，SDK 将确保您的 `__init__` 方法接收 [客户发送](/develop/python/client/temporal-client#start-workflow-execution) 的 Workflow 输入参数。
（Workflow 输入参数也会传递给您的 `@Workflow.run` 方法——无论您是否使用 `@Workflow.init` 装饰器，这种情况总是会发生。）
如果您有需要访问 Workflow 输入的消息处理程序，这非常有用：请参阅 [首先初始化Workflow](/handling-messages#workflow-initializers)。

这是一个例子。
请注意，`__init__` 和 `get_greeting` 必须具有相同的参数和相同的类型注释：

```python
@dataclass
class MyWorkflowInput:
    name: str


@workflow.defn
class WorkflowRunSeesWorkflowInitWorkflow:
    @workflow.init
    def __init__(self, workflow_input: MyWorkflowInput) -> None:
        self.name_with_title = f"Sir {workflow_input.name}"
        self.title_has_been_checked = False

    @workflow.run
    async def get_greeting(self, workflow_input: MyWorkflowInput) -> str:
        await workflow.wait_condition(lambda: self.title_has_been_checked)
        return f"Hello, {self.name_with_title}"

    @workflow.update
    async def check_title_validity(self) -> bool:
        # 👉 The handler is now guaranteed to see the workflow input
        # after it has been processed by __init__.
        is_valid = await workflow.execute_activity(
            check_title_validity,
            self.name_with_title,
            schedule_to_close_timeout=timedelta(seconds=10),
        )
        self.title_has_been_checked = True
        return is_valid
```

<a id="control-handler-concurrency"></a>
### 使用 `asyncio.Lock` 阻止并发处理程序执行 {#control-handler-concurrency}
并发进程可以以不可预测的方式交互。
当多个处理程序实例同时运行时，编写不正确的 [并发消息传递](/handling-messages#message-handler-concurrency) 代码可能无法正常工作。
这是一个病态病例的例子：

```python
@workflow.defn
class MyWorkflow:

    @workflow.signal
    async def bad_async_handler(self):
        data = await workflow.execute_activity(
            fetch_data, start_to_close_timeout=timedelta(seconds=10)
        )
        self.x = data.x
        # 🐛🐛 Bug!! If multiple instances of this handler are executing concurrently, then
        # there may be times when the Workflow has self.x from one Activity execution and self.y from another.
        await asyncio.sleep(1)  # or await anything else
        self.y = data.y
```

使用 `asyncio.Lock` 协调访问可更正此代码。
锁定可确保在任何给定时间只有一个处理程序实例可以执行特定的代码部分：

```python
@workflow.defn
class MyWorkflow:
    def __init__(self) -> None:
        ...
        self.lock = asyncio.Lock()
        ...

    @workflow.signal
    async def safe_async_handler(self):
        async with self.lock:
            data = await workflow.execute_activity(
                fetch_data, start_to_close_timeout=timedelta(seconds=10)
            )
            self.x = data.x
            # ✅ OK: the scheduler may switch now to a different handler execution, or to the main workflow
            # method, but no other execution of this handler can run until this execution finishes.
            await asyncio.sleep(1)  # or await anything else
            self.y = data.y
```

<a id="message-handler-troubleshooting"></a>
## 消息处理程序故障排除 {#message-handler-troubleshooting}
将 Signal、Update 或 Query 发送到 Workflow 时，您的客户端可能会遇到以下错误：

- **客户端无法联系服务器**：
  您将收到一个 [`Temporalio.service.RPCError`](https://python.temporal.io/temporalio.service.RPCError.html)，其 `status` 属性为 [`RPCStatusCode`](https://python.temporal.io/temporalio.service.RPCStatusCode.html) `UNAVAILABLE`（重试几次后；请参阅 [`Client.connect`](https://python.temporal.io/temporalio.client.Client.html#connect) 的 `retry_config` 参数）。

- **Workflow 不存在**：
  您将收到 [`Temporalio.service.RPCError`](https://python.temporal.io/temporalio.service.RPCError.html) 异常，其中 `status` 属性为 [`RPCStatusCode`](https://python.temporal.io/temporalio.service.RPCStatusCode.html) `NOT_FOUND`。

有关此主题的非特定于 Python 的讨论，请参阅 [消息处理程序中的异常](/handling-messages#exceptions)。

<a id="signal-problems"></a>
### 发送 Signal 时出现问题 {#signal-problems}
使用 Signal 时，您的请求在执行期间唯一会导致的异常是上述 `RPCError`。

对于查询和 Update，客户端等待来自 Worker 的响应，因此在 Worker 执行处理程序期间可能会出现其他错误。

<a id="update-problems"></a>
### 发送 Update 时出现问题 {#update-problems}
使用 Update 时，除了上述 `RPCError` 之外，您还可能会遇到以下错误：

- **没有 Workflow Worker 正在轮询 Task Queue**：
  SDK 客户端将无限期地重试您的请求。
  您可以使用 [`asyncio.timeout`](https://docs.python.org/3/library/asyncio-task.html#timeouts) 强制超时。
  这会引发 [`Temporalio.client.WorkflowUpdateRPCTimeoutOrCancelledError`](https://python.temporal.io/temporalio.client.WorkflowUpdateRPCTimeoutOrCancelledError.html) 异常。

- **Update 失败**：您将收到 [`Temporalio.client.WorkflowUpdateFailedError`](https://python.temporal.io/temporalio.client.WorkflowUpdateFailedError.html) 异常。
  有两种方式可以发生这种情况：

- Update 被 Workflow 中与 Update 处理程序一起定义的 Update 验证器拒绝。

- Update 被接受后失败。

Update 故障类似于 [Workflow 故障](/references/failures#errors-in-workflows)。
  导致 main 方法中的 Workflow 失败的问题也会导致 Update 处理程序中的 Update 失败。
  这些可能包括：

- 失败的Child Workflow
      - 失败的 Activity（如果 Activity 重试已设置为有限次数）
      - Workflow 作者筹集了 [`ApplicationError`](/references/failures#application-failure)
      - [Workflow_failure_exception_types](https://python.temporal.io/temporalio.worker.Worker.html) 中列出的任何错误（默认为空）

- **处理程序导致 Workflow 任务失败**：
  [Workflow 任务失败](/references/failures#errors-in-workflows) 导致服务器无限期地重试 Workflow 任务。您的 Update 请求会发生什么情况取决于其阶段：
  - 如果服务器未接受请求，您将收到 `FAILED_PRECONDITION` [`Temporalio.service.RPCError`](https://python.temporal.io/temporalio.service.RPCError.html) 异常。
  - 如果请求已被接受，则该请求是持久的。
    代码部署后，一旦 Workflow 再次正常运行，请使用 [`UpdateHandle`](https://python.temporal.io/temporalio.client.WorkflowUpdateHandle.html) 获取 Update 结果。

- **Workflow 已完成，而 Update 处理程序执行正在进行中**：
  您将收到 [`Temporalio.service.RPCError`](https://python.temporal.io/temporalio.service.RPCError.html) 异常，其 `status` 属性为 [`RPCStatusCode`](https://python.temporal.io/temporalio.service.RPCStatusCode.html) `NOT_FOUND`。
  如果 Workflow 完成而 Update 处理程序执行正在进行中，则会发生这种情况，例如因为

- Workflow 已取消或失败。

- Workflow 正常完成或继续如新，而 Workflow 作者没有 [等待处理程序完成](/handling-messages#finishing-message-handlers)。

<a id="query-problems"></a>
### 发送 Query 时出现问题 {#query-problems}
使用查询时，除了上述 `RPCError` 之外，您还可能会遇到以下错误：

- **没有 Workflow Worker 轮询 Task Queue**：
  您将收到 [`Temporalio.service.RPCError`](https://python.temporal.io/temporalio.service.RPCError.html) 异常，其中 `status` 属性为 [`RPCStatusCode`](https://python.temporal.io/temporalio.service.RPCStatusCode.html) `FAILED_PRECONDITION`。

- **Query 失败**：
  如果 Query 期间出现问题，您将收到 [`Temporalio.client.WorkflowQueryFailedError`](https://python.temporal.io/temporalio.client.WorkflowQueryFailedError.html) 异常。
  Query 处理程序中的任何异常都会触发此错误。
  这与 Signal 和 Update 请求不同，其中异常可能会导致 Workflow 任务失败。

- **处理程序导致 Workflow 任务失败。**
  例如，如果 Query 处理程序阻塞线程太长时间而没有屈服，就会发生这种情况。

<a id="dynamic-handler"></a>
## 动态组件 {#dynamic-handler}
动态 Workflow、Activity、Signal、Update 或 Query 是一种未命名项目。
通常，这些项目按名称注册到 Worker 并在运行时调用。
当未注册或无法识别的 Workflow、Activity 或带有已识别方法签名的消息请求到达时，Worker 可以使用预先注册的动态替代。

例如，您可以发送请求来启动名为​​“MyUnknownWorkflow”的 Workflow。
收到 Workflow 任务后，Worker 可能会发现没有已注册的该类型的 Workflow Definition。
然后，它检查是否存在已注册的动态 Workflow。
如果动态 Workflow 签名与传入的 Workflow 签名匹配，则 Worker 会调用该签名，就像调用非动态静态命名版本一样。

通过注册 Temporal 组件的动态版本，Worker 可以回退到这些名称不匹配的替代实现。

:::caution

明智地使用动态元素并将其作为后备机制，而不是主要设计。
它们可能会带来长期的可维护性和调试问题。
保留动态调用用于在编译时不知道或无法知道名称的情况。

:::

<a id="set-a-dynamic-signal"></a>
### 设置动态 Signal、Query 或 Update 处理程序 {#set-a-dynamic-signal}
动态 Signal、Query 或 Update 是指特殊的替代处理程序。
当未注册的处理程序请求到达时使用它。

考虑 Signal，您可以在其中发送类似 `Workflow.Signal(MyWorkflow.my_Signal_method, my_arg)` 的内容。
这是一种类型安全的编译器检查方法，可保证方法存在。
还有一种基于非类型安全字符串的形式：`Workflow.Signal('some-name', my_arg)`。
当发送到服务器时，仅在到达 Worker 后才检查名称。
这就是“动态处理程序”的用武之地。

在找不到名称和类型匹配的处理程序后，Worker 将检查已注册的动态替代处理程序。
如果找到，Worker 将使用它。

您必须选择处理程序进行动态访问。
将 `dynamic=True` 添加到处理程序装饰器（例如，`@Workflow.Signal(dynamic=True)`）以使处理程序动态化。
处理程序的签名必须接受 `(self, name: str, args: Sequence[RawValue])`。
使用 [负载转换器](https://python.temporal.io/temporalio.workflow.html#payload_converter) 函数将 `RawValue` 对象转换为您所需的类型。
例如：

```python
from typing import Sequence

from temporalio.common import RawValue
...

    @workflow.signal(dynamic=True)
    async def dynamic_signal(self, name: str, args: Sequence[RawValue]) -> None:
        ...
```

此示例创建 `dynamic_Signal` Signal。
当未注册或无法识别的 Signal 到达时带有匹配的签名，动态分配使用此处理程序来管理 Signal。
它负责将序列内容转换为方法逻辑可以处理和操作的可用数据。

<a id="set-a-dynamic-workflow"></a>
### 设置动态 Workflow {#set-a-dynamic-workflow}
动态Workflow是指特殊的替身Workflow Definition。
当未知的 Workflow Execution 请求到达时使用它。

考虑前面描述的“MyUnknownWorkflow”示例。
Worker 可能会发现没有该名称或类型的已注册 Workflow Definition。
在找不到具有匹配类型的 Workflow Definition 后，Worker 会寻找动态替身。
如果找到，它会调用它。

要参与，您的 Workflow 必须选择动态访问。
将 `dynamic=True` 添加到 `@Workflow.defn` 装饰器使得 Workflow Definition 有资格参与动态调用。
您必须先向 [Worker](https://python.temporal.io/temporalio.worker.html) 注册 Workflow，然后才能调用它。

Workflow Definition 的主要 Workflow 方法必须接受 `Sequence[Temporalio.common.RawValue]` 类型的单个参数。
使用 [负载转换器](https://python.temporal.io/temporalio.workflow.html#payload_converter) 函数将 `RawValue` 对象转换为您所需的类型。
例如：

<div style={{backgroundColor: '#e0e0e0',padding: '0px 15px',borderRadius: '5px',border: '1px solid #cccccc',display: 'inline-block'}}><a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/dynamic_handlers/your_dynamic_workflow_dacx.py"><tt>See full sample</tt></a></div>

```python
# ...
@workflow.defn(dynamic=True)
class DynamicWorkflow:
    @workflow.run
    async def run(self, args: Sequence[RawValue]) -> str:
        name = workflow.payload_converter().from_payload(args[0].payload, str)
        return await workflow.execute_activity(
            default_greeting,
            YourDataClass("Hello", name),
            start_to_close_timeout=timedelta(seconds=10),
        )
```

此 Workflow 将第一个 `Sequence` 元素转换为字符串，并使用它来执行 Activity。

<a id="set-a-dynamic-activity"></a>
### 设置动态 Activity {#set-a-dynamic-activity}
动态 Activity 是一个替代实现。
当 Worker 接收到未知 Activity 类型的 Activity 任务时，将使用它。

要参与，您的 Activity 必须选择动态访问。
将 `dynamic=True` 添加到 `@Activity.defn` 装饰器使得 Activity Definition 有资格参与动态调用。
您必须先向 [Worker](https://python.temporal.io/temporalio.worker.html) 注册 Activity，然后才能调用它。

然后，Activity Definition 必须接受 `Sequence[Temporalio.common.RawValue]` 类型的单个参数。
使用 [负载转换器](https://python.temporal.io/temporalio.activity.html#payload_converter) 函数将 `RawValue` 对象转换为您所需的类型。
例如：

<div style={{backgroundColor: '#e0e0e0',padding: '0px 15px',borderRadius: '5px',border: '1px solid #cccccc',display: 'inline-block'}}><a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/dynamic_handlers/your_dynamic_activity_dacx.py"><tt>See full sample</tt></a></div>

```python
# ...
@activity.defn(dynamic=True)
async def dynamic_greeting(args: Sequence[RawValue]) -> str:
    arg1 = activity.payload_converter().from_payload(args[0].payload, YourDataClass)
    return (
        f"{arg1.greeting}, {arg1.name}!\nActivity Type: {activity.info().activity_type}"
    )
# ...
@workflow.defn
class GreetingWorkflow:
    @workflow.run
    async def run(self, name: str) -> str:
        return await workflow.execute_activity(
            "unregistered_activity",
            YourDataClass("Hello", name),
            start_to_close_timeout=timedelta(seconds=10),
        )
```

此示例通过名称调用未注册的 Activity。
Worker 使用注册的动态 Activity 来解决它。
如果可能，优先使用编译器检查的类型安全参数而不是 Activity 名称字符串。
