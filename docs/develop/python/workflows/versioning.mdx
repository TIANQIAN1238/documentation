由于 Temporal 中的 Workflow Execution 可能会长期运行——有时长达数月甚至数年——即使在特定 Workflow Execution 进行期间，也经常需要对 Workflow Definition 进行修改。

Temporal Platform 要求 Workflow 代码具有[确定性](/workflow-definition#deterministic-constraints)。
如果对 Workflow 代码的修改会在 Replay 时导致非确定性行为，则需要使用我们的某种版本管理方法来优雅地更新正在运行的 Workflow。
使用版本管理，你可以修改 Workflow Definition，使新的执行使用更新后的代码，而现有的执行继续运行原始版本。
主要有两种版本管理方法可供使用：

- [Worker Versioning](/production-deployment/worker-deployments/worker-versioning)。Worker Versioning 功能允许你为 Worker 打标签，并以版本化部署的方式以编程方式滚动更新，使旧 Worker 能够运行旧代码路径，新 Worker 能够运行新代码路径。
- [基于 Patching 的版本管理](#patching)。此方法通过在代码中添加绑定到特定版本的分支来实现。它将代码变更应用于新的 Workflow Execution，同时避免对进行中的 Workflow Execution 造成破坏性影响。

:::danger
2025 年之前实验性 Worker Versioning 方法的支持将于 2026 年 3 月从 Temporal Server 中移除。请参阅[最新的 Worker Versioning 文档](/worker-versioning)获取指导。如有需要，你仍可参阅 [Worker Versioning 旧版本](worker-versioning-legacy)文档。
:::

## Worker Versioning

Temporal 的 [Worker Versioning](/production-deployment/worker-deployments/worker-versioning) 功能允许你为 Worker 打标签，并以 Deployment Version 的方式以编程方式滚动更新，使旧 Worker 能够运行旧代码路径，新 Worker 能够运行新代码路径。这样，你可以将 Workflow 固定到特定版本，从而避免 patching 的需求。

## 基于 Patching 的版本管理 {#patching}

### 添加 patch

Patch 在 Workflow 中为特定变更定义了一个逻辑分支，类似于功能开关。
它将代码变更应用于新的 Workflow Execution，同时避免对进行中的 Workflow Execution 造成破坏性影响。
当你想要对可能影响现有 Workflow Execution 的代码进行实质性修改时，请创建 patch。注意，[已固定的 Workflow](/worker-versioning) 无需 patch。

假设你有一个名为 `pre_patch_activity` 的初始 Workflow 版本：

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/version_your_workflows/workflow_1_initial_dacx.py">
    查看源代码
  </a>{' '}
  以了解完整的应用程序代码上下文。
</div>

```python
from datetime import timedelta

from temporalio import workflow

with workflow.unsafe.imports_passed_through():
    from activities import pre_patch_activity
# ...
@workflow.defn
class MyWorkflow:
    @workflow.run
    async def run(self) -> None:
        self._result = await workflow.execute_activity(
            pre_patch_activity,
            schedule_to_close_timeout=timedelta(minutes=5),
        )
```

现在，你想更新代码以运行 `post_patch_activity`。这代表你期望的最终状态。

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/version_your_workflows/workflow_4_patch_complete_dacx.py">
    查看源代码
  </a>{' '}
  以了解完整的应用程序代码上下文。
</div>

```python
from datetime import timedelta

from temporalio import workflow

with workflow.unsafe.imports_passed_through():
    from activities import post_patch_activity
# ...
@workflow.defn
class MyWorkflow:
    @workflow.run
    async def run(self) -> None:
        self._result = await workflow.execute_activity(
            post_patch_activity,
            schedule_to_close_timeout=timedelta(minutes=5),
        )
```

问题在于，在确定没有更多使用 `pre_patch_activity` 代码创建的正在运行的 Workflow 之前，无法直接部署 `post_patch_activity`，否则很可能导致非确定性错误。
因此，你需要部署 `post_patched_activity` 并使用 [patched](https://python.temporal.io/temporalio.workflow.html#patched) 函数来确定执行哪个版本的代码。

Patching 是一个三步骤流程：

1. 使用 `patched()` 函数引入新的更新代码，与旧代码并行运行。
2. 删除旧代码，并使用 `deprecate_patch()` 将特定 patch 标记为已废弃。
3. 一旦不再有该代码之前版本的开放 Workflow Execution，移除 `deprecate_patch()`。
   让我们按顺序逐步了解这一流程。

### 引入新代码的 patch {#using-patched-for-workflow-history-markers}

使用 `patched` 会在 Workflow History 中插入一个标记。
在 Replay 过程中，如果 Worker 遇到含有该标记的历史记录，而 Workflow 代码未产生相同的 patch 标记（此处为 `my-patch`），则会导致 Workflow 任务失败。
这确保你可以将 `post_patch_activity` 的代码作为"功能开关"与原始版本（`pre_patch_activity`）并行安全部署。

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/version_your_workflows/workflow_2_patched_dacx.py">
    查看源代码
  </a>{' '}
  以了解完整的应用程序代码上下文。
</div>

```python
# ...
@workflow.defn
class MyWorkflow:
    @workflow.run
    async def run(self) -> None:
        if workflow.patched("my-patch"):
            self._result = await workflow.execute_activity(
                post_patch_activity,
                schedule_to_close_timeout=timedelta(minutes=5),
            )
        else:
            self._result = await workflow.execute_activity(
                pre_patch_activity,
                schedule_to_close_timeout=timedelta(minutes=5),
            )
```

### 废弃 patch {#deprecated-patches}

在确认所有使用 `pre_patch_activity` 代码启动的 Workflow 均已超出保留期后，你可以[废弃该 patch](https://python.temporal.io/temporalio.workflow.html#deprecate_patch)。

一旦你的 Workflow 不再运行 pre-patch 代码路径，就可以部署带有 `deprecate_patch()` 的代码。
这些 Worker 将运行最新版本的 Workflow 代码，不再需要 patch。
已废弃的 patch 作为 patching 流程最终阶段与不再含有 patch 的最终状态之间的过渡桥梁。其工作方式与普通 patch 类似，也会向 Workflow History 添加标记。但是，当 Workflow 代码未产生该标记时，不会导致 replay 失败。

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/python/version_your_workflows/workflow_3_patch_deprecated_dacx.py">
    查看源代码
  </a>{' '}
  以了解完整的应用程序代码上下文。
</div>

```python
# ...
@workflow.defn
class MyWorkflow:
    @workflow.run
    async def run(self) -> None:
        workflow.deprecate_patch("my-patch")
        self._result = await workflow.execute_activity(
            post_patch_activity,
            schedule_to_close_timeout=timedelta(minutes=5),
        )
```

Patching 允许你对当前正在运行的 Workflow 进行修改。
这是一种引入兼容性变更而不产生非确定性错误的强大方法。

### Workflow 切换

要理解 Patching 的用处，可以通过演示整个 Workflow 的切换来加以说明。

由于不兼容的变更只影响同类型的已开启的 Workflow Execution，你可以通过在进行变更时创建全新的 Workflow 来避免确定性错误。
为此，你可以复制 Workflow Definition 函数并赋予不同的名称，然后在 Worker 中注册这两个名称。

例如，你可以将 `PizzaWorkflow` 复制为 `PizzaWorkflowV2`：

```python
@workflow.defn(name="PizzaWorkflow")
class PizzaWorkflow:
    @workflow.run
    async def run(self, name: str) -> str:
        # this function contains the original code

@workflow.defn(name="PizzaWorkflowV2")
class PizzaWorkflowV2:
    @workflow.run
    async def run(self, name: str) -> str:
        # this function contains the updated code
```

然后，你需要更新 Worker 配置以及其他标识符字符串，以注册这两种 Workflow 类型：

```python
worker = Worker(
    client,
    task_queue="your-task-queue",
    workflows=[PizzaWorkflow, PizzaWorkflowV2],
)
```

此方法的缺点是需要复制代码，并且需要更新用于启动 Workflow 的所有命令。
随着时间的推移，这可能变得不切实际。
此方法也无法对仍在运行的 Workflow 进行版本管理——它本质上只是一次切换，而不像 Patching 那样具有版本管理能力。

### 测试 Workflow 的 replay 安全性

要判断你的 Workflow 是否需要 patch，或者你的 patch 是否成功，应纳入 [Replay 测试](/develop/python/best-practices/testing-suite#replay)。
