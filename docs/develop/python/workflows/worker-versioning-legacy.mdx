##（已弃用）如何在 Python 中使用 Worker 版本控制 {#Worker-versioning}

:::caution

本节适用于已弃用的 Worker 版本控制 API。请将您的注意力转移到
[Worker 版本控制](/production-deployment/worker-deployments/worker-versioning)。

请参阅 [预发布自述文件](https://github.com/temporalio/temporal/blob/main/docs/worker-versioning.md) 了解更多信息
信息。

:::

构建 ID 对应于部署。如果您还没有，我们建议您将代码的哈希值（例如 Git SHA）与人类可读的时间戳相结合。
要使用 Worker 版本控制，您需要将构建 ID 传递给 Python Worker 并选择加入 Worker 版本控制。

<a id="worker-versioning"></a>
### 为您的 Worker 分配构建 ID 并选择加入 Worker 版本控制 {#worker-versioning}

在完成此步骤之前，您应该了解分配规则。请参阅
[Worker 版本控制预发布自述文件](https://github.com/temporalio/temporal/blob/main/docs/worker-versioning.md) 为
更多信息。

要为 Worker 启用 Worker 版本控制，请分配构建 ID（可能来自环境变量）并将其打开。

```python
# ...
worker = Worker(
  task_queue="your_task_queue_name",
  build_id=build_id,
  use_worker_versioning=True,
  # ... register workflows & activities, etc
)
# ...
```

:::warning

重要的是，当您启动此 Worker 时，在您设置分配规则之前，它不会接收任何任务。

:::

<a id="assign-a-build-id-to-your-worker-and-opt-in-to-worker-versioning"></a>
### 指定活动、Child Workflow 和 Continue-As-New Workflow 的版本 {#assign-a-build-id-to-your-worker-and-opt-in-to-worker-versioning}

:::caution

本节适用于已弃用的 Worker 版本控制 API。请将您的注意力转移到
[Worker 版本控制](/production-deployment/worker-deployments/worker-versioning)。

:::

默认情况下，活动、Child Workflow 和 Continue-As-New Workflow 在创建的 Workflow 的构建上运行
如果它们也配置为在同一 Task Queue 上运行。当配置为在单独的 Task Queue 上运行时，它们
将默认使用当前分配规则。

如果您想覆盖此行为，您可以通过 `versioning_intent` 上可用的参数指定您的意图
用于调用这些命令的方法。

例如，如果您希望 Activity 使用最新的分配规则而不是从其父级继承：

```python
# ...
await workflow.execute_activity(
    say_hello,
    "hi",
    versioning_intent=VersioningIntent.USE_ASSIGNMENT_RULES,
    start_to_close_timeout=timedelta(seconds=5),
)
# ...
```

<a id="specify-versions-for-activities-child-workflows-and-continue-as-new-workflows"></a>
### 告诉 Task Queue 您的 Worker 的内部版本 ID（已弃用） {#specify-versions-for-activities-child-workflows-and-continue-as-new-workflows}

:::caution

本节适用于已弃用的 Worker 版本控制 API。请将您的注意力转移到
[Worker 版本控制](/production-deployment/worker-deployments/worker-versioning)。

:::

现在您可以使用 SDK（或 Temporal CLI）向 Task Queue 告知您的 Worker 的构建 ID。你可能想做
这是 CI 部署过程的一部分。

```python
# ...
await client.update_worker_build_id_compatibility(
    "your_task_queue_name", BuildIdOpAddNewDefault("deadbeef")
)
```

此代码将 `deadbeef` 构建 ID 添加到 Task Queue 作为新版本集中的唯一版本，这将成为
队列的默认值。新的 Workflow 在具有此构建 ID 的 Worker 上执行，现有的将继续处理
通过适当兼容的 Worker。

相反，如果您想将构建 ID 添加到现有的兼容集中，您可以执行以下操作：

```python
# ...
await client.update_worker_build_id_compatibility(
    "your_task_queue_name", BuildIdOpAddNewCompatible("deadbeef", "some-existing-build-id")
)
```

此代码将 `deadbeef` 添加到包含 `some-existing-build-id` 的现有兼容集中，并将其标记为新的
该集的默认构建 ID。

您还可以将集合中的现有构建 ID 提升为该集合的默认值：

```python
# ...
await client.update_worker_build_id_compatibility(
    "your_task_queue_name", BuildIdOpPromoteBuildIdWithinSet("deadbeef")
)
```

您还可以将整个集合提升为队列的默认集合。新的 Workflow 将开始使用该套件的
默认构建。

```python
# ...
await client.update_worker_build_id_compatibility(
    "your_task_queue_name", BuildIdOpPromoteSetByBuildId("deadbeef")
)
```
