Temporal 通过重试和持久执行自动处理多种类型的故障。
本页向您展示如何基于这些功能为您的应用程序创建强大的错误处理。

**关键概念：**

并非所有故障都应该以相同的方式处理。
**暂时性故障**（例如短暂的网络中断）会自行解决，应立即重试。
**间歇性失败**（如速率限制）需要增加重试之间的延迟。
**永久性故障**（例如无效输入）无法通过重试解决，并且需要更改不同的数据或代码。

Temporal 区分 **Workflow 任务故障**（可以通过重新部署修复的错误）和 **Workflow Execution 故障**（应停止 Workflow 的业务逻辑故障）。
任务失败会自动重试，以便您可以修复和重新部署而不会丢失状态。
执行失败需要您显式引发 `ApplicationError`。

<a id="make-activities-idempotent"></a>
## 使活动幂等 {#make-activities-idempotent}

**如何使用 Temporal Python SDK 使活动幂等**

由于 Activity 可能会因失败而重试，因此强烈建议将其设为幂等。
幂等操作无论执行一次还是多次都会产生相同的结果。

活动遵循至少一次执行模型。
如果 Worker 成功执行 Activity 但在通知 Temporal 服务之前崩溃，则将重试 Activity。
如果没有幂等性，这可能会导致支付处理中重复收费或在基础设施配置中创建重复资源。

<a id="use-idempotency-keys"></a>
### 使用幂等键 {#use-idempotency-keys}

大多数外部服务都支持幂等键——防止重复操作的唯一标识符。
当服务收到带有已处理过的密钥的请求时，它会返回原始结果，而不是再次执行操作。

通过组合 Workflow 运行 ID 和 Activity ID 创建幂等密钥：

```python
from temporalio import activity

@activity.defn
async def process_payment(amount: float, account: str):
    info = activity.info()
    idempotency_key = f"{info.workflow_run_id}-{info.activity_id}"

    # Pass idempotency_key to your payment service
    result = await payment_service.charge(
        amount=amount,
        account=account,
        idempotency_key=idempotency_key
    )
    return result
```

该值在 Activity 重试中保持不变，但在所有 Workflow Execution 中是唯一的。

<a id="design-activities-to-be-atomic"></a>
### 将活动设计为原子的 {#design-activities-to-be-atomic}

活动是原子的——它们要么成功完成，要么失败。
如果 Activity 执行多个步骤并且最后一步失败，则会重试整个 Activity。

考虑这个 Activity：
1.在数据库中查找数据
2. 使用数据调用微服务
3. 将结果写入文件系统

如果步骤 3 失败，则重试时将再次执行所有三个步骤。
您可以将其拆分为三个单独的活动，以便仅重试失败的步骤，但要平衡此与更大的 Event History 和更多 Activity 执行。

## 从活动中引发异常 {#raise-exceptions-from-activities}

**如何使用 Temporal Python SDK 引发活动异常**

使用 `ApplicationError` 从活动中传达特定于应用程序的故障。显式提高它可以让您获得更多控制权。

:::note

Temporal Activity 中的 Python 代码引发的*任何*其他异常都将在内部转换为 `ApplicationError`。这样，错误的类型、严重性和任何其他详细信息都可以发送到 Temporal 服务，由 Web UI 进行索引，甚至可以跨语言边界进行序列化。

:::

```python
from temporalio import activity
from temporalio.exceptions import ApplicationError

@activity.defn
async def validate_charge(credit_card_number: str, amount: float):
    if not is_valid_card(credit_card_number):
        raise ApplicationError(
            f"Invalid credit card number: {credit_card_number}",
            type="InvalidCreditCard",
        )

    if amount <= 0:
        raise ApplicationError(
            f"Amount must be positive, got {amount}",
            type="InvalidAmount",
        )

    return True
```

当筹集 `ApplicationError` 时：
- 提供描述性的 `message`
- 可以选择提供 `type` 字符串来对故障进行分类
- 该错误作为 `ActivityTaskFailed` 事件出现在 Event History 中

当 Activity 失败时，Temporal 会将异常包装在 `ActivityError` 中，然后再将其呈现给 Workflow。
`ActivityError` 提供的上下文包括：
- Activity 类型失败
- 重试次数
- 原始原因（您提出的[`ApplicationError`](https://python.temporal.io/temporalio.exceptions.ApplicationError.html)，或`TimeoutError`、[`CancelledError`](https://python.temporal.io/temporalio.exceptions.CancelledError.html)等）

<a id="raise-exceptions-from-workflows"></a>
## 从 Workflows 引发异常 {#raise-exceptions-from-workflows}

**如何使用 Temporal Python SDK 从 Workflow 引发异常**

该行为取决于您引发的异常：

<a id="fail-a-workflow-execution"></a>
### Workflow Execution 失败 {#fail-a-workflow-execution}

要故意使 Workflow Execution 失败，请引发 `ApplicationError`：

```python
from temporalio import workflow
from temporalio.exceptions import ApplicationError

@workflow.defn
class PizzaDeliveryWorkflow:
    @workflow.run
    async def run(self, order):
        distance = await workflow.execute_activity(
            calculate_distance,
            order.address,
            start_to_close_timeout=timedelta(seconds=10)
        )

        if order.is_delivery and distance.kilometers > 25:
            workflow.logger.error("Customer outside service area")
            raise ApplicationError(
                "Customer lives outside the service area",
                type="CustomerOutsideServiceArea"
            )

        # Continue with order...
```

使用 Temporal `ApplicationError` 类的原因之一是它允许您设置附加的 `non_retryable` 参数。
这样，您可以决定 Temporal 是否不应自动重试错误。
这对于由于输入数据错误而故意使 Workflow 失败而不是等待超时结束非常有用。您也可以在 Activity [Retry Policy](/develop/python/activities/timeouts#activity-retries) 中指定不可重试的错误列表。

这会将 Workflow Execution 置于“失败”状态，并且不会自动重试。
将此用于重试无济于事的永久性故障，例如客户距离太远。

<a id="trigger-a-workflow-task-retry"></a>
### 触发 Workflow 任务重试 {#trigger-a-workflow-task-retry}

引发任何其他 Python 异常（如 `ValueError` 或 `TypeError`）会导致 Workflow 任务失败，该失败会自动重试：

```python
# This causes a Workflow Task failure (retries automatically)
raise ValueError("Unexpected condition")
```

这是故意的。
常规 Python 异常被视为可以通过代码部署修复的错误，而不是业务逻辑故障。
Workflow 任务无限期地重试，让您可以修复错误并重新部署，而不会丢失 Workflow 状态。

<a id="handle-exceptions-in-workflows"></a>
## 处理 Workflows 中的异常 {#handle-exceptions-in-workflows}

**如何使用 Temporal Python SDK 处理 Workflow 中的异常**

使用 Python 的 `try/except` 块来处理 Workflow 中的 Activity 故障：

```python
from temporalio import workflow
from temporalio.exceptions import ActivityError, ApplicationError
from datetime import timedelta

@workflow.defn
class MoneyTransferWorkflow:
    @workflow.run
    async def run(self, details):
        # Withdraw money
        try:
            withdraw_result = await workflow.execute_activity(
                withdraw,
                details,
                start_to_close_timeout=timedelta(seconds=10)
            )
        except ActivityError as e:
            raise ApplicationError(
                f"Withdrawal failed: {e.cause}",
                type="WithdrawalError"
            )

        # Deposit money
        try:
            deposit_result = await workflow.execute_activity(
                deposit,
                details,
                start_to_close_timeout=timedelta(seconds=10)
            )
        except ActivityError as e:
            # Deposit failed - attempt refund
            try:
                await workflow.execute_activity(
                    refund,
                    withdraw_result,
                    start_to_close_timeout=timedelta(seconds=10)
                )
                raise ApplicationError(
                    f"Deposit failed but money refunded to source account",
                    type="DepositError"
                )
            except ActivityError as refund_err:
                raise ApplicationError(
                    f"Deposit failed and refund also failed: {refund_err.cause}",
                    type="CriticalTransferError"
                )

        return f"Transfer complete: {withdraw_result}, {deposit_result}"
```

您可以在 Workflow 中捕获常见的 Temporal 异常：
- `ActivityError` - Activity 在用尽重试后失败
- `ChildWorkflowError` - Child Workflow 失败
- `CancelledError` - Workflow、Activity 或 Timer 已取消
- `TimeoutError` - 操作超出超时

如果这些异常未经处理地传播，则 Workflow Execution 将失败（或 `CancelledError` 进入“已取消”状态）。

## 配置自定义重试策略 {#configure-custom-retry-policies}

**如何使用 Temporal Python SDK 配置自定义重试策略**

活动具有默认的 Retry Policy，具有无限尝试和指数退避。
对此进行自定义以匹配您预期的故障模式。

```python
from temporalio import workflow
from temporalio.common import RetryPolicy
from datetime import timedelta

@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order):
        # Custom retry for rate-limited service
        retry_policy = RetryPolicy(
            initial_interval=timedelta(seconds=10),
            backoff_coefficient=3.0,
            maximum_interval=timedelta(minutes=5),
            maximum_attempts=20,
        )

        result = await workflow.execute_activity(
            call_external_service,
            order,
            start_to_close_timeout=timedelta(seconds=30),
            retry_policy=retry_policy,
        )
        return result
```

Retry Policy 属性：
- **`initial_interval`**：第一次重试之前的延迟（默认值：1 秒）
- **`backoff_coefficient`**：后续延迟的乘数（默认值：2.0）
- **`maximum_interval`**：重试延迟上限（默认值：100×初始间隔）
- **`maximum_attempts`**：最大重试次数（默认值：无限制）
- **`non_retryable_error_types`**：不应重试的错误类型（默认值：空）

<a id="match-your-retry-policy-to-failure-types"></a>
### 将您的 Retry Policy 与故障类型匹配 {#match-your-retry-policy-to-failure-types}

**对于暂时性故障**（短暂的网络问题）：使用默认值或较低的 `initial_interval` 和 `backoff_coefficient`。

**对于间歇性故障**（速率限制）：增加 `initial_interval` 和 `backoff_coefficient` 以间隔重试并解决问题。

**对于成本敏感的 API**：设置 `maximum_attempts` 以限制重试（罕见 - 通常更喜欢超时）。

<a id="use-different-policies-for-different-activities"></a>
### 针对不同的活动使用不同的策略 {#use-different-policies-for-different-activities}

您可以对不同的活动使用不同的重试策略，甚至可以对同一 Activity 使用多个策略：

```python
fast_retry = RetryPolicy(
    initial_interval=timedelta(seconds=1),
    backoff_coefficient=1.5,
)

slow_retry = RetryPolicy(
    initial_interval=timedelta(seconds=30),
    backoff_coefficient=3.0,
)

# Same Activity, different policies
await workflow.execute_activity(
    process_order,
    order,
    start_to_close_timeout=timedelta(seconds=10),
    retry_policy=fast_retry,
)

# Later, with different circumstances...
await workflow.execute_activity(
    process_order,
    order,
    start_to_close_timeout=timedelta(seconds=10),
    retry_policy=slow_retry,
)
```

<a id="dont-use-workflow-retry-policies"></a>
### 不要使用 Workflow 重试策略 {#dont-use-workflow-retry-policies}

与活动不同，Workflow 默认情况下不会重试，并且您通常不应添加 Retry Policy。
Workflow 是确定性的，不适用于容易发生故障的操作。
Workflow 故障通常表示代码错误或输入数据错误 - 重试整个 Workflow 会重复相同的逻辑，而不会解决根本问题。

如果您需要特定 Workflow 操作的重试逻辑，请在 Workflow 代码中实现它，而不是使用 Workflow Retry Policy。

## 将特定错误标记为不可重试 {#mark-errors-as-non-retryable}

**如何使用 Temporal Python SDK 将特定错误标记为不可重试**

有些故障是永久性的，无法通过重试解决。
将它们标记为不可重试以快速失败，而不是等待超时。

在引发 `ApplicationError` 时设置 `non_retryable` 标志：

```python
from temporalio import activity
from temporalio.exceptions import ApplicationError

@activity.defn
async def process_payment(card_number: str, amount: float):
    if not is_valid_card_format(card_number):
        # Invalid format will never become valid through retries
        raise ApplicationError(
            f"Invalid credit card format: {card_number}",
            type="InvalidCardFormat",
            non_retryable=True,
        )

    if amount <= 0:
        # Invalid amount won't be fixed by retrying
        raise ApplicationError(
            f"Amount must be positive: {amount}",
            type="InvalidAmount",
            non_retryable=True,
        )

    # Process payment...
```

无论 Retry Policy 如何，带有 `non_retryable=True` 的 `ApplicationError` 都不会重试。

使用不可重试的错误：
- 无效的输入数据阻止 Activity 继续进行
- 违反业务规则
- 授权失败

**谨慎使用这个。**
在大多数情况下，最好让 Retry Policy 根据时间或尝试来处理何时停止重试。

## 指定不可重试的错误类型 {#specify-non-retryable-error-types}

**如何使用 Temporal Python SDK 在重试策略中指定不可重试的错误类型**

有时您希望 Workflow（调用者）而不是 Activity（实现者）决定哪些错误类型不应重试。

列出不应在 Retry Policy 中重试的错误类型：

```python
from temporalio import workflow
from temporalio.common import RetryPolicy
from datetime import timedelta

@workflow.defn
class CheckoutWorkflow:
    @workflow.run
    async def run(self, payment_details):
        retry_policy = RetryPolicy(
            non_retryable_error_types=[
                "InvalidCardFormat",
                "InsufficientFunds",
                "AccountClosed",
            ]
        )

        try:
            result = await workflow.execute_activity(
                process_payment,
                payment_details,
                start_to_close_timeout=timedelta(seconds=30),
                retry_policy=retry_policy,
            )
            return result
        except ActivityError as e:
            workflow.logger.error(f"Payment failed: {e.cause}")
            # Handle the non-retryable error...
```

当 Activity 引发 `ApplicationError` 时，Temporal 检查其 `type` 是否在 `non_retryable_error_types` 中。
如果匹配，则 Activity 立即失败，无需重试。

<a id="when-to-use-each-approach"></a>
### 何时使用每种方法 {#when-to-use-each-approach}

**Activity 中的 `non_retryable=True`**：当 Activity 实施者知道错误永久不可恢复时使用。
这对所有调用者强制执行约束。

**Retry Policy 中的 `non_retryable_error_types`**：当调用者想要根据其业务逻辑决定哪些错误不可恢复时使用。
这使得不同的 Workflow 对同一 Activity 做出不同的决定。

## 使用 Saga 模式实现回滚逻辑 {#implement-saga-pattern}

**如何使用 Temporal Python SDK 实现 Saga 模式**

Saga 模式协调一系列操作，其中每个操作都有一个补偿操作来撤消其影响。
如果任何操作失败，则按照相反的顺序执行补偿操作，回滚之前的操作。

将其用于多步骤过程，例如：
- 电子商务结帐（付款、库存、运输）
- 跨服务的分布式事务
- 多级数据Updates

```python
from temporalio import workflow
from temporalio.exceptions import ActivityError
from datetime import timedelta

@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order):
        compensations = []

        try:
            # Reserve inventory
            compensations.append({
                "activity": revert_inventory,
                "input": order
            })
            await workflow.execute_activity(
                reserve_inventory,
                order,
                start_to_close_timeout=timedelta(seconds=10),
            )

            # Charge payment
            compensations.append({
                "activity": refund_payment,
                "input": order
            })
            payment_id = await workflow.execute_activity(
                charge_payment,
                order,
                start_to_close_timeout=timedelta(seconds=10),
            )

            # Create shipment
            compensations.append({
                "activity": cancel_shipment,
                "input": payment_id
            })
            shipment_id = await workflow.execute_activity(
                create_shipment,
                order,
                start_to_close_timeout=timedelta(seconds=10),
            )

            return {"payment_id": payment_id, "shipment_id": shipment_id}

        except ActivityError as e:
            workflow.logger.error(f"Order failed: {e.cause}, rolling back...")

            # Execute compensations in reverse order
            for compensation in reversed(compensations):
                try:
                    await workflow.execute_activity(
                        compensation["activity"],
                        compensation["input"],
                        start_to_close_timeout=timedelta(seconds=10),
                    )
                except ActivityError as comp_err:
                    # Log compensation failure but continue with others
                    workflow.logger.error(f"Compensation failed: {comp_err.cause}")

            # Re-raise the original error
            raise ApplicationError(
                f"Order failed: {e.cause}",
                type="OrderFailed"
            )
```

要点：
- 在执行每个 Activity 之前**将补偿操作添加到列表中
- 使用`reversed(compensations)`以正确的顺序撤消操作
- 优雅地处理补偿失败（它们也可能失败）
- Temporal 管理所有状态和重试逻辑，使 Saga 实现简单明了

## 了解 Temporal 的故障类型 {#understand-failure-types}

Temporal 使用专门的异常类型来表示不同的故障场景。
所有异常都继承自 [`TemporalError`](https://python.temporal.io/temporalio.exceptions.TemporalError.html)。

**不要扩展 `TemporalError` 或其子项。**
使用提供的异常类型来确保：
- 跨进程和语言边界的一致行为
- 与 Temporal 服务的兼容性
- 通过协议缓冲区进行正确的序列化

<a id="common-failure-types"></a>
### 常见故障类型 {#common-failure-types}

**`ApplicationError`**：由您的代码引发以指示特定于应用程序的故障。
这是您应该手动引发的唯一 Temporal 异常。
当您提出 `ApplicationError` 时，您可以选择提供 `type` 字符串并将其标记为 `non_retryable`。

**`ActivityError`**：包装从活动引发的异常。
`cause` 字段包含原始错误（`ApplicationError`、`TimeoutError`、`CancelledError` 等）。
在 Workflow 中捕获此内容以处理 Activity 故障。

**`TimeoutError`**：当 Activity 或 Workflow 超过其配置的超时时发生。

**`CancelledError`**：取消 Workflow、Activity 或 Timer 的结果。
您可以捕获并忽略它，以便在取消的情况下继续执行。

**`TerminatedError`**：当 Workflow Execution 被强制终止时发生。

**`ChildWorkflowError`**：当子 Workflow Execution 失败时引发。

**`WorkflowAlreadyStartedError`**：尝试使用已在运行的 ID 启动 Workflow 时引发。

**`ServerError`**：用于 Temporal 服务本身的异常（例如数据库故障）。

<a id="workflow-task-vs-workflow-execution-failures"></a>
### Workflow 任务与 Workflow Execution 失败 {#workflow-task-vs-workflow-execution-failures}

**Workflow 任务失败** 当 Workflow 代码引发非 Temporal 异常（如 `ValueError`、`TypeError` 或非确定性错误）时发生。
这些会自动重试，让您修复错误并重新部署，而不会丢失 Workflow 状态。

当 Workflow 代码引发 Temporal 异常（如 `ApplicationError`）时，**Workflow Execution 失败**发生。
这会将 Workflow 置于“失败”状态，并且不会自动重试。

Workflow 应该失败的永久性故障示例：

```python
if distance.kilometers > MAX_DELIVERY_DISTANCE:
    # Retrying won't change the distance - this is permanent
    raise ApplicationError(
        "Customer lives outside service area",
        type="OutsideServiceArea"
    )
```

<a id="protecting-sensitive-information"></a>
### 保护敏感信息 {#protecting-sensitive-information}

默认的故障转换器将异常消息和堆栈跟踪复制为 Web UI 中可见的纯文本。
如果您的异常可能包含敏感信息，请配置自定义故障转换器来加密此数据。
有关详细信息，请参阅 [保护应用程序数据课程](https://learn.temporal.io/courses/appdatasec/)。
