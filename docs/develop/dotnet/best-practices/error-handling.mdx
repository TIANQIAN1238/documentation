## 引发和处理异常 {#exception-handling}

在每个 Temporal SDK 中，错误处理都是按照语言约定惯用地实现的。
Temporal 在内部使用多个不同的错误类 - 例如，.NET SDK 中的 [`CancelledFailureException`](https://dotnet.temporal.io/api/temporalio.Exceptions.CanceledFailureException.html) 来处理 Workflow 取消。 
您不应手动提出或以其他方式实现这些，因为它们与 Temporal 平台逻辑相关。

您通常会故意提出的一个 Temporal 错误类别是 [`ApplicationFailureException`](https://dotnet.temporal.io/api/temporalio.Exceptions.ApplicationFailureException.html)。
事实上，Temporal Activity 中的 C# 代码引发的*任何*其他异常都将在内部转换为 `ApplicationFailureException`。
这样，错误的类型、严重性和任何其他详细信息都可以发送到 Temporal 服务，由 Web UI 进行索引，甚至可以跨语言边界进行序列化。

换句话说，这两个代码示例执行相同的操作：

```csharp
[Serializable]
public class InvalidDepartmentException : Exception
{
    public InvalidDepartmentException() : base() { }
    public InvalidDepartmentException(string message) : base(message) { }
    public InvalidDepartmentException(string message, Exception inner) : base(message, inner) { }
}

[Activity]
public Task<OrderConfirmation> SendBillAsync(Bill bill)
{
    throw new InvalidDepartmentException("Invalid department");
}
```

```csharp
[Activity]
public Task<OrderConfirmation> SendBillAsync(Bill bill)
{
    throw new ApplicationFailureException("Invalid department", errorType: "InvalidDepartmentException");
}
```

根据您的实施情况，您可以决定使用任一方法。
使用 Temporal `ApplicationFailureException` 类的原因之一是它允许您设置附加的 `non_retryable` 参数。
这样，您可以决定 Temporal 是否不应自动重试错误。
这对于由于输入数据错误而故意使 Workflow 失败非常有用，而不是等待超时结束：

```csharp
[Activity]
public Task<OrderConfirmation> SendBillAsync(Bill bill)
{
    throw new ApplicationFailureException("Invalid department", nonRetryable: true);
}
```

您也可以在 Activity [Retry Policy](/develop/dotnet/activities/timeouts#activity-retries) 中指定不可重试的错误列表。

## Workflow 失败 {#workflow-failure}

Temporal 的核心设计原则之一是 Activity 故障永远不会直接导致 Workflow 故障 — Workflow 永远不应该返回 Failed，除非故意。
与 Temporal 活动关联的默认 Retry Policy 是重试它们，直到达到某个超时阈值。
在满足此条件或另一个不可重试的条件之前，活动实际上不会向您的 Workflow“返回”失败。
此时，您可以决定如何处理 Activity 返回的错误，就像在任何其他程序中一样。
例如，您可以实现一个 [传奇模式](https://github.com/temporalio/samples-dotnet/tree/main/src/Saga)，它使用 `try`/`catch` 块来“展开”Workflow 在 Activity 故障点之前执行的一些步骤。

**您只能通过从 Workflow 代码手动引发 `ApplicationFailureException` 来使 Workflow 失败。**
如果 Activity 的失败意味着您的 Workflow 不应继续，您可以执行此操作以响应 Activity 失败：

```csharp
try
{
    await Workflow.ExecuteActivityAsync(
        (Activities act) => act.ValidateCreditCardAsync(order.Customer.CreditCardNumber),
        options);
}
catch (ActivityFailureException err)
{
    logger.LogError("Unable to process credit card: {Message}", err.Message);
    throw new ApplicationFailureException(message: "Invalid credit card number error");
}
```

这在 Workflow 中的工作方式与从活动中引发异常不同。
在 Activity 中，任何 C# 异常或自定义异常都会转换为 Temporal `ApplicationError`。
在 Workflow 中，除显式 Temporal `ApplicationError` 之外引发的任何异常都只会使该特定 [Workflow 任务](https://docs.temporal.io/tasks#workflow-task-execution) 失败并重试。
这包括自动引发的任何典型 C# `RuntimeError`。
这些错误被视为可以通过固定部署纠正的错误，而不是 Temporal Workflow Execution 意外返回的原因。
