Workflow 可以像有状态 Web 服务一样接收消息：查询、Signal 和 Update。
Workflow 实现通过可以对传入消息和返回值做出反应的处理程序方法定义这些端点。
Temporal CLIents 使用消息读取 Workflow 状态并控制执行。
有关此主题的一般概述，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing)。
本页介绍 Temporal .NET SDK 的这些功能。

<a id="writing-message-handlers"></a>
## 编写消息处理程序 {#writing-message-handlers}
:::info
以下代码是 [工作解决方案](https://github.com/temporalio/samples-dotnet/tree/main/src/MessagePassing) 的一部分。
:::

编写消息处理程序时请遵循以下准则：

- 消息处理程序被定义为 Workflow 类上的方法，使用以下三个属性之一：[`WorkflowQueryAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowQueryAttribute.html)、[`WorkflowSignalAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowSignalAttribute.html) 和 [`WorkflowUpdateAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowUpdateAttribute.html)。
- 处理程序和主 Workflow 函数的参数和返回值必须是 [可序列化](/dataconversion)。
- 优先选择数据类而不是多个输入参数。数据类参数允许您在不更改调用签名的情况下添加字段。请记住，如果新字段没有默认值，则序列化和反序列化可能会失败并显示默认的 Data Converter。

<a id="queries"></a>
### Query 处理程序 {#queries}
[Query](/sending-messages#sending-queries) 是从 Workflow Execution 检索状态的同步操作。
定义为方法：

```csharp
[Workflow]
public class GreetingWorkflow
{
    public enum Language
    {
        Chinese,
        English,
        French,
        Spanish,
        Portuguese,
    }

    public record GetLanguagesInput(bool IncludeUnsupported);

    // ...

    [WorkflowQuery]
    public IList<Language> GetLanguages(GetLanguagesInput input) =>
        Enum.GetValues<Language>().
            Where(language => input.IncludeUnsupported || Greetings.ContainsKey(language)).
            ToList();

    // ...
```

或者作为财产获取者：

```csharp
[Workflow]
public class GreetingWorkflow
{
    public enum Language
    {
        Chinese,
        English,
        French,
        Spanish,
        Portuguese,
    }

    // ...

    [WorkflowQuery]
    public Language CurrentLanguage { get; private set; } = Language.English;

    // ...
```

- Query 属性可以接受参数。
  请参阅 API 参考文档：[`WorkflowQueryAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowQueryAttribute.html)。
- Query 处理程序不得修改 Workflow 状态。
- 您无法执行异步阻塞操作，例如在 Query 处理程序中执行 Activity。

<a id="signals"></a>
### Signal 处理程序 {#signals}
[Signal](/sending-messages#sending-signals) 是发送到正在运行的 Workflow Execution 的异步消息，用于更改其状态并控制其流程：

```csharp
[Workflow]
public class GreetingWorkflow
{
    public record ApproveInput(string Name);

    // ...

    [WorkflowSignal]
    public async Task ApproveAsync(ApproveInput input)
    {
        approvedForRelease = true;
        approverName = input.Name;
    }

    // ...
```

- Signal 属性可以接受参数。
  请参阅 API 文档：[`WorkflowSignalAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowSignalAttribute.html)。

- 处理程序不应返回值。
  服务器立即发送响应，无需等待 Workflow 处理 Signal。

- Signal（和 Update）处理程序可以是异步和阻塞的。
  这允许您使用活动、Child Workflow、持久 [`Workflow.DelayAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_DelayAsync_System_Int32_System_Nullable_System_Threading_CancellationToken__)、Timer、[`Workflow.WaitConditionAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_WaitConditionAsync_System_Func_System_Boolean__System_Int32_System_Nullable_System_Threading_CancellationToken__) 条件等。
  有关安全使用异步 Signal 和 Update 处理程序的指南，请参阅 [异步处理程序](#async-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing)。

<a id="updates"></a>
### Update 处理程序和验证器 {#updates}
[Update](/sending-messages#sending-updates) 是发送到正在运行的 Workflow Execution 的可跟踪同步请求。
它可以更改 Workflow 状态、控制其流程并返回结果。
发送方必须等待，直到 Worker 接受或拒绝 Update。
如果出现问题，发送者可能会进一步等待接收返回值或异常：

```csharp
[Workflow]
public class GreetingWorkflow
{
    public enum Language
    {
        Chinese,
        English,
        French,
        Spanish,
        Portuguese,
    }

    // ...

    [WorkflowUpdateValidator(nameof(SetCurrentLanguageAsync))]
    public void ValidateLanguage(Language language)
    {
        if (!Greetings.ContainsKey(language))
        {
            throw new ApplicationFailureException($"{language} is not supported");
        }
    }

    [WorkflowUpdate]
    public async Task<Language> SetCurrentLanguageAsync(Language language)
    {
        var previousLanguage = CurrentLanguage;
        CurrentLanguage = language;
        return previousLanguage;
    }

    // ...
```

- Update 属性可以采用参数（例如 `Name`、`Dynamic` 和 `UnfinishedPolicy`），如 [`WorkflowUpdateAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowUpdateAttribute.html) 的 API 参考文档中所述。

- 关于验证器：
  - 在将 Update 写入历史记录之前，使用验证器拒绝该 Update。
    验证器始终是可选的。
    如果您不需要拒绝 Update，则可以跳过它们。
  - 使用 [`WorkflowUpdateValidatorAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowUpdateValidatorAttribute.html) 属性定义 Update 验证器。
    声明验证器时使用 Name 参数将其连接到其 Update。
    验证器必须是 `void` 类型，并接受与处理程序相同的参数类型。

- 使用验证器接受和拒绝 Update：
  - 要拒绝 Update，请在验证器中引发任何类型的异常。
  - 如果没有验证器，则始终接受 Update。
- 验证器和 Event History：
  - 无论是自动接受还是编程接受，`WorkflowExecutionUpdateAccepted` 事件都会写入历史记录。
  - 当验证器引发错误时，Update 会被拒绝，Update 不会运行，并且 `WorkflowExecutionUpdateAccepted` _不会_ 添加到 Event History 中。
    调用者收到“Update 失败”错误。

- 使用 [`Workflow.CurrentUpdateInfo`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_CurrentUpdateInfo) 获取有关当前 Update 的信息。
  这包括 Update ID，它在使用 Continue-As-New 时可用于重复数据删除：请参阅 [确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing)。
- Update（和 Signal）处理程序可以是异步和阻塞的。
  这允许您使用活动、Child Workflow、持久 [`Workflow.DelayAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_DelayAsync_System_Int32_System_Nullable_System_Threading_CancellationToken__)、Timer、[`Workflow.WaitConditionAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_WaitConditionAsync_System_Func_System_Boolean__System_Int32_System_Nullable_System_Threading_CancellationToken__) 条件等。
  有关安全使用异步 Update 和 Signal 处理程序的指南，请参阅 [异步处理程序](#async-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing)。

<a id="send-messages"></a>
## 发送消息 {#send-messages}
要发送查询、Signal 或 Update，您可以调用 [`WorkflowHandle`](https://dotnet.temporal.io/api/temporalio.Client.WorkflowHandle.html) 对象上的方法。
要获取WorkflowStub，您可以：

- 使用 [`TemporalClient.StartWorkflowAsync`](https://dotnet.temporal.io/api/temporalio.Client.TemporalClient.html#Temporalio_Client_TemporalClient_StartWorkflowAsync_System_String_System_Collections_Generic_IReadOnlyCollection_System_Object__Temporalio_Client_WorkflowOptions_) 启动 Workflow 并返回其句柄。
- 使用 [`TemporalClient.GetWorkflowHandle`](https://dotnet.temporal.io/api/temporalio.Client.TemporalClient.html#Temporalio_Client_TemporalClient_GetWorkflowHandle_System_String_System_String_System_String_) 方法通过 Workflow Id 检索 Workflow 句柄。

例如：

```csharp
var client = await TemporalClient.ConnectAsync(new("localhost:7233"));
var workflowHandle = await client.StartWorkflowAsync(
    (GreetingWorkflow wf) => wf.RunAsync(),
    new(id: "message-passing-workflow-id", taskQueue: "message-passing-sample"));
```

要检查发送消息时所需的参数类型以及查询和 Update 的返回类型，请参阅 Workflow Definition 中相应的处理程序方法。

:::warning Using Continue-as-New and Updates

- Temporal_不_支持 Update 处理程序中的 Continue-As-New 功能。
- 使用 Continue-As-New_之前_完成所有处理程序。
- 从您的主要 Workflow Definition 方法中使用 Continue-As-New，就像完成或失败 Workflow Execution 一样。

:::

<a id="send-query"></a>
### 发送 Query {#send-query}
使用 [`WorkflowHandle.QueryAsync`](https://dotnet.temporal.io/api/temporalio.Client.WorkflowHandle.html#Temporalio_Client_WorkflowHandle_QueryAsync__1_System_String_System_Collections_Generic_IReadOnlyCollection_System_Object__Temporalio_Client_WorkflowQueryOptions_) 调用 Query 方法：

```csharp
var supportedLanguages = await workflowHandle.QueryAsync(wf => wf.GetLanguages(new(false)));
```

- 发送 Query 不会将事件添加到 Workflow 的 Event History。

- 您可以在 Namespace 的 Workflow 保留期内向已关闭的 Workflow Execution 发送查询。
  这包括已完成、失败或超时的 Workflow。
  Querying 终止的 Workflow 不安全，因此不受支持。

- Worker 必须在线并轮询 Task Queue 才能处理 Query。

<a id="send-signal"></a>
### 发送 Signal {#send-signal}
您可以从 Temporal CLIent 或另一个 Workflow Execution 向 Workflow Execution 发送 Signal。
但是，您只能将 Signal 发送到尚未关闭的 Workflow Execution。

<a id="send-signal-from-client"></a>
#### 从客户端发送 Signal {#send-signal-from-client}
使用客户端代码中的 [`WorkflowHandle.SignalAsync`](https://dotnet.temporal.io/api/temporalio.Client.WorkflowHandle.html#Temporalio_Client_WorkflowHandle_SignalAsync_System_String_System_Collections_Generic_IReadOnlyCollection_System_Object__Temporalio_Client_WorkflowSignalOptions_) 发送 Signal：

```csharp
await workflowHandle.SignalAsync(wf => wf.ApproveAsync(new("MyUser")));
```

- 当服务器接受 Signal 时调用返回；它并不等待 Signal 被传送到 Workflow Execution。

- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在 Workflow 的 Event History 中。

<a id="send-signal-from-workflow"></a>
#### 从 Workflow 发送 Signal {#send-signal-from-workflow}
Workflow 可以将 Signal 发送到另一个 Workflow，称为_外部 Signal_。
在这种情况下，您需要获取外部 Workflow 的 Workflow 句柄。
使用 `Workflow.GetExternalWorkflowHandle`，传递正在运行的 Workflow Id，以检索类型化的 Workflow 句柄：

```csharp
// ...

[Workflow]
public class WorkflowB
{
    [WorkflowRun]
    public async Task RunAsync()
    {
        var handle = Workflow.GetExternalWorkflowHandle<WorkflowA>("workflow-a");
        await handle.SignalAsync(wf => wf.YourSignalAsync("signal argument"));
    }

    // ...
```

当发送外部 Signal 时：

- [Signal外部Workflow执行已启动](/references/events#signalexternalworkflowexecutioninitiated) 事件出现在发件人的 Event History 中。
- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在收件人的 Event History 中。

<a id="signal-with-start"></a>
#### Signal-带启动 {#signal-with-start}
Signal-With-Start 允许客户端将 Signal 发送到 Workflow Execution，如果尚未运行则启动执行。
如果有一个 Workflow 使用给定的 Workflow Id 运行，则它将是 Signaled。
如果没有，将启动新的 Workflow 并立即删除 Signal。
要使用 Signal-With-Start，请使用调用它的 lambda 表达式来调用 `SignalWithStart`：

```csharp
var client = await TemporalClient.ConnectAsync(new("localhost:7233"));
var options = new WorkflowOptions(id: "your-signal-with-start-workflow", taskQueue: "signal-tq");
options.SignalWithStart((GreetingWorkflow wf) => wf.SubmitGreetingAsync("User Signal with Start"));
await client.StartWorkflowAsync((GreetingWorkflow wf) => wf.RunAsync(), options);
```

<a id="send-update-from-client"></a>
### 发送 Update {#send-update-from-client}
Update 是同步、阻塞调用，可以更改 Workflow 状态、控制其流程并返回结果。

发送 Update 的客户端必须等待，直到服务器将 Update 传递给 Worker。
Worker 必须可用且响应迅速。
如果服务器收到请求后需要立即响应，请改用 Signal。
另请注意，您无法将 Update 发送到其他 Workflow Execution。

- 当Worker确认Update通过验证时，`WorkflowExecutionUpdateAccepted`被添加到Event History。
- 当Worker确认Update已完成时，`WorkflowExecutionUpdateCompleted`被添加到Event History。

要将 Update 发送到 Workflow Execution，您可以：

- 从客户端使用 `ExecuteUpdateAsync` 调用 Update 方法并等待 Update 完成。
  此代码获取 Update 结果：

  ```csharp
  var previousLanguage = await workflowHandle.ExecuteUpdateAsync(
    wf => wf.SetCurrentLanguageAsync(GreetingWorkflow.Language.Chinese));
  ```

2. 在接受 Update 后，立即使用 `StartUpdateAsync` 接收句柄。
     它返回 `UpdateHandle`

- 稍后使用此 `UpdateHandle` 获取结果。
  - 异步 Update 处理程序通常执行长时间运行的异步活动。
  - `StartUpdateAsync` 仅等待直到 Worker 接受或拒绝 Update，而不等待所有异步操作完成。

例如：

  ```csharp
  // Wait until the update is accepted
  var updateHandle = await workflowHandle.StartUpdateAsync(
      wf => wf.SetGreetingAsync(new HelloWorldInput("World")),
      new(waitForStage: WorkflowUpdateStage.Accepted));
  // Wait until the update is completed
  var updateResult = await updateHandle.GetResultAsync();
  ```

有关更多详细信息，请参阅“异步处理程序”部分。

#### Update-with-Start
:::tip

对于开源服务器用户，建议使用Temporal Server版本[Temporal Server 版本 1.28](https://github.com/temporalio/temporal/releases/tag/v1.28.0)。

:::

[Update-带启动](/sending-messages#update-with-start) 允许您 [发送 Update](/develop/dotnet/workflows/message-passing#send-update-from-client) 检查是否存在具有该 ID 的已运行的 Workflow：

- 如果 Workflow 存在，则处理 Update。
- 如果 Workflow 不存在，则使用给定 ID 启动新的 Workflow Execution，并在主 Workflow 方法开始执行之前处理 Update。

使用`ExecuteUpdateWithStartAsync`启动Update，一次性等待结果。

或者，使用 `StartUpdateWithStartAsync` 启动 Update 并接收 `WorkflowUpdateHandle`，然后使用 `await UpdateHandle.GetResultAsync()` 从 Update 检索结果。

一旦到达所请求的 Update 等待阶段或请求超时，这些调用就会返回。

- 您需要提供 `WithStartWorkflowOperation` 来定义必要时启动的 Workflow 及其参数。
- 创建 `WithStartWorkflowOperation` 时必须指定 [ID冲突策略](/workflow-execution/workflowid-runid#workflow-id-conflict-policy)。
  请注意，`WithStartWorkflowOperation` 只能使用一次。

以下是取自 [UpdateWithStartLazyInit](https://github.com/temporalio/samples-dotnet/blob/main/src/UpdateWithStartLazyInit/Program.cs) 样本的示例：

```csharp
async Task<AddCartItemResult> AddCartItemAsync(string sessionId, ShoppingCartItem item)
{
    // Issue an update-with-start that will create the workflow if it does not
    // exist before attempting the update

    // Create the start operation
    var startOperation = WithStartWorkflowOperation.Create(
        (ShoppingCartWorkflow wf) => wf.RunAsync(),
        new(id: $"cart-{sessionId}", taskQueue: TaskQueue)
        {
            IdConflictPolicy = Temporalio.Api.Enums.V1.WorkflowIdConflictPolicy.UseExisting,
        });

    // Issue the update-with-start, swallowing item-unavailable failure
    decimal? subtotal;
    try
    {
        subtotal = await client.ExecuteUpdateWithStartWorkflowAsync(
            (ShoppingCartWorkflow wf) => wf.AddItemAsync(item),
            new(startOperation));
    }
    catch (WorkflowUpdateFailedException e) when (
        e.InnerException is ApplicationFailureException appErr && appErr.ErrorType == "ItemUnavailable")
    {
        // Set subtotal to null if item was not found
        subtotal = null;
    }

    return new(await startOperation.GetHandleAsync(), subtotal);
}
```

<a id="async-handlers"></a>
### 添加异步处理程序以使用 `await` {#async-handlers}
Signal 和 Update 处理程序可以是异步的，也可以是阻塞的。
使用异步调用允许您执行 `await` 活动、Child Workflow、[`Workflow.DelayAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_DelayAsync_System_Int32_System_Nullable_System_Threading_CancellationToken__) Timer、[`Workflow.WaitConditionAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_WaitConditionAsync_System_Func_System_Boolean__System_Int32_System_Nullable_System_Threading_CancellationToken__) 等待条件等。
这扩展了处理程序可以完成的操作的可能性，但这也意味着处理程序执行和您的主 Workflow 方法都是同时运行的，并在调用 wait 时在它们之间进行切换。

为了安全地使用异步处理程序，了解可能出错的事情至关重要。
有关异步 Signal 和 Update 处理程序的安全使用指南，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing) 以及下面的 [控制处理程序并发](#control-handler-concurrency) 和 [等待消息处理程序完成](#wait-for-message-handlers) 部分。

以下代码执行模拟对远程服务的网络调用的 Activity：

```csharp
public class MyActivities
{
    private static readonly Dictionary<Language, string> Greetings = new()
    {
        [Language.Arabic] = "مرحبا بالعالم",
        [Language.Chinese] = "你好，世界",
        [Language.English] = "Hello, world",
        [Language.French] = "Bonjour, monde",
        [Language.Hindi] = "नमस्ते दुनिया",
        [Language.Spanish] = "Hola mundo",
    };

    [Activity]
    public async Task<string?> CallGreetingServiceAsync(Language language)
    {
        // Pretend that we are calling a remove service
        await Task.Delay(200);
        return Greetings.TryGetValue(language, out var value) ? value : null;
    }
}
```

以下代码修改 `WorkflowUpdate` 以异步使用前面的 Activity：

```csharp
[Workflow]
public class GreetingWorkflow
{
    private readonly Mutex mutex = new();

    // ...

    [WorkflowUpdate]
    public async Task<Language> SetLanguageAsync(Language language)
    {
        // 👉 Use a mutex here to ensure that multiple calls to SetLanguageAsync are processed in order.
        await mutex.WaitOneAsync();
        try
        {
            if (!greetings.ContainsKey(language))
            {
                var greeting = Workflow.ExecuteActivityAsync(
                    (MyActivities acts) => acts.CallGreetingServiceAsync(language),
                    new() { StartToCloseTimeout = TimeSpan.FromSeconds(10) });
                if (greeting == null)
                {
                    // 👉 An update validator cannot be async, so cannot be used to check that the remote
                    // CallGreetingServiceAsync supports the requested language. Throwing ApplicationFailureException
                    // will fail the Update, but the WorkflowExecutionUpdateAccepted event will still be
                    // added to history.
                    throw new ApplicationFailureException(
                        $"Greeting service does not support {language}");
                }
                greetings[language] = greeting;
            }
            var previousLanguage = CurrentLanguage;
            CurrentLanguage = language;
            return previousLanguage;
        }
        finally
        {
            mutex.ReleaseMutex();
        }
    }
}
```

更新异步调用的代码后，您的 Update 处理程序可以 Schedule 和 Activity 并等待结果。
尽管异步 Signal 处理程序可以启动类似的网络任务，但使用 Update 处理程序允许客户端在 Activity 完成后接收结果或错误。
这使您的客户端可以跟踪 Update 的活动、Child Workflow 等执行的异步工作的进度。

<a id="block-with-wait"></a>
### 添加等待条件来阻止 {#block-with-wait}
有时，异步 Signal 或 Update 处理程序需要满足某些条件才能继续。
将等待条件与 [`Workflow.WaitConditionAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_WaitConditionAsync_System_Func_System_Boolean__System_Int32_System_Nullable_System_Threading_CancellationToken__) 一起使用会设置一个函数，该函数会阻止代码继续执行，直到条件返回 `true`。
这是一个重要的功能，可以帮助您控制处理程序逻辑。

以下是 `Workflow.WaitConditionAsync` 的两个重要用例：

- 在处理程序中等待，直到适合继续。
- 在主 Workflow 中等待，直到所有活动处理程序完成。

您正在等待的条件状态可以是 Updated，并反映 Workflow 代码的任何部分。
这包括主 Workflow 方法、其他处理程序或由主 Workflow 方法生成的子协程等。

<a id="wait-in-handlers"></a>
#### 在处理程序中使用等待条件 {#wait-in-handlers}
有时，异步 Signal 或 Update 处理程序需要满足某些条件才能继续。
将等待条件与 [`Workflow.WaitConditionAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_WaitConditionAsync_System_Func_System_Boolean__System_Int32_System_Nullable_System_Threading_CancellationToken__) 一起使用会设置一个函数，该函数会阻止代码继续执行，直到条件返回 `true`。
这是一个重要的功能，可以帮助您控制处理程序逻辑。

考虑在 Update 处理程序执行之前运行的 `ReadyForUpdateToExecute` 方法。
[`Workflow.WaitConditionAsync`](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html?#Temporalio_Workflows_Workflow_WaitConditionAsync_System_Func_System_Boolean__System_Int32_System_Nullable_System_Threading_CancellationToken__) 调用将等待，直到满足您的条件：

```csharp
[WorkflowUpdate]
public async Task<string> MyUpdateAsync(UpdateInput updateInput)
{
    await Workflow.WaitConditionAsync(() => ReadyForUpdateToExecute(updateInput));
    // ...
}
```

请记住：处理程序可以在主 Workflow 方法启动之前执行。

<a id="wait-for-message-handlers"></a>
#### 确保您的处理程序在 Workflow 完成 {#wait-for-message-handlers} 之前完成
Workflow 等待条件可以确保您的处理程序在 Workflow 完成之前完成。
当您的 Workflow 使用异步 Signal 或 Update 处理程序时，您的主 Workflow 方法可以在处理程序仍在等待异步任务（例如 Activity 结果）时返回 或 Continue-As-New 。
Workflow 完成可能会在处理程序完成关键工作之前中断处理程序，并在尝试检索 Update 结果时导致客户端错误。
使用`Workflow.AllHandlersFinished`来解决这个问题，让你的Workflow顺利结束：

```csharp
[Workflow]
public class MyWorkflow
{
    [WorkflowRun]
    public async Task<string> RunAsync()
    {
        // ...
        await Workflow.WaitConditionAsync(() => Workflow.AllHandlersFinished);
        return "workflow-result";
    }

    // ...
```

默认情况下，当您允许 Workflow Execution 完成未完成的处理程序执行时，您的 Worker 将记录一条警告。
您可以通过将 `UnfinishedPolicy` 参数传递给 [`WorkflowSignalAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowSignalAttribute.html) / [`WorkflowUpdateAttribute`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowUpdateAttribute.html) 装饰器来在每个处理程序的基础上消除这些警告：

```csharp
[WorkflowUpdate(UnfinishedPolicy = HandlerUnfinishedPolicy.Abandon)]
public async Task MyUpdateAsync()
{
    // ...
```

请参阅 [在 Workflow 完成之前完成处理程序](/handling-messages#finishing-message-handlers) 了解更多信息。

<a id="use-workflowinit-to-operate-on-workflow-input-before-any-handler-executes"></a>
### 在执行任何处理程序之前，使用 `[WorkflowInit]` 对 Workflow 输入进行操作 {#use-workflowinit-to-operate-on-workflow-input-before-any-handler-executes}

`[WorkflowInit]` 属性使消息处理程序可以访问 [Workflow 输入](/handling-messages#workflow-initializers)。
当您在构造函数上使用 `[WorkflowInit]` 属性时，您可以为构造函数提供与 `[WorkflowRun]` 方法相同的 Workflow 参数。
然后，SDK 将确保您的构造函数接收到 [客户发送](/develop/dotnet/client/temporal-client#start-workflow) 的 Workflow 输入参数。
Workflow 输入参数也会传递到您的 `[WorkflowRun]` 方法 - 无论您是否使用 `[WorkflowInit]` 属性，这种情况总是会发生。

这是一个例子。
构造函数和 `RunAsync` 必须具有相同类型的相同参数：

```csharp
[Workflow]
public class WorkflowInitWorkflow
{
    public record Input(string Name);

    private readonly string nameWithTitle;
    private bool titleHasBeenChecked;

    [WorkflowInit]
    public WorkflowInitWorkflow(Input input) =>
        nameWithTitle = $"Sir {input.Name}";

    [WorkflowRun]
    public async Task<string> RunAsync(Input ignored)
    {
        await Workflow.WaitConditionAsync(() => titleHasBeenChecked);
        return $"Hello, {nameWithTitle}";
    }

    [WorkflowUpdate]
    public async Task<bool> CheckTitleValidityAsync()
    {
        // The handler is now guaranteed to see the workflow input after it has
        // been processed by the constructor.
        var valid = await Workflow.ExecuteActivityAsync(
            (MyActivities acts) -> acts.CheckTitleValidityAsync(nameWithTitle),
            new() { StartToCloseTimeout = TimeSpan.FromSeconds(10) });
        titleHasBeenChecked = true;
        return valid;
    }
}
```

<a id="control-handler-concurrency"></a>
### 使用锁来防止并发处理程序执行 {#control-handler-concurrency}
并发进程可以以不可预测的方式交互。
当多个处理程序实例同时运行时，编写不正确的 [并发消息传递](/handling-messages#message-handler-concurrency) 代码可能无法正常工作。
这是一个病态病例的例子：

```csharp
[Workflow]
public class MyWorkflow
{
    // ...

    [WorkflowSignal]
    public async Task BadHandlerAsync()
    {
        var data = await Workflow.ExecuteActivityAsync(
            (MyActivities acts) => acts.FetchDataAsync(),
            new() { StartToCloseTimeout = TimeSpan.FromSeconds(10) });
        this.x = data.X;
        // 🐛🐛 Bug!! If multiple instances of this handler are executing concurrently, then
        // there may be times when the Workflow has this.x from one Activity execution and this.y from another.
        await Workflow.DelayAsync(1000);
        this.y = data.Y;
    }
}
```

与互斥锁 [`Workflows.Mutex`](https://dotnet.temporal.io/api/temporalio.Workflows.Mutex.html) 协调访问可以更正此代码。
锁定可确保在任何给定时间只有一个处理程序实例可以执行特定的代码部分：

```csharp
[Workflow]
public class MyWorkflow
{
    private readonly Mutex mutex = new();

    // ...

    [WorkflowSignal]
    public async Task SafeHandlerAsync()
    {
        await mutex.WaitOneAsync();
        try
        {
            var data = await Workflow.ExecuteActivityAsync(
                (MyActivities acts) => acts.FetchDataAsync(),
                new() { StartToCloseTimeout = TimeSpan.FromSeconds(10) });
            this.x = data.X;
            // ✅ OK: the scheduler may switch now to a different handler execution, or to the main workflow
            // method, but no other execution of this handler can run until this execution finishes.
            await Workflow.DelayAsync(1000);
            this.y = data.Y;
        }
        finally
        {
            mutex.ReleaseMutex();
        }
    }
}
```

对于其他并发选项，您可以使用 [`Workflows.Semaphore`](https://dotnet.temporal.io/api/temporalio.Workflows.Semaphore.html)。
信号量管理对共享资源的访问并协调线程或进程的执行顺序。

<a id="message-handler-troubleshooting"></a>
## 消息处理程序故障排除 {#message-handler-troubleshooting}
将 Signal、Update 或 Query 发送到 Workflow 时，您的客户端可能会遇到以下错误：

- **客户端无法联系服务器**：
  您将收到 [`Temporalio.Exceptions.RpcException`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.html) 异常，其 `Code` 属性为 [`RpcException.StatusCode`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.StatusCode.html)，状态为 `Unavailable`（重试几次后）。

- **Workflow 不存在**：
  您将收到 [`Temporalio.Exceptions.RpcException`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.html) 异常，其 `Code` 属性为 [`RpcException.StatusCode`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.StatusCode.html)，状态为 `NotFound`。

有关此主题的非特定于 .NET 的讨论，请参阅 [消息处理程序中的异常](/handling-messages#exceptions)。

<a id="signal-problems"></a>
### 发送 Signal 时出现问题 {#signal-problems}
使用 Signal 时，您的请求在执行期间产生的唯一异常是 `RpcException`。
所有处理程序在处理程序请求生命周期的初始（Worker 之前）部分都可能会遇到其他异常。

对于查询和 Update，客户端等待来自 Worker 的响应。
如果在 Worker 执行处理程序期间出现问题，客户端可能会收到异常。

<a id="update-problems"></a>
### 发送 Update 时出现问题 {#update-problems}
使用 Update 时，您可能会遇到以下错误：

- **没有 Workflow Worker 正在轮询 Task Queue**：
  SDK 客户端将无限期地重试您的请求。
  使用 [远程过程调用选项](https://dotnet.temporal.io/api/temporalio.Client.WorkflowUpdateOptions.html#Temporalio_Client_WorkflowUpdateOptions_Rpc) 中的 `CancellationToken` 取消 Update。
  这会引发 [Temporalio.Exceptions.WorkflowUpdateRpcTimeoutOrCanceledException](https://dotnet.temporal.io/api/temporalio.Exceptions.WorkflowUpdateRpcTimeoutOrCanceledException.html) 异常。

- **Update 失败**：您将收到 [`Temporalio.Exceptions.WorkflowUpdateFailedException`](https://dotnet.temporal.io/api/temporalio.Exceptions.WorkflowUpdateFailedException.html) 异常。
  有两种方式可以发生这种情况：

- Update 被 Workflow 中与 Update 处理程序一起定义的 Update 验证器拒绝。

- Update 被接受后失败。

Update 故障类似于 [Workflow 故障](/references/failures)。
  导致 main 方法中的 Workflow 失败的问题也会导致 Update 处理程序中的 Update 失败。
  这些可能包括：

- 失败的Child Workflow
      - 失败的 Activity（如果 Activity 重试已设置为有限次数）
      - Workflow 作者抚养 `ApplicationFailure`
      - Worker 上的 [`TemporalWorkerOptions.WorkflowFailureExceptionTypes`](https://dotnet.temporal.io/api/temporalio.Worker.TemporalWorkerOptions.html#Temporalio_Worker_TemporalWorkerOptions_WorkflowFailureExceptionTypes) 或 Workflow 上的 [`WorkflowAttribute.FailureExceptionTypes`](https://dotnet.temporal.io/api/temporalio.Workflows.WorkflowAttribute.html#Temporalio_Workflows_WorkflowAttribute_FailureExceptionTypes) 中列出的任何错误（默认为空）

- **处理程序导致 Workflow 任务失败**：
  [Workflow 任务失败](/references/failures) 导致服务器无限期地重试 Workflow 任务。您的 Update 请求会发生什么情况取决于其阶段：
  - 如果服务器未接受请求，您将收到 `FAILED_PRECONDITION` [`Temporalio.Exceptions.RpcException`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.html) 异常。
  - 如果请求已被接受，则该请求是持久的。
    代码部署后，一旦 Workflow 再次正常运行，请使用 [`UpdateHandle`](https://dotnet.temporal.io/api/temporalio.Client.WorkflowUpdateHandle.html) 获取 Update 结果。

- **Workflow 已完成，而 Update 处理程序执行正在进行中**：
  您将收到 [`Temporalio.Exceptions.RpcException`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.html)“Workflow Execution 已完成”。

如果 Workflow 完成而 Update 处理程序执行正在进行中，则会发生这种情况，例如因为

- Workflow 已取消或失败。

- Workflow 正常完成或继续如新，而 Workflow 作者没有 [等待处理程序完成](/handling-messages#finishing-message-handlers)。

<a id="query-problems"></a>
### 发送 Query 时出现问题 {#query-problems}
使用查询时，您可能会遇到以下错误：

- **没有 Workflow Worker 轮询 Task Queue**：
  您将收到 [`Temporalio.Exceptions.RpcException`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.html)，其中 `Code` 是 [`RpcException.StatusCode`](https://dotnet.temporal.io/api/temporalio.Exceptions.RpcException.StatusCode.html)，状态为 `FailedPrecondition`。

- **Query 失败**：
  如果 Query 期间出现问题，您将收到 [`Temporalio.Exceptions.WorkflowQueryFailedException`](https://dotnet.temporal.io/api/temporalio.Exceptions.WorkflowQueryFailedException.html) 异常。
  Query 处理程序中的任何异常都会触发此错误。
  这与 Signal 和 Update 请求不同，其中异常可能会导致 Workflow 任务失败。

- **处理程序导致 Workflow 任务失败。**
  例如，如果 Query 处理程序阻塞线程太长时间而没有屈服，就会发生这种情况。

<a id="dynamic-handler"></a>
## 动态处理程序 {#dynamic-handler}
Temporal 支持动态查询、Signal、Update、Workflow 和活动。
这些是未命名的处理程序，如果不存在具有给定名称的其他静态定义的处理程序，则会调用这些处理程序。

动态处理程序可以灵活地处理运行时未知的查询、Signal、Update、Workflow 或活动的名称的情况。

:::caution

应明智地使用动态处理程序作为后备机制而不是主要方法。
过度使用它们可能会导致可维护性和调试问题。

相反，Signal、查询、Workflow 或活动应尽可能静态定义，并使用表明其用途的清晰名称。
使用静态定义作为构建 Workflow 的主要方式。

保留动态处理程序，用于处理程序名称在编译时未知且需要在运行时动态查找的情况。
它们旨在处理边缘情况并充当包罗万象的角色，而不是作为调用逻辑的主要方式。

:::

<a id="set-a-dynamic-query"></a>
### 设置动态 Query {#set-a-dynamic-query}
Temporal 中的动态 Query 是一个 Query 方法，如果没有注册其他同名的 Query，则在运行时动态调用该方法。
通过将 `[WorkflowQuery]` 属性上的 `Dynamic` 设置为 `true` 可以使 Query 成为动态的。
Workflow 上只能存在一个 Dynamic Query。

Query 处理程序参数必须接受 `string` 名称和 `Temporalio.Converters.IRawValue[]` 作为参数。
[Workflow.PayloadConverter](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_PayloadConverter) 属性用于使用 `Temporalio.Converters` Namespace 中的扩展方法将 `IRawValue` 对象转换为所需类型。

```csharp
[WorkflowQuery(Dynamic = true)]
public string DynamicQueryAsync(string queryName, IRawValue[] args)
{
    var input = Workflow.PayloadConverter.ToValue<MyStatusParam>(args.Single());
    return statuses[input.Type];
}
```

<a id="set-a-dynamic-signal"></a>
### 设置动态 Signal {#set-a-dynamic-signal}
Temporal 中的动态 Signal 是在运行时动态调用的 Signal，如果没有注册具有相同输入的其他 Signal。
通过将 `[WorkflowSignal]` 属性上的 `Dynamic` 设置为 `true` 可以使 Signal 成为动态的。
Workflow 上只能存在一个 Dynamic Signal。

Signal 处理程序参数必须接受 `string` 名称和 `Temporalio.Converters.IRawValue[]` 作为参数。
[Workflow.PayloadConverter](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_PayloadConverter) 属性用于使用 `Temporalio.Converters` Namespace 中的扩展方法将 `IRawValue` 对象转换为所需类型。

```csharp
[WorkflowSignal(Dynamic = true)]
public async Task DynamicSignalAsync(string signalName, IRawValue[] args)
{
    var input = Workflow.PayloadConverter.ToValue<DoSomethingParam>(args.Single());
    pendingThings.Add(input);
}
```

<a id="set-a-dynamic-update"></a>
### 设置动态 Update {#set-a-dynamic-update}
Temporal 中的动态 Update 是在运行时动态调用的 Update，如果没有注册具有相同输入的其他 Update。
通过将 `[WorkflowUpdate]` 属性上的 `Dynamic` 设置为 `true` 可以使 Update 成为动态的。

Update 处理程序参数必须接受 `string` 名称和 `Temporalio.Converters.IRawValue[]` 作为参数。
[Workflow.PayloadConverter](https://dotnet.temporal.io/api/temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_PayloadConverter) 属性用于使用 `Temporalio.Converters` Namespace 中的扩展方法将 `IRawValue` 对象转换为所需类型。

```csharp
[WorkflowUpdate(Dynamic = true)]
public async Task<string> DynamicUpdateAsync(string updateName, IRawValue[] args)
{
    var input = Workflow.PayloadConverter.ToValue<DoSomethingParam>(args.Single());
    pendingThings.Add(input);
    return statuses[input.Type];
}
```
