## 什么是 Continue-As-New？ {#what}

[Continue-As-New](/workflow-execution/continue-as-new) 让 Workflow Execution 成功关闭并创建一个新的 Workflow Execution。
当你的 Workflow 运行时间过长或接近某些扩展限制时，可以将其视为一个检查点。

新的 Workflow Execution 处于同一[执行链](/workflow-execution#workflow-execution-chain)中；它保留相同的 Workflow Id，但会获得新的 Run Id 和全新的 Event History。
它还会接收 Workflow 的常规参数。

## 如何使用 .NET SDK 实现 Continue-As-New {#how}

首先，设计你的 Workflow 参数，以便在 Continue-As-New 进入下一次 Workflow 运行时能够传入"当前状态"。
对于 Workflow 的原始调用方，该状态通常设置为 `None`。

### 含 Message Handler 的 Workflow 的注意事项 {#with-message-handlers}

如果你使用了 Update 或 Signal，请不要在处理器中调用 Continue-as-New。
应在主 Workflow 中等待处理器执行完毕后，再抛出 `CreateContinueAsNewException`。
参见 [`AllHandlersFinished`](message-passing#wait-for-message-handlers) 示例以获取指导。

## 什么时候适合使用 .NET SDK 的 Continue-as-New？ {#when}

当你的 Workflow 可能触及 [Event History 限制](/workflow-execution/event#event-history)时，使用 Continue-as-New。

Temporal 会追踪 Workflow 的进度是否接近这些限制，并告知你何时应该使用 Continue-as-New。
调用 `Workflow.ContinueAsNewSuggested` 来检查是否到了该使用的时候。

## 如何使用 .NET SDK 测试 Continue-as-New {#how-to-test}

对自然触发 Continue-as-New 的 Workflow 进行测试可能既耗时又消耗资源。
建议添加测试钩子，以便在自动化测试中更快速地检验 Workflow 的 Continue-as-New 行为。

例如，当 `TestContinueAsNew == true` 时，此示例会创建一个仅用于测试的变量 `maxHistoryLength`，并将其设置为较小的值。
Workflow 中的辅助变量会在每次考虑使用 Continue-as-New 时进行检查：

```csharp
private bool ShouldContinueAsNew =>
    // Don't continue as new while update running
    Workflow.AllHandlersFinished &&
    // Continue if suggested or, for ease of testing, max history reached
    (Workflow.ContinueAsNewSuggested || Workflow.CurrentHistoryLength > maxHistoryLength);
```
