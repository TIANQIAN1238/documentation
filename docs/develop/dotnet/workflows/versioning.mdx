由于 Temporal 中的 Workflow Execution 可能会长期运行——有时长达数月甚至数年——即使在特定 Workflow Execution 进行期间，也经常需要对 Workflow Definition 进行修改。

Temporal Platform 要求 Workflow 代码具有[确定性](/workflow-definition#deterministic-constraints)。
如果对 Workflow 代码的修改会在 Replay 时导致非确定性行为，则需要使用我们的某种版本管理方法来优雅地更新正在运行的 Workflow。
使用版本管理，你可以修改 Workflow Definition，使新的执行使用更新后的代码，而现有的执行继续运行原始版本。
主要有两种版本管理方法可供使用：

- [Worker Versioning](/production-deployment/worker-deployments/worker-versioning)。Worker Versioning 功能允许你为 Worker 打标签，并以版本化部署的方式以编程方式滚动更新，使旧 Worker 能够运行旧代码路径，新 Worker 能够运行新代码路径。
- [基于 Patching 的版本管理](#patching)。此方法通过在代码中添加绑定到特定版本的分支来实现。它将代码变更应用于新的 Workflow Execution，同时避免对进行中的 Workflow Execution 造成破坏性影响。

## Worker Versioning

Temporal 的 [Worker Versioning](/production-deployment/worker-deployments/worker-versioning) 功能允许你为 Worker 打标签，并以 Deployment Version 的方式以编程方式滚动更新，使旧 Worker 能够运行旧代码路径，新 Worker 能够运行新代码路径。这样，你可以将 Workflow 固定到特定版本，从而避免 patching 的需求。

## 基于 Patching 的版本管理 {#patching}

### 添加 patch

Patch 在 Workflow 中为特定变更定义了一个逻辑分支，类似于功能开关。
它将代码变更应用于新的 Workflow Execution，同时避免对进行中的 Workflow Execution 造成破坏性影响。
当你想要对可能影响现有 Workflow Execution 的代码进行实质性修改时，请创建 patch。注意，[已固定的 Workflow](/worker-versioning) 无需 patch。

假设你有一个名为 `PrePatchActivity` 的初始 Workflow 版本：

```csharp
[Workflow]
public class MyWorkflow
{
    [WorkflowRun]
    public async Task RunAsync()
    {
        this.result = await Workflow.ExecuteActivityAsync(
            (MyActivities a) => a.PrePatchActivity(),
            new() { StartToCloseTimeout = TimeSpan.FromMinutes(5) });

        // ...
    }
}
```

现在，你想更新代码以运行 `PostPatchActivity`。这代表你期望的最终状态。

```csharp
[Workflow]
public class MyWorkflow
{
    [WorkflowRun]
    public async Task RunAsync()
    {
        this.result = await Workflow.ExecuteActivityAsync(
            (MyActivities a) => a.PostPatchActivity(),
            new() { StartToCloseTimeout = TimeSpan.FromMinutes(5) });

        // ...
    }
}
```

问题在于，在确定没有更多使用 `PrePatchActivity` 代码创建的正在运行的 Workflow 之前，无法直接部署 `PostPatchActivity`，否则很可能导致非确定性错误。
因此，你需要部署 `PostPatchActivity` 并使用 [Patched](https://dotnet.temporal.io/api/Temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_Patched_System_String_) 方法来确定执行哪个版本的代码。

Patching 是一个三步骤流程：

1. 使用 [Patched](https://dotnet.temporal.io/api/Temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_Patched_System_String_) 引入新代码并与旧代码并行运行。
2. 删除旧代码并应用 [DeprecatePatch](https://dotnet.temporal.io/api/Temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_DeprecatePatch_System_String_)。
3. 待所有旧 Workflow 超出保留期后，删除 `DeprecatePatch`。

### 引入新代码的 patch {#using-patched-for-workflow-history-markers}

使用 `Patched` 会在 Workflow History 中插入一个标记。

在 Replay 过程中，如果 Worker 遇到含有该标记的历史记录，而 Workflow 代码未产生相同的 patch 标记（此处为 `my-patch`），则会导致 Workflow 任务失败。这确保你可以将 `PostPatchActivity` 的代码作为"功能开关"与原始版本（`PrePatchActivity`）并行安全部署。

```csharp
[Workflow]
public class MyWorkflow
{
    [WorkflowRun]
    public async Task RunAsync()
    {
        if (Workflow.Patched("my-patch"))
        {
            this.result = await Workflow.ExecuteActivityAsync(
                (MyActivities a) => a.PostPatchActivity(),
                new() { StartToCloseTimeout = TimeSpan.FromMinutes(5) });
        }
        else
        {
            this.result = await Workflow.ExecuteActivityAsync(
                (MyActivities a) => a.PrePatchActivity(),
                new() { StartToCloseTimeout = TimeSpan.FromMinutes(5) });
        }

        // ...
    }
}
```

### 废弃 patch {#deprecated-patches}

在所有使用 `PrePatchActivity` 代码启动的 Workflow 超出保留期后，你可以[废弃该 patch](https://dotnet.temporal.io/api/Temporalio.Workflows.Workflow.html#Temporalio_Workflows_Workflow_DeprecatePatch_System_String_)。

已废弃的 patch 作为 patching 流程最终阶段与不再含有 patch 的最终状态之间的过渡桥梁。其工作方式与普通 patch 类似，也会向 Workflow History 添加标记。但是，当 Workflow 代码未产生该标记时，不会导致 replay 失败。

如果在部署 `PostPatchActivity` 期间，仍有运行 `PrePatchActivity` 代码的活跃 Worker，且这些 Worker 处理了由 `PostPatchActivity` 生成的 Workflow 历史记录，它们将安全地使用已 patch 的分支。

```csharp
[Workflow]
public class MyWorkflow
{
    [WorkflowRun]
    public async Task RunAsync()
    {
        Workflow.DeprecatePatch("my-patch")
        this.result = await Workflow.ExecuteActivityAsync(
            (MyActivities a) => a.PostPatchActivity(),
            new() { StartToCloseTimeout = TimeSpan.FromMinutes(5) });

        // ...
    }
}
```

### 移除 patch {#deploy-postpatchactivity}

根据上述断言，一旦所有标记为 my-patch 或更早版本的 Workflow 超出保留期，即可安全部署 `PostPatchActivity`。

```csharp
[Workflow]
public class MyWorkflow
{
    [WorkflowRun]
    public async Task RunAsync()
    {
        this.result = await Workflow.ExecuteActivityAsync(
            (MyActivities a) => a.PostPatchActivity(),
            new() { StartToCloseTimeout = TimeSpan.FromMinutes(5) });

        // ...
    }
}
```

Patching 允许你对当前正在运行的 Workflow 进行修改。
这是一种引入兼容性变更而不产生非确定性错误的强大方法。

### Workflow 切换

要理解 Patching 的用处，可以通过演示整个 Workflow 的切换来加以说明。

由于不兼容的变更只影响同类型的已开启的 Workflow Execution，你可以通过在进行变更时创建全新的 Workflow 来避免确定性错误。为此，你可以复制 Workflow Definition 函数并赋予不同的名称，然后在 Worker 中注册这两个名称。

例如，你可以将 `SayHelloWorkflow` 复制为 `SayHelloWorkflowV2`：

```csharp
[Workflow]
public class SayHelloWorkflow
{
    [WorkflowRun]
    # this function contains the original code
}
        
[Workflow]
public class SayHelloWorkflowV2
{
    [WorkflowRun]
    # this function contains the updated code
}
```

然后，你需要更新 Worker 配置以及其他标识符字符串，以注册这两种 Workflow 类型：

```csharp
using var worker = new TemporalWorker(
    client,
    new TemporalWorkerOptions("greeting-tasks")
        .AddWorkflow<SayHelloWorkflow>()
        .AddWorkflow<SayHelloWorkflowV2>());
```

此方法的缺点是需要复制代码，并且需要更新用于启动 Workflow 的所有命令。随着时间的推移，这可能变得不切实际。此方法也无法对仍在运行的 Workflow 进行版本管理——它本质上只是一次切换，而不像 Patching 那样具有版本管理能力。

### 测试 Workflow 的 replay 安全性

要判断你的 Workflow 是否需要 patch，或者你的 patch 是否成功，应纳入 [Replay 测试](/develop/dotnet/best-practices/testing-suite#replay)。
