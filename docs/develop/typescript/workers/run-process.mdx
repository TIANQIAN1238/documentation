<a id="run-a-dev-worker"></a>
## 如何运行 Worker 进程 {#run-a-dev-worker}

[Worker 流程](/workers#worker-process) 是执行 Workflow 函数和 Activity 函数的地方。

- Worker 进程中的每个 [Worker 实体](/workers#worker-entity) 必须注册准确的 Workflow 类型和 Activity
  它可以执行的类型。
- 每个 Worker 实体还必须将其自身与一个 [Task Queue](/task-queue) 关联起来。
- 轮询相同 Task Queue 的每个 Worker 实体必须使用相同的 Workflow 类型和 Activity 类型进行注册。

[Worker 实体](/workers#worker-entity) 是 Worker 进程中侦听特定任务的组件
队列。

虽然多个 Worker 实体可以位于单个 Worker 进程中，但单个 Worker 实体 Worker 进程可以是
完全足够了。有关详细信息，请参阅 [Worker 调音指南](/develop/worker-performance)。

Worker 实体包含 Workflow Worker 和/或 Activity Worker，这在 Workflow Execution 上取得了进展，并且
Activity 分别执行。

<a id="run-a-worker-on-docker"></a>
## 如何使用 TypeScript 在 Docker 上运行 Worker {#run-a-worker-on-docker}

:::note

为了缩短 Worker 启动时间，我们建议提前准备 Workflow 捆绑包。看看我们的
[生产样品](https://github.com/temporalio/samples-typescript/tree/main/production) 了解详情。

:::

基于 TypeScript SDK 的 Worker 可以作为 Docker 容器部署和运行。

我们建议使用 LTS Node.js 版本，例如 18、20、22 或 24。`amd64` 和 `arm64` 架构均受支持。一个
需要基于 glibc 的镜像；不支持基于 musl 的图像（见下文）。

在 Docker 上部署 TypeScript SDK Worker 的最简单方法是从 `node:20-bullseye` 映像开始。例如：

```dockerfile
FROM node:20-bullseye

# For better cache utilization, copy package.json and lock file first and install the dependencies before copying the
# rest of the application and building.
COPY . /app
WORKDIR /app

# Alternatively, run npm ci, which installs only dependencies specified in the lock file and is generally faster.
RUN npm install --only=production \
    && npm run build

CMD ["npm", "start"]
```

对于较小的映像和/或更安全的部署，还可以使用 `-slim` Docker 映像变体（例如
`node:20-bullseye-slim`) 或 `distroless/nodejs` Docker 镜像（如 `gcr.io/distroless/nodejs20-debian11`）
以下警告。

<a id="using-nodeslim-images"></a>
### 使用 `node:slim` 图像 {#using-nodeslim-images}

`node:slim` 映像不包含常规映像中的一些常见包。这导致显着
较小的图像。

但是，TypeScript SDK 需要存在根 TLS 证书（`ca-certificates` 包），而该证书不存在
包含在 `slim` 图像中。即使连接到本地 Temporal Server 或 `ca-certificates` 包也是必需的
当使用未明确使用 TLS 的服务器连接配置时。

因此，在构建 Docker 镜像时必须安装 `ca-certificates` 包。例如：

```dockerfile
FROM node:20-bullseye-slim

RUN apt-get update \
    && apt-get install -y ca-certificates \
    && rm -rf /var/lib/apt/lists/*

# ... same as with regular image
```

未能安装此依赖项会导致 `[TransportError: transport error]` 运行时错误，因为
无法验证证书。

<a id="using-distrolessnodejs-images"></a>
### 使用 `distroless/nodejs` 图像 {#using-distrolessnodejs-images}

`distroless/nodejs` 映像仅包含执行 `node` 严格所需的文件。这导致甚至
较小的图像（大约是 `node:slim` 图像大小的一半）。它还显着减少了表面
黑客可能会在生成的 Docker 镜像中利用潜在的安全问题。

使用 `distroless/nodejs` 图像执行 TypeScript SDK Worker 通常是可能且安全的（除非您的代码
本身需要 `distroless/nodejs` 中未包含的依赖项）。

但是，构建过程所需的一些工具（特别是 `npm` 命令）并未包含在
`distroless/nodejs` 图像。这可能会导致 Docker 构建期间出现各种错误消息。

推荐的解决方案是使用多步骤 Dockerfile。例如：

```dockerfile
# -- BUILD STEP --

FROM node:20-bullseye AS builder

COPY . /app
WORKDIR /app

RUN npm install --only=production \
    && npm run build

# -- RESULTING IMAGE --

FROM gcr.io/distroless/nodejs20-debian11

COPY --from=builder /app /app
WORKDIR /app

CMD ["node", "build/worker.js"]
```

<a id="properly-configure-nodejs-memory-in-docker"></a>
### 在 Docker 中正确配置 Node.js 内存 {#properly-configure-nodejs-memory-in-docker}

默认情况下，`node` 将其最大旧代内存配置为其所在计算机的_物理内存_的 25%
执行，最大 4 GB。当在 Docker 环境中运行 Node.js 时，这可能不合适，并且可以
导致可用内存使用不足（`node` 仅使用分配给容器的内存的一小部分）
或过度使用（`node` 尝试使用比分配给容器的内存更多的内存，这最终会导致
进程被操作系统杀死）。

因此，我们建议您始终将 `--max-old-space-size` `node` 参数显式设置为大约 80%
您要分配给 `node` 进程的最大大小（以兆字节为单位）。您可能需要一些实验
根据您的具体应用进行调整以找到最合适的值。

在实践中，通常更容易通过
[`NODE_OPTIONS` 环境变量](https://nodejs.org/api/cli.html#node_optionsoptions)。

<a id="do-not-use-alpine"></a>
### 不要使用 Alpine {#do-not-use-alpine}

Alpine 用 musl 替换了 glibc，这与 TypeScript SDK 的 Rust 核心不兼容。如果您收到错误
如下所示，可能是因为您使用的是 Alpine。

```sh
Error: Error loading shared library ld-linux-x86-64.so.2: No such file or directory (needed by /opt/app/node_modules/@temporalio/core-bridge/index.node)
```

或者像这样：

```sh
Error: Error relocating /opt/app/node_modules/@temporalio/core-bridge/index.node: __register_atfork: symbol not found
```

<a id="run-a-temporal-cloud-worker"></a>
## 如何运行 Temporal Cloud Worker {#run-a-temporal-cloud-worker}

要运行使用 [Temporal Cloud](/cloud) 的 Worker，您需要提供额外的连接和客户端选项
包括以下内容：

- 包含您的 [云 Namespace 名称](/namespaces) 和端口号的地址：
  `<Namespace>.<ID>.tmprl.cloud:<port>`。
- mTLS CA 证书。
- mTLS 私钥。

有关管理和生成 Temporal Cloud 客户端证书的更多信息，请参阅
[如何在Temporal Cloud中管理证书](/cloud/certificates)。

有关配置 TLS 以保护 Temporal 服务的网络间和网络内通信的更多信息，请参阅
[Temporal 定制样品](https://github.com/temporalio/samples-server)。

### 如何注册类型 {#register-types}

必须注册所有监听相同 Task Queue 名称的 Worker 才能处理完全相同的 Workflow 类型和
Activity 类型。

如果 Worker 轮询任务以查找它不知道的 Workflow 类型或 Activity 类型，则该任务将失败。然而，
任务失败不会导致关联的 Workflow Execution 失败。

在开发中，使用
[`WorkflowsPath`](https://typescript.temporal.io/api/interfaces/worker.WorkerOptions/#workflowspath)：

<!--SNIPSTART typescript-Worker-create -->
[片段/src/Worker.ts](https://github.com/temporalio/samples-typescript/blob/main/snippets/src/worker.ts)
```ts
import { Worker } from '@temporalio/worker';
import * as activities from './activities';

async function run() {
  const worker = await Worker.create({
    workflowsPath: require.resolve('./workflows'),
    taskQueue: 'snippets',
    activities,
  });

  await worker.run();
}
```
<!--SNIPEND-->

在此代码片段中，Worker 在运行时捆绑 Workflow 代码。

在生产中，您可以通过提前捆绑来缩短 Worker 的启动时间：作为生产构建的一部分，调用
`bundleWorkflowCode`：

<!--SNIPSTART typescript-bundle-Workflow -->
[生产/src/scripts/build-Workflow-bundle.ts](https://github.com/temporalio/samples-typescript/blob/main/production/src/scripts/build-workflow-bundle.ts)
```ts
import { bundleWorkflowCode } from '@temporalio/worker';
import { writeFile } from 'fs/promises';
import path from 'path';

async function bundle() {
  const { code } = await bundleWorkflowCode({
    workflowsPath: require.resolve('../workflows'),
  });
  const codePath = path.join(__dirname, '../../workflow-bundle.js');

  await writeFile(codePath, code);
  console.log(`Bundle written to ${codePath}`);
}
```
<!--SNIPEND-->

然后可以将包传递给 Worker：

<!--SNIPSTART typescript-production-Worker-->
[生产/src/Worker.ts](https://github.com/temporalio/samples-typescript/blob/main/production/src/worker.ts)
```ts
const workflowOption = () =>
  process.env.NODE_ENV === 'production'
    ? {
        workflowBundle: {
          codePath: require.resolve('../workflow-bundle.js'),
        },
      }
    : { workflowsPath: require.resolve('./workflows') };

async function run() {
  const worker = await Worker.create({
    ...workflowOption(),
    activities,
    taskQueue: 'production-sample',
  });

  await worker.run();
}
```
<!--SNIPEND-->

<a id="shut-down-a-worker"></a>
## 如何关闭 Worker 并跟踪其状态 {#shut-down-a-worker}

如果 Worker 收到中列出的任何 Signal，则 Worker 将关闭
[关闭Signals](https://typescript.temporal.io/api/interfaces/worker.RuntimeOptions#shutdownsignals): `'SIGINT'`,
`'SIGTERM'`、`'SIGQUIT'` 和 `'SIGUSR2'`。

在开发过程中，我们用 `Ctrl+C` (`SIGINT`) 关闭 Worker 或
[节点监控器](https://github.com/temporalio/samples-typescript/blob/c37bae3ea235d1b6956fcbe805478aa46af973ce/hello-world/package.json#L10)
（`SIGUSR2`）。在生产中，您通常希望通过设置给 Worker 时间来完成任何正在进行的活动
[关机宽限时间](https://typescript.temporal.io/api/interfaces/worker.WorkerOptions#shutdowngracetime)。

一旦 Worker 收到关闭 Signal 或请求，Worker 就会停止轮询新任务并允许运行中的任务
达到 `shutdownGraceTime` 之前要完成的任务。此时仍在运行的任何 Activity 将停止
正在运行，并且当 Activity 超时发生时，将由 Temporal Server 重新 Scheduled。

如果必须保证Worker最终关闭，可以设置
[关闭强制时间](https://typescript.temporal.io/api/interfaces/worker.WorkerOptions#shutdownforcetime)。

您可能想要以编程方式关闭 Worker（使用
[Worker.shutdown()](https://typescript.temporal.io/api/classes/worker.Worker#shutdown)）在集成测试中或当
自动化 Worker 车队。

<a id="worker-states"></a>
### Worker 状态 {#worker-states}

您可以随时使用 Query Worker 状态
[Worker.getState()](https://typescript.temporal.io/api/classes/worker.Worker#getstate)。 Worker 始终位于其中之一
七个州：

- `INITIALIZED`：调用后Worker的初始状态
  [Worker.create()](https://typescript.temporal.io/api/classes/worker.Worker#create)并成功连接到
  服务器。
- `RUNNING`：调用了 [Worker.run()](https://typescript.temporal.io/api/classes/worker.Worker#run)，并且调用了 Worker
  轮询 Task Queue。
- `FAILED`：Worker遇到不可恢复的错误； `Worker.run()` 应拒绝并显示错误。
- 最后四种状态与Worker关闭过程相关：
  - `STOPPING`：Worker 收到关闭 Signal 或调用 `Worker.shutdown()` 的消息。 Worker将强制关闭
    `shutdownGraceTime` 到期后下降。
  - `DRAINING`：所有Workflow任务已被耗尽；等待活动和缓存的 Workflow 驱逐。
  - `DRAINED`：所有活动和 Workflow 均已完成；准备关闭。
  - `STOPPED`：关机完成； `Worker.run()` 解决。

如果您需要更多 Visibility 进入内部 Worker 状态，请参阅
API 参考中的 [Worker类](https://typescript.temporal.io/api/classes/worker.Worker)。
