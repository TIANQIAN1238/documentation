TypeScript 开发人员指南的可观察性部分涵盖了查看 [Temporal 应用](/temporal#temporal-application) 当前状态的多种方法，即查看 [Temporal平台](/workflow-execution) 跟踪哪些 [Workflow Executions](/temporal#temporal-platform) 以及任何指定 Workflow Execution 当前或执行点的状态的方法。

本节介绍与查看应用程序状态相关的功能，包括：

- [发出指标](#metrics)
- [设置追踪](#tracing)
- [从 Workflow 登录](#logging)
- [Visibility API](#visibility)

<a id="metrics"></a>
## 发出指标 {#metrics}
每个 Temporal SDK 都能够从客户端或 Worker 进程发出一组可选的指标。
有关能够发出的指标的完整列表，请参阅 [SDK指标参考](/references/sdk-metrics)。

- 有关 Prometheus 和 Grafana 集成的概述，请参阅 [监控](/self-hosted-guide/monitoring) 指南。
- 有关指标列表，请参阅 [SDK指标参考](/references/sdk-metrics)。
- 有关使用 TypeScript SDK 公开指标的端到端示例，请参阅 [样本打字稿](https://github.com/temporalio/samples-typescript/tree/main/interceptors-opentelemetry) 存储库。

Workers 可以发出指标和跟踪。有几个[遥测选项](https://typescript.temporal.io/api/interfaces/worker.TelemetryOptions)可以提供给[`Runtime.install`](https://typescript.temporal.io/api/classes/worker.Runtime/#install)。常见的选项有：

- `metrics: { otel: { url } }`：gRPC [打开遥测收集器](https://opentelemetry.io/docs/collector/) 的 URL。
- `metrics: { prometheus: { bindAddress } }`：Worker 主机上的地址，该主机将具有要抓取的 [普罗米修斯](https://prometheus.io/) 指标。

要设置 Workflow 和活动的跟踪，请使用我们的 `opentelemetry-interceptors` 包。
（有关详细信息，请参阅下一节。）

```typescript
telemetryOptions: {
    metrics: {
      prometheus: { bindAddress: '0.0.0.0:9464' },
    },
    logging: { forward: { level: 'DEBUG' } },
  },
```

<a id="tracing"></a>
## 设置跟踪 {#tracing}
跟踪允许您查看 Workflow 的调用图及其活动和任何 Child Workflow。

Temporal Web 的跟踪功能主要跟踪 Temporal 上下文中的 Activity 执行。如果您需要特定于您的用例的自定义跟踪，您应该利用上下文传播来相应地添加跟踪逻辑。

[`interceptors-opentelemetry`](https://github.com/temporalio/samples-typescript/tree/main/interceptors-opentelemetry) 示例展示了如何使用 SDK 的内置 OpenTelemetry 跟踪来跟踪从启动 Workflow 到 Workflow Execution 到从 Workflow 运行 Activity 的所有内容。

内置跟踪使用 Protobuf 消息标头（如启动 Workflow 时的 [这个](https://github.com/temporalio/api/blob/b2b8ae6592a8730dd5be6d90569d1aea84e1712f/temporal/api/workflowservice/v1/request_response.proto#L161)）将跟踪信息从客户端传播到 Workflow，并从 Workflow 传播到其后继者（当继续为新时）、子项和活动。
所有这些执行都与单个跟踪标识符链接，并具有正确的 `parent -> child` 跨度关系。

只要使用兼容的 [上下文传播器](https://opentelemetry.io/docs/concepts/context-propagation/)，不同 Temporal SDK 之间的跟踪就可以兼容。

**上下文传播**

TypeScript SDK 使用全局 OpenTelemetry 传播器。

要将默认传播器（[跟踪上下文](https://github.com/open-telemetry/opentelemetry-js/blob/main/packages/opentelemetry-core/README.md#w3ctracecontextpropagator-propagator) 和 [行李](https://github.com/open-telemetry/opentelemetry-js/blob/main/packages/opentelemetry-core/README.md#baggage-propagator) 传播器）扩展为也包含 [耶格传播者](https://www.npmjs.com/package/@opentelemetry/propagator-jaeger)，请执行以下步骤：

- `npm i @opentelemetry/propagator-jaeger`

- 在 Workflow 代码的顶层，添加以下行：

  ```js
  import { propagation } from '@opentelemetry/api';
  import {
    CompositePropagator,
    W3CBaggagePropagator,
    W3CTraceContextPropagator,
  } from '@opentelemetry/core';
  import { JaegerPropagator } from '@opentelemetry/propagator-jaeger';

  propagation.setGlobalPropagator(
    new CompositePropagator({
      propagators: [
        new W3CTraceContextPropagator(),
        new W3CBaggagePropagator(),
        new JaegerPropagator(),
      ],
    }),
  );
  ```

同样，您可以按照 `README.md` 文件的 [初始化SDK](https://github.com/open-telemetry/opentelemetry-js/tree/main/experimental/packages/opentelemetry-sdk-node#initialize-the-sdk) 部分中的说明自定义 OpenTelemetry `NodeSDK` 传播器。

<a id="logging"></a>
## 从 Workflow {#logging} 记录
日志记录使您能够在代码执行期间记录关键信息。
记录器创建审计跟踪并捕获有关 Workflow 操作的信息。
适当的日志记录级别取决于您的具体需求。
在开发或故障排除期间，您可能会使用调试甚至跟踪。
在生产中，您可以使用 info 或 warn 来避免日志量过多。

记录器支持以下日志记录级别：

| 等级   | 使用                                                                                                       |
| ------- | --------------------------------------------------------------------------------------------------------- |
| `TRACE` | 最详细的日志记录级别，用于非常细粒度的信息。                               |
| `DEBUG` | 详细信息，通常可用于调试目的。                                            |
| `INFO`  | 有关应用程序操作的一般信息。                                                    |
| `WARN`  | 指示不会阻止应用程序运行的潜在有害情况或小问题。 |
| `ERROR` | 指示可能仍允许应用程序继续运行的错误情况。                    |

Temporal SDK 核心通常使用 `WARN` 作为其默认日志记录级别。

<a id="logging-from-activities"></a>
### 从活动记录 {#logging-from-activities}

活动在标准 Node.js 环境中运行，因此可以直接使用任何 Node.js 记录器。

然而，Temporal SDK 提供了一个方便的 Activity 上下文记录器，它将日志消息传送到 [运行时的记录器](/develop/typescript/platform/observability#customizing-the-default-logger)。当前 Activity 上下文中的属性将作为元数据自动包含在使用 Activity 上下文记录器发出的每个日志条目中，并且会自动记录 Activity 生命周期的一些关键事件（对于大多数消息，在 DEBUG 级别；对于失败为 WARN）。

<details>
<summary>
使用 Activity 上下文记录器
</summary>

```ts
import { log } from '@temporalio/activity';

export async function greet(name: string): Promise<string> {
  log.info('Log from activity', { name });
  return `Hello, ${name}!`;
}
```

</details>

<a id="logging-from-workflows"></a>
### 从 Workflows 记录 {#logging-from-workflows}

Workflows 可能不使用常规 Node.js 记录器，因为：

1. Workflow 在沙盒环境中运行，无法执行任何 I/O。
1. Workflow 代码可能随时重播，这将导致日志消息重复。

然而，Temporal SDK 提供了一个 Workflow 上下文记录器，它将日志消息传送到 [运行时的记录器](/develop/typescript/platform/observability#customizing-the-default-logger)。当前 Workflow 上下文中的属性将作为元数据自动包含在使用 Workflow 上下文记录器发出的每个日志条目中，并且会自动记录 Workflow 生命周期的一些关键事件（对于大多数消息，在 DEBUG 级别；对于失败为 WARN）。

<details>
<summary>
使用 Workflow 上下文记录器
</summary>

```ts
import { log } from '@temporalio/workflow';

export async function myWorkflow(name: string): Promise<string> {
  log.info('Log from workflow', { name });
  return `Hello, ${name}!`;
}
```

</details>

Workflow 上下文记录器尝试避免在 Workflow 重放上重新发送日志消息。

<a id="what-is-the-runtimes-logger"></a>
### 什么是运行时的记录器 {#what-is-the-runtimes-logger}

Temporal Worker 可以通过多种方式发出日志，包括：

- 使用 [Workflow 上下文记录器](#logging) 发出的消息；
- 使用 [Activity 上下文记录器](#logging-from-activities) 发出的消息；
- TypeScript SDK Worker 本身发出的消息；
- 由底层 Temporal Core SDK（本机代码）发出的消息。

所有这些消息都在内部路由到单个记录器对象，称为运行时记录器。
默认情况下，运行时的记录器只是将消息写入控制台（即进程的 `STDOUT`）。

<a id="how-to-customize-the-runtimes-logger"></a>
#### 如何自定义运行时的记录器 {#how-to-customize-the-runtimes-logger}

当 SDK `Runtime` 实例化时，可以注册自定义运行时记录器。每个进程仅执行一次此操作。

要注册自定义运行时记录器，您必须使用 [`Runtime.install()`](https://typescript.temporal.io/api/classes/worker.Runtime/#install) 函数显式实例化运行时。
例如：

```typescript
import {
  DefaultLogger,
  makeTelemetryFilterString,
  Runtime,
} from '@temporalio/worker';

// This is your custom Logger.
const logger = new DefaultLogger('WARN', ({ level, message }) => {
  console.log(`Custom logger: ${level} — ${message}`);
});

Runtime.install({
  logger,
  // The following block is optional, but generally desired.
  // It allows capturing log messages emitted by the underlying Temporal Core SDK (native code).
  // The Telemetry Filter String determine the desired verboseness of messages emitted by the
  // Temporal Core SDK itself ("core"), and by other native libraries ("other").
  telemetryOptions: {
    logging: {
      filter: makeTelemetryFilterString({ core: 'INFO', other: 'INFO' }),
      forward: {},
    },
  },
});
```

一个常见的用例是将日志消息写入文件以供收集器服务（例如 [数据狗代理](https://docs.datadoghq.com/logs/log_collection/nodejs/?tab=winston30)）拾取。
例如：

```typescript
import {
  DefaultLogger,
  makeTelemetryFilterString,
  Runtime,
} from '@temporalio/worker';
import winston from 'winston';

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.json(),
  transports: [new transports.File({ filename: '/path/to/worker.log' })],
});

Runtime.install({
  logger,
  // The following block is optional, but generally desired.
  // It allows capturing log messages emitted by the underlying Temporal Core SDK (native code).
  // The Telemetry Filter String determine the desired verboseness of messages emitted by the
  // Temporal Core SDK itself ("core"), and by other native libraries ("other").
  telemetryOptions: {
    logging: {
      filter: makeTelemetryFilterString({ core: 'INFO', other: 'INFO' }),
      forward: {},
    },
  },
});
```

<a id="implementing-custom-logging-like-features-based-on-workflow-sinks"></a>
### 基于 Workflow 接收器实现自定义类似日志记录的功能 {#implementing-custom-logging-like-features-based-on-workflow-sinks}

接收器支持将日志、指标和跟踪从 Workflow 隔离区单向导出到 Node.js 环境。

{/*
Temporal 中的 Workflow 在恢复时可以从其历史记录的开头重播。为了使 Temporal 重新创建 Workflow 代码所处的确切状态，代码需要完全确定。为了防止破坏确定性，在 TypeScript SDK 中，Workflow 代码在隔离的执行环境中运行，并且不得使用任何 Node.js API 或直接与外界通信。 */}

接收器被写成带有方法的对象。
与活动类似，它们在 Worker 中声明，然后在 Workflow 代码中代理，这有助于在两者之间共享类型。

<a id="comparing-sinks-and-activities"></a>
#### 比较接收器和活动 {#comparing-sinks-and-activities}

接收器与活动类似，因为它们都在 Worker 上注册并代理到 Workflow 中。
然而，它们在一些重要方面与活动不同：

- 接收器函数不会将任何值返回到 Workflow，并且无法等待。
- 接收器调用不会记录在 Workflow Execution 的 Event History 中（无超时或重试）。
- 接收器函数_始终_在运行其调用的 Workflow Execution 的同一个 Worker 上运行。

<a id="declare-the-sink-interface"></a>
#### 声明接收器接口 {#declare-the-sink-interface}

显式声明接收器的接口是可选的，但对于确保后续步骤中的类型安全非常有用：

<!--SNIPSTART typescript-logger-sink-interface-->
[水槽/src/workflows.ts](https://github.com/temporalio/samples-typescript/blob/main/sinks/src/workflows.ts)
```ts
import { log, proxySinks, Sinks } from '@temporalio/workflow';

export interface AlertSinks extends Sinks {
  alerter: {
    alert(message: string): void;
  };
}

export type MySinks = AlertSinks;
```
<!--SNIPEND-->

<a id="implement-sinks"></a>
#### 实施水槽 {#implement-sinks}

实施接收器是一个两步过程。

实现 Sink 函数并将其注入到 Worker 中

<!--SNIPSTART typescript-logger-sink-Worker-->
[水槽/src/Worker.ts](https://github.com/temporalio/samples-typescript/blob/main/sinks/src/worker.ts)
```ts
import { InjectedSinks, Worker } from '@temporalio/worker';
import { MySinks } from './workflows';

async function main() {
  const sinks: InjectedSinks<MySinks> = {
    alerter: {
      alert: {
        fn(workflowInfo, message) {
          console.log('sending SMS alert!', {
            workflowId: workflowInfo.workflowId,
            workflowRunId: workflowInfo.runId,
            message,
          });
        },
        callDuringReplay: false, // The default
      },
    },
  };
  const worker = await Worker.create({
    workflowsPath: require.resolve('./workflows'),
    taskQueue: 'sinks',
    sinks,
  });
  await worker.run();
  console.log('Worker gracefully shutdown');
}

main().catch((err) => {
  console.error(err);
  process.exit(1);
});
```
<!--SNIPEND-->

- 接收器函数实现作为对象传递到 [Worker选项](https://typescript.temporal.io/api/interfaces/worker.WorkerOptions/#sinks)。
- 您可以通过设置 `callDuringReplay` 选项来指定是否希望在 Workflow 重放期间调用注入的函数。

<a id="proxy-and-call-a-sink-function-from-a-workflow"></a>
#### 从 Workflow 代理并调用接收器函数 {#proxy-and-call-a-sink-function-from-a-workflow}

<!--SNIPSTART typescript-logger-sink-Workflow-->
[水槽/src/workflows.ts](https://github.com/temporalio/samples-typescript/blob/main/sinks/src/workflows.ts)
```ts
const { alerter } = proxySinks<MySinks>();

export async function sinkWorkflow(): Promise<string> {
  log.info('Workflow Execution started');
  alerter.alert('alerter: Workflow Execution started');
  return 'Hello, Temporal!';
}
```
<!--SNIPEND-->

[注入Sink函数](https://typescript.temporal.io/api/interfaces/worker.InjectedSinkFunction) 接口的一些重要特性：

- **注入的 WorkflowInfo 参数：** Sink 函数实现的第一个参数是包含有用元数据的 [`WorkflowInfo` 对象](https://typescript.temporal.io/api/interfaces/workflow.WorkflowInfo/)。
- **有限的参数类型：** 剩余的 Sink 函数参数使用 [结构化克隆算法](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API/Structured_clone_algorithm) 在沙箱和 Node.js 环境之间复制。
- **无返回值：** 为了防止破坏确定性，Sink 函数不能将值返回到 Workflow。

**高级：性能考虑和非阻塞接收器**

注入的接收器函数会影响整个 Workflow 任务处理持续时间。

- 如果您有一个长时间运行的接收器功能，例如尝试与外部服务通信的接收器功能，您可能会开始看到 Workflow 任务超时。
- 使用 `callDuringReplay: true` 并重播长 Workflow 历史记录时，效果会成倍增加，因为 Workflow 任务 Timer 在第一个历史记录页面传送到 Worker 时启动。

<a id="logging-in-workers-and-clients"></a>
#### 登录 Worker 和客户端 {#logging-in-workers-and-clients}

Worker 附带一个默认记录器，默认使用 `console.error` 记录级别为 `INFO` 及更高级别的任何消息到 `STDERR`。
以下 [日志级别](https://typescript.temporal.io/api/namespaces/worker#loglevel) 按严重性升序列出。

<a id="customizing-the-default-logger"></a>
#### 自定义默认记录器 {#customizing-the-default-logger}

Temporal 使用实现基本接口的 [`DefaultLogger`](https://typescript.temporal.io/api/classes/worker.DefaultLogger/)：

```ts
import { DefaultLogger, Runtime } from '@temporalio/worker';

const logger = new DefaultLogger('WARN', ({ level, message }) => {
  console.log(`Custom logger: ${level} — ${message}`);
});
Runtime.install({ logger });
```

前面的代码示例将默认记录器设置为仅记录级别为 `WARN` 及更高级别的消息。

<a id="accumulate-logs-for-testing-and-reporting"></a>
#### 积累日志以进行测试和报告 {#accumulate-logs-for-testing-and-reporting}

```ts
import { DefaultLogger, LogEntry, LogLevel } from '@temporalio/worker';

const logs: LogEntry[] = [];
const logger = new DefaultLogger(LogLevel.TRACE, (entry) => logs.push(entry));

logger.debug('hey', { a: 1 });
logger.info('ho');
logger.warn('lets', { a: 1 });
logger.error('go');
```

常见的日志记录用例是记录到由 [数据狗代理](https://docs.datadoghq.com/logs/log_collection/nodejs/?tab=winston30) 等收集器拾取的文件。

```ts
import { Runtime } from '@temporalio/worker';
import winston from 'winston';

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.json(),
  transports: [new transports.File({ filename: '/path/to/worker.log' })],
});
Runtime.install({ logger });
```

<a id="visibility"></a>
## Visibility API {#visibility}
Temporal 平台中的术语 Visibility 是指使操作员能够查看当前存在于 Temporal 服务中的 Workflow Execution 的子系统和 API。

<a id="search-attributes"></a>
### 如何使用 Search Attribute {#search-attributes}
检索 Workflow Execution 的典型方法是通过其 Workflow Id。

但是，有时您需要根据另一属性检索一个或多个 Workflow Execution。例如，假设您想要获取某个时间范围内失败的某种类型的所有 Workflow Execution，以便您可以使用相同的参数启动新的 Workflow Execution。

您可以使用 [Search Attributes](/search-attribute) 来完成此操作。

- [默认 Search Attribute](/search-attribute#default-search-attribute) 与 `WorkflowType`、`StartTime` 和 `ExecutionStatus` 一样，会自动添加到 Workflow Execution。
- [定制 Search Attribute](/search-attribute#custom-search-attribute) 可以包含其自己的特定于域的数据（如 `customerId` 或 `numItems`）。

使用自定义 Search Attribute 的步骤是：

- 使用 `Temporal operator search-attribute create` 或云 UI 在 Temporal 服务中创建新的 Search Attribute。
- 设置 Workflow Execution 的 Search Attribute 值：
  - 在客户端上，在启动执行时将其作为选项包含在内。
  - 在 Workflow 中通过调用 `UpsertSearchAttributes`。
- 读取Search Attribute的值：
  - 在客户端上通过调用 `DescribeWorkflow`。
  - 在 Workflow 中查看 `WorkflowInfo`。
- Query Workflow Execution 由 Search Attribute 使用 [列表过滤器](/list-filter) 实现：
  - [使用 Temporal CLI](/cli/workflow#list)。
  - 在代码中调用 `ListWorkflowExecutions`。

以下是 Query Workflow Execution 的操作方法：

使用 [`WorkflowService.listWorkflowExecutions`](https://typescript.temporal.io/api/classes/proto.temporal.api.workflowservice.v1.WorkflowService-1#listworkflowexecutions)：

```typescript
import { Connection } from '@temporalio/client';

const connection = await Connection.connect();
const response = await connection.workflowService.listWorkflowExecutions({
  query: `ExecutionStatus = "Running"`,
});
```

其中 `Query` 是 [列表过滤器](/list-filter)。

<a id="custom-search-attributes"></a>
### 如何设置自定义 Search Attribute {#custom-search-attributes}
在 Temporal 服务（使用 `Temporal operator search-attribute create` 或云 UI）中创建自定义 Search Attribute 后，您可以在启动 Workflow 时设置自定义 Search Attribute 的值。

使用 [`WorkflowOptions.searchAttributes`](https://typescript.temporal.io/api/interfaces/client.WorkflowOptions#searchattributes)。

<!--SNIPSTART typescript-search-attributes-client-->
[搜索属性/src/client.ts](https://github.com/temporalio/samples-typescript/blob/main/search-attributes/src/client.ts)
```ts
const handle = await client.workflow.start(example, {
  taskQueue: 'search-attributes',
  workflowId: 'search-attributes-example-0',
  searchAttributes: {
    CustomIntField: [2],
    CustomKeywordListField: ['keywordA', 'keywordB'],
    CustomBoolField: [true],
    CustomDatetimeField: [new Date()],
    CustomTextField: [
      'String field is for text. When queried, it will be tokenized for partial match. StringTypeField cannot be used in Order By',
    ],
  },
});

const { searchAttributes } = await handle.describe();
```
<!--SNIPEND-->

`searchAttributes` 的类型为 `Record<string, string[] | 数字[] | 布尔值[] | 日期[]>`。

<a id="upsert-search-attributes"></a>
### 如何更新插入 Search Attributes {#upsert-search-attributes}
您可以更新插入 Search Attribute 以在 Workflow 代码中添加 Update Search Attribute。

在 Workflow 内部，我们可以读取 [`WorkflowInfo.searchAttributes`](https://typescript.temporal.io/api/interfaces/workflow.WorkflowInfo#searchattributes) 并调用 [`upsertSearchAttributes`](https://typescript.temporal.io/api/namespaces/workflow#upsertsearchattributes)：

<!--SNIPSTART typescript-search-attributes-Workflow -->
[搜索属性/src/workflows.ts](https://github.com/temporalio/samples-typescript/blob/main/search-attributes/src/workflows.ts)
```ts
export async function example(): Promise<SearchAttributes> {
  const customInt = (workflowInfo().searchAttributes.CustomIntField?.[0] as number) || 0;
  upsertSearchAttributes({
    // overwrite the existing CustomIntField: [2]
    CustomIntField: [customInt + 1],

    // delete the existing CustomBoolField: [true]
    CustomBoolField: [],

    // add a new value
    CustomDoubleField: [3.14],
  });
  return workflowInfo().searchAttributes;
}
```
<!--SNIPEND-->

<a id="remove-search-attribute"></a>
### 如何从 Workflow 中删除 Search Attribute {#remove-search-attribute}
要删除之前设置的 Search Attribute，请将其设置为空数组：`[]`。

```typescript
import { upsertSearchAttributes } from '@temporalio/workflow';

async function yourWorkflow() {
  upsertSearchAttributes({ CustomIntField: [1, 2, 3] });

  // ... later, to remove:
  upsertSearchAttributes({ CustomIntField: [] });
}
```
