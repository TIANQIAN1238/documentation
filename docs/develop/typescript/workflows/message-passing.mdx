Workflow 可以像有状态 Web 服务一样接收消息：查询、Signal 和 Update。
Workflow 实现通过可以对传入消息和返回值做出反应的处理程序方法定义这些端点。
Temporal CLIents 使用消息来读取 Workflow 状态并控制其执行。
有关此主题的一般概述，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing)。
本页介绍 Temporal TypeScript SDK 的这些功能。

<a id="writing-message-handlers"></a>
## 编写消息处理程序 {#writing-message-handlers}
:::info
下面的代码是工作 [消息传递样本](https://github.com/temporalio/samples-typescript/tree/main/message-passing/introduction) 的一部分。
:::

编写消息处理程序时请遵循以下准则：

- 使用 [`defineQuery`](https://typescript.temporal.io/api/namespaces/workflow#definequery)、[`defineSignal`](https://typescript.temporal.io/api/namespaces/workflow#definesignal) 或 [`defineUpdate`](https://typescript.temporal.io/api/namespaces/workflow#defineupdate) 将消息类型定义为全局变量。
  您的客户端代码将使用它向 Workflow 发送消息。
- 消息处理程序是通过在 Workflow 函数中调用 [`Workflow.setHandler`](https://typescript.temporal.io/api/namespaces/workflow#sethandler) 来定义的。
- 处理程序和主 Workflow 函数的参数和返回值必须是 [可序列化](/dataconversion)。
- 优先使用单个对象而不是多个输入参数。
  单个对象允许您添加字段而不更改签名。

<a id="queries"></a>
### Query 处理程序 {#queries}
[Query](/sending-messages#sending-queries) 是从 Workflow Execution 检索状态的同步操作：

```typescript
export enum Language {
  ARABIC = 'ARABIC',
  CHINESE = 'CHINESE',
  ENGLISH = 'ENGLISH',
  FRENCH = 'FRENCH',
  HINDI = 'HINDI',
  PORTUGUESE = 'PORTUGUESE',
  SPANISH = 'SPANISH',
}

interface GetLanguagesInput {
  includeUnsupported: boolean;
}

// 👉 Use the object returned by defineQuery to set the query handler in
// Workflow code, and when sending the Query in Client code.
export const getLanguages = wf.defineQuery<Language[], [GetLanguagesInput]>('getLanguages');

export async function greetingWorkflow(): Promise<string> {
  const greetings: Partial<Record<Language, string>> = {
    [Language.CHINESE]: '你好，世界',
    [Language.ENGLISH]: 'Hello, world',
  };

  wf.setHandler(getLanguages, (input: GetLanguagesInput): Language[] => {
    // 👉 A Query handler returns a value: it must not mutate the Workflow state
    // and can't perform async operations.
    if (input.includeUnsupported) {
      return Object.values(Language);
    } else {
      return Object.keys(greetings) as Language[];
    }
  });

  ...
}
```

- Query 处理程序不能是 `async`。
  您无法执行异步操作，例如在 Query 处理程序中执行 Activity。

- `setHandler` 可以采用 `QueryHandlerOptions`（例如 `description`），如 [`Workflow.setHandler`](https://typescript.temporal.io/api/namespaces/workflow#sethandler) 的 API 参考文档中所述。

<a id="signals"></a>
### Signal 处理程序 {#signals}
[Signal](/sending-messages#sending-signals) 是发送到正在运行的 Workflow Execution 的异步消息，用于更改其状态并控制其流程：

```typescript
// 👉 Use the object returned by defineSignal to set the Signal handler in
// Workflow code, and to send the Signal from Client code.
export const approve = wf.defineSignal<[ApproveInput]>('approve');

export async function greetingWorkflow(): Promise<string> {
  let approvedForRelease = false;
  let approverName: string | undefined;

  wf.setHandler(approve, (input) => {
    // 👉 A Signal handler mutates the Workflow state but cannot return a value.
    approvedForRelease = true;
    approverName = input.name;
  });

  ...
}
...
```

- 处理程序无法返回值。
  服务器立即发送响应，无需等待 Workflow 处理 Signal。

- Signal 和 Update 处理程序可以是 `async`。
  这允许您使用活动、Child Workflow、持久 [`Workflow.sleep`](https://typescript.temporal.io/api/namespaces/workflow#sleep)、Timer、[`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition) 条件等。
  有关安全使用异步 Signal 和 Update 处理程序的指南，请参阅 [异步处理程序](#async-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing)。

- 如果您的 Workflow 需要在处理 Signal 或 Update 之前执行一些异步初始化，请在处理程序内使用 [`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition) 等待初始化完成。

- `setHandler` 可以采用 `SignalHandlerOptions`（例如 `description` 和 `unfinishedPolicy`），如 [`Workflow.setHandler`](https://typescript.temporal.io/api/namespaces/workflow#sethandler) 的 API 参考文档中所述。

<a id="updates"></a>
### Update 处理程序和验证器 {#updates}
[Update](/sending-messages#sending-updates) 是发送到正在运行的 Workflow Execution 的可跟踪同步请求。
它可以更改 Workflow 状态、控制其流程并返回结果。
发送方必须等待，直到 Worker 接受或拒绝 Update。
如果出现问题，发送者可能会进一步等待接收返回值或异常：

```typescript
// 👉 Use the object returned by defineUpdate to set the Update handler in
// Workflow code, and to send Updates from Client code.
export const setLanguage = wf.defineUpdate<Language, [Language]>('setLanguage');

export async function greetingWorkflow(): Promise<string> {
  const greetings: Partial<Record<Language, string>> = {
    [Language.CHINESE]: '你好，世界',
    [Language.ENGLISH]: 'Hello, world',
  };

  let language = Language.ENGLISH;

wf.setHandler(
    setLanguage,
    (newLanguage: Language) => {
      // 👉 An Update handler can mutate the Workflow state and return a value.
      const previousLanguage = language;
      language = newLanguage;
      return previousLanguage;
    },
    {
      validator: (newLanguage: Language) => {
        // 👉 Update validators are optional
        if (!(newLanguage in greetings)) {
          throw new Error(`${newLanguage} is not supported`);
        }
      },
    }
  );

  ...
}
```

- `setHandler` 可以采用 `UpdateHandlerOptions`（例如 `validator`、`description` 和 `unfinishedPolicy`），如 [`Workflow.setHandler`](https://typescript.temporal.io/api/namespaces/workflow#sethandler) 的 API 参考文档中所述。

- 关于验证器：
  - 在将 Update 写入历史记录之前，使用验证器拒绝该 Update。
    验证器始终是可选的。
    如果您不需要拒绝 Update，则不需要验证器。
  - 要设置验证器，请在调用 [`Workflow.setHandler`](https://typescript.temporal.io/api/namespaces/workflow#sethandler) 时在 `UpdateHandlerOptions` 中传递验证器函数。
    验证器必须是非异步函数，它接受与处理程序相同的参数类型并返回 `void`。

- 使用验证器接受和拒绝 Update：
  - 要拒绝 Update，请在验证器中抛出任何类型的错误。
  - 如果没有验证器，则始终接受 Update。
- 验证器和 Event History：
  - `WorkflowExecutionUpdateAccepted` 事件被写入历史记录，无论接受是自动的还是由于验证器函数未抛出错误。
  - 当验证器抛出错误时，Update 会被拒绝，并且 `WorkflowExecutionUpdateAccepted` 不会被添加到 Event History 中。
    调用者收到“Update 失败”错误。

- 使用 [`Workflow.currentUpdateInfo`](https://typescript.temporal.io/api/namespaces/workflow#current_update_info) 获取有关当前 Update 的信息。
  这包括 Update ID，它在使用 Continue-As-New 时可用于重复数据删除：请参阅 [确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing)。
- Update 和 Signal 处理程序可以是 `async`，让它们使用活动、Child Workflow、持久 [`Workflow.sleep`](https://typescript.temporal.io/api/namespaces/workflow#sleep) Timer、[`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition) 条件等。
  请参阅 [异步处理程序](#async-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing) 了解安全使用指南。
- 如果您的 Workflow 需要在处理 Update 或 Signal 之前执行一些异步初始化，请在处理程序内使用 [`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition) 等待初始化完成。

<a id="send-messages"></a>
## 发送消息 {#send-messages}
要发送查询、Signal 或 Update，您可以调用 [Workflow手柄](https://typescript.temporal.io/api/namespaces/client#workflowhandle) 对象上的方法：

- 使用 [`client.Workflow.start`](https://typescript.temporal.io/api/classes/client.WorkflowClient#start) 并归还其手柄。

- 使用 [`client.Workflow.getHandle`](https://typescript.temporal.io/api/classes/client.WorkflowClient#gethandle) 通过 Workflow Id 检索 Workflow 句柄。

例如：

```typescript
const handle = await client.workflow.start(greetingWorkflow, {
  taskQueue: 'my-task-queue',
  args: [myArg],
  workflowId: 'my-workflow-id',
});
```

要检查发送消息时所需的参数类型以及查询和 Update 的返回类型，请参阅 Workflow Definition 中相应的处理程序方法。

:::warning Using Continue-as-New and Updates

- Temporal_不_支持 Update 处理程序中的 Continue-As-New 功能。
- 使用 Continue-As-New_之前_完成所有处理程序。
- 从您的主要 Workflow Definition 方法中使用 Continue-As-New，就像完成或失败 Workflow Execution 一样。

:::

<a id="send-query"></a>
### 发送 Query {#send-query}
使用 [`WorkflowHandle.Query`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#query) 将 Query 发送到 Workflow Execution：

```typescript
const supportedLanguages = await handle.query(getLanguages, {
  includeUnsupported: false,
});
```

- 发送 Query 不会将事件添加到 Workflow 的 Event History。

- 您可以在 Namespace 的 Workflow 保留期内向已关闭的 Workflow Execution 发送查询。
  这包括已完成、失败或超时的 Workflow。
  Querying 终止的 Workflow 不安全，因此不受支持。

- Worker 必须在线并轮询 Task Queue 才能处理 Query。

<a id="send-signal"></a>
### 发送 Signal {#send-signal}
您可以从 Temporal CLIent 或另一个 Workflow Execution 向 Workflow Execution 发送 Signal。
但是，您只能将 Signal 发送到尚未关闭的 Workflow Execution。

<a id="send-signal-from-client"></a>
#### 从客户端发送 Signal {#send-signal-from-client}
使用 [Workflow手柄.Signal](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle#signal) 发送 Signal：

```typescript
await handle.signal(greetingWorkflow.approve, { name: 'me' });
```

- 当服务器接受 Signal 时调用返回；它并不等待 Signal 被传送到 Workflow Execution。

- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在 Workflow 的 Event History 中。

<a id="send-signal-from-workflow"></a>
### 从 Workflow 发送 Signal {#send-signal-from-workflow}
Workflow 可以将 Signal 发送到另一个 Workflow，在这种情况下，它称为_外部 Signal_。
使用 [`getExternalWorkflowHandle`](https://typescript.temporal.io/api/namespaces/workflow#getExternalWorkflowHandle)：

```typescript
import { getExternalWorkflowHandle } from '@temporalio/workflow';
import { joinSignal } from './other-workflow';

export async function yourWorkflowThatSignals() {
  const handle = getExternalWorkflowHandle('workflow-id-123');
  await handle.signal(joinSignal, { userId: 'user-1', groupId: 'group-1' });
}
```

当发送外部 Signal 时：

- [Signal外部Workflow执行已启动](/references/events#signalexternalworkflowexecutioninitiated) 事件出现在发件人的 Event History 中。
- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在收件人的 Event History 中。

`getExternalWorkflowHandle` 方法有助于确保 Workflow 保持确定性。
回想一下，确定性 Workflow 的一方面意味着不直接从 Workflow 进行网络调用。
这意味着开发人员无法直接在 Workflow 代码中使用 Temporal CLIent 来发送 Signal 或启动其他 Workflow。
相反，为了在 Workflow 之间进行通信，我们使用 `getExternalWorkflowHandle` 来确保 Workflow 保持确定性，并且将这些交互记录为 Workflow 的 Event History 中的事件。

<a id="signal-with-start"></a>
### Signal-带启动 {#signal-with-start}
Signal-With-Start 允许客户端将 Signal 发送到 Workflow Execution，如果尚未运行则启动执行。
使用 [`Client.Workflow.SignalWithStart`](https://typescript.temporal.io/api/classes/client.WorkflowClient#signalwithstart)：

```typescript
import { Client } from '@temporalio/client';
import { joinSignal, yourWorkflow } from './workflows';

const client = new Client();

await client.workflow.signalWithStart(yourWorkflow, {
  workflowId: 'workflow-id-123',
  taskQueue: 'my-taskqueue',
  args: [{ foo: 1 }],
  signal: joinSignal,
  signalArgs: [{ userId: 'user-1', groupId: 'group-1' }],
});
```

Signal-With-Start 仅限客户使用。
不能从 Workflow 调用它。

<a id="send-update-from-client"></a>
### 发送 Update {#send-update-from-client}
Update 是同步、阻塞调用，可以更改 Workflow 状态、控制其流程并返回结果。

发送 Update 的客户端必须等待，直到服务器将 Update 传递给 Worker。
Worker 必须可用且响应迅速。
如果服务器收到请求后需要立即响应，请改用 Signal。
另请注意，您无法将 Update 发送到其他 Workflow Execution 或执行与 Signal-With-Start 等效的 Update。

- 当Worker确认Update通过验证时，`WorkflowExecutionUpdateAccepted`被添加到Event History。
- 当Worker确认Update已完成时，`WorkflowExecutionUpdateCompleted`被添加到Event History。

要将 Update 发送到 Workflow Execution，您可以：

- 呼叫 [`WorkflowHandle.executeUpdate`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#executeUpdate) 并等待 Update 完成。
  此代码获取 Update 结果：

  ```typescript
  let previousLanguage = await handle.executeUpdate(setLanguage, {
    args: [Language.CHINESE],
  });
  ```

- 一旦 Update 被接受或拒绝，请发送 [`WorkflowHandle.startUpdate`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#startUpdate) 以接收 [`WorkflowUpdateHandle`](https://typescript.temporal.io/api/interfaces/client.WorkflowUpdateHandle)。

- 稍后使用此 `UpdateHandle` 获取结果。
  - `async` Update 处理程序通常执行长时间运行的异步操作，例如调用 Activity。
  - `startUpdate` 仅等待直到 Worker 接受或拒绝 Update，而不等待所有异步操作完成。

例如：

  ```typescript
  const updateHandle = await handle.startUpdate(setLanguage, {
    args: [Language.ENGLISH],
    waitForStage: WorkflowUpdateStage.ACCEPTED,
  });
  previousLanguage = await updateHandle.result();
  ```

有关更多详细信息，请参阅“异步处理程序”部分。

要获取 Update 句柄，您可以：

- 使用 [`WorkflowHandle.startUpdate`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#startUpdate) 启动 Update 并返回句柄，如前面的示例所示。
- 使用 [`getUpdateHandle`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#getupdatehandle) 使用 Update ID 获取正在进行的 Update 的句柄。

<a id="update-with-start"></a>
#### Update-带启动 {#update-with-start}
:::tip

对于开源服务器用户，建议使用Temporal Server版本[Temporal Server 版本 1.28](https://github.com/temporalio/temporal/releases/tag/v1.28.0)。

:::

[Update-带启动](/sending-messages#update-with-start) 让您
[发送 Update](/develop/typescript/workflows/message-passing#send-update-from-client) 检查是否存在具有该 ID 的已运行的 Workflow：

- 如果 Workflow 存在，则处理 Update。
- 如果 Workflow 不存在，则使用给定 ID 启动新的 Workflow Execution，并在主 Workflow 方法开始执行之前处理 Update。

使用[`executeUpdateWithStart`](https://typescript.temporal.io/api/classes/client.WorkflowClient#executeUpdateWithStart)启动一个Update，一次性等待结果。

或者，使用 [`startUpdateWithStart`](https://typescript.temporal.io/api/classes/client.WorkflowClient#startUpdateWithStart) 启动 Update 并接收 [`WorkflowUpdateHandle`](https://typescript.temporal.io/api/interfaces/client.WorkflowUpdateHandle)，然后使用 `await UpdateHandle.result()` 从 Update 检索结果。

一旦到达所请求的 Update 等待阶段或请求超时，这些调用就会返回。

您需要提供 [`WithStartWorkflowOperation`](https://typescript.temporal.io/api/classes/client.WithStartWorkflowOperation) 来定义必要时启动的 Workflow 及其参数。
创建 `WithStartWorkflowOperation` 时必须指定 [WorkflowId冲突政策](/workflow-execution/workflowid-runid#workflow-id-conflict-policy)。
请注意，`WithStartWorkflowOperation` 只能使用一次。

以下是取自 [提前返回](https://github.com/temporalio/samples-typescript/tree/main/early-return) 样本的示例：

```typescript
const startWorkflowOperation = new WithStartWorkflowOperation.create(
  transactionWorkflow,
  {
    workflowId,
    args: [transactionID],
    taskQueue: 'early-return',
    workflowIdConflictPolicy: 'FAIL',
  },
);

const earlyConfirmation = await client.workflow.executeUpdateWithStart(
  getTransactionConfirmation,
  {
    startWorkflowOperation,
  },
);

const wfHandle = await startWorkflowOperation.workflowHandle();
const finalReport = await wfHandle.result();
```

:::info SEND MESSAGES WITHOUT TYPE SAFETY

在实际开发中，有时您可能无法导入 `defineQuery`、`defineSignal` 或 `defineUpdate` 定义的消息类型对象。
当您无权访问 Workflow Definition 或者它不是用 Typescript 编写时，您仍然可以使用非类型安全的 API 和动态方法调用。
将消息类型名称而不是消息类型对象传递给：

- [`client.Workflow.start`](https://typescript.temporal.io/api/classes/client.WorkflowClient#start)
- [`WorkflowHandle.Query`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#query)
- [Workflow手柄.Signal](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle#signal)
- [`WorkflowHandle.executeUpdate`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#executeUpdate)
- [`WorkflowHandle.startUpdate`](https://typescript.temporal.io/api/interfaces/client.WorkflowHandle/#startUpdate)

将 Workflow ID 传递给这些 API 以获取 Workflow 句柄：

- [`client.Workflow.getHandle`](https://typescript.temporal.io/api/classes/client.WorkflowClient#gethandle)
- [`getExternalWorkflowHandle`](https://typescript.temporal.io/api/namespaces/workflow#getExternalWorkflowHandle)。

:::

<a id="async-handlers"></a>
### 使用异步处理程序 {#async-handlers}
Signal 和 Update 处理程序可以是 `async` 函数。
使用 `async` 允许您将 `await` 与活动、Child Workflow、耐用 [`Workflow.sleep`](https://typescript.temporal.io/api/namespaces/workflow#sleep) Timer、[`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition) 条件等一起使用。
这扩展了处理程序可以完成的操作的可能性，但这也意味着处理程序执行和主 Workflow 方法都是同时运行的，并在 `await` 调用时在它们之间进行切换。
为了安全使用 `async` 处理程序，了解可能出现问题的情况至关重要。
有关异步 Signal 和 Update 处理程序、[安全消息处理程序](/encyclopedia/workflow-message-passing) 示例以及下面的 [控制处理程序并发](https://github.com/temporalio/samples-typescript/blob/main/message-passing/safe-message-handlers/README.md) 和 [等待消息处理程序完成](#control-handler-concurrency) 部分的安全使用指南，请参阅 [Workflow消息传递](#wait-for-message-handlers)。

以下代码执行 Activity 对远程服务进行网络调用。
它修改了本页前面的 Update 处理程序，将其转换为 `async` 函数：

```typescript
// 👉 Use the objects returned by defineUpdate to set the Update handler in
// Workflow code, and to send Updates from Client code.
export const setLanguageUsingActivity = wf.defineUpdate<Language, [Language]>('setLanguageUsingActivity');

export async function greetingWorkflow(): Promise<string> {
  const greetings: Partial<Record<Language, string>> = {
    [Language.CHINESE]: '你好，世界',
    [Language.ENGLISH]: 'Hello, world',
  };

  let language = Language.ENGLISH;

  const lock = new Mutex();
  wf.setHandler(setLanguageUsingActivity, async (newLanguage) => {
    // 👉 An Update handler can mutate the Workflow state and return a value.
    // 👉 Since this update handler is async, it can execute an activity.
    if (!(newLanguage in greetings)) {
      // 👉 Do the following with the lock held to ensure that multiple calls to set_language are processed in order.
      await lock.runExclusive(async () => {
        if (!(newLanguage in greetings)) {
          const greeting = await callGreetingService(newLanguage);
          if (!greeting) {
            // 👉 An update validator cannot be async, so cannot be used to check that the remote
            // call_greeting_service supports the requested language. Raising ApplicationError
            // will fail the Update, but the WorkflowExecutionUpdateAccepted event will still be
            // added to history.
            throw new wf.ApplicationFailure(`${newLanguage} is not supported by the greeting service`);
          }
          greetings[newLanguage] = greeting;
        }
      });
    }
    const previousLanguage = language;
    language = newLanguage;
    return previousLanguage;
  });
  ...
}
```

更新代码以使用 `async` 后，您的 Update 处理程序可以使用 Schedule 和 Activity 并等待结果。
尽管 `async` Signal 处理程序也可以执行 Activity，但使用 Update 处理程序允许客户端在 Activity 完成后接收结果或错误。
这使您的客户端可以跟踪 Update 的活动、Child Workflow 等执行的异步工作的进度。

<a id="add-wait-conditions-to-block"></a>
### 添加阻塞等待条件 {#add-wait-conditions-to-block}

有时，`async` Signal 或 Update 处理程序需要满足某些条件才能继续。
您可以使用 [`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition) 来阻止代码继续执行，直到条件成立为止。
您可以通过传递返回 `true` 或 `false` 的函数来指定条件。
这是一个重要的功能，可以帮助您控制处理程序逻辑。

以下是 `Workflow.condition` 的三个重要用例：

- 等待 Signal 或 Update 到达
- 在处理程序中等待，直到适合继续。
- 在主 Workflow 中等待，直到所有活动处理程序完成。

<a id="wait-for-a-signal-or-update-to-arrive"></a>
#### 等待 Signal 或 Update 到达 {#wait-for-a-signal-or-update-to-arrive}

通常使用 `Workflow.condition` 来等待客户端发送特定的 Signal 或 Update：

```typescript
export async function greetingWorkflow(): Promise<string> {
  let approvedForRelease = false;
  let approverName: string | undefined;

  wf.setHandler(approve, (input) => {
    approvedForRelease = true;
    approverName = input.name;
  });
  ...

  await wf.condition(() => approvedForRelease);
  ...
}
```

<a id="use-wait-conditions-in-handlers"></a>
#### 在处理程序中使用等待条件 {#use-wait-conditions-in-handlers}

在处理程序中使用 Workflow 等待条件是很常见的。
例如，假设您的 Workflow 有一个可变变量 `readyForUpdateToExecute`，它指示是否应允许您的 Update 处理程序开始执行。
您可以在处理程序中使用 `Workflow.condition` 使处理程序暂停，直到满足条件：

```typescript
let readyForUpdateToExecute = false;

wf.setHandler(myUpdate, async (input: MyUpdateInput): Promise<MyUpdateOutput> => {
  await wf.condition(() => readyForUpdateToExecute);
  ...
});
```

请记住：处理程序可以在主 Workflow 方法启动之前执行。

您还可以在处理程序中的任何其他位置使用等待条件来等待特定条件变为真。
这允许您编写在多个点暂停的处理程序，每次都等待所需的条件变为真。

<a id="wait-for-message-handlers"></a>
#### 确保您的处理程序在 Workflow 完成 {#wait-for-message-handlers} 之前完成
Workflow 等待条件可以确保您的处理程序在 Workflow 完成之前完成。
当您的 Workflow 使用 `async` Signal 或 Update 处理程序时，您的主 Workflow 方法可以在处理程序仍在等待异步任务（例如 Activity 结果）时返回 或 Continue-As-New 。
Workflow 完成可能会在处理程序完成关键工作之前中断处理程序，并在尝试检索 Update 结果时导致客户端错误。
使用[`Workflow.condition`](https://typescript.temporal.io/api/namespaces/workflow#condition)和[`allHandlersFinished`](https://typescript.temporal.io/api/namespaces/workflow#condition#allhandlersfinished)来解决这个问题，让你的Workflow顺利结束：

```typescript
export async function myWorkflow(): Promise<MyWorkflowOutput> {
  await wf.condition(wf.allHandlersFinished);
  return workflowOutput;
}
```

默认情况下，当您允许 Workflow Execution 完成未完成的处理程序执行时，您的 Worker 将记录一条警告。
您可以通过在调用 [`Workflow.setHandler`](https://typescript.temporal.io/api/namespaces/workflow#sethandler) 时在 `SignalHandlerOptions` 或 `UpdateHandlerOptions` 中设置 `unfinishedPolicy`，在每个处理程序的基础上使这些警告静音

请参阅 [在 Workflow 完成之前完成处理程序](/handling-messages#finishing-message-handlers) 了解更多信息。

<a id="control-handler-concurrency"></a>
### 使用锁来防止并发处理程序执行 {#control-handler-concurrency}
并发进程可以以不可预测的方式交互。
当多个处理程序实例同时运行时，编写不正确的 [并发消息传递](/handling-messages#message-handler-concurrency) 代码可能无法正常工作。
这是一个病态病例的例子：

```typescript
export async function myWorkflow(): Promise<MyWorkflowOutput> {
  let x = 0;
  let y = 0;
  wf.setHandler(mySignal, async () => {
    const data = await myActivity();
    x = data.x;

    // 🐛🐛 Bug!! If multiple instances of this handler are executing
    // concurrently, then there may be times when the Workflow has x from one
    // Activity execution and y from another.
    await wf.sleep(500); // or await anything else

    y = data.y;
  });
  ...
}
```

使用锁（也称为互斥体）协调访问可以更正此代码。
锁定可确保在任何给定时间只有一个处理程序实例可以执行特定的代码部分：

```typescript
import { Mutex } from 'async-mutex';  // https://github.com/DirtyHairy/async-mutex

...

export async function myWorkflow(): Promise<MyWorkflowOutput> {
  let x = 0;
  let y = 0;
  const lock = new Mutex();

  wf.setHandler(mySignal, async () => {
    await lock.runExclusive(async () => {
      const data = await myActivity();
      x = data.x;

      // ✅ OK: node's event loop may switch now to a different handler
      // execution, or to the main workflow function, but no other execution of
      // this handler can run until this execution finishes.
      await wf.sleep(500); // or await anything else

      y = data.y;
    });
  });
  return {
    name: 'hello',
  };
}
```

<a id="message-handler-troubleshooting"></a>
## 消息处理程序故障排除 {#message-handler-troubleshooting}
将 Signal、Update 或 Query 发送到 Workflow 时，您的客户端可能会遇到以下错误：

- **客户端无法联系服务器**：
  您将收到 [`client.ServiceError`](https://typescript.temporal.io/api/classes/client.ServiceError)，其 `cause.code` 属性为 [gRPC 状态码](https://grpc.io/docs/guides/status-codes/) 14 `UNAVAILABLE`（重试几次后）。

- **Workflow 不存在**：
  您将收到 [`common.WorkflowNotFoundError`](https://typescript.temporal.io/api/classes/common.WorkflowNotFoundError) 错误。

<a id="signal-problems"></a>
### 发送 Signal 时出现问题 {#signal-problems}
使用 Signal 时，上述两个错误是您的请求导致的唯一错误。

对于查询和 Update，客户端等待来自 Worker 的响应，因此在 Worker 执行处理程序期间可能会出现其他错误。

<a id="update-problems"></a>
### 发送 Update 时出现问题 {#update-problems}
使用 Update 时，您可能会遇到以下问题：

- **没有 Workflow Worker 正在轮询 Task Queue**：
  SDK 客户端将无限期地重试您的请求。

- **Update 失败**：您将收到 [`client.WorkflowUpdateFailedError`](https://typescript.temporal.io/api/classes/client.WorkflowUpdateFailedError) 异常。
  有两种方式可以发生这种情况：

- Update 被 Workflow 中与 Update 处理程序一起定义的 Update 验证器拒绝。

- Update 被接受后失败。

Update 故障类似于 [Workflow 故障](/references/failures#errors-in-workflows)。
  导致 main 方法中的 Workflow 失败的问题也会导致 Update 处理程序中的 Update 失败。
  这些可能包括：

- 失败的Child Workflow
      - 失败的 Activity（如果 Activity 重试已设置为有限次数）
      - Workflow 作者筹集了 `ApplicationFailure`

- **处理程序导致 Workflow 任务失败**：
  [Workflow 任务失败](/references/failures#errors-in-workflows) 导致服务器无限期地重试 Workflow 任务。您的 Update 请求会发生什么情况取决于其阶段：
  - 如果服务器尚未接受请求，您将收到 [`client.ServiceError`](https://typescript.temporal.io/api/classes/client.ServiceError)，其 `cause.code` 属性为 [gRPC 状态码](https://grpc.io/docs/guides/status-codes/) 9 `FAILED_PRECONDITION`（重试几次后）。
  - 如果请求已被接受，则该请求是持久的。
    代码部署后，一旦 Workflow 再次正常运行，请使用 [`WorkflowUpdateHandle`](https://typescript.temporal.io/api/interfaces/client.WorkflowUpdateHandle) 获取 Update 结果。

- **Workflow 已完成，而 Update 处理程序执行正在进行中**：
  您将收到 [`client.ServiceError`](https://typescript.temporal.io/api/classes/client.ServiceError)，其 `cause.code` 属性为 [gRPC 状态码](https://grpc.io/docs/guides/status-codes/) 5 `NOT_FOUND`。
  如果 Workflow 完成而 Update 处理程序执行正在进行中，则会发生这种情况，例如因为

- Workflow 已取消或失败。

- Workflow 正常完成或继续如新，而 Workflow 作者没有 [等待处理程序完成](/handling-messages#finishing-message-handlers)。

<a id="query-problems"></a>
### 发送 Query 时出现问题 {#query-problems}
使用查询时，您可能会遇到以下错误：

- **没有 Workflow Worker 轮询 Task Queue**：
  您将收到 [`client.ServiceError`](https://typescript.temporal.io/api/classes/client.ServiceError)，其 `cause.code` 属性为 [gRPC 状态码](https://grpc.io/docs/guides/status-codes/) 9 `FAILED_PRECONDITION`。

- **Query 失败**：
  如果 Query 期间出现问题，您将收到 [`client.QueryNotRegisteredError`](https://typescript.temporal.io/api/classes/client.QueryNotRegisteredError) 异常。
  Query 处理程序中的任何错误都会触发此错误。
  这与 Signal 和 Update 请求不同，其中错误可能会导致 Workflow 任务失败。

- **处理程序导致 Workflow 任务失败。**
  例如，如果 Query 处理程序阻塞线程太长时间而没有屈服，就会发生这种情况。

<a id="dynamic-handler"></a>
## 静态或动态定义 Signal 和查询 {#dynamic-handler}
- Signal 和查询的处理程序都可以采用参数，这些参数可以在 `setHandler` 逻辑内部使用。
- 只有 Signal 处理程序可以改变状态，并且只有 Query 处理程序可以返回值。

* [静态定义 Signal 和查询](#static-signals-and-queries)
* [动态定义 Signal 和查询](#dynamic-signals-and-queries)

<a id="static-signals-and-queries"></a>
### 静态定义 Signal 和查询 {#static-signals-and-queries}
如果您预先知道 Signal 和查询的名称，我们建议您在 Workflow Definition 之外声明它们。

<!--SNIPSTART typescript-blocked-Workflow-->
[Signals-查询/src/workflows.ts](https://github.com/temporalio/samples-typescript/blob/main/signals-queries/src/workflows.ts)
```ts
import * as wf from '@temporalio/workflow';

export const unblockSignal = wf.defineSignal('unblock');
export const isBlockedQuery = wf.defineQuery<boolean>('isBlocked');

export async function unblockOrCancel(): Promise<void> {
  let isBlocked = true;
  wf.setHandler(unblockSignal, () => void (isBlocked = false));
  wf.setHandler(isBlockedQuery, () => isBlocked);
  wf.log.info('Blocked');
  try {
    await wf.condition(() => !isBlocked);
    wf.log.info('Unblocked');
  } catch (err) {
    if (err instanceof wf.CancelledFailure) {
      wf.log.info('Cancelled');
    }
    throw err;
  }
}
```
<!--SNIPEND-->

此技术有助于提供类型安全，因为您可以导出要由客户端调用的 Signal 或 Query 的类型签名。

<a id="dynamic-signals-and-queries"></a>
### 动态定义 Signal 和查询 {#dynamic-signals-and-queries}
对于更灵活的用例，您可能需要动态 Signal（例如生成的 ID）。
您可以通过两种方式处理：

- 通过将所有 Signal 折叠到一个处理程序并将 ID 移至有效负载，避免使其动态化。
- 通过内联每个处理程序的 Signal 定义，实际上使 Signal 名称动态化。

```ts
import * as wf from '@temporalio/workflow';

// "fat handler" solution
wf.setHandler(`genericSignal`, (payload) => {
  switch (payload.taskId) {
    case taskAId:
      // do task A things
      break;
    case taskBId:
      // do task B things
      break;
    default:
      throw new Error('Unexpected task.');
  }
});

// "inline definition" solution
wf.setHandler(wf.defineSignal(`task-${taskAId}`), (payload) => {
  /* do task A things */
});
wf.setHandler(wf.defineSignal(`task-${taskBId}`), (payload) => {
  /* do task B things */
});

// utility "inline definition" helper
const inlineSignal = (signalName, handler) =>
  wf.setHandler(wf.defineSignal(signalName), handler);
inlineSignal(`task-${taskBId}`, (payload) => {
  /* do task B things */
});
```

<details>
  <summary>
API 设计常见问题解答
  </summary>

**为什么不是“新 Signal”和“新 Query”？**

`defineSignal` 和 `defineQuery` 的语义是有意的。
它们返回 Signal 和 Query **定义**，而不是 Signal 和查询本身的唯一实例
以下是他们的[整个源代码](https://github.com/temporalio/sdk-typescript/blob/fc658d3760e6653aec47732ab17a0062b7dd23fc/packages/workflow/src/workflow.ts#L883-L907)：

```ts
/**
 * Define a signal method for a Workflow.
 */
export function defineSignal<Args extends any[] = []>(
  name: string,
): SignalDefinition<Args> {
  return {
    type: 'signal',
    name,
  };
}

/**
 * Define a query method for a Workflow.
 */
export function defineQuery<Ret, Args extends any[] = []>(
  name: string,
): QueryDefinition<Ret, Args> {
  return {
    type: 'query',
    name,
  };
}
```

Signal 和查询仅在 `setHandler` 中实例化，并且特定于特定的 Workflow Execution。

这些区别可能看起来很小，但它们模拟了 Temporal 在底层的工作方式，因为 Signal 和查询是由“只是字符串”标识的消息，并且没有独立于具有侦听器来处理它们的 Workflow 的含义。
如果您参考客户端 API，这一点会更清楚。

**为什么使用 setHandler 而不是 OTHER_API？**

我们将其命名为 `setHandler` 而不是 `subscribe`，因为 Signal 或 Query 一次只能有一个“处理程序”，而 `subscribe` 可能意味着具有多个消费者的 Observable，并且是一种更高级别的构造。

```ts
wf.setHandler(MySignal, handlerFn1);
wf.setHandler(MySignal, handlerFn2); // replaces handlerFn1
```

如果您熟悉 [接收JS](https://rxjs.dev/)，则可以根据需要将 Signal 和查询包装到 Observable 中，或者您可以根据业务逻辑或 Workflow 状态动态重新分配侦听器。

</details>
