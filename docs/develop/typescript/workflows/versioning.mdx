由于 Temporal 中的 Workflow Execution 可以运行很长时间（有时是几个月甚至几年），因此通常需要对 Workflow Definition 进行更改，即使特定的 Workflow Execution 正在进行中也是如此。

Temporal 平台要求 Workflow 代码为 [确定性的](/workflow-definition#deterministic-constraints)。
如果您对 Workflow 代码进行的更改会导致重播时出现不确定行为，则您需要使用我们的版本控制方法之一来优雅地 Update 您正在运行的 Workflow。
通过版本控制，您可以修改 Workflow Definition，以便新执行使用 Updated 代码，而现有执行继续运行原始版本。
您可以使用两种主要的版本控制方法：

- [Worker 版本控制](/production-deployment/worker-deployments/worker-versioning)。 Worker 版本控制功能允许您标记 Worker 并以编程方式在版本化部署中推出它们，以便旧的 Worker 可以运行旧的代码路径，而新的 Worker 可以运行新的代码路径。
- [通过补丁进行版本控制](#patching)。此方法的工作原理是向与特定修订版相关的代码添加分支。它将代码更改应用于新的 Workflow Execution，同时避免对正在进行的 Workflow Execution 进行破坏性更改。

:::danger
2025 年之前对实验性 Worker 版本控制方法的支持将于 2026 年 3 月从 Temporal Server 中删除。请参阅 [最新的 Worker 版本控制文档](/worker-versioning) 获取指导。如果需要，您仍然可以参考 [Worker 版本控制旧版](/develop/typescript/worker-versioning-legacy) 文档。
:::

## 使用补丁进行版本控制 {#patching}

<a id="adding-a-patch"></a>
### 添加补丁 {#adding-a-patch}

补丁在 Workflow 中为特定更改定义逻辑分支，类似于功能标志。
它将代码更改应用于新的 Workflow Execution，同时避免对正在进行的 Workflow Execution 进行破坏性更改。
当您想要进行可能影响现有 Workflow Execution 的实质性代码更改时，请创建补丁。请注意，无需修补 [固定 Workflow](/worker-versioning)。

假设您有一个运行 `ActivityA` 的初始 Workflow：

```ts
// v1
export async function myWorkflow(): Promise<void> {
  await activityA();
  await sleep('1 days'); // arbitrary long sleep to simulate a long running workflow we need to patch
  await activityThatMustRunAfterA();
}
```

现在，您想要 Update 您的代码来运行 `ActivityB`。这代表您想要的最终状态。

```ts
// vFinal
export async function myWorkflow(): Promise<void> {
  await activityB();
  await sleep('1 days');
}
```

问题是，在确定不再有使用 `v1` 代码创建的正在运行的 Workflow 之前，您无法直接部署此 `vFinal` 修订版，否则可能会导致不确定性错误。
相反，您需要使用 [`patched`](https://typescript.temporal.io/api/namespaces/workflow#patched) 函数来检查应执行哪个版本的代码。

修补过程分为三个步骤：

1. 使用 `patched()` 函数修补任何新的 Updated 代码。将新的修补代码与旧代码一起运行。
2. 删除旧代码并使用 `deprecatePatch()` 将特定补丁标记为已弃用。
3. 一旦不再有任何打开的先前版本代码的 Workflow Execution，请删除 `deprecatePatch()`。
   让我们按顺序浏览一下这个过程。

<a id="patching-in-new-code"></a>
### 修补新代码 {#patching-in-new-code}

使用 `patched` 将标记插入到 Workflow 历史记录中。
在重播期间，如果 Worker 遇到带有该标记的历史记录，则当 Workflow 代码不生成相同的补丁标记（在本例中为 `your-change-id`）时，Workflow 任务将会失败。
这确保您可以安全地将 `v2` 中的代码作为“功能标志”与原始版本 (`v1`) 一起部署。

```ts
// v2
import { patched } from '@temporalio/workflow';
export async function myWorkflow(): Promise<void> {
  if (patched('my-change-id')) {
    await activityB();
    await sleep('1 days');
  } else {
    await activityA();
    await sleep('1 days');
    await activityThatMustRunAfterA();
  }
}
```

### 弃用补丁 {#deprecated-patches}

确保以 `v1` 代码开头的所有 Workflow 均已保留保留后，您可以使用 [弃用该补丁](https://typescript.temporal.io/api/namespaces/workflow#deprecatepatch)。

一旦您的 Workflow 不再运行修补程序前的代码路径，您就可以使用 `deprecatePatch()` 部署代码。
这些 Worker 将运行最新版本的 Workflow 代码，不再需要补丁。
已弃用的补丁充当修补过程的最后阶段和不再有补丁的最终状态之间的桥梁。它们的功能与常规补丁类似，通过在 Workflow 历史记录中添加标记来实现。但是，当 Workflow 代码不生成该标记时，该标记不会导致重放失败。

```ts
// v3
import { deprecatePatch } from '@temporalio/workflow';

export async function myWorkflow(): Promise<void> {
  deprecatePatch('my-change-id');
  await activityB();
  await sleep('1 days');
}
```

### 删除补丁 {#deploy-new-code}

一旦预补丁 Workflow 保留，您就可以安全地部署不再使用 `patched()` 或 `deprecatePatch()` 调用的 Worker：

修补允许您对当前运行的 Workflow 进行更改。
它是一种引入兼容更改而不引入非确定性错误的强大方法。

<a id="workflow-cutovers"></a>
### Workflow 切换 {#workflow-cutovers}

要了解修补为何有用，演示剪切整个 Workflow 会很有帮助。

由于不兼容的更改仅影响相同类型的打开的 Workflow Execution，因此您可以在进行更改时通过创建全新的 Workflow 来避免确定性错误。
为此，您可以复制 Workflow Definition 函数，为其指定一个不同的名称，然后将这两个名称注册到您的 Worker 中。

例如，您可以将 `PizzaWorkflow` 复制为 `PizzaWorkflowV2`：

```typescript
function pizzaWorkflow(order: PizzaOrder): Promise<OrderConfirmation> {
  // this function contains the original code
}

function pizzaWorkflowV2(order: PizzaOrder): Promise<OrderConfirmation> {
  // this function contains the updated code
}
```

然后，您需要 Update、Worker 配置以及任何其他标识符字符串来注册这两种 Workflow 类型：

```typescript
const worker = await Worker.create({
  workflowsPath: require.resolve('./workflows'),
  // other configurations
});
```

此方法的缺点是它需要您复制代码以及用于启动 Workflow 的任何命令到 Update。
随着时间的推移，这可能变得不切实际。
此方法也不提供对任何仍在运行的 Workflow 进行版本控制的方法——它本质上只是一个转换，与修补不同。

<a id="testing-a-workflow-for-replay-safety"></a>
### 测试 Workflow 的重放安全性 {#testing-a-workflow-for-replay-safety}

要确定您的 Workflow 是否需要修补程序，或者您已成功修补它，您应该合并 [重播测试](/develop/typescript/best-practices/testing-suite#replay)。

<a id="worker-versioning"></a>
## Worker 版本控制 {#worker-versioning}

Temporal 的 [Worker 版本控制](/production-deployment/worker-deployments/worker-versioning) 功能允许您标记 Worker 并以编程方式在部署版本中推出它们，以便旧的 Worker 可以运行旧的代码路径，而新的 Worker 可以运行新的代码路径。这样，您就可以将 Workflow 固定到特定版本，从而无需修补。
