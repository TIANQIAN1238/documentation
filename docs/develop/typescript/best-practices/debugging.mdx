Temporal TypeScript SDK 开发者指南的调试部分介绍了调试工具以及如何排查常见问题。

## 如何在开发环境中调试 {#debug-in-a-development-environment}

除了日志记录和调试器等常规开发工具外，还可以使用 [Web UI](/web-ui) 或 [Temporal CLI](/cli) 查看 Workflow 中发生的情况。

## 如何在生产环境中调试 {#debug-in-a-production-environment}

可以使用以下工具调试生产环境中的 Workflow：

- [Web UI](/web-ui)
- [Temporal CLI](/cli)
- [Replay](/develop/typescript/best-practices/testing-suite#replay)
- [Tracing](/develop/typescript/platform/observability#tracing)
- [Logging](/develop/typescript/platform/observability#logging)

可以使用指标和 [Worker 性能指南](/develop/worker-performance)来调试和调优 Worker 性能。有关设置 SDK 指标的信息，请参阅 TypeScript SDK 开发者指南可观测性部分的[指标](/develop/typescript/platform/observability#metrics)。

使用 [Cloud 指标](/cloud/metrics/)或[自托管 Server 指标](/self-hosted-guide/production-checklist#scaling-and-metrics)调试 Server 性能。

## 如何排查 TypeScript SDK 中的常见问题 {#troubleshoot-common-issues}

{/* The following was ported from \docs-src\typescript\troubleshooting.md */}

### 需要关注的两个地方

- Workflow 错误会反映在 Temporal Web 中。
- Worker 错误和日志会反映在终端中。

如果某些行为不符合预期，请务必检查这两个地方的错误消息。

### 过时的 Workflow

如果你在开发 Workflow 时发现代码没有按预期执行，首先要检查是否有旧的 Workflow 仍在运行。

如果那些旧的 Workflow 具有相同的名称且在相同的 Task Queue 上，Temporal 默认会尝试在新代码上继续执行它们。你可能会遇到莫名其妙的错误，原因可能是：

- Temporal 正在尝试执行已不存在于代码库中的旧 Workflow 代码，或者
- 新的 Client 代码期望 Temporal 执行尚不知晓的旧 Workflow/Activity 代码。

最明显的迹象是如果你发现 Temporal 的行为具有非确定性：运行同一 Workflow 两次得到不同的结果。

过时的 Workflow 通常不是大问题，因为产生的错误只是来自你不再想运行的代码的干扰。如果需要终止旧的过时 Workflow，可以通过 Temporal Web 或 Temporal CLI 来完成。

### Workflow/Activity 注册错误

**如果你的 Workflow 或 Activity 未正确导入或拼写错误**，以下是一些常见错误：

- `ApplicationFailure: 'MyFunction' is not a function`
- `Workflow did not register a handler for MyQuery`

请仔细检查 Worker 是否在正确的 Task Queue 上注册了正确的 Workflow 和 Activity 定义（函数名称）。

**如果你在 monorepo 中运行 Temporal**，则 `node_modules` 可能位于 Temporal 默认预期的位置之外，这会导致如下错误：

```bash
[ERROR] Module not found: Error: Can't resolve '@temporalio/workflow/lib/worker-interface.js' in '/src'
```

我们的 [Next.js 教程](https://learn.temporal.io/tutorials/typescript/nextjs)是为**在现有 monorepo 中**设置 Temporal 的人编写的，可能对此有所帮助。

当你传入 `workflowsPath` 时，Webpack 配置期望在相同或父/祖先目录中找到 `node_modules`。

**如果你自定义打包 Workflow**，可能会遇到如下错误：

```bash
[ERROR] Failed to activate workflow {
  runId: 'aaf84a83-51ce-462a-9ab7-6a641a703bff',
  error: ReferenceError: exports is not defined,
  workflowExists: false
}
```

Temporal Workflow Bundle 需要[导出一组符合 `@temporalio/workflow` 编译后 `worker-interface.ts` 的方法](https://github.com/temporalio/sdk-typescript/blob/eaa2d205c9bc5ff4a3b17c0b34f2dcf6b1e0264a/packages/worker/src/workflow/bundler.ts#L81)作为入口点。我们提供了 `bundleWorkflowCode` 方法来协助完成这项工作，尽管它使用了我们的 Webpack 设置。更多信息请参阅[注册类型](/develop/typescript/workers/run-worker-process#register-types)部分。

### Webpack 错误

TypeScript SDK 的 Worker 基于 `workflowsPath` 使用 [Webpack](https://webpack.js.org/) 打包 Workflow，并在 v8 隔离环境中运行。

如果 Webpack 未能创建 bundle，SDK 将抛出错误并使用 SDK 的[日志记录器](/develop/typescript/platform/observability#logging)输出 Webpack 日志。

如果终端中没有 Webpack 输出，请确认你没有禁用 SDK 日志记录（参见上面链接中关于 `Runtime.install()` 的说明）。

**TypeScript SDK 新手的一个常见错误是在 Workflow 代码中尝试使用 Node.js 内置模块和库。** 通常，最好的做法是将这些代码移到 Activity 中。

在 Workflow 隔离环境中**不起作用**的一些常见示例：

<details>
    <summary>
    导入 Node 内置模块
    </summary>

:::danger Antipattern

```ts
import fs from 'fs';

const config = fs.readFileSync('config.json', 'utf8');
```

:::

这是无效的，因为从文件系统读取是非确定性操作：文件可能在原始 Workflow Execution 时与 Workflow 重放时之间发生变化。

通常会在 Webpack 输出中看到如下形式的错误：

```
2021-10-14T19:22:00.606Z [INFO] Module not found: Error: Can't resolve 'fs' in '/Users/you/your-project/src'
2021-10-14T19:22:00.606Z [INFO] resolve 'fs' in '/Users/you/your-project/src'
2021-10-14T19:22:00.606Z [INFO]   Parsed request is a module
2021-10-14T19:22:00.606Z [INFO]   using description file: /Users/you/your-project/package.json (relative path: ./src)
2021-10-14T19:22:00.606Z [INFO]     Field 'browser' doesn't contain a valid alias configuration
```

</details>

<details>
    <summary>
    在 Workflow 代码中直接导入并调用 Activity
    </summary>

:::danger Antipattern

```ts
import { makeHTTPRequest } from './activities';

export async function yourWorkflow(): Promise<string> {
  return await makeHTTPRequest('https://temporal.io');
}
```

:::

这是无效的，因为 Activity 实现不应被 Workflow 代码直接引用。Activity 被 Workflow 使用以进行网络调用和从文件系统读取，这些操作本质上是非确定性的，因为它们依赖外部状态。Temporal 将 Activity 结果记录在 Workflow 历史中，如果 Workflow 需要重放，已完成的 Activity 不会重新运行，而是将记录的结果传递给 Workflow。

通常会在 Webpack 输出中看到如下形式的错误：

```
2021-10-14T19:46:52.731Z [INFO] ERROR in ./src/activities.ts 8:31-46
2021-10-14T19:46:52.731Z [INFO] Module not found: Error: Can't resolve 'http' in '/Users/you/your-project/src'
2021-10-14T19:46:52.731Z [INFO]
2021-10-14T19:46:52.731Z [INFO] BREAKING CHANGE: webpack < 5 used to include polyfills for node.js core modules by default.
2021-10-14T19:46:52.731Z [INFO] This is no longer the case. Verify if you need this module and configure a polyfill for it.
2021-10-14T19:46:52.731Z [INFO]
2021-10-14T19:46:52.731Z [INFO] If you want to include a polyfill, you need to:
2021-10-14T19:46:52.731Z [INFO]         - add a fallback 'resolve.fallback: { "http": require.resolve("stream-http") }'
2021-10-14T19:46:52.731Z [INFO]         - install 'stream-http'
2021-10-14T19:46:52.731Z [INFO] If you don't want to include a polyfill, you can use an empty module like this:
2021-10-14T19:46:52.731Z [INFO]         resolve.fallback: { "http": false }
```

要从 Workflow 代码中正确调用 Activity，请使用 `proxyActivities` 并确保只导入 Activity 类型。

```ts
import { proxyActivities } from '@temporalio/workflow';

import type * as activities from './activities';
const { makeHTTPRequest } = proxyActivities<typeof activities>();

export async function yourWorkflow(): Promise<string> {
  return await makeHTTPRequest('https://temporal.io');
}
```

### 开发环境正常但生产环境不正常

开发环境和生产环境差异的两个主要来源是打包和连接。

#### 生产环境打包

当调度 Workflow 时，你的 Client 可能会将被截断的名称作为 Workflow "Type" 发送。Webpack 可能会将 Workflow 的函数名称缩短。Temporal 无法处理缩短后的名称与预期的 Workflow 类型之间的不匹配。

你可能会遇到如下错误：

```
Error: 3 INVALID_ARGUMENT: WorkflowType is not set on request.
```

或者当 Webpack 将 Workflow 函数名称缩短（在本例中为单字母 `s`）时，你可能会在 Temporal Service 的 Web UI 中看到缩短后的名称：

<CaptionedImage
    src="/img/webui/stripped_workflow_types_in_webui.png"
    title="Temporal Web UI showing stripped 'Workflow Type' entries, in this case the single letter 's'"
/>

当打包工具截断 Workflow 函数名称时会出现此问题。Temporal 依赖这些名称在 Service Web UI 中设置"Workflow Type"。要防止构建过程缩短 Workflow 函数名称，请修改 Webpack 配置文件（`webpack.config.js`），在 `TerserPlugin` 配置部分设置保留原始名称的布尔值。将选项（`keep_fnames`）设置为 `true` 可防止名称被截断。

#### 连接到 Temporal Server

如果你在生产环境中尝试连接时遇到以下错误：

```bash
[TransportError: transport error]
```

这表明你的证书/密钥对出现了问题。将其打印出来并确认与预期完全匹配（通常，从生产密钥管理环境注入时，问题可能出在空白字符上）。

### 重置 Workflow 以处理逻辑错误

可以使用 Temporal CLI "回拨时间"，将 Workflow 历史重置到某个之前的时间点。可以阅读 Temporal CLI 文档：

- [按 ID 重启和重置 Workflow](/cli)
- [按二进制校验和标识符重置所有 Workflow](/cli)

如果需要以编程方式重置，TypeScript SDK 没有为此提供高级 API，但可以通过原始 gRPC 调用 [resetWorkflowExecution](https://typescript.temporal.io/api/classes/proto.temporal.api.workflowservice.v1.WorkflowService-1/#resetworkflowexecution) 来实现。

重置应仅用于处理代码中的严重逻辑错误：它不是用于处理瞬时失败（如下游服务不可达）的工具，也不应在正常应用流程中使用。

### gRPC 调用超时（context deadline exceeded）

不透明的 `context deadline exceeded` 错误来自 `gRPC`：

```
Error: 4 DEADLINE_EXCEEDED: context deadline exceeded
    at Object.callErrorFromStatus (/Users/swyx/Work/Temporal/samples-typescript/nextjs-oneclick/node_modules/@grpc/grpc-js/build/src/call.js:31:26)
    at Object.onReceiveStatus (/Users/swyx/Work/Temporal/samples-typescript/nextjs-oneclick/node_modules/@grpc/grpc-js/build/src/client.js:179:52)
    at Object.onReceiveStatus (/Users/swyx/Work/Temporal/samples-typescript/nextjs-oneclick/node_modules/@grpc/grpc-js/build/src/client-interceptors.js:336:141)
    at Object.onReceiveStatus (/Users/swyx/Work/Temporal/samples-typescript/nextjs-oneclick/node_modules/@grpc/grpc-js/build/src/client-interceptors.js:299:181)
    at /Users/swyx/Work/Temporal/samples-typescript/nextjs-oneclick/node_modules/@grpc/grpc-js/build/src/call-stream.js:145:78
    at processTicksAndRejections (node:internal/process/task_queues:78:11) {
  code: 4,
  details: 'context deadline exceeded',
  metadata: Metadata {
    internalRepr: Map(1) { 'content-type' => [Array] },
    options: {}
  },
  page: '/api/getBuyState'
}
```

多种情况可能导致此错误，包括网络抖动、超时时间设置过短以及服务器过载。查询一个查询处理器导致错误的 Workflow Execution 可能导致查询调用超时。

可以采取以下一些排查措施：

- 验证 Worker 与 Temporal Server 的连接是否正常，且延迟不异常高。
- 如果自己运行 Temporal Server，请检查[服务器指标](/self-hosted-guide/production-checklist#scaling-and-metrics)以确保其未过载。
- 如果超时的是查询，请检查 Worker 的日志，看它们是否在处理查询时遇到问题。

如果上述措施都无法帮助你找到超时原因，请尝试构造一个最小复现示例，我们很乐意为你提供帮助。
