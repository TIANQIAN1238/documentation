<a id="plugins"></a>
# 插件 {#plugins}

**插件**是一个抽象，允许您自定义 Temporal Worker 设置的任何方面，包括
注册 Workflow 和 Activity Definition、修改 Worker 和客户端选项等等。使用插件，您可以
为您公司的工程师构建可重用的开源库或构建附加组件。

本指南将教您如何创建插件，并为平台工程师提供使用和管理的一般指导
Temporal 的原语。

以下是插件的一些常见用例：

- AI代理SDK
- 可观察性、跟踪或日志记录中间件
- 添加可靠的内置功能，例如 LLM 呼叫、消息传递系统和支付基础设施
- 加密或合规中间件

<a id="how-to-build-a-plugin"></a>
## 如何构建插件 {#how-to-build-a-plugin}

开始构建插件的推荐方法是使用 `SimplePlugin`。这种抽象将解决绝大多数问题
人们想要编写的插件数量。

<a id="workflow-friendly-libraries"></a>
### Workflow 友好的库 {#workflow-friendly-libraries}

如果您想抽象出一些 Temporal 特定的详细信息，您可以提供一个在 Workflow 中使用的库。
您的用户。您的库将调用您包含在插件中的元素，例如活动、Child Workflows、Signals、
Update、查询、Nexus 操作、拦截器、Data Converter 以及任何其他代码，只要遵循这些代码即可
要求：

- 在 Workflow 上下文中运行的任何代码都必须是 [确定性的](/workflow-definition#deterministic-constraints)，
  这意味着重播时它会产生相同的命令和结果。例如，不调用系统时间API，生成
  随机值，或从 Workflow 上下文代码执行直接网络和文件 I/O；将该工作移至“活动”或
  Nexus 操作。
- 请参阅 [可观测性](/evaluate/development-production-features/observability) 以避免重复观察面
  Workflow重播时的效果。
- 将其他 Side Effect 放入活动或 [当地活动](/local-activity) 中。这有助于您的 Workflow 处理
  在与最初开始的地方不同的进程中重新启动、恢复或执行，而不会丢失正确性
  或状态一致性。
- 请参阅 [测试你的插件](#testing-your-plugin) 编写检查 Side Effect 问题的测试。
- 它应该运行得很快，因为它可能会在很长的 Workflow Execution 期间重播很多次。更昂贵的代码应该
  进入“活动”或“Nexus 操作”。

插件应允许用户将其 Workflow 分解为活动，以及 Child Workflow 和 Nexus 调用
当需要时。这使用户可以通过重试和超时进行精细控制，通过 Temporal UI 进行调试，
重置、暂停和取消的可操作性、效率和可恢复性的记忆以及使用任务的可扩展性
队列和 Worker。

用户将 Workflow 用于：

- 协调和决策
- InterActivity 通过 [消息传递](/evaluate/development-production-features/workflow-message-passing)
- 追踪和可观察性

<a id="making-changes-to-your-library"></a>
#### 对您的库进行更改 {#making-changes-to-your-library}

您的用户可能希望在 Worker 代码的部署中保持 Workflow 运行。如果他们的部署
包含新版本的插件，对插件的更改可能会破坏在新版本之前启动的 Workflow 代码
版本已部署。这可能是由于
[代码更改带来的非确定性行为](/workflow-definition#non-deterministic-change) 在您的插件中。

请参阅 [测试](#testing-your-plugin) 以了解如何对此进行测试。而且，如果您进行实质性更改，则需要使用
[修补](/patching)。

<a id="built-in-workflows"></a>
### 内置 Workflow {#built-in-workflows}

您可以在 `SimplePlugin` 中提供内置 Workflow。它可以作为 Child Workflow 或独立调用。当你想要的时候
要提供比 Activity 更复杂的功能，您可以：

- 使用直接在最终用户的 Workflow 中运行的 [Workflow 库](#workflow-friendly-libraries)
- 添加 Child Workflow

当满足以下一个或多个条件时，请考虑添加 Child Workflow：

- 那个孩子应该比父母活得更久。
- Workflow Event History 否则将在父级中 [没有规模](/workflow-execution/event#event-history-limits)
  Workflow。
- 当您想要为子级提供单独的 Workflow ID 以便其可以独立于父级状态进行操作时
  （取消、终止、暂停）。

任何 Workflow 都可以作为独立的 Workflow 或 Child Workflow 运行，因此在
`SimplePlugin` 与注册任何 Workflow 相同。

### 上下文传播器 {#context-propagators}

上下文传播器通过 Temporal 标头跨 Workflow、Activity 和 Child Workflow 边界传递自定义键值数据（例如跟踪 ID、租户 ID 或身份验证令牌）。有关其工作原理的详细信息，请参阅 [上下文传播](/encyclopedia/context-propagation)。

通过插件注册的传播器将附加到用户或以前的插件已设置的任何传播器。

```py
def workflow_runner(runner: WorkflowRunner | None) -> WorkflowRunner:
    if not runner:
        raise ValueError("No WorkflowRunner provided to the plugin.")

    # If in sandbox, add additional passthrough
    if isinstance(runner, SandboxedWorkflowRunner):
        return dataclasses.replace(
            runner,
            restrictions=runner.restrictions.with_passthrough_modules("module"),
        )
    return runner


plugin = SimplePlugin("organization.PluginName", workflow_runner=workflow_runner)
```

## 测试你的插件 {#typescript}

要测试您的插件，您将编写一个正常的 Temporal Workflow 测试，并将该插件包含在您的客户端中。

两个特别关注的问题是版本控制测试，当您对插件进行更改时，以及测试不需要的方面
影响。

<a id="testing-your-plugin"></a>
### 版本控制测试 {#testing-your-plugin}

当您在插件交付给用户后对其进行更改时，我们建议您在每个重要更改上设置 [重播测试](/develop/python/best-practices/testing-suite#replay)，以确保不会给用户造成非确定性错误。

<a id="versioning-tests"></a>
### Side Effects 测试 {#versioning-tests}

您的插件应该满足 Workflow 在与开始时不同的进程中恢复然后重播的需求
从一开始，这可能会发生，例如，在间歇性故障之后。

您可以通过关闭 Workflow 缓存来确保您不依赖于本地 Side Effect，这意味着
Workflow 每次进行时都会从顶部重播：
