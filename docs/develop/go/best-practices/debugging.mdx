在测试或执行 Workflow Definition 之前，您可以使用您最喜欢的 IDE 提供的调试器工具来调试它们。

Temporal Go SDK 包括死锁检测，如果代码阻塞超过一秒而不放弃执行控制，则该死锁检测会使 Workflow 任务失败。
因此，在调试期间单步执行 Workflow Definition 时，您经常会遇到 `PanicError: Potential deadlock detected`。

为了缓解此问题，您可以在调试 Workflow Definition 之前将 `Temporal_DEBUG` 环境变量设置为 `true`。

:::note

确保仅在调试期间将 `Temporal_DEBUG` 设置为 true。

:::

## 如何在开发环境中调试 {#debug-in-a-development-environment}

除了日志记录和调试器等常规开发工具之外，您还可以使用 [网页用户界面](/web-ui) 或 [Temporal CLI](/cli) 查看 Workflow 中发生的情况。

## 如何在 Go 中测试 Workflow Definition {#testing-and-debugging}

Temporal Go SDK 提供了一个测试框架来促进测试 Workflow 实现。

该框架适合于实现 Workflow 逻辑的单元测试和功能测试。

以下代码实现 `SimpleWorkflow` 示例的单元测试：

```go
package sample

import (
        "context"
        "errors"
        "testing"

        "github.com/stretchr/testify/mock"
        "github.com/stretchr/testify/suite"

        "go.temporal.io/sdk/activity"
        "go.temporal.io/sdk/testsuite"
)

type UnitTestSuite struct {
        suite.Suite
        testsuite.WorkflowTestSuite

        env *testsuite.TestWorkflowEnvironment
}

func (s *UnitTestSuite) SetupTest() {
        s.env = s.NewTestWorkflowEnvironment()
}

func (s *UnitTestSuite) AfterTest(suiteName, testName string) {
        s.env.AssertExpectations(s.T())
}

func (s *UnitTestSuite) Test_SimpleWorkflow_Success() {
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_success")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}

func (s *UnitTestSuite) Test_SimpleWorkflow_ActivityParamCorrect() {
        s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
          func(ctx context.Context, value string) (string, error) {
                s.Equal("test_success", value)
                return value, nil
        })
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_success")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}

func (s *UnitTestSuite) Test_SimpleWorkflow_ActivityFails() {
        s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
          "", errors.New("SimpleActivityFailure"))
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_failure")

        s.True(s.env.IsWorkflowCompleted())

        err := s.env.GetWorkflowError()
        s.Error(err)
        var applicationErr *temporal.ApplicationError
        s.True(errors.As(err, &applicationErr))
        s.Equal("SimpleActivityFailure", applicationErr.Error())
}

func TestUnitTestSuite(t *testing.T) {
        suite.Run(t, new(UnitTestSuite))
}
```

＃＃＃＃ 设置

为了运行单元测试，我们首先定义一个测试套件结构，它吸收了
[作证](https://pkg.go.dev/github.com/stretchr/testify/suite) 的基本套件功能
通过 `suite.Suite` 和 Temporal 测试框架的套件功能
`testsuite.WorkflowTestSuite`。因为这个测试套件中的每个测试都会测试我们的 Workflow，所以我们
向我们的结构添加一个属性来保存测试环境的实例。这允许我们初始化
设置方法中的测试环境。为了测试 Workflow，我们使用 `testsuite.TestWorkflowEnvironment`。

接下来，我们实现一个 `SetupTest` 方法来在每次测试之前设置一个新的测试环境。这样做
确保每个测试在其自己的隔离沙箱中运行。我们还实现了 `AfterTest` 函数
我们断言我们设置的所有模拟确实是通过调用 `s.env.AssertExpectations(s.T())` 来调用的。

整个测试的超时可以在 Workflow 或 Activity 环境中使用 `SetTestTimeout` 设置。

最后，我们创建一个由 `go test` 命令识别的常规测试函数，并将结构传递给 `suite.Run`。

<a id="setup"></a>
#### 一个简单的测试 {#setup}

我们可以编写的最简单的测试用例是让测试环境执行 Workflow，然后
评估结果。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_Success() {
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_success")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

调用 `s.env.ExecuteWorkflow(...)` 会执行 Workflow 逻辑以及
测试过程。 `s.env.ExecuteWorkflow(...)` 的第一个参数包含 Workflow 函数，
并且任何后续参数都包含 Workflow 声明的自定义输入参数的值
功能。

> 请注意，除非 Activity 调用被模拟或 Activity 实现
> 更换（见[Activity 模拟和重写](#activity-mocking-and-overriding)），测试环境
> 将执行实际的 Activity 代码，包括对外部服务的任何调用。

执行上例中的 Workflow 后，我们断言 Workflow 运行完成
通过调用 `s.env.IsWorkflowComplete()`。我们还断言断言没有返回任何错误
关于`s.env.GetWorkflowError()`的返回值。如果我们的 Workflow 返回一个值，我们可以有
通过调用 `s.env.GetWorkflowResult(&value)` 检索该值并对此进行附加断言
值。

<a id="a-simple-test"></a>
#### Activity 模拟和重写 {#a-simple-test}

在 Workflow 上运行单元测试时，我们希望单独测试 Workflow 逻辑。另外，
我们想在测试运行期间注入 Activity 错误。测试框架提供了两种机制
支持这些场景：Activity 模拟和 Activity 覆盖。这两种机制都允许
您可以更改 Workflow 调用的活动的行为，而无需修改实际的
Workflow 代码。

让我们看一下模拟通过“Activity 模拟”机制失败的测试的测试。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_ActivityFails() {
        s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
          "", errors.New("SimpleActivityFailure"))
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_failure")

        s.True(s.env.IsWorkflowCompleted())

        err := s.env.GetWorkflowError()
        s.Error(err)
        var applicationErr *temporal.ApplicationError
        s.True(errors.As(err, &applicationErr))
        s.Equal("SimpleActivityFailure", applicationErr.Error())
}
```

此测试模拟由我们的 Workflow 调用的 Activity `SimpleActivity` 的执行
`SimpleWorkflow` 返回错误。我们通过在测试环境上设置模拟来实现这一点
对于返回错误的 `SimpleActivity`。

```go
s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
  "", errors.New("SimpleActivityFailure"))
```

设置模拟后，我们现在可以通过 `s.env.ExecuteWorkflow(...)` 方法执行 Workflow 并
断言 Workflow 成功完成并返回预期错误。

简单地模拟执行以返回所需的值或错误是一种非常强大的机制
隔离 Workflow 逻辑。然而，有时我们想用替代实现替换 Activity
以支持更复杂的测试场景。假设我们要验证 Activity 是否被调用
使用正确的参数。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_ActivityParamCorrect() {
        s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
          func(ctx context.Context, value string) (string, error) {
                s.Equal("test_success", value)
                return value, nil
        })
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_success")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

在此示例中，我们提供函数实现作为 `Return` 的参数。这使我们能够
为 Activity `SimpleActivity` 提供替代实现。框架将执行此操作
每当调用 Activity 时都会调用函数，并将函数的返回值作为结果传递
Activity 调用的。此外，该框架将验证“模拟”的签名
函数与原始 Activity 函数的签名匹配。

由于这可以是一个完整的函数，因此我们在这里可以做什么没有限制。在这个
例如，我们断言 `value` 参数与我们传递给 Workflow 的值参数具有相同的内容。

<a id="activity-mocking-and-overriding"></a>
#### 查询 {#activity-mocking-and-overriding}

`TestWorkflowEnvironment` 实例具有 [`QueryWorkflow()`方法](https://pkg.go.dev/go.temporal.io/temporal/internal#TestWorkflowEnvironment.QueryWorkflow)，可让您获得 Query 当前运行的 Workflow 的状态。
例如，假设您有一个 Workflow，它允许您 Query 长时间运行的任务的进度，如下所示。

```go
func ProgressWorkflow(ctx workflow.Context, percent int) error {
	logger := workflow.GetLogger(ctx)

	err := workflow.SetQueryHandler(ctx, "getProgress", func(input []byte) (int, error) {
		return percent, nil
	})
	if err != nil {
		logger.Info("SetQueryHandler failed.", "Error", err)
		return err
	}

	for percent = 0; percent<100; percent++ {
                // Important! Use `workflow.Sleep()`, not `time.Sleep()`, because Temporal's
                // test environment doesn't stub out `time.Sleep()`.
		workflow.Sleep(ctx, time.Second*1)
	}

	return nil
}
```

Workflow 以百分比形式跟踪任务的当前进度，并每秒将百分比增加 1。
下面是如何编写查询此 Workflow 的测试用例。
请注意，在 `ExecuteWorkflow()` 完成后或在 `RegisterDelayedCallback()` 回调中，您应该始终使用 Query 和 Workflow，否则您会遇到 `runtime error` 恐慌。

```go
func (s *UnitTestSuite) Test_ProgressWorkflow() {
	value := 0

	// After 10 seconds plus padding, progress should be 10.
	// Note that `RegisterDelayedCallback()` doesn't actually make your test wait for 10 seconds!
	// Temporal's test framework advances time internally, so this test should take < 1 second.
	s.env.RegisterDelayedCallback(func() {
		res, err := s.env.QueryWorkflow("getProgress")
		s.NoError(err)
		err = res.Get(&value)
		s.NoError(err)
		s.Equal(10, value)
	}, time.Second*10+time.Millisecond*1)

	s.env.ExecuteWorkflow(ProgressWorkflow, 0)

	s.True(s.env.IsWorkflowCompleted())

	// Once the workflow is completed, progress should always be 100
	res, err := s.env.QueryWorkflow("getProgress")
	s.NoError(err)
	err = res.Get(&value)
	s.NoError(err)
	s.Equal(value, 100)
}
```

:::note

`RegisterDelayedCallback` 也可用于发送 [Signals](/sending-messages#sending-signals)。
使用“Signal-With-Start”时，将延迟设置为`0`。
:::
