Temporal 应用程序开发指南的测试部分描述了促进 Workflow 和集成测试的框架。

在 Temporal 的上下文中，您可以创建以下类型的自动化测试：

- **端到端：** 运行 Temporal Server 和 Worker 及其所有 Workflow 和活动；从客户端启动 Workflow 并与之交互。
- **集成：** 端到端和单元测试之间的任何内容。
  - 使用模拟上下文和其他 SDK 导入（通常是网络请求）运行活动。
  - 使用模拟活动运行 Worker，并使用客户端启动 Workflow。
  - 使用模拟 SDK 导入运行 Workflow。
- **单元：** 运行一段 Workflow 或 Activity 代码（函数或方法）并模拟它调用的任何代码。

我们通常建议将大部分测试编写为集成测试。

由于测试服务器支持跳过时间，因此请使用测试服务器与 Worker 进行端到端测试和集成测试。

## 测试框架 {#test-frameworks}

Temporal Go SDK 提供了一个测试框架来促进测试 Workflow 实现。

该框架适合于实现 Workflow 逻辑的单元测试和功能测试。

<a id="test-setup"></a>
## 测试设置 {#test-setup}

为了运行单元测试，我们首先定义一个测试套件结构，它通过 `suite.Suite` 吸收 [作证](https://pkg.go.dev/github.com/stretchr/testify/suite) 的基本套件功能，并通过 `testsuite.WorkflowTestSuite` 吸收 Temporal 测试框架的套件功能。

因为这个测试套件中的每个测试都会测试我们的 Workflow，所以我们
向我们的结构添加一个属性来保存测试环境的实例。这允许我们在设置方法中初始化测试环境。

为了测试 Workflow，我们使用 `testsuite.TestWorkflowEnvironment`。

```go
type UnitTestSuite struct {
        suite.Suite
        testsuite.WorkflowTestSuite

        env *testsuite.TestWorkflowEnvironment
}
```

接下来，我们实现一个 `SetupTest` 方法来在每次测试之前设置一个新的测试环境。
这样做可以确保每个测试都在自己的隔离沙箱中运行。

```go
func (s *UnitTestSuite) SetupTest() {
        s.env = s.NewTestWorkflowEnvironment()
}
```

我们还实现了一个 `AfterTest` 函数，在该函数中我们断言我们设置的所有模拟确实是通过调用 `s.env.AssertExpectations(s.T())` 来调用的。
整个测试的超时可以在 Workflow 或 Activity 环境中使用 `SetTestTimeout` 设置。

```go
func (s *UnitTestSuite) AfterTest(suiteName, testName string) {
        s.env.AssertExpectations(s.T())
}
```

最后，我们创建一个由 `go test` 命令识别的常规测试函数，并将结构传递给 `suite.Run`。

```go
func TestUnitTestSuite(t *testing.T) {
        suite.Run(t, new(UnitTestSuite))
}
```

## 测试活动 {#test-activities}

Activity 可以使用模拟 Activity 环境进行测试，该环境提供了一种模拟 Activity 上下文、监听心跳和取消 Activity 的方法。
此行为允许您通过直接调用 Activity 来单独测试 Activity，而无需创建 Worker 来运行 Activity。

<a id="mock-and-override-activities"></a>
## 模拟并覆盖活动 {#mock-and-override-activities}

在 Workflow 上运行单元测试时，我们希望单独测试 Workflow 逻辑。此外，我们希望在测试运行期间注入 Activity 错误。测试框架提供了两种支持这些场景的机制：Activity 模拟和 Activity 覆盖。这两种机制都允许您更改 Workflow 调用的活动的行为，而无需修改实际的 Workflow 代码。

让我们看一下模拟通过“Activity 模拟”机制失败的测试的测试。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_ActivityFails() {
        s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
          "", errors.New("SimpleActivityFailure"))
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_failure")

        s.True(s.env.IsWorkflowCompleted())

        err := s.env.GetWorkflowError()
        s.Error(err)
        var applicationErr *temporal.ApplicationError
        s.True(errors.As(err, &applicationErr))
        s.Equal("SimpleActivityFailure", applicationErr.Error())
}
```

此测试模拟由返回错误的 Workflow `SimpleWorkflow` 调用的 Activity `SimpleActivity` 的执行。我们通过在返回错误的 `SimpleActivity` 测试环境上设置模拟来实现此目的。

```go
s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
  "", errors.New("SimpleActivityFailure"))
```

设置模拟后，我们现在可以通过 `s.env.ExecuteWorkflow(...)` 方法执行 Workflow 并断言 Workflow 成功完成并返回预期的错误。

简单地模拟执行以返回所需的值或错误是隔离 Workflow 逻辑的非常强大的机制。
然而，有时我们想用替代实现替换 Activity 以支持更复杂的测试场景。
假设我们要验证是否使用正确的参数调用了 Activity。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_ActivityParamCorrect() {
        s.env.OnActivity(SimpleActivity, mock.Anything, mock.Anything).Return(
          func(ctx context.Context, value string) (string, error) {
                s.Equal("test_success", value)
                return value, nil
        })
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_success")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

在此示例中，我们提供函数实现作为 `Return` 的参数。
这使我们能够为 Activity `SimpleActivity` 提供替代实现。
每当调用 Activity 时，框架都会执行此函数，并将函数的返回值作为 Activity 调用的结果传递。

此外，框架将验证“mock”函数的签名是否与原始 Activity 函数的签名匹配。

由于这可以是一个完整的函数，因此我们在这里可以做什么没有限制。在此示例中，我们断言 `value` 参数与我们传递给 Workflow 的值参数具有相同的内容。

<a id="mock-and-override-nexus-operations"></a>
## 模拟并覆盖 Nexus 操作 {#mock-and-override-nexus-operations}

通过模拟 Nexus 操作，您可以测试执行 Nexus 操作的 Workflow，而无需
Nexus 处理程序运行实际的 Nexus 操作。您可以模拟 Nexus 操作或覆盖其
实现与测试Workflow环境。

考虑模拟 Nexus 操作调用的测试。在此示例中，Nexus 操作是
称为`sample-operation`，输入类型为`SampleInput`，输出类型为`SampleOutput`，并且
它属于 Nexus 服务 `sample-service`。

下面的示例模拟对 Nexus 同步操作的调用，由返回值指示
类型 `*Nexus.HandlerStartOperationResultSync[T]`。由于 `OnNexusOperation` 需要知道
操作的名称、输入类型和输出类型，并且您可能无权访问 Nexus 操作
在处理程序端，您可以使用 `Nexus.NewOperationReference` 创建 Nexus 操作引用
表示没有其实现的操作（基本上，它表示
Nexus 操作）。您还可以使用操作本身而不是创建操作
参考（如果有的话）。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_NexusSyncOperation() {
        s.env.OnNexusOperation(
                "sample-service",
                nexus.NewOperationReference[SampleInput, SampleOutput]("sample-operation"),
                SampleInput{},
                workflow.NexusOperationOptions{},
        ).Return(
                &nexus.HandlerStartOperationResultSync[SampleOutput]{
                        Value: SampleOutput{},
                },
                nil, // error if you want to simulate an error in the ExecuteOperation call
        )
        // You can also add a delay to return the mock values by calling After().
        // Eg: s.env.OnNexusOperation(...).Return(...).After(1*time.Second)

        s.env.ExecuteWorkflow(SimpleWorkflow, "test_nexus_operation")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

除了同步操作之外，您还可以模拟异步操作。下面的例子
演示如何测试执行 Nexus 异步操作的 Workflow。返回值
在这种情况下，类型必须是 `*Nexus.HandlerStartOperationResultAsync` 和 `OperationToken`，这可以
可以是您选择的任何字符串。此外，您必须致电 `RegisterNexusAsyncOperationCompletion`
注册由元组服务名称、操作标识的异步操作的结果
名称和操作令牌。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_NexusAsyncOperation() {
        s.env.OnNexusOperation(
                "sample-service",
                nexus.NewOperationReference[SampleInput, SampleOutput]("sample-operation"),
                SampleInput{},
                workflow.NexusOperationOptions{},
        ).Return(
                &nexus.HandlerStartOperationResultAsync{
                        OperationToken: "sample-operation-token",
                },
                nil, // error if you want to simulate an error in the ExecuteOperation call
        )
        err := env.RegisterNexusAsyncOperationCompletion(
                "sample-service",
                "sample-operation",
                "sample-operation-token", // must match the OperationToken above
                SampleOutput{},
                nil,            // error if you want to simulate an error in the operation
                2*time.Second,  // delay to simulate how long the operation takes after it starts
        )
        s.NoError(err)
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_nexus_operation")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

如果您的 Workflow 执行多个 Nexus 异步操作，您可以使用以下命令模拟它们中的每一个
不同的操作令牌，并使用相应的操作令牌注册完成结果。

如果模拟 Nexus 操作还不够，并且您需要在 Nexus 时运行一些自定义逻辑
操作执行后，您可以按如下方式覆盖它。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_NexusSyncOperation() {
        var SampleOperation = nexus.NewSyncOperation(
                "sample-operation",
                func(ctx context.Context, input SampleInput, options nexus.StartOperationOptions) (SampleOutput, error) {
                        // Custom logic here.
                        return SampleOutput{}, nil
                },
        )

        service := nexus.NewService("sample-service")
        s.NoError(service.Register(SampleOperation))
        env.RegisterNexusService(service)
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_nexus_operation")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

以下示例显示如何重写 Nexus 异步操作。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_NexusSyncOperation() {
        SampleHandlerWorkflow := func(_ workflow.Context, input SampleInput) (SampleOutput, error) {
                // Custom logic here.
                return SampleOutput{}, nil
        }
        SampleOperation := nexus.NewWorkflowRunOperation(
                "sample-operation",
                SampleHandlerWorkflow,
                func(ctx context.Context, input SampleInput, options nexus.StartOperationOptions) (client.StartWorkflowOptions, error) {
                        // Custom logic to build client.StartWorkflowOptions.
                        return client.StartWorkflowOptions{}, nil
                },
        )

        service := nexus.NewService("sample-service")
        s.NoError(service.Register(SampleOperation))
        env.RegisterNexusService(service)
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_nexus_operation")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

<a id="test-workflows"></a>
## 测试 Workflows {#test-workflows}

在 Workflow 上运行单元测试时，我们希望单独测试 Workflow 逻辑。
我们可以编写的最简单的测试用例是让测试环境执行 Workflow，然后评估结果。

```go
func (s *UnitTestSuite) Test_SimpleWorkflow_Success() {
        s.env.ExecuteWorkflow(SimpleWorkflow, "test_success")

        s.True(s.env.IsWorkflowCompleted())
        s.NoError(s.env.GetWorkflowError())
}
```

调用 `s.env.ExecuteWorkflow(...)` 会执行 Workflow 逻辑以及测试流程内任何调用的活动。 `s.env.ExecuteWorkflow(...)` 的第一个参数包含 Workflow 函数，任何后续参数都包含 Workflow 声明的自定义输入参数的值。

> 请注意，除非模拟 Activity 调用或替换 Activity 实现（请参阅 [Activity 模拟和重写](#mock-and-override-activities)），否则测试环境将执行实际的 Activity 代码，包括对外部服务的任何调用。

在执行上面示例中的 Workflow 后，我们断言 Workflow 通过调用 `s.env.IsWorkflowCompleted()` 完成运行。我们还断言通过断言 `s.env.GetWorkflowError()` 的返回值不会返回任何错误。
如果我们的 Workflow 返回一个值，我们可以通过调用 `s.env.GetWorkflowResult(&value)` 检索该值，并对该值进行附加断言。

<a id="query-tests"></a>
### Query 测试 {#query-tests}

`TestWorkflowEnvironment` 实例具有 [`QueryWorkflow()`方法](https://pkg.go.dev/go.temporal.io/temporal/internal#TestWorkflowEnvironment.QueryWorkflow)，可让您获得 Query 当前运行的 Workflow 的状态。
例如，假设您有一个 Workflow，它允许您 Query 长时间运行的任务的进度，如下所示。

```go
func ProgressWorkflow(ctx workflow.Context, percent int) error {
	logger := workflow.GetLogger(ctx)

	err := workflow.SetQueryHandler(ctx, "getProgress", func(input []byte) (int, error) {
		return percent, nil
	})
	if err != nil {
		logger.Info("SetQueryHandler failed.", "Error", err)
		return err
	}

	for percent = 0; percent<100; percent++ {
                // Important! Use `workflow.Sleep()`, not `time.Sleep()`, because Temporal's
                // test environment doesn't stub out `time.Sleep()`.
		workflow.Sleep(ctx, time.Second*1)
	}

	return nil
}
```

Workflow 以百分比形式跟踪任务的当前进度，并每秒将百分比增加 1。
下面是如何编写查询此 Workflow 的测试用例。
请注意，在 `ExecuteWorkflow()` 完成后或在 `RegisterDelayedCallback()` 回调中，您应该始终使用 Query 和 Workflow，否则您会遇到 `runtime error` 恐慌。

```go
func (s *UnitTestSuite) Test_ProgressWorkflow() {
	value := 0

	// After 10 seconds plus padding, progress should be 10.
	// Note that `RegisterDelayedCallback()` doesn't actually make your test wait for 10 seconds!
	// Temporal's test framework advances time internally, so this test should take < 1 second.
	s.env.RegisterDelayedCallback(func() {
		res, err := s.env.QueryWorkflow("getProgress")
		s.NoError(err)
		err = res.Get(&value)
		s.NoError(err)
		s.Equal(10, value)
	}, time.Second*10+time.Millisecond*1)

	s.env.ExecuteWorkflow(ProgressWorkflow, 0)

	s.True(s.env.IsWorkflowCompleted())

	// Once the workflow is completed, progress should always be 100
	res, err := s.env.QueryWorkflow("getProgress")
	s.NoError(err)
	err = res.Get(&value)
	s.NoError(err)
	s.Equal(value, 100)
}
```

:::note

`RegisterDelayedCallback` 也可用于发送 [Signals](/sending-messages#sending-signals)。
使用“Signal-With-Start”时，将延迟设置为`0`。
:::

### 如何跳过时间 {#skip-time}

一些长期运行的 Workflow 可以持续数月甚至数年。
实施测试框架允许您的 Workflow 代码跳过时间并在几秒钟内完成测试，而不是 Workflow 指定的数量。

例如，如果您的 Workflow 睡眠了一天，或者 Activity 发生了重试间隔较长的故障，则无需等待整个睡眠周期的长度来测试睡眠功能是否起作用。
相反，通过及时向前跳跃并及时完成测试来测试睡眠后发生的逻辑。

大多数 SDK 中包含的测试框架是 Temporal Server 的内存中实现，支持跳过时间。
时间是 `TestWorkflowEnvironment` 实例的全局属性：跳过时间（自动或手动）适用于所有当前正在运行的测试。
如果不同的测试需要不同的时间行为，请按系列运行测试或使用测试服务器的单独实例运行测试。
例如，您可以并行运行所有具有自动时间跳过功能的测试，然后串行运行所有具有手动时间跳过功能的测试，然后并行运行所有不带时间跳过功能的测试。

## 如何重播 Workflow Execution {#replay}

重播重新创建了 Workflow Execution 的确切状态。
您可以从 Event History 的开头重播 Workflow。

仅当 [Workflow Definition](/workflow-definition) 从确定性的角度来看与提供的历史记录兼容时，重播才会成功。

当您测试对 Workflow Definition 的更改时，我们建议执行以下操作作为 CI 检查的一部分：

1. 确定被测 Worker 代码将针对哪些 Workflow 类型或 Task Queue（或两者）。
2. 使用 SDK 客户端以编程方式或通过 Temporal CLI 从每个 Task Queue 下载一组代表性的最近打开和关闭的 Workflow 的事件历史记录。
3. 通过重播运行事件历史记录。
4. 如果重播期间遇到任何错误，则 CI 失败。

以下是获取和重播事件历史记录的示例：

使用 [Worker.Workflow重播](https://pkg.go.dev/go.temporal.io/sdk/worker#WorkflowReplayer) 从 Event History 重播现有的 Workflow Execution 以复制错误。

例如，以下代码检索 Workflow 的 Event History：

```go
import (
	"context"

	"go.temporal.io/api/enums/v1"
	"go.temporal.io/api/history/v1"
	"go.temporal.io/sdk/client"
)

func GetWorkflowHistory(ctx context.Context, client client.Client, id, runID string) (*history.History, error) {
	var hist history.History
	iter := client.GetWorkflowHistory(ctx, id, runID, false, enums.HISTORY_EVENT_FILTER_TYPE_ALL_EVENT)
	for iter.HasNext() {
		event, err := iter.Next()
		if err != nil {
			return nil, err
		}
		hist.Events = append(hist.Events, event)
	}
	return &hist, nil
}
```

然后可以使用该历史记录来_重播_。
例如，以下代码创建 `WorkflowReplayer` 并注册 `YourWorkflow` Workflow 函数。
然后它调用 `ReplayWorkflowHistory` 来_重播_ Event History 并返回错误代码。

```go
import (
	"context"

	"go.temporal.io/sdk/client"
	"go.temporal.io/sdk/worker"
)

func ReplayWorkflow(ctx context.Context, client client.Client, id, runID string) error {
	hist, err := GetWorkflowHistory(ctx, client, id, runID)
	if err != nil {
		return err
	}
	replayer := worker.NewWorkflowReplayer()
	replayer.RegisterWorkflow(YourWorkflow)
	return replayer.ReplayWorkflowHistory(nil, hist)
}
```

上面的代码将导致 Worker 使用原始 Event History 重新执行 Workflow 的 Workflow 函数。
如果遵循明显不同的代码路径或某些代码导致死锁，则会在错误代码中返回。
在本地重播 Workflow Execution 是准确查看给定输入和事件所采用的代码路径的好方法。

您可以通过注册所有需要的 Workflow 实现然后重复调用 `ReplayWorkflowHistory` 来重播许多事件历史记录。
