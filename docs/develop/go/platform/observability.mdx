本页介绍了查看 [Temporal 应用](/temporal#temporal-application) 当前状态的多种方法，即查看 [Temporal平台](/workflow-execution) 跟踪的 [Workflow Executions](/temporal#temporal-platform) 以及任何指定 Workflow Execution 当前或执行点的状态的方法。

本节介绍与查看应用程序状态相关的功能，包括：

- [指标](#metrics)
- [追踪](#tracing)
- [记录](#logging)
- [Visibility](#visibility)

<a id="metrics"></a>
## 如何发出指标 {#metrics}
**如何使用 Temporal Go SDK 发出应用程序指标。**

每个 Temporal SDK 都能够从客户端或 Worker 进程发出一组可选的指标。
有关能够发出的指标的完整列表，请参阅 [SDK指标参考](/references/sdk-metrics)。

- 有关 Prometheus 和 Grafana 集成的概述，请参阅 [监控](/self-hosted-guide/monitoring) 指南。
- 有关指标列表，请参阅 [SDK指标参考](/references/sdk-metrics)。
- 有关使用 Go SDK 公开指标的端到端示例，请参阅 [样品去](https://github.com/temporalio/samples-go/tree/main/metrics) 存储库。

要从 Go 中的 Temporal CLIent 发出指标，请从 [客户选项](https://pkg.go.dev/go.temporal.io/sdk/internal/common/metrics#Handler) 创建 [指标处理程序](https://pkg.go.dev/go.temporal.io/sdk@v1.15.0/internal#ClientOptions) 并指定 Prometheus 使用的侦听器地址。

```go
client.Options{
		MetricsHandler: sdktally.NewMetricsHandler(newPrometheusScope(prometheus.Configuration{
			ListenAddress: "0.0.0.0:9090",
			TimerType:     "histogram",
		}
```

Go SDK 目前支持 [相符](https://pkg.go.dev/go.temporal.io/sdk/contrib/tally) 库；但是，Tally 提供 [可扩展的自定义指标报告](https://github.com/uber-go/tally#report-your-metrics)，它通过 [`WithCustomMetricsReporter`](/references/server-options#withcustommetricsreporter) API 公开。

有关详细信息，请参阅 [获取指标样本](https://github.com/temporalio/samples-go/tree/main/metrics)。

<a id="tracing"></a>
## 追踪 {#tracing}
跟踪允许您查看 Workflow 的调用图及其活动、Nexus 操作和 Child Workflow。

Go SDK 为 [开放式遥测](https://pkg.go.dev/go.temporal.io/sdk/contrib/opentelemetry)、[开放式追踪](https://pkg.go.dev/go.temporal.io/sdk/contrib/opentracing) 和 [数据狗](https://pkg.go.dev/go.temporal.io/sdk/contrib/datadog/tracing) 提供跟踪拦截器。

首先，为客户端实例化创建一个跟踪拦截器。

```go
// OpenTelemetry
tracingInterceptor, err := opentelemetry.NewTracingInterceptor(opentelemetry.TracerOptions{})

// OpenTracing
tracingInterceptor, err := opentracing.NewInterceptor(opentracing.TracerOptions{})

// Datadog
tracingInterceptor, err := tracing.NewTracingInterceptor(tracing.TracerOptions{})
```

并将其传递给 [客户选项](https://pkg.go.dev/go.temporal.io/sdk/internal#ClientOptions) 进行注册：

```go
c, err := client.Dial(client.Options{
    Interceptors: []interceptor.ClientInterceptor{tracingInterceptor},
})
```

如果您正在构建可重用的库，您还可以通过 [插件](/develop/plugins-guide#interceptors) 注册拦截器。

每个跟踪拦截器都使用其库的本机传播机制将跟踪跨度序列化为 Temporal 标头。例如，OpenTelemetry 使用其 `TextMapPropagator` 和 W3C TraceContext 格式。 SDK 跨越 Workflow、Activity 和 Child Workflow 边界携带这些标头，因此跟踪库可以重建调用图。
有关更多信息，请参阅 [开放式遥测](https://opentelemetry.io/)、[开放式追踪](https://opentracing.io) 和 [数据狗](https://docs.datadoghq.com/tracing/) 的文档。

要构建自定义上下文传播（例如，租户 ID、身份验证令牌），请参阅 [上下文传播](/develop/go/best-practices/context-propagation)。

## 从 Workflow {#logging} 记录
**如何使用 Go SDK 从 Workflow 登录。**

将日志和错误发送到日志记录服务，以便当出现问题时，您可以看到发生了什么。

记录器创建审计跟踪并捕获有关 Workflow 操作的信息。
适当的日志记录级别取决于您的具体需求。
在开发或故障排除期间，您可能会使用调试甚至跟踪。
在生产中，您可以使用 info 或 warn 来避免日志量过多。

记录器支持以下日志记录级别：

| 等级   | 使用                                                                                                       |
| ------- | --------------------------------------------------------------------------------------------------------- |
| `TRACE` | 最详细的日志记录级别，用于非常细粒度的信息。                               |
| `DEBUG` | 详细信息，通常可用于调试目的。                                            |
| `INFO`  | 有关应用程序操作的一般信息。                                                    |
| `WARN`  | 指示不会阻止应用程序运行的潜在有害情况或小问题。 |
| `ERROR` | 指示可能仍允许应用程序继续运行的错误情况。                    |

Temporal SDK 核心通常使用 `WARN` 作为其默认日志记录级别。

在Workflow Definitions中，您可以使用[`Workflow.GetLogger(ctx)`](https://pkg.go.dev/go.temporal.io/sdk/workflow#GetLogger)写入日志。

```go
import (
	"context"
	"time"

	"go.temporal.io/sdk/activity"
	"go.temporal.io/sdk/workflow"
)

// Workflow is a standard workflow definition.
// Note that the Workflow and Activity don't need to care that
// their inputs/results are being compressed.
func Workflow(ctx workflow.Context, name string) (string, error) {
// ...

workflow.WithActivityOptions(ctx, ao)

// Getting the logger from the context.
	logger := workflow.GetLogger(ctx)
// Logging a message with the key value pair `name` and `name`
	logger.Info("Compressed Payloads workflow started", "name", name)

	info := map[string]string{
		"name": name,
	}

	logger.Info("Compressed Payloads workflow completed.", "result", result)

	return result, nil
}
```

<a id="custom-logger"></a>
### 提供自定义记录器 {#custom-logger}
**如何使用 Go SDK 向 Temporal CLIent 提供自定义记录器。**

此字段设置用于 Temporal CLIent 实例的所有日志记录操作的自定义记录器。

尽管 Go SDK 本身不支持大多数第三方日志记录解决方案，但 [我们万岁云的朋友](https://github.com/sagikazarmark) 构建了适配器包 [洛古尔](https://github.com/logur/logur)，这使得以最小的开销使用第三方日志记录器成为可能。
大多数流行的日志记录解决方案在 Logur 中都有现有适配器，但您可以找到完整列表 [在 Logur GitHub 项目中](https://github.com/logur?q=adapter-)。

下面是使用 Logur 支持 [洛鲁斯](https://github.com/sirupsen/logrus) 的示例：

```go
package main
import (
  "go.temporal.io/sdk/client"

	"github.com/sirupsen/logrus"
	logrusadapter "logur.dev/adapter/logrus"
	"logur.dev/logur"
)

func main() {
  // ...
  logger := logur.LoggerToKV(logrusadapter.New(logrus.New()))
  clientOptions := client.Options{
    Logger: logger,
  }
  temporalClient, err := client.Dial(clientOptions)
  // ...
}
```

## Visibility API {#visibility}
Temporal 平台中的术语 Visibility 是指使操作员能够查看当前存在于 Temporal 服务中的 Workflow Execution 的子系统和 API。

### Search Attributes
**如何通过 Go SDK 来使用 Search Attribute。**

检索 Workflow Execution 的典型方法是通过其 Workflow Id。

但是，有时您需要根据另一属性检索一个或多个 Workflow Execution。例如，假设您想要获取某个时间范围内失败的某种类型的所有 Workflow Execution，以便您可以使用相同的参数启动新的 Workflow Execution。

您可以使用 [Search Attributes](/search-attribute) 来完成此操作。

- [默认 Search Attribute](/search-attribute#default-search-attribute) 与 `WorkflowType`、`StartTime` 和 `ExecutionStatus` 一样，会自动添加到 Workflow Execution。
- [定制 Search Attribute](/search-attribute#custom-search-attribute) 可以包含其自己的特定于域的数据（如 `customerId` 或 `numItems`）。

使用自定义 Search Attribute 的步骤是：

- 使用 `Temporal operator search-attribute create` 或云 UI 在 Temporal 服务中创建新的 Search Attribute。
- 设置 Workflow Execution 的 Search Attribute 值：
  - 在客户端上，在启动执行时将其作为选项包含在内。
  - 在 Workflow 中通过调用 `UpsertSearchAttributes`。
- 读取Search Attribute的值：
  - 在客户端上通过调用 `DescribeWorkflow`。
  - 在 Workflow 中查看 `WorkflowInfo`。
- Query Workflow Execution 由 Search Attribute 使用 [列表过滤器](/list-filter) 实现：
  - [在Temporal CLI中](/cli/workflow#list)。
  - 在代码中调用 `ListWorkflowExecutions`。

以下是 Query Workflow Execution 的操作方法：

[列表Workflow()](https://pkg.go.dev/go.temporal.io/sdk/client#Client.ListWorkflow) 函数检索与给定 [列表过滤器](/workflow-execution) 的 [Search Attributes](/search-attribute) 匹配的 [Workflow Executions](/list-filter) 列表。
从 [Visibility](/temporal-service/visibility) 存储返回的元数据可用于从 [坚持](/temporal-service/persistence) 存储获取 Workflow Execution 的历史记录和详细信息。

使用列表过滤器定义要传递到 `ListWorkflow()` 的 `request`。

```go
request := &workflowservice.ListWorkflowExecutionsRequest{ Query: "CloseTime = missing" }
```

此 `request` 值仅返回打开的 Workflow。
有关更多列表过滤器示例，请参阅 [Temporal Visibility 指南中提供了列表过滤器的示例。](/list-filter#list-filter-examples)

```go
resp, err := temporalClient.ListWorkflow(ctx.Background(), request)
if err != nil {
  return err
}

fmt.Println("First page of results:")
for _, exec := range resp.Executions {
  fmt.Printf("Workflow ID %v\n", exec.Execution.WorkflowId)
}
```

<a id="custom-search-attributes"></a>
### 设置自定义 Search Attribute {#custom-search-attributes}
**如何使用 Go SDK 设置自定义 Search Attribute。**

在 Temporal 服务中创建自定义 Search Attribute 后（使用 `Temporal operator search-attribute create` 命令或云 UI），您可以在启动 Workflow 时设置自定义 Search Attribute 的值。

在 [`StartWorkflowOptions.SearchAttributes`](https://pkg.go.dev/go.temporal.io/sdk/internal#StartWorkflowOptions) 中提供键值对。

Search Attribute 表示为 `map[string]interface{}`。
地图中的值必须对应于 [Search Attribute 的值类型](/search-attribute#supported-types)：

- 布尔 = `bool`
- 日期时间 = `time.Time`
- 双 = `float64`
- 整数 = `int64`
- 关键字 = `string`
- 文本 = `string`

如果您有关键字类型的自定义 Search Attribute `CustomerId` 和文本类型的 `MiscData`，您将提供 `string` 值：

```go
func (c *Client) CallYourWorkflow(ctx context.Context, workflowID string, payload map[string]interface{}) error {
    // ...
    searchAttributes := map[string]interface{}{
        "CustomerId": payload["customer"],
        "MiscData": payload["miscData"]
    }
    options := client.StartWorkflowOptions{
        SearchAttributes:   searchAttributes
        // ...
    }
    we, err := c.Client.ExecuteWorkflow(ctx, options, app.YourWorkflow, payload)
    // ...
}
```

<a id="upsert-search-attributes"></a>
### 更新插入 Search Attributes {#upsert-search-attributes}
**如何使用 Go SDK 更新插入 Search Attribute。**

您可以更新插入 Search Attribute 以在 Workflow 代码中添加 Update Search Attribute。

在高级情况下，您可能希望随着 Workflow 的进展动态地 Update 这些属性。
[更新插入搜索属性](https://pkg.go.dev/go.temporal.io/sdk/workflow#UpsertSearchAttributes) 用于在 Workflow 代码中添加或添加 Update Search Attribute。

`UpsertSearchAttributes` 会将属性合并到 Workflow 中的现有地图。
考虑这个示例 Workflow 代码：

```go
func YourWorkflow(ctx workflow.Context, input string) error {

    attr1 := map[string]interface{}{
        "CustomIntField": 1,
        "CustomBoolField": true,
    }
    workflow.UpsertSearchAttributes(ctx, attr1)

    attr2 := map[string]interface{}{
        "CustomIntField": 2,
        "CustomKeywordField": "seattle",
    }
    workflow.UpsertSearchAttributes(ctx, attr2)
}
```

第二次调用 `UpsertSearchAttributes` 后，地图将包含：

```go
map[string]interface{}{
    "CustomIntField": 2, // last update wins
    "CustomBoolField": true,
    "CustomKeywordField": "seattle",
}
```

<a id="remove-search-attribute"></a>
### 从 Workflow 中删除 Search Attribute {#remove-search-attribute}
**如何使用 Go SDK 从 Workflow 中删除 Search Attribute。**

要删除之前设置的 Search Attribute，请将其设置为空数组：`[]`。

**不支持删除字段。**

但是，要实现类似的效果，请将该字段设置为某个占位符值。
例如，您可以将 `CustomKeywordField` 设置为 `impossibleVal`。
然后搜索 `CustomKeywordField != 'impossibleVal'` 将匹配 `CustomKeywordField` 不等于 `impossibleVal` 的 Workflow，其中包括未设置 `CustomKeywordField` 的 Workflow。
