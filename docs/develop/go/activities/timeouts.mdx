<a id="activity-timeouts"></a>
## 如何设置 Activity 超时 {#activity-timeouts}

每个 Activity 超时控制 Activity 执行不同方面的最大持续时间。

Activity 选项中提供以下超时。

- **[Schedule-关闭超时](/encyclopedia/detecting-activity-failures#schedule-to-close-timeout)：** 是整个 [Activity 执行](/activity-execution) 允许的最大时间量。
- **[开始到关闭超时](/encyclopedia/detecting-activity-failures#start-to-close-timeout)：** 是单个 [Activity 任务执行](/tasks#activity-task-execution) 允许的最长时间。
- **[Schedule-启动超时](/encyclopedia/detecting-activity-failures#schedule-to-start-timeout)：** 是从 [Activity 任务](/tasks#activity-task) 为 Scheduled 到 [Worker](/workers#worker) 启动该 Activity 任务所允许的最长时间。

Activity 执行必须设置“开始到关闭”或“Schedule 到关闭超时”。

要在 Go 中设置 Activity 超时，请从 `go.Temporal.io/sdk/Workflow` 包创建 `ActivityOptions` 实例，设置 Activity 超时字段，然后使用 `WithActivityOptions()` API 将选项应用到 `Workflow.Context` 实例。

可用的超时有：

- `StartToCloseTimeout`
- `ScheduleToClose`
- `ScheduleToStartTimeout`

```go
activityoptions := workflow.ActivityOptions{
  // Set Activity Timeout duration
  ScheduleToCloseTimeout: 10 * time.Second,
  // StartToCloseTimeout: 10 * time.Second,
  // ScheduleToStartTimeout: 10 * time.Second,
}
ctx = workflow.WithActivityOptions(ctx, activityoptions)
var yourActivityResult YourActivityResult
err = workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam).Get(ctx, &yourActivityResult)
if err != nil {
  // ...
}
```

<a id="activity-retries"></a>
### 设置自定义 Activity Retry Policy {#activity-retries}

Retry Policy 与超时配合提供精细控制以优化执行体验。

Activity 如果未提供自定义执行，则执行会自动与默认 [Retry Policy](/encyclopedia/retry-policies) 关联。

要设置 [重试策略](/encyclopedia/retry-policies)，请从 `go.Temporal.io/sdk/Workflow` 包创建 `ActivityOptions` 实例，设置 `RetryPolicy` 字段，然后使用 `WithActivityOptions()` API 将选项应用到 `Workflow.Context` 实例。

- 型号：[`RetryPolicy`](https://pkg.go.dev/go.temporal.io/sdk/temporal#RetryPolicy)
- 默认：

```go
retrypolicy := &temporal.RetryPolicy{
  InitialInterval:    time.Second,
  BackoffCoefficient: 2.0,
  MaximumInterval:    time.Second * 100, // 100 * InitialInterval
  MaximumAttempts: 0, // Unlimited
  NonRetryableErrorTypes: []string, // empty
}
```

此处提供 Retry Policy 是一种自定义，并覆盖各个字段默认值。

```go
retrypolicy := &temporal.RetryPolicy{
  InitialInterval:    time.Second,
  BackoffCoefficient: 2.0,
  MaximumInterval:    time.Second * 100,
}

activityoptions := workflow.ActivityOptions{
  RetryPolicy: retrypolicy,
}
ctx = workflow.WithActivityOptions(ctx, activityoptions)
var yourActivityResult YourActivityResult
err = workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam).Get(ctx, &yourActivityResult)
if err != nil {
  // ...
}
```

### 使用下次重试延迟覆盖重试间隔 {#next-retry-delay}

您可以返回设置了 `NextRetryDelay` 字段的 [应用失败](/references/failures#application-failure)。
该值将替换并覆盖 Retry Policy 上的任何重试间隔。

例如，如果在 Activity 中，您希望间隔基于尝试次数：

```go
attempt := activity.GetInfo(ctx).Attempt;

return temporal.NewApplicationErrorWithOptions(fmt.Sprintf("Something bad happened on attempt %d", attempt), "NextDelay", temporal.ApplicationErrorOptions{
  NextRetryDelay: 3 * time.Second * delay,
})
```

<a id="activity-heartbeats"></a>
## Activity 心跳 {#activity-heartbeats}

[Activity 心跳](/encyclopedia/detecting-activity-failures#activity-heartbeat) 是从正在执行 Activity 的 [Worker 流程](/workers#worker-process) 到 [Temporal 服务](/temporal-service) 的 ping。
每个心跳都会通知 Temporal 服务 [Activity 执行](/activity-execution) 正在取得进展并且 Worker 尚未崩溃。
如果Temporal服务在[心跳超时](/encyclopedia/detecting-activity-failures#heartbeat-timeout)时间段内没有收到Heartbeat，则Activity将被视为失败，并且根据Retry Policy，另一个[Activity 任务执行](/tasks#activity-task-execution)可能是Scheduled。

心跳可能并不总是发送到 Temporal 服务 — 它们可能是由 Worker 发送的 [节流](/encyclopedia/detecting-activity-failures#throttling)。

Activity 取消会在活动检测到时从 Temporal 服务传递到活动。没有心跳的活动无法收到取消。
心跳限制可能会导致取消的发送时间晚于预期。

心跳可以包含描述 Activity 当前进度的 `details` 字段。
如果重试 Activity，则 Activity 可以从发送到 Temporal 服务的最后一个心跳访问 `details`。

对于 Go 中 Activity 中的 [心跳](/encyclopedia/detecting-activity-failures#activity-heartbeat)，请使用 `RecordHeartbeat` API。

```go
import (
    // ...
    "go.temporal.io/sdk/workflow"
    // ...
)

func YourActivityDefinition(ctx, YourActivityDefinitionParam) (YourActivityDefinitionResult, error) {
    // ...
    activity.RecordHeartbeat(ctx, details)
    // ...
}
```

当 Activity 任务执行由于错过心跳而超时时，上述 `details` 变量的最后一个值将返回到 `TimeoutError` 的 `details` 字段中的调用 Workflow，并将 `TimeoutType` 设置为 `Heartbeat`。

您还可以从外部源对 Activity 进行心跳检测：

```go
// The client is a heavyweight object that should be created once per process.
temporalClient, err := client.Dial(client.Options{})
// Record heartbeat.
err := temporalClient.RecordActivityHeartbeat(ctx, taskToken, details)
```

`RecordActivityHeartbeat`函数的参数为​​：

- `taskToken`：内部检索的 `ActivityInfo` 结构的二进制 `TaskToken` 字段的值
  Activity。
- `details`：包含进度信息的可序列化有效负载。

如果 Activity 执行在失败之前检测其进度，则重试尝试将有权访问进度信息，以便 Activity 执行可以从失败状态恢复。
以下是如何实现这一点的示例：

```go
func SampleActivity(ctx context.Context, inputArg InputParams) error {
    startIdx := inputArg.StartIndex
    if activity.HasHeartbeatDetails(ctx) {
        // Recover from finished progress.
        var finishedIndex int
        if err := activity.GetHeartbeatDetails(ctx, &finishedIndex); err == nil {
            startIdx = finishedIndex + 1 // Start from next one.
        }
    }

    // Normal Activity logic...
    for i:=startIdx; i<inputArg.EndIdx; i++ {
        // Code for processing item i goes here...
        activity.RecordHeartbeat(ctx, i) // Report progress.
    }
}
```

### 设置心跳超时 {#heartbeat-timeout}

[心跳超时](/encyclopedia/detecting-activity-failures#heartbeat-timeout) 与 [Activity 心跳](/encyclopedia/detecting-activity-failures#activity-heartbeat) 配合使用。

要设置 [心跳超时](/encyclopedia/detecting-activity-failures#heartbeat-timeout)，请从 `go.Temporal.io/sdk/Workflow` 包创建 `ActivityOptions` 实例，设置 `HeartbeatTimeout` 字段，然后使用 `WithActivityOptions()` API 将选项应用到 `Workflow.Context` 实例。

```go
activityoptions := workflow.ActivityOptions{
  HeartbeatTimeout: 10 * time.Second,
}
ctx = workflow.WithActivityOptions(ctx, activityoptions)
var yourActivityResult YourActivityResult
err = workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam).Get(ctx, &yourActivityResult)
if err != nil {
  // ...
}
```
