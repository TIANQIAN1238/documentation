<a id="activity-execution"></a>
## 如何启动 Activity 执行 {#activity-execution}

对生成 [Activity 执行](/activity-execution) 的调用写入 [Workflow Definition](/workflow-definition) 中。
调用生成 Activity 执行会生成 [ScheduleActivity任务](/references/commands#scheduleactivitytask) 命令。
这会在 Workflow Execution Event History 中产生一组三个 [Activity 任务](/tasks#activity-task) 相关事件（[Activity任务Scheduled](/references/events#activitytaskscheduled)、[Activity任务已开始](/references/events#activitytaskstarted) 和 ActivityTask[Closed]）。

活动实现的单个实例在多个同时的 Activity 调用之间共享。
Activity 实现代码应该是_幂等的_。

通过调用参数传递给活动的值或通过结果值返回的值都记录在执行历史记录中。
当 Workflow 状态需要恢复时，整个执行历史记录将从 Temporal 服务传输到 Workflow Worker。
因此，大量执行历史记录可能会对 Workflow 的性能产生不利影响。

因此，请注意通过 Activity 调用参数或返回值传输的数据量。
否则，Activity 实现不存在其他限制。

要生成 [Activity 执行](/activity-execution)，请在 Workflow Definition 中调用 [`ExecuteActivity()`](https://pkg.go.dev/go.temporal.io/sdk/workflow#ExecuteActivity)。
    该 API 可从 [`go.Temporal.io/sdk/Workflow`](https://pkg.go.dev/go.temporal.io/sdk/workflow) 包中获取。
    `ExecuteActivity()` API 调用需要 `Workflow.Context` 的实例、Activity 函数名称以及要传递给 Activity 执行的任何变量。
        Activity 函数名称可以作为变量对象（无引号）或字符串提供。
        传递实际函数对象的好处是框架可以根据 Activity Definition 验证参数。
        `ExecuteActivity` 调用返回一个 Future，可用于获取 Activity 执行的结果。

```go
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) (*YourWorkflowResultObject, error) {
    // Set the options for the Activity Execution.
    // Either StartToClose Timeout OR ScheduleToClose is required.
    // Not specifying a Task Queue will default to the parent Workflow Task Queue.
    activityOptions := workflow.ActivityOptions{
        StartToCloseTimeout: 10 * time.Second,
    }
    ctx = workflow.WithActivityOptions(ctx, activityOptions)
    activityParam := YourActivityParam{
        ActivityParamX: param.WorkflowParamX,
        ActivityParamY: param.WorkflowParamY,
    }
    // Use a nil struct pointer to call Activities that are part of a struct.
    var a *YourActivityObject
    // Execute the Activity and wait for the result.
    var activityResult YourActivityResultObject
    err := workflow.ExecuteActivity(ctx, a.YourActivityDefinition, activityParam).Get(ctx, &activityResult)
    if err != nil {
        return nil, err
    }
// ...
}
```

### 如何设置所需的 Activity 超时 {#required-timeout}

Activity 执行语义依赖于多个参数。
唯一需要设置的值是 [Schedule-关闭超时](/encyclopedia/detecting-activity-failures#schedule-to-close-timeout) 或 [开始到关闭超时](/encyclopedia/detecting-activity-failures#start-to-close-timeout)。
这些值在 Activity 选项中设置。

要在 Go 中设置 Activity 超时，请从 `go.Temporal.io/sdk/Workflow` 包创建 `ActivityOptions` 实例，设置 Activity 超时字段，然后使用 `WithActivityOptions()` API 将选项应用到 `Workflow.Context` 实例。

可用的超时有：

- `StartToCloseTimeout`
- `ScheduleToClose`
- `ScheduleToStartTimeout`

```go
activityOptions := workflow.ActivityOptions{
  // Set Activity Timeout duration
  ScheduleToCloseTimeout: 10 * time.Second,
  // StartToCloseTimeout: 10 * time.Second,
  // ScheduleToStartTimeout: 10 * time.Second,
}
ctx = workflow.WithActivityOptions(ctx, activityOptions)
var yourActivityResult YourActivityResult
err = workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam).Get(ctx, &yourActivityResult)
if err != nil {
  // ...
}
```

### 转到 ActivityOptions 参考 {#activity-options-reference}

从 `go.Temporal.io/sdk/Workflow` 包创建 [`ActivityOptions`](https://pkg.go.dev/go.temporal.io/sdk/workflow#ActivityOptions) 的实例，并使用 [`WithActivityOptions()`](https://pkg.go.dev/go.temporal.io/sdk/workflow#WithActivityOptions) 将其应用到 `Workflow.Context` 的实例。

然后，`Workflow.Context` 的实例将传递给 `ExecuteActivity()` 调用。

| 场地                                               | 必需的                          | 类型                                                                        |
| --------------------------------------------------- | --------------------------------- | --------------------------------------------------------------------------- |
| [`ActivityID`](#activityid)                         | 不                                | `string`                                                                    |
| [`TaskQueueName`](#taskqueuename)                   | 不                                | `string`                                                                    |
| [`ScheduleToCloseTimeout`](#scheduletoclosetimeout) | 是（或 `StartToCloseTimeout`）    | `time.Duration`                                                             |
| [`ScheduleToStartTimeout`](#scheduletostarttimeout) | 不                                | `time.Duration`                                                             |
| [`StartToCloseTimeout`](#scheduletoclosetimeout)    | 是（或 `ScheduleToCloseTimeout`） | `time.Duration`                                                             |
| [`HeartbeatTimeout`](#heartbeattimeout)             | 不                                | `time.Duration`                                                             |
| [`WaitForCancellation`](#waitforcancellation)       | 不                                | `bool`                                                                      |
| [`OriginalTaskQueueName`](#originaltaskqueuename)   | 不                                | `string`                                                                    |
| [`RetryPolicy`](#retrypolicy)                       | 不                                | [`RetryPolicy`](https://pkg.go.dev/go.temporal.io/sdk/temporal#RetryPolicy) |

<a id="retrypolicy"></a>
#### 重试策略 {#retrypolicy}

要设置 [重试策略](/encyclopedia/retry-policies)，请从 `go.Temporal.io/sdk/Workflow` 包创建 `ActivityOptions` 实例，设置 `RetryPolicy` 字段，然后使用 `WithActivityOptions()` API 将选项应用到 `Workflow.Context` 实例。

- 型号：[`RetryPolicy`](https://pkg.go.dev/go.temporal.io/sdk/temporal#RetryPolicy)
- 默认：

```go
retryPolicy := &temporal.RetryPolicy{
  InitialInterval:    time.Second,
  BackoffCoefficient: 2.0,
  MaximumInterval:    time.Second * 100, // 100 * InitialInterval
  MaximumAttempts:    0, // Unlimited
  NonRetryableErrorTypes: []string, // empty
}
```

此处提供 Retry Policy 是覆盖各个字段默认值的自定义。

```go
retryPolicy := &temporal.RetryPolicy{
  InitialInterval:    time.Second,
  BackoffCoefficient: 2.0,
  MaximumInterval:    time.Second * 100,
}

activityOptions := workflow.ActivityOptions{
  RetryPolicy: retryPolicy,
}
ctx = workflow.WithActivityOptions(ctx, activityOptions)
var yourActivityResult YourActivityResult
err = workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam).Get(ctx, &yourActivityResult)
if err != nil {
  // ...
}
```

<a id="get-activity-results"></a>
### 如何获取 Activity 执行的结果 {#get-activity-results}

调用生成 [Activity 执行](/activity-execution) 会生成 [ScheduleActivity任务](/references/commands#scheduleactivitytask) 命令，并为 Workflow 提供 Awaitable。
Workflow Execution 可以阻止进度，直到结果通过 Awaitable 可用为止，或者继续进度，在结果可用时使用它。

`ExecuteActivity` API 调用返回 [`Workflow.Future`](https://pkg.go.dev/go.temporal.io/sdk/workflow#Futures) 的实例，该实例具有以下两个方法：

- `Get()`：采用传递给 Activity 执行的 `Workflow.Context` 实例，以及一个指针作为参数。
  与指针关联的变量由 Activity 执行结果填充。
  此调用会阻塞，直到结果可用为止。
- `IsReady()`：当 Activity 执行的结果准备就绪时，返回 `true`。

在 `Workflow.Future` 实例上调用 `Get()` 方法以获取 Activity 执行的结果。
结果参数的类型必须与 Activity 函数声明的返回值的类型匹配。

```go
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) (YourWorkflowResponse, error) {
 // ...
 future := workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam)
 var yourActivityResult YourActivityResult
 if err := future.Get(ctx, &yourActivityResult); err != nil {
   // ...
 }
 // ...
}
```

首先使用 `IsReady()` 方法确保 `Get()` 调用不会导致 Workflow Execution 等待结果。

```go
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) (YourWorkflowResponse, error) {
 // ...
 future := workflow.ExecuteActivity(ctx, YourActivityDefinition, yourActivityParam)
 // ...
 if(future.IsReady()) {
   var yourActivityResult YourActivityResult
   if err := future.Get(ctx, &yourActivityResult); err != nil {
     // ...
   }
 }
 // ...
}
```

从 Workflow 中调用多个 Activity 执行是惯用的做法。
因此，通常会阻止 Activity 执行的结果，或者继续执行其他逻辑，稍后检查 Activity 执行结果。
