[Temporal CLIent](/encyclopedia/temporal-sdks#temporal-client) 使您能够与
[Temporal 服务](/temporal-service)。与 Temporal 服务的通信允许您执行启动等操作
Workflow Executions，发送Signals到Workflow Executions，发送查询到Workflow Executions，得到结果
Workflow Execution，并提供 Activity 任务令牌。

本页向您展示如何使用 Go SDK 和 Temporal CLIent 执行以下操作：

- [连接到本地开发 Temporal 服务](#connect-to-development-service)
- [连接到 Temporal Cloud](#connect-to-temporal-cloud)
- [启动 Workflow Execution](#start-workflow-execution)
- [获取 Workflow 结果](#get-workflow-results)

:::caution

Temporal CLIent 无法在 Workflow 内部初始化和使用。然而，使用
Temporal CLIent 位于 Activity 内部，与 Temporal 服务进行通信。

:::

## 连接到开发 Temporal 服务 {#connect-to-development-service}

使用 [`Dial()`](https://pkg.go.dev/go.temporal.io/sdk/client#Dial) API 中提供的
[`go.Temporal.io/sdk/client`](https://pkg.go.dev/go.temporal.io/sdk/client) 包创建
[`Client`](https://pkg.go.dev/go.temporal.io/sdk/client#Client)。 `Dial()` API 需要连接选项，例如
Temporal Server 地址、要连接的 Namespace 以及传输层安全 (TLS) 配置。您可以指定
这些选项可以在函数调用中使用，或者使用环境变量或配置文件来指定它们。我们推荐
您可以使用环境变量或配置文件来安全地管理这些连接选项。

:::info Versioning Requirements

Go SDK v1.28.0 中添加了环境变量和配置文件支持。

:::

当您在本地运行 Temporal 服务时，例如
[Temporal CLI](https://docs.temporal.io/cli/server#start-dev)，您必须提供的连接选项很少。

如果您不提供 [`HostPort`](https://pkg.go.dev/go.temporal.io/sdk/internal#ClientOptions)，则客户端默认
地址和端口号为`127.0.0.1:7233`，这是开发Temporal服务的端口。如果你没有设置
在 Namespace 字段中自定义 Namespace 名称，客户端连接到默认的 Namespace。

您可以使用 TOML 配置文件来设置 Temporal CLIent 的连接选项。配置文件可以让你

您在配置文件和环境变量中设置相同的选项，即环境变量值
覆盖配置文件中设置的选项。

例如，以下 TOML 配置文件定义了两个配置文件：`default` 和 `prod`。每个配置文件都有自己的
连接选项集。

```toml
# Default profile for local development
[profile.default]
address = "localhost:7233"
namespace = "default"

# Custom gRPC headers
[profile.default.grpc_meta]
my-custom-header = "development-value"
trace-id = "dev-trace-123"

# Production profile for Temporal Cloud
[profile.prod]
address = "your-namespace.a1b2c.tmprl.cloud:7233"
namespace = "your-namespace"
api_key = "your-api-key-here"

# TLS configuration for production
[profile.prod.tls]
# TLS is auto-enabled when this TLS config or API key is present, but you can configure it explicitly
# disabled = false
# Use certificate files for mTLS
client_cert_path = "/etc/temporal/certs/client.pem"
client_key_path = "/etc/temporal/certs/client.key"

# Custom headers for production
[profile.prod.grpc_meta]
environment = "production"
service-version = "v1.2.3"
```

您可以使用配置文件中的特定配置文件创建 Temporal CLIent，如下所示：

```go
package main

import (
	"fmt"
	"log"

	"go.temporal.io/sdk/client"
	"go.temporal.io/sdk/contrib/envconfig"
)

func main() {
	// Load a specific profile from the TOML config file.
	// This requires a [profile.prod] section in your config.
	opts, err := envconfig.LoadClientOptions(envconfig.LoadClientOptionsRequest{
		ConfigFileProfile: "prod",
	})
	if err != nil {
		log.Fatalf("Failed to load 'prod' profile: %v", err)
	}

	c, err := client.Dial(opts)
	if err != nil {
		log.Fatalf("Failed to connect using 'prod' profile: %v", err)
	}
	defer c.Close()

	fmt.Printf("✅ Connected to Temporal namespace %q on %s using 'prod' profile\n", c.Options().Namespace, c.Options().HostPort)
}
```

## 连接到 Temporal Cloud {#connect-to-temporal-cloud}

您可以使用 [API密钥](/cloud/api-keys) 或通过 mTLS 连接到 Temporal Cloud。连接至 Temporal
与连接到不安全的云或任何安全的 Temporal 服务相比，云或任何安全的 Temporal 服务需要额外的连接选项
本地开发实例：

- 您的身份验证凭据。
  - 如果您使用 API 密钥，请提供 API 密钥值。
  - 如果您使用 mTLS，请提供 mTLS CA 证书和 mTLS 私钥。
- 您的 _Namespace 和帐户 ID_ 组合，格式为 `<Namespace_id>.<account_id>`。
- 建议的_端点_是 gRPC Namespace 端点：`<Namespace>.<account>.tmprl.cloud:7233`。
  此端点适用于所有 Namespace，并自动将流量定向到具有 [高可用性](/cloud/high-availability) 的 Namespace 的活动区域。
  有关端点选项的更多信息，请参阅 [访问 Namespace](/cloud/namespaces#access-namespaces)。

您可以在 Namespaces 选项卡上找到 Namespace 和账户 ID 以及终端节点：

![左侧为 Namespace 和账户 ID 组合，右侧为区域终端节点](/img/cloud/apikeys/namespaces-and-regional-endpoints.png)

有关管理和生成 Temporal Cloud 客户端证书的更多信息，请参阅
[如何在Temporal Cloud中管理证书](/cloud/certificates)。

您可以使用环境变量、配置文件或直接在代码中提供这些连接选项。

例如，以下 TOML 配置文件定义了 `cloud` 配置文件，其中包含必要的连接选项
通过 API 密钥连接到 Temporal Cloud：

```toml
# Cloud profile for Temporal Cloud
[profile.cloud]
address = "your-namespace.a1b2c.tmprl.cloud:7233"
namespace = "your-namespace"
api_key = "your-api-key-here"
```

如果您想使用 mTLS 身份验证而不是 API 密钥，请将 `api_key` 字段替换为您的 mTLS 证书，然后
私钥：

```toml
# Cloud profile for Temporal Cloud
[profile.cloud]
address = "your-namespace.a1b2c.tmprl.cloud:7233"
namespace = "your-namespace"
tls_client_cert_data = "your-tls-client-cert-data"
tls_client_key_path = "your-tls-client-key-path"
```

使用配置文件中定义的连接选项，使用 `envconfig` 中的 `LoadClientOptions` 函数
使用 `cloud` 配置文件创建 Temporal CLIent 的软件包，如下所示：

```go {13-17}
package main

import (
	"fmt"
	"log"

	"go.temporal.io/sdk/client"
	"go.temporal.io/sdk/contrib/envconfig"
)

func main() {
	// Replace with the actual path to your TOML file.
	configFilePath := "/Users/yourname/.config/my-app/temporal.toml"

	opts, err := envconfig.LoadClientOptions(envconfig.LoadClientOptionsRequest{
    ConfigFilePath: configFilePath,
    ConfigFileProfile: "cloud",
	})
	if err != nil {
		log.Fatalf("Failed to load client config from custom file: %v", err)
	}

	c, err := client.Dial(opts)
	if err != nil {
		log.Fatalf("Failed to connect using custom config file: %v", err)
	}
	defer c.Close()

	fmt.Printf("✅ Connected using custom config at: %s\n", configFilePath)
}
```

## 启动 Workflow Execution {#start-workflow-execution}

**如何使用 Go SDK 启动 Workflow Execution**

[Workflow Execution](/workflow-execution) 语义依赖于几个参数，即启动 Workflow Execution
您必须提供将用于任务的 Task Queue（Worker 正在轮询的任务）、Workflow 类型、
特定于语言的上下文数据和 Workflow 函数参数。

在下面的示例中，所有 Workflow Execution 均使用 Temporal CLIent 启动。从中生成 Workflow Execution
在另一个 Workflow Execution 中，使用 [Child Workflow](/develop/go/workflows/child-workflows) 或外部 Workflow
API。

请参阅 [定制Workflow型](/develop/go/workflows/basics#customize-workflow-type) 部分以了解如何自定义
Workflow 类型的名称。

生成 Workflow Execution 的请求会导致 Temporal 服务创建第一个事件
([Workflow执行开始](/references/events#workflowexecutionstarted)) 在 Workflow Execution Event History 中。的
然后，Temporal 服务创建第一个 Workflow 任务，从而产生第一个
[Workflow任务Scheduled](/references/events#workflowtaskscheduled) 事件。

要生成 [Workflow Execution](/workflow-execution)，请在 Go SDK 上使用 `ExecuteWorkflow()` 方法
[`Client`](https://pkg.go.dev/go.temporal.io/sdk/client#Client)。

`ExecuteWorkflow()` API 调用需要 [`context.Context`](https://pkg.go.dev/context#Context) 的实例，
[`StartWorkflowOptions`](https://pkg.go.dev/go.temporal.io/sdk/client#StartWorkflowOptions) 的实例，Workflow 类型
名称以及要传递给 Workflow Execution 的所有变量。 `ExecuteWorkflow()` 调用返回一个 Future，它可以
用于获取 Workflow Execution 的结果。

```go
package main

import (
  // ...

  "go.temporal.io/sdk/client"
)

func main() {
  temporalClient, err := client.Dial(client.Options{})
  if err != nil {
    // ...
  }
  defer temporalClient.Close()
  // ...
  workflowOptions := client.StartWorkflowOptions{
    // ...
  }
  workflowRun, err := temporalClient.ExecuteWorkflow(context.Background(), workflowOptions, YourWorkflowDefinition, param)
  if err != nil {
    // ...
  }
  // ...
}

func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) (YourWorkflowResponse, error) {
  // ...
}
```

如果调用过程可以直接访问该函数，则可以传递 Workflow 类型名称参数，就好像
函数名称是一个变量，不带引号。

```go
workflowRun, err := temporalClient.ExecuteWorkflow(context.Background(), workflowOptions, YourWorkflowDefinition, param)
```

如果调用进程无法直接访问静态定义的 Workflow Definition，例如，如果
Workflow Definition 位于不可导入的包中，或者它是用完全不同的语言编写的，那么
Workflow 类型可作为 `string` 提供。

```go
workflowRun, err := c.ExecuteWorkflow(context.Background(), workflowOptions, "YourWorkflowDefinition", param)
```

### 设置 Workflow Task Queue {#set-task-queue}

**如何使用 Go SDK 设置 Workflow 的 Task Queue**

在大多数 SDK 中，唯一必须设置的 Workflow 选项是 [Task Queue](/task-queue) 的名称。

对于要执行的任何代码，必须运行包含正在轮询同一任务的 Worker 实体的 Worker 进程
队列名称。

创建一个实例
[`StartWorkflowOptions`](https://pkg.go.dev/go.temporal.io/sdk@v1.10.0/client#StartWorkflowOptions) 来自
`go.Temporal.io/sdk/client` 包，设置 `TaskQueue` 字段，并将实例传递给 `ExecuteWorkflow` 调用。

- 型号：`string`
- 默认：无，这是开发人员设置的必填字段

```go
workflowOptions := client.StartWorkflowOptions{
  // ...
  TaskQueue: "your-task-queue",
  // ...
}
workflowRun, err := c.ExecuteWorkflow(context.Background(), workflowOptions, YourWorkflowDefinition)
if err != nil {
  // ...
}
```

**获取最后的完成结果**

对于 [Temporal 计划任务](/cron-job)，您可能需要获取之前 Workflow 运行的结果并使用
它在当前的 Workflow 中运行。

为此，请使用 [`HasLastCompletionResult`](https://pkg.go.dev/go.temporal.io/sdk/workflow#HasLastCompletionResult)
和 [`GetLastCompletionResult`](https://pkg.go.dev/go.temporal.io/sdk/workflow#GetLastCompletionResult) API，可用
从 [`go.Temporal.io/sdk/Workflow`](https://pkg.go.dev/go.temporal.io/sdk/workflow) 包中，直接在您的
Workflow 代码。

```go
type CronResult struct {
 Count int
}

func YourCronWorkflowDefinition(ctx workflow.Context) (CronResult, error) {
 count := 1

 if workflow.HasLastCompletionResult(ctx) {
   var lastResult CronResult
   if err := workflow.GetLastCompletionResult(ctx, &lastResult); err == nil {
     count = count + lastResult.Count
   }
 }

 newResult := CronResult {
   Count: count,
 }
 return newResult, nil
}
```

即使 cron Workflow 运行之一失败，这也将起作用。下一次 Workflow 运行将获取上一次的结果
成功完成 Workflow 运行。
