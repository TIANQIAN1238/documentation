## 启动启用了 Nexus 的 Temporal 开发 Server {#run-the-temporal-nexus-development-server}

前置条件：

- [安装最新的 Temporal CLI](/develop/run-a-development-server)（推荐 v1.3.0 或更高版本）
- [安装最新的 Temporal Go SDK](/develop/go/set-up-your-local-go)（推荐 v1.33.0 或更高版本）

使用 Temporal Nexus 的第一步是启动启用了 Nexus 的 Temporal Server。

```
temporal server start-dev
```

此命令会自动启动带有 Web UI 的 Temporal 开发 Server，并创建 `default` Namespace。它使用内存数据库，因此不要在实际使用场景中使用。

现在应可以通过 [http://localhost:8233](http://localhost:8233) 访问 Temporal Web UI，Temporal Server 也可以在 `localhost:7233` 上接受客户端连接。

## 创建 caller 和 handler Namespace {#create-caller-handler-namespaces}

在设置 Nexus Endpoint 之前，为 caller 和 handler 分别创建独立的 Namespace。

```
temporal operator namespace create --namespace my-target-namespace
temporal operator namespace create --namespace my-caller-namespace
```

`my-target-namespace` 将包含 Nexus Operation handler，我们将在 `my-caller-namespace` 中使用 Workflow 来调用该 Operation handler。我们使用不同的 Namespace 来演示跨 Namespace 的 Nexus 调用。

## 创建 Nexus Endpoint 将请求从 caller 路由到 handler {#create-nexus-endpoint}

建立 caller 和 handler Namespace 后，下一步是创建 Nexus Endpoint 来路由请求。

```
temporal operator nexus endpoint create \
  --name my-nexus-endpoint-name \
  --target-namespace my-target-namespace \
  --target-task-queue my-handler-task-queue
```

也可以使用 Web UI 创建 Namespace 和 Nexus Endpoint。

## 开发 Nexus Service 和 Operation handler {#develop-nexus-service-operation-handlers}

Nexus Operation handler 通常与其所抽象的底层 Temporal 原语定义在同一 Worker 中。Operation handler 可以决定给定的 Nexus Operation 是同步还是异步的。它们可以使用 Temporal SDK Client 调用底层 Temporal 原语（如 Query、Signal 或 Update），或运行其他可靠代码。Handler 应当可靠，因为[熔断器](/nexus/operations#circuit-breaking)在连续发生 5 次可重试错误后会触发，阻止所有来自 caller 到该 Endpoint 的 Operation。

`temporalnexus` 包提供了创建 Nexus Operation 和其他帮助编写 Operation handler 的构建器：

- `NewWorkflowRunOperation` \- 将 Workflow 作为异步 Nexus Operation 运行
- `GetClient` \- 获取 Worker 初始化时使用的 Temporal Client，用于由 Temporal 原语（如 Signal 和 Query）支持的同步 handler

本教程从使用 `nexus.NewSyncOperation` 方法的同步 Operation handler 示例开始，然后展示如何创建使用 `NewWorkflowRunOperation` 从 Nexus Operation 启动 handler Workflow 的异步 Operation handler。

### 开发同步 Nexus Operation handler

`nexus.NewSyncOperation` 构建函数用于暴露简单的 RPC handler。使用 `temporalnexus.GetClient(ctx)` 获取 Temporal Client，用于 Signal、Query 和列出 Workflow。实现也可以进行其他调用，但 handler 应当可靠，以避免触发[熔断器](/nexus/operations#circuit-breaking)。

<!--SNIPSTART samples-go-nexus-handler {"selectedLines": ["2-23"]}-->
[nexus/handler/app.go](https://github.com/temporalio/samples-go/blob/main/nexus/handler/app.go)
```go
// ...

import (
	"context"
	"fmt"

	"github.com/nexus-rpc/sdk-go/nexus"

	"go.temporal.io/sdk/client"
	"go.temporal.io/sdk/temporalnexus"
	"go.temporal.io/sdk/workflow"

	"github.com/temporalio/samples-go/nexus/service"
)

// NewSyncOperation is a meant for exposing simple RPC handlers.
var EchoOperation = nexus.NewSyncOperation(service.EchoOperationName, func(ctx context.Context, input service.EchoInput, options nexus.StartOperationOptions) (service.EchoOutput, error) {
	// Use temporalnexus.GetClient to get the client that the worker was initialized with to perform client calls
	// such as signaling, querying, and listing workflows. Implementations are free to make arbitrary calls to other
	// services or databases, or perform simple computations such as this one.
	return service.EchoOutput(input), nil
})

```
<!--SNIPEND-->

### 在同步 handler 中使用 Temporal Client 处理 Signal、Query 和 Update

在同步 handler 中使用 Temporal Client 来对 Workflow 进行 Signal、Query 或 Update 是一种常见模式。你也可以使用 Signal-With-Start 或 Update-With-Start 来确保 Workflow 已启动并向其发送 Signal 或 Update。所有调用必须在 [Nexus 请求超时](/cloud/limits#nexus-operation-request-timeout)内完成。传递给 handler 的 ctx 会自动设置此截止时间，因此直接将其传递给 Temporal Client 调用可以正确传播超时。Update 应是短暂的，以保持在此截止时间内。

### 开发用于启动 Workflow 的异步 Nexus Operation handler

使用 `NewWorkflowRunOperation` 构造函数，这是将 Workflow 作为 Operation 暴露的最简便方式。更多替代方案请参阅[此处](https://pkg.go.dev/go.temporal.io/sdk/temporalnexus)。

<!--SNIPSTART samples-go-nexus-handler {"selectedLines": ["26-35"]}-->
[nexus/handler/app.go](https://github.com/temporalio/samples-go/blob/main/nexus/handler/app.go)
```go
// ...
var HelloOperation = temporalnexus.NewWorkflowRunOperation(service.HelloOperationName, HelloHandlerWorkflow, func(ctx context.Context, input service.HelloInput, options nexus.StartOperationOptions) (client.StartWorkflowOptions, error) {
	return client.StartWorkflowOptions{
		// Workflow IDs should typically be business meaningful IDs and are used to dedupe workflow starts.
		// For this example, we're using the request ID allocated by Temporal when the caller workflow schedules
		// the operation, this ID is guaranteed to be stable across retries of this operation.
		ID: options.RequestID,
		// Task queue defaults to the task queue this operation is handled on.
	}, nil
})

```
<!--SNIPEND-->

Workflow ID 通常应是具有业务意义的 ID，用于对 Workflow 启动进行去重。对于 `HelloOperation`，`input.ID` 作为 Nexus Service 契约的一部分传递。

:::tip RESOURCES

使用 Use-Existing 的冲突策略[将多个 Nexus caller 附加到 handler Workflow](/nexus/operations#attaching-multiple-nexus-callers)。

:::

#### 将 Nexus Operation 输入映射到多个 Workflow 参数

Nexus Operation 只能接受一个输入参数。如果想让 Nexus Operation 启动一个接受多个参数的 Workflow，请使用 `NewWorkflowRunOperationWithOptions` 或 `MustNewWorkflowRunOperationWithOptions`。

<!--SNIPSTART samples-go-nexus-handler-multiargs-->
[nexus-multiple-arguments/handler/app.go](https://github.com/temporalio/samples-go/blob/main/nexus-multiple-arguments/handler/app.go)
```go
var HelloOperation = temporalnexus.MustNewWorkflowRunOperationWithOptions(temporalnexus.WorkflowRunOperationOptions[service.HelloInput, service.HelloOutput]{
	Name: service.HelloOperationName,
	Handler: func(ctx context.Context, input service.HelloInput, options nexus.StartOperationOptions) (temporalnexus.WorkflowHandle[service.HelloOutput], error) {
		return temporalnexus.ExecuteUntypedWorkflow[service.HelloOutput](
			ctx,
			options,
			client.StartWorkflowOptions{
				// Workflow IDs should typically be business meaningful IDs and are used to dedupe workflow starts.
				// For this example, we're using the request ID allocated by Temporal when the caller workflow schedules
				// the operation, this ID is guaranteed to be stable across retries of this operation.
				ID: options.RequestID,
			},
			HelloHandlerWorkflow,
			input.Name,
			input.Language,
		)
	},
})

```

### 在 Worker 中注册 Nexus Service

开发好异步 Nexus Operation handler 以启动 Workflow 后，下一步是在 Worker 中注册 Nexus Service。

<!--SNIPSTART samples-go-nexus-handler-worker-->
[nexus/handler/worker/main.go](https://github.com/temporalio/samples-go/blob/main/nexus/handler/worker/main.go)
```go
package main

import (
	"log"
	"os"

	"go.temporal.io/sdk/client"
	"go.temporal.io/sdk/worker"

	"github.com/nexus-rpc/sdk-go/nexus"
	"github.com/temporalio/samples-go/nexus/handler"
	"github.com/temporalio/samples-go/nexus/options"
	"github.com/temporalio/samples-go/nexus/service"
)

const (
	taskQueue = "my-handler-task-queue"
)

func main() {
	// The client and worker are heavyweight objects that should be created once per process.
	clientOptions, err := options.ParseClientOptionFlags(os.Args[1:])
	if err != nil {
		log.Fatalf("Invalid arguments: %v", err)
	}
	c, err := client.Dial(clientOptions)
	if err != nil {
		log.Fatalln("Unable to create client", err)
	}
	defer c.Close()

	w := worker.New(c, taskQueue, worker.Options{})
	service := nexus.NewService(service.HelloServiceName)
	err = service.Register(handler.EchoOperation, handler.HelloOperation)
	if err != nil {
		log.Fatalln("Unable to register operations", err)
	}
	w.RegisterNexusService(service)
	w.RegisterWorkflow(handler.HelloHandlerWorkflow)

	err = w.Run(worker.InterruptCh())
	if err != nil {
		log.Fatalln("Unable to start worker", err)
	}
}
```

## 开发使用 Nexus Service 的 caller Workflow {#develop-caller-workflow-nexus-service}

导入包含必要 Service 和 Operation 名称以及输入/输出类型的 Service API 包，以便从 caller Workflow 执行 Nexus Operation：

<!--SNIPSTART samples-go-nexus-caller-workflow-->
[nexus/caller/workflows.go](https://github.com/temporalio/samples-go/blob/main/nexus/caller/workflows.go)
```go
package caller

import (
	"github.com/temporalio/samples-go/nexus/service"
	"go.temporal.io/sdk/workflow"
)

const (
	TaskQueue    = "my-caller-workflow-task-queue"
	endpointName = "my-nexus-endpoint-name"
)

func EchoCallerWorkflow(ctx workflow.Context, message string) (string, error) {
	c := workflow.NewNexusClient(endpointName, service.HelloServiceName)

	fut := c.ExecuteOperation(ctx, service.EchoOperationName, service.EchoInput{Message: message}, workflow.NexusOperationOptions{})

	var res service.EchoOutput
	if err := fut.Get(ctx, &res); err != nil {
		return "", err
	}

	return res.Message, nil
}

func HelloCallerWorkflow(ctx workflow.Context, name string, language service.Language) (string, error) {
	c := workflow.NewNexusClient(endpointName, service.HelloServiceName)

	fut := c.ExecuteOperation(ctx, service.HelloOperationName, service.HelloInput{Name: name, Language: language}, workflow.NexusOperationOptions{})
	var res service.HelloOutput

	// Optionally wait for the operation to be started. NexusOperationExecution will contain the operation token in
	// case this operation is asynchronous, which is a handle that can be used to perform additional actions like
	// cancelling an operation.
	var exec workflow.NexusOperationExecution
	if err := fut.GetNexusOperationExecution().Get(ctx, &exec); err != nil {
		return "", err
	}
	if err := fut.Get(ctx, &res); err != nil {
		return "", err
	}

	return res.Message, nil
}

```

### 设置 Nexus Operation 超时

Nexus Operation 支持[三种类型的超时](/nexus/operations#timeouts)，用于控制 caller 在 Operation 生命周期各阶段愿意等待的时长。调用 `ExecuteOperation` 时，在 `NexusOperationOptions` 中设置这些超时。

#### Schedule-to-Close 超时

[Schedule-to-Close 超时](/nexus/operations#schedule-to-close-timeout)限制 Operation 从调度到完成的总持续时间。Nexus Machinery 会自动重试失败的请求，直到超过此超时。

```go
fut := c.ExecuteOperation(ctx, service.HelloOperationName, service.HelloInput{Name: name, Language: language}, workflow.NexusOperationOptions{
	ScheduleToCloseTimeout: 10 * time.Minute,
})
```

#### Schedule-to-Start 超时

[Schedule-to-Start 超时](/nexus/operations#schedule-to-start-timeout)限制 caller 等待 Operation 由 handler 启动的时长。如果未设置，则不强制执行 Schedule-to-Start 超时。

```go
fut := c.ExecuteOperation(ctx, service.HelloOperationName, service.HelloInput{Name: name, Language: language}, workflow.NexusOperationOptions{
	ScheduleToStartTimeout: 2 * time.Minute,
})
```

#### Start-to-Close 超时

[Start-to-Close 超时](/nexus/operations#start-to-close-timeout)限制 caller 等待异步 Operation 启动后完成的时长。此超时仅适用于异步 Operation。如果未设置，则不强制执行 Start-to-Close 超时。

```go
fut := c.ExecuteOperation(ctx, service.HelloOperationName, service.HelloInput{Name: name, Language: language}, workflow.NexusOperationOptions{
	StartToCloseTimeout: 5 * time.Minute,
})
```

### 开发 starter 以启动 caller Workflow

要启动 caller Workflow，需要一个 starter 程序。

<!--SNIPSTART samples-go-nexus-caller-starter-->
[nexus/caller/starter/main.go](https://github.com/temporalio/samples-go/blob/main/nexus/caller/starter/main.go)
```go
package main

import (
	"context"
	"log"
	"os"
	"time"

	"go.temporal.io/sdk/client"

	"github.com/temporalio/samples-go/nexus/caller"
	"github.com/temporalio/samples-go/nexus/options"
	"github.com/temporalio/samples-go/nexus/service"
)

func main() {
	clientOptions, err := options.ParseClientOptionFlags(os.Args[1:])
	if err != nil {
		log.Fatalf("Invalid arguments: %v", err)
	}
	c, err := client.Dial(clientOptions)
	if err != nil {
		log.Fatalln("Unable to create client", err)
	}
	defer c.Close()
	runWorkflow(c, caller.EchoCallerWorkflow, "Nexus Echo 👋")
	runWorkflow(c, caller.HelloCallerWorkflow, "Nexus", service.ES)
}

func runWorkflow(c client.Client, workflow interface{}, args ...interface{}) {
	ctx := context.Background()
	workflowOptions := client.StartWorkflowOptions{
		ID:        "nexus_hello_caller_workflow_" + time.Now().Format("20060102150405"),
		TaskQueue: caller.TaskQueue,
	}

	wr, err := c.ExecuteWorkflow(ctx, workflowOptions, workflow, args...)
	if err != nil {
		log.Fatalln("Unable to execute workflow", err)
	}
	log.Println("Started workflow", "WorkflowID", wr.GetID(), "RunID", wr.GetRunID())

	// Synchronously wait for the workflow completion.
	var result string
	err = wr.Get(context.Background(), &result)
	if err != nil {
		log.Fatalln("Unable get workflow result", err)
	}
	log.Println("Workflow result:", result)
}
```

## 在开发 Server 上进行跨 Namespace 的 Nexus 调用 {#nexus-calls-across-namespaces-dev-server}

按照以下步骤运行 Nexus handler Worker、Nexus caller Worker 和 starter。

### 运行连接到本地开发 Server 的 Worker

运行 Nexus handler Worker：

```
cd handler
go run ./worker \
  -target-host localhost:7233 \
  -namespace my-target-namespace
```

在另一个终端窗口中，运行 Nexus caller Worker：

```
cd caller
go run ./worker \
  -target-host localhost:7233 \
  -namespace my-caller-namespace
```

### 启动 caller Workflow

Worker 运行后，本地开发流程的最后一步是启动 caller Workflow。

运行 starter：

```
cd caller
go run ./starter \
  -target-host localhost:7233 \
  -namespace my-caller-namespace
```

结果如下：

```
2024/10/04 19:57:40 Workflow result: Nexus Echo 👋
2024/10/04 19:57:40 Started workflow WorkflowID nexus_hello_caller_workflow_20240723195740 RunID c9789128-2fcd-4083-829d-95e43279f6d7
2024/10/04 19:57:40 Workflow result: ¡Hola! Nexus 👋
```

### 取消 Nexus Operation {#canceling-a-nexus-operation}

要从 Workflow 内部取消 Nexus Operation，请使用 `workflow.WithCancel` API 创建一个 Go context。这会返回一个新的 context 和一个函数，调用该函数会取消 context 以及传入该 context 的任何 SDK 方法。`NexusClient.ExecuteOperation` 返回的 future 在 Operation 完成时（无论是成功、失败、超时还是被取消）被解析。

在 Nexus 中，只有异步 Operation 可以被取消，因为取消是通过 Operation 令牌发送的。支撑该 Operation 的 Workflow 或其他资源可以选择忽略取消请求。如果被忽略，Operation 可能进入终止状态。

caller Workflow 完成后，caller 的 Nexus Machinery 不会再尝试取消仍在运行的 Operation。在某些使用场景中，保持 Operation 运行是可以接受的。为确保取消被送达，请在退出 Workflow 前等待所有待处理的 Operation 完成。

参考示例，请参阅 [Nexus 取消示例](https://github.com/temporalio/samples-go/tree/main/nexus-cancelation)。

## 在 Temporal Cloud 上进行跨 Namespace 的 Nexus 调用 {#nexus-calls-across-namespaces-temporal-cloud}

本节假设你已熟悉[如何将 Worker 连接到 Temporal Cloud](/develop/go/workers/cloud-worker)。本节使用相同的[源代码](https://github.com/temporalio/samples-go/tree/main/nexus)，但将使用 `tcld` CLI 创建 Namespace 和 Nexus Endpoint，并使用 mTLS 客户端证书将 caller 和 handler Worker 安全地连接到各自的 Temporal Cloud Namespace。

### 安装最新版 `tcld` CLI 并生成证书

要安装最新版 `tcld` CLI，在 macOS 上运行以下命令：

```
brew install temporalio/brew/tcld
```

如果还没有证书，可以使用以下命令生成用于 mTLS Worker 身份验证的证书：

```
tcld gen ca --org $YOUR_ORG_NAME --validity-period 1y --ca-cert ca.pem --ca-key ca.key
```

这些证书有效期为一年。

### 创建 caller 和 handler Namespace

在部署到 Temporal Cloud 之前，请确保已为 caller 和 handler 创建了合适的 Namespace。如果已有这些 Namespace，则无需执行此步骤。

```
tcld login

tcld namespace create \
  --namespace <your-caller-namespace> \
  --cloud-provider aws \
  --region us-west-2 \
  --ca-certificate-file 'path/to/your/ca.pem' \
  --retention-days 1

tcld namespace create \
  --namespace <your-target-namespace> \
  --cloud-provider aws \
  --region us-west-2 \
  --ca-certificate-file 'path/to/your/ca.pem' \
  --retention-days 1
```

或者，可以通过 UI 创建 Namespace：[https://cloud.temporal.io/Namespaces](https://cloud.temporal.io/Namespaces)。

### 创建 Nexus Endpoint 将请求从 caller 路由到 handler

要创建 Nexus Endpoint，必须拥有 Developer 或更高级别的账户角色，并对 `--target-namespace` 拥有 NamespaceAdmin 权限。

```
tcld nexus endpoint create \
  --name <my-nexus-endpoint-name> \
  --target-task-queue my-handler-task-queue \
  --target-namespace <my-target-namespace.account> \
  --description-file description.md
```

或者，可以通过 UI 创建 Nexus Endpoint：[https://cloud.temporal.io/nexus](https://cloud.temporal.io/nexus)。

### 运行使用 TLS 证书连接到 Temporal Cloud 的 Worker

运行 handler Worker：

```
cd handler

go run ./worker \
  -target-host <your-target-namespace.account>.tmprl.cloud:7233 \
  -namespace <your-target-namespace.account> \
  -client-cert 'path/to/your/ca.pem' \
  -client-key 'path/to/your/ca.key'
```

运行 caller Worker：


## 可观测性

### Web UI

同步 Nexus Operation 在 caller Workflow 中的表现如下，caller 的 Workflow 历史中只有 `NexusOperationScheduled` 和 `NexusOperationCompleted` 事件：

<CaptionedImage
    src="/img/cloud/nexus/go-sdk-observability-sync.png"
    title="Observability Sync"
/>

异步 Nexus Operation 在 caller Workflow 中的表现如下，caller 的 Workflow 历史中包含 `NexusOperationScheduled`、`NexusOperationStarted` 和 `NexusOperationCompleted`：

<CaptionedImage
    src="/img/cloud/nexus/go-sdk-observability-async.png"
    title="Observability Async"
/>

### Temporal CLI

使用 `workflow describe` 命令显示 caller Workflow 中待处理的 Nexus Operation 以及 handler Workflow 上附加的任何回调：

```
temporal workflow describe -w <ID>
```

Nexus 事件包含在 caller 的 Workflow 历史中：

```
temporal workflow show -w <ID>
```

对于**异步 Nexus Operation**，以下内容记录在 caller 的历史中：

- `NexusOperationScheduled`
- `NexusOperationStarted`
- `NexusOperationCompleted`

对于**同步 Nexus Operation**，以下内容记录在 caller 的历史中：

- `NexusOperationScheduled`
- `NexusOperationCompleted`

:::note

对于同步 Operation，`NexusOperationStarted` 不会记录在 caller 的历史中。

:::
