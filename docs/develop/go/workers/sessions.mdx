Worker 会话是一项为 [任务路由](/task-routing) 提供简单 API 的功能，以确保使用相同的 Worker 执行 Activity 任务，而无需您手动指定 Task Queue 名称。

## 启用 Worker 会话 {#enable-sessions}

在 Worker 选项中将 `EnableSessionWorker` 设置为 `true`。

  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/go/features/sessions/worker/main.go">
查看源代码
  </a>{' '}
在应用程序代码的其余部分的上下文中。

```go
// ...
func main() {
// ...
	// Enable Sessions for this Worker.
	workerOptions := worker.Options{
		EnableSessionWorker: true,
// ...
	}
	w := worker.New(temporalClient, "fileprocessing", workerOptions)
	w.RegisterWorkflow(sessions.SomeFileProcessingWorkflow)
	w.RegisterActivity(&sessions.FileActivities{})
	err = w.Run(worker.InterruptCh())
// ...
}
```

<a id="max-concurrent-sessions"></a>
### 更改 Worker 的最大并发会话数。 {#max-concurrent-sessions}

您可以调整 Worker 的最大并发会话数。

要限制 Worker 上运行的并发会话数，请将 `Worker.Options` 的 `MaxConcurrentSessionExecutionSize` 字段设置为所需的值。
默认情况下，该字段设置为一个很大的值，因此如果不需要限制，则无需手动设置。

如果 Worker 达到此限制，则在现有会话之一完成之前，它不会接受任何新的 `CreateSession()` 请求。
如果无法在 `CreationTimeout` 内创建会话，`CreateSession()` 将返回错误。

  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/go/features/sessions/worker/main.go">
查看源代码
  </a>{' '}
在应用程序代码的其余部分的上下文中。
</div>

```go
func main() {
// ...
	workerOptions := worker.Options{
// ...
		// This configures the maximum allowed concurrent sessions.
		// Customize this value only if you need to.
		MaxConcurrentSessionExecutionSize: 1000,
// ...
}
// ...
```

## 创建 Worker 会话 {#create-a-session}

在 Workflow 代码中，使用 Workflow API 与选择第一个 Activity 任务的 Worker 创建会话。

使用 [`CreateSession`](https://pkg.go.dev/go.temporal.io/sdk/workflow#CreateSession) API 创建一个 Context 对象，该对象可以传递给调用以生成 Activity 执行。

将 `Workflow.Context` 和 [`SessionOptions`](https://pkg.go.dev/go.temporal.io/sdk/workflow#SessionOptions) 的实例传递给 `CreateSession` API 调用，并获取包含会话元数据信息的会话上下文。

使用会话上下文生成应属于会话的所有 Activity 执行。
然后，所有关联的 Activity 任务均由同一 Worker 实体处理。
调用 `CreateSession` API 时，将使用在 `ActivityOptions` 中指定的 Task Queue 名称（如果在 `ActivityOptions` 中未指定 Task Queue 名称，则在 `StartWorkflowOptions` 中指定），并使用轮询该 Task Queue 的 Worker 之一创建会话。

如果执行此会话的 Worker 死亡或调用 `CompleteSession()`，则会话上下文将被取消。
当使用返回的会话上下文生成 Activity 执行时，如果会话框架检测到执行此会话的 Worker 已死亡，则会返回 `Workflow.ErrSessionFailed` 错误。
Activity 执行的失败不会影响会话的状态，因此您仍然需要处理从活动返回的错误，并在必要时调用 `CompleteSession()`。

如果传入的上下文已包含打开的会话，则 `CreateSession()` 返回错误。
如果所有 Worker 当前都很忙并且无法处理新会话，则框架会不断重试，直到您在 `SessionOptions` 中指定的 `CreationTimeout` 时段过去，然后才会返回错误。
（有关更多详细信息，请查看“并发会话限制”部分。）

`CompleteSession()` 释放在 Worker 上保留的资源，因此当您不再需要会话时立即调用它非常重要。
它取消会话上下文，从而取消使用该会话上下文的所有 Activity 执行。
在失败的会话上调用 `CompleteSession()` 是安全的，这意味着您可以在成功创建会话后从 `defer` 函数调用它。

如果 Worker 在活动之间出现故障，则用于会话 Worker 的任何 Scheduled 活动都将被取消。
如果不是，则当从该 Workflow 进行下一次调用 `Workflow.ExecuteActivity()` 时，您会收到 `Workflow.ErrSessionFailed` 错误。

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/go/features/sessions/workflow.go">
查看源代码
  </a>{' '}
在应用程序代码的其余部分的上下文中。
</div>

```go
package sessions

import (
	"time"

	"go.temporal.io/sdk/workflow"
)

// ...
// SomeFileProcessingWorkflow is a Workflow Definition.
func SomeFileProcessingWorkflow(ctx workflow.Context, param FileProcessingWFParam) error {
	activityOptions := workflow.ActivityOptions{
		StartToCloseTimeout: time.Minute,
	}
	ctx = workflow.WithActivityOptions(ctx, activityOptions)
// ...
	sessionOptions := &workflow.SessionOptions{
		CreationTimeout:  time.Minute,
		ExecutionTimeout: time.Minute,
	}
	// Create a Session with the Worker so that all Activities execute with the same Worker.
	sessionCtx, err := workflow.CreateSession(ctx, sessionOptions)
	if err != nil {
		return err
	}
	defer workflow.CompleteSession(sessionCtx)
// ...
	err = workflow.ExecuteActivity(sessionCtx, a.DownloadFile, param).Get(sessionCtx, &downloadResult)
// ...
	err = workflow.ExecuteActivity(sessionCtx, a.ProcessFile, processParam).Get(sessionCtx, &processResult)
// ...
	err = workflow.ExecuteActivity(sessionCtx, a.UploadFile, uploadParam).Get(sessionCtx, nil)
// ...
}
```

## 附加 Session 使用信息 {#session-metadata}

```go
type SessionInfo struct {
 // A unique Id for the session
 SessionID         string
 // The hostname of the worker that is executing the session
 HostName          string
 // ... other unexported fields
}

func GetSessionInfo(ctx Context) *SessionInfo
```

会话上下文还存储一些会话元数据，可以通过 `GetSessionInfo()` API 检索这些元数据。
如果传入的 Context 不包含任何 Session 元数据，则此 API 将返回 `nil` 指针。

<a id="recreate-session"></a>
### 重新创建会话 {#recreate-session}

对于长时间运行的会话，当所有活动都需要由同一个 Worker 执行时，您可能需要使用 `ContinueAsNew` 功能将 Workflow 拆分为多个运行。
`RecreateSession()` API 专为此类用例而设计。

```go
func RecreateSession(ctx Context, recreateToken []byte, sessionOptions *SessionOptions) (Context, error)
```

它的用法与 `CreateSession()` 相同，只是它还接受 `recreateToken`，这是在与前一个会话相同的 Worker 上创建新会话所需的。
您可以通过调用`SessionInfo`对象的`GetRecreateToken()`方法来获取令牌。

```go
token := workflow.GetSessionInfo(sessionCtx).GetRecreateToken()
```

**有完整的例子吗？**

是的，[Temporalio/样本-go](https://github.com/temporalio/samples-go/tree/master/fileprocessing) 存储库中的 [文件处理示例](https://github.com/temporalio/samples-go) 已被 Updated 使用会话框架。

**如果 Worker 死亡，我的 Activity 会怎样？**

如果您的Activity已经是Scheduled，它将被取消。
否则，当您调用 `Workflow.ExecuteActivity()` 时，您将收到 `Workflow.ErrSessionFailed` 错误。

**并发会话限制是每个进程还是每个主机？**

它是针对每个 Worker 进程的，因此如果您计划使用此功能，请确保主机上仅运行一个 Worker 进程。

**未来的工作**

- 现在，如果 Worker 进程终止，则会话被视为失败。
  但是，对于某些用例，您可能只关心 Worker 主机是否处于活动状态。
  对于这些用例，如果重新启动 Worker 进程，应自动重新建立会话。

- 当前的实现假设所有会话都消耗相同类型的资源，并且只有一个全局限制。
  我们的计划是允许您指定您的会话将消耗什么类型的资源，并对不同类型的资源实施不同的限制。
