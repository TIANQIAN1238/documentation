- [将选择器与 Future 结合使用](#use-selectors-with-futures)

- [将选择器与通道一起使用](#use-selectors-with-channels)

在 Go 中，`select` 语句让 goroutine 等待多个通信操作。
`select` **会阻塞，直到其中一个案例可以运行**，然后执行该案例。
如果准备好多个，它会随机选择一个。

然而，由于随机性，普通的 Go select 语句不能直接在 Workflow 内部使用。
Temporal 的 Go SDK `Selector` 类似，可作为替代品。
他们可以阻止从通道发送和接收，但作为奖励可以监听未来的推迟工作。
为了确保确定性的 Workflow 代码执行，必须使用选择器来推迟和处理工作（代替 Go 的 `select`）（尽管在 Activity 代码中使用 `select` 很好）。

## 完整的 API 示例 {#api-example}

该 API 与 `select` 有很大不同，因此需要记录：

```go
func SampleWorkflow(ctx workflow.Context) error {
	// standard Workflow setup code omitted...

	// API Example: declare a new selector
	selector := workflow.NewSelector(ctx)

	// API Example: defer code execution until the Future that represents Activity result is ready
	work := workflow.ExecuteActivity(ctx, ExampleActivity)
	selector.AddFuture(work, func(f workflow.Future) {
		// deferred code omitted...
	})

	// more parallel timers and activities initiated...

	// API Example: receive information from a Channel
	var signalVal string
	channel := workflow.GetSignalChannel(ctx, channelName)
	selector.AddReceive(channel, func(c workflow.ReceiveChannel, more bool) {
		// matching on the channel doesn't consume the message.
	 	// So it has to be explicitly consumed here
		c.Receive(ctx, &signalVal)
		// do something with received information
	})

	// API Example: block until the next Future is ready to run
	// important! none of the deferred code runs until you call selector.Select
	selector.Select(ctx)

	// Todo: document selector.HasPending
}
```

<a id="use-selectors-with-futures"></a>
## 将选择器与 Future 一起使用 {#use-selectors-with-futures}

通常在 `Activities` 之后添加 `Futures`：

```go
// API Example: defer code execution until after an activity is done
work := workflow.ExecuteActivity(ctx, ExampleActivity)
selector.AddFuture(work, func(f workflow.Future) {
	// deferred code omitted...
})
```

单独调用 `ExecuteActivity` 不会导致实际的 Activity 执行。当 Workflow Definition 中有屈服点时，它会导致一个命令与其他命令一起批处理。在您调用 `selector.Select(ctx)` 之前，您的 `Futures` 不会运行。因此，在调用 `selector.Select` 之前，对 `ExecuteActivity` 的调用不会导致命令发送到服务器。

对 `selector.Select(ctx)` 的调用实际上并不执行 `Future`，但它_请求_执行 `Future`。因此，此时 SDK 将向服务器传达执行 Activity 的命令。

```go
  // API Example: blocking conditionally
  if somecondition != nil {
    selector.Select(ctx)
  }

  // API Example: popping off all remaining Futures
  for i := 0; i < len(someArray); i++ {
    selector.Select(ctx) // this will wait for one branch
    // you can interrupt execution here
  }
```

即使多次调用 Select，Future 也只会在每个 Selector 实例中匹配一次。
如果有多个项目可用，则不定义匹配顺序。

<a id="use-selectors-with-timers"></a>
## 将选择器与 Timer 一起使用 {#use-selectors-with-timers}

future 的一个重要用例是在 Timer 和待处理的 Activity 之间设置竞争，从而有效地添加“软”超时，不会导致该 Activity 出现任何错误或重试。

例如，[Timer 样本](https://github.com/temporalio/samples-go/blob/master/timer) 展示了如何编写长时间运行的订单处理操作，其中：

- 如果处理时间太长，我们会向用户发送有关延迟的通知电子邮件，但我们不会取消操作
- 如果操作在 Timer 触发之前完成，那么我们要取消 Timer。

```go
var processingDone bool
f := workflow.ExecuteActivity(ctx, OrderProcessingActivity)
selector.AddFuture(f, func(f workflow.Future) {
	processingDone = true
	// cancel timerFuture
	cancelHandler()
})

// use timer future to send notification email if processing takes too long
timerFuture := workflow.NewTimer(childCtx, processingTimeThreshold)
selector.AddFuture(timerFuture, func(f workflow.Future) {
	if !processingDone {
		// processing is not done yet when timer fires, send notification email
		_ = workflow.ExecuteActivity(ctx, SendEmailActivity).Get(ctx, nil)
	}
})

// wait the timer or the order processing to finish
selector.Select(ctx)
```

我们使用 `Workflow.NewTimer` API 创建 Timer。

<a id="use-selectors-with-channels"></a>
## 将选择器与通道一起使用 {#use-selectors-with-channels}

`selector.AddReceive(channel, func(c Workflow.ReceiveChannel, more bool) {})` 是从 `Channels` 接收消息的主要机制。

```go
// API Example: receive information from a Channel
var signalVal string
channel := workflow.GetSignalChannel(ctx, channelName)
selector.AddReceive(channel, func(c workflow.ReceiveChannel, more bool) {
	c.Receive(ctx, &signalVal)
	// do something with received information
})
```

仅在通道上进行匹配不会消耗消息；它必须通过 `c.Receive(ctx, &SignalVal)` 调用显式使用。
