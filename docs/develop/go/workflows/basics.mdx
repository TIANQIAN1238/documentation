<a id="develop-workflows"></a>
## 如何开发基本的 Workflow {#develop-workflows}

Workflow 是 Temporal 应用程序的基本单元，这一切都始于 [Workflow Definition](/workflow-definition) 的开发。

在 Temporal Go SDK 编程模型中，[Workflow Definition](/workflow-definition) 是可导出函数。
以下是基本 Workflow Definition 的示例。

```go

import (
    "time"

    "go.temporal.io/sdk/workflow"
)
// ...

// YourSimpleWorkflowDefinition is the most basic Workflow Definition.
func YourSimpleWorkflowDefinition(ctx workflow.Context) error {
    // ...
    return nil
}
```

<a id="workflow-parameters"></a>
### 如何定义 Workflow 参数 {#workflow-parameters}

Temporal Workflow 可以具有任意数量的自定义参数。
但是，我们强烈建议将对象用作参数，以便可以在不破坏 Workflow 签名的情况下更改对象的各个字段。
所有 Workflow Definition 参数都必须可序列化。

基于 Go 的 Workflow Definition 的第一个参数必须是 [`Workflow.Context`](https://pkg.go.dev/go.temporal.io/sdk/workflow#Context) 类型。
Temporal Go SDK 使用它来传递 Workflow Execution 上下文，并且几乎所有可从 Workflow 调用的 Go SDK API 都需要它。
它是从 [`go.Temporal.io/sdk/Workflow`](https://pkg.go.dev/go.temporal.io/sdk/workflow) 包中获取的。

`Workflow.Context` 实体的运行方式与 Go 提供的标准 `context.Context` 实体类似。
`Workflow.Context` 和 `context.Context` 之间的唯一区别是 `Workflow.Context` 提供的 `Done()` 函数返回 `Workflow.Channel` 而不是标准的 Go `chan`。

调用 Workflow 时，可以将其他参数传递给它。
Workflow Definition 可能支持多个自定义参数，也可能不支持。
这些参数可以是常规类型变量或安全指针。
但是，最佳实践是传递 `struct` 类型的单个参数，因此如果添加新参数，则可以具有一定的向后兼容性。

所有 Workflow Definition 参数必须可序列化，并且不能是通道、函数、变量或不安全指针。

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/go/yourapp/your_workflow_definition_dacx.go">
查看源代码
  </a>{' '}
在应用程序代码的其余部分的上下文中。
</div>

```go
package yourapp

import (
    "time"

    "go.temporal.io/sdk/workflow"
)

// YourWorkflowParam is the object passed to the Workflow.
type YourWorkflowParam struct {
    WorkflowParamX string
    WorkflowParamY int
}
// ...
// YourWorkflowDefinition is your custom Workflow Definition.
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) (*YourWorkflowResultObject, error) {
// ...
}
```

<a id="workflow-return-values"></a>
### 如何定义 Workflow 返回参数 {#workflow-return-values}

Workflow 返回值也必须是可序列化的。
返回结果、返回错误或引发异常在每种受支持的语言中都是相当惯用的。
但是，必须用于获取 Workflow Execution 结果的 Temporal API 将仅收到结果或错误之一。

基于 Go 的 Workflow Definition 可以仅返回 `error` 或 `customValue, error` 组合。
同样，此处的最佳实践是使用 `struct` 类型来保存所有自定义值。
用 Go 编写的 Workflow Definition 可以返回自定义值和错误。
但是，在调用过程中不可能同时接收自定义值和错误，这在 Go 中是正常的。
呼叫者将收到其中之一。
从 Workflow 返回非零 `error` 表示在执行过程中遇到错误，并且 Workflow Execution 应终止，并且系统将忽略任何自定义返回值。

```go
type YourWorkflowResultObject struct {
    WFResultFieldX string
    WFResultFieldY int
}
// ...
// YourWorkflowDefinition is your custom Workflow Definition.
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) (*YourWorkflowResultObject, error) {
// ...
    if err != nil {
        return nil, err
    }
    // Make the results of the Workflow Execution available.
    workflowResult := &YourWorkflowResultObject{
        WFResultFieldX: activityResult.ResultFieldX,
        WFResultFieldY: activityResult.ResultFieldY,
    }
    return workflowResult, nil
}
```

<a id="customize-workflow-type"></a>
### 如何在 Go 中自定义 Workflow 类型 {#customize-workflow-type}

在 Go 中，默认情况下，Workflow 类型名称与函数名称相同。

要自定义 Workflow 类型，请在使用 Worker 注册 Workflow 时使用 `RegisterOptions` 设置 `Name` 参数。

<div className="copycode-notice-container">
  <a href="https://github.com/temporalio/documentation/blob/main/sample-apps/go/yourapp/worker/main_dacx.go">
查看源代码
  </a>{' '}
在应用程序代码的其余部分的上下文中。
</div>

```go
package main

import (
    "log"

    "go.temporal.io/sdk/activity"
    "go.temporal.io/sdk/client"
    "go.temporal.io/sdk/worker"
    "go.temporal.io/sdk/workflow"

    "documentation-samples-go/yourapp"
)
// ...
func main() {
// ...
    yourWorker := worker.New(temporalClient, "your-custom-task-queue-name", worker.Options{})
// ...
    // Use RegisterOptions to set the name of the Workflow Type for example.
    registerWFOptions := workflow.RegisterOptions{
        Name: "JustAnotherWorkflow",
    }
    yourWorker.RegisterWorkflowWithOptions(yourapp.YourSimpleWorkflowDefinition, registerWFOptions)
// ...
}
```

<a id="workflow-logic-requirements"></a>
### 如何开发 Workflow 逻辑 {#workflow-logic-requirements}

Workflow 逻辑受 [确定性执行要求](/workflow-definition#deterministic-constraints) 约束。
因此，每种语言都仅限于使用某些惯用技术。
但是，每个 Temporal SDK 都提供了一组 API，可在 Workflow 内部使用这些 API 来与外部（对于 Workflow）应用程序代码进行交互。

在 Go 中，Workflow Definition 代码不能直接执行以下操作：

- 使用 `range` 迭代地图，因为使用 `range` 地图迭代的顺序是随机的。
  相反，您可以收集映射的键，对它们进行排序，然后迭代排序的键以访问映射。
  该技术提供了确定性结果。
  您还可以使用 Side Effect 或 Activity 来处理地图。
- 调用外部 API、执行文件 I/O 操作、与其他服务通信等（使用 Activity 执行这些操作。）

Temporal Go SDK 具有处理等效 Go 结构的 API：

- `Workflow.Now()` 这是 `time.Now()` 的替代品。
- `Workflow.Sleep()` 这是 `time.Sleep()` 的替代品。
- `Workflow.GetLogger()` 这可确保提供的记录器在重放期间不会重复日志。
- `Workflow.Go()` 这是 `go` 语句的替代。
- `Workflow.Channel` 这是原生 `chan` 类型的替代品。
  Temporal 提供对缓冲和非缓冲通道的支持。
- `Workflow.Selector` 这是 `select` 语句的替代。
  请访问 [Go SDK 选择器](https://legacy-documentation-sdks.temporal.io/go/selectors) 页面了解更多信息。
- `Workflow.Context` 这是 `context.Context` 的替代品。
  有关上下文传播的更多信息，请参阅 [追踪](/develop/go/platform/observability#tracing)。
