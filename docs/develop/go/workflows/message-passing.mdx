Workflow 可以像有状态 Web 服务一样接收消息：查询、Signal 和 Update。
Workflow 实现通过可以对传入查询和 Update 做出反应的处理程序方法以及通过 Signal 通道定义这些端点。
Temporal CLIents 使用消息来读取 Workflow 状态并控制其执行。
有关此主题的一般概述，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing)。
本页介绍 Temporal Go SDK 的这些功能。

<a id="handling-messages"></a>
## 处理消息 {#handling-messages}
:::info
下面的代码是传递 [样本](https://github.com/temporalio/samples-go/tree/message-passing/message-passing-intro) 的工作消息的一部分。
:::

编写消息处理程序时请遵循以下准则：

- 消息中发送的值以及消息处理程序和主 Workflow 函数的返回值必须是 [可序列化](/dataconversion)。
- 优先使用单个结构而不是多个输入参数。
  这允许您添加字段而不更改调用签名。

<a id="queries"></a>
### Query 处理程序 {#queries}
[Query](/sending-messages#sending-queries) 是从 Workflow Execution 检索状态的同步操作：

```go
type Language string

const Chinese Language = "chinese"
const English Language = "english"
const French Language = "french"
const Spanish Language = "spanish"
const Portuguese Language = "portuguese"

const GetLanguagesQuery = "GetLanguages"

type GetLanguagesInput struct {
	IncludeUnsupported bool
}

func GreetingWorkflow(ctx workflow.Context) (string, error) {
    ...
    greeting := map[Language]string{English: "Hello", Chinese: "你好，世界"}
    err := workflow.SetQueryHandler(ctx, GetLanguagesQuery, func(input GetLanguagesInput) ([]Language, error) {
        // 👉 A Query handler returns a value: it can inspect but must not mutate the Workflow state.
        if input.IncludeUnsupported {
            return []Language{Chinese, English, French, Spanish, Portuguese}, nil
        } else {
            // Range over map is a nondeterministic operation.
            // It is OK to have a non-deterministic operation in a query function.
            //workflowcheck:ignore
            return maps.Keys(greeting), nil
        }
    })
    ...
}
```

- 使用 [`SetQueryHandler`](https://pkg.go.dev/go.temporal.io/sdk/workflow#SetQueryHandler) 设置按名称侦听 Query 的 Query 处理程序。
- 处理程序必须是返回两个值、可序列化结果和错误的函数。
- 您无法执行异步操作，例如在 Query 处理程序中执行 Activity。

<a id="signals"></a>
### Signal 频道 {#signals}
[Signal](/sending-messages#sending-signals) 是发送到正在运行的 Workflow Execution 的异步消息，用于更改其状态并控制其流程。
通过从频道接收消息来处理 Signal 消息：

```go
const ApproveSignal = "approve"

type ApproveInput struct {
	Name string
}

func GreetingWorkflow(ctx workflow.Context) error {
    logger := workflow.GetLogger(ctx)
	approverName := ""
	...
	// Block until the language is approved
	var approveInput ApproveInput
	workflow.GetSignalChannel(ctx, ApproveSignal).Receive(ctx, &approveInput)
	approverName = approveInput.Name
	logger.Info("Received approval", "Approver", approverName)
	...
}
```

- 将 Signal 的名称传递给 [`GetSignalChannel`](https://pkg.go.dev/go.temporal.io/sdk/workflow#GetSignalChannel) 以获取侦听该类型 Signal 的 Signal 通道。

或者，您可能希望 Workflow 继续运行并且仍然能够处理外部 Signal。

```go
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) error {
   var signal MySignal
   signalChan := workflow.GetSignalChannel(ctx, "your-signal-name")
 	workflow.Go(ctx, func(ctx workflow.Context) {
 		for {
 			selector := workflow.NewSelector(ctx)
 			selector.AddReceive(signalChan, func(c workflow.ReceiveChannel, more bool) {
 				c.Receive(ctx, &signal)
 			})
 			selector.Select(ctx)
 		}
 	})
   // You could now submit an activity; any signals will still be received while the activity is pending.
 }
```

在上面的示例中，Workflow 代码使用 `Workflow.GetSignalChannel` 打开 Signal 类型的 `Workflow.Channel`（由 Signal 名称标识）。

- 在完成 Workflow 或使用 [Continue-As-New](/develop/go/workflows/continue-as-new) 之前，请确保在 Signal 通道上执行异步耗尽。
  否则，Signal 将丢失。
  [批量滑动窗口](https://github.com/temporalio/samples-go/tree/main/batch-sliding-window) 示例包含一个示例：
- 延迟调用 `Workflow.GetSignalChannel`，直到处理 Signal 通道所需的 Workflow 初始化完成。
  这是安全的，因为当没有为 Signal 创建通道时，SDK 会缓冲它们。

```go
reportCompletionChannel := workflow.GetSignalChannel(ctx, "ReportCompletion")
// Drain signals async
for {
	var recordId int
	ok := reportCompletionChannel.ReceiveAsync(&recordId)
	if !ok {
		break
	}
	s.recordCompletion(ctx, recordId)
}
```

<a id="updates"></a>
### Update 处理程序和验证器 {#updates}
[Update](/sending-messages#sending-updates) 是发送到正在运行的 Workflow Execution 的可跟踪同步请求。
它可以更改 Workflow 状态、控制其流程并返回结果。
发送方必须等待，直到 Worker 接受或拒绝 Update。
如果出现问题，发送者可能会进一步等待接收返回值或异常：

```go
type Language string

const SetLanguageUpdate = "set-language"

func GreetingWorkflow(ctx workflow.Context) error {
	language := English

	err = workflow.SetUpdateHandlerWithOptions(ctx, SetLanguageUpdate, func(ctx workflow.Context, newLanguage Language) (Language, error) {
		// 👉 An Update handler can mutate the Workflow state and return a value.
		var previousLanguage Language
		previousLanguage, language = language, newLanguage
		return previousLanguage, nil
	}, workflow.UpdateHandlerOptions{
		Validator: func(ctx workflow.Context, newLanguage Language) error {
			if _, ok := greeting[newLanguage]; !ok {
				// 👉 In an Update validator you return any error to reject the Update.
				return fmt.Errorf("%s unsupported language", newLanguage)
			}
			return nil
		},
	})
  ...
}
```

- 使用 [Workflow.SetUpdateHandler](https://pkg.go.dev/go.temporal.io/sdk/workflow#SetUpdateHandler) 或 [Workflow.SetUpdateHandlerWithOptions](https://pkg.go.dev/go.temporal.io/sdk/workflow#SetUpdateHandlerWithOptions) 为给定名称注册 Update 处理程序。
- 处理程序必须是接受 `Workflow.Context` 作为其第一个参数的函数。
- 该函数可以返回带有错误的可序列化值或仅返回错误。

- 关于验证器：
  - 在将 Update 写入历史记录之前，使用验证器拒绝该 Update。
    验证器始终是可选的。
    如果您不需要拒绝 Update，则不需要验证器。
  - 要设置验证器，请在调用 [Workflow.SetUpdateHandlerWithOptions](https://pkg.go.dev/go.temporal.io/sdk@v1.29.1/internal#UpdateHandlerOptions) 时在 [Workflow.UpdateHandlerOptions](https://pkg.go.dev/go.temporal.io/sdk/workflow#SetUpdateHandlerWithOptions) 中传递验证器函数。
    验证器必须是一个接受与处理程序相同的参数类型并返回错误类型的单个值的函数。

- 使用验证器接受和拒绝 Update：
  - 要拒绝 Update，您必须在验证器中返回错误或恐慌。
    Workflow 的 `WorkflowPanicPolicy` 确定 Handler 函数内部如何处理恐慌。
  - 如果没有验证器，则始终接受 Update。
- 验证器和 Event History：
  - `WorkflowExecutionUpdateAccepted` 事件被写入历史记录，无论接受是自动的还是由于验证器功能未引发错误或恐慌。
  - 当验证器抛出错误时，Update 会被拒绝，并且 `WorkflowExecutionUpdateAccepted` 不会被添加到 Event History 中。
    调用者收到“Update 失败”错误。

- 使用 [`Workflow.GetCurrentUpdateInfo`](https://pkg.go.dev/go.temporal.io/sdk/workflow#GetCurrentUpdateInfo) 获取有关当前 Update 的信息。
  这包括 Update ID，它在使用 Continue-As-New 时可用于重复数据删除：请参阅 [确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing)。
- Update 处理程序可以使用活动、Child Workflow、持久 [Workflow.睡眠](https://pkg.go.dev/go.temporal.io/sdk/workflow#Sleep)、Timer、[`Workflow.Await`](https://pkg.go.dev/go.temporal.io/sdk/workflow#Await) 条件等。
  请参阅 [阻塞处理程序](#blocking-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing) 了解安全使用指南。
- 延迟调用 [`Workflow.SetUpdateHandler`](https://pkg.go.dev/go.temporal.io/sdk/workflow#SetUpdateHandler)，直到 Update 处理程序所需的 Workflow 初始化完成。
  这是安全的，因为当没有注册处理程序时，SDK 会缓冲消息。
  请注意，[`Workflow.SetUpdateHandler`](https://pkg.go.dev/go.temporal.io/sdk/workflow#SetUpdateHandler) 将立即调用具有匹配类型的缓冲 Update 的处理程序。
  这可能会导致不同类型的消息处理无序。

<a id="send-messages"></a>
## 发送消息 {#send-messages}
要发送查询、Signal 或 Update，您可以调用 Temporal [客户](https://pkg.go.dev/go.temporal.io/sdk/client#Client) 上的方法。
要检查发送消息时所需的参数类型以及查询和 Update 的返回类型，请参阅 Workflow Definition 中相应的处理程序方法。

:::warning Using Continue-as-New and Updates

- Temporal_不_支持 Update 处理程序中的 Continue-As-New 功能。
- 使用 Continue-As-New_之前_完成所有处理程序。
- 从您的主 Workflow 功能中使用 Continue-As-New，就像您完成或失败 Workflow Execution 一样。

:::

<a id="send-query"></a>
### 发送 Query {#send-query}
查询从 Temporal CLIent 发送。

使用 [`Client.QueryWorkflow`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.QueryWorkflow) 或 [`Client.QueryWorkflowWithOptions`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.QueryWorkflowWithOptions)。

```go
// ...
supportedLangResult, err := temporalClient.QueryWorkflow(context.Background(), we.GetID(), we.GetRunID(), message.GetLanguagesQuery, message.GetLanguagesInput{IncludeUnsupported: false})
if err != nil {
    log.Fatalf("Unable to query workflow: %v", err)
}
var supportedLang []message.Language
err = supportedLangResult.Get(&supportedLang)
if err != nil {
    log.Fatalf("Unable to get query result: %v", err)
}
log.Println("Supported languages:", supportedLang)
// ...
```

- 发送 Query 不会将事件添加到 Workflow 的 Event History。

- 您可以在 Namespace 的 Workflow 保留期内向已关闭的 Workflow Execution 发送查询。
  这包括已完成、失败或超时的 Workflow。
  Query 不支持终止 Workflow。

- Worker 必须在线并轮询 Task Queue 才能处理 Query。

<a id="send-signal"></a>
### 发送 Signal {#send-signal}
您可以从 Temporal CLIent 或另一个 Workflow Execution 向 Workflow Execution 发送 Signal。
但是，您只能将 Signal 发送到尚未关闭的 Workflow Execution。

<a id="send-signal-from-client"></a>
#### 从客户端发送 Signal {#send-signal-from-client}
使用 [`Client.SignalWorkflow`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.SignalWorkflow)。

传入 [Workflow ID](/workflow-execution/workflowid-runid#workflow-id) 和 [运行 ID](/workflow-execution/workflowid-runid#run-id) 以唯一标识 Workflow Execution。
如果仅提供 Workflow Id（提供空字符串作为运行 Id 参数），则正在运行的 Workflow Execution 会收到 Signal。

```go
// ...
err = temporalClient.SignalWorkflow(context.Background(), we.GetID(), we.GetRunID(), message.ApproveSignal, message.ApproveInput{Name: ""})
if err != nil {
    log.Fatalf("Unable to signal workflow: %v", err)
}
// ...
```

- 当服务器接受 Signal 时调用返回；它并不等待 Signal 被传送到 Workflow Execution。

- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在 Workflow 的 Event History 中。

<a id="send-signal-from-workflow"></a>
#### 从 Workflow 发送 Signal {#send-signal-from-workflow}
Workflow 可以将 Signal 发送到另一个 Workflow，在这种情况下，它称为外部 Signal。

```go
// ...
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) error {
  ...
  signal := MySignal {
    Message: "Some important data",
  }
  err :=  workflow.SignalExternalWorkflow(ctx, "some-workflow-id", "", "your-signal-name", signal).Get(ctx, nil)
  if err != nil {
    // ...
  }
// ...
}
```

当发送外部 Signal 时：

- [Signal外部Workflow执行已启动](/references/events#signalexternalworkflowexecutioninitiated) 事件出现在发件人的 Event History 中。
- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在收件人的 Event History 中。

<a id="signal-with-start"></a>
#### Signal-带启动 {#signal-with-start}
Signal-With-Start 从客户端使用。
它需要 Workflow Id、Workflow 参数、Signal 名称和 Signal 参数。

如果有一个 Workflow 使用给定的 Workflow Id 运行，则它将是 Signaled。如果没有，将启动新的 Workflow 并立即删除 Signal。

使用 [`Client.SignalWithStartWorkflow`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.SignalWithStartWorkflow) API 启动 Workflow Execution（如果尚未运行）并同时向其传递 Signal。

由于 Workflow Execution 可能不存在，因此此 API 不采用 Run Id 作为参数

```go
// ...
signal := MySignal {
  Message: "Some important data",
}
err = temporalClient.SignalWithStartWorkflow(context.Background(), "your-workflow-id", "your-signal-name", signal)
if err != nil {
	log.Fatalln("Error sending the Signal", err)
	return
}
```

<a id="send-update-from-client"></a>
### 发送 Update {#send-update-from-client}
Update 是同步、阻塞调用，可以更改 Workflow 状态、控制其流程并返回结果。

发送 Update 的客户端必须等待，直到服务器将 Update 传递给 Worker。
Worker 必须可用且响应迅速。
使用上下文设置超时对客户端等待响应的时间进行了严格限制。
如果服务器收到请求后需要立即响应，请改用 Signal。

- 当Worker确认Update通过验证时，`WorkflowExecutionUpdateAccepted`被添加到Event History。
- 当Worker确认Update已完成时，`WorkflowExecutionUpdateCompleted`被添加到Event History。

使用 [`Client.UpdateWorkflow`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.UpdateWorkflow) API 将 Update 发送到 Workflow Execution。

您必须提供 Workflow Id，但指定运行 Id 是可选的。
如果您仅提供 Workflow Id（并提供空字符串作为运行 Id 参数），则运行的 Workflow Execution 会收到 Update。

致电 `UpdateWorkflow()` 时，您必须提供 `WaitForStage`。
此参数控制 Update 在将句柄返回给调用者之前必须到达的阶段：

- 如果 `WaitForStage` 设置为 `WorkflowUpdateStageCompleted`，则在 Update 完成后返回句柄。
- 如果 `WaitForStage` 设置为 `WorkflowUpdateStageAccepted`，则在接受 Update 后（即在验证器运行后，如果有验证器的话）返回句柄。

另请注意，您无法将 Update 发送到其他 Workflow Execution。

```go
ctxWithTimeout, cancel := context.WithTimeout(context.Background(), 15*time.Second)
defer cancel()

updateHandle, err := temporalClient.UpdateWorkflow(ctxWithTimeout, client.UpdateWorkflowOptions{
    WorkflowID:   we.GetID(),
    RunID:        we.GetRunID(),
    UpdateName:   message.SetLanguageUpdate,
    WaitForStage: client.WorkflowUpdateStageAccepted,
    Args:         []interface{}{message.Chinese},
})
if err != nil {
    log.Fatalf("Unable to update workflow: %v", err)
}

var previousLang message.Language
err = updateHandle.Get(ctxWithTimeout, &previousLang)
if err != nil {
    log.Fatalf("Unable to get update result: %v", err)
}
```

<a id="update-with-start"></a>
#### Update-带启动 {#update-with-start}
:::tip

对于开源服务器用户，建议使用Temporal Server版本[Temporal Server 版本 1.28](https://github.com/temporalio/temporal/releases/tag/v1.28.0)。

:::

[Update-带启动](/sending-messages#update-with-start) 让您
[发送 Update](/develop/go/workflows/message-passing#send-update-from-client) 检查是否存在具有该 ID 的已运行的 Workflow：

- 如果 Workflow 存在，则处理 Update。
- 如果 Workflow 不存在，则使用给定 ID 启动新的 Workflow Execution，并在主 Workflow 方法开始执行之前处理 Update。

使用 [`Client.UpdateWithStartWorkflow`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.UpdateWithStartWorkflow) API 调用。
一旦到达所请求的 Update 等待阶段，它就会返回；或者当提供的上下文超时时。
使用 [`WorkflowUpdateHandle`](https://pkg.go.dev/go.temporal.io/sdk/client#WorkflowUpdateHandle) 从 Update 检索结果。

您需要提供：

- [`StartWorkflowOptions`](https://pkg.go.dev/go.temporal.io/sdk/internal#StartWorkflowOptions)。
  需要 [Workflow ID 冲突政策](/workflow-execution/workflowid-runid#workflow-id-conflict-policy)。
  选择“使用现有”并使用幂等 Update 处理程序以确保在客户端发生故障时可以再次执行您的代码。
  并非所有 `StartWorkflowOptions` 都是允许的。
  例如，指定 Cron Schedule 将导致错误。
  请参阅 [API文档](https://pkg.go.dev/go.temporal.io/sdk/internal#StartWorkflowOptions) 了解更多详细信息。

- [`UpdateWorkflowOptions`](https://pkg.go.dev/go.temporal.io/sdk/internal#UpdateWorkflowOptions)。
  与 [Update Workflow](/develop/go/workflows/message-passing#send-update-from-client) 相同，必须指定 Update 名称和 Update 等待阶段。
  对于 Update-with-Start，Workflow Id 是可选的。
  指定时，Id 必须与 `StartWorkflowOptions` 中使用的 ID 匹配。
  由于正在运行的 Workflow Execution 可能尚不存在，因此您无法设置运行 ID。

- [`Client.NewWithStartWorkflowOperation`](https://pkg.go.dev/go.temporal.io/sdk/client#Client.NewWithStartWorkflowOperation)。
  指定 Workflow 选项、方法和参数。
  请注意，`WithStartWorkflowOperation` 只能使用一次。
  重新使用以前使用过的操作会返回 `UpdateWithStartWorkflow` 错误。

以下示例显示了 UpdateWithStart 的创建、配置和使用：

```go
ctxWithTimeout, cancel := context.WithTimeout(context.Background(), 15*time.Second)
defer cancel()

workflowOptions := client.StartWorkflowOptions{
    ID:                       "some-workflow-id",
    TaskQueue:                "some-task-queue",
    WorkflowIDConflictPolicy: enumspb.WORKFLOW_ID_CONFLICT_POLICY_USE_EXISTING,
}

updateOptions := client.UpdateWorkflowOptions{
    UpdateName:   message.SetLanguageUpdate,
    WaitForStage: client.WorkflowUpdateStageCompleted,
}

startWorkflowOp := temporalClient.NewWithStartWorkflowOperation(workflowOptions, MyWorkflow)
updateHandle, err := temporalClient.UpdateWithStartWorkflow(
	ctxWithTimeout,
	client.UpdateWithStartWorkflowOptions{
        StartWorkflowOperation: startWorkflowOp,
        UpdateOptions:          updateOptions,
    })
if err != nil {
    log.Fatalf("Unable to execute update-with-start: %v", err)
}

var previousLang message.Language
err = updateHandle.Get(ctxWithTimeout, &previousLang)
if err != nil {
    log.Fatalf("Unable to obtain update result: %v", err)
}

workflowRun, err := startWorkflowOp.Get(ctxWithTimeout)
if err != nil {
    log.Fatalf("Unable to obtain workflow run: %v", err)
}
```

有关更多示例，请参阅 [获取提前返回模式样本](https://github.com/temporalio/samples-go/tree/main/early-return)。

## 消息处理程序模式 {#message-handler-patterns}
本节介绍常见的写入操作，例如 Signal 和 Update 处理程序。
它不适用于纯读取操作，例如查询或 Update 验证器。

:::tip

有关其他信息，请参阅演示安全阻止消息处理的 [将工作注入主 Workflow](/handling-messages#injecting-work-into-main-workflow)、[确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing) 和 [这个样本](https://github.com/temporalio/samples-go/blob/main/safe_message_handler/README.md)。

<a id="blocking-handlers"></a>
### 阻塞处理程序 {#blocking-handlers}
Signal 和 Update 处理程序可以阻止。
这允许您使用活动、Child Workflow、持久 [Workflow.睡眠](https://pkg.go.dev/go.temporal.io/sdk/workflow#Sleep)、Timer、[`Workflow.Await`](https://pkg.go.dev/go.temporal.io/sdk/workflow#Await) 条件等。
这扩展了处理程序可以完成的操作的可能性，但这也意味着处理程序执行和您的主 Workflow 方法都是同时运行的，并在调用 wait 时在它们之间进行切换。

为了安全地使用阻塞处理程序，了解可能出错的事情至关重要。
有关安全使用阻止 Signal 和 Update 处理程序的指南，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing) 以及下面的 [控制处理程序并发](#control-handler-concurrency) 和 [等待消息处理程序完成](#wait-for-message-handlers) 部分。

以下代码修改了本页前面的 Update 处理程序。
Update 处理程序现在进行阻塞调用来执行 Activity：

```go
func GreetingWorkflow(ctx workflow.Context) error {
	language := English

	err = workflow.SetUpdateHandler(ctx, SetLanguageUpdate, func(ctx workflow.Context, newLanguage Language) (Language, error) {
        if _, ok := greeting[newLanguage]; !ok {
            ao := workflow.ActivityOptions{
                StartToCloseTimeout: 10 * time.Second,
            }
            ctx = workflow.WithActivityOptions(ctx, ao)

            var greeting string
            err := workflow.ExecuteActivity(ctx, CallGreetingService, newLanguage).Get(ctx, &greeting)
            if err != nil {
                return nil, err
            }
            greeting[newLanguage] = greeting
        }
		var previousLanguage Language
		previousLanguage, language = language, newLanguage
		return previousLanguage, nil
	})
  ...
}
```

<a id="block-with-wait"></a>
### 添加阻塞等待条件 {#block-with-wait}
有时，阻塞 Signal 或 Update 处理程序需要满足某些条件才能继续。
您可以使用 [`Workflow.Await`](https://pkg.go.dev/go.temporal.io/sdk/workflow#Await) 来阻止代码继续执行，直到条件成立为止。
您可以通过传递返回 `true` 或 `false` 的函数来指定条件。
这是一个重要的功能，可以帮助您控制处理程序逻辑。

以下是 `Workflow.await` 的三个重要用例：

- 等待特定的 Update 到达。
- 在处理程序中等待，直到适合继续。
- 在主 Workflow 中等待，直到所有活动处理程序完成。

```go
err = workflow.SetUpdateHandler(ctx, "UpdateHandler", func(ctx workflow.Context, input UpdateInput) error {
    workflow.Await(ctx, updateUnblockedFunc)
    ...
})
```

如果您的 Update 处理程序需要首先完成主 Workflow 函数中的某些操作，则这是必要的，因为 Update 处理程序可以与主 Workflow 函数同时执行。

您还可以在处理程序中的任何其他位置使用 `Workflow.await` 来等待特定条件变为真。
这允许您编写在多个点暂停的处理程序，每次都等待所需的条件变为真。

<a id="wait-for-message-handlers"></a>
#### 确保您的处理程序在 Workflow 完成 {#wait-for-message-handlers} 之前完成
`Workflow.await` 可以确保您的处理程序在 Workflow 完成之前完成。
当您的 Workflow 使用阻塞 Update 处理程序时，您的主 Workflow 方法可以在处理程序仍在等待异步任务（例如 Activity）时返回 或 Continue-As-New。
Workflow 完成可能会在处理程序完成关键工作之前中断处理程序，并在尝试检索 Update 结果时导致客户端错误。
使用[`Workflow.Await`](https://pkg.go.dev/go.temporal.io/sdk/workflow#Await)等待[`AllHandlersFinished`](https://pkg.go.dev/go.temporal.io/sdk/workflow#AllHandlersFinished)返回`true`来解决这个问题，让你的Workflow顺利结束：

```go
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) error {
    ...
	err = workflow.Await(ctx, func() bool {
		return workflow.AllHandlersFinished(ctx)
	})
    return nil
}
```

默认情况下，如果您允许 Workflow Execution 完成未完成的 Update 处理程序执行，您的 Worker 将记录一条警告。
您可以通过在 [`Workflow.UpdateHandlerOptions`](https://pkg.go.dev/go.temporal.io/sdk/workflow#UpdateHandlerOptions) 结构上设置 `UnfinishedPolicy` 字段来在每个处理程序的基础上消除这些警告：

```go
err = workflow.SetUpdateHandlerWithOptions(ctx, UpdateHandlerName, UpdateFunc, workflow.UpdateHandlerOptions{
       UnfinishedPolicy: workflow.HandlerUnfinishedPolicyAbandon,
})
```

请参阅 [在 Workflow 完成之前完成处理程序](/handling-messages#finishing-message-handlers) 了解更多信息。

<a id="control-handler-concurrency"></a>
#### 使用 `Workflow.Mutex` 阻止并发处理程序执行 {#control-handler-concurrency}
参见 [消息处理程序并发](/handling-messages#message-handler-concurrency)。

并发进程可以以不可预测的方式交互。
当多个处理程序实例同时运行时，编写不正确的 [并发消息传递](/handling-messages#message-handler-concurrency) 代码可能无法正常工作。
这是一个病态病例的例子：

```go
// ...
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) error {
    ...
    err := workflow.SetUpdateHandler(ctx, "BadUpdateHandler", func(ctx workflow.Context) error {
        ao := workflow.ActivityOptions{
            StartToCloseTimeout: 10 * time.Second,
        }
        ctx = workflow.WithActivityOptions(ctx, ao)

        var result Data
        err := workflow.ExecuteActivity(ctx, FetchData, name).Get(ctx, &result)
        x = result.x
        // 🐛🐛 Bug!! If multiple instances of this handler are executing concurrently, then
        // there may be times when the Workflow has self.x from one Activity execution and self.y from another.
        err = workflow.Sleep(ctx, time.Second)
        if err != nil {
            return err
        }
        y = result.y
    })
    ...
}
```

与 `Workflow.Mutex` 协调访问可更正此代码。
锁定可确保在任何给定时间只有一个处理程序实例可以执行特定的代码部分：

```go
func YourWorkflowDefinition(ctx workflow.Context, param YourWorkflowParam) error {
    ...
    err := workflow.SetUpdateHandler(ctx, "SafeUpdateHandler", func(ctx workflow.Context) error {
        err := mutex.Lock(ctx)
        if err != nil {
            return err
        }
        defer mutex.Unlock()
        ao := workflow.ActivityOptions{
            StartToCloseTimeout: 10 * time.Second,
        }
        ctx = workflow.WithActivityOptions(ctx, ao)

        var result Data
        err := workflow.ExecuteActivity(ctx, FetchData, name).Get(ctx, &result)
        x = data.x
        // ✅ OK: the scheduler may switch now to a different handler execution, or to the main workflow
        // method, but no other execution of this handler can run until this execution finishes.
        err = workflow.Sleep(ctx, time.Second)
        if err != nil {
            return err
        }
        self.y = data.y
    })
    ...
}
```

<a id="troubleshooting"></a>
## 故障排除 {#troubleshooting}

有关此主题的非特定于 Go 的讨论，请参阅 [消息处理程序中的异常](/handling-messages#exceptions)。

将 Signal、Update 或 Query 发送到 Workflow 时，您的客户端可能会遇到以下错误：

- **客户端无法联系服务器**

- **Workflow 不存在**

与 Signal 不同，对于查询和 Update，客户端等待来自 Worker 的响应。
如果 Worker 执行处理程序期间出现问题，客户端可能会收到异常。

<a id="problems-when-sending-an-update"></a>
### 发送 Update 时出现问题 {#problems-when-sending-an-update}

- **没有 Workflow Worker 轮询 Task Queue**

SDK 客户端将重试您的请求，直到调用上下文被取消。

- **Update 失败。**

Update 故障类似于 [Workflow 故障](/references/failures)。
  导致 main 方法中的 Workflow 失败的问题也会导致 Update 处理程序中的 Update 失败。
  这些可能包括：

- 失败的Child Workflow
  - 如果 Activity 重试次数已设置为有限次数，则 Activity 失败
  - Workflow 作者返回 `error`
  - 处理程序中出现恐慌，具体取决于 `WorkflowPanicPolicy`

- **处理程序导致 Workflow 任务失败**
  [Workflow 任务失败](/references/failures) 导致服务器无限期地重试 Workflow 任务。您的 Update 请求会发生什么情况取决于其阶段：
  - 如果服务器未接受请求，您将收到 [`FAILED_PRECONDITION`](https://pkg.go.dev/go.temporal.io/api/serviceerror#FailedPrecondition) 错误。
  - 如果请求已被接受，则该请求是持久的。
    代码部署后，一旦 Workflow 再次正常运行，请使用 [`WorkflowUpdateHandle`](https://pkg.go.dev/go.temporal.io/sdk/client#WorkflowUpdateHandle) 获取 Update 结果。

- **Workflow 已完成，而 Update 处理程序执行正在进行中**：
  您将收到 [`ServiceError`](https://pkg.go.dev/go.temporal.io/api/serviceerror#ServiceError)“Workflow Execution 已完成”`。

如果 Workflow 完成而 Update 处理程序执行正在进行中，则会发生这种情况，例如因为

- Workflow 已取消或失败。

- Workflow 正常完成或继续如新，而 Workflow 作者没有 [等待处理程序完成](/handling-messages#finishing-message-handlers)。

<a id="problems-when-sending-a-query"></a>
### 发送 Query 时出现问题 {#problems-when-sending-a-query}

- **没有 Workflow Worker 轮询 Task Queue**

您将收到 [`ServiceError`](https://pkg.go.dev/go.temporal.io/api/serviceerror#ServiceError)，其中 `status` 为 `FAILED_PRECONDITION`。

- **Query 失败。**
  您将收到 [`QueryFailed`](https://pkg.go.dev/go.temporal.io/api/serviceerror#QueryFailed) 错误。
  Query 处理程序中的任何恐慌都会触发此错误。
  这与 Signal 和 Update 不同，其中恐慌可能会导致 Workflow 任务失败。

- **处理程序导致 Workflow 任务失败。**
  例如，如果 Query 处理程序阻塞线程太长时间而没有屈服，就会发生这种情况。
