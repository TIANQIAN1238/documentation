<a id="develop-workflows"></a>
## 如何开发 Workflow {#develop-workflows}

Workflow 是 Temporal 应用程序的基本单元，这一切都始于 [Workflow Definition](/workflow-definition) 的开发。

在 Temporal Java SDK 编程模型中，Workflow Definition 包括用 `@WorkflowInterface` 注释的 Workflow 接口和实现 Workflow 接口的 Workflow 实现。

Workflow接口是一个Java接口，注释为`@WorkflowInterface`。
每个 Workflow 接口只能有一种用 `@WorkflowMethod` 注释的方法。

```java
// Workflow interface
@WorkflowInterface
public interface YourWorkflow {

    @WorkflowMethod
    String yourWFMethod(Arguments args);
}
```

但是，使用动态 Workflow 时，不要指定 `@WorkflowMethod`，而直接在 Workflow 实现代码中实现 `DynamicWorkflow`。

`@WorkflowMethod` 标识作为 Workflow Execution 起点的方法。
当此方法完成时，Workflow Execution 也完成。

您可以创建 [接口继承层次结构](#interface-inheritance) 以跨其他 Workflow 接口重用组件。
接口继承方法不适用于 `@WorkflowMethod` 注释。

Workflow 实现实现 Workflow 接口。

```java
// Define the Workflow implementation which implements our getGreeting Workflow method.
  public static class GreetingWorkflowImpl implements GreetingWorkflow {
      ...
    }
  }
```

要调用 Workflow 中的活动，请调用 Activity 实现。

使用 `ExternalWorkflowStub` 启动 Signal 或将 Workflow 中的 Signal 发送到其他正在运行的 Workflow Execution。

您还可以在 Workflow Definition 中使用 `Workflow.newChildWorkflowStub()` 或 `Workflow.newUntypedChildWorkflowStub()` 将其他 Workflow 调用为 Child Workflow。

## Workflow 接口继承 {#interface-inheritance}

Workflow 接口可以形成继承层次结构。
它对于创建跨多个可重用的组件可能很有用
Workflow 接口。
例如，想象一个 UI 或 CLI 按钮允许在任何 Workflow 上使用 `retryNow` Signal。要实现此功能，您可以重新设计如下界面：

```java
public interface Retryable {
    @SignalMethod
    void retryNow();
}

@WorkflowInterface
public interface FileProcessingWorkflow extends Retryable {

    @WorkflowMethod
    String processFile(Arguments args);

    @QueryMethod(name="history")
    List<String> getHistory();

    @QueryMethod
    String getStatus();

    @SignalMethod
    void abandon();
}
```

然后其他一些 Workflow 接口可以仅扩展 `Retryable`，例如：

```java
@WorkflowInterface
public interface MediaProcessingWorkflow extends Retryable {

    @WorkflowMethod
    String processBlob(Arguments args);
}
```

现在，如果我们有两个正在运行的 Workflow，一个实现 `FileProcessingWorkflow` 接口，另一个实现 `MediaProcessingWorkflow` 接口，我们可以使用 Signal 的通用接口并了解其 WorkflowId，例如：

```java
Retryable r1 = client.newWorkflowStub(Retryable.class, firstWorkflowId);
Retryable r2 = client.newWorkflowStub(Retryable.class, secondWorkflowId);
r1.retryNow();
r2.retryNow();
```

相同的技术可用于使用基本 Workflow 接口的 Query Workflow。

请注意，此方法不适用于 `@WorkflowMethod` 注释，这意味着在使用基本接口时，它不应包含任何 `@WorkflowMethod` 方法。
为了说明这一点，假设我们定义了以下**无效**代码：

```java
// INVALID CODE!
public interface BaseWorkflow {
    @WorkflowMethod
    void retryNow();
}

@WorkflowInterface
public interface Workflow1 extends BaseWorkflow {}

@WorkflowInterface
public interface Workflow2 extends BaseWorkflow {}
```

任何使用 Worker 注册这两个实现的尝试都将失败。
假设我们有：

```java
worker.registerWorkflowImplementationTypes(
    Workflow1Impl.class, Workflow2Impl.class);
```

此注册将失败并显示：

```text
java.lang.IllegalStateException: BaseWorkflow workflow type is already registered with the worker
```

<a id="workflow-parameters"></a>
## 定义 Workflow 参数 {#workflow-parameters}

Temporal Workflow 可以具有任意数量的自定义参数。
但是，我们强烈建议将对象用作参数，以便可以在不破坏 Workflow 签名的情况下更改对象的各个字段。
所有 Workflow Definition 参数都必须可序列化。

用 `@WorkflowMethod` 注释的方法可以有任意数量的参数。

我们建议传递包含所有输入字段的单个参数，以允许以向后兼容的方式添加字段。

请注意，所有输入都应该可由默认的 Jackson JSON 负载转换器进行序列化。

您可以创建自定义对象并将其传递给 Workflow 方法，如以下示例所示。

```java
//...
@WorkflowInterface
public interface YourWorkflow {
    @WorkflowMethod
    String yourWFMethod(CustomObj customobj);
// ...
}
```

<a id="workflow-return-values"></a>
## 定义 Workflow 返回参数 {#workflow-return-values}

Workflow 返回值也必须是可序列化的。
返回结果、返回错误或引发异常在每种受支持的语言中都是相当惯用的。
但是，必须用于获取 Workflow Execution 结果的 Temporal API 将仅收到结果或错误之一。

Workflow 方法参数和返回值必须可使用提供的 [`DataConverter`](https://www.javadoc.io/static/io.temporal/temporal-sdk/1.17.0/io/temporal/common/converter/DataConverter.html) 进行序列化和反序列化。

`DynamicWorkflow` 的 `execute` 方法可以返回 Object 类型。
确保您的客户端可以处理对象类型返回或能够转换对象类型响应。

相关参考：

- [Data Converter](/dataconversion)
- [Java 数据转换器参考](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/common/converter/DataConverter.html)

<a id="workflow-type"></a>
## 自定义您的 Workflow 类型 {#workflow-type}

Workflow 具有称为 Workflow 名称的类型。

以下示例演示如何为 Workflow 类型设置自定义名称。

Workflow 类型默认为 Workflow 接口的短名称。
在以下示例中，Workflow 类型默认为 `NotifyUserAccounts`。

```java
  @WorkflowInterface

  public interface NotifyUserAccounts {
    @WorkflowMethod
    void notify(String[] accountIds);
}
```

要覆盖此默认命名并分配自定义 Workflow 类型，请使用带有 `name` 参数的 `@WorkflowMethod` 注释。
在以下示例中，Workflow 类型设置为 `your-Workflow`。

```java
@WorkflowInterface

  public interface NotifyUserAccounts {
  @WorkflowMethod(name = "your-workflow")
  void notify(String[] accountIds);
  }
```

当您以这种方式设置 Workflow 类型时，`name` 参数的值不必以大写字母开头。

<a id="workflow-logic-requirements"></a>
## Workflow 逻辑要求 {#workflow-logic-requirements}

Workflow 逻辑受 [确定性执行要求](/workflow-definition#deterministic-constraints) 约束。
因此，每种语言都仅限于使用某些惯用技术。
但是，每个 Temporal SDK 都提供了一组 API，可在 Workflow 内部使用这些 API 来与外部（对于 Workflow）应用程序代码进行交互。

使用 Temporal Java SDK 定义 Workflow 时，必须编写 Workflow 代码以有效执行一次并完成。

写入 Workflow Definition 时适用以下约束：

- 不要在 Workflow 实现中使用可变全局变量。
  这将确保多个 Workflow 实例完全隔离。
- 您的 Workflow 代码必须是确定性的。
  不要直接从 Workflow 代码调用非确定性函数（例如非种子随机函数或 `UUID.randomUUID()`）。
  Temporal SDK 提供了用于调用 Workflow 中的非确定性代码的特定 API。
- 不要使用依赖系统时间的编程语言结构。
  例如，仅使用 `Workflow.currentTimeMillis()` 来获取 Workflow 内的当前时间。
- 不要使用本机 Java `Thread` 或任何其他多线程类，如 `ThreadPoolExecutor`。
  使用Temporal SDK提供的`Async.function`或`Async.procedure`异步执行代码。
- 除了 Workflow 类提供的类之外，请勿使用同步、锁或其他标准 Java 阻塞并发相关类。
  不需要显式同步，因为 Workflow 内的多线程代码一次执行一个线程，并且在全局锁下执行。
  - 呼叫 `Workflow.sleep` 而不是 `Thread.sleep`。
  - 使用 `Promise` 和 `CompletablePromise` 代替 `Future` 和 `CompletableFuture`。
  - 使用 `WorkflowQueue` 代替 `BlockingQueue`。
- 对 Workflow 代码进行任何更改时使用 `Workflow.getVersion`。
  如果没有这个，Updated Workflow 代码的任何部署都可能会破坏已经运行的 Workflow。
- 不要直接从 Workflow 访问配置 API，因为配置更改可能会影响 Workflow Execution 路径。
  将其作为参数传递给 Workflow 函数或使用 Activity 加载它。
- 当您需要一个默认的 Workflow 来处理未使用 Worker 注册的所有 Workflow 类型时，请使用 `DynamicWorkflow`。
  单个实现可以实现 Workflow 类型，根据定义，该类型是从某些外部源动态加载的。
  所有标准 `WorkflowOptions` 和确定性规则均适用于动态 Workflow 实现。
