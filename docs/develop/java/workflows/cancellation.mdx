您可以通过以下方式之一中断 Workflow Execution：

- [取消](#cancellation)：取消 Workflow 提供了一种停止 Workflow Execution 的优雅方法。
- [终止](#termination)：终止 Workflow 会强制停止 Workflow Execution。

终止 Workflow 会强制停止 Workflow Execution。此操作类似于终止进程。

- 系统在 Workflow 历史记录中记录 `WorkflowExecutionTerminated` 事件。
- 强制终止并立即停止 Workflow Execution。
- Workflow 代码没有机会处理终止。
- Workflow 任务不会获得 Scheduled。

在大多数情况下，取消是更好的选择，因为它允许 Workflow 优雅地完成。仅在以下情况下终止
Workflow被卡住，无法正常取消。

## 取消 Workflow Execution {#cancellation}

取消 Workflow 提供了一种停止 Workflow Execution 的优雅方法。此操作类似于将 `SIGTERM` 发送到
过程。

- 系统在 Workflow 历史记录中记录 `WorkflowExecutionCancelRequested` 事件。
- Workflow 任务获取 Scheduled 来处理取消。
- Workflow 代码可以处理取消并执行任何清理逻辑。
- 系统不会强制停止Workflow。

要在 Java 中取消 Workflow Execution，请使用
[取消（）](<https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#cancel()>)
WorkflowStub 上的功能。

```java
WorkflowStub workflowStub = WorkflowStub.fromTyped(workflow);
workflowStub.cancel();
```

## Java 中的取消范围 {#cancellation-scopes}

在 Java SDK 中，Workflow 在内部由取消范围树表示，每个取消范围都有取消行为
你可以指定。默认情况下，一切都在“root”范围内运行。

范围是使用创建的
[Workflow.newCancellationScope](<https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#newCancellationScope(io.temporal.workflow.Functions.Proc1)>)
构造函数

取消应用于取消范围，可以包含整个 Workflow 或仅包含其中的一部分。范围可以
被嵌套，并且取消从外部作用域传播到内部作用域。 Workflow 的方法在最外层作用域中运行。
取消是通过捕获可取消​​操作抛出的 `CanceledFailure` 来处理的。

您还可以使用以下 API：

- `CancellationScope.current()`：获取当前范围。
- `scope.cancel()`：取消`scope`内的所有操作。
- `scope.getCancellationRequest()`：当请求范围取消时解决的承诺，例如当
  Workflow 代码调用 `cancel()` 或整个 Workflow 被外部客户端取消。

当 `CancellationScope` 被取消时，它会在任何子作用域和任何可取消操作中传播取消
在其中创建，例如以下内容：

- 活动
- Timers（使用
  [睡觉](<https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#sleep(java.time.Duration)>)
  函数）
- Child Workflows
- Nexus 操作

### 从 Workflow 取消 Activity {#cancel-activity}

从 Workflow 内取消 Activity 要求 Activity 执行发送心跳并设置心跳
超时。如果不调用Heartbeat，Activity将无法接收取消请求。当任何非立即
Activity 被执行，Activity 执行应该发送心跳并设置
[心跳超时](/encyclopedia/detecting-activity-failures#heartbeat-timeout) 确保服务器知道它是
仍在工作。

当取消 Activity 时，在下一个可用机会时 Activity 中会引发错误。如果清理逻辑
需要执行，可以在 `finally` 子句中或在捕获的取消错误内完成。然而，对于 Activity
要显示已取消，需要重新引发异常。

:::note

与常规活动不同，[当地活动](/local-activity) 目前不支持取消。

:::

要从 Workflow Execution 取消 Activity，请调用
[取消（）](<https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/CancellationScope.html#cancel()>)
Activity 启动所在的 CancellationScope 上的方法。

```java
public class GreetingWorkflowImpl implements GreetingWorkflow {
    @Override
    public String getGreeting(String name) {
      List<Promise<String>> results = new ArrayList<>(greetings.length);

      /*
       * Create our CancellationScope. Within this scope we call the workflow activity
       * composeGreeting method asynchronously for each of our defined greetings in different
       * languages.
       */
      CancellationScope scope =
          Workflow.newCancellationScope(
              () -> {
                for (String greeting : greetings) {
                  results.add(Async.function(activities::composeGreeting, greeting, name));
                }
              });

      /*
       * Execute all activities within the CancellationScope. Note that this execution is
       * non-blocking as the code inside our cancellation scope is also non-blocking.
       */
      scope.run();

      // We use "anyOf" here to wait for one of the activity invocations to return
      String result = Promise.anyOf(results).get();

      // Trigger cancellation of all uncompleted activity invocations within the cancellation scope
      scope.cancel();

      /*
       *  Wait for all activities to perform cleanup if needed.
       *  For the sake of the example we ignore cancellations and
       *  get all the results so that we can print them in the end.
       *
       *  Note that we cannot use "allOf" here as that fails on any Promise failures
       */
      for (Promise<String> activityResult : results) {
        try {
          activityResult.get();
        } catch (ActivityFailure e) {
          if (!(e.getCause() instanceof CanceledFailure)) {
            throw e;
          }
        }
      }
      return result;
    }
}
```

## 终止 Workflow Execution {#termination}

终止 Workflow 会强制停止 Workflow Execution。此操作类似于终止进程。

- 系统在 Workflow 历史记录中记录 `WorkflowExecutionTerminated` 事件。
- 强制终止并立即停止 Workflow Execution。
- Workflow 代码没有机会处理终止。
- Workflow 任务不会获得 Scheduled。

要在 Java 中终止 Workflow Execution，请使用
[终止（）](<https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#terminate(java.lang.String,java.lang.Object...)>)
WorkflowStub 上的功能。

```java
WorkflowStub untyped = WorkflowStub.fromTyped(myWorkflowStub);
untyped.terminate("Sample reason");
```

## 重置 Workflow Execution {#reset}

重置 Workflow Execution 将终止当前的 Workflow Execution 并从
您在其 Event History 中指定的点。当 Workflow 由于非确定性错误或其他原因而被阻止时，请使用重置
阻止其完成的问题。

当您重置 Workflow 时，重置点之前的 Event History 将被复制到新的 Workflow Execution，并且
Workflow 从该点使用当前代码恢复。仅当您解决了根本问题（例如
删除不确定性代码。重置点之后取得的任何进度都将被丢弃。给出一个理由，当
重置，因为它将记录在 Event History 中。

<Tabs>

<TabItem value="web-ui" label="Web UI">

1. 导航至Workflow Execution详细信息页面，
2. 单击右上角下拉菜单中的 **重置** 按钮，
3. 选择要重置的事件 ID，
4. 提供重置原因，
5. 确认重置。

Web UI 显示可用的重置点，并在重置完成后创建指向新 Workflow Execution 的链接。

</TabItem>

<TabItem value="cli" label="Temporal CLI">

使用 `Temporal Workflow reset` 命令重置 Workflow Execution：

```bash
temporal workflow reset \
    --workflow-id <workflow-id> \
    --event-id <event-id> \
    --reason "Reason for reset"
```

例如：

```bash
temporal workflow reset \
    --workflow-id my-background-check \
    --event-id 4 \
    --reason "Fixed non-deterministic code"
```

默认情况下，该命令重置 `default` Namespace 中最新的 Workflow Execution。使用 `--run-id` 重置
具体运行。使用 `--Namespace` 指定不同的 Namespace：

```bash
temporal workflow reset \
    --workflow-id my-background-check \
    --event-id 4 \
    --reason "Fixed non-deterministic code" \
    --namespace my-namespace \
    --tls-cert-path /path/to/cert.pem \
    --tls-key-path /path/to/key.pem
```

重置后监视新的 Workflow Execution 以确保其成功完成。

</TabItem>

</Tabs>
