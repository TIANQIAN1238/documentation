Workflow 可以像有状态 Web 服务一样接收消息：查询、Signal 和 Update。
Workflow 实现通过可以对传入消息和返回值做出反应的处理程序方法定义这些端点。
Temporal CLIents 使用消息读取 Workflow 状态并控制执行。
有关此主题的一般概述，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing)。
本页介绍 Temporal Java SDK 的这些功能。

<a id="writing-message-handlers"></a>
## 编写消息处理程序 {#writing-message-handlers}
编写消息处理程序时请遵循以下准则：

- 消息处理程序使用以下三个注释之一定义为 Workflow 类上的方法：[`@QueryMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/QueryMethod.html)、[`@SignalMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/SignalMethod.html) 和 [`@UpdateMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/UpdateMethod.html)。
- 处理程序和主 Workflow 函数的参数和返回值必须是 [可序列化](/dataconversion)。
- 优先使用具有多个字段的单个类，而不是使用多个输入参数。
  类允许您添加字段而不更改调用签名。

<a id="queries"></a>
### Query 处理程序 {#queries}
[Query](/sending-messages#sending-queries) 是从 Workflow Execution 检索状态的同步操作：

```java
public class MessagePassingIntro {

    public enum Language {
        CHINESE,
        ENGLISH,
        FRENCH,
        SPANISH,
        PORTUGUESE,
    }

    public static class GetLanguagesInput {
        public boolean includeUnsupported;

        public GetLanguagesInput() {
            this.includeUnsupported = false;
        }

        public GetLanguagesInput(boolean includeUnsupported) {
            this.includeUnsupported = includeUnsupported;
        }
    }

    @WorkflowInterface
    public interface GreetingWorkflow {
        ...
        // 👉 Use the @QueryMethod annotation to define a Query handler in the
        // Workflow interface.
        @QueryMethod
        List<Language> getLanguages(GetLanguagesInput input);
    }

    public static class GreetingWorkflowImpl implements GreetingWorkflow {
        ...
        @Override
        public List<Language> getLanguages(GetLanguagesInput input) {
            // 👉 The Query handler returns a value: it must not mutate the Workflow state
            // or perform blocking operations.
            if (input.includeUnsupported) {
                return Arrays.asList(Language.values());
            } else {
                return new ArrayList(greetings.keySet());
            }
        }
    }

}
```

- Query 处理程序不得修改 Workflow 状态。
- 您无法执行阻塞操作，例如在 Query 处理程序中执行 Activity。
- Query 注释接受参数 (`name`)，如 [`@QueryMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/QueryMethod.html) 的 API 参考文档中所述。

<a id="signals"></a>
### Signal 处理程序 {#signals}
[Signal](/sending-messages#sending-signals) 是发送到正在运行的 Workflow Execution 的异步消息，用于更改其状态并控制其流程：

```java
public class MessagePassingIntro {

    public static class ApproveInput {
        private String name;

        public ApproveInput() {}

        public ApproveInput(String name) {
            this.name = name;
        }
    }

    @WorkflowInterface
    public interface GreetingWorkflow {
        ...
        // 👉 Use the @SignalMethod annotation to define a Signal handler in the
        // Workflow interface.
        @SignalMethod
        void approve(ApproveInput input);
    }

    public static class GreetingWorkflowImpl implements GreetingWorkflow {
        ...
        private Boolean approvedForRelease;
        private String approverName;

        @Override
        public void approve(ApproveInput input) {
            // 👉 The Signal handler mutates the Workflow state but cannot return a value.
            this.approvedForRelease = true;
            this.approverName = input.name;
        }
    }
}
```

- 处理程序不应返回值。
  服务器立即发送响应，无需等待 Workflow 处理 Signal。

- Signal 注释接受参数（`name` 和 `unfinished_policy`），如 [`@SignalMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/SignalMethod.html) 的 API 参考文档中所述。

- Signal（和 Update）处理程序可能会阻塞。
  这允许您使用活动、Child Workflow、耐用的 [`Workflow.sleep`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#sleep(java.time.Duration)) Timer、[`Workflow.await`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#await(java.time.Duration,java.util.function.Supplier)) 等。
  有关安全使用阻止 Signal 和 Update 处理程序的指南，请参阅 [阻塞处理程序](#blocking-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing)。

<a id="updates"></a>
### Update 处理程序和验证器 {#updates}
[Update](/sending-messages#sending-updates) 是发送到正在运行的 Workflow Execution 的可跟踪同步请求。
它可以更改 Workflow 状态、控制其流程并返回结果。
发送方必须等待，直到 Worker 接受或拒绝 Update。
如果出现问题，发送者可能会进一步等待接收返回值或异常：

```java
public class MessagePassingIntro {
    @WorkflowInterface
    public interface GreetingWorkflow {
        ...
        // 👉 Use the @UpdateMethod annotation to define an Update handler in the
        // Workflow interface.
        @UpdateMethod
        Language setLanguage(Language language);

        // 👉 Update validators are optional
        @UpdateValidatorMethod(updateName = "setLanguage")
        void setLanguageValidator(Language language);
    }

    public static class GreetingWorkflowImpl implements GreetingWorkflow {
        ...
        @Override
        public Language setLanguage(Language language) {
            // 👉 The Update handler can mutate the Workflow state and return a value.
            Language previousLanguage = this.language;
            this.language = language;
            return previousLanguage;
        }

        @Override
        public void setLanguageValidator(Language language) {
            // 👉 The Update validator performs validation but cannot mutate the Workflow state.
            if (!greetings.containsKey(language)) {
                throw new IllegalArgumentException("Unsupported language: " + language);
            }
        }
    }
}
```

- Update 注释接受参数（`name` 和 `unfinished_policy`），如 [`@UpdateMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/UpdateMethod.html) 的 API 参考文档中所述。

- 关于验证器：
  - 在将 Update 写入历史记录之前，使用验证器拒绝该 Update。
    验证器始终是可选的。
    如果您不需要拒绝 Update，则可以跳过它们。
  - 使用 `@UpdateValidatorMethod` 注释定义 Update 验证器。
    声明验证器时使用 `UpdateName` 参数将其连接到其 Update。
    验证器必须返回 `void` 并接受与处理程序相同的参数类型。

- 使用验证器接受和拒绝 Update：
  - 要拒绝 Update，请在验证器中抛出任何类型的异常。
  - 如果没有验证器，则始终接受 Update。
- 验证器和 Event History：
  - 无论是自动接受还是编程接受，`WorkflowExecutionUpdateAccepted` 事件都会写入历史记录。
  - 当验证器抛出错误时，Update 会被拒绝，Update 不会运行，并且 `WorkflowExecutionUpdateAccepted` _不会_ 添加到 Event History 中。
    调用者收到“Update 失败”错误。

- 使用[`getCurrentUpdateInfo`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/internal/sync/WorkflowInternal.html#getCurrentUpdateInfo()) 获取有关当前Update 的信息。这包括 Update ID，它在使用 Continue-As-New 时可用于重复数据删除：请参阅 [确保您的消息仅被处理一次](https://docs.temporal.io/handling-messages#exactly-once-message-processing)。

- Signal（和 Update）处理程序可以是阻塞的，让它们使用活动、Child Workflow、持久 [`Workflow.sleep`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#sleep(java.time.Duration)、Timer、[`Workflow.await`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#await(java.time.Duration,java.util.function.Supplier)）条件等。
  请参阅 [阻塞处理程序](#blocking-handlers) 和 [Workflow消息传递](/encyclopedia/workflow-message-passing) 了解安全使用指南。

<a id="send-messages"></a>
## 发送消息 {#send-messages}
要发送查询、Signal 或 Update，您可以调用 `WorkflowInterface` 上的方法，通常称为“WorkflowStub”。

使用 [newWorkflow存根](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowClient.html#newWorkflowStub(java.lang.Class,io.temporal.client.WorkflowOptions)) 获取 WorkflowStub。

例如：

```java
WorkflowServiceStubs service = WorkflowServiceStubs.newLocalServiceStubs();

WorkflowClient client = WorkflowClient.newInstance(service);

WorkflowOptions workflowOptions =
    WorkflowOptions.newBuilder().setTaskQueue(TASK_QUEUE).setWorkflowId(WORKFLOW_ID).build();

// Create the workflow client stub. It is used to start the workflow execution.
GreetingWorkflow workflow = client.newWorkflowStub(GreetingWorkflow.class, workflowOptions);

// Start workflow asynchronously and call its getGreeting workflow method
WorkflowClient.start(workflow::getGreetings);
```

要检查发送消息时所需的参数类型以及查询和 Update 的返回类型，请参阅 Workflow Definition 中相应的处理程序方法。

:::warning Using Continue-as-New and Updates

- Temporal_不_支持 Update 处理程序中的 Continue-As-New 功能。
- 使用 Continue-As-New_之前_完成所有处理程序。
- 从您的主要 Workflow Definition 方法中使用 Continue-As-New，就像完成或失败 Workflow Execution 一样。

:::

<a id="send-query"></a>
### 发送 Query {#send-query}
从客户端代码中创建的 `WorkflowStub` 调用 Workflow 中定义的 Query 方法，以将 Query 发送到 Workflow Execution：

```java
List<Language> languages = workflow.getLanguages(new GetLanguagesInput(false));
System.out.println("Supported languages: " + languages);
```

- 发送 Query 不会将事件添加到 Workflow 的 Event History。

- 您可以在 Namespace 的 Workflow 保留期内向已关闭的 Workflow Execution 发送查询。
  这包括已完成、失败或超时的 Workflow。
  Querying 终止的 Workflow 不安全，因此不受支持。

- Worker 必须在线并轮询 Task Queue 才能处理 Query。

<a id="send-signal"></a>
### 发送 Signal {#send-signal}
您可以从 Temporal CLIent 或另一个 Workflow Execution 向 Workflow Execution 发送 Signal。
但是，您只能将 Signal 发送到尚未关闭的 Workflow Execution。

<a id="send-signal-from-client"></a>
#### 从客户端发送 Signal {#send-signal-from-client}
要从客户端代码发送 Signal，请在 WorkflowStub 上调用 Signal 方法：

```java
workflow.approve(new ApproveInput("Me"));
```

- 当服务器接受 Signal 时调用返回；它并不等待 Signal 被传送到 Workflow Execution。

- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在 Workflow 的 Event History 中。

<a id="send-signal-from-workflow"></a>
#### 从 Workflow 发送 Signal {#send-signal-from-workflow}
Workflow 可以将 Signal 发送到另一个 Workflow，称为_外部 Signal_。
在_当前_ Workflow 中使用 [`Workflow.newExternalWorkflowStub`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#newExternalWorkflowStub(java.lang.Class,io.temporal.api.common.v1.WorkflowExecution)) 为另一个 Workflow 创建 `ExternalWorkflowStub`。
在外部存根上调用 Signal 方法到 Signal 和其他 Workflow：

```java
OtherWorkflow other = Workflow.newExternalWorkflowStub(OtherWorkflow.class, otherWorkflowID);
other.mySignalMethod();
```

当发送外部 Signal 时：

- [Signal外部Workflow执行已启动](/references/events#signalexternalworkflowexecutioninitiated) 事件出现在发件人的 Event History 中。
- [Workflow执行Signaled](/references/events#workflowexecutionsignaled) 事件出现在收件人的 Event History 中。

<a id="signal-with-start"></a>
#### Signal-带启动 {#signal-with-start}
Signal-With-Start 允许客户端将 Signal 发送到 Workflow Execution，如果尚未运行则启动执行。
如果有一个 Workflow 使用给定的 Workflow Id 运行，则它将是 Signaled。
如果没有，将启动新的 Workflow 并立即删除 Signal。
要使用 Signal-With-Start，请调用 `SignalWithStart` 并传递 Signal 的名称及其参数：

```java
public static void signalWithStart() {
    // WorkflowStub is a client-side stub to a single Workflow instance
    WorkflowStub untypedWorkflowStub = client.newUntypedWorkflowStub("GreetingWorkflow",
    WorkflowOptions.newBuilder()
            .setWorkflowId(workflowId)
            .setTaskQueue(taskQueue)
            .build());

    untypedWorkflowStub.signalWithStart("setCustomer", new Object[] {customer2}, new Object[] {customer1});

    String greeting = untypedWorkflowStub.getResult(String.class);
}
```

这是上一个示例的 `WorkflowInterface`。
使用 Signal-With-Start 时，Signal 处理程序 (`setCustomer`) 将在 Workflow 方法 (`greet`) 之前执行。

```java
@WorkflowInterface
public interface GreetingWorkflow {
    @WorkflowMethod
    String greet(Customer customer);

    @SignalMethod
    void setCustomer(Customer customer);

    @QueryMethod
    Customer getCustomer();
}
```

<a id="send-update-from-client"></a>
### 发送 Update {#send-update-from-client}
Update 是同步、阻塞调用，可以更改 Workflow 状态、控制其流程并返回结果。

发送 Update 的客户端必须等待，直到服务器将 Update 传递给 Worker。
Worker 必须可用且响应迅速。
如果服务器收到请求后需要立即响应，请改用 Signal。
另请注意，您无法将 Update 发送到其他 Workflow Execution。

- 当Worker确认Update通过验证时，`WorkflowExecutionUpdateAccepted`被添加到Event History。
- 当Worker确认Update已完成时，`WorkflowExecutionUpdateCompleted`被添加到Event History。

要将 Update 发送到 Workflow Execution，您可以：

- 在客户端代码中调用 WorkflowStub 上的 Update 方法并等待 Update 完成。
  此代码获取 Update 结果：

  ```java
  Language previousLanguage = workflow.setLanguage(Language.CHINESE);
  ```

- 发送 [`startUpdate`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#startUpdate(io.temporal.client.UpdateOptions,java.lang.Object...))，以便在 Update 被接受或拒绝后立即收到 [`WorkflowUpdateHandle`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowUpdateHandle.html)。

- 稍后使用此 `WorkflowUpdateHandle` 获取结果。
  - 阻塞 Update 处理程序通常执行长时间运行的异步操作。
  - `startUpdate` 仅等待直到 Worker 接受或拒绝 Update，而不等待所有异步操作完成。

例如：

  ```java
  WorkflowUpdateHandle<Language> handle =
      WorkflowStub.fromTyped(workflow)
          .startUpdate(
              "setLanguage", WorkflowUpdateStage.ACCEPTED, Language.class, Language.ENGLISH);
  previousLanguage = handle.getResultAsync().get();
  ```

有关更多详细信息，请参阅“阻止处理程序”部分。

要获取 Update 句柄，您可以：

- 使用 [`startUpdate`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#startUpdate(io.temporal.client.UpdateOptions,java.lang.Object...)) 启动 Update 并返回句柄，如前面的示例所示。
- 使用 [`getUpdateHandle`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#getUpdateHandle(java.lang.String,java.lang.Class)) 使用 Update ID 和 Workflow ID 获取正在进行的 Update 的句柄。

您可以使用 `WorkflowUpdateHandle` 获取有关 Update 的信息：

- `getExecution()`：返回此 Update 发送到的 Workflow Execution。
- `getId()`：返回 Update 的唯一 ID，这对于使用 Continue-As-New 时的重复数据删除很有用：请参阅 [确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing)。
- `getResultAsync()`：返回 `CompletableFuture`，可用于等待 Update 完成。

<a id="update-with-start"></a>
#### Update-带启动 {#update-with-start}
:::tip

对于开源服务器用户，建议使用Temporal Server版本[Temporal Server 版本 1.28](https://github.com/temporalio/temporal/releases/tag/v1.28.0)。

:::

[Update-带启动](/sending-messages#update-with-start) 让您
[发送 Update](/develop/java/workflows/message-passing#send-update-from-client) 检查是否存在具有该 ID 的已运行的 Workflow：

- 如果 Workflow 存在，则处理 Update。
- 如果 Workflow 不存在，则使用给定 ID 启动新的 Workflow Execution，并在主 Workflow 方法开始执行之前处理 Update。

使用 [`startUpdateWithStart`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowClient.html#startUpdateWithStart(io.temporal.workflow.Functions.Func,io.temporal.client.UpdateOptions,io.temporal.client.WithStartWorkflowOperation)) WorkflowClient API。
一旦到达所请求的 Update 等待阶段，它就会返回；或者当请求超时时。
使用 [`WorkflowUpdateHandle`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowUpdateHandle.html) 从 Update 检索结果。

您需要提供：

- 从 [`WorkflowOptions`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowOptions.html) 创建的 WorkflowStub。
  `WorkflowOptions` 需要指定 [Workflow ID 冲突政策](/workflow-execution/workflowid-runid#workflow-id-conflict-policy)。
  选择“使用现有”并使用幂等 Update 处理程序以确保在客户端发生故障时可以再次执行您的代码。
  并非所有 `WorkflowOptions` 都是允许的。
  例如，指定 Cron Schedule 将导致错误。

- [`UpdateOptions`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/UpdateOptions.html)。
  与 [Update Workflow](/develop/java/workflows/message-passing#send-update-from-client) 相同，必须指定 Update 等待阶段。
  对于 Update-with-Start，Workflow Id 是可选的。
  指定时，Id 必须与 `WorkflowOptions` 中使用的 ID 匹配。
  由于正在运行的 Workflow Execution 可能尚不存在，因此您无法设置运行 ID。

- [`WithStartWorkflowOperation`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WithStartWorkflowOperation.html)。
  指定 Workflow 方法。
  请注意，`WithStartWorkflowOperation` 只能使用一次。
  重新使用以前使用过的操作会返回 `startUpdateWithStart` 错误。

例如：

```java
WorkflowUpdateHandle<Language> handle =
    WorkflowClient.startUpdateWithStart(
        workflow::setLanguage,
        Language.ENGLISH,
        UpdateOptions.<Language>newBuilder().setWaitForStage(WorkflowUpdateStage.ACCEPTED).build(),
        new WithStartWorkflowOperation<>(workflow::getGreetings));

Language previousLanguage = handle.getResultAsync().get();
```

要直接获取 Update 结果，请使用 [`executeUpdateWithStart`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowClient.html#executeUpdateWithStart(io.temporal.workflow.Functions.Func,io.temporal.client.UpdateOptions,io.temporal.client.WithStartWorkflowOperation)) WorkflowClient API。
一旦 Update 结果可用，它就会返回；或者当 API 调用超时时。
`UpdateOptions` 上的 Update 等待阶段是可选的。
指定时，它必须是 `WorkflowUpdateStage.COMPLETED`。

例如：

```java
Language previousLanguage =
    WorkflowClient.executeUpdateWithStart(
        workflow::setLanguage,
        Language.ENGLISH,
        UpdateOptions.<Language>newBuilder().build(),
        new WithStartWorkflowOperation<>(workflow::getGreetings));
```

有关更多示例，请参阅 [提前返回模式的 Java 示例](https://github.com/temporalio/samples-java/tree/main/core/src/main/java/io/temporal/samples/earlyreturn)。

:::info NON-TYPE SAFE API CALLS

在实际开发中，有时您可能无法导入 Workflow Definition 方法签名。
当您无权访问 Workflow Definition 或者它不是用 Java 编写时，您可以使用这些非类型安全 API 来获取无类型的 WorkflowStub：

- [`WorkflowClient.newUntypedWorkflowStub`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowClient.html#newUntypedWorkflowStub(java.lang.String,io.temporal.client.WorkflowOptions)）
- [`Workflow.newUntypedExternalWorkflowStub`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#newUntypedExternalWorkflowStub(java.lang.String)）。

将方法名称而不是方法对象传递给：

- [`WorkflowStub.Query`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#query(java.lang.String,java.lang.Class,java.lang.Object...)）
- [`WorkflowStub.Signal`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#signal(java.lang.String,java.lang.Object...)）
- [`WorkflowStub.Update`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#update(java.lang.String,java.lang.Class,java.lang.Object...))
- [`WorkflowStub.startUpdateWithStart`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#startUpdateWithStart(io.temporal.client.UpdateOptions,java.lang.Object%5B%5D,java.lang.Object%5B%5D))
- [`WorkflowStub.executeUpdateWithStart`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowStub.html#executeUpdateWithStart(io.temporal.client.UpdateOptions,java.lang.Object%5B%5D,java.lang.Object%5B%5D))

:::

<a id="message-handler-patterns"></a>
## 消息处理程序模式 {#message-handler-patterns}
本节介绍常见的写入操作，例如 Signal 和 Update 处理程序。
它不适用于纯读取操作，例如查询或 Update 验证器。

:::tip

有关其他信息，请参阅 [将工作注入主 Workflow](/handling-messages#injecting-work-into-main-workflow) 和 [确保您的消息仅被处理一次](/handling-messages#exactly-once-message-processing)。

:::

<a id="blocking-handlers"></a>
### 在处理程序中执行阻塞操作 {#blocking-handlers}
Signal 和 Update 处理程序可以阻止。
这允许您使用 [`Workflow.await`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#await(java.time.Duration,java.util.function.Supplier))、活动、Child Workflow、[`Workflow.sleep`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#sleep(java.time.Duration)) Timer 等。
这扩展了处理程序可以完成的操作的可能性，但这也意味着处理程序执行和您的主 Workflow 方法都是同时运行的，并在调用 wait 时在它们之间进行切换。

为了安全地使用阻塞处理程序，了解可能出错的事情至关重要。
有关安全使用阻止 Signal 和 Update 处理程序的指南，请参阅 [Workflow消息传递](/encyclopedia/workflow-message-passing) 以及下面的 [控制处理程序并发](#control-handler-concurrency) 和 [等待消息处理程序完成](#wait-for-message-handlers) 部分。

以下代码修改了本页前面的 Update 处理程序。
Update 处理程序现在进行阻塞调用来执行 Activity：

```java
public static class GreetingWorkflowImpl implements GreetingWorkflow {
    ...
    @Override
    public Language setLanguage(Language language) {
        if (!greetings.containsKey(language)) {
            String greeting = activity.greetingService(language);
            if (greeting == null) {
                // 👉 An update validator cannot be blocking, so cannot be used to check that the remote
                // greetingService supports the requested language. Throwing an ApplicationFailure
                // will fail the Update, but the WorkflowExecutionUpdateAccepted event will still be
                // added to history.
                throw ApplicationFailure.newFailure("Greeting service does not support: " + language, "GreetingFailure")
            }
            greetings.put(language, greeting);
        }
        Language previousLanguage = this.language;
        this.language = language;
        return previousLanguage;
    }
}
```

尽管 Signal 处理程序也可以进行这样的阻塞调用，但使用 Update 处理程序允许客户端在 Activity 完成后接收结果或错误。
这使您的客户端可以跟踪 Update 的活动、Child Workflow 等执行的异步工作的进度。

<a id="block-with-wait"></a>
### 添加阻塞等待条件 {#block-with-wait}
有时，阻塞 Signal 或 Update 处理程序需要满足某些条件才能继续。
您可以使用 [`Workflow.await`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#await(java.time.Duration,java.util.function.Supplier)) 来阻止代码继续执行，直到条件成立为止。
您可以通过传递返回 `true` 或 `false` 的函数来指定条件。
这是一个重要的功能，可以帮助您控制处理程序逻辑。

以下是 `Workflow.await` 的两个重要用例：

- 在处理程序中等待，直到适合继续。
- 在主 Workflow 中等待，直到所有活动处理程序完成。

<a id="wait-in-handlers"></a>
#### 等待处理程序中的条件 {#wait-in-handlers}
在处理程序中使用 `Workflow.await` 是很常见的。
例如，假设您的 Workflow 类具有 `UpdateReadyToExecute` 方法，该方法指示是否应允许您的 Update 处理程序开始执行。
您可以在处理程序中使用 `Workflow.wait_condition` 使处理程序暂停，直到满足条件：

```java
@Override
public String setLanguage(UpdateInput input) {
    Workflow.await(() -> this.updateReadyToExecute(input));
    ...
}
```

请记住：处理程序可以在主 Workflow 方法启动之前执行。

您还可以在处理程序中的任何其他位置使用 `Workflow.await` 来等待特定条件变为真。
这允许您编写在多个点暂停的处理程序，每次都等待所需的条件变为真。

<a id="wait-for-message-handlers"></a>
#### 确保您的处理程序在 Workflow 完成 {#wait-for-message-handlers} 之前完成
`Workflow.await` 可以确保您的处理程序在 Workflow 完成之前完成。
当您的 Workflow 使用阻塞 Signal 或 Update 处理程序时，您的主 Workflow 方法可以在处理程序仍在等待异步任务（例如 Activity）时返回 或 Continue-As-New。
Workflow 完成可能会在处理程序完成关键工作之前中断处理程序，并在尝试检索 Update 结果时导致客户端错误。
使用`Workflow.await`等待[`Workflow.isEveryHandlerFinished`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#isEveryHandlerFinished()返回`true`来解决这个问题，让你的Workflow顺利结束：

```java
public class MyWorkflowImpl implements MyWorkflow {
    ...
    @Override
    public String run() {
        ...
        Workflow.await(() -> Workflow.isEveryHandlerFinished());
        return "workflow-result";
    }
}
```

默认情况下，当您允许 Workflow Execution 完成未完成的处理程序执行时，您的 Worker 将记录一条警告。
您可以通过将 `unfinishedPolicy` 参数传递给 [`@SignalMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/SignalMethod.html) / [`@UpdateMethod`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/UpdateMethod.html) 注释来在每个处理程序的基础上消除这些警告：

```java
@WorkflowInterface
public interface MyWorkflow {
    ...
    @UpdateMethod(unfinishedPolicy = HandlerUnfinishedPolicy.ABANDON)
    void myUpdate();
}
```

请参阅 [在 Workflow 完成之前完成处理程序](/handling-messages#finishing-message-handlers) 了解更多信息。

<a id="use-workflowinit-to-operate-on-workflow-input-before-any-handler-executes"></a>
### 在执行任何处理程序之前，使用 `@WorkflowInit` 对 Workflow 输入进行操作 {#use-workflowinit-to-operate-on-workflow-input-before-any-handler-executes}

通常，您的 Workflows 构造函数不会有任何参数。
但是，如果您在构造函数上使用 `@WorkflowInit` 注释，则可以为其指定与 `@WorkflowMethod` 相同的 [Workflow参数](/develop/java/workflows/basics#workflow-parameters)。
然后，SDK 将确保您的构造函数接收到 [客户发送](/develop/java/client/temporal-client#start-workflow-execution) 的 Workflow 输入参数。
Workflow 输入参数也会传递到您的 `@WorkflowMethod` 方法 - 无论您是否使用 `@WorkflowInit` 注释，这种情况总是会发生。
如果您有需要访问 Workflow 输入的消息处理程序，这非常有用：请参阅 [首先初始化Workflow](/handling-messages#workflow-initializers)。

:::caution

不要从 `@WorkflowInit` 方法内进行阻塞调用。这可能会导致 Workflow 在开始时未完全初始化，这意味着 Signal、Query 和 Update 处理程序注册将被延迟。

:::

这是一个例子。
请注意，构造函数和 `getGreeting` 必须具有相同的参数：

```java
public class GreetingExample {
    @WorkflowInterface
    public interface GreetingWorkflow {
        @WorkflowMethod
        String getGreeting(String input);

        @UpdateMethod
        boolean checkTitleValidity();
    }

    public static class GreetingWorkflowImpl implements GreetingWorkflow {
        private final String nameWithTitle;
        private boolean titleHasBeenChecked;
        ...
        // Note the annotation is on a public constructor
        @WorkflowInit
        public GreetingWorkflowImpl(String input) {
          this.nameWithTitle = "Sir " + input;
          this.titleHasBeenChecked = false;
        }

        @Override
        public String getGreeting(String input) {
          Workflow.await(() -> titleHasBeenChecked)
          return "Hello " + nameWithTitle;
        }

        @Override
        public boolean checkTitleValidity() {
          // 👉 The handler is now guaranteed to see the workflow input
          // after it has been processed by the constructor.
          boolean isValid = activity.checkTitleValidity(nameWithTitle);
          titleHasBeenChecked = true;
          return isValid;
        }
    }
}
```

<a id="control-handler-concurrency"></a>
### 使用锁来防止并发处理程序执行 {#control-handler-concurrency}
并发进程可以以不可预测的方式交互。
当多个处理程序实例同时运行时，编写不正确的 [并发消息传递](/handling-messages#message-handler-concurrency) 代码可能无法正常工作。
这是一个病态病例的例子：

```java
public class DataWorkflowImpl implements DataWorkflow {
    ...
    @Override
    public void badSignalHandler() {
        Data data = activity.fetchData();
        this.x = data.x;
        // 🐛🐛 Bug!! If multiple instances of this method are executing concurrently, then
        // there may be times when the Workflow has self.x from one Activity execution and self.y from another.
        Workflow.sleep(Duration.ofSeconds(1));
        this.y = data.y;
    }
}
```

与 `WorkflowLock` 协调访问可更正此代码。
锁定可确保在任何给定时间只有一个处理程序实例可以执行特定的代码部分：

```java
public class DataWorkflowImpl implements DataWorkflow {
    WorkflowLock lock = Workflow.newWorkflowLock();
    ...
    @Override
    public void safeSignalHandler() {
        try {
            lock.lock();
            Data data = activity.fetchData();
            this.x = data.x;
            // ✅ OK: the scheduler may switch now to a different handler execution,
            // or to the main workflow method, but no other execution of this handler
            // can run until this execution finishes.
            Workflow.sleep(Duration.ofSeconds(1));
            this.y = data.y;
        } finally {
            lock.unlock()
        }
    }
}
```

<a id="message-handler-troubleshooting"></a>
## 消息处理程序故障排除 {#message-handler-troubleshooting}
将 Signal、Update 或 Query 发送到 Workflow 时，您的客户端可能会遇到以下错误：

- **客户端无法联系服务器**：
  您将收到一个 [`WorkflowServiceException`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowServiceException.html)，其中 `cause` 是 [`StatusRuntimeException`](https://grpc.github.io/grpc-java/javadoc/io/grpc/StatusRuntimeException.html)，`status` 是 `UNAVAILABLE`（重试几次后）。

- **Workflow 不存在**：
  您将收到 [`WorkflowNotFoundException`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowNotFoundException.html)。

有关此主题的非特定于 Java 的讨论，请参阅 [消息处理程序中的异常](/handling-messages#exceptions)。

<a id="signal-problems"></a>
### 发送 Signal 时出现问题 {#signal-problems}
使用 Signal 时，上述 `WorkflowException` 是请求导致的唯一异常类型。

相反，对于查询和 Update，客户端等待来自 Worker 的响应。
如果 Worker 执行处理程序期间出现问题，客户端可能会收到异常。

<a id="update-problems"></a>
### 发送 Update 时出现问题 {#update-problems}
使用 Update 时，您可能会遇到以下错误：

- **没有 Workflow Worker 正在轮询 Task Queue**：
  SDK 客户端将无限期地重试您的请求。
  您可以使用带有超时参数的 `CompletableFuture.get()` 方法强制超时。
  这会在过期时引发 `java.util.concurrent.TimeoutException` 异常。

- **Update 失败**：您将收到 `WorkflowUpdateException` 异常。
  有两种方式可以发生这种情况：

- Update 被 Workflow 中与 Update 处理程序一起定义的 Update 验证器拒绝。

- Update 被接受后失败。

Update 故障类似于 [Workflow 故障](/references/failures)。
  导致 main 方法中的 Workflow 失败的问题也会导致 Update 处理程序中的 Update 失败。
  这些可能包括：

- 失败的Child Workflow
      - 失败的 Activity（如果 Activity 重试已设置为有限次数）
      - Workflow作者扔`ApplicationFailure`
      - [getFailWorkflowExceptionTypes](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/worker/WorkflowImplementationOptions.html#getFailWorkflowExceptionTypes()) 中列出的任何错误（默认为空）

- **处理程序导致 Workflow 任务失败**：
  [Workflow 任务失败](/references/failures) 导致服务器无限期地重试 Workflow 任务。您的 Update 请求会发生什么情况取决于其阶段：
  - 如果服务器未接受请求，您将收到 `FAILED_PRECONDITION` [`WorkflowServiceException`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowServiceException.html) 异常。
  - 如果请求已被接受，则该请求是持久的。
    代码部署后，一旦 Workflow 再次正常运行，请使用 [`WorkflowUpdateHandle`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowUpdateHandle.html) 获取 Update 结果。

- **Workflow 已完成，而 Update 处理程序执行正在进行中**：
  您将收到 [`WorkflowServiceException`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowServiceException.html)“Workflow Execution 已完成”`。

如果 Workflow 完成而 Update 处理程序执行正在进行中，则会发生这种情况，例如因为

- Workflow 已取消或失败。

- Workflow 正常完成或继续如新，而 Workflow 作者没有 [等待处理程序完成](/handling-messages#finishing-message-handlers)。

<a id="query-problems"></a>
### 发送 Query 时出现问题 {#query-problems}
使用查询时，您可能会遇到以下错误：

- **没有 Workflow Worker 轮询 Task Queue**：
  您将收到一个 [`WorkflowServiceException`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowServiceException.html)，其中 `cause` 是 [`StatusRuntimeException`](https://grpc.github.io/grpc-java/javadoc/io/grpc/StatusRuntimeException.html)，`status` 是 `FAILED_PRECONDITION`。

- **Query 失败**：
  如果 Query 期间出现问题，您将收到 [`WorkflowQueryException`](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowQueryException.html) 异常。
  Query 处理程序中的任何异常都会触发此错误。
  这与 Signal 和 Update 请求不同，其中异常可能会导致 Workflow 任务失败。

- **处理程序导致 Workflow 任务失败。**
  例如，如果 Query 处理程序阻塞线程太长时间而没有屈服，就会发生这种情况。

<a id="dynamic-handler"></a>
## 动态组件 {#dynamic-handler}
动态 Workflow、Activity、Signal、Update 或 Query 是一种未命名项目。
通常，这些项目按名称注册到 Worker 并在运行时调用。
当未注册或无法识别的 Workflow、Activity 或带有已识别方法签名的消息请求到达时，Worker 可以使用预先注册的动态替代。

例如，您可以发送请求来启动名为​​“MyUnknownWorkflow”的 Workflow。
收到 Workflow 任务后，Worker 可能会发现没有已注册的该类型的 Workflow Definition。
然后，它检查是否存在已注册的动态 Workflow。
如果动态 Workflow 签名与传入的 Workflow 签名匹配，则 Worker 会调用该签名，就像调用非动态静态命名版本一样。

通过注册 Temporal 组件的动态版本，Worker 可以回退到这些名称不匹配的替代实现。

:::caution

明智地使用动态元素并将其作为后备机制，而不是主要设计。
它们可能会带来长期的可维护性和调试问题。
保留动态调用用于在编译时不知道或无法知道名称的情况。

:::

<a id="set-a-dynamic-workflow"></a>
### 设置动态 Workflow {#set-a-dynamic-workflow}
使用 [`DynamicWorkflow`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/DynamicWorkflow.html) 动态实现 Workflow 类型。
注册一个扩展 `DynamicWorkflow` 的 Workflow 实现类型，以实现未向 Worker 显式注册的任何 Workflow 类型。

动态Workflow接口是用`execute`方法实现的。此方法接受 `EncodedValues`，它们是 Workflow Execution 的输入。
这些输入可以由客户端在调用 Workflow Execution 时指定。

```java
public class MyDynamicWorkflow implements DynamicWorkflow {
   @Override
    public Object execute(EncodedValues args) {
    }
}
```

<a id="set-a-dynamic-activity"></a>
### 如何设置动态 Activity {#set-a-dynamic-activity}
要处理没有显式注册处理程序的 Activity 类型，您可以直接实现动态 Activity。

使用 `DynamicActivity` 动态实现任意数量的 Activity 类型。
当注册扩展 `DynamicActivity` 的 Activity 实现时，任何没有显式注册处理程序的 Activity 类型调用都会调用它。

动态Activity接口是通过`execute`方法实现的，如下例所示。

```java
// Dynamic Activity implementation
 public static class DynamicGreetingActivityImpl implements DynamicActivity {
   @Override
   public Object execute(EncodedValues args) {
     String activityType = Activity.getExecutionContext().getInfo().getActivityType();
     return activityType
         + ": "
         + args.get(0, String.class)
         + " "
         + args.get(1, String.class)
         + " from: "
         + args.get(2, String.class);
   }
 }
```

使用 `Activity.getExecutionContext()` 获取有关应动态实现的 Activity 类型的信息。

<a id="set-a-dynamic-signal"></a>
### 如何设置动态 Signal {#set-a-dynamic-signal}
您还可以动态实现 Signal 处理程序。这对于库级代码和 DSL 的实现非常有用。

使用 `Workflow.registerListener(Object)` 在 Workflow 实现代码中注册 `DynamicSignalListener` 的实现。

```java
Workflow.registerListener(
  (DynamicSignalHandler)
      (signalName, encodedArgs) -> name = encodedArgs.get(0, String.class));
```

注册后，发送到 Workflow 且没有定义处理程序的任何 Signal 将被传送到 `DynamicSignalHandler`。
请注意，每个 Workflow Execution 只能注册一个 `Workflow.registerListener(Object)`。
`DynamicSignalHandler` 可以在常规和动态 Workflow 实现中实现。

<a id="set-a-dynamic-query"></a>
### 如何设置动态 Query {#set-a-dynamic-query}
您还可以动态实现 Query 处理程序。这对于库级代码和 DSL 的实现非常有用。

使用 `Workflow.registerListener(Object)` 在 Workflow 实现代码中注册 `DynamicQueryListener` 的实现。

```java
Workflow.registerListener(
  (DynamicQueryHandler)
      (queryName, encodedArgs) -> name = encodedArgs.get(0, String.class));
```

注册后，任何发送到 Workflow 且没有定义处理程序的查询都将被传送到 `DynamicQueryHandler`。
请注意，每个 Workflow Execution 只能注册一个 `Workflow.registerListener(Object)`。
`DynamicQueryHandler` 可以在常规和动态 Workflow 实现中实现。

<a id="set-a-dynamic-update"></a>
### 如何设置动态 Update {#set-a-dynamic-update}
您还可以动态实现 Update 处理程序。
这对于库级代码和 DSL 的实现非常有用。

```java
Workflow.registerListener(
  (DynamicUpdateHandler)
      (updateName, encodedArgs) -> encodedArgs.get(0, String.class));
```

注册后，任何发送到 Workflow 且没有定义处理程序的 Update 将被传送到 `DynamicUpdateHandler`。
每个 Workflow Execution 只能注册一个 `Workflow.registerListener(Object)`。
`DynamicUpdateHandler` 可以在常规和动态 Workflow 实现中实现。
