<a id="start-child-workflow"></a>
## 启动一个子 Workflow Execution {#start-child-workflow}

[儿童 Workflow Execution](/child-workflows) 是一个 Workflow Execution，它是使用 Child Workflow API 来自另一个 Workflow 的 Scheduled。

使用 Child Workflow API 时，Child Workflow 相关事件（[StartChildWorkflowExecutionInitiated](/references/events#startchildworkflowexecutioninitiated)、[子WorkflowExecutionStarted](/references/events#childworkflowexecutionstarted)、[ChildWorkflowExecutionCompleted](/references/events#childworkflowexecutioncompleted) 等）将记录在 Workflow Execution Event History 中。

始终阻止进度，直到 [子WorkflowExecutionStarted](/references/events#childworkflowexecutionstarted) 事件记录到 Event History，以确保子 Workflow Execution 已启动。
此后，可以使用 Child Workflow 选项中设置的 _Abandon_ [家长关闭政策](/parent-close-policy) 放弃子 Workflow Execution。

要确保 Child Workflow Execution 已启动，请首先在 Child Workflow future 实例上调用 Child Workflow Execution 方法，该方法返回不同的 future。

然后获取一个对象的值，该对象充当最初未知的结果的代理，该结果将等待子 Workflow Execution 生成。

<a id="async-child-workflows"></a>
### 异步 Child Workflows {#async-child-workflows}

对 Child Workflow 存根的第一次调用必须始终是其 Workflow 方法（用 `@WorkflowMethod` 注释的方法）。
与活动类似，可以使用 `Async#function` 或 `Async#procedure` 使调用 Child Workflow 方法变得同步或异步。
同步调用将阻塞，直到 Child Workflow 方法完成。
异步调用返回 `Promise`，可用于等待 Child Workflow 方法完成，如下例所示：

```java
GreetingChild child = Workflow.newChildWorkflowStub(GreetingChild.class);
Promise<String> greeting = Async.function(child::composeGreeting, "Hello", name);
// ...
greeting.get()
```

要异步执行非类型化 Child Workflow，请在 `ChildWorkflowStub` 上调用 `executeAsync`，如以下示例所示。

```java
//...
ChildWorkflowStub childUntyped =
    Workflow.newUntypedChildWorkflowStub(
        "GreetingChild", // your workflow type
        ChildWorkflowOptions.newBuilder().setWorkflowId("childWorkflow").build());

Promise<String> greeting =
    childUntyped.executeAsync(String.class, String.class, "Hello", name);
String result = greeting.get();
//...
```

以下示例显示如何生成 Child Workflow：

- 从 Workflow 生成 Child Workflow

  ```java
  // Child Workflow interface
  @WorkflowInterface
  public interface GreetingChild {
  @WorkflowMethod
  String composeGreeting(String greeting, String name);
  }
  // Child Workflow implementation not shown

  // Parent Workflow implementation
  public class GreetingWorkflowImpl implements GreetingWorkflow {

  @Override
  public String getGreeting(String name) {
      GreetingChild child = Workflow.newChildWorkflowStub(GreetingChild.class);

      // This is a blocking call that returns only after child has completed.
      return child.composeGreeting("Hello", name );
  }
  }
  ```

- 并行生成两个 Child Workflow（具有相同类型）：

  ```java
  // Parent Workflow implementation
  public class GreetingWorkflowImpl implements GreetingWorkflow {

      @Override
      public String getGreeting(String name) {

          // Workflows are stateful, so a new stub must be created for each new child.
          GreetingChild child1 = Workflow.newChildWorkflowStub(GreetingChild.class);
          Promise<String> greeting1 = Async.function(child1::composeGreeting, "Hello", name);

          // Both children will run concurrently.
          GreetingChild child2 = Workflow.newChildWorkflowStub(GreetingChild.class);
          Promise<String> greeting2 = Async.function(child2::composeGreeting, "Bye", name);

          // Do something else here.
          ...
          return "First: " + greeting1.get() + ", second: " + greeting2.get();
      }
  }
  ```

- 从父级发送 Signal 到 Child Workflow：

  ```java
  // Child Workflow interface
  @WorkflowInterface
  public interface GreetingChild {
      @WorkflowMethod
      String composeGreeting(String greeting, String name);

      @SignalMethod
      void updateName(String name);
  }

  // Parent Workflow implementation
  public class GreetingWorkflowImpl implements GreetingWorkflow {

      @Override
      public String getGreeting(String name) {
          GreetingChild child = Workflow.newChildWorkflowStub(GreetingChild.class);
          Promise<String> greeting = Async.function(child::composeGreeting, "Hello", name);
          child.updateName("Temporal");
          return greeting.get();
      }
  }
  ```

- 不支持从父 Workflow 代码中将 Query 发送到 Child Workflow。但是，您可以使用 `WorkflowClient` 从活动中将 Query 发送到 Child Workflow。

## 父关闭策略 {#parent-close-policy}

[家长关闭政策](/parent-close-policy) 确定如果子 Workflow Execution 的父级更改为关闭状态（已完成、失败或超时），子级 Workflow Execution 会发生什么情况。

默认“父关闭策略”选项设置为终止子 Workflow Execution。

使用 [`ChildWorkflowOptions.newBuilder().setParentClosePolicy`](/parent-close-policy) 在 `ChildWorkflowOptions` 的实例上设置 [家长关闭政策](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/ChildWorkflowOptions.Builder.html)。

- 型号：`ChildWorkflowOptions.Builder`
- 默认：`PARENT_CLOSE_POLICY_TERMINATE`

```java
 public void parentWorkflow() {
     ChildWorkflowOptions options =
        ChildWorkflowOptions.newBuilder()
            .setParentClosePolicy(ParentClosePolicy.PARENT_CLOSE_POLICY_ABANDON)
            .build();
     MyChildWorkflow child = Workflow.newChildWorkflowStub(MyChildWorkflow.class, options);
     Async.procedure(child::<workflowMethod>, <args>...);
     Promise<WorkflowExecution> childExecution = Workflow.getWorkflowExecution(child);
     // Wait for child to start
     childExecution.get()
}
```

在这个例子中，我们是：

1. 创建 Child Workflow 存根时将 `ChildWorkflowOptions.ParentClosePolicy` 设置为 `ABANDON`。
2. 使用 `Async.function` 或 `Async.procedure` 异步启动子 Workflow Execution。
3. 在子存根上调用 `Workflow.getWorkflowExecution(…)`。
4.等待`getWorkflowExecution`返回的`Promise`完成。
   这表明Child Workflow是否启动成功（或失败）。
5. 异步完成父Workflow Execution。

需要执行步骤 3 和 4 以确保子 Workflow Execution 在父级关闭之前启动。
如果父进程启动子进程 Workflow Execution，然后立即完成，则 Child Workflow 将永远不会执行。
