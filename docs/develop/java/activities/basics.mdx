<a id="develop-an-activity"></a>
## 开发 Activity {#develop-an-activity}

Workflow 所做的主要事情之一是协调活动的执行。
Activity 是正常的函数或方法执行，旨在执行单个、明确定义的操作（短期或长期运行），例如 Query 数据库、调用第三方 API 或转码媒体文件。
Activity 可以与 Temporal 平台之外的世界交互，或使用 Temporal CLIent 与 Temporal 服务交互。
为了使 Workflow 能够执行 Activity，我们必须定义 [Activity Definition](/activity-definition)。

[Activity Definition](/activities) 是实现特殊注释接口的 Temporal Java SDK [Activity](https://www.javadoc.io/static/io.temporal/temporal-sdk/0.19.0/io/temporal/activity/Activity.html) 类的组合。

Activity 接口用 `@ActivityInterface` 进行注释，并且 Activity 实现实现此 Activity 接口。
要处理没有显式注册处理程序的 Activity 类型，您可以直接实现动态 Activity。

```java
@ActivityInterface
public interface GreetingActivities {
    String composeGreeting(String greeting, String language);
}
```

Activity 接口中定义的每个方法都定义了单独的 Activity 方法。
您可以使用 `@ActivityMethod` 注释来注释 Activity 接口中的每个方法，但这完全是可选的。
以下示例对上一示例中定义的方法使用 `@ActivityMethod` 注释。

```java
@ActivityInterface
public interface GreetingActivities {
    @ActivityMethod
    String composeGreeting(String greeting, String language);
}
```

Activity 实现是一个实现 Activity 带注释的接口的 Java 类。

```java
// Implementation for the GreetingActivities interface example from in the previous section
 static class GreetingActivitiesImpl implements GreetingActivities {
    @Override
    public String composeGreeting(String greeting, String name) {
      return greeting + " " + name + "!";
    }
  }
```

<a id="activity-parameters"></a>
## 定义 Activity 参数 {#activity-parameters}

[Activity Definition](/activity-definition) 可以支持的参数总数没有明确的限制。
但是，最终编码到 gRPC 消息有效负载中的数据的总大小是有限制的。

单个参数的最大大小限制为 2 MB。
gRPC 消息的总大小（包括所有参数）限制为最大 4 MB。

另外，请记住，所有有效负载数据都记录在 [Workflow Execution Event History](/workflow-execution/event#event-history) 中，大型事件历史记录可能会影响 Worker 性能。
这是因为整个 Event History 可以转移到带有 [Workflow 任务](/tasks#workflow-task) 的 Worker 进程。

某些 SDK 要求您传递上下文对象，而其他 SDK 则不需要。
当涉及到您的应用程序数据（即序列化并编码到有效负载中的数据）时，我们建议您使用单个对象作为包装传递给活动的应用程序数据的参数。
这样您就可以更改传递给 Activity 的数据，而不会破坏函数或方法签名。

Activity 接口可以有任意数量的参数。
所有输入都应该可以通过默认的 Jackson JSON 负载转换器进行序列化。

实施活动时，请注意使用 Activity 调用参数或返回值传输的数据量，因为这些数据记录在 Workflow Execution 事件历史记录中。
大型事件历史记录可能会对性能产生不利影响。

您可以创建自定义对象，并将其传递到 Activity 接口，如以下示例所示。

```java
@ActivityInterface
public interface YourActivities {
    String getCustomObject(CustomObj customobj);
    void sendCustomObject(CustomObj customobj, String abc);
}
```

动态 Activity 接口实现中的 `execute` 方法接受 `EncodedValues`，这些 `EncodedValues` 是 Activity 执行的输入，如以下示例所示。

```java
// Dynamic Activity implementation
 public static class DynamicActivityImpl implements DynamicActivity {
   @Override
   public Object execute(EncodedValues args) {
     String activityType = Activity.getExecutionContext().getInfo().getActivityType();
     return activityType
         + ": "
         + args.get(0, String.class)
         + " "
         + args.get(1, String.class)
         + " from: "
         + args.get(2, String.class);
   }
 }
```

有关更多详细信息，请参阅 [动态 Activity 参考](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/activity/DynamicActivity.html)。

<a id="activity-return-values"></a>
## 定义 Activity 返回值 {#activity-return-values}

从 Activity 返回的所有数据都必须是可序列化的。

Activity 返回值受 Temporal 中的有效负载大小限制。默认有效负载大小限制为 2MB，并且 Event History 事务 ([请参阅此处的云限制](https://docs.temporal.io/cloud/limits#per-message-grpc-limit)) 中的任何 gRPC 消息大小都有 4MB 的硬限制。请记住，所有返回值都记录在 [Workflow Execution Event History](/workflow-execution/event#event-history) 中。

Activity 返回值必须可通过提供的 [`DataConverter`](https://www.javadoc.io/static/io.temporal/temporal-sdk/1.17.0/io/temporal/common/converter/DataConverter.html) 进行序列化和反序列化。

`DynamicActivity` 的 `execute` 方法可以返回 Object 类型。
确保您的 Workflow 或客户端可以处理对象类型返回或能够转换对象类型响应。

- [Data Converter](/dataconversion)
- Java DataConverter 参考：[https://www.javadoc.io/doc/io.Temporal/temporal-sdk/latest/io/temporal/common/converter/DataConverter.html](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/common/converter/DataConverter.html)

<a id="activity-type"></a>
## 自定义您的 Activity 类型 {#activity-type}

每个 Activity 都有一个类型，也可以称为 Activity“名称”。
此名称显示在每个 Activity 任务的“摘要”选项卡中的 Workflow Execution Event History 中。
该名称可让您识别执行期间调用的 Activity 类型。

自定义 Activity 类型名称可防止接口和 Workflow 之间的名称冲突。
它们提供描述性的 Activity 方法名称，无需担心在项目的其他地方重复使用这些名称。
它们还支持代码管理，特别是在具有许多活动的大型项目中。
例如，您可以使用前缀将相关活动分组在一起。
自定义名称还可以区分用于收集指标的键，而不会发生名称冲突。

以下示例显示如何为 Activity 类型设置自定义名称。

<a id="default-behavior"></a>
### 默认行为 {#default-behavior}

默认情况下，Activity 类型是第一个字母大写的方法名称：

```java
@ActivityInterface
public interface GreetingActivities {
    String sendMessage(String input);

    @ActivityMethod
    String composeGreeting(String greeting, String language);
}
```

- 方法名称：`sendMessage`
- Activity 型号：`SendMessage`
- 方法名称：`composeGreeting`
- Activity 型号：`ComposeGreeting`

<a id="custom-prefix"></a>
### 自定义前缀 {#custom-prefix}

在 `@ActivityInterface` 注释中使用 `namePrefix` 参数会为接口中提到的每个 Activity 类型名称添加前缀，除非专门覆盖该前缀：

```java
@ActivityInterface(namePrefix = "Messaging_")
public interface GreetingActivities {
    String sendMessage(String input);

    @ActivityMethod
    String composeGreeting(String greeting, String language);
}
```

- 方法名称：`sendMessage`
- Activity 型号：`Messaging_SendMessage`
- 方法名称：`composeGreeting`
- Activity 型号：`Messaging_ComposeGreeting`

即使使用前缀，Activity 类型也是大写的。

<a id="custom-name"></a>
### 自定义名称 {#custom-name}

要覆盖默认名称和任何继承的前缀，请在 `@ActivityMethod` 注释中使用 `name` 参数：

```java
@ActivityInterface(namePrefix = "Messaging_")
public interface GreetingActivities {
    String sendMessage(String input);

    @ActivityMethod
    String composeGreeting(String greeting, String language);
    @ActivityMethod(name = "farewell")
    String composeFarewell(String farewell, String language);
}
```

使用 `name` 参数不会自动将结果大写：

- 方法名称：`sendMessage`
- Activity 型号：`Messaging_SendMessage`
- 方法名称：`composeGreeting`
- Activity 型号：`Messaging_ComposeGreeting`
- 方法名称：`composeFarewell`
- Activity 型号：`farewell`

请谨慎对待包含特殊字符的名称，因为它们可以用作指标标签。
Prometheus 等系统可能会忽略带有使用不受支持字符的标签的指标。
