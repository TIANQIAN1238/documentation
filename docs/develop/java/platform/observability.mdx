Temporal 开发人员指南的可观察性部分涵盖了查看 [Temporal 应用](/temporal#temporal-application) 当前状态的多种方法，即查看 [Temporal平台](/workflow-execution) 跟踪哪些 [Workflow Executions](/temporal#temporal-platform) 以及任何指定 Workflow Execution 当前或执行点的状态的方法。

本节介绍与查看应用程序状态相关的功能，包括：

- [发出指标](#metrics)
- [设置追踪](#tracing)
- [从 Workflow 登录](#logging)
- [Visibility API](#visibility)

<a id="metrics"></a>
## 发出指标 {#metrics}
每个 Temporal SDK 都能够从客户端或 Worker 进程发出一组可选的指标。
有关能够发出的指标的完整列表，请参阅 [SDK指标参考](/references/sdk-metrics)。

- 有关 Prometheus 和 Grafana 集成的概述，请参阅 [监控](/self-hosted-guide/monitoring) 指南。
- 有关指标列表，请参阅 [SDK指标参考](/references/sdk-metrics)。
- 有关使用 Java SDK 公开指标的端到端示例，请参阅 [样本-java](https://github.com/temporalio/samples-java/tree/main/core/src/main/java/io/temporal/samples/metrics) 存储库。

要使用 Java SDK 发出指标，请使用 [`MicrometerClientStatsReporter`](https://github.com/temporalio/sdk-java/blob/55ee7894aec427d7e384c3519732bdd61119961a/src/main/java/io/temporal/common/reporter/MicrometerClientStatsReporter.java#L34) 类与为指标后端配置的 Micrometer MeterRegistry 集成。
[千分尺](https://micrometer.io/docs) 是一个流行的 Java 框架，提供与 Prometheus 和其他后端的集成。

以下示例演示如何使用 `MicrometerClientStatsReporter` 定义指标范围并使用 `WorkflowServiceStubsOptions` 进行设置。

```java
//...
   // see the Micrometer documentation for configuration details on other supported monitoring systems.
   // in this example shows how to set up Prometheus registry and stats reported.
   PrometheusMeterRegistry registry = new PrometheusMeterRegistry(PrometheusConfig.DEFAULT);
   StatsReporter reporter = new MicrometerClientStatsReporter(registry);
    // set up a new scope, report every 10 seconds
     Scope scope = new RootScopeBuilder()
             .reporter(reporter)
             .reportEvery(com.uber.m3.util.Duration.ofSeconds(10));
   // for Prometheus collection, expose a scrape endpoint.
   //...
   // add metrics scope to WorkflowServiceStub options
   WorkflowServiceStubsOptions stubOptions =
       WorkflowServiceStubsOptions.newBuilder().setMetricsScope(scope).build();
//...
```

有关更多详细信息，请参阅 [Java SDK 样品](https://github.com/temporalio/samples-java/tree/637c2e66fd2dab43d9f3f39e5fd9c55e4f3884f0/core/src/main/java/io/temporal/samples/metrics)。
有关使用 Micrometer 配置 Prometheus 抓取端点的详细信息，请参阅 [千分尺普罗米修斯配置](https://docs.micrometer.io/micrometer/reference/implementations/prometheus.html#_configuring) 文档。

<a id="tracing"></a>
## 设置跟踪 {#tracing}
跟踪允许您查看 Workflow 的调用图及其活动、Nexus 操作和任何 Child Workflow。

Temporal Web 的跟踪功能主要跟踪 Temporal 上下文中的 Activity 执行。如果您需要特定于您的用例的自定义跟踪，您应该利用上下文传播来相应地添加跟踪逻辑。

要在 Java 中配置跟踪，请注册 `OpenTracingClientInterceptor()` 拦截器。
您可以在 Temporal CLIent 端和 Worker 端注册拦截器，或者如果您正在构建可重用库，则可以通过 [插件](/develop/plugins-guide#interceptors) 注册拦截器。

以下代码示例演示了 Temporal CLIent 上的 `OpenTracingClientInterceptor()`。

```java
WorkflowClientOptions.newBuilder()
   //...
   .setInterceptors(new OpenTracingClientInterceptor())
   .build();
```

```java
WorkflowClientOptions clientOptions =
    WorkflowClientOptions.newBuilder()
        .setInterceptors(new OpenTracingClientInterceptor(JaegerUtils.getJaegerOptions(type)))
        .build();
WorkflowClient client = WorkflowClient.newInstance(service, clientOptions);
```

以下代码示例演示了 Worker 上的 `OpenTracingClientInterceptor()`。

```java
WorkerFactoryOptions.newBuilder()
   //...
   .setWorkerInterceptors(new OpenTracingWorkerInterceptor())
   .build();
```

```java
WorkerFactoryOptions factoryOptions =
    WorkerFactoryOptions.newBuilder()
        .setWorkerInterceptors(
            new OpenTracingWorkerInterceptor(JaegerUtils.getJaegerOptions(type)))
        .build();
WorkerFactory factory = WorkerFactory.newInstance(client, factoryOptions);
```

有关详细信息，请参阅 Temporal [开放跟踪模块](https://github.com/temporalio/sdk-java/blob/master/temporal-opentracing/README.md)。

<a id="context-propagation-over-nexus-operation-calls"></a>
### 通过 Nexus 操作调用进行上下文传播 {#context-propagation-over-nexus-operation-calls}

Nexus 不使用标准上下文传播器标头结构。
相反，它依赖于与 Temporal 无关的协议，旨在连接任意系统。
为了通过 Nexus 操作调用传播上下文，上下文被序列化为 `Map<String, String>`。
该映射很特殊，因为它将所有键标准化为小写。

由于 Nexus 使用此自定义格式，并且 Nexus 调用可能涉及外部系统，因此 `ContextPropagator` 接口不适用于 Nexus 标头。
上下文必须通过拦截器显式传播，如 [Nexus 上下文传播示例](https://github.com/temporalio/samples-java/tree/main/core/src/main/java/io/temporal/samples/nexuscontextpropagation) 中所示。

<a id="logging"></a>
## 从 Workflow {#logging} 记录
日志记录使您能够在代码执行期间记录关键信息。
记录器创建审计跟踪并捕获有关 Workflow 操作的信息。
适当的日志记录级别取决于您的具体需求。
在开发或故障排除期间，您可能会使用调试甚至跟踪。
在生产中，您可以使用 info 或 warn 来避免日志量过多。

记录器支持以下日志记录级别：

| 等级   | 使用                                                                                                       |
| ------- | --------------------------------------------------------------------------------------------------------- |
| `TRACE` | 最详细的日志记录级别，用于非常细粒度的信息。                               |
| `DEBUG` | 详细信息，通常可用于调试目的。                                            |
| `INFO`  | 有关应用程序操作的一般信息。                                                    |
| `WARN`  | 指示不会阻止应用程序运行的潜在有害情况或小问题。 |
| `ERROR` | 指示可能仍允许应用程序继续运行的错误情况。                    |

Temporal SDK 核心通常使用 `WARN` 作为其默认日志记录级别。

要在 Workflow 代码中获取标准 `slf4j` 记录器，请使用 [`Workflow.getLogger`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html) 方法。

```java
private static final Logger logger = Workflow.getLogger(DynamicDslWorkflow.class);
```

除非 [`WorkerFactoryOptions.Builder.setEnableLoggingInReplay(boolean)`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/worker/WorkerFactoryOptions.Builder.html#setEnableLoggingInReplay(boolean)) 方法设置为 true，否则重播模式下的日志将被忽略。

<a id="search-attributes"></a>
### 如何使用 Search Attribute {#search-attributes}
检索 Workflow Execution 的典型方法是通过其 Workflow Id。

但是，有时您需要根据另一属性检索一个或多个 Workflow Execution。例如，假设您想要获取某个时间范围内失败的某种类型的所有 Workflow Execution，以便您可以使用相同的参数启动新的 Workflow Execution。

您可以使用 [Search Attributes](/search-attribute) 来完成此操作。

- [默认 Search Attribute](/search-attribute#default-search-attribute) 与 `WorkflowType`、`StartTime` 和 `ExecutionStatus` 一样，会自动添加到 Workflow Execution。
- [定制 Search Attribute](/search-attribute#custom-search-attribute) 可以包含其自己的特定于域的数据（如 `customerId` 或 `numItems`）。

使用自定义 Search Attribute 的步骤是：

- 使用 `Temporal operator search-attribute create` 或云 UI 在 Temporal 服务中创建新的 Search Attribute。
- 设置 Workflow Execution 的 Search Attribute 值：
  - 在客户端上，在启动执行时将其作为选项包含在内。
  - 在 Workflow 中通过调用 `upsertTypedSearchAttributes`。
- 读取Search Attribute的值：
  - 在客户端上通过调用 `DescribeWorkflow`。
  - 在 Workflow 中查看 `WorkflowInfo`。
- Query Workflow Execution 由 Search Attribute 使用 [列表过滤器](/list-filter) 实现：
  - [在Temporal CLI中](/cli/workflow#list)。
  - 在代码中调用 `ListWorkflowExecutions`。

<a id="custom-search-attributes"></a>
### 如何设置自定义 Search Attribute {#custom-search-attributes}
在 Temporal 服务（使用 `Temporal operator search-attribute create` 或云 UI）中创建自定义 Search Attribute 后，您可以在启动 Workflow 时设置自定义 Search Attribute 的值。

当您的客户端启动 Workflow Execution 时，请在使用 `WorkflowOptions.newBuilder().setTypedSearchAttributes()` 的选项中包含自定义 Search Attribute：

```java
    // In a shared constants file, so all files have access

    public static final SearchAttributeKey<Boolean> IS_ORDER_FAILED = SearchAttributeKey.forBoolean("isOrderFailed");
...
    // In main
    WorkflowOptions options = WorkflowOptions.newBuilder()
        .setWorkflowId(workflowID)
        .setTaskQueue(Constants.TASK_QUEUE_NAME)
        .setTypedSearchAttributes(generateSearchAttributes())
        .build();

    PizzaWorkflow workflow = client.newWorkflowStub(PizzaWorkflow.class, options);
...

    // Further down in the file
    private static Map<String, Object> generateSearchAttributes(){
        return SearchAttributes.newBuilder().set(Constants.IS_ORDER_FAILED, false).build();
    }
```

每个`SearchAttribute`对象代表一个自定义属性名称，其值为代表特定类型的[`SearchAttributeKey`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/common/SearchAttributeKey.html#forBoolean(java.lang.String)）。目前支持以下类型：

- 布尔值
- 双人间
- 长
- 关键词
- 关键词列表
- 文字

在此示例中，`isOrderFailed` 设置为 Search Attribute。这个属性是
对于基于客户订单的成功/失败的 Querying Workflow 非常有用。

<a id="upsert-search-attributes"></a>
### 如何更新插入 Search Attributes {#upsert-search-attributes}
在 Workflow 代码中，您可以使用 [`upsertTypedSearchAttributes`](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/workflow/Workflow.html#upsertTypedSearchAttributes(io.temporal.common.SearchAttributeUpdate...) 动态添加 Update 或 Search Attribute。
此方法对于属性需要根据内部逻辑或外部事件更改的 Workflow 特别有用。

```java
  import io.temporal.workflow.Workflow;

  ...

  // Existing Workflow Logic

  Map<String, Object> searchAttribute = new HashMap<>();

  Distance distance;
  try {
    distance = activities.getDistance(address);
    searchAttribute.put("isOrderFailed", false);
    Workflow.upsertTypedSearchAttributes(Constants.IS_ORDER_FAILED.valueSet(false));
  } catch (NullPointerException e) {
    searchAttribute.put("isOrderFailed", true);
    Workflow.upsertTypedSearchAttributes(Constants.IS_ORDER_FAILED.valueSet(true));
    throw new NullPointerException("Unable to get distance");
  }
```

<a id="remove-search-attribute"></a>
### 如何从 Workflow 中删除 Search Attribute {#remove-search-attribute}
要删除之前设置的 Search Attribute，请将其设置为空地图。

```java
    // In a shared constants file, so all files have access

    public static final SearchAttributeKey<Boolean> IS_ORDER_FAILED = SearchAttributeKey.forBoolean("isOrderFailed");

    ...

    Workflow.upsertTypedSearchAttributes(Constants.IS_ORDER_FAILED.valueUnset());
```
