[Temporal CLIent](/encyclopedia/temporal-sdks#temporal-client) 使您能够与
[Temporal 服务](/temporal-service)。与 Temporal 服务的通信允许您执行启动等操作
Workflow Executions，发送Signals到Workflow Executions，发送查询到Workflow Executions，得到结果
Workflow Execution，并提供 Activity 任务令牌。

本页向您展示如何使用 Java SDK 和 Temporal CLIent 执行以下操作：

- [连接到本地开发 Temporal 服务](#connect-to-development-service)
- [连接到 Temporal Cloud](#connect-to-temporal-cloud)
- [启动 Workflow Execution](#start-workflow-execution)
- [获取 Workflow 结果](#get-workflow-results)

:::caution

Temporal CLIent 无法在 Workflow 内部初始化和使用。然而，使用
Temporal CLIent 位于 Activity 内部，与 Temporal 服务进行通信。

:::

## 连接到开发 Temporal 服务 {#connect-to-development-service}

使用 `newLocalServiceStubs` 方法创建指向 Temporal 开发服务的存根，然后使用
[`WorkflowClient.newInstance`方法](https://javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/client/WorkflowClient.html#newInstance(io.temporal.serviceclient.WorkflowServiceStubs)）
创建 Temporal CLIent。

<!--SNIPSTART javasdk-build-caller-app-using-local-client {"selectedLines": ["11-17"], "highlightedLines": ["3","7"]}-->
[示例应用程序/java/client/devserver-client-sample/src/main/java/clientsample/YourCallerApp.java](https://github.com/temporalio/documentation/blob/main/sample-apps/java/client/devserver-client-sample/src/main/java/clientsample/YourCallerApp.java)
```java {3,7}
// ...
        // Create an instance that connects to a Temporal Service running on the local
        // machine, using the default port (7233)
        WorkflowServiceStubs serviceStub = WorkflowServiceStubs.newLocalServiceStubs();

        // Initialize the Temporal Client
        // This application uses the Client to communicate with the local Temporal Service
        WorkflowClient client = WorkflowClient.newInstance(serviceStub);
```
<!--SNIPEND-->

当您使用 `newLocalServiceStubs` 实例创建新客户端时，客户端会连接到默认本地端口
在端口 7233 处。当您未指定自定义 Namespace 时，客户端将连接到 `default` Namespace。

要连接到自定义 Namespace，请使用 `WorkflowClientOptions.Builder.setNamespace` 方法设置 Namespace。然后
将 `clientOptions` 传递给 `WorkflowClient.newInstance` 方法。

<TabItem value="config-file" label="Configuration File">

您可以使用 TOML 配置文件来设置 Temporal CLIent 的连接选项。配置文件可以让你
配置多个配置文件，每个配置文件都有自己的一组连接选项。然后，您可以指定在以下情况下使用哪个配置文件：
创建 Temporal CLIent。您可以使用环境变量 `Temporal_CONFIG_FILE` 来指定
TOML 文件或直接在代码中提供文件的路径。如果不提供配置文件路径，SDK
在路径 `~/.config/temporalio/temporal.toml` 中查找它。有关所有可用配置选项的列表，请参阅
至 [环境配置](/references/client-environment-configuration)

:::info

配置文件中设置的连接选项的优先级低于环境变量。这意味着如果
您在配置文件和环境变量中设置相同的选项，即环境变量值
覆盖配置文件中设置的选项。

:::

例如，以下 TOML 配置文件定义了两个配置文件：`default` 和 `prod`。每个配置文件都有自己的
连接选项集。

```toml
# Default profile for local development
[profile.default]
address = "localhost:7233"
namespace = "default"

# Custom gRPC headers
[profile.default.grpc_meta]
my-custom-header = "development-value"
trace-id = "dev-trace-123"

# Production profile for Temporal Cloud
[profile.prod]
address = "your-namespace.a1b2c.tmprl.cloud:7233"
namespace = "your-namespace"
api_key = "your-api-key-here"

# TLS configuration for production
[profile.prod.tls]
# TLS is auto-enabled when this TLS config or API key is present, but you can configure it explicitly
# disabled = false
# Use certificate files for mTLS
client_cert_path = "/etc/temporal/certs/client.pem"
client_key_path = "/etc/temporal/certs/client.key"

# Custom headers for production
[profile.prod.grpc_meta]
environment = "production"
service-version = "v1.2.3"
```

您可以使用配置文件中的特定配置文件创建 Temporal CLIent，如下所示。首次使用
`ClientConfigProfile.load` 从配置文件加载配置文件。然后使用
`profile.toWorkflowServiceStubsOptions` 和 `profile.toWorkflowClientOptions` 将配置文件转换为
分别为 `WorkflowServiceStubsOptions` 和 `WorkflowClientOptions`。然后使用`WorkflowClient.newInstance`创建
Temporal CLIent。

```java {21-25,32-34}
import io.temporal.client.WorkflowClient;
import io.temporal.client.WorkflowClientOptions;
import io.temporal.envconfig.ClientConfigProfile;
import io.temporal.envconfig.LoadClientConfigProfileOptions;
import io.temporal.serviceclient.WorkflowServiceStubs;
import io.temporal.serviceclient.WorkflowServiceStubsOptions;
import java.nio.file.Paths;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class LoadFromFile {

  private static final Logger logger = LoggerFactory.getLogger(LoadFromFile.class);

  public static void main(String[] args) {
    try {

      String configFilePath =
          Paths.get(LoadFromFile.class.getResource("/config.toml").toURI()).toString();

      ClientConfigProfile profile =
          ClientConfigProfile.load(
              LoadClientConfigProfileOptions.newBuilder()
                  .setConfigFilePath(configFilePath)
                  .build());

      WorkflowServiceStubsOptions serviceStubsOptions = profile.toWorkflowServiceStubsOptions();
      WorkflowClientOptions clientOptions = profile.toWorkflowClientOptions();

      try {
        // Create the workflow client using the loaded configuration
        WorkflowClient client =
            WorkflowClient.newInstance(
                WorkflowServiceStubs.newServiceStubs(serviceStubsOptions), clientOptions);

        // Test the connection by getting system info
        var systemInfo =
            client
                .getWorkflowServiceStubs()
                .blockingStub()
                .getSystemInfo(
                    io.temporal.api.workflowservice.v1.GetSystemInfoRequest.getDefaultInstance());

        logger.info("✅ Client connected successfully!");
        logger.info("   Server version: {}", systemInfo.getServerVersion());

      } catch (Exception e) {
        logger.error("❌ Failed to connect: {}", e.getMessage());
      }

    } catch (Exception e) {
      logger.error("Failed to load configuration: {}", e.getMessage(), e);
      System.exit(1);
    }
  }
}
```

## 连接到 Temporal Cloud {#connect-to-temporal-cloud}

您可以使用 [API密钥](/cloud/api-keys) 或通过 [mTLS](/cloud/certificates) 连接到 Temporal Cloud。
与 Temporal Cloud 或任何安全的 Temporal 服务的连接需要额外的连接选项
连接到不安全的本地开发实例：

- 您的身份验证凭据。
  - 如果您使用 API 密钥，请提供 API 密钥值。
  - 如果您使用 mTLS，请提供 mTLS CA 证书和 mTLS 私钥。
- 您的 _Namespace 和帐户 ID_ 组合，格式为 `<Namespace_id>.<account_id>`。
- 建议的_端点_是 gRPC Namespace 端点：`<Namespace>.<account>.tmprl.cloud:7233`。
  此端点适用于所有 Namespace，并自动将流量定向到具有 [高可用性](/cloud/high-availability) 的 Namespace 的活动区域。
  有关端点选项的更多信息，请参阅 [访问 Namespace](/cloud/namespaces#access-namespaces)。

您可以在 Namespaces 选项卡上找到 Namespace 和账户 ID 以及终端节点：

![左侧为 Namespace 和账户 ID 组合，右侧为区域终端节点](/img/cloud/apikeys/namespaces-and-regional-endpoints.png)

您可以使用环境变量、配置文件或直接在代码中提供这些连接选项。

<TabItem value="config-file" label="Configuration File">

您可以使用 TOML 配置文件来设置 Temporal CLIent 的连接选项。配置文件可以让你
配置多个配置文件，每个配置文件都有自己的一组连接选项。然后，您可以指定在以下情况下使用哪个配置文件：
创建 Temporal CLIent。有关可在 TOML 文件中设置的所有可用配置选项的列表，请参阅
[环境配置](/references/client-environment-configuration)。

您可以使用环境变量 `Temporal_CONFIG_FILE` 指定 TOML 文件的位置或提供路径
直接在代码中写入文件。如果您不提供配置文件的路径，SDK 将在
默认路径 `~/.config/temporalio/temporal.toml`。

:::info

配置文件中设置的连接选项的优先级低于环境变量。这意味着如果
您在配置文件和环境变量中设置相同的选项，即环境变量值
覆盖配置文件中设置的选项。

:::

例如，以下 TOML 配置文件定义了 `cloud` 配置文件，其中包含必要的连接选项
通过 API 密钥连接到 Temporal Cloud：

```toml
# Cloud profile for Temporal Cloud
[profile.cloud]
address = "your-namespace.a1b2c.tmprl.cloud:7233"
namespace = "your-namespace"
api_key = "your-api-key-here"
```

如果您想使用 mTLS 身份验证而不是 API 密钥，请将 `api_key` 字段替换为您的 mTLS 证书，然后
私钥：

```toml
# Cloud profile for Temporal Cloud
[profile.cloud]
address = "your-namespace.a1b2c.tmprl.cloud:7233"
namespace = "your-namespace"
tls_client_cert_data = "your-tls-client-cert-data"
tls_client_key_path = "your-tls-client-key-path"
```

使用配置文件中定义的连接选项，使用 `envconfig` 中的 `LoadClientOptions` 函数
包以使用 `cloud` 配置文件创建 Temporal CLIent，如下所示。加载配置文件后，您还可以
在创建客户端之前以编程方式覆盖特定的连接选项。

```java {25-30,33-35,42-44}
import io.temporal.client.WorkflowClient;
import io.temporal.client.WorkflowClientOptions;
import io.temporal.envconfig.ClientConfigProfile;
import io.temporal.envconfig.LoadClientConfigProfileOptions;
import io.temporal.serviceclient.WorkflowServiceStubs;
import io.temporal.serviceclient.WorkflowServiceStubsOptions;
import java.nio.file.Paths;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class LoadProfile {

  private static final Logger logger = LoggerFactory.getLogger(LoadProfile.class);

  public static void main(String[] args) {
    String profileName = "cloud";

    try {
      String configFilePath =
          Paths.get(LoadProfile.class.getResource("/config.toml").toURI()).toString();

      logger.info("--- Loading '{}' profile from {} ---", profileName, configFilePath);

      // Load specific profile from file with environment variable overrides
      ClientConfigProfile profile =
          ClientConfigProfile.load(
              LoadClientConfigProfileOptions.newBuilder()
                  .setConfigFilePath(configFilePath)
                  .setConfigFileProfile(profileName)
                  .build());

      // Demonstrate programmatic override - fix the incorrect address from staging profile
      ClientConfigProfile.Builder profileBuilder = profile.toBuilder();
      profileBuilder.setAddress("localhost:7233"); // Override the incorrect address
      profile = profileBuilder.build();

      WorkflowServiceStubsOptions serviceStubsOptions = profile.toWorkflowServiceStubsOptions();
      WorkflowClientOptions clientOptions = profile.toWorkflowClientOptions();

      try {
        // Create the workflow client using the loaded configuration

            WorkflowClient.newInstance(
                WorkflowServiceStubs.newServiceStubs(serviceStubsOptions), clientOptions);

        // Test the connection by getting system info

            client
                .getWorkflowServiceStubs()
                .blockingStub()
                .getSystemInfo(
                    io.temporal.api.workflowservice.v1.GetSystemInfoRequest.getDefaultInstance());

        logger.info("✅ Client connected successfully!");
        logger.info("   Server version: {}", systemInfo.getServerVersion());

      } catch (Exception e) {
        logger.error("❌ Failed to connect: {}", e.getMessage());
      }

    } catch (Exception e) {
      logger.error("Failed to load configuration: {}", e.getMessage(), e);
      System.exit(1);
    }
  }
}
```

连接到 Temporal Cloud 需要以下环境变量：

- `Temporal_Namespace`：您的 Namespace 和帐户 ID 组合，格式为 `<Namespace_id>.<account_id>`。
- `Temporal_ADDRESS`：Temporal Cloud Namespace 的 gRPC 端点。

  如果您使用 mTLS 身份验证，则为必需。

  如果您使用 mTLS 身份验证，则为必需。

## 启动 Workflow Execution {#start-workflow-execution}

[Workflow Execution](/workflow-execution) 语义依赖于几个参数，即启动 Workflow Execution
您必须提供将用于任务的 Task Queue（Worker 正在轮询的任务）、Workflow 类型、
特定于语言的上下文数据和 Workflow 函数参数。

在下面的示例中，所有 Workflow Execution 均使用 Temporal CLIent 启动。从中生成 Workflow Execution
在另一个 Workflow Execution 中，使用 [Child Workflow](/develop/java/workflows/child-workflows) 或外部 Workflow
API。

请参阅 [定制Workflow型](/develop/java/workflows/basics#workflow-type) 部分以了解如何自定义名称
Workflow 型的。

生成 Workflow Execution 的请求会导致 Temporal 服务创建第一个事件
([Workflow执行开始](/references/events#workflowexecutionstarted)) 在 Workflow Execution Event History 中。的
然后，Temporal 服务创建第一个 Workflow 任务，从而产生第一个
[Workflow任务Scheduled](/references/events#workflowtaskscheduled) 事件。

使用 `WorkflowStub` 从客户端内启动 Workflow Execution，并使用 `ExternalWorkflowStub` 启动不同的
来自 Workflow 内的 Workflow Execution。

请参阅 [`SignalwithStart`](/develop/java/workflows/message-passing#signal-with-start) 启动 Workflow Execution 来接收
Signal from within another Workflow.

**使用 `WorkflowStub`**

`WorkflowStub` 是由 `WorkflowClient` 生成的代理。每次启动新的 Workflow Execution 时，都会生成一个实例
Workflow 实现对象的创建。然后，其中一种方法（取决于 Workflow 的类型）
可以调用用 `@WorkflowMethod` 注释的实例）。此方法一返回，Workflow Execution 就会
被认为是完整的。

您可以在客户端代码中使用类型化或非类型化的 `WorkflowStub`。

- 类型化的 `WorkflowStub` 很有用，因为它们是类型安全的，并允许您调用 Workflow 方法，例如
  直接 `@WorkflowMethod`、`@QueryMethod` 和 `@SignalMethod`。
- 非类型化 `WorkflowStub` 不使用 Workflow 接口，并且不是类型安全的。它更加灵活，因为它
  具有来自 `WorkflowStub` 接口的方法，例如 `start`、`SignalWithStart`、`getResults`（同步和异步），
  `Query`、`Signal`、`cancel` 和 `terminate`。请注意，Temporal Java SDK 还提供类型为 `WorkflowStub`
  这些方法的版本。当使用无类型的 `WorkflowStub` 时，我们依赖于 Workflow 类型、Activity 类型、子代
  Workflow 类型，以及 Query 和 Signal 名称。有关详细信息，请参见 [Temporal CLIent](#connect-to-development-service)。

Workflow Execution 可以同步或异步启动。

- 同步调用启动 Workflow，然后等待其完成。如果启动 Workflow 的进程
  崩溃或停止等待，Workflow 继续执行。因为 Workflow 可能会长时间运行，并且客户端
  崩溃发生，在生产使用中并不常见。以下示例是类型安全的方法
  同步启动 Workflow Execution。

  ```java
    NotifyUserAccounts workflow = client.newWorkflowStub(
          NotifyUserAccounts.class,
          WorkflowOptions.newBuilder()
                  .setWorkflowId("notifyAccounts")
                  .setTaskQueue(taskQueue)
                  .build()
          );

  // start the Workflow and wait for a result.
    workflow.notify(new String[] { "Account1", "Account2", "Account3", "Account4", "Account5",
                  "Account6", "Account7", "Account8", "Account9", "Account10"});
      }
  // notify(String[] accountIds) is a Workflow method defined in the Workflow Definition.
  ```

- 异步启动启动 Workflow Execution 并立即返回给调用者。这是最常见的方式
  在生产代码中启动 Workflows。的
  [`WorkflowClient`](https://github.com/temporalio/sdk-java/blob/master/temporal-sdk/src/main/java/io/temporal/client/WorkflowClient.java)
  提供了一些静态方法，例如 `start`、`execute`、`SignalWithStart` 等，帮助您启动
  Workflow 异步。

以下示例显示如何使用类型化或非类型化异步启动 Workflow Execution
  `WorkflowStub`。
  - **键入 WorkflowStub 示例**

    ```java
    // create typed Workflow stub
    FileProcessingWorkflow workflow = client.newWorkflowStub(FileProcessingWorkflow.class,
          WorkflowOptions.newBuilder()
                  .setTaskQueue(taskQueue)
                  .setWorkflowId(workflowId)
                  .build());
    // use WorkflowClient.execute to return future that contains Workflow result or failure, or
    // use WorkflowClient.start to return WorkflowId and RunId of the started Workflow).
    WorkflowClient.start(workflow::greetCustomer);
    ```

- **无类型 WorkflowStub 示例**

    ```java
    WorkflowStub untyped = client.newUntypedWorkflowStub("FileProcessingWorkflow",
          WorkflowOptions.newBuilder()
                  .setWorkflowId(workflowId)
                  .setTaskQueue(taskQueue)
                  .build());

    // blocks until Workflow Execution has been started (not until it completes)
    untyped.start(argument);
    ```

您可以使用非类型化 `WorkflowStub` 调用动态 Workflow 实现。下面的例子展示了如何调用
客户端代码中的动态 Workflow 实现。

```java
    WorkflowClient client = WorkflowClient.newInstance(service);
    /**
      * Note that for this part of the client code, the dynamic Workflow implementation must
      * be known to the Worker at runtime in order to dispatch Workflow tasks, and may be defined
      * in the Worker definition as:*/
    // worker.registerWorkflowImplementationTypes(DynamicGreetingWorkflowImpl.class);

    /* Create the Workflow stub to call the dynamic Workflow.
    * Note that the Workflow Type is not explicitly registered with the Worker.*/
    WorkflowOptions workflowOptions =
        WorkflowOptions.newBuilder().setTaskQueue(TASK_QUEUE).setWorkflowId(WORKFLOW_ID).build();
    WorkflowStub workflow = client.newUntypedWorkflowStub("DynamicWF", workflowOptions);
```

`DynamicWorkflow` 可用于调用不同的 Workflow 类型。检查 Dynamic 时运行的类型
Workflow `execute`方法运行，在实现代码中使用`getWorkflowType()`。

```java
String type = Workflow.getInfo().getWorkflowType();
```

有关如何获取 Workflow Execution 结果的详细信息，请参阅 [Workflow Execution 结果](#get-workflow-results)。

**使用 `ExternalWorkflowStub`**

在 Workflow 中使用 `ExternalWorkflowStub` 来调用 Signal，并将 Signal 按类型发送到其他 Workflow。

这对于执行用其他语言 SDK 编写的 Workflow 特别有帮助，如以下示例所示。

```java
@Override
  public String yourWFMethod(String name) {
      ExternalWorkflowStub callOtherWorkflow = Workflow.newUntypedExternalWorkflowStub("OtherWFId");
    }
```

有关执行 Workflow 的示例，请参阅 [Temporal 多语言](https://github.com/tsurdilo/temporal-polyglot) 代码
用其他语言的 SDK 编写。

#### WorkflowID重用策略 {#workflowidreusepolicy}

- 型号：`WorkflowIdReusePolicy`
- 默认：`AllowDuplicate`
- 价值观：
  - `enums.AllowDuplicateFailedOnly`：如果较早的 Workflow Execution 失败、取消或，则 Workflow 可以启动
    终止。
  - `AllowDuplicate`：无论先前执行的关闭状态如何，Workflow 都可以启动。
  - `RejectDuplicate`：如果有较早的运行，则 Workflow 无法启动。

```java

    WorkerGreet.greetclient.newWorkflowStub(
        GreetWorkflowInterface.class,
        WorkflowOptions.newBuilder()
                .setWorkflowId("GreetWF")
                .setTaskQueue(WorkerGreet.TASK_QUEUE)
                // Set Workflow Id Reuse Policy
                .setWorkflowIdReusePolicy(
                        WorkflowIdReusePolicy.WORKFLOW_ID_REUSE_POLICY_REJECT_DUPLICATE)
                .build());
```

<a id="get-workflow-results"></a>
### 如何在 Java 中获取 Workflow Execution 的结果 {#get-workflow-results}

如果启动 Workflow Execution 的调用成功，您将获得对 Workflow Execution 的运行 ID 的访问权限。

Workflow Id、Run Id 和 Namespace 可用于唯一标识系统中的 Workflow Execution 并获取其
结果。

可以阻止结果的进展（同步执行）或在其他时间点获取结果
（异步执行）。

在 Temporal 平台中，也可以使用查询作为访问状态和数据的首选方法。
Workflow Execution 的结果。

同步 Workflow Execution 会阻塞您的客户端线程，直到 Workflow Execution 完成（或失败）并获取
结果（或失败时的错误）。

以下示例是用于获取同步 Workflow Execution 结果的类型安全方法。

```java
 FileProcessingWorkflow workflow = client.newWorkflowStub(
                FileProcessingWorkflow.class,
                WorkflowOptions.newBuilder()
                        .setWorkflowId(workflowId)
                        .setTaskQueue(taskQueue)
                        .build();

// start sync and wait for results (or failure)
String result = workflow.processfile(new Argument());
```

异步 Workflow Execution 立即向调用者返回一个值。

以下示例演示如何通过类型化和非类型化 `WorkflowStub` 获取 Workflow Execution 的结果。

- **键入 WorkflowStub 示例**

  ```java
  // create typed Workflow stub
  FileProcessingWorkflow workflow = client.newWorkflowStub(FileProcessingWorkflow.class,
                WorkflowOptions.newBuilder()
                        .setTaskQueue(taskQueue)
                        .setWorkflowId(workflowId)
                        .build());
  // use WorkflowClient.execute (if your Workflow takes in arguments) or WorkflowClient.start (for zero arguments)
  WorkflowClient.start(workflow::greetCustomer);
  ```

- **无类型 WorkflowStub 示例**

  ```java
  WorkflowStub untyped = client.newUntypedWorkflowStub("FileProcessingWorkflow",
                  WorkflowOptions.newBuilder()
                          .setWorkflowId(workflowId)
                          .setTaskQueue(taskQueue)
                          .build());

  // blocks until Workflow Execution has been started (not until it completes)
  untyped.start(argument);
  ```

如果您需要在异步启动后等待 Workflow Execution 完成，最直接的方法是
再次调用阻塞的 Workflow 实例。

请注意，如果 `WorkflowOptions.WorkflowIdReusePolicy` 未设置为 `AllowDuplicate`，则不抛出
`DuplicateWorkflowException`，它重新连接到现有的 Workflow 并等待其完成。

以下示例显示如何从启动 Workflow Execution 的进程不同的进程执行此操作。

```java
YourWorkflow workflow = client.newWorkflowStub(YourWorkflow.class, workflowId);

// Returns the result after waiting for the Workflow to complete.
String result = workflow.yourMethod();
```

连接到现有 Workflow 并等待另一个进程完成的另一种方法是使用
`UntypedWorkflowStub`。例如：

```java
WorkflowStub workflowStub = client.newUntypedWorkflowStub(workflowType, workflowOptions);

// Returns the result after waiting for the Workflow to complete.
String result = untyped.getResult(String.class);
```

**获取最后（成功）完成结果**

对于 Temporal Cron 作业，使用 `GetLastCompletionResult()` 获取之前成功运行的结果。方法
如果没有先前的完成，则返回 `null`。以下示例显示如何在 Workflow 中实现此功能。

```java
public String cronWorkflow() {
    String lastProcessedFileName = Workflow.getLastCompletionResult(String.class);

    // Process work starting from the lastProcessedFileName.
    // Business logic implementation goes here.
    // Updates lastProcessedFileName to the new value.

    return lastProcessedFileName;
}
```

请注意，即使 Cron Schedule 之一运行失败，此操作也有效。下一个Schedule仍然会取得最后的成功
如果它至少成功完成一次，就会得到结果。例如，对于每日 cron Workflow，如果运行成功
第一天失败，第二天失败，然后第三天运行将使用这些获得第一天运行的结果
API。
