<a id="worker-versioning"></a>
## 如何在 Java 中使用 Worker 版本控制（已弃用） {#worker-versioning}

:::caution

本节适用于已弃用的 Worker 版本控制 API。请将您的注意力转向[Worker 版本控制](/production-deployment/worker-deployments/worker-versioning)。

请参阅 [预发布自述文件](https://github.com/temporalio/temporal/blob/main/docs/worker-versioning.md) 了解更多信息。

:::
构建 ID 对应于部署。如果您还没有，我们建议您将代码的哈希值（例如 Git SHA）与人类可读的时间戳相结合。
要使用 Worker 版本控制，您需要将构建 ID 传递给 Java Worker 并选择加入 Worker 版本控制。

<a id="assign-a-build-id-to-your-worker-and-opt-in-to-worker-versioning"></a>
### 为您的 Worker 分配构建 ID 并选择加入 Worker 版本控制 {#assign-a-build-id-to-your-worker-and-opt-in-to-worker-versioning}

在完成此步骤之前，您应该了解分配规则。
请参阅 [Worker 版本控制预发布自述文件](https://github.com/temporalio/temporal/blob/main/docs/worker-versioning.md) 了解更多信息。

要为 Worker 启用 Worker 版本控制，请分配构建 ID（可能来自环境变量）并将其打开。

```java
// ...
WorkerOptions workerOptions = WorkerOptions.newBuilder()
    .setBuildId(buildId)
    .setUseBuildIdForVersioning(true)
    // ...
    .build();
Worker w = workerFactory.newWorker("your_task_queue_name", workerOptions);
// ...
```

:::warning

重要的是，当您启动此 Worker 时，在您设置分配规则之前，它不会接收任何任务。

:::

<a id="specify-versions-for-activities-child-workflows-and-continue-as-new"></a>
### 指定活动、Child Workflow 和 Continue-As-New 的版本 {#specify-versions-for-activities-child-workflows-and-continue-as-new}

:::caution

Java对此功能的支持正在建设中！

:::

默认情况下，如果活动、Child Workflow 和 Continue-As-New Workflow 也配置为在同一 Task Queue 上运行，则它们将在创建它们的 Workflow 的版本上运行。
当配置为在单独的 Task Queue 上运行时，它们将默认使用当前分配规则。

如果要覆盖此行为，可以通过 `ActivityOptions`、`ChildWorkflowOptions` 或 `ContinueAsNewOptions` 对象上的 `setVersioningIntent` 方法指定您的意图。

例如，如果您希望 Activity 使用最新的分配规则而不是从其父级继承：

```java
// ...
private final MyActivity activity =
    Workflow.newActivityStub(
        MyActivity.class,
        ActivityOptions.newBuilder()
          .setScheduleToCloseTimeout(Duration.ofSeconds(10))
          .setVersioningIntent(VersioningIntent.VERSIONING_INTENT_USE_ASSIGNMENT_RULES)
          // ...other options
          .build()
    );
// ...
```

<a id="tell-the-task-queue-about-your-workers-build-id-deprecated"></a>
### 告诉 Task Queue 您的 Worker 的内部版本 ID（已弃用） {#tell-the-task-queue-about-your-workers-build-id-deprecated}

:::caution

本节适用于已弃用的 Worker 版本控制 API。请将您的注意力转向[Worker 版本控制](/production-deployment/worker-deployments/worker-versioning)。

:::

现在您可以使用 SDK（或 Temporal CLI）向 Task Queue 告知您的 Worker 的构建 ID。
您可能希望将其作为 CI 部署过程的一部分来执行。

```java
// ...
workflowClient.updateWorkerBuildIdCompatability(
    "your_task_queue_name", BuildIdOperation.newIdInNewDefaultSet("deadbeef"));
```

此代码将 `deadbeef` 构建 ID 添加到 Task Queue 作为新版本集中的唯一版本，这将成为队列的默认值。
新的 Workflow 在具有此构建 ID 的 Worker 上执行，现有的 Workflow 将继续由适当兼容的 Worker 进行处理。

相反，如果您想将构建 ID 添加到现有的兼容集中，您可以执行以下操作：

```java
// ...
workflowClient.updateWorkerBuildIdCompatability(
    "your_task_queue_name", BuildIdOperation.newCompatibleVersion("deadbeef", "some-existing-build-id"));
```

此代码将 `deadbeef` 添加到包含 `some-existing-build-id` 的现有兼容集，并将其标记为该集的新默认构建 ID。

您还可以将集合中的现有构建 ID 提升为该集合的默认值：

```java
// ...
workflowClient.updateWorkerBuildIdCompatability(
    "your_task_queue_name", BuildIdOperation.promoteBuildIdWithinSet("deadbeef"));
```

您还可以将整个集合提升为队列的默认集合。新的 Workflow 将开始使用该集的默认值。

```java
// ...
workflowClient.updateWorkerBuildIdCompatability(
    "your_task_queue_name", BuildIdOperation.promoteSetByBuildId("deadbeef"));
```
