Temporal 应用程序开发指南的测试部分描述了促进 Workflow 和集成测试的框架。

在 Temporal 的上下文中，您可以创建以下类型的自动化测试：

- **端到端：** 运行 Temporal Server 和 Worker 及其所有 Workflow、活动和 Nexus 操作；从客户端启动 Workflow 并与之交互。
- **集成：** 端到端和单元测试之间的任何内容。
  - 使用模拟上下文和其他 SDK 导入（通常是网络请求）运行活动。
  - 通过模拟活动和 Nexus 操作运行 Worker，并使用客户端启动 Workflow。
  - 使用模拟 SDK 导入运行 Workflow。
- **单位：** 运行一段 Workflow、Activity 或 Nexus 操作代码（函数或方法）并模拟它调用的任何代码。

我们通常建议将大部分测试编写为集成测试。

由于测试服务器支持跳过时间，因此请使用测试服务器与 Worker 进行端到端测试和集成测试。

## 测试框架 {#test-frameworks}

Temporal Java SDK 提供了一个测试框架来促进 Workflow 单元和集成测试。
测试框架提供了一个 `TestWorkflowEnvironment` 类，其中包括内存中实现
支持自动跳时的Temporal服务。这可以让您
在几秒钟内轻松测试长时间运行的 Workflow，而无需更改您的 Workflow 代码。

您可以将提供的 `TestWorkflowEnvironment` 与您选择的 Java 单元测试框架一起使用，
例如JUnit。

<a id="setup-testing-dependency"></a>
### 设置测试依赖 {#setup-testing-dependency}

要开始使用Java SDK测试框架，您需要添加[`io.Temporal:Temporal-testing`](https://search.maven.org/artifact/io.temporal/temporal-testing)
作为您项目的依赖项：

**[阿帕奇Maven](https://maven.apache.org/)：**

```maven
<dependency>
    <groupId>io.temporal</groupId>
    <artifactId>temporal-testing</artifactId>
    <version>1.17.0</version>
    <scope>test</scope>
</dependency>
```

**[Gradle Groovy DSL](https://gradle.org/)：**

```groovy
testImplementation ("io.temporal:temporal-testing:1.17.0")
```

确保设置的版本与您的 [Temporal Java SDK](https://github.com/temporalio/sdk-java) 依赖项版本相匹配。

<a id="sample-unit-tests"></a>
### 单元测试示例 {#sample-unit-tests}

以下代码实现 `HelloActivity` 示例的单元测试：

```java
public class HelloActivityTest {

    private TestWorkflowEnvironment testEnv;
    private Worker worker;
    private WorkflowClient client;

    // Set up the test workflow environment
    @Before
    public void setUp() {
        testEnv = TestWorkflowEnvironment.newInstance();
        worker = testEnv.newWorker(TASK_QUEUE);
        // Register your workflow implementations
        worker.registerWorkflowImplementationTypes(GreetingWorkflowImpl.class);

        client = testEnv.getWorkflowClient();
    }

    // Clean up test environment after tests are completed
    @After
    public void tearDown() {
        testEnv.close();
    }

    @Test
    public void testActivityImpl() {
        // This uses the actual activity impl
        worker.registerActivitiesImplementations(new GreetingActivitiesImpl());

        // Start test environment
        testEnv.start();

        // Create the workflow stub
        GreetingWorkflow workflow =
                client.newWorkflowStub(
                        GreetingWorkflow.class, WorkflowOptions.newBuilder().setTaskQueue(TASK_QUEUE).build());

        // Execute our workflow waiting for it to complete
        String greeting = workflow.getGreeting("World");
        assertEquals("Hello World!", greeting);
    }
}
```

如果您不希望在执行期间执行实际的 Activity 或 Nexus 操作实施
单元测试时，您可以使用 Mockito 等框架来模拟它们。

以下代码实现了 `HelloActivity` 示例的单元测试，显示
如何模拟活动：

```java
public class HelloActivityTest {

    private TestWorkflowEnvironment testEnv;
    private Worker worker;
    private WorkflowClient client;

    // Set up the test workflow environment
    @Before
    public void setUp() {
        testEnv = TestWorkflowEnvironment.newInstance();
        worker = testEnv.newWorker(TASK_QUEUE);
        // Register your workflow implementations
        worker.registerWorkflowImplementationTypes(GreetingWorkflowImpl.class);

        client = testEnv.getWorkflowClient();
    }

    // Clean up test environment after tests are completed
    @After
    public void tearDown() {
        testEnv.close();
    }

    @Test
    public void testMockedActivity() {
        // Mock our workflow activity
        GreetingActivities activities = mock(GreetingActivities.class);
        when(activities.composeGreeting("Hello", "World")).thenReturn("Hello Mocked World!");
        worker.registerActivitiesImplementations(activities);

        // Start test environment
        testEnv.start();

        // Create the workflow stub
        GreetingWorkflow workflow =
                client.newWorkflowStub(
                        GreetingWorkflow.class, WorkflowOptions.newBuilder().setTaskQueue(TASK_QUEUE).build());

        // Execute our workflow waiting for it to complete
        String greeting = workflow.getGreeting("World");
        assertEquals("Hello Mocked World!", greeting);
    }
}
```

<a id="testing-with-junit4"></a>
### 使用 JUnit4 进行测试 {#testing-with-junit4}

对于 Junit4 测试，Temporal 提供了 TestWorkflowRule 类，该类简化了 Temporal 测试环境设置以及
在测试中创建和关闭 Workflow Worker。

确保设置的版本与您的 [Temporal Java SDK](https://github.com/temporalio/sdk-java) 依赖项版本相匹配。

我们现在可以重写上面提到的“HelloActivityTest”测试类，如下所示：

```java
public class HelloActivityJUnit4Test {
    @Rule
    public TestWorkflowRule testWorkflowRule =
            TestWorkflowRule.newBuilder()
                    .setWorkflowTypes(GreetingWorkflowImpl.class)
                    .setActivityImplementations(new GreetingActivitiesImpl())
                    .build();

    @Test
    public void testActivityImpl() {
        // Get a workflow stub using the same task queue the worker uses.
        GreetingWorkflow workflow =
                testWorkflowRule
                        .getWorkflowClient()
                        .newWorkflowStub(
                                GreetingWorkflow.class,
                                WorkflowOptions.newBuilder().setTaskQueue(testWorkflowRule.getTaskQueue()).build());
        // Execute a workflow waiting for it to complete.
        String greeting = workflow.getGreeting("World");
        assertEquals("Hello World!", greeting);

        testWorkflowRule.getTestEnvironment().shutdown();
    }
}
```


## 测试活动 {#test-activities}

模拟隔离了正在测试的代码，因此重点仍然放在代码上，而不是外部依赖项或其他状态。您可以使用模拟的 Activity 环境来测试活动。

此方法提供了一种模拟 Activity 上下文、监听心跳和取消 Activity 的方法。您可以单独测试 Activity，直接调用它，而不需要创建 Worker 来运行它。

Temporal 提供 `TestActivityEnvironment` 和 `TestActivityExtension` 类，用于测试 Workflow 范围之外的活动。测试
活动类似于测试非 Temporal Java 代码。

例如，您可以测试 Activity：

- 调用 Activity 执行时引发异常。
- 检查 Activity 执行结果时引发异常。
- Activity 的返回值。检查返回值是否与预期值匹配。

<a id="run-an-activity"></a>
### 运行 Activity {#run-an-activity}

在隔离测试期间，如果 Activity 引用其上下文，您将需要模拟该上下文。
模拟信息代表上下文，使您可以将测试集中在 Activity 的代码上。

<a id="listen-to-heartbeats"></a>
### 聆听心跳 {#listen-to-heartbeats}

活动通常会发出周期性的心跳，这是一种广播重复生命证明 Update 的功能。
每个 ping 都显示 Activity 正在取得进展，而 Worker 尚未崩溃。
心跳可能包括在 Activity Worker 崩溃时报告任务进度的详细信息。

在测试支持心跳的活动时，请确保您可以在测试代码中看到这些心跳。
提供适当的测试覆盖率。
这使您能够验证心跳的内容和行为。

<a id="cancel-an-activity"></a>
### 取消 Activity {#cancel-an-activity}

Activity 取消让活动知道它们不需要继续工作，并为 Activity 提供时间来清理其创建的任何资源。如果基于 Java 的活动发出心跳，您可以取消它们。要测试 Activity 对取消的反应，请确保 Activity 正确反应并取消。

<a id="mock-nexus-operations"></a>
### 如何模拟 Nexus 操作 {#mock-nexus-operations}

当使用 Worker 集成测试 Workflow 时，您可以通过向 Worker 提供模拟 Nexus 服务处理程序来模拟 Nexus 操作。
或者，您可以模拟 Nexus 服务本身。

您可以在 [这个测试包](https://github.com/temporalio/samples-java) 的 [Temporal Java 示例](https://github.com/temporalio/samples-java/tree/main/core/src/test/java/io/temporal/samples/nexus/caller) 存储库中找到 Nexus 的示例单元测试。
这些示例展示了如何使用 Temporal 测试包在测试中调用 Nexus 服务，以及如何针对 JUnit 4 和 5 模拟它们。
详细的解释性注释包含在存储库的代码中。

要模拟 Nexus 处理程序，请从 Temporal 测试包创建规则（对于 JUnit4）或扩展（对于 JUnit5），就像在 [单元测试样本](#sample-unit-tests) 中一样，并向构建器添加对 `setNexusServiceImplementation` 的调用。
这将设置测试所需的 Nexus 端点以及由 Nexus 服务实现定义的 Nexus 处理程序 Workflow。
一切都是由 Temporal 测试包创建和设置的，因此不需要更多的工作！

您需要像往常一样使用 `setWorkflowTypes`（对于 JUnit4）或 `registerWorkflowImplementationTypes`（对于 JUnit5）为每个处理程序创建 Worker。
完成后，您就可以像任何其他 Workflow 一样模拟 Nexus 端点 - 再次，就像上面的 [示例单元测试](#sample-unit-tests) 一样。

以下是来自 [测试包](https://github.com/temporalio/samples-java/tree/main/core/src/test/java/io/temporal/samples/nexus/caller) 的示例来证明这一点。

#### 使用 JUnit4 模拟 Nexus 处理程序 {#mocking-nexus-handlers-with-junit4}

[核心/src/test/java/io/temporal/samples/nexus/caller/CallerWorkflowMockTest.java](https://github.com/temporalio/samples-java/blob/main/core/src/test/java/io/temporal/samples/nexus/caller/CallerWorkflowMockTest.java)

```java
public class CallerWorkflowMockTest {
  @Rule
  public TestWorkflowRule testWorkflowRule =
      TestWorkflowRule.newBuilder()
          .setNexusServiceImplementation(new SampleNexusServiceImpl())
          .setWorkflowTypes(HelloCallerWorkflowImpl.class)

  @Test

              return wf;

    // Now create the caller workflow

        testWorkflowRule
            .getWorkflowClient()
            .newWorkflowStub(
                HelloCallerWorkflow.class,
                WorkflowOptions.newBuilder().setTaskQueue(testWorkflowRule.getTaskQueue()).build());
    String greeting = workflow.hello("World", SampleNexusService.Language.EN);
    assertEquals("Hello Mock World", greeting);

  }
```

<a id="mocking-the-nexus-service-with-junit4"></a>
#### 使用 JUnit4 模拟 Nexus 服务 {#mocking-the-nexus-service-with-junit4}
{/* SNIPSTART java-nexus-service-sample-junit4-mock */}
[核心/src/test/java/io/temporal/samples/nexus/caller/nexusServiceMockTest.java](https://github.com/temporalio/samples-java/blob/main/core/src/test/java/io/temporal/samples/nexus/caller/nexusServiceMockTest.java)
```java
public class NexusServiceMockTest {

  private final SampleNexusService mockNexusService = mock(SampleNexusService.class);

  /**
   * A test-only Nexus service implementation that delegates to the Mockito mock defined above. The
   * operation is implemented as a synchronous handler that forward calls to the mock, allowing
   * full control over return values and verification of inputs.
   */
  @ServiceImpl(service = SampleNexusService.class)
  public class TestNexusServiceImpl {

    @OperationImpl
    @SuppressWarnings("DirectInvocationOnMock")
    public OperationHandler<SampleNexusService.HelloInput, SampleNexusService.HelloOutput> hello() {
      return OperationHandler.sync((ctx, details, input) -> mockNexusService.hello(input));
    }
  }

  // Using OperationHandler.sync for the operation bypasses the need for a backing workflow,
  // returning results inline just like a synchronous call.

  @Rule
  public TestWorkflowRule testWorkflowRule =
      TestWorkflowRule.newBuilder()
          .setNexusServiceImplementation(new TestNexusServiceImpl())
          .setWorkflowTypes(HelloCallerWorkflowImpl.class)
          .build();

  @Test
  public void testHelloCallerWithMockedService() {
    when(mockNexusService.hello(any()))
        .thenReturn(new SampleNexusService.HelloOutput("Bonjour World"));

    HelloCallerWorkflow workflow =
        testWorkflowRule
            .getWorkflowClient()
            .newWorkflowStub(
                HelloCallerWorkflow.class,
                WorkflowOptions.newBuilder().setTaskQueue(testWorkflowRule.getTaskQueue()).build());

    String result = workflow.hello("World", SampleNexusService.Language.FR);
    assertEquals("Bonjour World", result);

    // Verify the Nexus service was called with the correct name and language
    verify(mockNexusService)
        .hello(
            argThat(
                input ->
                    "World".equals(input.getName())
                        && SampleNexusService.Language.FR == input.getLanguage()));

    // Verify the operation was called exactly once and no other operations were invoked
    verify(mockNexusService, times(1)).hello(any());
  }
}
```

#### 使用 JUnit5 模拟 Nexus 服务 {#mocking-the-nexus-service-with-junit5}

[核心/src/test/java/io/temporal/samples/nexus/caller/nexusServiceJunit5Test.java](https://github.com/temporalio/samples-java/blob/main/core/src/test/java/io/temporal/samples/nexus/caller/nexusServiceJunit5Test.java)
```java
public class NexusServiceJunit5Test {

  private final SampleNexusService mockNexusService = mock(SampleNexusService.class);

  /**
   * A test-only Nexus service implementation that delegates to the Mockito mock defined above. The
   * operation is implemented as a synchronous handler that forward calls to the mock, allowing
   * full control over return values and verification of inputs.
   */
  @ServiceImpl(service = SampleNexusService.class)
  public class TestNexusServiceImpl {

    @OperationImpl
    @SuppressWarnings("DirectInvocationOnMock")
    public OperationHandler<SampleNexusService.HelloInput, SampleNexusService.HelloOutput> hello() {
      return OperationHandler.sync((ctx, details, input) -> mockNexusService.hello(input));
    }
  }

  // Using OperationHandler.sync for both operations bypasses the need for a backing workflow,
  // returning results inline just like a synchronous call.


  @RegisterExtension
  public final TestWorkflowExtension testWorkflowExtension =
      TestWorkflowExtension.newBuilder()
          // If a Nexus service is registered as part of the test as in the following line of code,
          // the TestWorkflowExtension will, by default, automatically create a Nexus service
          // endpoint and workflows registered as part of the TestWorkflowExtension will
          // automatically inherit the endpoint if none is set.
          .setNexusServiceImplementation(new TestNexusServiceImpl())
          // registerWorkflowImplementationTypes will take the classes given and create workers for
          // them, enabling workflows to run.
          // Since both operations are mocked with OperationHandler.sync, no backing workflow is
          // needed for hello — only the caller workflow types need to be registered.
          .registerWorkflowImplementationTypes(HelloCallerWorkflowImpl.class)
          // The workflow will start before each test, and will shut down after each test.
          // See CallerWorkflowTest for an example of how to control this differently if needed.
          .build();

  // The TestWorkflowExtension extension in the Temporal testing library creates the
  // arguments to the test cases and initializes them from the extension setup call above.
  @Test
  public void testHelloWorkflow(
      TestWorkflowEnvironment testEnv, Worker worker, HelloCallerWorkflow workflow) {

    // Set the mock value to return
    when(mockNexusService.hello(any()))
        .thenReturn(new SampleNexusService.HelloOutput("Hello Mock World"));

    // Execute a workflow waiting for it to complete.
    String greeting = workflow.hello("World", SampleNexusService.Language.EN);
    assertEquals("Hello Mock World", greeting);

    // Verify the operation was called exactly once and no other operations were invoked
    verify(mockNexusService, times(1)).hello(any());
    // Verify the Nexus service was called with the correct input
    verify(mockNexusService).hello(argThat(input -> "World".equals(input.getName())));

    verifyNoMoreInteractions(mockNexusService);
  }
}
```

### 如何跳过时间 {#skip-time}

一些长期运行的 Workflow 可以持续数月甚至数年。
实施测试框架允许您的 Workflow 代码跳过时间并在几秒钟内完成测试，而不是 Workflow 指定的数量。

例如，如果您的 Workflow 睡眠了一天，或者 Activity 发生了重试间隔较长的故障，则无需等待整个睡眠周期的长度来测试睡眠功能是否起作用。
相反，通过及时向前跳跃并及时完成测试来测试睡眠后发生的逻辑。

大多数 SDK 中包含的测试框架是 Temporal Server 的内存中实现，支持跳过时间。
时间是 `TestWorkflowEnvironment` 实例的全局属性：跳过时间（自动或手动）适用于所有当前正在运行的测试。
如果不同的测试需要不同的时间行为，请按系列运行测试或使用测试服务器的单独实例运行测试。
例如，您可以并行运行所有具有自动时间跳过功能的测试，然后串行运行所有具有手动时间跳过功能的测试，然后并行运行所有不带时间跳过功能的测试。

#### 设置时间跳过 {#setting-up}

在您选择的 SDK 中设置跳时测试框架。

{/* #### Skip time automatically {#automatic-method} */}

您可以在您选择的 SDK 中自动跳过时间。
启动一个测试服务器进程，根据需要跳过时间。
例如，在跳时模式下，Timer（包括睡眠和条件超时）会快进，除非“活动”或“Nexus 操作”处理程序正在运行。 Nexus 操作处理程序在 10 秒后超时，并且在等待重试时允许跳过时间。

## 如何重播 Workflow Execution {#replay}

重播重新创建了 Workflow Execution 的确切状态。
您可以从 Event History 的开头重播 Workflow。

仅当 [Workflow Definition](/workflow-definition) 从确定性的角度来看与提供的历史记录兼容时，重播才会成功。

当您测试对 Workflow Definition 的更改时，我们建议执行以下操作作为 CI 检查的一部分：

1. 确定被测 Worker 代码将针对哪些 Workflow 类型或 Task Queue（或两者）。
2. 使用 SDK 客户端以编程方式或通过 Temporal CLI 从每个 Task Queue 下载一组代表性的最近打开和关闭的 Workflow 的事件历史记录。
3. 通过重播运行事件历史记录。
4. 如果重播期间遇到任何错误，则 CI 失败。

以下是获取和重播事件历史记录的示例：

要重播 Workflow Execution，请使用 `Temporal-testing` 包中的 [Workflow重播器](https://www.javadoc.io/doc/io.temporal/temporal-testing/latest/io/temporal/testing/WorkflowReplayer.html) 类。

在以下示例中，从服务器下载事件历史记录，然后重播。
请注意，这需要启用高级 Visibility。

```java
// Note we assume you already have a WorkflowServiceStubs (`service`) and WorkflowClient (`client`)
// in scope.
ListWorkflowExecutionsRequest listWorkflowExecutionRequest =
    ListWorkflowExecutionsRequest.newBuilder()
        .setNamespace(client.getOptions().getNamespace())
        .setQuery("TaskQueue = 'mytaskqueue'")
        .build();
ListWorkflowExecutionsResponse listWorkflowExecutionsResponse =
    service.blockingStub().listWorkflowExecutions(listWorkflowExecutionRequest);
List<WorkflowExecutionHistory> histories =
    listWorkflowExecutionsResponse.getExecutionsList().stream()
        .map(
            (info) -> {
              GetWorkflowExecutionHistoryResponse weh =
                  service.blockingStub().getWorkflowExecutionHistory(
                      GetWorkflowExecutionHistoryRequest.newBuilder()
                          .setNamespace(testEnvironment.getNamespace())
                          .setExecution(info.getExecution())
                          .build());
              return new WorkflowExecutionHistory(
                  weh.getHistory(), info.getExecution().getWorkflowId());
            })
        .collect(Collectors.toList());

WorkflowReplayer.replayWorkflowExecutions(
    histories, true, WorkflowA.class, WorkflowB.class, WorkflowC.class);
```

在下一个示例中，从磁盘上的 JSON 文件加载单个历史记录：

```java
File file = new File("my_history.json");
WorkflowReplayer.replayWorkflowExecution(file, MyWorkflow.class);
```

在这两个示例中，如果 Event History 是不确定的，则会引发错误。
您可以通过将 `failFast` 参数设置为 `false` 来选择等待，直到使用 `replayWorkflowExecutions` 重播所有历史记录。
