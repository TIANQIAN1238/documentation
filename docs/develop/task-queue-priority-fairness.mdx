[Task Queue 优先级](#task-queue-priority)和 [Task Queue 公平性](#task-queue-fairness)是管理 Task Queue 内工作分配的两种方式。优先级关注 Task 在单个 Task Queue 内的优先级排序。公平性旨在防止在同一优先级级别内某组 Task 阻塞其他 Task。

您可以在 Task Queue 中单独使用优先级和公平性，也可以将它们一起用于更复杂的场景。

:::tip 支持、稳定性和依赖信息

优先级和公平性目前处于
[公开预览（Public Preview）](/evaluate/development-production-features/release-stages#public-preview)阶段。优先级在 Temporal Cloud 和自托管 Temporal Service 中是免费功能。

公平性将是 Temporal Cloud 中的__付费__功能，计费将在不久的将来启用。您可以在[这里](/cloud/actions#fairness)找到有关公平性定价结构的更多详情。

:::

## Task Queue 优先级

**Task Queue 优先级**允许您在_单个_ Task Queue 内根据分配的优先级值控制 Workflow、Activity 和 Child Workflow 的执行顺序。每个优先级级别充当一个子队列，将单个 Task Queue 内的 Task 分开，以便高优先级 Task 可以排在低优先级 Task 前面。

<EnlargeImage
  src="/img/develop/task-queue-priority-fairness/priority-details.png"
  alt="Detailed view of how Priority works"
/>

### 何时使用优先级

如果您需要一种方式来指定 Task 的执行顺序，可以使用优先级来管理。优先级允许您区分不同的 Task，例如批处理和实时 Task，这样您就可以使用单一 Worker 池进行高效的资源分配，同时确保实时 Task 在批处理 Task 之前得到处理。

您也可以将此用作立即运行紧急 Task 并覆盖其他 Task 的方式。例如，如果您运营一个电商平台，您可能希望在处理内部库存管理等时效性较低的 Task 之前先处理与支付相关的 Task。

### 如何使用优先级

优先级适用于自托管 Temporal 实例和 Temporal Cloud，并将自动启用。如果您应用优先级键，该功能将处于使用状态。

:::info

您可以通过将动态配置 `matching.useNewMatcher` 设置为 `true` 或 `false` 在 Task Queue、Namespace 或全局级别切换优先级启用状态。

:::

您可以通过将_优先级键_设置为整数范围 `[1,5]` 内的值来选择优先级级别。较低的值意味着较高的优先级，因此 `1` 是最高优先级级别。如果您不指定优先级，Task 默认为优先级 `3`。如果没有单独设置 Activity 优先级，Activity 将继承其 Workflow 的优先级级别。

当您在 Task Queue 中设置优先级级别时，这意味着它们将**全部**按优先级顺序处理。
例如，所有优先级级别 `1` 的 Task 都将在优先级级别 `2` 的 Task 之前执行，依此类推。因此，较低优先级的 Task 将被阻塞，直到较高优先级的 Task 完成运行。相同优先级级别的 Task 按先进先出（FIFO）顺序调度运行。如果您需要更灵活地为相同类型的 Task 分配资源（例如处理多个电商平台的支付），请查看[公平性部分](#task-queue-fairness)。

您可以通过 Temporal CLI 设置 Workflow 上的优先级键参数：

```
temporal workflow start \
  --type ChargeCustomer \
  --task-queue my-task-queue \
  --workflow-id my-workflow-id \
  --input '{"customerId":"12345"}' \
  --priority-key 1
```

## Task Queue 公平性

Task Queue 公平性允许您基于 Task Queue 内的_公平性键_及其_公平性权重_来分配 Task。公平性键用于描述您的 Task 结构。每个公平性键在 Task Queue 中创建自己的"虚拟队列"，允许您将 Task 组织成租户、应用程序或工作负载类型等逻辑组。

这些虚拟队列使用轮询派发机制运行，即系统在选择下一个要派发的 Task 时依次循环遍历每个公平性键。轮询方法从本质上防止任何单个公平性键占用 Worker 容量。这确保了一个公平性键的 Task 不能完全阻塞来自其他键的 Task，即使一个键的积压比其他键多得多。

当 Worker 变得可用时，每个公平性键都有机会派发 Task，而不是在移动到下一个键之前处理完一个键的所有 Task。这创造了一种更公平的分配，使较小的租户或低流量工作负载仍能获得定期处理时间，而不是排在高流量操作之后等待。

与每个公平性键相关联的 Task 可以根据分配给该键的_公平性权重_进行派发。每个公平性键在单个 Task Queue 中应只分配一个公平性权重。为一个公平性键设置多个公平性权重将导致未指定的行为。

公平性权重可用于为公平性键分配更多或更少的资源。默认情况下，公平性键的公平性权重为 1.0。属于权重为 2.0 的公平性键的 Task 将比权重为 1.0 的其他键派发频率高一倍。

使用公平性键及其相应的公平性权重，您可以定义具有加权容量的级别。例如，您可以拥有免费、基础和高级级别，公平性确保高级 Task 的涌入不会压倒您的资源并阻塞免费和基础 Task。

:::info

当您开始使用公平性键时，它会将您的活跃 Task Queue 切换到公平性模式。这意味着新 Task 是在公平性下创建的，而现有排队的 Task 会在任何新 Task 之前得到处理。

:::

### 何时使用公平性

公平性旨在解决常见情况，例如：

- 拥有大小租户的多租户应用程序，其中小租户不应被大租户阻塞。
- 将 Task 分配到不同的容量带，然后，例如，在一个带不为空时，从一个带派发 80%，从另一个带派发 20%，而不限制总体容量。

它使用基于以下内容的加权分配，以概率方式在 Task Queue 中对 Task 进行排序：

- 您设置的公平性权重
- Task 的当前积压
- 跟踪您为不同公平性键分配 Task 情况的数据结构

举个例子，假设有一个工作负载包含三个租户：_tenant-big_、_tenant-mid_ 和 _tenant-small_，它们的 Task 数量随时间变化。您的 _tenant-big_ 有大量 Task，可能会压倒您的 Task Queue 并阻止 _tenant-mid_ 和 _tenant-small_ 运行其 Task。使用公平性，您可以为每个租户提供不同的公平性键，以确保 _tenant-big_ 不使用所有 Task Queue 资源并阻塞其他租户。

在这种情况下，_tenant-mid_ 和 _tenant-small_ 的 Task 将在 _tenant-big_ 的 Task 之间运行，以便它们被"公平地"执行。不过，如果您的所有 Task 都可以立即派发，那么您就不需要使用公平性。

同样的场景也适用于某些作业比其他作业运行更频繁的批处理作业，或者来自多个供应商的订单处理（其中几个供应商拥有大多数订单）。

公平性在 Task 派发时根据通过 Task Queue 的 Task 信息进行应用，并在派发前将每个 Task 视为等成本。它不考虑 Worker 当前正在执行的任何 Task。因此，如果您查看 Worker 正在处理的 Task，您可能看不到租户之间的"公平性"。例如，如果 _tenant-big_ 的 Task 已经在处理中，当 _tenant-small_ 的 Task 被派发时，仍然可能看起来 _tenant-big_ 在使用最多的资源。

有两种使用 Task Queue 公平性的方式：不带优先级和带优先级。

### 公平性的工作原理

如果您在不带优先级的情况下使用公平性，具有不同公平性键的 Task 将使用基于公平性权重的加权分配来分配 Task Queue 中的资源。例如，假设您有三个公平性键来描述客户层级：_free-tier_、_basic-tier_ 和 _premium-tier_。您给 _premium-tier_ 分配公平性权重 5.0，_basic-tier_ 分配 3.0，_free-tier_ 分配 2.0。使用公平性，这意味着 50% 的时间派发 _premium-tier_ Task，30% 的时间派发 _basic-tier_ Task，20% 的时间从 Task Queue 积压中派发 _free-tier_ Task。

如果 Task Queue 积压中有具有相同公平性键的 Task，则按 [FIFO 顺序](/task-queue#task-ordering)派发。

这就是您能够确保一个层级不使用所有资源并阻塞 Task Queue 积压中其他 Task 派发的方式。

当您更新公平性键或公平性权重时，Task Queue 仅反映尚未派发的 Task 的这些变化。

<EnlargeImage
  src="/img/develop/task-queue-priority-fairness/fairness-details.png"
  alt="High-level of how fairness works"
/>

没有设置 `fairness_key` 的 Task 被归入隐式空字符串键下。
所有未设置键的 Task 共享这个单一的默认桶，并与命名的公平性键一起参与相同的轮询派发，默认权重为 1.0。

这意味着公平性的采用可以是渐进式的：您可以为某些租户分配公平性键，而不为其他租户分配。
未设置键的 Task 不会绕过公平性；它们作为一个组与所有明确设置键的 Task 竞争。

### 同时使用公平性和优先级

当您同时使用公平性和优先级时，优先级决定 Task 进入哪个子队列。实现了优先级的单个 Task Queue 将根据优先级级别具有不同的子队列。公平性将应用于每个优先级级别内的 Task。

### 如何使用公平性

公平性适用于自托管 Temporal 实例和 Temporal Cloud。

如果您在 API 调用中开始使用_公平性键_，它将在 Temporal Cloud 中自动启用。

如果您是自托管 Temporal，请使用最新的预发布开发服务器，并在相关 Task Queue 或 Namespace 的[动态配置](https://github.com/temporalio/temporal/blob/a3a53266c002ae33b630a41977274f8b5b587031/common/dynamicconfig/constants.go#L1345-L1348)中将 `matching.useNewMatcher` 和 `matching.enableFairness` 设置为 `true`。您还需要将 `matching.enableMigration` 设置为 `true`，以支持在启用公平性后清空现有积压中的 Task。

启用 `matching.useNewMatcher` 和 `matching.enableFairness` 仅适用于自托管 Temporal 实例。即将在 Temporal Cloud 中推出一个切换开关，用于在 Namespace 级别启用优先级和公平性。

:::info

公平性将是 Temporal Cloud 中的付费功能，计费将在不久的将来启用。在为您的 Namespace 启用计费之前，您将收到通知。当 Temporal 开始对此功能计费时，您将能够在 Namespace 级别启用或禁用公平性，并且计费将在禁用后的下一个整点小时后停止。

:::

### 在 Task Queue 级别设置速率限制

当您开始扩展 Temporal Service 时，您可能决定设置[每秒请求数（RPS）](/references/dynamic-configuration#service-level-rps-limits)限制来测试您的工作负载或试验配置基准。您可以通过 CLI 中的 `queue-rps-limit` 在 Task Queue 级别设置 RPS 限制。

整个队列的速率限制了 Task 的派发速率，与公平性键无关。Task 的派发速度不会超过规定的限制（以几秒钟为平均值），但由于分区，您可能会看到小幅突发。

:::note

整个队列的速率限制与 SDK 中通过 Worker 选项提供的功能相同。如果通过 API 设置，该限制优先于通过 Worker 选项设置的限制。

:::

如果您想确保特定的公平性键有限制来节流 Task，您还可以在 CLI 中使用 `fairness-key-rps-limit-default` 基于公平性键设置 RPS 限制。这可以是您区分客户层级的方式，仅允许该层级处理定义数量的 Task。

```
temporal task-queue config set \
    --task-queue my-task-queue \
    --task-queue-type activity \
    --namespace my-namespace \
    --queue-rps-limit 500 \
    --queue-rps-limit-reason "overall limit" \
    --fairness-key-rps-limit-default 33.3 \
    --fairness-key-rps-limit-reason "per-key limit"
```

每公平性键速率限制与 Task Queue 公平性协同工作。如果您将公平性视为将队列划分为每个键一个虚拟队列，那么每公平性键速率限制就是对每个单独虚拟队列的限制。关于每公平性键限制的一些重要说明：

- 整个队列限制和每公平性键限制可以独立设置：可以不设置、设置其一或两者都设置。如果两者都设置，则应用更严格的限制。
- 键的每公平性键限制会根据分配给该键的公平性权重进行缩放。因此，如果队列的每公平性键限制设置为 10，那么所有具有默认权重（1.0）的键将具有 10 个 Task/秒的限制。但如果某个特定键被给予 2.5 的权重，则该键的每键速率限制将为 25 个 Task/秒。
- 由于每个键的派发速率应与其权重成比例，如果任何键达到每键限制，那么几乎所有键都会达到。工作方式是：如果下一个要派发的 Task 达到每键限制，则派发将等待直到可以继续。
- 通常实际上没有任何阻塞，但当一个键的公平性权重在 Task 被调度和派发之间发生变化时，可能会发生阻塞。如果一个键的公平性权重被降低，例如，新的较低每键速率限制将被遵守。由于这些 Task 最初是以较高速率调度的，因此在派发时它们会阻塞其他 Task。这个限制将在未来得到改进。

### 公平性权重覆盖

在许多情况下，在公平性键的同时提供权重很简单直接。在某些情况下，将权重从客户端代码中删除，而通过配置 API 控制少数键的权重更为方便。

您可以通过配置 API 覆盖最多 1000 个键的权重。当为一个键设置覆盖时，通过 Workflow 或 Activity 优先级元数据附加到 Task 的权重将被忽略，而改用覆盖的权重。

权重覆盖按 Task Queue（包括类型）存储，因此对于 Workflow 和 Activity Task Queue 都需要设置，以便对两者都生效。

### 公平性的限制

- 您可以使用的公平性键数量没有限制，但随着添加更多键，其准确性可能会降低。
- Task Queue 在内部被[分区](/task-queue#task-ordering)，Task 随机分配到分区。这可能会干扰公平性。根据您的用例，您可以联系 Temporal 支持将您的 Task Queue 设置为单个分区。
- 公平性权重在调度时而非派发时应用。因此，它只影响新调度的 Task，而不影响当前积压的 Task。这意味着如果您需要节流现有积压中某个公平性键的 Task，您将无法做到。
- 当您使用 Worker Versioning 并将 Workflow 从一个版本迁移到另一个版本时，版本之间仍然会应用优先级。版本之间不保证公平性。例如，您可能有最初在 Worker 版本 _alpha_ 上排队的 Task、在 Worker 版本 _beta_ 上排队的 Task，以及一些从 _alpha_ 移动到 _beta_ 的 Task。只有当 Task 最初在同一 Worker 版本上排队时，才能保证公平性。因此，从 _alpha_ 移动到 _beta_ 的 Task 可能会有一些差异。
- 积压的 Task 是持久的，不受服务器重启影响。公平性排序在重启后对最活跃的键得到保留；不太活跃的键可能会在排序正常化之前暂时将新 Task 派发到其现有积压之前。
