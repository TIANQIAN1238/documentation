<a id="develop-workflows"></a>
## 如何开发基本的 Workflow {#develop-workflows}

Workflow 是 Temporal 应用程序的基本单元，这一切都始于 [Workflow Definition](/workflow-definition) 的开发。

在 Temporal PHP SDK 编程模型中，Workflow 是类方法。类必须实现用 `#[WorkflowInterface]` 注释的接口。 Workflow 方法必须用 `#[WorkflowMethod]` 进行注释。

```php
use Temporal\Workflow\YourWorkflowInterface;
use Temporal\Workflow\WorkflowMethod;

#[WorkflowInterface]
interface FileProcessingWorkflow
{
    #[WorkflowMethod]
    public function processFile(Argument $args);

}
```

<a id="workflow-parameters"></a>
### 如何定义 Workflow 参数 {#workflow-parameters}

Temporal Workflow 可以具有任意数量的自定义参数。
但是，我们强烈建议将对象用作参数，以便可以在不破坏 Workflow 签名的情况下更改对象的各个字段。
所有 Workflow Definition 参数都必须可序列化。

用 `#[WorkflowMethod]` 注释的方法可以有任意数量的参数。

我们建议传递包含所有输入字段的单个参数，以允许以向后兼容的方式添加字段。

请注意，所有输入都应可使用提供的 [数据转换器](https://github.com/temporalio/sdk-php/blob/master/src/DataConverter/DataConverterInterface.php) 接口序​​列化为字节数组。
默认实现使用 JSON 序列化器，但可以轻松配置替代实现。
您可以创建自定义对象并将其传递给 Workflow 方法，如以下示例所示：

```php
#[WorkflowInterface]
interface FileProcessingWorkflow {
    #[WorkflowMethod]
    public function processFile(Argument $args);
}
```

<a id="workflow-return-values"></a>
### 如何定义 Workflow 返回参数 {#workflow-return-values}

Workflow 返回值也必须是可序列化的。
返回结果、返回错误或引发异常在每种受支持的语言中都是相当惯用的。
但是，必须用于获取 Workflow Execution 结果的 Temporal API 将仅收到结果或错误之一。

Workflow 方法返回一个生成器。
要在客户端代码中正确类型转换 Workflow 的返回值，请使用 `#[ReturnType()]` 属性。

```php
#[WorkflowInterface]
interface FileProcessingWorkflow {

    #[WorkflowMethod]
    #[ReturnType("string")]
    public function processFile(Argument $args);
}
```

<a id="workflow-type"></a>
### 如何自定义您的 Workflow 类型 {#workflow-type}

Workflow 具有称为 Workflow 名称的类型。

以下示例演示如何为 Workflow 类型设置自定义名称。

要自定义 Workflow 类型，请使用 `WorkflowMethod` 属性指定 Workflow 的名称。

```php
#[WorkflowMethod(name)]
```

如果未指定 Workflow 类型，则 Workflow 类型默认为接口名称，在本例中为 `YourWorkflowDefinitionInterface`。

```php
#[WorkflowInterface]
interface YourWorkflowDefinitionInterface
{
    #[WorkflowMethod]
    public function processFile(Argument $args);
}
```

<a id="workflow-logic-requirements"></a>
### 如何开发 Workflow 逻辑 {#workflow-logic-requirements}

Workflow 逻辑受 [确定性执行要求](/workflow-definition#deterministic-constraints) 约束。
因此，每种语言都仅限于使用某些惯用技术。
但是，每个 Temporal SDK 都提供了一组 API，可在 Workflow 内部使用这些 API 来与外部（对于 Workflow）应用程序代码进行交互。

\*\*Temporal 使用 [Microsoft Azure 事件溯源模式](https://docs.microsoft.com/en-us/azure/architecture/patterns/event-sourcing) 恢复 Workflow 对象的状态，包括其局部变量值。

本质上，每次需要恢复 Workflow 状态时，其代码都会从头开始重新执行。
重播时，Side Effect（例如 Activity 调用）将被忽略，因为它们已记录在 Workflow Event History 中。
在编写 Workflow 逻辑时，重放是不可见的，因此应该编写代码，因为它只执行一次。
此设计对 Workflow 实现施加以下约束：

- 不要使用任何可变全局变量，因为 Workflow 的多个实例是并行执行的。
- 不要直接从 Workflow 代码调用任何非确定性函数，例如非种子随机函数或 `UUID`。

始终在 Workflow 实现代码中执行以下操作：

- 不要执行任何 IO 或服务调用，因为它们通常不是确定性的。为此使用活动。
- 仅使用 `Workflow::now()` 获取 Workflow 内的当前时间。
- 呼叫 `yield Workflow::Timer()` 而不是 `sleep()`。
- 不要在 **Workflow 代码**中使用 PHP 提供的任何阻止 SPL（即 `fopen`、`PDO` 等）。
- 对 Workflow 代码进行任何更改时使用 `yield Workflow::getVersion()`。如果没有这个，任何 Updated Workflow 代码的部署
  可能会破坏已经打开的 Workflow。
- 不要直接从 Workflow 访问配置 API，因为配置中的更改可能会影响 Workflow Execution 路径。
  将其作为参数传递给 Workflow 函数或使用 Activity 加载它。

Workflow 方法参数和返回值可使用提供的 [数据转换器](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/common/converter/DataConverter.html) 接口序​​列化为字节数组。
默认实现使用 JSON 序列化程序，但您可以使用任何替代序列化机制。

确保使用 `ReturnType` 注释您的 `WorkflowMethod` 以指定具体的返回类型。

> 您不能使用默认返回类型声明，因为 Workflow 方法是生成器。

通过调用参数传递给 Workflow 或通过结果值返回的值都记录在执行历史记录中。
对于 Workflow 逻辑需要处理的每个事件，整个执行历史记录都会从 Temporal 服务传输到 Workflow Worker。
因此，大量执行历史记录可能会对 Workflow 的性能产生不利影响。
因此，请注意通过 Activity 调用参数或返回值传输的数据量。
否则，Activity 实现不存在其他限制。\*\*
