## Worker 性能概念 {#worker-performance-concepts}

Worker 的性能特征受以下因素（但不限于此）影响。

### Task 槽位 {#slots}

**Worker Task 槽位**代表 Temporal Worker 执行单个并发 Task 的容量。槽位对于管理 Temporal 应用中 Worker 的工作负载和性能至关重要。它们用于 Workflow 和 Activity Task。当 Worker 开始处理 Task 时，它会占用一个槽位。可用槽位的数量直接影响 Worker 可以同时处理多少个 Task。

### Slot Supplier {#slot-suppliers}

**Slot Supplier** 定义了为 Worker 提供槽位的策略，增加或减少 Worker 的槽位数量。Supplier 决定何时可以开始新 Task。每个 Supplier 管理一种槽位类型。槽位类型包括 Activity、Workflow、Nexus 或 Local Activity Task。可用槽位决定 Worker 是否愿意轮询并执行该类型的新 Task。

Slot Supplier 策略包括手动分配固定槽位数量和基于资源的"自动调整器"分配。基于资源的 Supplier 根据 CPU 和内存资源调整槽位数量。可用的 Slot Supplier 包括：

- **固定大小 Slot Supplier（Fixed Size Slot Suppliers）**：
  发放槽位直到预设限制。
  如果您对 Task 将消耗多少资源有具体了解，并且可以轻松确定同时运行数量的上限，这非常有用。
  当您需要绝对最佳性能时，请查看您的硬件和环境特性。
  这些信息可让您计算适当的固定大小限制。
  评估您可以支持的最大槽位数量，而不会造成过度订阅或内存溢出（"OOM"）。
  将该值与固定大小 Supplier 结合使用可提供最优结果，且开销最小。

- **基于资源的 Slot Supplier（Resource-Based Slot Suppliers）**：
  根据实时 CPU 和内存使用情况发放槽位。
  您设置 CPU 和内存的目标利用率，Slot Supplier 会尝试在负载下达到这些值而不超过它们。
  基于资源的 Supplier 将考虑容器化环境中施加的内存限制。
  它根据当前系统资源动态调整不同 Task 类型的可用槽位数量。

:::info

在容器化环境中运行时，所有 SDK 对 CPU 和内存都使用 cgroups。CPU 在容器级别计算。

:::

- **自定义 Slot Supplier（Custom Slot Suppliers）**：
  根据您定义的自定义逻辑发放槽位。
  当您需要完全控制 Worker 何时接受和执行 Task 时，请使用此方法。
  有关实现详情，请参阅[实现自定义 Slot Supplier](#custom-slot-implementation)。

:::caution

- 您无法保证基于资源的 Supplier 的目标永远不会被超过。
  Task 消耗的资源在开始前无法预知。

- 在选择之前，请阅读[选择合适的 Slot Supplier 类型](#choosing-slot-supplier-types)。

- Worker tuner 取代了现有的 `maxConcurrentXXXTask` 风格的 Worker 选项。
  同时使用两种风格将在 Worker 初始化时导致错误。

:::

### Worker 调优 {#worker-tuning}

Worker 调优是为 Worker 的不同 Task 槽位定义自定义 Slot Supplier 以精细调整其性能的过程。
您使用称为 **Worker tuner** 的特殊类型，将 Slot Supplier 分配给各种 Task 类型，包括 Worker、Activity、Nexus 和 Local Activity Task。

有关如何配置和使用 Worker tuner 的更多信息，请参阅 [Worker 运行时性能调优](#worker-performance-tuning)。

:::caution

Worker tuner 取代了现有的 `maxConcurrentXXXTask` 风格的 Worker 选项。
同时使用两种风格将在 Worker 初始化时导致错误。

:::

### Task Poller

Worker 的 **Task Poller** 在 Temporal 架构中发挥着关键作用，通过高效地将工作接入 Worker 来支持可扩展、弹性的 Workflow Execution。
Poller 创建与 Temporal Service 的长轮询连接，并主动轮询 Task Queue 以获取要处理的 Task。当 Task Poller 收到 Task 时，它将 Task 传递给适当的执行器槽位进行处理。

Temporal SDK 实现了 *Poller 自动扩缩容（Poller Autoscaling）*支持，它根据给定数量的 Worker 和 Task 积压大小动态调整正在使用的轮询器数量，以最大化吞吐量。
Temporal 建议在大多数用例中使用 Poller Autoscaling，因为手动将轮询器数量设置得过高或过低都会导致性能下降。
要配置 Poller Autoscaling，请参阅[配置轮询器选项](#configuring-poller-options)以及每个 Temporal SDK 的示例。

### 即时 Task 执行

:::caution
即时启动不遵循 Worker Versioning。即时启动的 Workflow 可能在任何可用的本地 Worker 上运行，即使该 Worker 不是其 Worker Deployment 的 Current 或 Ramping 版本。
:::

作为延迟优化，Activity 和 Workflow Task 在合适情况下可以在本地 Worker 中即时启动。

#### 即时 Activity 启动

如果处理 Workflow Task 的 Worker 也注册了被调用的 Activity 定义，可能会自动发生即时 Activity 启动。
如果发生这种情况，它可能会尝试为 Activity 的执行保留一个 Activity 槽位，服务器可能会在 Workflow Task 完成时响应 Activity Task，供 Worker 立即执行。

#### 即时 Workflow 启动

:::tip 支持、稳定性和依赖信息

即时 Workflow 启动在 Go、Java、Python 和 .NET SDK 中处于[公开预览（Public Preview）](/evaluate/development-production-features/release-stages#public-preview)阶段。
即时 Workflow 启动对所有 Temporal Cloud 用户和自托管 Temporal Server 1.29.0+ 启用。无需额外配置或访问请求。但是，在启动每个 Workflow 时必须将 Request-Eager-Start 设置为 true 才能使用即时 Workflow 启动。

:::

即时 Workflow 启动减少了启动 Workflow Execution 所需的延迟。
推荐用于使用 Local Activity 与外部服务交互的短暂 Workflow，特别是在这些交互在第一个 Workflow Task 中发起且 Workflow 部署在靠近 Temporal Server 的位置以最小化网络延迟时。

该功能对于"正常路径"必须在几十毫秒内开始外部交互、同时仍然依赖 Temporal 服务器驱动的重试和补偿机制以确保故障场景可靠性的 Workflow 特别有益。

**快速入门**

即时 Workflow 启动要求 Starter 和 Worker 共享位于同一进程中的 Client，并在 Start Workflow 调用中将 `request_eager_start`（或类似名称）设置为 true。
设置后，如果 Worker 有可用的 Workflow Task 槽位并且已注册 Workflow 定义，Worker 可以在本地执行 Workflow 的第一个 Task，而无需先往返 Temporal Server。
这通常与在第一个 Workflow Task 中执行的 Local Activity 结合使用最为有效，因为其他需要等待某些事情的 Workflow API 调用将强制往返。

:::tip 资源

- [Go SDK - 代码示例](https://github.com/temporalio/samples-go/tree/main/eager-workflow-start)
- [Java SDK - 代码示例](https://github.com/temporalio/samples-java/blob/main/core/src/main/java/io/temporal/samples/hello/HelloEagerWorkflowStart.java)
- Python SDK - 在调用 `start_workflow` 或 `execute_workflow` 时使用 `request_eager_start`
- .NET SDK - 在启动 Workflow 时在 `WorkflowOptions` 中使用 `RequestEagerStart`
- [博客：使用即时 Workflow 启动改善延迟](https://temporal.io/blog/improving-latency-with-eager-workflow-start)

:::

**工作原理**

启动 Workflow 的传统方式通过在 Starter 程序和 Worker 之间共享 Task Queue 名称来解耦它们，类似于发布/订阅模式。
这有许多优点：例如，我们可以在没有运行中的 Worker 的情况下可靠地调度 Workflow Execution，或者将 Worker 和 Workflow 实现与 Starter 应用分离并独立托管。

但解耦也使得优化延迟变得更加困难。
相反，当 **Starter 和 Worker 共存于同一进程**并且彼此感知时，它们可以在绕过服务器的情况下交互，节省一些耗时的操作。

<CaptionedImage
    src="/img/develop/worker-performance/eager-workflow-start-flow.png"
    title="即时 Workflow 启动"
/>

上图展示了即时 Workflow 启动的过程：

1. 进程开始时，Starter 在 Start Workflow 选项中将 `request_eager_start`（或类似名称）设置为 true。
1. SDK 将尝试找到愿意执行第一个 Workflow Task 的本地 Worker，并为其保留执行槽位。
1. 如果成功，SDK 将向服务器提示新 Workflow 首选即时模式。
1. 服务器不仅在历史记录中注册 Workflow 的启动，还将第一个 Workflow Task 分配给 Starter，所有这些都在同一次数据库更新中完成。
1. 第一个 Task 包含在服务器响应中，无需匹配步骤。
1. SDK 从响应中提取 Task，并将其分发给本地 Worker。

为了从错误中恢复，即时 Workflow 启动会回退到非即时路径。例如，当第一个 Task 被即时返回，但本地 Worker 在处理 Task 时失败或超时，服务器会在 WorkflowTaskTimeout 后非即时地重试此 Task。

## 调优性能指标 {#metrics}

Temporal SDK 从 Temporal Client 使用情况和 Worker 进程中发出指标。
性能调优使用三个重要的 SDK 指标组：

### 槽位可用性指标

Temporal 的 [`worker_task_slots_available`](/references/sdk-metrics#worker_task_slots_available) 和 `worker_task_slots_used` 仪表可以报告某个 Worker 类型当前可用和未占用的执行器"槽位"数量。
使用 `worker_type=WorkflowWorker` 标记 Workflow Task Worker，使用 `worker_type=ActivityWorker` 标记 Activity Worker。

:::tip

与 `worker_task_slots_used` 不同，`worker_task_slots_available` 只能与固定大小的 Slot Supplier 一起使用，不能与基于资源的 Slot Supplier 一起使用。

:::

### 延迟指标

Temporal 提供两个延迟计时器：[`workflow_task_schedule_to_start_latency`](/references/sdk-metrics#workflow_task_schedule_to_start_latency) 用于 Workflow Task，[`activity_schedule_to_start_latency`](/references/sdk-metrics#activity_schedule_to_start_latency) 用于 Activity。
Schedule-To-Start 延迟是从 Task 被调度（即放入队列）到 Worker 开始（即从 Task Queue 取出）该 Task 的时间。
这些指标有助于确保 Task 从队列中及时处理。
有关 `schedule_to_start` 超时和延迟的更多信息，请参阅 [Schedule-To-Start Timeout](/encyclopedia/detecting-activity-failures#schedule-to-start-timeout)。

### 缓存指标

[`sticky_cache_size`](/references/sdk-metrics#sticky_cache_size) 和 [`workflow_active_thread_count`](/references/sdk-metrics#workflow_active_thread_count) 指标报告 Workflow 缓存的大小和缓存的 Workflow 线程数量。

## Worker 性能选项 {#configuration}

每个 Worker 可以通过在实例化时提供自定义 Worker 选项（`WorkerOptions`）来配置。
选项特定于单个 Worker，不影响集群中的其他成员。

### 执行器槽位选项

`maxConcurrentWorkflowTaskExecutionSize` 和 `maxConcurrentActivityExecutionSize` 选项定义了 Worker 的 Workflow Task 和 Activity Task 槽位总数。

:::caution

- Worker tuner 取代了现有的 `maxConcurrentXXXTask` 风格的 Worker 选项。
  同时使用两种风格将在 Worker 初始化时导致错误。

:::

### 配置轮询器选项

#### 推荐方法

Temporal SDK 支持 Poller Autoscaling，它根据需要自动选择适当数量的轮询器。使用此功能可以更有效地使用轮询器，提高吞吐量，并改善 Schedule-to-Start 延迟。您可以通过将 `*_task_poller_behavior` 选项设置为 `PollerBehaviorAutoscaling` 来启用此功能。名称可能因 SDK 而略有不同。有关启用 Poller Autoscaling 的具体示例，请参阅下方的 SDK 示例部分。Poller Autoscaling 将成为未来版本 Temporal SDK 的默认配置。

:::tip

`PollerBehaviorAutoscaling` 仅在 Temporal Server v1.28.0 及更高版本中启用。

:::

#### 手动配置

有一些选项可用于手动配置最小、最大和初始轮询器数量，但不建议在生产用例中手动设置这些值。要手动设置这些值，可以使用以下选项：

- `maxConcurrentWorkflowTaskPollers`（在 Java SDK 中：`workflowPollThreadCount`）
- `maxConcurrentActivityTaskPollers`（在 Java SDK 中：`activityPollThreadCount`）

这些选项分别定义在 Workflow 和 Activity Task Queue 上执行轮询请求的轮询器最大数量。

### 缓存选项（Java SDK）{#cache-options}

Workflow 缓存在单个主机上的所有 Worker 之间创建和共享。
它旨在限制每个主机/进程的缓存资源使用。
这些选项在 `WorkerFactoryOptions` 上定义：

- `WorkerFactoryOptions#workflowCacheSize` 定义了缓存的 Workflow Execution 的最大数量。
  每个缓存的 Workflow 至少包含一个 Workflow 线程及其资源（内存等）。
- `maxWorkflowThreadCount` 定义了在任何时间可以同时存在的最大 Workflow 线程数。

这些缓存选项限制了内存中 Workflow 缓存的资源消耗。
Workflow 缓存选项在所有 Worker 之间共享，因为 Workflow 缓存与整个主机的资源消耗紧密集成。
这包括内存和总线程数，应该按主机/JVM 进行限制。

对于 Go，使用 [`SetStickyWorkflowCacheSize`](https://pkg.go.dev/go.temporal.io/sdk/worker#SetStickyWorkflowCacheSize)。对于 Python，使用 `max_cached_workflows` Worker 选项。

### "大值"的缺点

在"所有地方都使用大值"时存在缺点。
与任何多线程系统一样，在不使用 SDK 和系统指标监控的情况下指定过大的值会导致持续的资源争用/抢占，
这会降低系统的总吞吐量并增加延迟抖动。

### 不变量（仅 Java SDK）{#invariants}

这些属性对于 Worker 的配置应始终为真。

在调整 Worker 设置后执行此合理性检查。

1. `workflowCacheSize` 应该 ≤ `maxWorkflowThreadCount`。每个 Workflow 至少有一个 Workflow 线程。
2. `maxConcurrentWorkflowTaskExecutionSize` 应该 ≤ `maxWorkflowThreadCount`。拥有比 Workflow 缓存大小更多的 Worker 槽位会导致执行器之间的资源争用/抢占和不可预测的延迟。建议 `maxWorkflowThreadCount` 至少是 `maxConcurrentWorkflowTaskExecutionSize` 的 2 倍。
3. `maxConcurrentWorkflowTaskPollers` 应该显著 ≤ `maxConcurrentWorkflowTaskExecutionSize`。`maxConcurrentActivityTaskPollers` 应该显著 ≤ `maxConcurrentActivityExecutionSize`。轮询器数量应始终低于执行器数量。

## Worker 运行时性能调优 {#worker-performance-tuning}

Worker 调优管理 Slot Supplier 的分配。
每个 Worker 存在一个 **Worker Tuner** 实例，为不同的槽位类型（Activity、Workflow、Nexus 或 Local Activity Task）提供 Slot Supplier。
Tuner 为每种槽位类型分配不同的 Supplier。
例如，它可能为 Workflow 提供固定分配的 Slot Supplier，并为 Activity 使用基于资源的 Supplier。

### 选择 Slot Supplier 类型 {#choosing-slot-supplier-types}

Temporal 提供三种类型的 Slot Supplier：固定分配、基于资源和自定义。
以下是如何根据系统需求和工作负载特性选择最佳方法。

在选择固定分配还是基于资源的 Supplier 时，请考虑：

- Workflow Task 对 CPU 的需求极小，通常不消耗太多内存。
  它们非常适合固定大小的 Slot Supplier。
- 当非常低的 Task 完成延迟和最大吞吐量很重要时，避免使用基于资源的自动调整 Slot Supplier。
- 将自动调整的基于资源的 Slot Supplier 保留给资源使用模式您不完全了解且不想深入研究的工作负载部署。它可以为您不想花时间分析的工作负载提供**足够好**的性能。

具有可变或非常高的每任务资源需求的场景应依赖固定大小的 Supplier 和手动调优。基于资源的 Tuner 永远无法达到具有适当配置的固定大小 Tuner 的性能，但可以在不需要分析的情况下提供合理的性能。

以下用例非常适合基于资源的自动调整 Slot Supplier：

- **您希望以最小努力获得可接受的性能**：
  基于资源的 Supplier 可以在不需要分析工作负载的情况下提供合理的性能。
  它们是开始新工作负载的好方法。如果工作负载变得非常性能敏感，我们建议您对其进行分析并选择适当的固定大小数量。
- **低每任务消耗的波动工作负载**：
  基于资源的 Supplier 在每个 Task 消耗很少资源但可能运行相对较长时间时效果很好。
  例如：HTTP 调用或其他大部分时间等待外部事件的阻塞 I/O。
- **在不可预测的每任务消耗下防止内存溢出和过度订阅**：
  您的 Task 是否经常消耗不可预测数量的资源？
  您是否希望避免崩溃而不设置过于保守的固定限制？
  在这些情况下，基于资源的 Supplier 是一个好选择。
  请记住，自动调整永远无法做到_完美_，有时可能会超过您请求的 CPU 和内存系统限制。

对于最高级别的槽位分配控制，请考虑自定义 Slot Supplier。
这允许您根据系统需求定制槽位分配逻辑。
自定义 Supplier 提供了灵活性，以优化固定分配和基于资源的 Supplier 可能无法完全解决的特定用例。

选择合适的 Slot Supplier 取决于您的工作负载复杂性和您需要对资源分配的控制程度。
对于可预测的任务、可变工作负载或复杂的动态场景，Temporal Slot Supplier 可以满足您的需求。

### 实现自定义 Slot Supplier {#custom-slot-implementation}

实现您自己的 Slot Supplier，以控制 Worker 如何分配 Task 以及管理 Workflow、Activity 和 Nexus 操作的处理。
自定义 Slot Supplier 让您可以根据应用程序的需求精细调整 Task 处理。

Slot Supplier 发放 `SlotPermit`。
这些代表使用特定类型槽位（即 Workflow、Activity、Local Activity 或 Nexus）的权利。
您通过发放或扣留许可来控制 Worker 是否可以执行某些 Task。

自定义 Slot Supplier 必须实现以下函数：

- `reserveSlot` - 在轮询新 Task 之前调用。您的实现可以阻塞，并且一旦决定接受新工作就必须返回一个 Slot Permit。
- `tryReserveSlot` - 用于即时 Activity 处理等情况下的槽位预留。这不能阻塞。
- `markSlotUsed` - 当槽位即将用于 Task 时调用（不是在轮询期间持有时）。它提供有关 Task 的信息。
- `releaseSlot` - 当不再需要槽位时调用，无论是否已使用。

自定义策略需要更多工作，但提供对 Task 处理的更精细控制。
通过实现您自己的 Slot Supplier，您可以定制 Workflow、Activity 和 Nexus 操作的处理方式，为您的特定需求优化性能。

### Slot Supplier 节流

自动调整的 Supplier 可能会偏离请求的阈值。
给定 Task 将使用的资源在开始前无法预知。
在 Slot Supplier 愿意接受 Task 的速度和遵守定义阈值的能力之间存在根本性的权衡。

槽位节流是一种控制并发 Task 的新槽位可用于处理的速率的机制。
这个概念是 Worker 基于资源的自动调整功能的一部分。
通过在使槽位可用之间等待短暂时间，Worker 可以评估自上次 Task 开始处理以来资源使用情况的变化。

这个节流在基于资源的 Slot Supplier 的 SDK 选项中称为 `rampThrottle`。
它定义了 Worker 在超过最小槽位数量后，在发放新槽位之间等待的最短时间。

**更高的 `rampThrottle` 以性能换取安全性。**

例如：

如果一个刚启动的 Worker 没有节流，并且有 Task 积压，它可能立即接受 100 个 Task。
如果每个 Task 分配 1GB RAM，Worker 可能很快耗尽内存并崩溃。
节流在超过最小占用槽位数量后强制等待再发放新槽位，这样您就可以测量新消耗的资源。

## Workflow 缓存调优

当 `sticky_cache_size` 报告的缓存 Workflow Execution 数量达到 `workflowCacheSize` _或_ `workflow_active_thread_count` 指标仪表报告的线程数量达到 `maxWorkflowThreadCount` 时，Workflow Execution 将开始从缓存中被驱逐。
被驱逐的 Workflow Execution 在获得任何可能推进其进展的操作时，都需要重放。

如果上述 Workflow 缓存限制被触发，而 Worker 主机有足够的空闲 RAM 且未接近合理的线程限制，那么您可以选择增加 `workflowCacheSize` 和 `maxWorkflowThreadCount` 限制，以降低系统中 Replay 的总延迟和成本。
如果出现相反情况，请考虑降低限制。

:::note

在基于 CoreSDK 的 SDK（如 TypeScript）中，此指标的工作方式不同，应按 Worker 和 Task Queue 进行监控和调整。

`maxWorkflowThreadCount` 和 `workflow_active_thread_count` 参数仅适用于 Java SDK。

:::

## 可用的 Task Queue 信息 {#task-queue-metrics}

:::tip 支持、稳定性和依赖信息

本节列出的信息可使用 [Go SDK](https://github.com/temporalio/sdk-go/blob/74320648ab0e4178b1fedde01672f9b5b9f6c898/client/client.go) 中的 `DescribeTaskQueueEnhanced` 方法、[Temporal CLI](https://github.com/temporalio/cli/releases/tag/v1.1.0) 的 `task-queue describe` 命令以及通过 RPC 使用 `DescribeTaskQueue` 来读取。

:::

Temporal Service 分别报告每种 Task Queue 类型（非聚合）的信息。
使用以下 Task Queue 属性来检索和评估有关 Task Queue 健康状况和性能的信息。
可用数据包括：

- [`ApproximateBacklogCount`](#ApproximateBacklogCountAndAge) 和 [`ApproximateBacklogAge`](#ApproximateBacklogCountAndAge)
- [`TasksAddRate`](#TasksAddRate-and-TasksDispatchRate) 和 [`TasksDispatchRate`](#TasksAddRate-and-TasksDispatchRate)
- [`BacklogIncreaseRate`](#BacklogIncreaseRate)（由 [`TasksAddRate`](#TasksAddRate-and-TasksDispatchRate) 和 [`TasksDispatchRate`](#TasksAddRate-and-TasksDispatchRate) 导出）

### `ApproximateBacklogCount` 和 `ApproximateBacklogAge` {#ApproximateBacklogCountAndAge}

`ApproximateBacklogCount` 表示当前在此 Task Queue 中积压的 Task 的大致数量。
该数量可能包括已过期的 Task 以及活跃的 Task，但最终会随着时间的推移收敛到正确的计数。

`ApproximateBacklogAge` 返回积压中最旧 Task 的大致时间。
该时间基于队列头部 Task 的创建时间。

您可以在做扩展决策时依赖这两个计数。

请注意：[粘性队列](https://docs.temporal.io/sticky-execution)会影响这些值，但只会影响几秒钟。
这是因为发送到粘性队列的 Task 不包含在 `ApproximateBacklogCount` 和 `ApproximateBacklogAge` 的返回值中。
随着积压增长，不准确性会减少。

### `TasksAddRate` 和 `TasksDispatchRate` {#TasksAddRate-and-TasksDispatchRate}

报告每秒添加到或从 Task Queue 派发的大致 Task 数量。
该速率是最近 30 秒时间间隔内的平均值。
计算包括添加到积压或从积压派发的 Task，以及立即派发并绕过积压（同步匹配）的 Task。

实际 Task 交付数量可能明显高于这两个值报告的数量：

- 即时派发是指 SDK 使用一次 Workflow Task 完成响应请求 Activity 的 Temporal 功能。
  使用即时派发的 Task 不经过 Task Queue。
- 传递到粘性 Task Queue 的 Task 不包含在 `TasksAddRate` 和 `TasksDispatchRate` 的返回值中。

### `BacklogIncreaseRate` {#BacklogIncreaseRate}

近似计算每秒添加到积压的_净_ Task 数，以最近 30 秒为平均值。
计算方式为：

```
TasksAddRate - TasksDispatchRate
```

- `X` 的正值表示积压以大约每秒 `X` 个 Task 的速率增长。
- `X` 的负值表示积压以大约每秒 `X` 个 Task 的速率缩减。

虽然由于即时和粘性 Task Queue 的原因，个别 `add` 和 `dispatch` 速率可能不准确，但 `BacklogIncreaseRate` 可靠地反映了积压超过几秒钟后的缩减或增长速率。

## 评估 Task Queue 性能 {#evaluate-worker-loads}

[Task Queue](https://docs.temporal.io/task-queue) 是一个轻量级、动态分配的队列。
[Worker Entity](/workers#worker-entity) 轮询队列以获取 [Task](https://docs.temporal.io/tasks#task) 并检索要处理的 Task。
Task 是 Worker 使用特定 Workflow Execution、Activity Execution 或 Nexus Task Execution 推进的上下文。
每种 Task Queue 类型向兼容的 Worker 提供其 Task 以完成 Task。
Temporal Service 动态创建不同的 [Task Queue 类型](/task-queue)，包括 Activity Task Queue、Workflow Task Queue 和 Nexus Task Queue。

通过准确估计积压 Task，您可以确定要部署的最优 Worker 数量。
平衡 Worker 数量与 Task 数量以实现最佳性能。
这种方法最大程度地减少 Task 积压饱和度并减少空闲 Worker。

Task Queue 数据提供了对 Task Queue 活动和积压特性的数字洞察。
使用这些数字来调整您的生产部署。
评估您的 Worker 负载并评估是否需要扩大或缩减 Worker 部署。

:::note 速率限制

[Visibility API 速率限制](/cloud/limits#visibility-api-rate-limit)适用于 Task Queue 性能数据请求。

:::

### 评估 Worker 可用性和容量问题 {#worker-capacity-issues}

每个 Temporal [Server](https://docs.temporal.io/temporal-service/temporal-server) 记录每次轮询请求的最后时间。
此时间显示在 `temporal task-queue describe` 输出中。

- `LastAccessTime` 值超过一分钟可能表示 Worker 集群已达到容量上限，或 Worker 已关闭或被移除。

- 低于 5 分钟的值通常表明 Worker 集群已达到容量上限。
  "达到容量上限"意味着所有 Workflow 和 Activity 槽位都已满。

- 自上次轮询请求以来超过 5 分钟的值通常表明 Worker 已关闭或被移除。
  如果最后一次轮询请求已过去 5 分钟，Worker 将被移除。

### 管理 Worker 集群 {#manage-your-worker-fleet}

您可以调整 Worker 数量以提高 Workflow Execution 性能并管理集群规模。
例如，大量积压的 Task 和过少的 Worker 会减慢 Workflow Execution 完成速度并降低处理效率。
添加更多 Worker 可以加快完成速度并提高吞吐量。
空积压表示 Worker 利用率低，允许您减少集群及相关成本。

`temporal task-queue describe` 提供的值可以帮助您管理 Worker 集群部署：

- `ApproximateBacklogAge` 显示 Task 等待派发的时间。
  如果这段时间增长过长，更多 Worker 可以提升 Workflow 效率。

- 通过将积压 Task 数量（`ApproximateBacklogCount`）除以 Worker 数量来计算每个 Worker 的需求。
  使用每个 Worker 的需求（每个 Worker 还需处理多少 Task）、积压消耗速率（`TasksDispatchRate`，Worker 处理 Task 的速率）以及派发延迟（`ApproximateBacklogAge`，最旧 Task 等待分配给 Worker 的时间）来确定您的 Task 处理速率是否在您需求的可接受范围内。

- 积压增长率（`BacklogIncreaseRate`）显示 Worker 随时间变化的需求变化。
  随着该速率增加，您可能需要添加更多 Worker，直到需求和容量平衡。
  随着它减少，您可以减少 Worker 集群规模。

## Task Queue 处理调优 {#task-queues-processing-tuning}

以下步骤限制由于 Worker 不足或不均衡导致的 Task Queue 处理延迟。
如果您注意到高 `schedule_to_start` 指标，请查看这些步骤。

这些步骤按推荐的执行顺序排列。

### 主机和资源配置

如果当前配置的 Worker 主机已完全利用（接近满 CPU 使用率、高平均负载等），则需要配置额外的 Worker 主机以增加 Worker 池的容量。

**可能有太多 Worker**

监控轮询成功率（`poll_success`/`poll_success_sync`）和轮询超时 `poll_timeouts` 服务器指标计数器。

轮询成功率 = (`poll_success` + `poll_success_sync`) / (`poll_success` + `poll_success_sync` + `poll_timeouts`)

在大多数稳定负载的系统中，轮询成功率应 >90%。对于高吞吐量和低延迟，尝试目标 >95%。

如果您同时看到：

1. 低轮询成功率，
2. 低 `schedule_to_start_latency`，
3. 低 Worker 主机资源利用率，

那么您可能有太多 Worker，考虑缩小规模。

### Worker 执行器槽位大小

调优时关注的主要领域是 Worker 执行器槽位的数量。
如果同时满足以下两个条件，请通过调整 `maxConcurrentWorkflowTaskExecutionSize` 或 `maxConcurrentActivityExecutionSize` 来增加工作槽位的最大数量：

1. Worker 主机未充分利用（CPU、平均负载等无瓶颈）。
2. 来自相应 Worker 类型的 `worker_task_slots_available` 指标频繁显示可用 Worker 槽位数量耗尽。

或者，考虑使用[此处](#slot-suppliers)描述的基于资源的 Slot Supplier。

### 轮询器数量

有时，增加 Task 轮询器的数量是合适的。
这在 Worker 与服务器通信时延迟较高的情况下更为常见。
您可以简单地使用自动化的轮询器调优来自动处理这个问题。

在以下情况下考虑手动调整：

1. Worker 主机未充分利用，例如，CPU、平均负载等无瓶颈。
2. 来自相应 Worker 类型的 `worker_task_slots_available` 指标显示，定期有大量百分比的 Worker 槽位可用。
3. `schedule_to_start` 指标异常高。

然后考虑通过调整 `maxConcurrentWorkflowTaskPollers` 或 `maxConcurrentActivityTaskPollers` 来增加轮询器数量，具体取决于哪种 `schedule_to_start` 指标偏高。

### 速率限制

如果在按照前面的说明调整轮询器和执行器数量后，您仍然观察到高 `schedule_to_start`、Worker 主机未充分利用或高 `worker_task_slots_available`，您可能需要检查以下内容：

- 如果通过 `WorkerOptions#maxTaskQueueActivitiesPerSecond` 设置了每个 Task Queue 的服务器端速率限制，请删除该限制或向上调整该值。（参见 [Go](/develop/go/workers/run-worker-process#taskqueueactivitiespersecond) 和 [Java](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/worker/WorkerOptions.Builder.html)。）
- 如果通过 `WorkerOptions#maxWorkerActivitiesPerSecond` 设置了每个 Worker 的 Worker 端速率限制，请删除该限制。（参见 [Go](/develop/go/workers/run-worker-process#workeractivitiespersecond)、[TypeScript](https://typescript.temporal.io/api/interfaces/worker.WorkerOptions#maxconcurrentactivitytaskexecutions) 和 [Java](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/worker/WorkerOptions.Builder.html)。）
