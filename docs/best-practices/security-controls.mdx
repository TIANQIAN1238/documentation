Temporal Cloud 提供自托管 Temporal 的功能作为托管服务；它不管理您的应用程序或 Worker。使用 Temporal SDK 编写的应用程序和服务仍然在您的计算环境中运行，并且您可以完全控制如何保护应用程序和服务。

这些最佳实践可确保您的 Temporal Cloud 环境遵守我们团队推荐的安全准则。您还可以了解有关我们的安全实践、合规状况的更多信息，并订阅漏洞 (CVE) Update：https://trust.Temporal.io/.

如果您有任何疑虑或疑问，请通过 security@Temporal.io 联系您的客户经理或我们的安全团队。

:::tip 

**留在 Updated 上 Temporal 安全公告：** 
订阅 [Temporal 信任门户](https://trust.temporal.io/) 上的 Temporal 安全 Update，以便您了解任何补丁或 CVE。虽然 Temporal Cloud 服务器端 Update 由供应商处理，但您的 Temporal SDK（在应用程序代码中）应保持最新。

:::

<a id="identity-and-access-management"></a>
## 身份和访问管理 {#identity-and-access-management}

Temporal Cloud 中强大的身份管理对于确保 Temporal 帐户的安全访问至关重要。至关重要的是，只有授权用户和服务才能访问您的 Temporal Cloud 帐户，并且每个用户和服务都拥有其角色所需的最低必要权限。

<a id="best-practices"></a>
### 最佳实践： {#best-practices}

<a id="1-enable-saml-single-sign-on-sso-for-user-access"></a>
#### 1. 启用 [SAML 单点登录](https://docs.temporal.io/cloud/saml) (SSO) 用户访问 {#1-enable-saml-single-sign-on-sso-for-user-access}

通过 SAML 2.0 将 Temporal Cloud 与您组织的身份提供商集成以进行集中身份验证。 SSO 允许您强制执行公司登录策略（MFA、密码复杂性等）。当您使用 Temporal Cloud 配置 SAML 时，您可以通过打开支持票证来禁用社交登录（即 Microsoft、Google）。

<a id="2-use-least-privilege-roles-for-temporal-cloud-users"></a>
#### 2. 对 Temporal Cloud 用户使用最低权限角色 {#2-use-least-privilege-roles-for-temporal-cloud-users}

Temporal Cloud 提供 [预配置的帐户级角色](https://docs.temporal.io/cloud/users)（账户所有者、财务管理员、全局管理员、开发人员、只读）和 Namespace 级别的权限。为用户分配他们需要的最低访问级别。例如，仅允许开发人员访问他们所处理的 Namespace，并为审核员或审阅者使用只读角色。定期检查用户角色并删除或降级不再需要的帐户

<a id="3-leverage-scim-or-automated-user-provisioning"></a>
#### 3. 利用 SCIM 或自动用户配置 {#3-leverage-scim-or-automated-user-provisioning}

如果适用，请使用 [SCIM](https://docs.temporal.io/cloud/scim) 或 Temporal Cloud 用户管理 API 自动添加和删除用户帐户。这可以确保当人们改变角色或离开组织时及时删除访问权限。

<a id="use-service-accounts-for-automation"></a>
#### 使用服务帐户实现自动化 {#use-service-accounts-for-automation}

对于非人类访问（CI/CD 管道、后端服务），请使用 [Temporal Cloud 服务帐户](https://docs.temporal.io/cloud/service-accounts) 而不是共享用户登录。服务帐户是可以被授予特定权限的机器身份，而与个人无关。为不同的应用程序或微服务创建具有唯一 API 密钥的单独服务帐户，并对每个应用程序或微服务应用最小权限（例如，只能访问一个 Namespace 的服务帐户）。

<a id="secure-application-authentication-and-api-access"></a>
## 安全应用程序身份验证和 API 访问 {#secure-application-authentication-and-api-access}

客户端与 Temporal 服务交互以启动和管理 Workflow，而 Worker 在您自己的环境中执行 Workflow 和活动中定义的业务逻辑。

加强使用的一个重要方面是确保这些交互的安全。 Temporal Cloud 为您的应用程序提供两种身份验证方法：相互 TLS 证书和 API 密钥。

<a id="best-practices"></a>
### 最佳实践： {#best-practices}

<a id="1-using-mutual-tls-mtls-provides-comprehensive-security"></a>
#### 1. 使用相互 TLS (mTLS) 提供全面的安全性 {#1-using-mutual-tls-mtls-provides-comprehensive-security}

Temporal Cloud 通过相互 TLS 根据 Namespace 保护其 gRPC 端点。这意味着您为 Namespace 提供证书颁发机构 (CA) 证书，并且所有 Temporal CLIents/Worker 必须提供由该 CA 签名的客户端证书。

我们建议您启用mTLS，为客户提供强有力的身份保证；它确保只有持有有效证书（由您信任的 CA 颁发）的系统才能连接。生成私钥和 CA 证书（或使用您的企业 CA）并将 CA 上传到 Temporal Cloud。不要在授权服务之外共享这些证书和关联密钥。

<a id="2-proactively-manage-and-rotate-certificates"></a>
#### 2. 主动管理和轮换证书 {#2-proactively-manage-and-rotate-certificates}

跟踪您的客户端和 [证书颁发机构证书](https://docs.temporal.io/cloud/certificates) 的到期日期。 Temporal Cloud信任上传的CA；如果过期，所有客户端授权都将失败。建立并自动化证书轮换 Schedule（例如，每季度轮换一次客户端证书，每年轮换一次 CA 证书，早在到期之前）。 Temporal 支持将新的 CA 证书与旧的 CA 证书一起上传，以实现无缝翻转。如果可能，请始终在临时环境中测试新证书。

<a id="3-if-youre-using-api-keys-handle-them-with-strict-care"></a>
#### 3. 如果您使用 API 密钥，请严格小心处理它们 {#3-if-youre-using-api-keys-handle-them-with-strict-care}

Temporal Cloud API 密钥是 mTLS 的替代方案，用于 SDK、CLI 和自动化的身份验证。如果您选择 API 密钥，请通过制定以下做法来严格谨慎地处理它们： 
- 保密：存储在秘密管理器中，而不是存储在代码或 Git 中。
- 至少每 90 天轮换一次：Temporal 可让您创建一把新钥匙，将其换入，然后删除旧钥匙。
- 每个服务/人一把钥匙：不可共享或重复使用。
- 监控使用情况并撤销异常：将 Temporal 审核日志提供给 SIEM。
- 可选：如果您的策略是“仅限 mTLS”，管理员可以禁用所有用户 API 密钥。

<a id="network-configuration-and-isolation"></a>
## 网络配置和隔离 {#network-configuration-and-isolation}

尽管 Temporal Cloud 是 SaaS 产品，但您仍保留对其网络配置的控制，从而可以采取定制的安全措施。通过最大限度地减少公共互联网暴露并将 Temporal Workflow 划分为合适的网络区域，您可以显着增强安全性并减少潜在的漏洞。这种方法可确保您的 Workflow 在您定义的网络边界内得到隔离和保护，即使同时利用基于云的服务的优势也是如此。

<a id="best-practices"></a>
### 最佳实践： {#best-practices}

<a id="1-use-private-connectivity"></a>
#### 1. 使用专用连接 {#1-use-private-connectivity}

Temporal Cloud 支持专用连接选项，例如 [AWS 私有链接](https://docs.temporal.io/cloud/connectivity/aws-connectivity) 和 [Google 云私有服务连接](https://docs.temporal.io/cloud/connectivity/gcp-connectivity)。如果您的基础设施位于 AWS 或 GCP 中，请为 Temporal Cloud 配置 PrivateLink/PSC 终端节点。这允许您的 Worker 和应用程序通过专用网络路径到达 Temporal Cloud，从而避免遍历公共互联网。专用连接减少了中间人攻击的范围，并且可以满足严格的网络安全策略。

<a id="2-separate-environments-by-namespace"></a>
#### 2.通过Namespace分隔环境 {#2-separate-environments-by-namespace}

使用 [Temporal Namespaces](https://learn.temporal.io/best_practice_guides/managing_a_namespace/#2-use-domain-service-and-environment-to-name-namespaces) 为不同环境或团队（例如开发、登台、生产）隔离 Workflow。每个Namespace在逻辑上都是隔离的，默认情况下不能与其他人交互，从而提供了安全边界。

确保您的生产 Namespace 使用更严格的网络控制（例如只能从产品网络访问），并且其凭据与非产品 Namespace 分开。这限制了较低环境中任何妥协的影响，并且由于 Workflow 数据仅对有权访问该 Namespace 的用户可见，因此通过 Namespace 分隔环境还强制执行数据-Visibility 边界。

<a id="data-protection-and-encryption"></a>
## 数据保护和加密 {#data-protection-and-encryption}

Temporal 的数据加密功能可确保您的 Workflow 的安全性和机密性，并在不影响性能的情况下提供保护。保护您发送到 Temporal Cloud 并存储在 Temporal Cloud 中的数据是我们的共同责任。 Temporal Cloud 已对服务器端的所有静态数据进行加密，但您可以添加额外的加密和控制层。

<a id="best-practices"></a>
### 最佳实践： {#best-practices}

<a id="1-enable-client-side-encryption-for-workflow-data"></a>
#### 1. 为 Workflow 数据启用客户端加密 {#1-enable-client-side-encryption-for-workflow-data}

Temporal提供可选的[数据转换框架](https://docs.temporal.io/dataconversion)（Data Converter）和负载编解码器接口；客户必须实施、部署和操作自己的自定义编解码器并管理加密密钥。

实际上，这意味着您可以在将任何敏感数据发送到 Temporal Cloud 之前对其进行加密，并且仅在客户端/Worker 端对其进行解密。由于加密密钥由您控制，因此您负责密钥生成、安全存储、轮换和版本控制。实现这一点涉及在 Temporal SDK 中开发自定义编解码器插件，并可选择（如果您需要在 Web UI 或 CLI 中检查解密的有效负载）部署专用的 Codec Server。

<a id="2-encode-workflow-failure-details-with-a-failure-converter"></a>
#### 2. 使用 [故障转换器](https://docs.temporal.io/failure-converter) 编码 Workflow 故障详细信息 {#2-encode-workflow-failure-details-with-a-failure-converter}

Temporal 的默认行为将错误消息和调用堆栈复制为纯文本，并且可以在 Workflow Execution 的消息字段中直接访问此文本。

如果您的失败消息和堆栈跟踪包含敏感信息，建议您配置 [故障转换器](https://docs.temporal.io/failure-converter) 对错误信息进行加密。这将加密有效负载中的 `message` 和 `stack_trace` 字段。

<a id="3-leverage-namespace-data-retention-policies"></a>
#### 3. 利用 Namespace 数据保留策略 {#3-leverage-namespace-data-retention-policies}

Temporal Cloud Namespace 具有 Workflow 历史记录（1 至 90 天）的 [保留期限](https://docs.temporal.io/temporal-service/temporal-server#retention-period) 设置。设置适当的保留期限以平衡操作需求和安全性。较短的保留意味着更快地清除已完成的 Workflow 数据（历史记录、有效负载），从而减少随时存储在云中的敏感数据量。记录您的保留选择，以符合公司的数据保留政策和监管要求。对于超过 90 天的保留期，可以将它们导出到您自己的 GCS 或 S3 存储桶。

<a id="availability-and-disaster-recovery"></a>
### 可用性和灾难恢复 {#availability-and-disaster-recovery}

Temporal Cloud 的平台专为开箱即用的容错而设计，但您可以确定哪些 Namespace 值得最高的可用性保证。使用下表来决定何时开启不同的高可用性模型以及如何实施它们。

| Namespace范围 | 使用案例 | 正常运行时间服务等级协议 | 恢复时间目标 (RTO) | 恢复点目标 (RPO) |
|-----------------|----------|------------|--------------------------------|--------------------------------|
| **单区域** | **如果您的应用程序是针对一个区域构建的，并且没有严格的高可用性或灾难恢复要求。** | 99.9% | ≤8小时 | ≤8小时 |
| **同区域复制** | **如果您想要更高的可用性，但您的应用程序是针对单个区域设计的，或者跨区域延迟不符合应用程序的 SLA** | 99.99% | ≤20分钟 | 接近于零（≈秒） |
| **多区域复制** | **如果 Workflow 的中断会导致收入损失、最终用户体验不佳或法规合规性问题。** | 99.99% | ≤20分钟 | 接近于零（≈秒） |
| **多云复制** | **如果您需要最高级别的容灾能力，防止整个云提供商（例如 AWS 或 GCP）发生中断** | 99.99% | ≤20分钟 | 接近于零（≈秒） |

<a id="best-practices"></a>
### 最佳实践： {#best-practices}

<a id="1-identify-availability-sensitive-namespaces"></a>
#### 1. 识别可用性敏感的 Namespace {#1-identify-availability-sensitive-namespaces}

运行业务影响分析来标记 Workflow，其中区域中断将导致重大客户、收入或安全影响。识别对可用性敏感的 Namespace，其中区域中断可能会产生巨大的业务影响，例如收入损失、客户体验不佳或无法履行法律义务。

<a id="2-enable-high-availability-for-business-critical-use-cases"></a>
#### 2. 为业务关键用例启用高可用性 {#2-enable-high-availability-for-business-critical-use-cases}

对于许多组织来说，由于严格的正常运行时间要求、合规性和监管需求，需要确保高可用性 (HA)。

对于这些关键用例，请为 [99.99% 合同 SLA](https://docs.temporal.io/cloud/high-availability#high-availability-features) 的特定 Namespace 启用高可用性功能。在 [同地域、多地域、多云复制](https://docs.temporal.io/cloud/high-availability) 之间进行选择时，建议使用多区域/多云复制来跨区域分发依赖项。使用物理上分离的区域可以提高应用程序的容错能力。

默认情况下，Temporal Cloud 为所有 Namespace 提供针对服务错误的 [99.9% 合同 SLA 保证](https://docs.temporal.io/cloud/high-availability)。

注：[为 Namespace 启用 HA 功能将使消耗成本增加 2 倍。](https://docs.temporal.io/cloud/pricing#high-availability-features)
