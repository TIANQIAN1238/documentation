:::info Applies to both open source and Temporal Cloud
本页介绍了适用于**开源 Temporal 和 Temporal Cloud 的 Namespace 最佳实践。
特定于平台的指南在全文中都有明确的标签。

有关参考文档，请参阅：
- [Namespace 概念](/namespaces)
- [管理 Namespace（开源）](/self-hosted-guide/namespaces)
- [Namespace (Temporal Cloud)](/cloud/namespaces)
:::

[Namespace](/namespaces) 是 Temporal 平台内的一个隔离单元。
它确保 Workflow Execution、Task Queue 和资源在逻辑上分离，防止冲突并实现安全的多租户使用。

<a id="naming-conventions"></a>
## 命名约定 {#naming-conventions}

<a id="use-lowercase-and-hyphens"></a>
### 使用小写字母和连字符 {#use-lowercase-and-hyphens}

在 Namespace 名称中使用小写字母和连字符 (`-`) 作为分隔符。

- **Temporal Cloud**：Namespace 名称不区分大小写，因此 `MyNamespace` 和 `myNamespace` 引用相同的 Namespace。
- **开源**：Namespace 名称区分大小写，因此 `MyNamespace` 和 `myNamespace` 是不同的 Namespace。

为了避免跨环境混淆，请始终使用小写。

**示例**：`payment-checkout-prd`

<a id="follow-a-consistent-naming-pattern"></a>
### 遵循一致的命名模式 {#follow-a-consistent-naming-pattern}

使用 `<use-case>-<domain>-<environment>` 之类的模式来命名 Namespace：

| 成分 | 最大长度 | 示例 |
|-----------|------------|----------|
| 使用案例 | 10 个字符 | `payments`、`fulfill`、`orders` |
| 领域 | 10 个字符 | `checkout`、`notify`、`inventory` |
| 环境 | 3 个字符 | `dev`、`stg`、`prd` |

**示例**：`payments-checkout-dev`、`fulfill-notify-prd`、`orders-inventory-stg`

**为什么是这种模式？**
- 简单易懂
- 清楚地区分环境
- 在域下对相关服务进行分组
- 允许平台团队向应用程序团队实施退款
- Namespace级别限制在不同服务和环境之间隔离

:::tip Temporal Cloud
云 Namespace 名称仅限于 [39 个字符](/cloud/namespaces#Temporal-cloud-Namespace-name)。
如果您需要包含区域，请使用短代码（例如 `aps1`、`use1`）。
:::

<a id="organizational-patterns"></a>
## 组织模式 {#organizational-patterns}

<a id="pattern-1-namespace-per-use-case-and-environment"></a>
### 模式 1：每个用例和环境的 Namespace {#pattern-1-namespace-per-use-case-and-environment}

适用于没有多个服务或团队边界的简单配置。

**命名约定**：`<use-case>_<environment>`

**示例**：`payments_prod`、`orders_dev`

<a id="pattern-2-namespace-per-use-case-service-and-environment"></a>
### 模式 2：每个用例、服务和环境 Namespace {#pattern-2-namespace-per-use-case-service-and-environment}

当属于同一用例的多个服务通过 API (HTTP/gRPC) 与 Temporal 进行外部通信时。

**命名约定**：`<use-case>_<service>_<environment>`

**示例**：`payments_gateway_prod`、`payments_processor_prod`

<a id="pattern-3-namespace-per-use-case-domain-and-environment"></a>
### 模式 3：每个用例、域和环境的 Namespace {#pattern-3-namespace-per-use-case-domain-and-environment}

当多个服务需要相互通信时，使用 [Temporal Nexus](/nexus) 跨 Namespace 边界连接 Workflow。
与共享 Namespace 相比，这提供了更好的安全性、故障隔离和模块化性。

**命名约定**：`<use-case>_<domain>_<environment>`

**示例**：`payments_checkout_prod`、`payments_refunds_prod`

对于没有 Nexus 的系统，服务可以通过同一 Namespace 内的 [Signals](/sending-messages#sending-Signals) 或 [Child Workflows](/child-workflows) 进行通信。

:::note Workflow ID uniqueness
当多个团队共享 Namespace 时，请为每个 Workflow ID 添加特定于服务的字符串作为前缀，以确保唯一性。
Task Queue 名称在 Namespace 中也必须是唯一的。
:::

<a id="production-safeguards"></a>
## 生产保障 {#production-safeguards}

### 使用授权者（仅限开源） {#authorizer}

在前端服务上使用自定义 [授权人](/self-hosted-guide/security#authorizer-plugin) 来设置对谁可以创建 Update 或弃用 Namespace 的限制。

如果未设置授权者，Temporal 将使用无条件允许所有 API 调用的 `nopAuthority` 授权者。

在Temporal Cloud、[基于角色的访问控制](/cloud/manage-access/roles-and-permissions#Namespace-level-permissions)上提供Namespace级别的授权，无需自定义配置。

### 启用删除保护（仅限 Temporal Cloud） {#deletion-protection}

[启用删除保护](/cloud/namespaces#delete-protection) 用于生产 Namespace，以防止意外删除。

### 启用高可用性（仅限 Temporal Cloud） {#high-availability}

对于具有严格正常运行时间要求的关键业务用例，请为 [99.99% 合同 SLA](/cloud/high-availability#high-availability-features) 启用 [高可用性功能](/cloud/high-availability)。

### 使用基础设施即代码（仅限 Temporal Cloud） {#terraform}

使用 [Temporal Cloud Terraform 提供商](/cloud/terraform-provider) 管理 Namespace。
如果 Terraform 不合适，针对 [云运营 API](/ops) 或 [tcld](/cloud/tcld) 编写脚本是一个不错的选择。

这提供了：
- 每个 Namespace 的用途和所有者的文档
- 防止基础设施漂移
- 版本控制的配置更改

在 Terraform 配置中使用 `prevent_destroy = true` 可以防止通过 Terraform 意外删除 Namespace。
这与 [Temporal Cloud删除保护](/cloud/namespaces#delete-protection) 是分开的，[Temporal Cloud删除保护](/cloud/namespaces#delete-protection) 可以防止通过任何接口进行删除。

**参考**：[Terraform 配置示例](https://github.com/kawofong/temporal-terraform)

## 标记（仅限 Temporal Cloud） {#tagging}

[标签](/cloud/namespaces#tag-a-Namespace) 是键值元数据对，可帮助组织、跟踪和管理 Namespace。

标签通过添加不适合 Namespace 名称的元数据来补充您的命名约定。
虽然名称捕获了用例、领域和环境，但标签可以捕获其他维度，例如团队所有权、数据敏感性或业务关键性。

<a id="recommended-tag-categories"></a>
### 推荐标签类别 {#recommended-tag-categories}

| 标签键 | 目的 | 示例 |
|---------|---------|----------|
| `environment` | 部署阶段 | `dev`、`staging`、`production` |
| `team` | 所属团队 | `platform`、`payments`、`identity` |
| `division` | 事业部 | `engineering`、`finance`、`ops` |
| `criticality` | 商业重要性 | `high`、`medium`、`low` |
| `data-sensitivity` | 数据分类 | `pii`、`pci`、`public` |
| `latency-sensitivity` | 性能层 | `realtime`、`batch`、`async` |

有关标签结构、限制和管理说明，请参阅 [如何标记 Namespace](/cloud/namespaces#tag-a-Namespace)。

<a id="sdk-client-configuration"></a>
## SDK客户端配置 {#sdk-client-configuration}

在 SDK 客户端中设置 Namespace 以隔离 Workflow Execution。
如果不设置 Namespace，则使用客户端启动的所有 Workflow Execution 将与 `default` Namespace 关联。

在将 Namespace 设置到客户端之前，您必须先注册它。

详细配置请参见：
- [Namespace 概念](/namespaces)
- [Namespace (Temporal Cloud)](/cloud/namespaces#access-Namespaces)
