Worker Versioning 是一项 Temporal 功能，允许您在不破坏 Workflow 的情况下，自信地将新更改部署到在 Worker 上运行的 Workflow。Temporal 通过帮助您管理不同的构建版本（正式称为 [Worker Deployment Versions](/worker-versioning#deployment-versions)）来实现这一点。

对于大多数团队，Worker Versioning 应该是在生产中部署 Workflow 代码更改的默认推荐方案。如果您可以运行版本化的 Worker 部署，请优先选择 Worker Versioning 而非 Patching。

Worker Versioning 为[蓝绿或彩虹部署](#deployment-systems)的用户解锁了重要优势：

- 逐步将流量引导至新的 Worker Deployment Version。
- 在将生产流量发送到新 Deployment Version 之前，先用测试进行验证。
- 检测到新 Deployment Version 出现问题时立即回滚。
- 采用时改善错误率。

此外，Worker Versioning 还引入了 **Workflow 固定（Workflow Pinning）**。对于已固定的 Workflow 类型，每次执行完全在其启动时所在的 Worker Deployment Version 上运行。您无需担心对正在运行的已固定 Workflow 进行破坏性代码更改。

要使用 Workflow 固定，我们建议使用[彩虹部署](#deployment-systems)。

## Worker Versioning 入门 {#definition}

要开始使用 Worker Versioning，您应该了解一些关于版本控制和部署的概念。

- **Worker Deployment** 是跨多个版本的部署或服务。在彩虹部署中，可以同时运行两个以上的活跃 Deployment Version。
- **Worker Deployment Version** 是部署或服务的一个版本。它可以有多个 Worker 在多个 Task Queue 上轮询，但它们都运行相同的构建。
- **Build ID** 与 Worker Deployment 名称结合使用，标识单个 Worker Deployment Version。
- 当版本化的 Worker 在 Task Queue 上轮询时，该 Task Queue 成为该 Worker 版本的一部分。该版本的 Worker Deployment 控制 Task Queue 如何将 Workflow Task 与 Worker 匹配。
- 使用 **Workflow 固定（Workflow Pinning）**，您可以将每种 Workflow 类型声明为具有**版本控制行为（Versioning Behavior）**，可以是 Pinned 或 Auto-Upgrade。
  - **Pinned** Workflow 保证在单个 Worker Deployment Version 上完成。
  - **Auto-Upgrade** Workflow 将随着您推出新代码版本而自动迁移到新代码版本，具体地说是其目标 Worker Deployment Version（下面定义）。因此，Auto-Upgrade Workflow 不限于单个 Deployment Version，需要通过[Patching](/workflow-definition#workflow-versioning) 手动保持重放安全性。
  - Pinned 和 Auto-Upgrade Workflow 都保证只在其 Worker Deployment 的 Current 或 Ramping Version 上启动。
  - Pinned Workflow 设计用于彩虹部署。参见[部署系统](#deployment-systems)。
  - Pinned Workflow 不需要 Patching，因为它们在完成之前都运行在相同的 Worker 和构建上。
  - 如果您期望您的 Workflow 运行时间比您希望 Worker Deployment Version 存在的时间更长，您应该将 Workflow 类型标记为 Auto-Upgrade。
- 每个 Worker Deployment 都有一个单一的[**Current Version**](/worker-versioning#versioning-definitions)，除非 Workflow 之前在不同版本上被固定，否则 Workflow 都被路由到此版本。
- 每个 Worker Deployment 可以有一个[**Ramping Version**](/worker-versioning#versioning-definitions)，可配置百分比的 Workflow 被路由到此版本，除非它们之前被固定在不同版本上。
- 对于给定的 Workflow，其[**目标 Worker Deployment Version**](/worker-versioning#versioning-definitions)是它将迁移到的下一个版本。

## 设置您的部署系统 {#deployment-systems}

如果您还没有，您需要为 Worker 选择一个容器部署解决方案。

您还需要在三种常见的部署策略中选择一种：

- **滚动部署（rolling deployment）**策略就地升级 Worker，对切换速度控制有限，Worker 回滚也很慢。滚动部署的资源占用最小，但往往提供比其他策略更低的可用性，且与 Worker Versioning 不兼容。
- **蓝绿部署（blue-green deployment）**策略同时维护两个"颜色"或 Worker Deployment Version，可以控制流量在它们之间的路由方式。这允许您通过即时回滚和流量灰度等功能最大化正常运行时间。Worker Versioning 启用了蓝绿部署所需的路由控制。
- **彩虹部署（rainbow deployment）**策略类似于蓝绿部署，但有更多颜色，允许 Workflow 固定。您可以自由部署 Workflow 的新修订版，同时旧版本在排空中。使用 Worker Versioning，Temporal 会在给定版本的所有 Workflow 都排空时通知您，以便您可以将其下线。

:::note

如果您使用 Kubernetes，您也可以选择使用 [Temporal Worker Controller](/production-deployment/worker-deployments/kubernetes-controller) 自动实现 Worker 的彩虹部署。

:::

如果您还不能支持蓝绿或彩虹风格的部署，请使用 [Patching](/patching) 作为临时方案，同时逐步迁移到版本化的部署模型。

## 为 Versioning 配置 Worker

您需要向 Worker 添加一些额外的配置参数以启用 Worker Versioning。有三个新参数，根据语言不同名称有所不同：

- `UseVersioning`：为此 Worker 启用 Versioning 功能。
- 一个 `Version`，用于标识此 Worker 将被允许执行的修订版。这是部署名称和 Build ID 编号的组合。
- （可选）[默认版本控制行为](#definition)。如果未设置，您将需要在每个 Workflow 上指定行为。或者您可以默认为 Pinned 或 Auto-Upgrade。

## 选择版本控制行为 {#choosing-behavior}

正确的版本控制行为取决于您的 Workflow 运行时间相对于部署频率的关系。

### 决策指南 {#decision-guide}

| Workflow 持续时间                   | 使用 Continue-as-New？ | 推荐行为                                                  | 是否需要 Patching？ |
| ----------------------------------- | ---------------------- | --------------------------------------------------------- | ------------------- |
| **短暂**（在下次部署前完成）        | 不适用                 | `PINNED`                                                  | 从不                |
| **中等**（跨越多次部署）            | 否                     | `AUTO_UPGRADE`                                            | 是                  |
| **长期**（数周至数年）              | 是                     | `PINNED` + [在 Continue-as-New 时升级](#upgrade-on-continue-as-new) | 从不      |
| **长期**（数周至数年）              | 否                     | `AUTO_UPGRADE` + Patching                                 | 是                  |

### 按 Workflow 类型分类的示例 {#behavior-examples}

| Workflow 类型       | 持续时间    | 推荐行为                  | 备注                           |
| ------------------- | ----------- | ------------------------- | ------------------------------ |
| 订单处理            | 分钟        | `PINNED`                  | 在下次部署前完成               |
| 支付重试            | 小时        | `PINNED` 或 `AUTO_UPGRADE` | 取决于部署频率                |
| 订阅计费            | 天          | `AUTO_UPGRADE`            | 可能跨越多次部署               |
| 客户实体            | 月至年      | `PINNED` + 在 CaN 时升级  | 使用 Continue-as-New 模式      |
| AI 代理 / 聊天机器人 | 周          | `PINNED` + 在 CaN 时升级  | 长时间休眠，使用 CaN           |
| 合规性审计          | 月          | `AUTO_UPGRADE` + Patching | 不能使用 CaN（需要完整历史）   |

:::info 使用 Continue-as-New 的长期运行 Workflow

如果您的 Workflow 使用 Continue-as-New 来管理历史大小，您可以在 CaN 边界处升级到新的 Worker Deployment Version，而无需 Patching。参见下面的[在 Continue-as-New 时升级](#upgrade-on-continue-as-new)。

:::

### 默认版本控制行为注意事项

如果您使用蓝绿部署，您应该默认使用 Auto-Upgrade 且不应使用 Workflow 固定。
否则，如果您的 Worker 和 Workflow 是新的，我们建议不要提供 `DefaultVersioningBehavior`。

一般来说，每种 Workflow 类型都应注明为 Auto-Upgrade 或 Pinned。如果您所有的 Workflow 在可预见的未来都是短期运行的，您可以默认为 Pinned。

许多正在迁移到 Worker Versioning 的用户会先默认为 Auto-Upgrade，直到他们有时间注释其 Workflow。这个默认值与旧版行为最相似。一旦每种 Workflow 类型都被注释，您就可以删除 `DefaultVersioningBehavior`。

如果有灰度发布（ramp），但 Current 或 Ramping 版本之一宕机或容量不足，新的或 Auto-Upgrade Workflow 可能出现队列阻塞限制。这会导致其他版本无法获取 Task 或速度变慢。

例如，您有一个 Current Version 和一个 50% 的 Ramping Version。如果您所有的 Current Version Worker 都宕机，您会期望至少 50% 的新 Workflow 进入 Ramping Version。但这不会发生，因为 Current Version 的 Task 正在阻塞队列。

:::note

请记住，父 Workflow 或之前的 Auto-Upgrade Workflow 的 Child Workflow 默认为 Auto-Upgrade 行为，而不是 Unspecified。

:::

您还需要确保了解 Activity 在不同 Worker Deployment Version 之间的工作方式。有关更多详情，请参阅 [Worker Versioning Activity 行为文档](#choosing-behavior)。

## 使用 CLI 推出更改

接下来，使用额外的配置参数部署您的 Worker。在做出任何 Workflow 修订之前，您可以使用 `temporal` CLI 检查当前正在轮询的 Worker 版本：

您可以使用 `temporal worker deployment describe` 查看属于 Deployment 的版本：

```bash
temporal worker deployment describe --name="$MY_DEPLOYMENT"
```

要激活 Deployment Version，使用 `temporal worker deployment set-current-version`，指定部署名称和 Build ID：

```bash
temporal worker deployment set-current-version \
    --deployment-name "YourDeploymentName" \
    --build-id "YourBuildID"
```

要将 Deployment Version 逐步扩展到您整体 Worker 集群的某个百分比，使用 `set-ramping version`，加上相同的参数和灰度百分比：

```bash
temporal worker deployment set-ramping-version \
    --deployment-name "YourDeploymentName" \
    --build-id "YourBuildID" \
    --percentage=5
```

您可以通过 `describe -w YourWorkflowID` 验证 Workflow 是否已切换到该版本：

```bash
temporal workflow describe -w YourWorkflowID
```

这将返回 Workflow 正在运行的新版本：

```
Versioning Info:

  Behavior               AutoUpgrade
  Version                llm_srv.2.0
  OverrideBehavior       Unspecified
```

## 移动已固定的 Workflow

有时您需要手动将一组已固定的 Workflow 从有 Bug 的版本迁移到修复版本。

如果您需要将已固定的 Workflow 迁移到新版本，请使用 `temporal workflow update-options`：

```bash
temporal workflow update-options \
    --workflow-id "$WORKFLOW_ID" \
    --versioning-override-behavior pinned \
    --versioning-override-deployment-name "$TARGET_DEPLOYMENT" \
    --versioning-override-build-id "$TARGET_BUILD_ID"
```

您可以一次移动多个匹配 `--query` 参数的 Workflow：

```bash
temporal workflow update-options \
  --query="TemporalWorkerDeploymentVersion=$TARGET_DEPLOYMENT:$BAD_BUILD_ID" \
  --versioning-override-behavior pinned \
  --versioning-override-deployment-name "$TARGET_DEPLOYMENT" \
  --versioning-override-build-id "$FIXED_BUILD_ID"
```

在这种情况下，您可能还需要使用其他 [Versioning API](/workflow-definition#workflow-versioning) 在"修复"构建中对您的 Workflow 进行 Patching，以便目标 Worker 可以正确处理移动的 Workflow。如果您对 Workflow 进行了[版本不兼容的更改](/workflow-definition#deterministic-constraints)，并且想要回滚到较早的版本，则无法对其进行 Patching。可以考虑将 [Workflow Reset](/workflow-execution/event#reset) 与您的迁移结合使用。

"Reset-with-Move"允许您以原子方式重置 Workflow 并在新重置的 Workflow 上设置版本控制覆盖，因此当它恢复执行时，所有新的 Workflow Task 将在您的新 Worker 上执行。

```bash
temporal workflow reset with-workflow-update-options \
    --workflow-id "$WORKFLOW_ID" \
    --event-id "$EVENT_ID" \
    --reason "$REASON" \
    --versioning-override-behavior pinned \
    --versioning-override-deployment-name "$TARGET_DEPLOYMENT" \
    --versioning-override-build-id "$TARGET_BUILD_ID"
```

## 将 Workflow 从 Pinned 迁移到 Auto-Upgrade

有时您需要将 Workflow 从 Pinned 迁移到 Auto-Upgrade，因为您错误地配置了 Workflow 类型的行为，或者您意外地固定了一个长期运行的 Workflow。

已固定的 Workflow 可能会阻塞版本排空，特别是当它们运行很长时间时。您可以将 Workflow 移动到新的构建，但这只会将问题推到下一个构建。

为了进行此更改，您需要将 Workflow 的版本控制行为从 Pinned 更改为 Auto-Upgrade。您可以使用 `temporal workflow update-options`：

```bash
temporal workflow update-options \
    --workflow-id "$WORKFLOW_ID" \
    --versioning-override-behavior auto_upgrade
```

如果您想将某种类型的所有 Workflow 移动到这个新配置，可以使用此命令：

```bash
temporal workflow update-options \
    --query="WorkflowType='$WORKFLOW_TYPE'" \
    --versioning-override-behavior auto_upgrade
```

您也可以按特定的 Build ID 进行过滤以限制应用的 Workflow 数量：

```bash
temporal workflow update-options \
    --query="WorkflowType='$WORKFLOW_TYPE' AND TemporalWorkerDeploymentVersion='$TARGET_DEPLOYMENT:$OLD_VERSION'" \
    --versioning-override-behavior auto_upgrade
```

:::note

当您将行为更改为 Auto-Upgrade 时，Workflow 将在其目标版本上恢复工作。因此，如果 Workflow 的目标版本与之前的 Pinned 版本不同，您应该确保[对 Workflow 代码进行 Patching](/patching#patching)。

:::

## 在 Continue-as-New 时升级 {#upgrade-on-continue-as-new}

使用 [Continue-as-New](/workflow-execution/continue-as-new) 的长期运行 Workflow 可以在 Continue-as-New 边界处升级到更新的 Worker Deployment Version，而无需 Patching。

此模式非常适合：

- 运行数月或数年的**实体 Workflow（Entity Workflows）**
- 使用 Continue-as-New 进行检查点的**批处理** Workflow
- 等待用户输入的长时间休眠的 **AI 代理 Workflow**

:::note 公开预览

此功能作为实验性 SDK 级别选项处于公开预览阶段。

:::

### 工作原理 {#upgrade-on-can-how-it-works}

默认情况下，已固定的 Workflow 在执行 Continue-as-New 时仍保留在其原始 Worker Deployment Version 上。启用升级选项后：

1. 每次 Workflow 运行都保持固定在其版本上（在一次运行中无需 Patching）
2. Temporal Server 在新的[目标版本](/worker-versioning#versioning-definitions)可用时通知 Workflow
3. 当 Workflow 使用升级选项执行 Continue-as-New 时，新的运行将在[目标版本](/worker-versioning#versioning-definitions)上启动

### 检查新版本 {#checking-for-new-versions}

当新的 Worker Deployment Version 成为 Current 或 Ramping 时，活跃的 Workflow 可以通过 `target_worker_deployment_version_changed` 检测到这一点：

```go
func (w *Workflows) ContinueAsNewWithVersionUpgradeV1(
  ctx workflow.Context,
  attempt int,
) (string, error) {
  if attempt > 0 {
    return "v1.0", nil
  }

	// Check GetTargetWorkerDeploymentVersionChanged periodically.
	// GetTargetWorkerDeploymentVersionChanged is refreshed after each WFT completes.
  for {
	// Trigger a WFT when timer expires, thereby refreshing the GetTargetWorkerDeploymentVersionChanged flag.
	// Since this is just a test workflow, we aren't doing any real work. In a real workflow regularly
	// doing non-sleep workflow tasks, you would not need to artificially trigger a WFT to refresh the
	// GetTargetWorkerDeploymentVersionChanged flag. You could choose to check the field periodically, or you
	// might want to check before accepting updates, starting activities, or starting child workflows.
	err := workflow.Sleep(ctx, 10*time.Millisecond)
	if err != nil {
	  return "", err
	}
	info := workflow.GetInfo(ctx)
	if info.GetTargetWorkerDeploymentVersionChanged() {
	  return "", workflow.NewContinueAsNewErrorWithOptions(
		ctx,
		workflow.ContinueAsNewErrorOptions{
		  // Pass InitialVersioningBehavior=workflow.ContinueAsNewVersioningBehaviorAutoUpgrade
		  // to make the new run start with AutoUpgrade behavior and use the Target Version of
		  // its Worker Deployment.
		  InitialVersioningBehavior: workflow.ContinueAsNewVersioningBehaviorAutoUpgrade,
		},
		"ContinueAsNewWithVersionUpgrade",
		attempt+1,
	  )
	}
  }
}

func (w *Workflows) ContinueAsNewWithVersionUpgradeV2(
  ctx workflow.Context,
  attempt int,
) (string, error) {
  return "v2.0", nil
}
```

### 限制 {#upgrade-on-can-limitations}

:::caution 当前限制

- **仅惰性移动：** Workflow 必须通过执行步骤来接收目标版本已更改的信息。
  休眠中的 Workflow 不会主动获取目标版本已更改的信息。如果您有想要唤醒的空闲 Workflow 以便它们可以检查 `GetTargetWorkerDeploymentVersionChanged`，可以向它们发送 Signal。
- **接口兼容性：** 在以 Continue-as-New 方式切换到不同版本时，确保前一个版本的 Workflow 定义提供的 Workflow 输入与新版本的 Workflow 定义兼容。如果不兼容，新的运行可能在其第一个 Workflow Task 上失败。

:::

## 下线旧的 Deployment Version

Worker Deployment Version 经历以下状态：

1. **非活跃（Inactive）**：该版本之所以存在，是因为具有该版本的 Worker 曾轮询过服务器。如果此版本从未变为 Active，它将永远不会是 Draining 或 Drained 状态。
2. **活跃（Active）**：该版本是 Current 或 Ramping，因此它正在接受新 Workflow 和现有的 Auto-Upgrade Workflow。
3. **排空中（Draining）**：该版本停止成为 Current 或 Ramping，并且它上面有开放的已固定 Workflow 在运行。由于排空状态是周期性更新的，可能存在短暂时期处于 Draining 状态但没有开放的已固定 Workflow。
4. **已排空（Drained）**：该版本正在排空，现在所有在其上运行的已固定 Workflow 都已关闭。

您可以在描述 Worker Deployment 时的每个 `VersionSummary` 的 `WorkerDeploymentVersionStatus` 中看到这些状态，或者直接描述该版本。当某个版本处于 Draining 或 Drained 状态时，在名为 `DrainageStatus` 的值中显示。Temporal Service 会定期刷新此状态，通过统计使用该版本的任何开放已固定 Workflow 来更新。

每次刷新时，`DrainageInfo.last_checked_time` 会更新。最终，`DrainageInfo` 会报告该版本已完全排空。此时，没有 Workflow 仍在该版本上运行，也没有更多会自动路由到它，因此您可以考虑关闭正在运行的 Worker。

您可以通过检查 `WorkerDeploymentInfo.VersionSummaries` 或使用 `temporal worker deployment describe-version` 来监控这一点：

```bash
temporal worker deployment describe-version \
    --deployment-name "YourDeploymentName" \
    --build-id "YourBuildID"
```

## 垃圾回收

Worker Deployment 永远不会被垃圾回收，但 _Worker Deployment Version_（通常称为 Version、Worker Version、Deployment Version）会被回收。

Version 被删除是为了将一个 Worker Deployment 中的版本总数保持在 [`matching.maxVersionsInDeployment`](https://github.com/temporalio/temporal/blob/a3a53266c002ae33b630a41977274f8b5b587031/common/dynamicconfig/constants.go#L1317-L1321) 以内，该值在 Temporal Cloud 中目前设置为 100，但这是一个保守的数字，如果需要可以增加。

例如，当您在 Worker Deployment 中部署第 101 个 Worker Version 时，服务器会查看 Worker Deployment 中最旧的已排空版本。如果该版本在过去 5 分钟内没有轮询器，服务器将删除它。如果该版本仍有轮询器，服务器将尝试下一个最旧的版本。如果 100 个版本中没有一个符合删除条件（即，没有一个是没有轮询器的已排空版本），则不会删除任何版本，第 101 个版本的轮询将失败。

此时，要成功部署第 101 个版本，您需要增加 `matching.maxVersionsInDeployment` 或停止从一个旧的已排空版本轮询，使其符合清理条件。

如果您想重新部署之前删除的版本，请使用与已删除版本具有相同 Build ID 和 Deployment Name 的 Worker 开始轮询，服务器将重新创建它。

这涵盖了使用 Worker Versioning 工作的完整生命周期。我们将继续改进此功能，欢迎通过侧边栏链接提供任何反馈或功能请求！
