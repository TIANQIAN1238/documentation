本页面介绍了自行托管 Temporal 的客户面临的常见挑战，并分享了缓解这些问题的建议。

Temporal 的核心是耐用性和可靠性。
为了确保这种耐用性和可靠性，必须根据最佳实践部署 Temporal 服务。

本指南提供了一种途径来证明 Temporal 消费者可以对 Temporal 服务充满信心，并提供了您作为用户应针对该服务执行的关键测试列表。

<a id="self-hosting-challenge-areas"></a>
## 自托管挑战区域 {#self-hosting-challenge-areas}

需要大量的工程设计和持续的努力来解决几个潜在的挑战：

- 峰值或不断增长的工作负载的可扩展性
- 全球托管
- 正常运行时间、可用性和可靠性
- 管理和控制平面
- 延迟，必须保持较低且一致
- [安全](/self-hosted-guide/security)
- 维护和升级
- 为服务用户提供专家支持
- 成本管理

其中每个组件都是构建关键任务 Temporal 服务的重要组成部分。
如果没有证明建筑耐用性，Temporal 的 [持久的执行力](https://temporal.io/how-it-works) 型号的价值就会受到影响。

## 可变或不断增长的工作负载的可扩展性 {#scaling-and-metrics}

工作负载可能变化很大，您可能会遇到持续的工作负载峰值。
Temporal 建议将集群扩展至远高于平均吞吐量。
有关该主题的介绍，请参阅 [缩放 Temporal：基础知识](https://temporal.io/blog/scaling-temporal-the-basics)。

Temporal Server 吞吐量通常受到为 Temporal 服务配置的 [碎片](/temporal-service/temporal-server#history-shard) 数量的限制。
分片是 Temporal 服务中的一个单元，通过它可以扩展并发 Workflow Execution 吞吐量。
分片容量（通常是整个集群吞吐量）是在集群构建时设置的，并且该集群设置以后无法调整。
如果需要添加更多分片需要重建集群并迁移到新集群。

您的 Temporal 服务的要求将根据您的预期生产工作负载而有很大差异。
您将需要运行自己的概念验证测试并观察关键指标以了解系统运行状况和扩展需求。

**负载测试。** 您可以使用[Omes 基准测试工具](https://github.com/temporalio/omes/)，看看我们自己如何[压力测试Temporal](https://temporal.io/blog/temporal-deep-dive-stress-testing/)，或者编写您自己的。

服务器发出的所有指标均为 [列在 Temporal 的源中](https://github.com/temporalio/temporal/blob/main/common/metrics/defs.go)。
您还可以从客户端配置等效指标。
在较高层面上，您需要跟踪以下 3 类指标：

- **服务指标**：对于服务处理程序发出的每个请求，我们都会发出带有 `type`、`operation` 和 `Namespace` 标签的 `service_requests`、`service_errors` 和 `service_latency` 指标。
  这为您提供了基本的服务使用情况，并允许您查看跨服务、Namespace 甚至操作的请求率。
- **持久性指标**：服务器为每个持久性操作发出 `persistence_requests`、`persistence_errors` 和 `persistence_latency` 指标。
  这些指标包括 `operation` 标签，以便您可以获得每个操作的请求率、错误率或延迟。
  这些对于识别数据库引起的问题非常有用。
- **Workflow Execution 统计**：服务器还会在 Workflow Execution 完成时发出计数器。
  这些对于获取有关 Workflow Execution 完成情况的总体统计数据非常有用。
  对每种类型的 Workflow Execution 完成使用 `Workflow_success`、`Workflow_failed`、`Workflow_timeout`、`Workflow_terminate` 和 `Workflow_cancel` 计数器。
  其中包括 `Namespace` 标签。

<a id="availability"></a>
## 可用性 {#availability}

高水平的可用性和可靠性 (99.99%) 是关键任务部署的要求。
Temporal 建议在负载测试时测试此可用性级别。
我们还建议在进行服务器升级时验证此级别的可靠性，以确保服务可用性不会丢失。

Temporal集群可以根据需要部署在多个区域，以满足不同的需求：

- 数据驻留
- 延迟
- 安全/隔离
  这会增加实施和维护集群的工作量。

[Temporal Cloud 在各个云提供商区域均可用](/cloud/service-availability)。

<a id="management-and-control-plane"></a>
## 管理和控制平面 {#management-and-control-plane}

Temporal 的成功导致了更大规模的 Temporal 部署。
需求可能会增加，并且可能从在单个区域中拥有一两个生产用例到在多个区域中拥有许多用例。
运行多个 Temporal 服务是一项复杂的工作，因为每个服务都需要自己的设置、调整和配置。

需要以统一的方式监控和管理所有 Temporal 服务会导致运营管理的痛苦。
考虑在 Temporal 之上添加一个层来管理多个 Temporal 服务：控制平面。
控制平面管理和引导数据流，决定数据包应发送到何处。
Temporal 服务数据平面可以简化操作并提高效率。
由于 Temporal 没有提供自己的开源数据平面，因此自行构建可能会很复杂并且需要花费精力来添加。

Temporal Cloud 正是提供了这种支持。
借助 Temporal Cloud，可以从单个视图管理所有区域中的所有 Namespace。
[Temporal Cloud](https://temporal.io/cloud) 还具有 RBAC 功能，可以委派各个 Namespace 的职责。

自托管 Temporal 不支持开箱即用的 RBAC 或审核日志记录。
Temporal Cloud 提供 RBAC 和 SSO 支持、审核日志记录、数据加密、第三方渗透测试验证以及 SOC 2-Type II 和 HIPAA 合规性。

<a id="maintenance-and-upgrades"></a>
## 维护和升级 {#maintenance-and-upgrades}

Temporal 建议您的服务器版本保持最新并且不要落后。

Temporal Server 是主动 Updated，并且每两周发布一次。
Temporal 推荐 [依次升级](/self-hosted-guide/upgrade-server)，不跳过任何次要版本，尽管您可以跳过补丁版本。
不保证对 Temporal Server 的支持，但非常旧的服务器甚至社区都很难支持，因此我们鼓励您保持最新状态。
您必须创建和维护基础架构来托管和运行自托管 Temporal 安装（例如 Kubernetes）以及用于持久性的数据存储。

服务器升级可能会对自托管 Temporal 服务可用性产生负面影响。
Temporal 建议在升级过程中进行负载和可用性测试，以了解性能影响。

Temporal Cloud Update 由 Temporal Cloud 团队管理；云升级是无缝的。

<a id="cost-management"></a>
## 成本管理 {#cost-management}

运行关键任务的全球 Temporal 服务可能会很昂贵。
Temporal Server 是一个运行和扩展复杂的系统。
Temporal 建议随着性能需求的变化进行性能测试和规划扩展。
遵循我们的指南可能会使您的自托管 Temporal Server 安装规模过大，但这对于处理不可预测的峰值工作负载是必要的。
性能测试可以帮助您调整环境规模。
将关键任务 Temporal 作为服务运行需要多个 Temporal 集群来实现高可用性和全球覆盖。

最好由经过培训、经验丰富、熟悉 Temporal 服务架构的管理员来维护您的 Temporal Server 并提供关键任务服务。
人员配置、培训和技能开发可能是维护 Temporal 服务的巨大成本。

[Temporal Cloud](https://temporal.io/cloud) 的设置和扩展成本显着降低。
