[Visibility](/temporal-service/visibility) 商店是您的一部分
[持久性存储](/temporal-service/persistence) 用于启用列出和过滤有关 Workflow Execution 的详细信息
存在于您的 Temporal 服务上。

Temporal 服务设置中需要 Visibility 存储，因为它由 [Temporal 网页用户界面](/web-ui) 使用，并且
[Temporal CLI](/cli) 用于提取 [Workflow Execution](/workflow-execution) 数据并启用批量操作等功能
在一组 Workflow Execution 上。

通过 Visibility 商店，您可以使用 [列表过滤器](/list-filter) 和 [Search Attributes](/search-attribute) 来列出
并过滤您想要查看或执行操作的 Workflow Execution。

支持的 Visibility 商店包括：

- Elasticsearch v7 与 Temporal Server v1.7 及更高版本
- Elasticsearch v8 与 Temporal Server v1.18 及更高版本
- OpenSearch 2+ 与 Temporal Server v1.30.1 及更高版本
- MySQL v8.0.17 及更高版本以及 Temporal Server v1.20 及更高版本
- PostgreSQL v12 及更高版本，带有 Temporal Server v1.20 及更高版本
- SQLite v3.31.0 及更高版本以及 Temporal Server v1.20 及更高版本

## 当前和旧版 Visibility 支持 {#supported-databases}

[高级Visibility](/visibility#advanced-visibility) 是当前一代的 Temporal Visibility。它支持
现代 Query 型号，包括 [定制 Search Attribute](/search-attribute#custom-search-attribute)。

此页面还包括旧版指南（在 Temporal Server v1.21 中弃用并在 v1.24 中删除）
[标准 Visibility](#legacy-standard-visibility) 模型适用于较旧的部署和迁移工作。在此背景下，
“高级”和“标准（旧版）”分别指当前和旧版 Temporal Visibility。

以下兼容性矩阵总结了每个商店支持哪一代 Visibility 和 Temporal Server
所需版本：

| 店铺                                                                                                               | 高级 Visibility（当前）                                                                                                                                    | 标准 Visibility（旧版）                                          |
| :------------------------------------------------------------------------------------------------------------------ | :--------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------- |
| [弹性搜索](#elasticsearch) <br/><br/>推荐用于产生多个 Workflow Execution 的任何设置 | Temporal Server v1.7+ 上的 v7、Temporal Server v1.18+ 上的 v8                                                                                                        | 不支持                                                         |
| [开放搜索](#elasticsearch)                                                                                        | Temporal Server v1.30.1+ 2+                                                                                                                                   | 不支持                                                         |
| [MySQL](#mysql)                                                                                                     | Temporal Server v1.20+ 上的 v8.0.17+                                                                                                                               | Temporal Server v1.24 之前的旧部署上的 v5.7+               |
| [PostgreSQL](#postgresql)                                                                                           | Temporal Server v1.20+ 上的 v12+                                                                                                                                   | Temporal Server v1.24 之前的旧部署上的 v9.6+               |
| [SQLite](#sqlite)                                                                                                   | Temporal Server v1.20+ 上的 v3.31.0+                                                                                                                               | 不支持                                                         |
| [卡桑德拉](#cassandra)                                                                                             | 不支持。<br/><br/>要从 Cassandra 迁移到受支持的高级 Visibility 存储，请参阅 [正在迁移 Visibility 数据库](#migrating-visibility-database)。 | 在 Temporal Server v1.21 中弃用，在 Temporal Server v1.24 中删除 |

您可以将支持的数据库的任意组合用于持久性和 Visibility 存储。对于 Update，请检查
[服务器发行说明](https://github.com/temporalio/temporal/releases)。

Temporal Server v1.21 引入了对 Temporal 服务中辅助 Visibility 存储的支持，以启用
[双 Visibility](/dual-visibility)。这对于迁移 Visibility 商店数据库非常有用。

## 如何设置MySQL Visibility 存储 {#mysql}

:::tip Support, stability, and dependency info

- MySQL v5.7 及更高版本。
- 高级 Visibility 可在 MySQL v8.0.17 及更高版本上与 Temporal Server v1.20 及更高版本一起使用。
- MySQL v5.7 支持应用于 Temporal Server v1.24 之前的旧标准 Visibility 部署。

:::

您可以将 MySQL 设置为您的 [Visibility 商店](/temporal-service/visibility)。验证
[支持的版本](/self-hosted-guide/visibility)，然后再继续。

如果使用 MySQL v8.0.17 或更高版本作为 Visibility 存储以及 Temporal Server v1.20 和更高版本，则任何
您创建的 [定制 Search Attribute](/search-attribute#custom-search-attribute) 必须与
Temporal 服务中的 Namespace。

<a id="persistence-configuration"></a>
### 持久化配置 {#persistence-configuration}

在持久性配置的 `VisibilityStore` 参数中设置 MySQL Visibility 存储名称，然后
在 `datastores` 下定义 Visibility 存储配置。

以下示例显示如何设置 Visibility 存储 `mysql-Visibility` 并在中定义数据存储配置
您的 Temporal 服务配置 YAML。

```yaml
#...
persistence:
  #...
  visibilityStore: mysql-visibility
  #...
  datastores:
    default:
      #...
    mysql-visibility:
      sql:
        pluginName: 'mysql8' # For MySQL v8.0.17 and later. For earlier versions, use "mysql" plugin.
        databaseName: 'temporal_visibility'
        connectAddr: ' ' # Remote address of this database; for example, 127.0.0.0:3306
        connectProtocol: ' ' # Protocol example: tcp
        user: 'username_for_auth'
        password: 'password_for_auth'
        maxConns: 2
        maxIdleConns: 2
        maxConnLifetime: '1h'
#...
```

有关配置参数和值的详细信息，请参见
[Temporal 服务配置](/references/configuration#sql)。

要在 MySQL Visibility 存储上启用高级 Visibility 功能，请使用 Temporal 升级到 MySQL v8.0.17 或更高版本
服务器 v1.20 或更高版本。请参阅 [升级服务器](/self-hosted-guide/upgrade-server#upgrade-server) 了解如何升级您的
Temporal Server 和数据库架构。

例如配置模板，请参见
[MySQL Visibility 存储配置](https://github.com/temporalio/temporal/blob/main/config/development-mysql8.yaml)。

<a id="database-schema-and-setup"></a>
### 数据库架构和设置 {#database-schema-and-setup}

Visibility 数据存储在名为 `executions_Visibility` 的数据库表中，并且必须使用以下架构创建
[MySQL v8.0.17 及更高版本](https://github.com/temporalio/temporal/tree/main/schema/mysql/v8/visibility)。

以下示例展示了如何使用以下命令将 MySQL 设置为持久性存储和 Visibility 存储：
`Temporal-sql-tool`。请参阅
[样本服务器存储库](https://github.com/temporalio/samples-server/tree/main/compose/scripts) 了解更多示例
与不同的数据库。

{/* SNIPSTART compose-mysql-setup */}
[撰写/脚本/setup-mysql.sh](https://github.com/temporalio/samples-server/blob/main/compose/scripts/setup-mysql.sh)
```sh
set -eu

# Validate required environment variables
: "${MYSQL_SEEDS:?ERROR: MYSQL_SEEDS environment variable is required}"
: "${MYSQL_USER:?ERROR: MYSQL_USER environment variable is required}"

echo 'Starting MySQL schema setup...'
echo 'Waiting for MySQL port to be available...'
nc -z -w 10 ${MYSQL_SEEDS} ${DB_PORT:-3306}
echo 'MySQL port is available'

# Create and setup temporal database
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal create
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal setup-schema -v 0.0
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal update-schema -d /etc/temporal/schema/mysql/v8/temporal/versioned

# Create and setup visibility database
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal_visibility create
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal_visibility setup-schema -v 0.0
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal_visibility update-schema -d /etc/temporal/schema/mysql/v8/visibility/versioned

echo 'MySQL schema setup complete'
```
{/* SNIPEND */}

请注意，该脚本使用
[Temporal-sql-工具](https://github.com/temporalio/temporal/blob/3b982585bf0124839e697952df4bba01fe4d9543/tools/sql/main.go)
运行安装程序。

## 如何设置 SQLite Visibility 存储 {#sqlite}

:::tip Support, stability, and dependency info

- SQLite v3.31.0 及更高版本。

:::

您可以将 SQLite 设置为 [Visibility 商店](/temporal-service/visibility)。验证
[支持的版本](/self-hosted-guide/visibility)，然后再继续。

Temporal 仅支持带有 SQLite 的内存数据库；这意味着数据库会在以下情况下自动创建：
Temporal Server 启动并在 Temporal Server 停止时被销毁。

您可以更改配置以使用基于文件的数据库，以便在 Temporal Server 停止时保留它。
但是，如果您使用基于文件的 SQLite 数据库，请升级数据库架构以启用高级 Visibility 功能
不支持；在这种情况下，您必须删除数据库并重新创建才能升级。

如果使用 SQLite v3.31.0 及更高版本作为 Visibility 存储以及 Temporal Server v1.20 及更高版本，则任何
您创建的 [定制 Search Attribute](/search-attribute#custom-search-attribute) 必须与
Temporal 服务中的 Namespace。

<a id="persistence-configuration"></a>
### 持久化配置 {#persistence-configuration}

在持久性配置的 `VisibilityStore` 参数中设置 SQLite Visibility 存储名称，然后
在 `datastores` 下定义 Visibility 存储配置。

以下示例显示如何设置 Visibility 存储 `sqlite-Visibility` 并在中定义数据存储配置
您的 Temporal 服务配置 YAML。

```yaml
persistence:
  # ...
  visibilityStore: sqlite-visibility
  # ...
  datastores:
    # ...
    sqlite-visibility:
      sql:
        user: 'username_for_auth'
        password: 'password_for_auth'
        pluginName: 'sqlite'
        databaseName: 'default'
        connectAddr: 'localhost'
        connectProtocol: 'tcp'
        connectAttributes:
          mode: 'memory'
          cache: 'private'
        maxConns: 1
        maxIdleConns: 1
        maxConnLifetime: '1h'
        tls:
          enabled: false
          caFile: ''
          certFile: ''
          keyFile: ''
          enableHostVerification: false
          serverName: ''
```

SQLite（v3.31.0 及更高版本）默认启用高级 Visibility。

<a id="legacy-standard-visibility"></a>
## 旧标准 Visibility 配置 {#legacy-standard-visibility}

以下部分适用于仍使用标准 Visibility 的较旧的自托管部署。对于新部署，
使用本页前面描述的高级 Visibility 后端之一。

### 如何设置 Cassandra Visibility 商店 {#cassandra}

:::tip Support, stability, and dependency info

- Cassandra 仅支持标准 Visibility。标准 Visibility 在 Temporal Server v1.21 中已弃用并删除
  在 v1.24 中。对于 Update，请检查 [Temporal Server 发行说明](https://github.com/temporalio/temporal/releases)。
- 我们建议从 Cassandra 迁移到 Visibility 的任何其他支持的数据库。

:::

Cassandra 不支持高级 Visibility。

要启用当前的 Visibility 功能，请使用 MySQL、PostgreSQL、SQLite、Elasticsearch 或 OpenSearch 作为您的 Visibility
商店。对于处理多个 Workflow 的任何 Temporal 服务设置，我们建议使用 Elasticsearch 或 OpenSearch
执行，因为这些后端支持 Visibility 请求加载并有助于优化性能。

要从 Cassandra 迁移到支持的 SQL 数据库，请参阅
[正在迁移 Visibility 数据库](#migrating-visibility-database)。

<a id="persistence-configuration"></a>
### 持久化配置 {#persistence-configuration}

在持久性配置的 `VisibilityStore` 参数中设置 Cassandra Visibility 存储名称，然后
在 `datastores` 下定义 Visibility 存储配置。

以下示例显示如何设置 Visibility 存储 `cass-Visibility` 并在中定义数据存储配置
您的 Temporal 服务配置 YAML。

```yaml
#...
persistence:
  #...
  visibilityStore: cass-visibility
  #...
  datastores:
    default:
    #...
    cass-visibility:
      cassandra:
        hosts: '127.0.0.1'
        keyspace: 'temporal_visibility'
#...
```

<a id="database-schema-and-setup"></a>
### 数据库架构和设置 {#database-schema-and-setup}

Visibility 数据存储在名为 `executions_Visibility` 的数据库表中，必须根据
https://github.com/temporalio/temporal/tree/main/schema/cassandra/visibility. 中定义的模式（由支持的版本）

以下示例演示如何使用 `Temporal-cassandra-tool` 设置 Cassandra Visibility 存储。了解更多
不同数据库的示例，请参阅
[样本服务器存储库](https://github.com/temporalio/samples-server/tree/main/compose/scripts)。

```bash
#...
# set your Cassandra environment variables
: "${KEYSPACE:=temporal}"
: "${VISIBILITY_KEYSPACE:=temporal_visibility}"

: "${CASSANDRA_SEEDS:=}"
: "${CASSANDRA_PORT:=9042}"
: "${CASSANDRA_USER:=}"
: "${CASSANDRA_PASSWORD:=}"
: "${CASSANDRA_TLS_ENABLED:=}"
: "${CASSANDRA_CERT:=}"
: "${CASSANDRA_CERT_KEY:=}"
: "${CASSANDRA_CA:=}"
: "${CASSANDRA_REPLICATION_FACTOR:=1}"
#...
# set connection details
#...
# set up Cassandra schema
setup_cassandra_schema() {
  #...
  # use valid schema for the version of the database you want to set up for Visibility
    VISIBILITY_SCHEMA_DIR=${TEMPORAL_HOME}/schema/cassandra/visibility/versioned
    if [[ ${SKIP_DB_CREATE} != true ]]; then
        temporal-cassandra-tool --ep "${CASSANDRA_SEEDS}" create -k "${VISIBILITY_KEYSPACE}" --rf "${CASSANDRA_REPLICATION_FACTOR}"
    fi
    temporal-cassandra-tool --ep "${CASSANDRA_SEEDS}" -k "${VISIBILITY_KEYSPACE}" setup-schema -v 0.0
    temporal-cassandra-tool --ep "${CASSANDRA_SEEDS}" -k "${VISIBILITY_KEYSPACE}" update-schema -d "${VISIBILITY_SCHEMA_DIR}"
  #...
}
```

## 如何将 Elasticsearch 或 OpenSearch 集成到 Temporal 服务 {#elasticsearch}

您可以将 Elasticsearch 或 OpenSearch 与 Temporal 服务集成作为您的 Visibility 商店。我们建议使用
用于 Temporal 服务大规模操作的后端之一。

要将 Elasticsearch 或 OpenSearch 与您的 Temporal 服务集成，请编辑您的 `persistence` 部分
`development.yaml` 配置文件，将搜索后端添加为 `VisibilityStore`，并运行索引架构设置
命令。

使用以下版本指南：

- Temporal Server v1.7 及更高版本支持 Elasticsearch v7。
- Temporal Server v1.18 及更高版本支持 Elasticsearch v8。
- Temporal Server v1.30.1 及更高版本支持 OpenSearch 2+。

本节中的示例使用 Elasticsearch。对于 OpenSearch，请使用相同的数据存储配置形状并
操作流程，除非您的目标 Temporal Server 版本的发行说明另有说明。

<a id="persistence-configuration"></a>
### 持久化配置 {#persistence-configuration}

在持久性配置的 `VisibilityStore` 参数中设置 Visibility 商店名称，然后定义
在`datastores`下搜索后端配置。

以下示例演示如何设置名为 `es-Visibility` 的 Visibility 存储并定义 Elasticsearch 数据存储
Temporal 服务配置 YAML 中的配置。

```yaml
persistence:
  ...
  visibilityStore: es-visibility
  datastores:
    ...
    es-visibility: # Define the Elasticsearch datastore connection information under the `es-visibility` key
      elasticsearch:
        version: "v7"
        url:
          scheme: "http"
          host: "127.0.0.1:9200"
        indices:
          visibility: temporal_visibility_v1_dev
```

### 索引架构和索引 {#index-schema-and-index}

`Temporal-elasticsearch-tool`。有关不同数据库的更多示例，请参阅

{/* SNIPSTART compose-mysql-es-setup */}

```sh
: "${ES_VERSION:?ERROR: ES_VERSION environment variable is required}"

: "${MYSQL_SEEDS:?ERROR: MYSQL_SEEDS environment variable is required}"
: "${MYSQL_USER:?ERROR: MYSQL_USER environment variable is required}"

echo 'Starting MySQL and Elasticsearch schema setup...'
echo 'Waiting for MySQL port to be available...'
nc -z -w 10 ${MYSQL_SEEDS} ${DB_PORT:-3306}
echo 'MySQL port is available'

# Create and setup temporal database
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal create
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal setup-schema -v 0.0
temporal-sql-tool --plugin mysql8 --ep ${MYSQL_SEEDS} -u ${MYSQL_USER} -p ${DB_PORT:-3306} --db temporal update-schema -d /etc/temporal/schema/mysql/v8/temporal/versioned

# Setup Elasticsearch index
# temporal-elasticsearch-tool is available in v1.30+ server releases
if [ -x /usr/local/bin/temporal-elasticsearch-tool ]; then
  echo 'Using temporal-elasticsearch-tool for Elasticsearch setup'
  temporal-elasticsearch-tool --ep "$ES_SCHEME://$ES_HOST:$ES_PORT" setup-schema
  temporal-elasticsearch-tool --ep "$ES_SCHEME://$ES_HOST:$ES_PORT" create-index --index $ES_VISIBILITY_INDEX
else
  echo 'Using curl for Elasticsearch setup'
  echo 'WARNING: curl will be removed from admin-tools in v1.30.'
  echo 'Waiting for Elasticsearch to be ready...'
  max_attempts=30
  attempt=0
  until curl -s -f "$ES_SCHEME://$ES_HOST:$ES_PORT/_cluster/health?wait_for_status=yellow&timeout=1s"; do
    attempt=$((attempt + 1))
    if [ $attempt -ge $max_attempts ]; then
      echo "ERROR: Elasticsearch did not become ready after $max_attempts attempts"
      echo "Last error from curl:"
      curl "$ES_SCHEME://$ES_HOST:$ES_PORT/_cluster/health?wait_for_status=yellow&timeout=1s" 2>&1 || true
      exit 1
    fi
    echo "Elasticsearch not ready yet, waiting... (attempt $attempt/$max_attempts)"
    sleep 2
  done
  echo ''
  echo 'Elasticsearch is ready'
  echo 'Creating index template...'
  curl -X PUT --fail "$ES_SCHEME://$ES_HOST:$ES_PORT/_template/temporal_visibility_v1_template" -H 'Content-Type: application/json' --data-binary "@/etc/temporal/schema/elasticsearch/visibility/index_template_$ES_VERSION.json"
  echo ''
  echo 'Creating index...'
  curl --head --fail "$ES_SCHEME://$ES_HOST:$ES_PORT/$ES_VISIBILITY_INDEX" 2>/dev/null || curl -X PUT --fail "$ES_SCHEME://$ES_HOST:$ES_PORT/$ES_VISIBILITY_INDEX"
  echo ''
fi

echo 'MySQL and Elasticsearch setup complete'
```

<a id="dual-visibility"></a>
## 如何设置双 Visibility {#dual-visibility}

要启用 [双 Visibility](/dual-visibility)，请使用主 Visibility 商店设置辅助 Visibility 商店，
并配置您的 Temporal 服务以启用辅助 Visibility 存储上的读取和/或写入操作。

使用双 Visibility，您一次只能从一个 Visibility 存储中读取数据，但可以将您的 Temporal 服务配置为
仅写入主要存储、仅写入辅助存储或同时写入主要存储和辅助存储。

<a id="set-up-secondary-visibility-store"></a>
#### 设置二级 Visibility 商店 {#set-up-secondary-visibility-store}

在持久性配置中使用 `secondaryVisibilityStore` 配置密钥设置辅助存储，并且
然后在 `datastores` 下定义辅助 Visibility 存储配置。

您可以将任何 [支持的数据库](/self-hosted-guide/visibility) 配置为您的辅助存储。

示例：

要将 MySQL 配置为辅助存储，而 Cassandra 作为主存储，请执行以下操作。

```yaml
persistence:
  visibilityStore: cass-visibility # This is your primary Visibility store
  secondaryVisibilityStore: mysql-visibility # This is your secondary Visibility store
  datastores:
    cass-visibility:
      cassandra:
        hosts: '127.0.0.1'
        keyspace: 'temporal_primary_visibility'
    mysql-visibility:
      sql:
        pluginName: 'mysql8' # Verify supported versions. Use a version of SQL that supports advanced Visibility.
        databaseName: 'temporal_secondary_visibility'
        connectAddr: '127.0.0.1:3306'
        connectProtocol: 'tcp'
        user: 'temporal'
        password: 'temporal'
```

要将 Elasticsearch 配置为主存储和辅助存储，请使用配置密钥
`elasticsearch.indices.secondary_Visibility`，如下例所示。

```yaml
persistence:
  visibilityStore: es-visibility
  datastores:
    es-visibility:
      elasticsearch:
        version: 'v7'
        logLevel: 'error'
        url:
          scheme: 'http'
          host: '127.0.0.1:9200'
        indices:
          visibility: temporal_visibility_v1
          secondary_visibility: temporal_visibility_v1_new
        closeIdleConnectionsInterval: 15s
```

```bash

: "${ENABLE_ES:=false}"

# Set up Elasticsearch index
setup_es_index() {
```

<a id="migrating-visibility-database"></a>
## 如何迁移 Visibility 数据库 {#migrating-visibility-database}

要迁移您的 Visibility 数据库，[设立二级Visibility商店](#dual-visibility) 以启用
[双 Visibility](/dual-visibility) 和 Update 将 Temporal 服务中的动态配置读取到 Update
以及 Visibility 存储的写入操作。

双 Visibility 设置是可选的，但对于逐步将 Visibility 数据迁移到另一个数据库非常有用。

在开始之前，请验证 [支持的数据库和版本](/self-hosted-guide/visibility) 是否有 Visibility 商店。

以下步骤描述如何迁移 Visibility 数据库。

对 [Temporal 服务配置](/temporal-service/configuration) 进行任何更改后，请确保您
重新启动您的服务。

<a id="set-up-secondary-visibility-store"></a>
#### 设置二级 Visibility 商店 {#set-up-secondary-visibility-store}

1. 在您的 Temporal 服务配置中，
   [添加辅助 Visibility 商店](/references/configuration#secondaryvisibilitystore) 到您的 Visibility 设置下
   持久性配置。

示例：要从 Cassandra 迁移到 Elasticsearch，请将 Elasticsearch 添加为辅助数据库并进行设置。对于
   详情请参见[辅助 Visibility 数据库架构和设置](#dual-visibility)。

   ```yaml
   persistence:
   visibilityStore: cass-visibility
   secondaryVisibilityStore: es-visibility
   datastores:
     cass-visibility:
     cassandra:
       hosts: '127.0.0.1'
       keyspace: 'temporal_visibility'
     es-visibility:
     elasticsearch:
       version: 'v7'
       logLevel: 'error'
       url:
       scheme: 'http'
       host: '127.0.0.1:9200'
       indices:
       visibility: temporal_visibility_v1_dev
       closeIdleConnectionsInterval: 15s
   ```

1. 自托管上的 Update 和 [动态配置](/temporal-service/configuration#dynamic-configuration) 密钥
   Temporal 启用对辅助存储的写入操作并禁用读取操作的服务。例子：

   ```yaml
   system.secondaryVisibilityWritingMode:
   - value: "dual"
   constraints: {}
   system.enableReadFromSecondaryVisibility:
   - value: false
   constraints: {}
   ```

此时，从主存储读取 Visibility 数据，并将所有 Visibility 数据写入主存储
和二级商店。此设置仅适用于启用 Dual Visibility 后生成的新 Visibility 数据。它
不会将主存储中的任何现有数据迁移到辅助存储。

有关向辅助存储写入选项的详细信息，请参阅
[二次Visibility动态配置参考](/references/dynamic-configuration#secondary-visibility-settings)。

<a id="run-in-dual-mode"></a>
#### 在双模式下运行 {#run-in-dual-mode}

当您启用辅助存储时，只有新的 Visibility 数据才会写入主存储和辅助存储中。初级
存储仍保留设置辅助存储之前的 Workflow Execution 数据。

在双模式下运行可让您在设置辅助存储之前规划关闭和打开的 Workflow Executions 数据
在您的自托管 Temporal 服务中。

例子：

- 要管理关闭的 Workflow Execution 数据，请在双模式下运行，直到 Namespace
  已达到 [保留期限](/temporal-service/temporal-server#retention-period)。保留期结束后，
  Workflow Execution 数据已从持久性和 Visibility 存储中删除。如果你想保持关闭的Workflow
  在设定的保留期限后执行数据，必须设置[档案](/self-hosted-guide/archival)。
- 要管理所有打开的 Workflow Execution 的数据，请以双模式运行，直到所有 Workflow Execution 之前启动
  启用双 Visibility 模式已关闭。 Workflow Execution 关闭后，验证保留期并
  如果您需要在保留期限后保留数据，请设置存档。

您可以无限期地以双模式运行 Visibility 设置，或者直到您准备好弃用主模式
完全存储并移动到二级存储，而不会丢失数据。

<a id="deprecate-primary-visibility-store"></a>
#### 弃用主 Visibility 存储 {#deprecate-primary-visibility-store}

当您准备好弃用主商店时，请按照以下步骤操作。

1. Update 动态配置 YAML 以启用从辅助存储的读取操作。例子：

   ```yaml
   system.secondaryVisibilityWritingMode:
   - value: "dual"
   constraints: {}
   system.enableReadFromSecondaryVisibility:
   - value: true
   constraints: {}
   ```

此时，仅从辅助存储读取Visibility数据。验证二级存储上的数据是否存在
   正确。

1. 当二级商店经过审查并准备好取代您当前的主要商店时，更改您的 Temporal 服务
   配置将辅助存储设置为主存储，并删除前面设置的动态配置
   步骤。例子：

   ```yaml
   persistence:
   visibilityStore: es-visibility
   datastores:
     es-visibility:
     elasticsearch:
       version: 'v7'
       logLevel: 'error'
       url:
       scheme: 'http'
       host: '127.0.0.1:9200'
       indices:
       visibility: temporal_visibility_v1_dev
       closeIdleConnectionsInterval: 15s
   ```

## 管理自定义 Search Attribute {#custom-search-attributes}

要管理 Temporal Cloud 上的自定义 Search Attribute，请使用 [`tcld`](/cloud/tcld/namespace#search-attributes) CLI 工具。
使用 Temporal Cloud，您可以创建和重命名自定义 Search Attribute。如果您需要删除自定义 Search Attribute，请通过 [支持.Temporal.io](https://support.temporal.io) 联系支持人员。
要管理自托管 Temporal 服务上的自定义 Search Attribute，请使用 [Temporal CLI](/cli/operator#search-attribute)。
通过自托管 Temporal 服务，您可以创建和删除自定义 Search Attribute。

如果您是自托管，请在继续之前验证您的 [Visibility 数据库](/self-hosted-guide/visibility#supported-databases) 版本是否支持自定义 Search Attribute。

:::caution Do not use sensitive data or PII in Search Attributes

请勿在 Search Attribute **名称或值**中包含敏感数据、秘密或个人身份信息 (PII)。
Search Attribute 值未加密地存储在 Visibility 存储中，并且不由自定义 [有效负载编解码器](/payload-codec#payload-codec) 处理。
Temporal Server 必须能够以纯文本形式读取这些值以支持过滤和排序，因此在不破坏搜索功能的情况下不可能进行加密。

属性名称在 Namespace 配置、Query 表达式和 Temporal UI 中也可见。
在名称或值中使用敏感数据可能会导致任何具有 Namespace 访问权限的人面临暴露风险，并且可能违反 GDPR、HIPAA 或 SOC 2 等数据保护法规。

:::

### 如何创建自定义 Search Attribute {#create-custom-search-attributes}

在 Visibility 商店中创建自定义 Search Attribute 使其可在 Workflow 元数据中使用，并且
[列表过滤器](/list-filter)。

**关于Temporal Cloud**

要在 Temporal Cloud 上创建自定义 Search Attribute，请使用
[`tcld Namespace search-attributes add`](/cloud/tcld/namespace/#search-attributes)。例如，添加自定义搜索
将“CustomSA”赋予您的 Temporal Cloud Namespace “YourNamespace”，运行以下命令。
`tcld Namespace search-attributes add --Namespace YourNamespace --search-attribute "CustomSA"`

**在自托管 Temporal 服务上**

要在自托管 Temporal 服务 Visibility 存储中创建自定义 Search Attribute，请使用
`Temporal operator search-attribute create` 与 `--name` 和 `--type` 命令选项。

例如，要创建 `Keyword` 类型、名为 `CustomSA` 的 Search Attribute，请运行以下命令：

```
temporal operator search-attribute create --name="CustomSA" --type="Keyword"
```

请注意，如果您使用具有高级 Visibility 功能的 SQL 数据库，则需要在以下情况下指定 Namespace：
创建自定义 Search Attribute。例如：

```
temporal operator search-attribute create --name="CustomSA" --type="Keyword" --namespace="yournamespace"
```

您还可以在设置 Visibility 商店时创建多个自定义 Search Attribute。

以下示例显示如何在 SQL 数据库的 Visibility 存储设置期间创建自定义 Search Attribute。
有关设置示例，请参阅 [样本服务器存储库](https://github.com/temporalio/samples-server)

```bash
add_custom_search_attributes() {
    until temporal operator search-attribute list --namespace "${DEFAULT_NAMESPACE}"; do
      echo "Waiting for namespace cache to refresh..."
      sleep 1
    done
    echo "Namespace cache refreshed."

    echo "Adding Custom*Field search attributes."

    temporal operator search-attribute create --namespace "${DEFAULT_NAMESPACE}" --yes \
        --name="CustomKeywordField" --type="Keyword" \
        --name="CustomStringField" --type="Text" \
        --name="CustomTextField" --type="Text" \
        --name="CustomIntField" --type="Int" \
        --name="CustomDatetimeField" --type="Datetime" \
        --name="CustomDoubleField" --type="Double" \
        --name="CustomBoolField" --type="Bool"
}
```

对于 Temporal Server v1.19 及更早版本，或者如果使用 Elasticsearch 进行高级 Visibility，您可以创建自定义搜索
没有 Namespace 关联的属性，如以下示例所示。

{/* CHECK FOR ACCURACY */}

```bash
add_custom_search_attributes() {
       echo "Adding Custom*Field search attributes."
       temporal operator search-attribute create \
        --name="CustomKeywordField" --type="Keyword" \

        --name="CustomTextField" --type="Text" \
```

代码。

### 如何删除自定义 Search Attribute {#remove-custom-search-attributes}

要从自托管 Temporal 服务 Visibility 存储中删除 Search Attribute 密钥，请使用以下命令
`Temporal operator search-attribute remove`。 Temporal Cloud 不支持删除 Search Attribute。

例如，如果使用 Elasticsearch 进行高级 Visibility，则删除名为 `CustomSA` 的自定义 Search Attribute
type 关键字使用以下命令：

```
temporal operator search-attribute remove \
    --name="your_custom_attribute"
```

如果在 Temporal Server v1.20 及更高版本上使用 SQL 数据库进行高级 Visibility，则需要指定 Namespace
在您的命令中，如以下命令所示：

```
temporal operator search-attribute remove \
    --name="your_custom_attribute" \
    --namespace="your_namespace"
```

要检查 Search Attribute 是否已删除，请运行

```
temporal operator search-attribute list
```

并检查清单。

如果您使用的是 Temporal Server v1.20 及更高版本，请指定从中删除 Search Attribute 的 Namespace。例如，

```
temporal search-attribute list --namespace="yournamespace"
```

请注意，如果您将 [SQL数据库](/self-hosted-guide/visibility) 与 Temporal Server v1.20 及更高版本一起使用，则新的自定义
Search Attribute 映射到 Visibility 存储 `custom_search_attributes` 表中的数据库字段名称。去除
此自定义 Search Attribute 删除与数据库字段名称的映射，但不删除数据。如果你
删除自定义 Search Attribute 并添加新的，新的自定义 Search Attribute 可能会映射到数据库
最近被删除的字段。当您使用 List API 检索时，这可能会导致意外结果
使用新的自定义 Search Attribute 的结果。如果您使用 Elasticsearch，这些限制不适用。
