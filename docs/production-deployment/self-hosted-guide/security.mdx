Temporal Platform 在设计时考虑了安全性，您可以使用许多功能来保护平台本身和用户数据的安全。

安全的 Temporal Server 对其网络通信进行了加密，并为其接收的 API 调用设置了认证和授权协议。
没有这些措施，您的服务器可能会被未经授权的实体访问。

本页记录的是 Temporal 内置的可选安全措施。
但用户也可以选择使用反向代理设计自己的安全架构，或在 VPC 环境中运行不安全的实例。

### 服务器示例

https://github.com/temporalio/samples-server 仓库提供了两个示例，下面将进一步说明：

- **TLS：** 如何配置传输层安全（TLS）以加密与 Temporal Service 之间以及 Temporal Service 内部的网络通信。
- **Authorizer：** 如何注入一个低级授权器组件，该组件可以控制对所有 API 调用的访问。

### 使用 mTLS 加密传输中的数据

Temporal 支持双向传输层安全（mTLS）作为加密 Temporal Service 各服务之间以及应用程序进程与 Temporal Service 之间网络流量的方式。
自签名或正式颁发的证书都可以用于 mTLS。
mTLS 在 Temporal 的 [TLS 配置](/references/configuration#tls)中设置。
该配置包含两个部分，使得 Temporal Service 内部流量和外部流量可以使用不同的证书集和设置进行加密：

- `internode`：用于加密 Temporal Service 节点之间通信的配置。
- `frontend`：用于加密 Frontend 公共端点的配置。

可以使用 [WithConfig](/references/server-options#withconfig) 或 [WithConfigLoader](/references/server-options#withconfigloader) Server 选项传入自定义配置。

有关更多详情，请参阅 [TLS 配置参考](/references/configuration#tls)。

### 认证

有几种认证协议可用于防止未授权访问，例如服务器、客户端和用户的认证。

### 服务器

为了防止欺骗和[中间人攻击](https://en.wikipedia.org/wiki/Man-in-the-middle_attack)，您可以在各自 mTLS 配置的 `client` 部分指定 `serverName`。
这使已建立的连接能够对端点进行认证，确保呈现给任何连接客户端的服务器证书在其 CN 属性中具有适当的服务器名称。
它可以用于 `internode` 和 `frontend` 端点。

有关 mTLS 设置的更多指导，请参阅 [`samples-server` 仓库](https://github.com/temporalio/samples-server/tree/main/tls)，您也可以联系我们获取进一步指导。

### 客户端连接

为了限制客户端对 Temporal Service 端点的网络访问，您可以将其限制为仅允许具有特定证书颁发机构（CA）颁发证书的客户端。
在 [mTLS 配置](/references/configuration#tls)的 `internode` 和 `frontend` 部分中使用 `clientCAFiles`/`clientCAData` 和 `requireClientAuth` 属性。

### 用户

为了限制特定用户的访问，认证和授权通过可扩展点和插件执行，如下面的[授权](#authorization)部分所述。

#### 授权

Temporal 提供两个插件接口用于实现 API 调用授权：

- [`ClaimMapper`](#claim-mapper)

提供这些接口时，Frontend 会在执行请求的操作之前调用这些接口的实现。

请参阅 https://github.com/temporalio/samples-server/blob/main/extensibility/authorizer 获取示例实现。

### 单点登录集成

Temporal 可以通过使用 `ClaimMapper` 和 `Authorizer` 插件与单点登录（SSO）体验集成。
默认的 JWT `ClaimMapper` 实现可以直接使用，也可以作为类似插件的自定义实现的基础。

### Temporal UI

要在 Temporal UI 中使用环境凭证启用 SSO 认证，您需要使用定义身份提供者和 OAuth 设置的特定环境变量来配置 UI 容器。
在您的 docker-compose.yaml 中，设置 `TEMPORAL_AUTH_ENABLED=true` 以激活认证。
接下来，使用以下环境变量指定所需的 OAuth 凭证和端点：

- `TEMPORAL_AUTH_CLIENT_ID`
- `TEMPORAL_AUTH_CLIENT_SECRET`
- `TEMPORAL_AUTH_PROVIDER_URL`
- `TEMPORAL_AUTH_CALLBACK_URL`

这些值对应于您的 OAuth 身份提供者（如 Google、Auth0、Okta）提供的客户端凭证和端点。
正确配置后，Temporal UI 将将用户重定向到您的 SSO 登录页面并强制执行访问认证。
这种方法不需要任何额外的配置文件，非常适合使用安全环境变量注入的容器化环境。

```yaml
temporal-ui:
  container_name: temporal-ui
  depends_on:
    - temporal
  environment:
    - TEMPORAL_GRPC_ENDPOINT=temporal:7233
    - TEMPORAL_ADDRESS=temporal:7233
    - TEMPORAL_AUTH_ENABLED=true
    - TEMPORAL_AUTH_PROVIDER_URL=https://example.com
    - TEMPORAL_AUTH_CLIENT_ID=xxxxxxxxxxxxxx
    - TEMPORAL_AUTH_CLIENT_SECRET=xxxxxxxxxxxxxx
    - TEMPORAL_AUTH_CALLBACK_URL=https://your-domain/auth/sso/callback
    - TEMPORAL_AUTH_SCOPES=openid profile email
  image: temporalio/ui:latest
  networks:
    - temporal-network
  ports:
    - 8080:8080
```

有关配置的更多一般指导，请参阅 [Temporal UI README](https://github.com/temporalio/ui?tab=readme-ov-file#configuration)。
有关 Docker 配置的更多详情，请参阅 [Temporal UI 配置](https://github.com/temporalio/ui/blob/c95265ee6431fd0f6cf78ae06373885d66a8ee0c/server/docker/config-template.yaml)。

## Temporal Service 插件 {#plugins}

Temporal Service 支持一些可插拔组件。

### 什么是 ClaimMapper 插件？{#claim-mapper}

Claim Mapper 组件是一个可插拔组件，用于从 JSON Web Token（JWT）中提取声明（Claims）。

该过程通过 `GetClaims` 方法实现，该方法将来自调用方的 `AuthInfo` 结构体转换为关于调用方在 Temporal 中角色的 `Claims`。

Temporal 中的 `Role` 是一个位掩码，组合了一个或多个角色常量。
在以下示例中，角色被分配了允许调用方读取和写入信息的常量。

```go
role := authorization.RoleReader | authorization.RoleWriter
```

`GetClaims` 是可自定义的，可以通过 `temporal.WithClaimMapper` 服务器选项进行修改。
Temporal 还为您提供了一个默认的 JWT `ClaimMapper`。

典型的方法是让 `ClaimMapper` 解释来自调用方 JWT 的自定义 `Claims`（例如群组成员身份），并将其映射到用户的 Temporal 角色。
来自调用方 mTLS 证书的主体信息也可以作为确定角色的参数。

#### `AuthInfo`

`AuthInfo` 是传递给 `GetClaims` 的结构体。`AuthInfo` 包含从 gRPC 请求的 `authorization` 标头中提取的授权令牌。

`AuthInfo` 包含一个指向 `pkix.Name` 结构体的指针。
此结构体包含来自调用方 mTLS 证书的 [x.509](https://www.ibm.com/docs/en/ibm-mq/7.5?topic=certificates-distinguished-names) 可分辨名称（Distinguished Name）。

#### `Claims`

`Claims` 是包含授予调用方的权限声明信息的结构体。

`Authorizer` 假设调用方已经正确认证，并在做出授权决定时信任 `Claims`。

#### 默认 JWT ClaimMapper

Temporal 提供了一个默认的 JWT `ClaimMapper`，用于提取形成 Temporal `Claims` 所需的信息。
此插件需要一个公钥来验证数字签名。

要获取默认 JWT `ClaimMapper` 的实例，请调用 `NewDefaultJWTClaimMapper` 并提供：

- 一个 `TokenKeyProvider` 实例
- 一个 `config.Authorization` 指针
- 一个日志记录器

默认 `ClaimMapper` 的代码也可以用于构建自定义 `ClaimMapper`。

#### Token 密钥提供者

`TokenKeyProvider` 从指定发行者的 URI 获取公钥，这些 URI 遵循特定格式。
默认的 JWT `ClaimMapper` 使用此组件随时间获取和刷新公钥。

Temporal 提供了一个 `defaultTokenKeyProvider`。
该组件动态获取遵循 [JWKS 格式](https://tools.ietf.org/html/rfc7517)的公钥。
它支持 `RSA` 和 `ECDSA` 等格式。

```go
provider := authorization.NewDefaultTokenKeyProvider(cfg, logger)
```

:::note

`KeySourceURIs` 是返回 [JWKS 格式](https://tools.ietf.org/html/rfc7517)令牌发行者公钥的 HTTP 端点。
`RefreshInterval` 定义刷新密钥的频率。
例如，[Auth0](https://auth0.com/) 公开了 `https://YOUR_DOMAIN/.well-known/jwks.json` 等端点。

:::

默认情况下，使用"permissions"作为 `permissionsClaimName` 值的名称。

使用 `config.Config.Global.Authorization.JWTKeyProvider` 配置插件。

#### JSON Web Token 格式

默认的 JWT `ClaimMapper` 期望授权令牌格式如下：

```
Bearer <token>
```

JWT Token 中的权限声明（Permissions Claim）预期为单个权限声明（Individual Permission Claim）的集合。
每个单个权限声明的格式必须如下：

```
<namespace> : <permission>
```

然后，这些权限将被转换为调用方的 Temporal 角色。
可以是 Temporal 的以下四个值之一：

- read（读取）
- write（写入）
- worker（Worker）
- admin（管理员）

同一 Namespace 的多个权限将被 `ClaimMapper` 覆盖。

##### 默认 JWT ClaimMapper 的有效负载示例

```
{
   "permissions":[
      "temporal-system:read",
      "namespace1:write"
   ],
   "aud":[
      "audience"
   ],
   "exp":1630295722,
   "iss":"Issuer"
}
```

### 什么是 Authorizer 插件？{#authorizer-plugin}

`Authorizer` 插件包含一个 `Authorize` 方法，该方法对每个传入的 API 调用进行调用。
`Authorize` 接收有关 API 调用的信息，以及调用方的角色和权限声明。

`Authorizer` 允许广泛的授权逻辑，包括调用目标、角色/权限声明以及系统可用的其他数据。

#### 配置

必须将以下参数传递给 `Authorizer`：

- `context.Context`：调用的一般上下文。
- `authorization.Claims`：关于分配给调用方的角色的声明。其预期用途在本页前面的 [`Claims`](#claims) 部分中描述。
- `authorization.CallTarget`：API 调用的目标。

然后 `Authorizer` 返回以下两个决定之一：

- `DecisionDeny`：不调用请求的 API 调用，并向调用方返回错误。
- `DecisionAllow`：调用请求的 API 调用。

:::warning 安全警告

如果您**没有**明确配置 `Authorizer`，Temporal 将使用默认的 `noopAuthorizer`。此默认设置允许**每个** API 请求，没有任何认证或访问控制。任何能够访问您的 Temporal Server 的人都可以调用任何 API，包括敏感的管理操作。
这对生产环境或任何可被不可信客户端访问的环境（例如通过互联网）**都是不安全的**。

**要保护您的 Temporal Server，您必须配置带有相应 `ClaimMapper` 的 `Authorizer` 插件。** 没有这些配置，您的部署实际上对任何具有网络访问权限的人都是开放的。

:::

使用 [`temporal.WithAuthorizer`](/references/server-options#withauthorizer) 服务器选项配置您的 `Authorizer`，使用 [`temporal.WithClaimMapper`](/references/server-options#withclaimmapper) 服务器选项配置您的 `ClaimMapper`。

```go
temporalServer, err := temporal.NewServer(
	temporal.WithAuthorizer(newCustomAuthorizer()),
	temporal.WithClaimMapper(func(cfg *config.Config) authorization.ClaimMapper {
		return newCustomClaimMapper(cfg)
	}),
)
```

#### 如何授权 SDK API 调用 {#authorize-api-calls}

启用认证后，您可以授权对 Frontend Service 进行的 API 调用。

如果配置了[请求授权](#authorization)，Temporal Server [期望](#authentication) API 调用时在 gRPC 标头中传递带有授权令牌的 `authorization` 标头。

通过实现 `io.temporal.authorization.AuthorizationTokenSupplier` 接口，可以向 Temporal Java SDK 提供授权令牌。
该实现应用于创建 `io.temporal.authorization.AuthorizationGrpcMetadataProvider`，可以在 ServiceStub gRPC 拦截器列表上配置。

该实现对每个 SDK gRPC 请求调用，可以提供动态令牌。

**JWT**

可以通过这种方式传递的令牌类型之一是 JWT 令牌。
Temporal Server 提供了 [JWT 认证的默认实现](#default-jwt-claimmapper)。

**示例**

```java
AuthorizationTokenSupplier tokenSupplier =
  //your implementation of token supplier
  () -> "Bearer <Base64 url-encoded value of the token for default JWT ClaimMapper>";
WorkflowServiceStubsOptions serviceStubOptions =
  WorkflowServiceStubsOptions.newBuilder()
    //other service stub options
    .addGrpcMetadataProvider(new AuthorizationGrpcMetadataProvider(tokenSupplier))
    .build();
WorkflowServiceStubs service = WorkflowServiceStubs.newServiceStubs(serviceStubOptions);
WorkflowClient client = WorkflowClient.newInstance(service);
```

## Data Converter {#data-converter}

每个 Temporal SDK 都提供一个 [Data Converter](/dataconversion)，可以使用自定义 [Payload Codec](/payload-codec) 进行自定义，以编码和保护您的数据。

有关可以编码哪些数据、如何保护数据以及使用加密时需要考虑的内容，请参阅[数据加密](/production-deployment/data-encryption)。

#### Codec Server

您可以使用带有自定义 Payload Codec 的 [Codec Server](/codec-server)，通过远程端点在本地解码您在 Web UI 和 CLI 上看到的数据。
但是，在使用 Codec Server 之前，请确保您考虑了[远程数据编码](/remote-data-encoding)的所有安全影响。

有关如何设置 Codec Server 的详情，请参阅 [Codec Server 设置](/production-deployment/data-encryption#codec-server-setup)。
