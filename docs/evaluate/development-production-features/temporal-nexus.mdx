Temporal Nexus 现已[正式发布（Generally Available）](/evaluate/development-production-features/release-stages#general-availability)，适用于 [Temporal Cloud](/cloud/nexus) 和[自托管部署](/production-deployment/self-hosted-guide/nexus)。

随着 Temporal 在各团队中的普及，组织将应用程序划分为隔离的 Namespace 以实现安全性和故障隔离。
Nexus 跨越这些边界，通过内置的持久执行、可观测性和访问控制，将跨 Namespace、跨地区和跨云的 Temporal 应用连接起来。
每个团队保留对自己 Namespace 的所有权，同时通过简洁的服务契约共享能力。
观看 [Nexus 概览](https://www.youtube.com/watch?v=tJ1OwSFokOg&t=117s) 了解详细介绍。

## Nexus 之前的方式
连接 Namespace 是可行的，但很痛苦。它需要大量配置，增加了运维开销，并且通常依赖额外的基础设施：

- **Child Workflow** - 仅限于同一 Namespace。跨 Namespace 使用会暴露底层实现细节，要求调用方管理目标 Namespace、Task Queue 和 Workflow 选项。
- **Activity 包装器** - 每个目标都需要 mTLS 客户端，增加了配置和证书管理开销。通常权限过大，缺乏内置可观测性，且需要容易出错的异步结果样板代码。
- **额外的网关基础设施** - 不具备持久性，跨服务难以调试，且增加了另一个需要管理和修补的服务。

Nexus 用调用方和处理方之间的简洁服务契约替代了所有这些方式，减少了代码量，并提供了一等可观测性。

## 优势

通过以下特性，跨团队、跨域、跨地区和跨云连接 Temporal 应用：

- **更强的安全态势** - 服务契约的内置访问控制，而非宽泛的 Namespace 访问。每个团队控制自己的 Namespace、Worker 和部署生命周期。
- **更高的可靠性** - 持久的原子切换消除了请求丢失。故障被隔离，使得行为异常的 Worker 不会影响其他团队。
- **更易于构建和维护** - 减少样板代码、自定义重试和去重逻辑以及持续维护。团队专注于业务逻辑而非基础设施。
- **可扩展的平台模式** - 无需集中所有权即可实现跨团队和跨地区的编排。
- **降低跨团队用例的门槛** - 通过内置可发现性，轻松逐步构建和采用共享服务。
- **合规性和数据隔离** - 隔离的 Namespace 支持可审计性、数据驻留要求以及针对敏感数据（PCI、PII）的专用加密和访问控制。
