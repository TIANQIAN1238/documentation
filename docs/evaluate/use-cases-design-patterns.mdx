### 人在循环中 {#general-use-cases}

“人在循环”系统在某些步骤中需要人工交互，例如客户加入、表格或发票审批。
这些是由人类生成事件的事件驱动系统，并且由于人类与系统其他部分之间的时间或不可靠连接而可能难以实现。
他们可以使用 Schedule 和 Timer 提示用户输入。

现代开发团队经常根据他们的专业知识和项目要求使用不同的编程语言。 Temporal 通过内置的多语言功能支持这一点，允许团队在协同工作时继续使用他们喜欢的语言。

下面的示例展示了用不同语言编写的 Workflow Execution 如何相互发送消息。此示例中包含 Go、Java、PHP 和 TypeScript SDK。它还展示了如何正确传播错误，包括如何在用不同语言编写的 Workflow 之间执行此操作。

**示例**：[多语言示例](https://github.com/temporalio/temporal-polyglot)。

<a id="polyglot-systems"></a>
### 长时间运行的任务 {#polyglot-systems}

此用例与电子商务应用程序中的购物车 Workflow 等场景特别相关，您可以在其中高效处理长时间运行的任务，而无需在单独的数据库中管理状态。
它一次处理一条消息，确保每条消息仅处理一次。

此方法解决了消息处理时间过长可能出现的问题，这在其他系统中可能会导致使用者故障转移（通常具有默认的 5 分钟消息轮询超时），并可能导致多个使用者重复消息处理。
Temporal 处理延长任务持续时间的能力使其非常适合此类场景。
[心跳](/encyclopedia/detecting-activity-failures#activity-heartbeat) 功能可让您知道 Activity 仍在工作，从而深入了解长时间运行的流程的进度。

**示例**：[电子商务示例](https://learn.temporal.io/tutorials/go/build-an-ecommerce-app/)。

**代码示例**：[Temporal 电子商务](https://github.com/temporalio/temporal-ecommerce)

<a id="long-running-tasks"></a>
## 设计模式 {#long-running-tasks}

<a id="design-patterns"></a>
### 传奇 {#design-patterns}

Saga 模式是一种设计模式，用于通过将事务分解为一系列较小的、可管理的子事务来管理和处理复杂 Workflow 中的故障。
如果 Workflow 中的某个步骤失败，Saga 模式会通过执行特定操作来撤消前面的步骤来补偿此失败。
这确保了即使发生故障，系统也可以恢复到一致的状态。

**示例：**

- [使用 Python 构建旅行预订应用程序](https://learn.temporal.io/tutorials/python/trip-booking-app/)。
- [Saga 模式与 Temporal 白皮书](https://pages.temporal.io/download-saga-pattern-made-easy)
- [编排或编排你的传奇，这就是问题所在](https://temporal.io/blog/to-choreograph-or-orchestrate-your-saga-that-is-the-question)
- [佐贺网络研讨会](https://pages.temporal.io/on-demand-webinar-what-is-a-saga.html)

<a id="saga"></a>
### 状态机 {#saga}

状态机是一种软件设计模式，用于修改系统的行为以响应其状态的变化。
虽然状态机广泛应用于软件开发中，但将它们应用于复杂的业务流程可能是一项艰巨的任务。
Temporal 通过为 Workflow 开发提供结构化方法来简化状态机的复杂性，避免状态机所需的复杂状态管理代码。

**示例**：[状态机简化白皮书](https://pages.temporal.io/download-state-machines-simplified.html)
