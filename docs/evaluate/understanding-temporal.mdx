Temporal 提供了一种构建可扩展且可靠的应用程序的全新方法。

<a id="build-invincible-apps"></a>
## 构建无敌的应用程序 {#build-invincible-apps}

在任何复杂的系统中，故障都是必然会发生的。
软件工程师花费大量时间来确保他们构建的内容能够承受潜在的故障。
默认情况下，Temporal 使您的代码执行可靠且持久。

通常，如果发生崩溃，那么应用程序的执行状态就会丢失。
应用程序不记得故障之前发生的情况，需要大量的错误处理逻辑和复杂的恢复代码才能恢复。
该过程耗时且容易出错，难以确保可靠性。

Temporal 跟踪您的申请进度。
如果出现问题（例如断电），它可以保证您的应用程序可以从中断处继续运行 - 就像拥有终极自动保存功能一样。
将故障管理的责任从应用程序转移到平台，无需进行大量的恢复编码、测试和维护任务。

<a id="durable-execution"></a>
### 持久执行 {#durable-execution}

Temporal 是一个耐用的执行平台。
持久执行通过保证应用程序运行完成，确保应用程序在不利条件下仍能正确运行。
这种转变简化了开发过程。如果发生故障或崩溃，您的业务流程将继续无缝运行而不会中断。
开发人员将注意力转移到业务逻辑而不是基础设施问题上，并创建本质上可扩展和可维护的应用程序。

成千上万的开发人员信任 Temporal 的订单处理、客户引导和付款处理等用例，因为它使他们能够构建坚韧、耐用且“正常工作”的无敌应用程序。
有了 Temporal，无论发生什么情况，您的应用程序都可以保持运行。

<a id="temporal-application-the-building-blocks"></a>
## Temporal 应用：构建块 {#temporal-application-the-building-blocks}

### Workflow

从概念上讲，Workflow 是一系列步骤。
您可能在日常生活中遇到过 Workflow，无论是：

- 使用移动应用程序转账
- 预订假期
- 提交费用报告
- 创建新员工入职流程
- 部署云基础设施
- 训练人工智能模型

Temporal Workflow 是您的业务逻辑，在代码中定义，概述了流程中的每个步骤。

Temporal 不是无代码 Workflow 引擎 - 它是 **Workflows-as-Code**。
您无需在可视化界面中拖放步骤，而是使用您最喜欢的编程语言、代码编辑器和其他工具编写 Workflow 代码。
无代码引擎最终会达到其局限性，但是，Temporal 使您可以完全控制业务流程并具有灵活性。
这使您能够准确构建您需要的内容。

＃＃＃ 活动

活动是 Workflow 中的各个工作单元。
活动被定义为函数或方法，具体取决于编程语言。
活动通常涉及与外界交互，例如发送电子邮件、发出网络请求、写入数据库或调用 API，这些都容易失败。
您可以直接从 Workflow 代码调用活动。

如果 Activity 失败，Temporal 会根据您的配置自动重试。
由于 Activity 通常依赖于外部系统，因此可能会出现暂时性问题。
其中包括临时但关键的问题，例如网络故障、超时或服务中断。
您可以完全控制每个 Activity 重试的频率和次数。

<a id="activities"></a>
### SDK {#activities}

开发人员通过编写代码来创建 Temporal 应用程序，就像创建任何其他软件一样。

Temporal SDK（软件开发套件）是一个开源库，开发人员将其添加到其应用程序中以使用 Temporal。
它提供了以特定编程语言构建 Workflow、活动和各种其他 Temporal 功能所需的一切。

Temporal 提供七种 SDK：.NET、Go、Java、PHP、Python、Ruby、TypeScript。
由于 Temporal 支持多种编程语言，因此您可以为多语言团队混合搭配多种语言。
您可以轻松地将任何 Temporal SDK 添加到当前项目中，而无需更改已用于构建和部署的工具。
Temporal 非常适合您现有的技术堆栈。

<a id="sdk"></a>
## Temporal 服务 {#sdk}

Temporal 有两个主要部分：

1. 您的申请
2. Temporal 服务（一组服务和组件）

Temporal 架构的核心是 Temporal 服务，它为您的应用程序提供耐用性、可扩展性和可靠性。
您的应用程序与 Temporal 服务通信，Temporal 服务监督关键任务的执行，例如进行 API 调用，然后记录其完成情况。
它维护每个事件的详细历史记录，并将其可靠地保存到数据库中。

Temporal 服务的最大优势之一是它处理故障的方式。
Temporal 服务会详细记录您 Workflow 中的每一步。
通过保留 Workflow 中每一步的历史记录，它可以确保即使出现问题，您的 Workflow 也可以从上一个成功点继续。
Temporal 服务确切地知道在哪里恢复，而不会丢失任何工作。
这使您不必自己编写复杂的错误处理代码或艰苦的恢复机制。

您可以在自己的基础设施上运行 Temporal 服务，或使用 Temporal Cloud，这是一项托管服务，可处理运营开销并提供可扩展性和专家支持。

<a id="temporal-service"></a>
## Workers {#temporal-service}

Temporal 的真正优势来自于您的应用程序和 Temporal 服务的结合。
每当您的应用程序需要执行任务（例如发送通知或处理付款）时，Temporal 服务都会协调需要完成的操作。
Worker 是您的应用程序的一部分，由 Temporal SDK 提供，然后执行 Workflow 中定义的任务。

Worker 轮询 Temporal 服务以查看是否有可用任务，以及 Temporal 服务将该任务与 Worker 相匹配。
Worker 根据任务中指定的详细信息运行 Workflow 代码。

这种协作对于构建可靠、可扩展且持久的应用程序至关重要。
您可以运行多个 Worker（通常是数十个、数百个甚至数千个）来提高应用程序性能和可扩展性。

一个常见的误解是 Temporal 服务运行您的代码。
事实上，Worker 运行您的代码并直接处理您的数据。
Temporal 应用程序的设计是安全的。
Workflow 和活动无缝部署在您的基础设施中，并完全集成到您的应用程序中。
您的数据还受到加密库和密钥的保护。
您可以从头到尾保持对应用程序安全性的完全控制。

<a id="temporal-cli"></a>
### Event History {#temporal-cli}

借助 Temporal，您的 Workflow 可以从崩溃中无缝恢复。这是通过 [Event History](https://docs.Temporal.io/workflow-execution/event) 实现的，[Event History](https://docs.Temporal.io/workflow-execution/event) 是 Workflow Execution 生命周期中发生的所有事件的完整且持久的日志，以及 Temporal 服务在重播期间持久保留事件的能力。

Temporal 使用 Event History 记录沿途采取的每一步。例如，每次 Workflow Definition 进行 API 调用以执行 Activity 或启动 Timer 时，它不会直接执行该操作。相反，它会向 Temporal 服务发送命令。

命令是 Workflow 任务执行完成后由 Worker 向 Temporal 服务发出的请求操作。 Temporal 服务将根据这些命令执行操作，例如调度 Activity 或调度 Timer。然后，这些命令被映射到事件，这些事件在发生故障时会被保留。例如，如果 Worker 崩溃，Worker 使用 Event History 重播代码并将 Workflow Execution 的状态重新创建为崩溃前的状态。然后，它从故障点恢复进度，就好像故障从未发生过一样。

要深入了解 Event History 或命令，请访问 Temporal [百科页面](/encyclopedia/event-history) 或注册 [我们的课程](https://learn.Temporal.io/courses/) 之一。

<a id="event-history"></a>
## 像重力一样可靠 {#event-history}

Temporal 提供轻松的耐用性，即使底层基础设施发生故障，应用程序也可以不间断地运行数天、数周甚至数年。
这就是我们所说的_持久执行_。 Temporal 还代表了软件开发的范式转变。
这不仅仅是为了让现有的模式更加可靠；而是为了让现有的模式变得更加可靠。它是关于启用全新的方法来构建复杂的分布式系统。

Temporal 简化了状态管理，开发人员不必编写大量额外的代码来处理所有可能出错的事情。
凭借内置的可扩展性，Temporal 可确保您的应用程序顺利运行，无论其大小或复杂程度如何。
