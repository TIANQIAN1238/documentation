操作跟踪 Workflow 的进度（例如 Workflow 启动、Schedule 启动、Workflow 重置）和更广泛的进度
由 Temporal Cloud 启用的功能。

Temporal Cloud 操作是 Temporal Cloud 基于消耗的定价的主要单位。他们跟踪计费
Temporal Cloud 服务中的操作，例如启动 Workflow、记录心跳或发送消息。

以下结果导致对 Temporal Cloud 执行操作：

## Workflows

- **Workflow 开始**。通过客户端启动发生，[Continue-As-New](/workflow-execution/continue-as-new)，
  [Child Workflow](/child-workflows)启动。如果 Workflow 启动失败，则不会记录操作。去重 Workflow
  共享 Workflow ID 的启动_不_算作操作。
- **Workflow 重置**。当 [Workflow](/workflows) 复位时发生。 （在 a 之前发生的动作
  即使 [重置](/workflow-execution/event#reset) 在中不再可见，也会被计算在内
  [Event History](/workflow-execution/event#event-history)。）
- **Timer 开始**。包括设置超时时由 Temporal SDK 启动的隐式 Timer，例如
  Go 中的 `AwaitWithTimeout` 或 TypeScript 中的 `condition`。
- **Search Attribute 已请求更新插入**。每次调用 `UpsertSearchAttributes` 命令时都会发生。多重搜索
  单个 `UpsertSearchAttributes` 命令中的属性 Updated 计为一个操作。 Search Attribute 指定
  在 Workflow 启动期间，从操作计数中_排除_。 `TemporalChangeVersion` Search Attribute，用于
  Workflow 版本控制也不受操作计数的影响。
- **Signal 已发送**。每个 [Signal](/sending-messages#sending-signals) 都会发生一个操作，无论是从客户端发送还是从
  来自 Workflow。此外，任何 [Signal-带启动](/sending-messages#signal-with-start) 都会发生一个总动作，
  无论 Workflow 是否启动。
- **Query 由 Worker 收到**。每个 [Query](/sending-messages#sending-queries) 都会发生一个操作，包括查看
  Temporal Cloud UI 中的调用堆栈，这会在幕后生成 Query。
- **Update 由 Worker 收到**。每个成功的 [Update](/sending-messages#sending-updates) 都会发生一个操作，并且
  每个 [被拒绝](/handling-messages#update-validators) Update。这包括
  [Update-带启动](/sending-messages#update-with-start)，并且是除了启动动作之外的情况下，当
  Workflow 也启动。共享 Update ID 的重复数据删除 Update _不_算作操作。
- **Side Effect 已记录**。对于可变的 [Side Effect](/workflow-execution/event#side-effect)，仅发生一个操作
  当值改变时。
- **Workflow Execution 选项 Updated。** 每个操作都会发生一个操作
  [Workflow-执行-选项-Updated](/references/events#workflowexecutionoptionsupdated)事件。这包括
  附加 Workflow 完成回调或修改 Workflow 版本控制覆盖。

## Child Workflows

- **启动 Child Workflow** 和 **子 Workflow Execution**。当父 Workflow 持久记录意图
  启动 Child Workflow，它会产生两个操作，一个用于启动 Workflow，另一个用于尝试
  执行。

＃＃ 活动

- **Activity 已启动或重试**。每次启动或重试 Activity 时发生。
- **本地 Activity 开始**。与一个 Workflow 任务关联的所有 [当地活动](/local-activity) 都算作一个
  单一动作。这是因为 Temporal Cloud 计算了每个中的所有 [记录标记](/references/commands#recordmarker)
  Workflow 任务作为一项操作，而不是 _N_ 个操作。请注意：
  - 之后每个额外的 Workflow 任务心跳都算作一个额外的操作。
  - Workflow 任务心跳后重试的本地活动计为一项操作（上限为 100 项操作）。
- **Activity 心跳记录**。仅当来自 Activity 代码的 Heartbeat 调用到达
  [Temporal Server](/temporal-service/temporal-server)。 Temporal SDK 节流阀
  [Activity 心跳](/encyclopedia/detecting-activity-failures#activity-heartbeat)。默认油门为 80%
  [心跳超时](/encyclopedia/detecting-activity-failures#heartbeat-timeout)。心跳不适用于本地
  活动。

<a id="activities"></a>
## Schedules {#activities}

[Schedules](/schedule) 允许您在特定时间启动 Schedule 和 Workflow。每次执行 Schedule
累积三个行动：

- **Schedule 开始**。这说明了两个动作。
- **Workflow 开始**。这是启动目标 Workflow 的单个操作。它包括初始 Search Attribute 作为
  启动请求的一部分。

＃＃ 出口

[Workflow 历史导出](/cloud/export) 使您能够将关闭的 Workflow 历史记录导出到您的云存储接收器
选择。

- **Workflow 已导出**。导出的每个 Workflow 都会累积一个操作。
- 不包括在 APS 计算中。

<a id="schedules"></a>
## Temporal Nexus {#schedules}

- 对于 [Nexus 操作 Scheduled](/references/events#nexusoperationscheduled)，调用者 Workflow 启动 Nexus
  操作结果是调用者 Namespace 上的一个操作。
- 对于 [Nexus 操作已取消](/references/events#nexusoperationcanceled)，呼叫者 Workflow 取消 Nexus
  操作结果是调用者 Namespace 上的一个操作。
- 由 Nexus 操作处理程序创建的基础 Temporal 原语（例如 Workflow、活动和 Signal）
  （直接或间接）导致这些原语的正常操作记入处理程序的 Namespace。这个
  包括对底层 Temporal 原语（如“活动”）的重试，但_不_用于处理 Nexus 操作本身
  或重试 Nexus 操作本身。

<a id="export"></a>
## 公平性 {#export}

对于 Namespace 启用 [公平性](/develop/task-queue-priority-fairness#task-queue-fairness) 功能的每一小时，将针对 Namespace 中的每个操作收取额外的 `0.1` 操作费用。
详细计费示例请参见 [公平定价](/cloud/pricing#fairness-pricing)。

＃＃ 容量

- 对于 Namespace 容量 Temporal 资源单位 (TRU)，将生成最多包含的每小时分配的操作
  在设置了 TRU 并且实际使用量低于所包含的每小时操作分配的任何小时内的 TRU。
- 不包括在 APS 计算中。

＃＃ 用法

操作使用情况在 [使用仪表板](https://cloud.temporal.io/usage) 中的帐户中进行跟踪，并且对以下人员可见
帐户所有者、财务管理员和全局管理员。对于单个Namespace，用法可以参见
[Namespace 总结](https://cloud.temporal.io/namespaces/) 用于特定的 Namespace。

<a id="temporal-nexus"></a>
## Workflow 中的操作 {#temporal-nexus}

查看 Workflow 历史记录时，代表可计费操作的事件会用该操作所消耗的数量进行注释
**可计费操作**列中的事件。这些操作总结在 Workflow 的顶部。

![Temporal Cloud 显示聚合计费操作的使用情况仪表板](/img/cloud/billing/aggregate-billable-actions.png)

![Temporal Cloud 使用情况仪表板显示与事件关联的各个可计费操作](/img/cloud/billing/individual-billable-actions.png)

此计费操作估算对于预测 Workflow 的成本非常有用。例如，如果您运行测试 Workflow
生成了 20 个可计费操作并预计一个月内每天运行 100 次，您可以预期
Workflow 生成 20 个操作 x 100 次运行/天 x 30 天 = 每月 60,000 个计费操作。您还可以使用
可计费操作估算可通过更好地了解您的成本驱动因素来帮助优化 Workflow。

:::tip Excluded Billable Actions

可计费操作估算是一项实验性功能，仅测量 Workflow 中存在的可计费操作
事件历史记录。一些计费概念不包含在这些计算中，例如：

- Query
- Activity 心跳
- 拒绝 Update Workflow Execution
- 出口
- Schedule
- 发生在一个重复的动作
  [Namespace复制](../../cloud/high-availability/index.mdx#high-availability-features)

此外，带有 `TemporalNamespaceDivision` Search Attribute 套装的 Workflow 可能没有准确的计费操作
估计。估计的可计费操作数仅应被视为估计值。如果计费事件存在于
Event History，操作计数可能会更高。

:::
