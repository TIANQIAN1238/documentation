**Temporal Cloud 提供什么样的安全性？**

[Temporal Cloud](/cloud) 的安全模型涵盖应用程序、数据和 Temporal Cloud 平台本身。

:::info General platform security

有关 Temporal 一般安全功能的信息，请参阅我们的 [平台安全页面](/security)。

:::

## 应用程序和数据 {#applications-and-data}

**Temporal Cloud 中应用程序和数据的安全模型是什么？**

<a id="code-execution-boundaries"></a>
### 代码执行边界 {#code-execution-boundaries}

Temporal Cloud 将 Temporal Server 的功能提供为托管服务；它不管理您的应用程序或 [Workers](/workers#worker)。
使用 [Temporal SDK](/encyclopedia/temporal-sdks) 编写的应用程序和服务在您的计算环境中运行，例如容器（Docker、Kubernetes）或虚拟机（在任何托管环境中）。
您可以完全控制如何保护您的应用程序和服务。

<a id="data-converter-client-side-encryption"></a>
### Data Converter：客户端加密 {#data-converter-client-side-encryption}

Temporal 平台的可选 [数据转换](/dataconversion) 功能可让您在将数据发送到 Temporal Cloud 之前透明地对其进行加密，并在数据传出时对其进行解密。

数据转换在 Temporal、Worker 和 [客户](/encyclopedia/temporal-sdks#temporal-client) 上运行； Temporal Cloud 无法查看或解密您的数据。
如果您使用此功能，即使服务本身受到损害，Temporal Cloud 中存储的数据仍保持加密状态。

通过部署 [Codec Server](/production-deployment/data-encryption)，您可以安全地解密 [Temporal 网页用户界面](/web-ui) 中的数据，而无需与 Temporal 共享加密密钥。

## 平台 {#the-platform}

**Temporal Cloud 平台的安全模型是什么？**

<a id="namespace-isolation"></a>
### Namespace 隔离 {#namespace-isolation}

Temporal 环境中的基本隔离单元是 [Namespace](/namespaces)。
每个 Temporal Cloud 帐户可以有多个 Namespace，并且每个 Namespace 都是隔离的，以确保您的工作负载保持安全和高性能。

＃＃＃＃ 验证

每个 Namespace 均通过您选择的身份验证方法进行保护：
- **mTLS 证书** - 用于相互 TLS 身份验证的 Namespace 特定 X.509 证书
- **API 密钥** - 用于身份验证的 Namespace 范围 API 密钥

有关为 Namespace 配置身份验证的更多详细信息，请参阅 [API 密钥](/cloud/api-keys) 和 [mTLS 证书](/cloud/certificates)。

<a id="authentication"></a>
#### 速率限制 {#authentication}

Temporal Cloud 通过单独的速率限制保护每个 Namespace，以防止嘈杂的邻居问题：
- **每秒操作数 (APS)** - 限制 [行动](/best-practices/managing-aps-limits) 在 Workflow 中执行的速率
- **每秒操作数 (OPS)** - 限制在 Temporal Server 上创建负载的所有 [运营](/references/operation-list) 的速率

这些每个 Namespace 的速率限制可确保遇到流量峰值的一个 Namespace 不会影响其他 Namespace 的性能或可靠性，无论这些 Namespace 属于单个 Temporal Cloud 帐户还是单独的帐户。

有关 Temporal Cloud 限制的更多信息，请参阅 [速率限制](/cloud/limits)，有关监控最佳实践的 [监控趋势与限制](/cloud/service-health#rps-aps-rate-limits) 的更多信息。

<a id="rate-limiting"></a>
#### Namespace 间通信 {#rate-limiting}

默认情况下，Namespace 是隔离的。一个 Namespace 中的 Workflow 与另一个 Namespace 中的 Workflow 交互的唯一方法是通过 [Temporal Nexus](/nexus)，它通过 Nexus 端点提供受控、安全的跨 Namespace 通信。

有关 Nexus 如何实现 Namespace 间安全通信的详细信息，请参阅 [Nexus 安全](/nexus/security)。

<a id="inter-namespace-communication"></a>
#### 逻辑隔离 {#inter-namespace-communication}

Temporal Cloud 是一项多租户服务。
同一环境中的Namespace在逻辑上是隔离的。
Namespace 不跨区域边界共享数据处理或数据存储。

<a id="logical-segregation"></a>
### 私人连接 {#logical-segregation}

Temporal Cloud 支持专用连接，使您能够从安全网络连接到 Temporal Cloud。

请参阅 [连接性](/cloud/connectivity) 页面，了解有关将 AWS PrivateLink 和 GCP Private Service Connect 与 Temporal Cloud 结合使用的更多信息和详细信息。

<a id="private-connectivity"></a>
### Temporal Nexus {#private-connectivity}

与 Namespace 类似，Nexus 端点是帐户范围的资源，在 Temporal Cloud 帐户内是全局的。
帐户中的任何开发人员角色（或更高级别），同时也是端点目标 Namespace 上的 Namespace 管理员，都可以管理（创建/Update/删除）Nexus 端点。
帐户中具有只读角色（或更高角色）的所有用户都可以查看和浏览端点的完整列表。

从调用者 Namespace 中的 Workflow 到 Nexus 端点的运行时访问由 Nexus API 注册表中每个端点的（调用者 Namespace 的）白名单策略控制。
Worker 使用 Temporal Cloud 进行身份验证，就像现在使用 mTLS 证书或 API 密钥（如 Namespace 配置允许的）一样。
Nexus 请求通过安全的多区域 mTLS Envoy 网格从调用者的 Namespace 发送到处理程序的 Namespace。

对于有效负载加密，DataConverter 对 Nexus 操作的工作方式与对 Worker 和 Temporal Cloud 之间发送的其他有效负载的工作方式相同。

请参阅 [Nexus 安全](/nexus/security) 了解更多信息。

<a id="temporal-nexus"></a>
### 加密 {#temporal-nexus}

:::tip TLS vs mTLS

**TLS**（传输层安全）对传输中的数据进行加密。 **mTLS**（双向 TLS）是一种身份验证方法，客户端和服务器均提供证书来验证身份。所有 Temporal Cloud 连接都使用 TLS 加密。当您选择“mTLS 身份验证”时，您选择的是如何证明您的身份，而不是您的连接是否加密。

:::

**传输中**：与 Temporal Cloud 的所有连接都使用 TLS 1.3 加密，无论您的身份验证方法如何（[API 密钥](/cloud/api-keys) 或 [mTLS 证书](/cloud/certificates)）。

**静态**：数据存储在两个位置：Elasticsearch 实例（在 SDK 客户端、[命令行界面](/cloud/tcld) 或 Web UI 中过滤 Workflow 时使用）和核心 Temporal Cloud 持久层。
两者均使用 AES-256-GCM 进行静态加密。

＃＃＃ 身份

对 Temporal Cloud gRPC 端点的身份验证支持两种方法：

- **[API 密钥](/cloud/api-keys)**：使用不记名令牌进行基于身份的身份验证。推荐用于大多数用例。
- **[mTLS 证书](/cloud/certificates)**：使用 CA 颁发的客户端证书进行相互 TLS 身份验证。

两种方法都提供与 Temporal Cloud 的安全、加密连接。根据您组织的安全要求和密钥管理首选项进行选择。

有关 Temporal Cloud UI 的用户身份验证，请参阅 [如何使用 Temporal Cloud 管理 SAML 身份验证](/cloud/saml)。

＃＃＃ 使用权

授权在帐户和 Namespace 级别进行管理。
用户和系统被分配一个或多个预配置的角色。
用户持有管理员、开发者、只读用户的[账户级角色](/cloud/manage-access/roles-and-permissions#account-level-roles)。
系统和应用程序进程拥有各自不同的角色。

<a id="encryption"></a>
### 监控 {#encryption}

除了针对操作和可用性要求进行广泛的系统监控之外，我们还收集和监控来自 AWS 环境的审核日志以及对 gRPC API（由 SDK、CLI 和 Web UI 使用）的所有调用。
这些审核日志可用于提取到您的安全监控系统中。

<a id="identity"></a>
### 测试 {#identity}

我们与第三方签订合同，每年进行一次全面的渗透测试（社会工程除外）。
此外，我们还会根据需要执行有针对性的第三方和内部测试，例如在发布重要功能时。

<a id="access"></a>
### 内部 Temporal 访问 {#access}

我们将生产系统的访问权限限制为维护我们生产基础设施的一小部分员工。
我们记录对生产系统的所有访问；不允许共享帐户。
通过 SSO 访问所有生产系统，并启用 MFA。

仅在有限的时间内授予对我们的云环境的访问权限，最多 8 小时。
（有关更多信息，请参阅博客文章 [推出 Temporal 的访问时间](https://temporal.io/blog/rolling-out-access-hours-at-temporal)。）

所有 Temporal 工程系统均由 GitHub 凭据保护，这需要 Temporal GitHub 组织的成员资格和 MFA。
访问补助金每季度审查一次。

＃＃＃ 遵守

Temporal Technologies 经过 SOC 2 Type 2 认证，符合 GDPR 和 HIPAA 法规。
Compliance audits are available by request through our [接触](https://pages.temporal.io/contact-us) page.
