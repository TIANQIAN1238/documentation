Temporal Cloud 的操作范围包括吞吐量、延迟和限制。
服务区域列在 [本页](/cloud/regions) 上。
如果您需要更多详细信息，[联系我们](https://pages.temporal.io/contact-us)。

## 吞吐量预期 {#throughput}

**使用 Temporal Cloud 可以获得什么样的吞吐量？**

Temporal 中的每个 Namespace 都有一个速率限制，以每秒 [行动](/cloud/pricing#action) 为单位进行测量。 
Temporal 提供两种不同的容量调整模式：按需容量或预置容量。
通过按需容量，Namespace 容量会随着使用情况自动增加。
借助预配置容量，您可以通过请求 Temporal 资源单元 (TRU) 来控制容量限制。

:::tip Support, stability, and dependency info

预配置容量当前为 [公开预览](/evaluate/development-production-features/release-stages#public-preview)。

:::

## 延迟服务级别目标 (SLO) {#latency}

**Temporal Cloud 会有什么样的延迟？**

Temporal Cloud 每个区域的 p99 延迟 SLO 为 200 毫秒。

正常 Worker 请求（命令和轮询）的相同 SLO 适用于调用方和处理程序 Namespace 中的 Nexus。

<a id="historical-latency-data"></a>
### 历史延迟数据 {#historical-latency-data}

启动 Signaling Workflow Execution 的一周延迟如下：

<a id="january-2026"></a>
#### 2026 年 1 月 {#january-2026}

| 手术                          |  p50   | p90  |  p99 |
| :--------------------------------- | :----: | :--: | ---: |
| `StartWorkflowExecution`           | 14毫秒 | 21毫秒 | 69毫秒 |
| `SignalWorkflowExecution`          | 11毫秒 | 19毫秒 | 46毫秒 |
| `SignalWithStartWorkflowExecution` | 19毫秒 | 37毫秒 | 95毫秒 |

<a id="march-2024"></a>
#### 2024 年 3 月 {#march-2024}

| 手术                          | p90  |  p99 |
| :--------------------------------- | :--: | ---: |
| `StartWorkflowExecution`           | 24毫秒 | 54毫秒 |
| `SignalWorkflowExecution`          | 14毫秒 | 40毫秒 |
| `SignalWithStartWorkflowExecution` | 24毫秒 | 61毫秒 |

从 Temporal CLIent 观察到的延迟受到其他系统组件（例如 Codec Server、出口代理和网络本身）的影响。
此外，同一 Workflow Execution 上的并发操作可能会导致更高的延迟。
