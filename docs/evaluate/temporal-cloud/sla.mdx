**Temporal Cloud 的服务水平协议是什么？ SLA？**

Temporal Cloud 提供两个可用性级别：[服务可用性](https://en.wikipedia.org/wiki/Reliability,_availability_and_serviceability) 和合同 [服务水平协议](https://en.wikipedia.org/wiki/Service-level_agreement) (SLA)。
这些级别由您的部署模式设置：

- **Temporal Cloud 具有标准单区域部署**：
  标准 Temporal Cloud 部署提供 99.99% 的可用性和 99.9% 防止服务错误的合同服务级别协议 (SLA)。
- **Temporal Cloud 具有高可用性功能 Namespace 部署**：
  使用高可用性功能的 Temporal Cloud Namespace 提供 99.99% 的可用性和 99.99% 的合同服务级别协议 (SLA)，以防止服务错误。

正常 Worker 请求（命令和轮询）的相同 SLA 适用于调用方和处理程序 Namespace 中的 Nexus。

为了计算服务错误率，Temporal Cloud 捕获五分钟间隔内到达 Namespace 的所有请求。
我们记录发生的 gRPC 服务错误的数量。
对于每个 Namespace，我们将服务错误率计算为 1 -（错误计数/请求计数）。
费率按月计算平均值，并每季度重置。

根据 SLA 记录的错误是服务错误，例如 `UNAVAILABLE` [gRPC 状态码](https://grpc.github.io/grpc/core/md_doc_statuscodes.html)。
以下错误不计入 SLA：

- `ClientVersionNotSupported`
- `InvalidArgument`
- `NamespaceAlreadyExists`
- `NamespaceInvalidState`
- `NamespaceNotActive`
- `NamespaceNotFound`
- `NotFound`
- `PermissionDenied`
- `QueryFailed`
- `RetryReplication`
- `StickyWorkerUnavailable`
- `TaskAlreadyStarted`
- `Throttling (resources exhausted; triggers retry)`
- `WorkflowExecutionAlreadyStarted`
- `WorkflowNotReady`

我们的内部警报系统基于针对所有错误的 [服务水平目标](https://en.wikipedia.org/wiki/Service-level_objective) (SLO)，而不仅仅是针对 SLA 的错误。
当我们收到未满足 SLO 的警报时，我们会呼叫待命工程师，这通常意味着问题在问题变得明显之前就已得到解决。

在内部，我们的组件分布在每个区域至少三个可用区。
我们实现了单元架构。
每个单元都包含托管 Namespace 所需的软件和服务。
在每个单元内，组件分布在每个区域至少三个可用区中。

有关当前系统状态和有关最近事件的信息，请参阅 [Temporal 状态](https://status.temporal.io)。
