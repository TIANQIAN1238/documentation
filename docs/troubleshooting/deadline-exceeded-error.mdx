客户或 Worker 向 [Temporal 服务](/temporal-service) 发出的所有请求均为 [gRPC 请求](https://grpc.io/docs/what-is-grpc/core-concepts/#deadlines)。
有时，当这些前端请求无法完成时，您会看到以下特定错误消息：`Context: deadline exceeded`。
网络中断、超时、服务器过载和 Query 错误是导致此错误的部分原因。

以下各节讨论此错误的性质以及如何解决该错误。

<a id="check-system-clocks"></a>
### 检查系统时钟 {#check-system-clocks}

时序偏差可能会导致 Worker 上的系统时钟漂移到 Temporal 服务的系统时钟之后。
如果两个时钟之间的差异超过 Activity 的启动到关闭超时，则会发生 `Activity complete after timeout` 错误。

如果您收到 `Activity complete after timeout` 错误和 `Context: deadline exceeded`，请检查 Temporal 服务系统和发送该错误的 Worker 系统上的时钟。
如果 Worker 的时钟与 Temporal 服务不匹配，请将所有时钟同步到 NTP 服务器。

<a id="check-frontend-service-logs"></a>
### 检查前端服务日志 {#check-frontend-service-logs}

:::note

云用户无法访问诊断错误来源所需的某些日志。

如果您使用的是 Temporal Cloud，请创建一个包含尽可能多信息的 [支持票](/cloud/support#support-ticket)，包括发生问题的一些 Workflow Execution 的 Namespace 名称和 Workflow Id。

:::

[前端服务](/temporal-service/temporal-server#frontend-service) 日志可以显示 Temporal 服务的哪些部分不工作。
要出现该错误，服务容器或容器必须启动并运行。

OSS用户可以使用Temporal CLI验证前端服务是否已连接并正在运行。

```
temporal operator cluster health --address 127.0.0.1:7233
```

使用 [`gRPC-health-probe`](https://github.com/grpc-ecosystem/grpc-health-probe) 检查前端服务、[配对服务](/temporal-service/temporal-server#matching-service) 和 [历史服务](/temporal-service/temporal-server#history-service)。

```
./grpc-health-probe -addr=frontendAddress:frontendPort -service=temporal.api.workflowservice.v1.WorkflowService

./grpc-health-probe -addr=matchingAddress:matchingPort -service=temporal.api.workflowservice.v1.MatchingService

./grpc-health-probe -addr=historyAddress:historyPort -service=temporal.api.workflowservice.v1.HistoryService
```

日志还可用于查找失败的客户端 [Query](/sending-messages#sending-queries) 请求。

<a id="check-your-temporal-service-metrics"></a>
### 检查您的 Temporal 服务指标 {#check-your-temporal-service-metrics}

Temporal 服务指标可用于检测影响 Temporal 服务运行状况的问题（例如 `resource exhausted`）。
`resource exhausted` 错误可能会导致您的客户端请求失败，从而提示 `deadline exceeded` 错误。

使用以下 Query 检查指标仪表板上的 `RpsLimit`、`ConcurrentLimit` 和 `SystemOverloaded` 中是否有错误。

```
sum(rate(service_errors_resource_exhausted{}[1m])) by (resource_exhausted_cause)
```

查找高延迟、短超时和其他异常 [Temporal 服务指标](/references/cluster-metrics)。
如果指标来自特定服务（例如历史记录服务），请检查该服务的运行状况和性能。

<a id="check-workflow-logic"></a>
### 检查 Workflow 逻辑 {#check-workflow-logic}

检查您的 [客户端和 Worker 配置](/references/configuration) 文件是否缺少或无效的目标值，如下所示：

- 服务器名称
- 网络或主机地址
- 证书

无效目标还会导致 `connection refused` 错误和 `deadline exceeded` 错误。
更新文件后检查客户端是否连接。

<a id="advanced-troubleshooting"></a>
### 高级故障排除 {#advanced-troubleshooting}

除了前面部分中列出的步骤之外，还请检查以下每个场景中提到的区域。

<a id="after-enabling-mtls"></a>
### 启用 mTLS 后 {#after-enabling-mtls}

使用 `Temporal operator cluster health` 检查 Temporal 服务的运行状况。

```
temporal operator cluster health --address [SERVER_ADDRESS]
```

将任何缺失的 [环境变量](/references/web-ui-environment-variables) 添加到配置文件中，并更正任何不正确的值。
前端和节点间的服务器名称和证书必须匹配。

<a id="after-restarting-the-temporal-service"></a>
### 重新启动 Temporal 服务后 {#after-restarting-the-temporal-service}

您可能没有给 Temporal 服务足够的时间来响应和重新连接。
重新启动服务器，等待，然后检查所有服务的连接情况和进一步的错误。

如果错误仍然存​​在，请在继续排除故障之前查看您的 Workflow Execution 历史记录和服务器日志以了解更具体的原因。

<a id="when-executing-or-scheduling-workflows"></a>
### 执行或调度 Workflow 时 {#when-executing-or-scheduling-workflows}

一项或多项服务可能无法连接到 [前端服务](/temporal-service/temporal-server#frontend-service)。
Workflow 可能无法在给定的连接时间内完成请求。

增加 `frontend.keepAliveMaxConnectionAge` 的值，以便可以在连接终止之前完成请求。

:::note

如果增加 `frontend.keepAliveMaxConnectionAge` 值，请考虑监视服务器性能的负载。

:::
