<a id="task-routing"></a>
## 什么是任务路由？ {#task-routing}

任务路由就是 Task Queue 与一个或多个 Worker 配对时的情况，主要用于 Activity 任务执行。

这也可能意味着采用多个 Task Queue，每个 Worker 进程都配对。

任务路由有许多适用的用例。

某些 SDK 提供 [会话API](#worker-session)，它提供了一种简单的方法来确保使用相同的 Worker 执行 Activity 任务，而无需您手动指定 Task Queue 名称。
它还包括并发会话限制和 Worker 故障检测等功能。

<a id="flow-control"></a>
### 流量控制 {#flow-control}

从 Task Queue 消耗的 Worker 仅在具有可用容量时才请求 Activity 任务，因此它永远不会因请求峰值而过载。
如果 Activity 任务的创建速度快于 Worker 的处理速度，则它们会积压在 Task Queue 中。

<a id="throttling"></a>
### 节流 {#throttling}

每个 Activity Worker 轮询和处理 Activity 任务的速率可根据 Worker 进行配置。
即使 Worker 有闲置容量，也不会超过此速率。
还支持全局 Task Queue 速率限制。
此限制适用于给定 Task Queue 的所有 Worker。
它经常用于限制 Activity 调用的下游服务的负载。

<a id="specific-environments"></a>
### 特定环境 {#specific-environments}

在某些情况下，您可能需要在专用环境中执行活动。
要将 Activity 任务发送到此环境，请使用专用的 Task Queue。

<a id="route-activity-tasks-to-a-specific-host"></a>
#### 将 Activity 任务路由到特定主机 {#route-activity-tasks-to-a-specific-host}

在某些用例中，例如文件处理或机器学习模型训练，Activity 任务必须路由到特定的 Worker 进程或 Worker 实体。

例如，假设您有一个 Workflow，其中包含以下三个单独的活动：

- 下载文件。
- 以某种方式处理文件。
- 将文件上传到另一个位置。

下载文件的第一个 Activity 可能发生在任何主机上的任何 Worker 上。
但是，第二个和第三个活动必须由第一个 Activity 下载文件的同一主机上的 Worker 执行。

在现实场景中，您可能有许多 Worker 进程在许多主机上扩展。
您需要开发 Temporal 应用程序，以便在需要时将任务路由到特定的 Worker 流程。

代码示例：

- [Go文件处理示例](https://github.com/temporalio/samples-go/tree/main/fileprocessing)
- [Java文件处理示例](https://github.com/temporalio/samples-java/tree/main/core/src/main/java/io/temporal/samples/fileprocessing)
- [PHP文件处理示例](https://github.com/temporalio/samples-php/tree/master/app/src/FileProcessing)

<a id="route-activity-tasks-to-a-specific-process"></a>
#### 将 Activity 任务路由到特定进程 {#route-activity-tasks-to-a-specific-process}

某些活动会加载大型数据集并在此过程中缓存它们。
依赖这些数据集的活动应路由到同一进程。

在这种情况下，涉及的每个 Worker 进程都会存在唯一的 Task Queue。

<a id="workers-with-different-capabilities"></a>
#### Worker 具有不同的功能 {#workers-with-different-capabilities}

与非 GPU 盒子相比，某些 Worker 可能存在于 GPU 盒子上。
在这种情况下，每种类型的盒子都有自己的 Task Queue，而 Workflow 可以选择一个来发送 Activity 任务。

<a id="multiple-priorities"></a>
### 多个优先级 {#multiple-priorities}

如果您的用例涉及多个优先级，您可以为每个优先级创建一个 Task Queue，并为每个优先级创建一个 Worker 池。

<a id="versioning"></a>
### 版本控制 {#versioning}

任务路由是对代码进行版本控制的最简单方法。

如果您有新的向后不兼容的 Activity Definition，请首先使用不同的 Task Queue。

<a id="worker-session"></a>
## 什么是 Worker 会话？ {#worker-session}

Worker 会话是某些 SDK 提供的一项功能，它为 [任务路由](#task-routing) 提供简单的 API，以确保使用相同的 Worker 执行 Activity 任务，而无需您手动指定 Task Queue 名称。
它还包括并发会话限制和 Worker 故障检测等功能。

- [如何使用 Worker 会话](/develop/go/workers/sessions)
