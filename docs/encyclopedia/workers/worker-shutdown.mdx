<a id="worker-shutdown-behavior"></a>
# Worker 关闭行为 {#worker-shutdown-behavior}

当 Worker 关闭时，它会停止轮询新任务并开始关闭序列。
对于正在进行的 Workflow 任务，如果在尝试完 Retry Policy 尝试后未及时完成，关闭可能会导致它们失败。

可能会发生两种类型的关闭行为，具体取决于是否配置了“正常关闭”的概念。

<a id="graceful-shutdown"></a>
## 优雅关闭 {#graceful-shutdown}

正常关闭配置 Worker 在关闭之前必须完成当前任务的时间。
Activity 能够通过 Activity 上下文确定其运行的 Worker 正在关闭。

> 核心 SDK - `graceful_shutdown_period`

> 去 - `WorkerStopTimeout`

> Java - `shutdown()` 后跟 `awaitTermination(timeout, unit)`

<a id="workflow-tasks"></a>
### Workflow 任务 {#workflow-tasks}

任何飞行中的 Workflow 任务均已（尝试）完成。
它们可能不会立即发生的唯一原因是 Workflow 代码是否（错误地）阻塞，或者由于本地活动（见下文）。

＃＃＃ 活动

允许在正常关闭期间完成活动。

<a id="activities"></a>
### 当地活动 {#activities}

由于本地活动在 Workflow 任务内运行，因此假设没有其他命令需要屈服，则将允许同一 Workflow 任务内的当前和未来本地活动运行并完成。

如果本地 Activity 无法在正常关闭期限内完成，则会向本地 Activity 尝试发送取消 Signal。
在这种情况下，不会重试或启动任何新的本地活动，并且 Worker 将关闭。
Worker 仍然等待当前的 Workflow 任务完成，这意味着您最终可以达到 Workflow 任务或执行超时，除非另一个 Worker 启动。

<a id="local-activities"></a>
## 非优雅期关闭 {#local-activities}

此行为适用于未指定宽限期，或者关闭时间超过配置的宽限期的情况。
在所有情况下，当当前 Workflow 任务完成（成功或失败）时，Activity 上下文都会被取消，并且 Worker 将完成关闭。

:::note
当我们超过任务超时且 Activity 或本地 Activity 仍在运行时，Go 和 Core SDK 的行为有所不同：

**开始** - 关闭完成，但 Activity 将继续运行并使用插槽。

**核心** - 当 Activity 完成时，Worker 关闭不会完成。
:::

<a id="non-graceful-period-shutdown"></a>
### 当地活动 {#non-graceful-period-shutdown}

向本地 Activity 发送取消 Signal，然后 Workflow 任务心跳停止，并且不会重试或启动任何新的本地活动。
Worker 仍然等待当前的 Workflow 任务完成，这意味着您最终可以达到 Workflow 任务或执行超时，除非另一个 Worker 启动。

<a id="local-activities"></a>
## 一般开发者指南 {#local-activities}

- 确保活动和本地活动**荣誉上下文取消**或其他关闭 Signal。
- 预计**长时间或挂起的本地活动可能会阻止关闭**，除非您提前失败。
  建议本地活动通常已用于短期活动。
