## 什么是 Worker？{#worker}

在日常对话中，Worker 一词用于指代 [Worker Program](#worker-program)、[Worker Process](#worker-process) 或 [Worker Entity](/workers#worker-entity)。
Temporal 文档旨在明确区分这三者。

## 什么是 Worker Program？{#worker-program}

Worker Program 是定义 Worker Process 约束的静态代码，使用 Temporal SDK 的 API 开发。

## 什么是 Worker Entity？{#worker-entity}

Worker Entity 是 Worker Process 中监听特定 Task Queue 的单个 Worker。

Worker Entity 监听并轮询单个 Task Queue。
Worker Entity 包含 Workflow Worker 和/或 Activity Worker，分别推进 Workflow Execution 和 Activity Execution 的进展。

**Worker 能处理比其缓存大小或支持线程数更多的 Workflow Execution 吗？**

可以。
但代价是增加了延迟。

Worker 是无状态的，因此任何处于阻塞状态的 Workflow Execution 都可以从 Worker 中安全地移除。
之后，当需要时（以外部事件的形式），它可以在相同或不同的 Worker 上恢复。
因此，单个 Worker 可以处理数百万个开放的 Workflow Execution，前提是它可以处理更新速率，且稍高的延迟不是问题。

## 什么是 Worker Identity？{#worker-identity}

Worker 有一个关联的标识符，用于帮助识别特定的 Worker 实例。
默认情况下，Temporal SDK 将 Worker Identity 设置为 `${process.pid}@${os.hostname()}`，它结合了 Worker 的进程 ID（`process.pid`）和运行 Worker 的机器的主机名（`os.hostname()`）。

Worker Identity 在各种上下文中可见，例如 Workflow 历史记录和 Task Queue 上的轮询器列表。

您可以使用 Worker Identity 来帮助调试操作问题。
通过提供用户分配的标识符，您可以将问题追溯到特定的 Worker 实例。

**默认 Identity 有哪些限制？**

虽然默认的 Identity 格式看似合理，但在云环境中通常用处有限。
一些常见问题包括：

- **Docker 容器**：在 Docker 容器内运行 Worker 时，进程 ID 始终是 `1`，因为每个容器通常运行单个进程。这使得进程标识符对于识别目的毫无意义。
- **随机主机名**：在某些云环境中，例如 Amazon ECS（弹性容器服务），主机名是随机生成的字符串，不提供有关 Worker 执行上下文的任何有意义信息。
- **临时 IP 地址**：在某些情况下，主机名可能被设置为临时 IP 地址，该地址可能随时间变化且不能唯一标识 Worker 实例。

**有哪些推荐方法？**

建议您确保 Worker Identity 可以链接回相应的机器、进程、执行上下文或日志流。
在某些执行环境中，这可能需要您明确指定 Worker Identity。

以下是一些方法：

- **使用特定于环境的标识符**：选择特定于您的执行环境的标识符。例如，在 Amazon ECS 上运行 Worker 时，您可以将 Worker Identity 设置为 ECS Task ID，它唯一标识运行 Worker 的任务。
- **包含相关上下文**：纳入有助于建立 Worker 上下文的信息，例如部署环境（`staging` 或 `production`）、区域或其他相关详情。
- **确保唯一性**：确保 Worker Identity 在您的系统中是唯一的，以避免在调试问题时出现歧义。
- **保持简洁**：虽然包含相关信息很重要，但请尽量保持 Worker Identity 简洁且易于阅读，以便快速识别和故障排除。

## 什么是 Worker Process？{#worker-process}

<CaptionedImage
    src="/diagrams/worker-and-server-component.svg"
    title="Worker Process 和 Temporal Server 的组件图"
    />

Worker Process 负责轮询 [Task Queue](/task-queue)、出队 [Task](/tasks#task)、执行您的代码以响应 Task，并将结果返回给 [Temporal Service](/temporal-service)。

更正式地说，Worker Process 是任何实现了 Task Queue 协议和 Task Execution 协议的进程。

- 如果进程实现了 Workflow Task Queue 协议并执行 Workflow Task Execution 协议以推进 Workflow Execution 的进展，则该进程是 Workflow Worker Process。
  Workflow Worker Process 可以监听任意数量的 Workflow Task Queue，并可以执行任意数量的 Workflow Task。
- 如果进程实现了 Activity Task Queue 协议并执行 Activity Task Processing 协议以推进 Activity Execution 的进展，则该进程是 Activity Worker Process。
  Activity Worker Process 可以监听任意数量的 Activity Task Queue，并可以执行任意数量的 Activity Task。

**Worker Process 在 Temporal Service 外部运行。**
Temporal 应用程序开发者负责开发 [Worker Program](#worker-program) 和运营 Worker Process。
换句话说，[Temporal Service](/temporal-service)（包括 Temporal Cloud）不会在 Temporal Service 机器上执行您的任何代码（Workflow 和 Activity 定义）。Temporal Service 仅负责编排[状态转换](/workflow-execution#state-transition)并将 Task 提供给下一个可用的 [Worker Entity](/workers#worker-entity)。

虽然 Event 历史记录中传输的数据[通过 mTLS 进行了安全保护](/self-hosted-guide/security#encryption-in-transit-with-mtls)，但默认情况下，它在 Temporal Service 中静止时仍然是可读的。

为了解决这个问题，Temporal SDK 提供了一个 [Data Converter API](/dataconversion)，您可以使用它来自定义进出 Worker Entity 的数据序列化，其净效果是保证 Temporal Service 无法读取敏感业务数据。

在我们的许多教程中，我们展示了如何在本地开发时在同一台机器上同时运行 Temporal Service 和一个 Worker。
但是，生产级 Temporal 应用程序通常拥有一个_集群_的 Worker Process，所有这些进程都在 Temporal Service 外部的主机上运行。
Temporal 应用程序可以根据需要拥有任意数量的 Worker Process。

Worker Process 既可以是 Workflow Worker Process，也可以是 Activity Worker Process。
许多 SDK 支持在单个 Worker Process 中拥有多个 Worker Entity 的能力。
（Worker Entity 的创建和管理在不同 SDK 之间有所不同。）
单个 Worker Entity 只能监听单个 Task Queue。
但如果一个 Worker Process 有多个 Worker Entity，则该 Worker Process 可以监听多个 Task Queue。

<CaptionedImage
    src="/diagrams/worker-and-server-entity-relationship.svg"
    title="Worker Process、Task Queue 和 Task 的实体关系图（元模型）"
/>

执行 Activity Task 的 Worker Process 必须能够访问执行 Activity 定义中定义的操作所需的任何资源，例如：

- 用于外部 API 调用的网络访问。
- 用于基础设施配置的凭证。
- 用于机器学习工具的专用 GPU。

Temporal Service 本身有用于系统 Workflow Execution 的[内部 Worker](https://temporal.io/blog/workflow-engine-principles/#system-workflows-1910)。
但是，这些内部 Worker 对开发者不可见。
