## Worker 部署 {#deployments}

Worker 部署是将相似的 Worker 分组在一起进行统一管理的逻辑服务。
每个部署都有一个名称（例如您的服务名称），并支持通过一系列 Worker 部署版本进行版本控制。

## Worker 部署版本 {#deployment-versions}

Worker 部署版本表示 Worker 部署的迭代。
每个部署版本都由共享相同代码构建和环境的 Worker 组成。
当 Worker 开始轮询 Workflow 和 Activity 任务时，它会将其部署版本报告给 Temporal Server。

## 版本控制行为 {#versioning-behaviors}

您可以使用 SDK 或 CLI 在 Workflow 配置中声明每个 Workflow 类型具有**版本控制行为**（固定或自动升级）。

要了解有关实施 Worker 版本控制的更多信息，请参阅我们的 [Worker 生产中的版本控制](production-deployment/worker-deployments/worker-versioning) 页面。

### 固定 Workflows {#pinned}

**固定** Workflow 保证在单个 Worker 部署版本上完成。您可以在注册 Workflow 类型时通过添加附加固定参数将其标记为固定。如果您需要将固定的 Workflow 移至新版本，请使用 [`Temporal Workflow Update-options`](/cli/workflow#update-options)。

### 自动升级 Workflows {#auto-upgrade}

每当您更改当前版本时，**自动升级** Workflow 将自动移动到最新的 Worker 部署版本。自动升级 Workflow 不限于单个部署版本，需要手动保持重放安全，即使用 [修补](/workflow-definition#workflow-versioning)。

<a id="activity-behavior-across-versions"></a>
### Activity 跨版本行为 {#activity-behavior-across-versions}

当您处理 Worker 部署版本时，需要考虑一些活动方案。

- 活动通常从其 Workflow 的 Worker 部署版本开始，这意味着：
  - 对于固定的 Workflow，Activity 从固定版本开始。
  - 对于自动升级 Workflow，Activity 在 Workflow 的目标 Worker 部署版本上启动。在这种情况下，如果目标版本与上次使用的版本不同，则 Workflow Execution 在启动 Activity 之前立即移动到其目标版本。 Workflow 的目标 Worker 部署版本是 Workflow 的 Task Queue 的当前版本或渐变版本，具体取决于渐变百分比和 Workflow ID。

有一个例外，您将有**独立活动**。独立活动特定于 Worker 版本控制。他们从自己的 Task Queue 的当前版本或斜坡版本开始，独立于 Workflow。

- 对于固定的 Workflow，独立活动是在 Task Queue 上启动的活动，该 Task Queue 不是调用 Workflow 的固定 Worker 部署版本的成员。
- 对于自动升级 Workflow，独立活动是在 Task Queue 上启动的活动，该 Task Queue 不是调用 Workflow 的目标 Worker 部署版本的成员。

由于独立活动不是 Workflow 版本的一部分，因此它们可以通过几种不同的方式运行：

- Activity Task Queue 在仅具有独立 Activity 的单独 Worker 部署中运行。
- 独立 Activity 位于未版本化的 Task Queue 中。
- 独立 Activity 位于单独的 Worker 部署中，该部署拥有自己的 Workflow，但其他 Workflow 重复使用其他 Worker 部署中的 Activity。

<a id="versioning-definitions"></a>
## 版本控制定义 {#versioning-definitions}

- **当前 Worker 部署版本**：Workflow 路由到的版本，除非它们之前固定在不同版本上。其他版本可以继续轮询以允许固定的 Workflow 完成执行或以防万一您需要回滚。如果未指定当前版本，则默认为无版本。
- **渐变 Worker 部署版本**：将可配置百分比的 Workflow 路由到的版本，除非它们之前固定在不同版本上。斜坡百分比可以在 [0, 100] 范围内。不进入斜坡版本的 Workflow 将进入当前版本。如果未指定渐变版本，则 100% 的新 Workflow 和自动升级 Workflow 将转到当前版本。
- **目标 Worker 部署版本**：您的 Workflow 将升级到下一个版本。这可能是部署的当前版本或渐变版本。例如，如果自动升级 Workflow 在版本 A 上运行，当前版本为 B，并且有 5% 的升级到 C，则其目标版本为 B 的可能性为 95%，为 C 的可能性为 5%。Workflow ID 确定 Workflow 属于 95% 组还是 5% 组。

## 版本控制状态 {#versioning-statuses}

Worker 部署版本会经历以下状态：

1. **非活动**：该版本存在，因为具有该版本的 Worker 已轮询服务器。如果此版本从未变为“Active”，则它永远不会处于“Draining”或“Drained”状态。
2. **活动**：版本为当前版本或正在升级版本，因此它接受新的 Workflow 和现有的自动升级 Workflow。
3. **耗尽**：该版本已打开固定的 Workflow 在其上运行，但不再是当前或斜坡，通常是因为已部署了较新的版本。由于排水状态仅定期为 Updated，因此可能会在短时间内处于排水状态并且没有开路固定的 Workflow。
4. **耗尽**：该版本正在耗尽，现在在其上运行的所有固定的 Workflow 都已关闭。如果关闭的 Workflow 的 [保留期限](https://docs.temporal.io/sending-messages#sending-queries) 中的 [查询](https://docs.temporal.io/temporal-service/temporal-server#retention-period) 和 Worker 仍在轮询，则它们仍可能重新运行某些代码路径。

## Continue-As-New、Child Workflow 和重试语义 {#inheritance-semantics}

当 Workflow 开始新的运行（例如，通过继续新运行或重试）时，新运行可能会继承其版本控制行为。本节介绍继承如何在不同的 Workflow Execution 模式中发挥作用。

<a id="ways-workflows-start-new-runs"></a>
### Workflow 开始新运行的方式 {#ways-workflows-start-new-runs}

Workflow 可以通过以下方式开始新的运行：

- 启动 [Child Workflow](https://docs.temporal.io/child-workflows)
- 调用 [Continue-As-New](https://docs.temporal.io/workflow-execution/continue-as-new)
- 按照 [Retry Policy](https://docs.temporal.io/encyclopedia/retry-policies) 重试
- 开始 [计划任务](https://docs.temporal.io/cron-job) 的另一次迭代（由 [Schedules](https://docs.temporal.io/schedule) 取代）

<a id="inheritance-rules-overview"></a>
### 继承规则概述 {#inheritance-rules-overview}

自动升级 Workflow 永远不会继承版本。
默认情况下，固定的 Workflow 会将其版本传递给任何固定的子项。

本节提供有关特定继承方案的更多详细信息。

<a id="inheritance-by-scenario"></a>
### 按场景继承 {#inheritance-by-scenario}

#### Child Workflows

**当家长被固定时：**

- 如果子项的 Task Queue 属于该版本，则子项继承父项的版本
- 子级的第一个 Workflow 任务与其父级执行相同的版本
- 如果子项也被固定：子项在其生命周期中保持固定到继承的版本
- 如果孩子是自动升级：第一个任务完成后，孩子的行为将更改为自动升级
- 如果子项的 Task Queue 与父项不在同一个 Worker 部署中：不会发生继承，子项从其 Task Queue 的当前版本开始

**当父级自动升级时：**

- 子项不继承初始版本控制行为
- 子项像所有新的 Workflow Execution 一样从当前版本的 Worker 部署开始

#### Continue-As-New

**当原始 Workflow 被固定时：**

- 固定版本在 Continue-As-New 链中继承
- 如果新运行的 Task Queue 与原始 Workflow 不在同一 Worker 部署中：不会发生继承，新运行将在其 Task Queue 的当前版本上启动

**原版Workflow自动升级时：**

- 不发生版本继承

<a id="retries"></a>
#### 重试 {#retries}

**继承条件（必须全部满足）：**

- 重试运行在重试时被有效固定
- 重试的运行在启动时继承了固定版本（即，它是固定父级的子级，或固定运行的 Continue-As-New）
- 重试运行在继承版本的 Task Queue 上运行

**当条件不满足时：**

- 不发生版本继承

<a id="cron-jobs"></a>
#### 计划任务 {#cron-jobs}

- **永远不要继承**版本控制行为或版本

<a id="versioning-override-inheritance"></a>
### 版本控制覆盖继承 {#versioning-override-inheritance}

- 子进程、crons、重试和 Continue-As-New 继承源运行的覆盖 **if**：
  - 覆盖已固定，**并且**
  - 新Workflow的Task Queue属于覆盖版本
- 覆盖继承是单独评估的，并且优先于继承的基本版本
