Retry Policy 是一组设置，用于告知 Temporal 在 Workflow Execution 或 Activity Task Execution 中发生失败后如何以及何时进行重试。

## 概述

Temporal 的默认行为是自动重试失败的 Activity，因此瞬时或间歇性故障无需你采取任何措施。此行为由 Retry Policy 定义。

Retry Policy 是声明式的。你不需要实现自己的重试处理逻辑；只需指定所需的行为，Temporal 就会提供相应功能。

与其包含的 Activity 不同，Workflow Execution 本身默认不关联 Retry Policy。这看起来可能有些违反直觉，但 Workflow 和 Activity 扮演不同的角色。Activity 旨在执行可能失败的操作，因此拥有默认的 Retry Policy 可以提高它们最终成功完成的可能性，即使初始尝试失败了。另一方面，Workflow 必须是确定性的，不应执行容易失败的操作。虽然可以为 Workflow Execution 指定 Retry Policy，但这不是默认设置，且通常不常见。

Retry Policy 不适用于 Workflow Task Execution，后者会以指数退避方式重试，直到 Workflow Execution Timeout（默认无限制），最大间隔为 10 分钟。Retry Policy 指导 Temporal Service 如何重试 [Workflow Execution](/workflow-execution) 或 [Activity Task Execution](/tasks#activity-task-execution) 的失败。

尝试使用 [Activity 重试模拟器](/develop/activity-retry-simulator)来直观展示 Retry Policy 的工作原理。

## 默认行为

Temporal 中的 Activity 默认关联 Retry Policy，而 Workflow 则没有。Temporal SDK 提供了一个具有默认行为的 Retry Policy 实例。虽然此对象并非专属于 Workflow 或 Activity，但你需要使用不同的方法将其应用于各自的执行。

本节详细介绍 Activity 和 Workflow 的默认重试行为，为进一步自定义提供背景信息。

### Activity Execution

Temporal 的默认行为是自动重试 Activity，每次尝试之间有短暂延迟且呈指数增长，直到 Activity 成功或被取消。当后续请求成功时，Workflow 代码将继续执行，就好像之前的失败从未发生过一样。

当 Activity Task Execution 被重试时，Temporal Service 将一个新的 [Activity Task](/tasks#activity-task) 放入其对应的 [Activity Task Queue](/task-queue)，从而产生新的 Activity Task Execution。

默认 Retry Policy 使用指数退避，退避系数为 2.0，初始间隔为 1 秒，最大间隔上限为 100 秒。默认情况下，最大重试次数设置为零（即无限制），不可重试错误默认为空。有关所有 Retry Policy 属性及其默认值的详细信息，请参阅[属性](#properties)部分。

### Workflow Execution

与 Activity 不同，Workflow Execution 默认不进行重试。当 Workflow Execution 被触发时，它不关联默认的 Retry Policy，因此默认不重试。

Temporal 为 Activity 代码的幂等性提供指导，预期 Activity 在失败后需要重新执行；这通常对 Workflow 不适用。在大多数使用场景中，Workflow 失败表明应用的设计或部署存在问题，例如可能需要不同输入数据的永久性故障。

由于 Temporal 的确定性设计，不建议重试整个 Workflow Execution。由于 Workflow 会重放相同的事件序列以达到相同的状态，重试整个 Workflow 将重复相同的逻辑而不解决导致失败的根本问题。这种重复不能解决与外部依赖或未改变条件相关的问题，还会导致不必要的资源消耗和更高的成本。相反，更有效的做法是仅重试失败的 Activity。这种方法针对特定的故障点，让 Workflow 能够在不做冗余操作的情况下继续执行，从而节省资源并确保更有针对性和有效的错误恢复过程。如果需要重试 Workflow Definition 的某些部分，建议在 Workflow 代码中实现此逻辑。

## 自定义 Retry Policy

要使用自定义 Retry Policy，在启动 Workflow Execution 或 Activity Execution 时将其作为选项参数提供。只有某些特定场景才适合使用自定义 Retry Policy 启动 Workflow Execution，例如：

- [Temporal Cron Job](/cron-job) 或其他无状态、持续运行的 Workflow Execution，可以从重试中获益。
- 将文件下载到主机的文件处理或媒体编码 Workflow Execution。

## 属性

### Retry Policy 的默认值

```
Initial Interval     = 1 second
Backoff Coefficient  = 2.0
Maximum Interval     = 100 × Initial Interval
Maximum Attempts     = ∞
Non-Retryable Errors = []
```

### 初始间隔（Initial Interval）

- **描述：** 第一次重试发生前必须经过的时间量。
  - **默认值为 1 秒。**
- **使用场景：** 用作[退避系数](#backoff-coefficient)的基础间隔时间。

### 退避系数（Backoff Coefficient）

- **描述：** 该值决定_重试间隔_增加的幅度。
  - **默认值为 2.0。**
  - 退避系数为 1.0 意味着重试间隔始终等于[初始间隔](#initial-interval)。
- **使用场景：** 使用此属性增加重试间隔。退避系数大于 1.0 时，前几次重试发生得相对较快以克服间歇性故障，而后续重试间隔越来越长以应对持续时间较长的故障。使用[最大间隔](#maximum-interval)属性防止系数将重试间隔增加过多。

### 最大间隔（Maximum Interval）

- **描述：** 指定重试之间的最大间隔。
  - **默认值为[初始间隔](#initial-interval)的 100 倍。**
- **使用场景：** 此属性对于大于 1.0 的[退避系数](#backoff-coefficient)非常有用，因为它可以防止重试间隔无限增长。

### 最大尝试次数（Maximum Attempts）

- **描述：** 指定在存在失败的情况下可以进行的最大执行尝试次数。
  - **默认为无限制。**
  - 如果超过此限制，执行将失败且不再重试。发生这种情况时，会返回一个错误。
  - 将值设置为 0 也表示无限制。
  - 将值设置为 1 表示单次执行尝试，不进行重试。
  - 将值设置为负整数会在调用执行时导致错误。
- **使用场景：** 使用此属性确保重试不会无限期继续。在大多数情况下，我们建议使用 Workflow Execution Timeout（针对 [Workflow](/workflows)）或 Schedule-To-Close Timeout（针对 Activity）来限制重试总时长，而不是使用此属性。

### 不可重试错误（Non-Retryable Errors） {#non-retryable-errors}

不可重试错误指定不应重试的错误类型。默认情况下，未指定任何错误。错误将与 [Application Failure](/references/failures#application-failure) 的 `type` 字段进行匹配。如果出现这些错误之一，则不会进行重试。如果你知道某些错误不应触发重试，可以指定这些错误，这样一旦发生，就不会重试执行。

#### Activity 的不可重试错误

在编写软件应用程序时，你会遇到三种类型的故障：瞬时、间歇性和永久性。瞬时和间歇性故障可能在重试后自行解决，而永久性故障则不会。永久性故障根据定义需要你对逻辑或输入进行更改。因此，最好是将其暴露出来而不是重试。

不可重试错误是指无论 Retry Policy 如何都不会重试的错误。

例如，检查错误的输入数据是使用不可重试错误的合理场景。如果 Activity 无法使用所提供的输入继续执行，则应立即暴露该错误，以便在下一次尝试时更正输入。

如果应用程序的维护责任分散在多个维护者之间，或者你正在开发一个要集成到别人应用程序中的库，则可以将硬编码不可重试错误的决定视为遵循"调用者与实现者"二分法。任何调用你的 Activity 的人都可以决定他们的 Retry Policy，但只有实现者才能决定某个错误是否默认不可重试。

## 重试间隔

重试前的等待时间即为_重试间隔_。重试间隔是以下两个值中较小的一个：

- [初始间隔](#initial-interval)乘以[退避系数](#backoff-coefficient)的重试次数次幂。
- [最大间隔](#maximum-interval)。

### 按错误设置下次重试延迟

有时，你的 Activity 或 Workflow 会引发一个需要与 Retry Policy 中不同重试间隔的特殊异常。为此，可以抛出一个设置了下次重试延迟字段的 [Application Failure](/references/failures#application-failure)。此值将替换并覆盖 Retry Policy 中原本的重试间隔。请注意，重试仍然受 Retry Policy 的最大尝试次数限制，以及整体超时约束。对于 Activity，适用 Schedule-to-Close Timeout；对于 Workflow，适用 Execution Timeout。

## Event History

当 Retry Policy 起作用时，事件记录到 Event History 的方式存在一些细微差别。

- 对于 Activity Execution，[ActivityTaskStarted](/references/events#activitytaskstarted) 事件在 Activity Execution 完成或失败（已耗尽所有重试次数）之前不会出现在 Workflow Execution Event History 中。这是为了避免用噪音填满 Event History。使用 Describe API 获取待处理 Activity Execution 的尝试次数。

- 当 Workflow 失败且有 Retry Policy 时，失败的运行以 `WorkflowExecutionFailed`（`retryState=IN_PROGRESS` 且设置了 `newExecutionRunId`）结束，Temporal Service 会启动新的 Workflow Execution。新的 Workflow Execution 会立即创建，但直到退避时长耗尽后才会调度第一个 Workflow Task。该时长记录在新运行的 `WorkflowExecutionStartedEventAttributes` 的 `first_workflow_task_backoff` 字段中。
