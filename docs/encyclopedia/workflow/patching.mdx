本页讨论 [打补丁](#patching)。

<a id="patching"></a>
## 什么是修补？ {#patching}

补丁在 Workflow 中为特定更改定义逻辑分支，类似于功能标志。
它将代码更改应用于新的 Workflow Execution，同时避免对正在进行的 Workflow Execution 进行破坏性更改。
当您想要进行可能影响现有 Workflow Execution 的实质性代码更改时，请创建补丁。请注意，无需修补 [固定 Workflow](/worker-versioning)。

<a id="detailed-description-of-the-patched-function"></a>
### `patched()` 功能详细说明 {#detailed-description-of-the-patched-function}

这适用于 Python、.NET 和 Ruby SDK 中的 `patched()` 函数。

<a id="behavior-when-not-replaying"></a>
#### 不重播时的行为 {#behavior-when-not-replaying}

如果执行不是重播，当它遇到对 `patched()` 的调用时，它首先检查 Event History。

- 如果补丁 ID 不在 Event History 中，则执行会向 Event History 添加标记，更新插入 Search Attribute，并返回 `true`。这发生在补丁 ID 的第一个块中。
- 如果补丁 ID 在 Event History 中，则执行不会修改历史记录，并返回 `true`。
  这种情况发生在补丁 ID 的后续块中，因为第一个块中的 Event History 是 Updated。

对于这种行为有一个警告，我们将在下面介绍。

<a id="behavior-when-replaying-with-marker-before-or-at-current-location"></a>
#### 在当前位置之前或当前位置重放标记时的行为 {#behavior-when-replaying-with-marker-before-or-at-current-location}

如果执行正在重播并且调用了 `patched()`，并且 Event History 在同一位置具有来自对 `patched()` 的调用的标记（这意味着它将与原始 Event History 匹配），则它将一个标记写入重播 Event History 并返回 `true`。

这与非重放情况的行为类似，并且也发生在给定补丁 ID 的第一个块中。

如果代码调用了 `patched()`，并且 Event History 具有历史记录中较早的带有该补丁 ID 的标记，则它将返回 `true` 并且不会修改重播 Event History。

这也类似于非重放情况的行为，并且也发生在给定补丁 ID 的后续块中。

<a id="behavior-when-replaying-with-marker-after-current-location"></a>
#### 在当前位置后使用标记重放时的行为 {#behavior-when-replaying-with-marker-after-current-location}

如果Event History的标记事件在当前执行点之后，则意味着新补丁为时过早。
执行将在原始补丁之前遇到新补丁。
该执行将尝试将标记写入重播 Event History，但会引发非确定性异常，因为重播和原始事件历史记录不匹配。

<a id="behavior-when-replaying-with-no-marker-for-that-patch-id"></a>
#### 在没有该补丁 ID 的标记的情况下重放时的行为 {#behavior-when-replaying-with-no-marker-for-that-patch-id}

在重放期间，如果给定补丁 ID 没有标记，执行将返回 `false` 并且不会向 Event History 添加标记。此外，以后所有使用该 ID 对 `patched()` 的调用都将返回 `false`——即使在重放完成并正在运行新代码之后也是如此。

[前一节](#behavior-when-not-replaying) 声明，如果执行未重播，则 `patched()` 函数将始终返回 `true`。
如果标记不存在，则会添加它，如果标记已存在，则不会重新添加。

但是，如果重播代码中已存在对 `patched()` 的调用（但不在 Event History 中），则不会发生此行为。
在这种情况下，该函数不会返回 `true`。

<a id="a-summary-of-the-two-potentially-unexpected-behaviors"></a>
#### 两种潜在意外行为的总结 {#a-summary-of-the-two-potentially-unexpected-behaviors}

回顾重播期间可能发生的潜在意外行为：

如果执行命中对 `patched()` 的调用，但补丁 ID 不在 Event History 中的“该点或之前”，您可能没有意识到当前执行位置“之后”的 Event History 很重要。
出现此行为的原因是：

- 如果该补丁 ID 稍后存在，您会收到非确定性错误
- 如果补丁稍后不存在，则不会收到非确定性错误，并且调用返回 `false`

如果执行使用历史记录中不存在的 ID 调用 `patched()`，那么它不仅会在该情况下返回 `false`，而且如果执行超过重播阈值并且正在运行新代码，它还会返回 `false`。

<a id="implications-of-the-behaviors"></a>
#### 行为的影响 {#implications-of-the-behaviors}

如果您在 Workflow 执行时部署新代码，则任何正在执行的 Workflow 都将重播到 Worker 关闭时的位置。
当他们执行此重播时，他们不会遵循代码中的 `patched()` 分支。
对于重播到部署之前的点并重新启动 Worker 后的其余执行，它们将：

- 如果重放代码中没有调用 `patched()`，则使用新代码
- 如果重放代码中有对 `patched()` 的调用，他们将
  在重播期间和之后运行未修补的代码

这可能听起来很奇怪，但它实际上正是所需要的，因为这意味着如果未来修补的代码依赖于早期修补的代码，那么它将不会使用新代码 - 它将使用旧代码。

但是，如果将来有新代码，并且主体中之前没有需要新补丁的代码，那么它可以切换到新代码，它会这样做。

请注意，此行为意味着 Workflow _并不总是运行最新的代码_。
仅当未重播或超过重播并且在整个重播期间没有调用 `patched()`（具有该 ID）时，它才会执行此操作。

<a id="recommendations"></a>
#### 建议 {#recommendations}

基于此行为及其含义，在修补新代码时，请始终将最新代码放在 if-patched-block 的顶部。

```python
if patched('v3'):
    # This is the newest version of the code.
    # put this at the top, so when it is running
    # a fresh execution and not replaying,
    # this patched statement will return true
    # and it will run the new code.
    pass
elif patched('v2'):
    pass
else:
    pass
```

以下示例显示了 `patched()` 在以不同方式排列的条件块中的行为方式。
在这种情况下，代码的条件块顶部没有最新代码。
由于 `patched()` 在不重播时将返回 `True`（前面的注意事项除外），因此此代码段将在新执行中运行 `v2` 分支而不是 `v3`。

```python
if patched('v2'):
    # This is bad because when doing a new execution (i.e. not replaying),
    # patched statements evaluate to True (and put a marker
    # in the event history), which means that new executions
    # will use v2, and miss v3 below
    pass
elif patched('v3'):
    pass
else:
  pass
```
