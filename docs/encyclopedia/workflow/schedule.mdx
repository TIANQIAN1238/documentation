<a id="schedule"></a>
## 什么是 Schedule？ {#schedule}

Schedule 包含在特定时间启动 [Workflow Execution](/workflow-execution) 的指令。
Schedule 提供比 [Temporal 计划任务](/cron-job) 更灵活、更用户友好的方法。

- [如何启用 Schedule](#limitations)
- [如何使用 Temporal CLI 操作 Schedule](/cli/schedule)

Schedule 具有身份并且独立于 Workflow Execution。
这与 Temporal Cron 作业不同，Temporal Cron 作业依赖于 cron Schedule 作为 Workflow Execution 的属性。

:::info

要在特定的一次性未来点而不是在重复的 Schedule 上触发 Workflow Execution，应使用 [启动延迟](/workflow-execution/timers-delays#delay-workflow-execution) 选项而不是 Schedule。

:::

＃＃＃ 行动

Schedule 的操作是建立 Workflow Execution 属性的地方，例如 Workflow 类型、Task Queue、参数和超时。

由 Schedule 启动的 Workflow Execution 具有以下附加属性：

- 操作的时间戳附加到 Workflow Id。
- `TemporalScheduledStartTime` [Search Attribute](/search-attribute) 已添加到 Workflow Execution 中。
  该值是操作的时间戳。
- `TemporalScheduledById` Search Attribute 已添加到 Workflow Execution 中。
  该值为 Schedule Id。

<a id="action"></a>
### 规格 {#action}

Schedule 规范定义了何时应采取操作。
除非许多 Schedule 同时有操作 Scheduled，否则操作通常应在指定时间的 1 秒内启动。
Schedule规格有两种：

- 一个简单的间隔，例如“每 30 分钟”（与 Unix 纪元开始对齐，并且可以选择包括相位偏移）。
- 基于日历的表达式，类似于许多软件支持的“cron 表达式”，包括较旧的 Temporal Cron 功能。

这两种类型有多种表示形式，具体取决于您使用的接口或 SDK，但它们都支持相同的功能。

例如，在 Temporal CLI 中，间隔被指定为字符串，如 `45m` 表示每 45 分钟一次，或 `6h/5h` 表示每 6 小时一次，但在每个时间段内的第五个小时开始。

在 Temporal CLI 中，日历表达式可以指定为具有五个（或六个或七个）位置字段的传统 cron 字符串，也可以指定为具有命名字段的 JSON：

```json
{
  "year": "2022",
  "month": "Jan,Apr,Jul,Oct",
  "dayOfMonth": "1,15",
  "hour": "11-14"
}
```

以下日历 JSON 字段可用：

- `year`
- `month`
- `dayOfMonth`
- `dayOfWeek`
- `hour`
- `minute`
- `second`
- `comment`

每个字段可以包含以逗号分隔的范围列表（或 `*` 通配符），并且每个范围可以包含一个斜杠，后跟一个跳过值。
`hour`、`minute` 和 `second` 字段默认为 `0`，而其他字段默认为 `*`，因此您只需几个字段即可描述许多有用的规格。

对于 `month`，可以使用月份名称代替整数（不区分大小写，允许缩写）。
对于 `dayOfWeek`，可以使用星期几名称。

`comment` 字段是可选的，可用于包含日历规范意图的自由格式描述，这对于复杂的规范很有用。

无论您提供哪种形式，日历和间隔规范都会转换为规范表示。
当您“描述”或“列出”Schedule 时看到的内容可能与您输入的内容不完全一样，但它具有相同的含义。

<a id="spec"></a>
#### 相位偏移 {#spec}

默认情况下，间隔与 Unix 纪元（1970 年 1 月 1 日午夜 UTC）对齐，因此如果没有相位偏移，7 天的间隔将在每周四午夜 UTC 等于您当地时间的任何时间触发，这对您的用例来说可能不方便。

当您需要重复出现的 Schedule 在特定的、对人类有意义的时间（例如每隔 10 天在 UTC 的特定日期下午 5 点）触发时，请使用相位偏移，而不是接受纪元对齐产生的任意时间。

您可以设置 `offset` 参数，以便 `interval` 规范在预定的日期和时间准确触发。以下示例说明了如何计算 `offset` 并将其应用于您的 Schedule。

时间戳是用 UNIX 时间测量的。这意味着 7 天是 604,800 秒 (7 * 24 * 60 * 60)，5 天是 432,000 (5 * 24 * 60 * 60)，等等。当以下条件成立时，Schedule 会触发：

```
((currentTimestampUTC - phase offset) % interval) == 0
```

因此，使用 7 天的间隔和 0 的相位偏移，这意味着 Schedule 在 `December 28th 2023, at midnight UTC` 上触发：

```
((1703721600 - 0) % 604800) = 0
```

对于 5 天的间隔和 0 的相位偏移，Schedule 在 `December 29th 2023` 上触发

```
((1703808000 - 0) % 432000) = 0
```

您可以计算任何间隔所需的相位偏移。获取您希望 Schedule 执行的任何日期和时间的 UNIX 时间戳，然后计算以下内容：

```
phase offset = referenceTimestampUTC % interval
```

例如，如果您希望 Schedule 每 10 天触发一次，包括 UTC 时间 2024 年 6 月 19 日下午 5 点，您需要计算 UNIX 时间戳：

```
UNIX Timestamp of 2024-06-19T17:00:00 UTC = 1718816400
```

然后您需要计算间隔的秒数：

```
Interval of 10 days = (10 * 24 * 60 * 60) = 864000
```

最后，您可以计算所需的相位偏移：

```
phase offset = 1718816400 % 864000 = 320400
```

因此，您将 Schedule 间隔设置为 `864000s`（10 天），并将阶段设置为 `320400s`，并且您将在 UTC 下午 5 点精确地每 10 天获得一次。

<a id="phase-offset"></a>
#### 其他规格功能 {#phase-offset}

**多个间隔/日历表达式：** 规范可以具有多个间隔和/或日历表达式的组合来定义特定的 Schedule。

**时间范围：** 通过规范提供绝对开始或结束时间（或两者），以确保在开始时间之前或结束时间之后不采取任何操作。

**排除：** 规范可以包含零个或多个日历表达式形式的排除。
这可以用来表达日程安排，例如“除节假日外每周一中午”。
您必须提供自己的排除集并将其包含在每个 Schedule 中；没有预定义的集合。
（此功能当前未在 Temporal CLI 或 Temporal Web UI 中公开。）

**抖动：** 如果给定，则将零和最大抖动之间的随机偏移添加到每个操作时间（但受下一个 Scheduled 操作之前的时间限制）。

**时区：** 默认情况下，基于日历的表达式以 UTC 进行解释。
Temporal 建议使用 UTC 以避免时区的各种令人惊讶的特性。
如果您不想使用 UTC，则可以提供时区名称。
时区定义从磁盘或二进制文件中嵌入的回退加载到 Temporal Server Worker 服务上。

为了获得更多操作控制，请将时区数据库文件的内容嵌入到 Schedule 规范本身中。
（注意：当前未在 Temporal CLI 或 Web UI 中公开。）

＃＃＃ 暂停

Schedule 可以暂停。
当 Schedule 暂停时，规范不起作用。
但是，您仍然可以使用 [Temporal Schedule 触发器](/cli/schedule#trigger) 命令强制执行手动操作。

为了帮助开发人员和操作人员之间的沟通，可以在暂停或恢复时使用“注释”字段 Updated 来存储当前状态的说明。

<a id="other-spec-features"></a>
### 回填 {#other-spec-features}

Schedule 可以回填。
当 Schedule 回填时，现在将执行在指定时间段内执行的所有操作（如果使用 `AllowAll` [重叠政策](#overlap-policy)，则并行执行；如果使用 `BufferAll`，则按顺序执行）。
您可以使用它来填充 Schedule 由于现已解决的外部条件而暂停的时间段或创建 Schedule 之前的时间段的运行。

<a id="pause"></a>
### 限制操作数量 {#pause}

一个 Schedule 可以被限制为一定数量的 Scheduled 操作（即不立即触发）。
之后，它将表现得好像暂停了一样。

<a id="backfill"></a>
### 政策 {#backfill}

Schedule 支持一组启用自定义行为的策略。

<a id="limit-number-of-actions"></a>
#### 重叠政策 {#limit-number-of-actions}

重叠策略控制当需要启动 Workflow Execution 但之前启动的 Workflow Execution 仍在运行时会发生什么情况。
可以使用以下选项：

- `Skip`：**默认**。
  什么也没有发生； Workflow Execution 未启动。
- `BufferOne`：当前 Workflow Execution 完成后立即启动。
  缓冲区仅限于一个。
  如果另一个 Workflow Execution 应该启动，但其中一个已经在缓冲区中，则只有缓冲区中的一个最终启动。
- `BufferAll`：允许缓冲无限数量的 Workflow。
  它们按顺序启动。
- `CancelOther`：取消正在运行的Workflow Execution，待旧的Workflow Execution取消完成后，再启动新的Workflow Execution。
- `TerminateOther`：终止正在运行的 Workflow Execution 并立即启动新的 Workflow Execution。
- `AllowAll` 启动任意数量的并发 Workflow Execution。
  通过此策略（且仅此策略），由 Schedule 启动的多个 Workflow Execution 可以同时运行。

<a id="policies"></a>
#### 追赶窗口 {#policies}

当 Schedule 应采取操作时，Temporal 服务可能已关闭或不可用。
当它恢复时，追赶窗口控制此时应采取哪些错过的操作。
默认值为一年，这意味着除非晚了一年以上，否则将采取行动。
如果您的操作对时间更加敏感，您可以将追赶窗口设置为较小的值（最少十秒），接受超过窗口时间的中断可能会导致错过操作。
（但您始终可以 [回填](#backfill)。）

<a id="overlap-policy"></a>
#### 失败时暂停 {#overlap-policy}

如果设置此策略，则由 Schedule 启动并以失败或超时（但不是取消或终止）结束的 Workflow Execution 会导致 Schedule 自动暂停。

请注意，对于 `AllowAll` 重叠策略，此暂停可能不适用于下一个 Workflow Execution，因为下一个 Workflow Execution 可能在失败的 Workflow Execution 完成之前就开始了。
它仅适用于在失败的 Scheduled 完成后启动的 Workflow Execution。

<a id="catchup-window"></a>
### 最后完成结果 {#catchup-window}

由Schedule启动的Workflow可以获得最近一次成功运行的完成结果。
（如何执行此操作取决于您使用的 SDK。）

对于不允许重叠的重叠策略，“最近成功的运行”很容易定义。
对于 `AllowAll` 策略，它指的是在相关运行开始时最近完成的运行。
考虑以下重叠运行：

```
time -------------------------------------------->
 A     |----------------------|
 B               |-------|
 C                          |---------------|
 D                                |--------------T
```

如果D在时间T询问最后的完成结果，它得到A的结果。
不是 B，尽管 B 开始得更晚，因为 A 完成得更晚。
而不是 C，即使 C 在 A 之后完成，因为 D 的结果是在 D 启动时捕获的，而不是在查询时捕获的。

失败和超时不会影响最后的完成结果。

:::note

当 Schedule 触发成功完成并产生结果的 Workflow 时，后续 Scheduled 执行可以通过 `LastCompletionResult` 访问初始 Schedule 执行的结果。

请注意，如果在后续运行期间，Workflow 使用 [Continue-As-New](/workflow-execution/continue-as-new) 功能，则此新的 Workflow 迭代将无法访问 `LastCompletionResult`。

需要注意的是，后续运行的 [地位](/workflow-execution#workflow-execution-status) 被标记为 `Continued-As-New`，而不是 `Completed`。

:::

:::caution

Scheduled Workflow Execution 可能会完成最大 Blob 大小（默认为 2 MiB）的结果。
但是，由于内部限制，在此限制的 1 KiB 以内的结果无法传递到下一次执行。
例如，Workflow Execution 返回大小为 2,096,640 字节的结果（高于 2MiB - 1KiB 限制）
将允许成功完成，但该值将不能作为最后的完成结果。
此限制将来可能会取消。

:::

<a id="pause-on-failure"></a>
### 最后一次失败 {#pause-on-failure}

由 Schedule 启动的 Workflow 可以获得在启动相关 Workflow 时结束的最近一次运行的失败详细信息。与上次完成结果不同，_成功_运行_确实_重置上次失败。

<a id="last-completion-result"></a>
### 限制 {#last-completion-result}

在内部，Schedule 被实现为 Workflow。
如果您使用 Elasticsearch，这些 Workflow Execution 在普通视图中是隐藏的。
