Temporal Service 通过将 Event 相关信息（例如 Workflow Execution 的开始或结束时间）追加到与该执行关联的 Event 历史记录中，来跟踪每个 Workflow Execution 的进度。
这些信息不仅使开发者能够了解发生了什么，而且对于提供持久执行至关重要，因为它使 Workflow Execution 能够从崩溃中恢复并继续推进进度。
为了维持高性能，Temporal Service 对每个 Workflow Execution 的 Event 历史记录中的项目数量和大小都设置了限制。

## 什么是 Event？{#event}

Event 由 Temporal Service 响应外部事件以及 Workflow Execution 生成的 Command 而创建。
每个 Event 对应于[服务器 API](https://github.com/temporalio/api/blob/master/temporal/api/enums/v1/event_type.proto) 中定义的 `enum`。

所有 Event 都记录在 [Event 历史记录](#event-history)中。

[Event 参考](/references/events)中提供了可能出现在 Workflow Execution Event 历史记录中的所有可能 Event 的列表。

### Activity Events {#activity-events}

在 Activity Execution 的不同时间点，七个与 Activity 相关的 Event 会被添加到 Event 历史记录中：

- 当 [Workflow Task Execution](/tasks#activity-task-execution) 执行到启动/执行 Activity 的代码行后，Worker 将 Activity 类型和参数发送给 Temporal Service，Temporal Service 将 [ActivityTaskScheduled](/references/events#activitytaskscheduled) Event 添加到 Event 历史记录。
- 当 `ActivityTaskScheduled` 被添加到历史记录时，Temporal Service 将相应的 Activity Task 添加到 Task Queue。
- 轮询该 Task Queue 的 Worker 获取 Activity Task 并运行 Activity 函数或方法。
- 如果 Activity 函数返回，Worker 向 Temporal Service 报告完成，Temporal Service 将 [ActivityTaskStarted](/references/events#activitytaskstarted) 和 [ActivityTaskCompleted](/references/events#activitytaskcompleted) 添加到 Event 历史记录。
- 如果 Activity 函数抛出[不可重试的 Failure](/references/failures#non-retryable)，Temporal Service 将 [ActivityTaskStarted](/references/events#activitytaskstarted) 和 [ActivityTaskFailed](/references/events#activitytaskfailed) 添加到 Event 历史记录。
- 如果 Activity 函数抛出错误或可重试的 Failure，Temporal Service 会调度 Activity Task 重试以添加到 Task Queue（除非您已达到 [Retry Policy](/encyclopedia/retry-policies) 的最大尝试次数，此时 Temporal Service 会将 [ActivityTaskStarted](/references/events#activitytaskstarted) 和 [ActivityTaskFailed](/references/events#activitytaskfailed) 添加到 Event 历史记录）。
- 如果 Activity 的 [Start-to-Close Timeout](/encyclopedia/detecting-activity-failures#start-to-close-timeout) 在 Activity 函数返回或抛出之前过期，Temporal Service 会调度重试。
- 如果 Activity 的 [Schedule-to-Close Timeout](/encyclopedia/detecting-activity-failures#schedule-to-close-timeout) 在 Activity Execution 完成之前过期，或者 [Schedule-to-Start Timeout](/encyclopedia/detecting-activity-failures#schedule-to-start-timeout) 在 Worker 获取 Activity Task 之前过期，Temporal Service 会将 [ActivityTaskTimedOut](/references/events#activitytasktimedout) 写入 Event 历史记录。
- 如果 Activity 被[取消](/activity-execution#cancellation)，Temporal Service 将 [ActivityTaskCancelRequested](/references/events#activitytaskcancelrequested) 写入 Event 历史记录，如果 Activity 接受取消，Temporal Service 将写入 [ActivityTaskCanceled](/references/events#activitytaskcanceled)。

:::note

当 Activity 正在运行和重试时，[ActivityTaskScheduled](/references/events#activitytaskscheduled) 是历史记录中唯一与 Activity 相关的 Event：[ActivityTaskStarted](/references/events#activitytaskstarted) 与像 [ActivityTaskCompleted](/references/events#activitytaskcompleted) 或 [ActivityTaskFailed](/references/events#activitytaskfailed) 这样的终止 Event 一起写入。

:::

### 什么是 Event 历史记录？{#event-history}

应用程序 [Event](#event) 的仅追加日志。

- Event 历史记录由 Temporal Service 持久存储，使您的应用程序能够从崩溃或故障中无缝恢复状态。
- 它还可作为调试的审计日志。

### Event 历史记录限制 {#event-history-limits}

Temporal Service 存储 Workflow Execution 整个生命周期的完整 Event 历史记录。

Temporal Service 在[超过 10,240 个 Event 时记录警告](/workflow-execution/limits)，并在添加新 Event 时定期记录额外警告。

当 Event 历史记录出现以下情况时，Workflow Execution 将被终止：

- 超过 51,200 个 Event。
- 包含超过 2000 个 Update。
- 包含超过 10000 个 Signal。

为了避免达到这些限制，您可以使用 [Continue-As-New](/workflow-execution/continue-as-new) 功能关闭当前 Workflow Execution 并创建新的 Workflow Execution。

### Event 循环 {#event-loop}

Workflow Execution 由称为 [Event 历史记录](#event-history)的 [Event](#event) 序列组成。
Event 由 Temporal Service 响应 Command 或 Temporal Client 请求的操作（例如生成 Workflow Execution 的请求）而创建。

## 时间约束 {#time-constraints}

**Workflow 的运行时间有限制吗？**

没有，Workflow Execution 的运行时间没有时间约束。

但是，如果您的 Workflow 将执行许多操作，或者将接收许多消息，它可能会遇到 [Event 历史记录限制](#event-history-limits)。

它还可能遇到 [Workflow Versioning](/workflow-definition#workflow-versioning) 和其他向后不兼容性问题。

出于这些原因，定期使用 [Continue-As-New](/workflow-execution/continue-as-new) 可能是个好主意。

## 什么是 Reset？{#reset}

Reset 终止一个 [Workflow Execution](/workflow-execution) 并创建一个具有相同 [Workflow Type](/workflow-definition#workflow-type) 和 [Workflow ID](/workflow-execution/workflowid-runid) 的新 Workflow Execution。
[Event 历史记录](/workflow-execution/event#event-history) 从原始执行复制，直到并包括重置点。
新执行从重置点继续。有效的重置点为：`WorkflowTaskStarted`、`WorkflowTaskCompleted`、`WorkflowTaskTimedOut` 和 `WorkflowTaskFailed`。
原始历史记录中的 Signal 可以选择性地复制到新历史记录，无论它们是否出现在重置点之后。

## 什么是 Side Effect？{#side-effect}

:::note

Side Effect 包含在 Go、Java 和 PHP SDK 中。
其他 SDK 不包含它。
[Local Activity](/local-activity) 适用于相同的用例，且资源消耗稍少。

:::

Side Effect 是一种执行短小的非确定性代码片段（例如生成 UUID）的方式，它执行所提供的函数一次，并将其结果记录到 Workflow Execution Event 历史记录中。

Side Effect 在重放时不会重新执行，而是返回已记录的结果。

永远不要有可能失败的 Side Effect，因为失败可能导致 Side Effect 函数被执行多次。
如果提供给 Side Effect 的代码有任何失败的可能，请使用 Activity。
