## 什么是Workflow Execution？ {#workflow-execution}

虽然 Workflow Definition 是定义 Workflow 的代码，但 Workflow Execution 是通过执行该代码创建的。
Temporal Workflow Execution 是持久、可靠且可扩展的功能执行。
它是[Temporal应用程序](/temporal#temporal-application)的主要执行单元。

- [如何使用时间启动Workflow Execution](/cli/workflow#start)
- [如何使用Go SDK启动Workflow Execution](/develop/go/client/temporal-client#start-workflow-execution)
- [如何使用Java SDK启动Workflow Execution](/develop/java/client/temporal-client#start-workflow-execution)
- [如何使用PHP SDK启动Workflow Execution](/develop/php/client/temporal-client#start-workflow-execution)
- [如何使用Python SDK启动Workflow Execution](/develop/python/client/temporal-client#start-workflow-execution)
- [How to start a Workflow Execution using the TypeScript SDK](/develop/typescript/client/temporal-client#start-workflow-execution)
- [如何使用.NET SDK启动Workflow Execution](/develop/dotnet/client/temporal-client#start-workflow)

每个 Temporal Workflow Execution 都可以独占访问其本地状态。
它与所有其他 Workflow Execution 并发执行，并通过 [Signals](/sending-messages#sending-signals) 与其他 Workflow Execution 通信，并通过 [活动](/activities) 与环境通信。
虽然单个 Workflow Execution 在大小和吞吐量上有限制，但 Temporal 应用程序可以由数百万到数十亿个 Workflow Execution 组成。

**Durability**

耐久性是指没有强加的时间限制。

Workflow Execution 是持久的，因为它执行 Temporal Workflow Definition（也称为 Temporal Workflow 函数）、您的应用程序代码，一次有效并完成 — 无论您的代码执行几秒还是几年。

**Reliability**

可靠性是出现故障时的响应能力。

Workflow Execution 是可靠的，因为它在发生故障后可以完全恢复。
Temporal 平台确保 Workflow Execution 的状态在出现故障和中断时持续存在，并从最新状态恢复执行。

**Scalability**

可扩展性是负载存在时的响应能力。

单个 Workflow Execution 的大小和吞吐量受到限制，但具有可扩展性，因为它可以 [Continue-As-New](/workflow-execution/continue-as-new) 响应负载。
Temporal 应用程序是可扩展的，因为 Temporal 平台能够支持数百万至数十亿个 Workflow Execution 同时执行，这是通过 [Temporal 服务](/temporal-service) 和 [Worker 进程](/workers#worker-process) 的设计和性质实现的。

### Replays {#replay}

重播是 Workflow Execution 恢复进度的方法。在重放期间，将根据现有 Event History 检查生成的命令。重播是必要的，并且经常会产生 Workflow Execution 可恢复、可靠且持久的效果。

有关更多信息，请参阅[确定性约束](/workflow-definition#deterministic-constraints)。

如果发生故障，Workflow Execution 将拾取 Event History 中最后记录的事件发生的位置。

### 命令和等待 {#commands-awaitables}

Workflow Execution 有两件事：

1. 发出[命令](#command)。
2. 等待 Awaitables（通常称为 Futures）。

<CaptionedImage
src="/diagrams/workflow-execution-progession-simple.svg"
title="命令生成并等待" />

通过使用 [Workflow Definition](/workflow-definition) 中的 Workflow API 来发出命令并提供等待。

每当执行 Workflow 功能时都会生成命令。
Worker 进程监督命令生成并确保其映射到当前的 Event History。
（更多信息请参见[确定性约束](/workflow-definition#deterministic-constraints)。）
Worker 进程对命令进行批处理，然后每当 Workflow 函数到达如果没有可等待结果就无法再继续进行的位置时，就会暂停进度以将命令发送到 Temporal 服务。

Workflow Execution 只能阻止通过 Temporal SDK API 提供的 Awaitable 的进度。
使用 API 时会提供以下可等待项：

- 等待：使用显式“等待”API 可以阻止进度。
- 请求取消另一个 Workflow Execution：在确认另一个 Workflow Execution 已取消时，进度可能会阻塞。
- 发送 [Signal](/sending-messages#sending-signals)：确认 Signal 发送后，进度可能会阻塞。
- 生成 [子 Workflow Execution](/child-workflows)：在确认子 Workflow Execution 已启动以及子 Workflow Execution 的结果时，进度可能会阻塞。
- 生成 [Activity 执行](/activity-execution)：进度可能会阻塞 Activity 执行的结果。
- 启动 Timer：进程可能会阻塞，直到 Timer 触发。

### 什么是命令？ {#command}

命令是在 [Workflow 任务执行](/tasks#workflow-task-execution) 完成后，由 [Worker](/workers#worker) 向 [Temporal 服务](/temporal-service) 发出的请求操作。

Temporal 服务采取的操作作为 [事件](/workflow-execution/event) 记录在 [Workflow Execution's](#workflow-execution) [Event History](/workflow-execution/event#event-history) 中。
Workflow Execution 可以等待某些命令导致的某些事件。

命令是通过在代码中使用 Workflow API 生成的。在 Workflow 任务执行期间可能会生成多个命令。
在 Workflow 任务通过 Workflow 功能尽可能进展之后，命令将被批量发送到 Temporal 服务，作为 Workflow 任务执行完成请求的一部分。
当有 Workflow 任务执行完成请求时，Event History 中总会有 [WorkflowTaskStarted](/references/events#workflowtaskstarted) 和 [WorkflowTaskCompleted](/references/events#workflowtaskcompleted) 事件。

<CaptionedImage
src="/diagrams/commands.svg"
title="命令是通过在代码中使用 Workflow API 生成的" />

命令在[命令参考](/references/commands)中描述，并在[Temporal gRPC API](https://github.com/temporalio/api/blob/master/temporal/api/command/v1/message.proto).

### 地位 {#workflow-execution-status}

Workflow Execution 可以是_Open_ 或_Closed_。

<CaptionedImage
src="/diagrams/workflow-execution-statuses.svg"
title="Workflow Execution 状态" />

#### Open

_Open_ 状态意味着 Workflow Execution 能够取得进展。

- 运行中：Workflow Execution 的唯一开放状态。
当 Workflow Execution 正在运行时，它要么正在积极进行，要么正在等待某些事情。

#### Closed

_Closed_状态意味着Workflow Execution由于以下原因之一而无法取得进一步进展：

- 已取消：Workflow Execution 成功处理取消请求。
- 已完成：Workflow Execution 已成功完成。
- 续如新：Workflow Execution [续如新](/workflow-execution/continue-as-new)。
- 失败：Workflow Execution 返回错误并失败。
- 终止：Workflow Execution 已终止。
- 超时：Workflow Execution 达到超时限制。

### Workflow Execution链 {#workflow-execution-chain}

Workflow Execution 链是共享相同 Workflow Id 的 Workflow Execution 序列。
链中的每个环节通常称为 Workflow 运行。
序列中的每个 Workflow Run 由以下之一连接：

- [Continue-As-New](/workflow-execution/continue-as-new)
- [重试](/encyclopedia/retry-policies)
- [Temporal 定时作业](/cron-job)

Workflow Execution 由其 [Namespace](/namespaces)、[Workflow Id](/workflow-execution/workflowid-runid#workflow-id) 和 [Run Id](/workflow-execution/workflowid-runid#run-id) 唯一标识。

[Workflow Execution 超时](/encyclopedia/detecting-workflow-failures#workflow-execution-timeout) 适用于 Workflow Execution 链。
[Workflow 运行超时](/encyclopedia/detecting-workflow-failures#workflow-run-timeout) 适用于单个 Workflow Execution（Workflow 运行）。

## 什么是备忘录？ {#memo}

备忘录是一组非索引的 Workflow Execution 元数据，开发人员在启动时或在 Workflow 代码中提供，并在您描述或列出 Workflow Execution 时返回。

使用备忘录的主要目的是加强 Workflow Execution 的组织和管理。
将您自己的元数据（例如注释或描述）添加到 Workflow Execution，这样您就可以根据开发人员定义的标准对 Workflow Execution 进行注释和分类。
在处理大量 Workflow Execution 时，此功能特别有用，因为它有助于添加上下文、提醒或任何其他有助于理解或跟踪 Workflow Execution 的相关信息。

:::note Use Memos judiciously

备忘录不应存储对 Workflow 执行至关重要的数据，原因如下：

- 与 Workflow 输入不同，备忘录缺乏类型安全性
- 备忘录取决于最终一致性，可能无法立即可用
- 过度依赖备忘录隐藏了 Workflow Execution 历史中的可变状态

:::

## 什么是状态转换？ {#state-transition}

状态转换是 [Workflow Execution](#workflow-execution) 取得的进展单位。
每个状态转换都记录在持久性存储中。

某些操作，例如 [Activity 心跳](/encyclopedia/detecting-activity-failures#activity-heartbeat)，每个操作仅需要一到两个状态转换。对于 Activity 心跳，有两种：Activity 心跳和 Timer。

大多数操作需要多个状态转换。

例如，具有两个连续 [Activity 任务](/tasks#activity-task)（且无重试）的简单 Workflow 会产生 11 个状态转换：两个用于 Workflow 启动，四个用于每个 Activity，一个用于 Workflow 完成。
