<a id="timer"></a>
## 什么是 Timer？ {#timer}

Temporal SDK 提供 Timer API，以便 Workflow Execution 在处理时间值时具有确定性。

Temporal 中的 Timer 会持续存在，这意味着即使您的 Worker 或 Temporal 服务在时间段结束时关闭，只要您的 Worker 和 Temporal 服务可用，等待 Workflow 代码中的 Timer 的呼叫就会解决，从而导致执行才能继续进行。
Timer 可靠且高效。
Worker 在等待 Timer 触发时不会消耗额外的资源，因此单个 Worker 可以同时等待数百万个 Timer。

Timer 的持续时间是固定的，您的 Workflow 可能指定短至一秒或长至几年的值。
尽管可以指定极其精确的持续时间，例如 36 毫秒或 15.072 分钟，但您的 Workflow 不应依赖于 Timer 的亚秒精度。
我们建议您将持续时间视为最短时间，由于调度和触发 Timer 所涉及的延迟，该时间会稍微向上舍入。
例如，将 Timer 设置为 11.97 秒可以保证至少延迟执行那么长的时间，但实际上可能更接近 12 秒。

<a id="delay-workflow-execution"></a>
## 什么是启动延迟？ {#delay-workflow-execution}

:::tip COMPATIBILITY

启动延迟 Workflow Execution 与 [Schedules](/schedule) 和 [计划任务](/cron-job) 不兼容。

:::

启动延迟确定启动 Workflow Execution 之前等待的时间量。
如果您有一个 Workflow，并且希望将来将 Schedule 输出，但只希望它执行一次：与使用 Schedule 重复出现的 Workflow 相比，这非常有用。

如果 Workflow 在延迟期间收到 Signal-With-Start 或 Update-With-Start，它将分派 Workflow 任务，并绕过剩余的延迟。
如果 Workflow 在延迟期间收到不是 Signal-With-Start 的 Signal，则 Signal 不会中断延迟，并且 Workflow 继续延迟，直到延迟到期或收到 Signal-With-Start。

您可以通过在所选 SDK 的 Workflow 选项字段中设置此选项来延迟初始 Workflow Execution 的调度。
此延迟仅适用于初始 Workflow Execution，不会影响后续执行，例如当 Workflow Continues-as-New 时。
