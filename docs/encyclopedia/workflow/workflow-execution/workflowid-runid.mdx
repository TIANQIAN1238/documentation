每个 Workflow Execution 都与用户定义的 [Workflow ID](#workflow-id) 相关联，该值通常带有一些业务含义（例如订单号或客户号）。
Temporal 保证在任何时间点最多可以有一个具有给定 ID 的 Workflow Execution 运行，这一约束有助于防止意外重复。
在某些情况下，例如使用 Schedule 功能以重复间隔运行相同的 Workflow 时，在一段时间内可能会多次“运行”单个 Workflow Execution。
在这种情况下，所有运行都将具有相同的 Workflow ID。
但是，每次运行都会有一个唯一的系统生成的 [运行ID](#run-id)。

<a id="run-id"></a>
## 什么是运行 ID？ {#run-id}

Run Id 是 [Workflow Execution](/workflow-execution) 的全局唯一平台级标识符。

当前运行 ID 是可变的，并且可以在 [Workflow 重试](/encyclopedia/retry-policies) 期间更改。您不应依赖于存储当前的运行 Id，或将其用于任何逻辑选择，因为 Workflow 重试会更改运行 Id，并可能导致不确定性问题。

Temporal 保证在任何给定时间只有一个具有给定 [Workflow ID](#workflow-id) 的 Workflow Execution 可以处于打开状态。
但是，当 Workflow Execution 达到“关闭”状态时，可能有另一个具有相同 Workflow Id 的处于“打开”状态的 Workflow Execution。
例如，Temporal Cron 作业是一系列 Workflow Execution，它们都具有相同的 Workflow Id。
链中的每个 Workflow Execution 都被视为_Run_。

运行 Id 唯一标识 Workflow Execution，即使它与其他 Workflow Execution 共享 Workflow Id。

<a id="which-operations-lead-to-non-determinism-issues"></a>
### 哪些操作会导致非确定性问题？ {#which-operations-lead-to-non-determinism-issues}

`ContinueAsNew`、`Retry`、`Cron` 和 `Reset` 等操作会创建由 [`first_execution_run_id`](/workflow-execution#workflow-execution-chain) 标识的 [Workflow Execution链条](https://github.com/temporalio/api/blob/master/temporal/api/history/v1/message.proto)。

每个操作都会在链运行中创建一个新的 Workflow Execution，并将其信息保存为 `first_execution_run_id`。
因此，在 Workflow Execution 上的每次操作期间，运行 ID 为 Updated。

- `first_execution_run_id` 是链运行中第一个 Workflow Execution 的运行 ID。
- `original_execution_run_id` 是 `WorkflowExecutionStarted` 事件发生时的运行 ID。

Workflow `Reset` 更改第一个执行运行 Id，但保留原始执行运行 Id。
例如，当链中的新 Workflow Execution 启动时，它将其运行 ID 存储在 `original_execution_run_id` 中。
重置不会更改该字段，但当前运行 ID 为 Updated。

:::caution

由于这种行为，您不应依赖代码中的当前运行 ID 来做出逻辑选择。

:::

<a id="workflow-id"></a>
## Workflow ID 是什么？ {#workflow-id}

Workflow Id 是 [Workflow Execution](/workflow-execution) 的可自定义应用程序级标识符，对于 [Namespace](/namespaces) 中的 Open Workflow Execution 是唯一的。

- [如何设置 Workflow Id](/develop/go/client/temporal-client#workflow-id)

Workflow Id 是指业务流程标识符，例如客户标识符或订单标识符。

:::caution Do not use sensitive data or PII in Workflow Ids

请勿将敏感数据、秘密或个人身份信息 (PII) 包含在 Workflow ID 中。
Workflow ID 以纯文本形式存储，**不**由自定义 [有效负载编解码器](/payload-codec#payload-codec) 处理，并且在 Temporal Web UI、CLI 输出、Event History 和系统日志中可见。
这同样适用于其他用户定义的标识符，例如 Workflow 类型名称、Task Queue 名称、Activity 名称和 Signal/Query/Update 名称。

在这些标识符中使用敏感数据可能会导致任何具有 Namespace 访问权限的人面临暴露的风险，并且可能违反 GDPR、HIPAA 或 SOC 2 等数据保护法规。

:::

Temporal 平台根据 Workflow Id 重用策略保证 [Namespace](/namespaces) 中 Workflow Id 的唯一性。

[Workflow ID 重用政策](#workflow-id-reuse-policy) 可用于管理是否可以重复使用已关闭的 Workflow 中的 Workflow Id。

[Workflow ID 冲突政策](#workflow-id-conflict-policy) 可用于决定如何解决 Workflow Id 与正在运行的 Workflow 的冲突。

Workflow Execution 可以通过其 [Namespace](/namespaces)、Workflow Id 和 [运行 ID](#run-id) 在所有 Namespace 中唯一标识。

<a id="workflow-id-reuse-policy"></a>
### 什么是 Workflow ID 重用政策？ {#workflow-id-reuse-policy}

Workflow Id 重用策略确定是否允许使用特定 Workflow Id 生成 Workflow（如果该 Workflow Id 已与先前且现已关闭的 Workflow Execution 一起使用）。

无论 Workflow Id 重用策略如何，都不可能生成具有与另一个打开的 Workflow Execution 相同的 Workflow Id 的新 Workflow Execution。

请参阅 [Workflow ID 冲突政策](#workflow-id-conflict-policy) 以解决 Workflow Id 冲突。

Workflow ID 重用策略可以具有以下值之一：

- **允许重复：** 无论具有相同 Workflow Id 的先前 Workflow Execution 的关闭状态如何，都允许 Workflow Execution 存在。
  **如果未指定，这是默认策略。**
  当 Workflow Execution 可以与之前但现已关闭的 Workflow Execution 具有相同的 Workflow Id 时，请使用此选项。
- **仅允许重复失败：** 仅当具有相同 Workflow Id 的先前 Workflow Execution 不具有“已完成”状态时，才允许 Workflow Execution 存在。
  当需要重新执行失败、超时、终止或取消的 Workflow Execution 并保证已完成的 Workflow Execution 不会被重新执行时，请使用此策略。
- **拒绝重复：** 如果先前的 Workflow Execution 具有相同的 Workflow Id，则无论关闭状态如何，Workflow Execution 都不能存在。
  当给定保留期内 Namespace 内的每个 Workflow Id 只能有一个 Workflow Execution 时，请使用此选项。

Workflow ID 重用策略的前三个值（允许重复、仅允许失败的重复和拒绝重复）适用于保留在 Namespace 中的已关闭 Workflow Execution。
例如，给定默认保留期，Temporal 服务只能根据 Workflow ID 重用策略对照过去 30 天的关闭的 Workflow Execution 检查生成的 Workflow Execution 的 Workflow ID。

如果您仅需要在特定实施尚未启动的情况下启动 Workflow，请确保您的保留期足够长以供检查。
如果这变得难以处理，请考虑使用 [Workflow消息传递](/encyclopedia/workflow-message-passing) 而不是尝试以原子方式启动 Workflow。

Workflow ID 重用策略的第四个值“运行时终止”仅适用于当前在 Namespace 中打开的 Workflow Execution。
对于“运行时终止”，保留期不是此策略的考虑因素。

如果尝试使用不允许的 Workflow ID 重用策略生成 Workflow Execution，服务器将阻止 Workflow Execution 生成。

<a id="workflow-id-conflict-policy"></a>
### 什么是 Workflow ID 冲突政策？ {#workflow-id-conflict-policy}

Workflow Id 冲突策略确定在生成新 Workflow Execution 与现有 Open Workflow Execution 使用的特定 Workflow Id 时如何解决冲突。
请参阅 [Workflow ID 重用政策](#workflow-id-reuse-policy)，了解管理已关闭 Workflow 的 Workflow Id 的重用。

默认情况下，这会导致 `Workflow Execution already started` 错误。

:::note

Go SDK 中的默认 [开始Workflow选项](https://pkg.go.dev/go.temporal.io/sdk/internal#StartWorkflowOptions) 行为是当使用与 Open Workflow Execution 相同的 Workflow Id 尝试新的 Workflow Execution 时不返回错误。
相反，它返回一个 WorkflowRun 实例，表示 Open Workflow Execution 的当前或上次运行。

要返回 `Workflow Execution already started` 错误，请将 `WorkflowExecutionErrorWhenAlreadyStarted` 设置为 `true`。

:::

Workflow ID 冲突策略可以具有以下值之一：

- **失败：** 阻止 Workflow Execution 生成并返回 `Workflow Execution already started` 错误。
  **如果未指定，这是默认策略。**
- **使用现有：** 防止 Workflow Execution 生成并返回带有打开 Workflow Execution 运行 ID 的成功响应。
- **终止现有：** 终止打开的 Workflow Execution，然后生成具有相同 Workflow Id 的新 Workflow Execution。
