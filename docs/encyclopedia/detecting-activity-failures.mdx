## Schedule-启动超时 {#schedule-to-start-timeout}

**Temporal 中的 Schedule 启动超时是什么？**

Schedule 启动超时是从 [Activity 任务](/tasks#activity-task) 被调度（即，放置在 Task Queue 中）到 [Worker](/workers#worker) 启动（即从 Task Queue 中获取）该 Activity 任务所允许的最大时间量。
换句话说，它是 Activity 任务排队时间的限制。

  <RelatedReadItem path="/develop/typescript/activities/timeouts" text="Set a Schedule-To-Start Timeout using the TypeScript SDK" archetype="feature-guide" />
  <RelatedReadItem path="/develop/dotnet/activities/timeouts" text="Set a Schedule-To-Start Timeout using the .NET SDK" archetype="feature-guide" />

出于 Schedule 启动超时和相关指标的目的，Worker 从 Task Queue 中选择任务的那一刻被视为 Activity 任务执行的开始。
“开始”的这种定义避免了 Temporal 服务和 Worker 之间的时钟差异可能产生的问题。

<CaptionedImage
src="/diagrams/schedule-to-start-timeout.svg"
title="Schedule-启动超时时间" />

Schedule-To-Start 和 Schedule-To-Close 中的“Schedule”具有不同的频率保证。

每个 Activity 任务都会强制执行 Schedule-To-Start 超时，而每个 Activity 执行都会强制执行一次 Schedule-To-Close 超时。
因此，Schedule-To-Start中的“Schedule”指的是组成Activity执行的Activity任务序列中的_every_ Activity任务的调度时刻，而
Schedule-To-Close 中的“Schedule”指的是该序列中的第一个 Activity 任务。

附加到 Activity 执行的 [Retry Policy](/encyclopedia/retry-policies) 会重试 Activity 任务。

<CaptionedImage
src="/diagrams/schedule-to-start-timeout-with-retry.svg"
title="开始到关闭重试超时时间" />

此超时有两个主要用例：

1. 检测单个Worker是否崩溃。
2. 检测轮询 Task Queue 的 Worker 队列是否无法跟上 Activity 任务的速率。

**默认的 Schedule-To-Start 超时为 ∞（无穷大）。**

如果使用此超时，我们建议将此超时设置为 Workflow Execution 在所有可能的 Worker 中断的情况下愿意等待 Activity 执行的最长时间，并制定具体计划将 Activity 任务重新路由到不同的 Task Queue。
无论 Retry Policy 为何，此超时 **不会** 触发任何重试，因为重试会将 Activity 任务放回到相同的 Task Queue 中。
我们不建议使用此超时，除非您知道自己在做什么。

在大多数情况下，我们建议监控 `temporal_activity_schedule_to_start_latency` 指标以了解 Worker 何时减慢拾取 Activity 任务的速度，而不是设置此超时。

## 开始到关闭超时 {#start-to-close-timeout}

**Temporal 中的开始到关闭超时是什么？**

开始到关闭超时是单个 [Activity 任务执行](/tasks#activity-task-execution) 允许的最长时间。

  <RelatedReadItem path="/develop/dotnet/activities/timeouts" text="Set a Start-To-Close Timeout using the .NET SDK" archetype="feature-guide" />
</RelatedReadContainer>

**默认的开始到关闭超时与默认的[Schedule-To-Close Timeout](#schedule-to-close-timeout)相同。**

Activity 执行必须设置此超时（开始到关闭）或 [Schedule 到关闭超时](#schedule-to-close-timeout)。
我们建议始终设置此超时；但是，请确保启动到关闭超时始终设置为长于完成 Activity 执行的最大可能时间。
对于长时间运行的 Activity 执行，我们建议还使用 [Activity 心跳](#activity-heartbeat) 和 [心跳超时](#heartbeat-timeout)。

:::tip

我们强烈建议设置开始到关闭超时。

当 Worker 与服务器失去通信或崩溃时，Temporal Server 不会检测到故障。
因此，Temporal Server 依靠启动到关闭超时来强制 Activity 重试。

:::

开始到关闭超时的主要用例是检测 Worker 在开始执行 Activity 任务后何时崩溃。

src="/diagrams/start-to-close-timeout.svg"
title="开始到关闭超时时间" />

附加到 Activity 执行的 [Retry Policy](/encyclopedia/retry-policies) 会重试 Activity 任务执行。
因此，开始到关闭超时适用于 Activity 执行中的每个 Activity 任务执行。

如果第一个 Activity 任务执行第一次返回错误，则完整的 Activity 执行可能如下所示：

<CaptionedImage
src="/diagrams/start-to-close-timeout-with-retry.svg"
title="开始到关闭重试超时时间" />

如果达到此超时，则会发生以下操作：

- [ActivityTaskTimedOut](/references/events#activitytasktimedout) 事件被写入 Workflow Execution 的可变状态。
- 如果 Retry Policy 指示重试，则 Temporal 服务会安排另一个 Activity 任务。
  - 在 Workflow Execution 的可变状态下，尝试计数加 1。
  - 启动-关闭超时计时器被重置。

## Activity 心跳 {#activity-heartbeat}

**Temporal中的Activity心跳是什么？**

Activity 心跳是从正在执行 Activity 的 Worker 到 Temporal 服务的 ping。
每个 ping 都会通知 Temporal 服务 Activity 执行正在取得进展，并且 Worker 尚未崩溃。

Activity 心跳与[心跳超时](#heartbeat-timeout) 结合使用。

Activity 心跳在 Activity Definition 内实现。
自定义进度信息可以包含在心跳中，然后在重试发生时可以由 Activity 执行使用。

可以根据需要多次记录 Activity 心跳（例如，每分钟一次或每次循环迭代一次）。
除了最短的 Activity 执行之外，对任何东西进行心跳检测通常是一个很好的做法。
Temporal SDK 控制向 Temporal 服务发送心跳的速率。

[本地活动](/local-activity)不需要心跳，并且不执行任何操作。

对于_长时间运行_的活动，我们建议使用相对较短的心跳超时和频繁的心跳。
这样，如果 Worker 失败，可以及时处理。

心跳可以包含可用于保存 Activity 执行进度的应用程序层有效负载。
如果 [Activity 任务执行](/tasks#activity-task-execution) 由于错过心跳而超时，下一个 Activity 任务可以访问并继续该有效负载。

当活动心跳时，Activity 取消会从 Temporal 服务传递到活动。没有心跳的活动无法收到取消。
心跳限制可能会导致取消的发送时间晚于预期。

### Throttling

心跳可能并不总是发送到 Temporal 服务 - 它们可能会受到 Worker 的限制。
节流间隔是以下较小者：

- 如果提供`heartbeatTimeout`，则`heartbeatTimeout * 0.8`；否则，`defaultHeartbeatThrottleInterval`
- `maxHeartbeatThrottleInterval`

`defaultHeartbeatThrottleInterval`默认为30秒，`maxHeartbeatThrottleInterval`默认为60秒。
每个都可以在 Worker 选项中设置。

节流的实现方式如下：

- 发送心跳后，Worker 会设置一个计时器来控制节流间隔。
- Worker 停止发送心跳，但继续从 Activity 接收心跳并记住最近的心跳。
- 当定时器触发时，Worker：
  - 发送最近的心跳。
  - 再次设置定时器。

限制允许 Worker 通过抑制不必要的心跳来减少 Temporal 服务上的网络流量和负载，以防止心跳超时。
在 Activity 失败的情况下，限制不适用于最终的心跳消息。
如果 Activity 在心跳消息中记录进度信息后立即失败，则该进度信息将在下一次重试尝试期间可用，前提是 Worker 本身在将其传递到 Temporal 服务之前没有崩溃。

### 哪些活动应该心跳？

最好不要从时间的角度来考虑心跳，而是从“你怎么知道你正在取得进展？”的角度来考虑。
对于短期操作，不需要进度更新。
然而，检查长时间运行的 Activity 执行的进度和状态几乎总是有用的。

设置 Activity 心跳时请考虑以下事项：

- 您的基本任务必须能够报告明确的进度。
请注意，当 Activity 仍在执行时，您的 Workflow 无法读取此进度信息（或者必须将其存储在 Event History 中）。
如果需要向用户公开，您可以向外部源报告进度。

- 您的 Activity 执行是长时间运行的，您需要验证正在处理您的 Activity 的 Worker 是否仍然存在，并且没有耗尽内存或无提示崩溃。

例如，以下场景适合使用Heartbeating：

- 从 Amazon S3 读取大文件。
- 在一些本地 GPU 上运行 ML 训练作业。

并且以下场景不适合Heartbeating：

- 进行快速 API 调用。
- 从磁盘读取一个小文件。
