Temporal 是一个可扩展且可靠的运行时，用于执行称为 [Temporal Workflow Execution](/workflow-execution) 的持久化函数。

换句话说，它是一个能够保证应用代码 [Durable Execution](#durable-execution) 的平台。

有了 Temporal，你可以像“不会发生故障”一样去开发应用。
即使应用遇到网络中断、服务器崩溃等问题——而这些问题对传统应用往往是灾难性的——你的应用依然可以可靠运行。
Temporal 平台会接管这类故障处理，让你把精力集中在业务逻辑上，而不是自己编写检测故障与恢复执行的代码。

## Durable Execution {#durable-execution}

在 Temporal 的语境里，Durable Execution 指的是 Workflow Execution 即使在故障、崩溃或服务器不可用的情况下，也依然能够保持自身状态并继续推进。

这一能力依赖于 Temporal 使用 [Event History](/workflow-execution/event#event-history) 记录 Workflow Execution 在每一步的状态。
当故障发生时，Workflow Execution 可以从最近一次已记录的事件恢复，确保执行进度不会丢失。

这正是 Temporal Workflow Execution 可靠且具备韧性的关键特性。
无论你的应用代码运行几秒、几天还是几年，它都能有效地、持续地执行直到完成。

## 什么是 Temporal 平台？ {#temporal-platform}

Temporal 平台由 [Temporal Service](/temporal-service) 和 [Worker 进程](/workers#worker-process) 组成。
它们共同为 Workflow Execution 提供运行时。

你也可以把 Temporal 平台理解为：一套通常称为 [Temporal Service](/temporal-service) 的调度与管理软件，加上以 Worker 进程形式运行的应用代码。
两者协同，为你的应用提供可靠的执行环境。

一个 Temporal Service 由用 Go 编写的 [Temporal Server](https://github.com/temporalio/temporal) 和数据库组成。

## 什么是 Temporal Application？ {#temporal-application}

Temporal Application 是一组 [Temporal Workflow Execution](/workflow-execution) 的集合。
每个 Temporal Workflow Execution 都拥有独占的本地状态，与其他 Workflow Execution 并发执行，并通过消息传递与其他 Workflow Execution 及外部环境通信。

一个 Temporal Application 可以包含数百万乃至数十亿个 Workflow Execution。
Workflow Execution 是轻量级的。
单个 Workflow Execution 占用的计算资源很少；实际上，当它处于等待状态而被挂起时，几乎不会消耗任何计算资源。

**可重入进程（Reentrant Process）**

Temporal Workflow Execution 本质上是一种可重入进程。可重入进程具备以下特征：可恢复、可继续、可响应。

- 可继续：当执行因某个 _awaitable_ 被挂起后，进程仍可继续执行。
- 可恢复：当执行因某个 _failure_ 被中断后，进程仍可恢复继续执行。
- 可响应：进程能够对外部事件作出反应。

因此，Temporal Workflow Execution 会执行一个 [Temporal Workflow Definition](/workflow-definition)（也叫 Temporal Workflow Function），也就是你的应用代码，并且做到**准确地执行并最终完成**——无论这段代码运行几秒还是几年，也无论它面对的是任意规模的负载还是任意类型的故障。

## 什么是 Failure？ {#failure}

[Temporal Failures](/references/failures) 是对系统中各种错误类型的统一表示形式（体现在各 SDK 和 Event History 中）。

Failure 处理是开发中的关键组成部分。
如果你想进一步了解应用层故障与平台层故障的区别，可以阅读 [Handling Failure From First Principles](https://dominik-tornow.medium.com/handling-failures-from-first-principles-1ed976b1b869)。
如果想了解这些概念在 Temporal 中如何落地实践，可以参考 [Failure Handling in Practice](https://temporal.io/blog/failure-handling-in-practice)。

对于会抛出错误（或异常）的语言来说：
如果在 Workflow 中抛出的错误不是 Temporal Failure，那么失败的是 Workflow Task（并且该 Task 会不断重试直到成功）；
而如果抛出的是 Temporal Failure（或者让某个来自 Temporal 调用的 Temporal Failure 向外传播，例如 Activity 调用返回的 [Activity Failure](/references/failures#activity-failure)），那么失败的就是整个 Workflow Execution。

更多信息请参见 [Application Failure](/references/failures#application-failure)。
