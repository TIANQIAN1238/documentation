<a id="key-management"></a>
## 什么是密钥管理？ {#key-management}

密钥管理是使用加密密钥的基本部分。

生成和轮换密钥涉及许多计算和逻辑方面，这通常需要堆栈中的专用应用程序。以下是使用 Temporal 应用程序加密密钥的一些一般建议：

- [对称加密](https://en.wikipedia.org/wiki/Symmetric-key_algorithm) 通常比非对称速度更快，并且产生的有效载荷更小。通常，非对称加密的优点是它允许您单独分发加密和解密密钥，但根据您的基础设施，这可能不会为 Temporal 提供任何安全优势。

- 基于AES的算法有[Go 中的硬件加速](https://pkg.go.dev/crypto/aes)等语言。 AES 算法经过广泛审查和信任，有许多不同的变体可以满足您的要求。使用 `ALG_AES_256_GCM_HKDF_SHA512_COMMIT_KEY` 的负载测试表现良好。

- 以与存储密码、配置详细信息和其他敏感数据相同的方式存储加密密钥。如果可能，请将密钥加载到您的应用程序中，这样您就不需要进行网络调用来检索它。尽可能为每个环境或 Namespace 提供单独的密钥。

- 确保您制定了密钥轮换策略，以防您的密钥遭到泄露或因其他原因需要更换。考虑使用专用秘密引擎或密钥管理系统 (KMS)。请注意，当您轮换密钥时，您可能还需要保留 Query 旧 Workflow 的旧密钥。

<a id="key-rotation"></a>
### 按键轮换 {#key-rotation}

美国国家标准与技术研究所 (NIST) 指南建议定期轮换加密密钥。对于 AES-GCM 密钥，应遵循 NIST 出版物 800-38D 的指南，在密钥版本执行大约 2^32 次加密之前进行轮换。

建议操作员估计密钥的加密率，并使用它来确定防止达到指导限制的轮换频率。例如，如果确定估计速率为每天 4000 万次操作，那么每三个月轮换一次密钥就足够了。

密钥轮换通常对于 Temporal Data Converter 实现来说应该是透明的。 Temporal 的 `Encode()` 和 `Decode()` 步骤只需要按预期触发，并且 Temporal 不知道您如何或何时生成加密密钥。

您应该设计编码和解码步骤以接受密钥管理的所有必要参数，例如密钥版本以及有效负载。与 Data Converter 一样，键应映射到 Temporal 中的 Namespace。
