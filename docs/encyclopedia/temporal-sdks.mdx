Temporal SDK（软件开发套件）是支持 Temporal 应用程序开发的工具、库和 API 的开源集合。

他们提供 [Temporal 客户端](#temporal-client) 与 [Temporal 服务](/temporal-service) 交互、用于开发 [Temporal 应用程序](#temporal-application) 的 API 以及用于运行水平可扩展的 [Workers](/workers#worker) 的 API。

然而，SDK 不仅仅是一个开发工具。
SDK API 使开发人员能够以反映现实世界流程的特定模式编写代码。
SDK 的内部实现与 Temporal 服务协作，逐步执行该代码，保证应用程序运行时的执行进度。

## Temporal应用 {#temporal-application}

Temporal 应用程序是您编写的代码，由 [Workflow Definitions](/workflow-definition)、[Activity Definitions](/workflow-definition)、用于配置 [Temporal 客户端](#temporal-client) 的代码以及用于配置和启动 [Workers](/workers#worker) 的代码组成。
开发人员使用[官方 Temporal SDK](#official-sdks) 创建 Temporal 应用程序。

考虑到 Workflow Definition 代码可以重复执行。
Temporal平台可以同时支持数百万到数十亿个Workflow Execution，每个Workflow Execution代表一个被调用的Workflow Definition。

此外，Temporal Workflow Execution 既可恢复又可恢复，并且可以对外部事件做出反应。

- 可恢复：进程在 _awaitable_ 上挂起后恢复执行的能力。
- 可恢复：进程在因失败而挂起后恢复执行的能力。
- 反应性：流程响应外部事件的能力。

因此，Temporal 应用程序可以在任意负载和故障的情况下运行数秒或数年。

## 为什么使用Temporal SDK？ {#why-use-an-sdk}

Temporal SDK 使开发人员能够专注于创建可靠且可扩展的业务逻辑，从而无需构建本地监控系统以确保可靠性和容错能力。这是可能的，因为 Temporal SDK 提供了一个统一的库，抽象了 Temporal 如何处理分布式系统的复杂性。

### 发展格局

通过抽象复杂性和简化样板代码，开发人员可以编写直接与其业务逻辑一致的简单代码，从而增强代码可读性并提高开发人员的工作效率。

考虑银行贷款申请。
开发者可以使用Temporal SDK设计银行贷款的业务逻辑。
Workflow 将总体业务逻辑定义为活动，包括验证申请人信息、信用检查、贷款审批和申请人通知等任务。

:::caution Do not copy and use code

以下是伪代码。有关测试示例，请参阅您的语言 SDK 的开发人员指南。

:::

```
func LoanApplicationWorkflow {

    sdk.ExecuteActivity(CreditCheck)

    sdk.ExecuteActivity(AutomatedApproval)

    sdk.ExecuteActivity(NotifyApplicant)

    // ...
}
```

例如，Temporal SDK 内置了对处理失败、超时和重试的支持。
如果 Activity 失败，SDK 将根据开发人员在 SDK 内建立的可配置策略自动启动重试。这种简化的流程简化了容错机制到应用程序的集成。

:::caution Do not copy and use code

以下是伪代码。有关测试示例，请参阅您的语言 SDK 的开发人员指南。

:::

```
func LoanApplicationWorkflow {

    options = {
        MaxAttempts: 3,
        StartToCloseTimeout: 30min,
        HeartbeatTimeout: 10min,
    }

    sdk.ExecuteActivity(CreditCheck, options)

    sdk.ExecuteActivity(AutomatedApproval)

    sdk.ExecuteActivity(NotifyApplicant)

    // ...
}
```

Temporal 客户端充当您的应用程序和 Temporal 服务之间通信的桥梁。

```go
// LoanApplicationWorkflow is the workflow definition.
func LoanApplicationWorkflow(ctx workflow.Context, applicantName string, loanAmount int) (string, error) {
	// Step 1: Notify the applicant that the application process has started
	err := workflow.ExecuteActivity(ctx, NotifyApplicantActivity, applicantName, "Application process started").Get(ctx, nil)
	if err != nil {
		return "", err
	}

	// Step 2: Perform a credit check
	var creditCheckResult string
	err = workflow.ExecuteActivity(ctx, LoanCreditCheckActivity, loanAmount).Get(ctx, &creditCheckResult)
	if err != nil {
		return "", err
	}

	// Step 3: Perform an automatic approval check
	var approvalCheckResult string
	err = workflow.ExecuteActivity(ctx, AutomaticApprovalCheckActivity, creditCheckResult).Get(ctx, &approvalCheckResult)
	if err != nil {
		return "", err
	}

	// Step 4: Notify the applicant of the decision
	var notificationResult string
	err = workflow.ExecuteActivity(ctx, NotifyApplicantActivity, applicantName, approvalCheckResult).Get(ctx, &notificationResult)
	if err != nil {
		return "", err
	}

	return notificationResult, nil
}
```

此外，Workflow 的状态会自动保留，因此如果发生故障，它会从中断处恢复。

Worker 具有水平可扩展性，通常与 Kubernetes 等系统一起运行，并根据应用程序的需求进行配置。

### 用于自定义 Activity 执行行为的 API

默认情况下，如果 Activity 尝试与另一个系统通信并遇到暂时性故障（例如网络问题），Temporal 将确保自动再次尝试 Activity。

## SDK与Temporal服务关系 {#sdk-and-cluster-relationship}

**Temporal SDK如何与Temporal服务配合使用？**

Temporal服务更像是编舞者而不是指挥者。 Temporal 服务不是直接将任务分配给 Worker，而是将任务安排到 Task Queue，同时 Worker 轮询 Task Queue。开发人员可以创建一组 Worker 并对其进行调整，以便任务一可用就被拾取。如果 Worker 发生故障，任务可以等待，直到下一个 Worker 可用。

Workflow 可能请求执行 Activity、启动 Timer 或启动 Child Workflow，其中每个都转换为命令，分派到 Temporal 服务。
除了执行这些命令之外，Temporal 服务还通过将相应的事件附加到 Workflow Execution 的 Event History 中来记录交互。

以执行 Activity 的调用为例。当 Workflow 调用它时，Worker 不会立即执行该 Activity 代码。相反，它会生成 ScheduleActivityTask 命令，并将其分派到集群。作为响应，集群将新的 Activity 任务排队。仅当 Worker 发现自己空闲时，它才会收集任务并开始执行 Activity 代码。

Temporal 服务会保留 Workflow Execution Event History，因此如果发生故障，SDK Worker 能够重播执行并从中断处恢复。

这就是 Workflow 代码的确定性约束发挥作用的地方，需要使用活动来创建副作用并与外界交互。

让我们看一下带有单个 Activity 的 Workflow 示例。

```go
func LoanApplication(ctx workflow.Context, input *LoanApplicationWorkflowInput) (*LoanApplicationWorkflowResult, error) {

	ctx = workflow.WithActivityOptions(ctx, workflow.ActivityOptions{
		StartToCloseTimeout: time.Minute,
	})

	var result activities.NotifyApplicantActivityResult
	f := workflow.ExecuteActivity(ctx, a.NotifyApplicantActivity, NotifyApplicantActivityInput(*input))

	err := f.Get(ctx, &result)

	// Return the results
	return &l.LoanApplicationState, nil
}

type Activities struct {}

func (a *Activities) NotifyApplicantActivity(ctx context.Context, input *NotifyApplicantActivityInput) (*NotifyApplicantActivityResult, error) {
	var result NotifyApplicantActivityResult

	// Call the thirdparty API and handle the result

	return &result, err
}
```

上面的 Activity 正在执行对外部 API 的单个调用。由于调用可能会因暂时性问题而失败，因此我们将其定义在 Workflow 之外，并为其提供重试选项。

当您创建新的 Worker 进程时，Worker 会创建与 Temporal 服务的持久连接，轮询 Task Queue 以获得与其能够执行的代码相关的任务。

<CaptionedImage
src="/diagrams/how-sdk-works-1.svg"
title="Worker 任务长轮询" />

尽管 Worker 现在正在运行，但除非显式启动 Workflow，否则 Task Queue 上没有任何任务，因此不会执行任何代码。
我们可以使用 Temporal 客户端（在 Temporal SDK 和 Temporal CLI 中提供）来启动新的 Workflow。

<CaptionedImage
src="/diagrams/how-sdk-works-2.svg"
title="使用 Temporal 客户端启动 Workflow" />

启动 Workflow Execution 会创建一个新事件 WorkflowExecutionStarted，并将其添加到 Workflow Execution 的 Event History 中。

然后，Temporal 服务通过将 Workflow 任务添加到 Task Queue 来调度 Workflow 任务。
当Worker有容量时，它会拿起这个任务，并开始执行代码。

任务的每个步骤（例如 Scheduled、开始和完成）都会记录到 Event History 中。

- Scheduled 表示 Temporal 服务已向 Task Queue 添加了任务。
- Started 意味着 Worker 已使任务出列。
- 完成意味着Worker通过响应Temporal服务完成了任务的执行。

当评估调用 Activity 的调用时，Worker 暂停执行代码并向 Temporal 服务发送命令以调度 Activity 任务。

<CaptionedImage
src="/diagrams/how-sdk-works-3.svg"
title="Worker 暂停代码执行并向 Temporal 服务发送命令" />

当 Worker 进程可以执行更多工作时，它会拿起 Activity 任务并开始执行 Activity 代码，其中包括对外部 API 的调用。

如果 Activity 失败，比如 API 出现故障，Temporal 将自动重试 Activity，间隔一秒（如配置所定义），无限次，直到 Activity 成功或被取消。

在调用成功并且代码完成的情况下，Worker 告诉 Temporal 服务 Activity 任务已完成。

<CaptionedImage
src="/diagrams/how-sdk-works-activity.svg"
title="Worker 报告 Activity 执行已完成" />

其中包括从 Activity 返回的任何数据（API 调用的结果），这些数据随后保留在 Workflow Execution Event History 中，并且现在可供 Workflow 代码访问。

Temporal 服务创建一个新的 Workflow 任务，Worker 将拾取该任务。

<CaptionedImage
src="/diagrams/how-sdk-works-1.svg"
title="Worker 领取新任务" />

此时，SDK Worker 会重放 Workflow 代码，并使用 Event History 作为预期的指导。如果重放遇到与代码预期不匹配的事件，则会引发[非确定性](/references/errors#non-deterministic-error)错误。

如果存在对齐，则 Worker 继续评估代码。

假设 Activity 执行成功，Workflow 现在具有 Activity 的结果，并且 Worker 能够完成评估和执行 Workflow 代码，并在完成时响应 Temporal 服务。

现在可以使用 Temporal 客户端检索 Workflow 的结果。

<CaptionedImage
src="/diagrams/how-sdk-works-4.svg"
title="Temporal 客户端现在可以访问 Workflow 的结果" />

这就是 Temporal Worker 和 Temporal 服务协同工作的方式。
