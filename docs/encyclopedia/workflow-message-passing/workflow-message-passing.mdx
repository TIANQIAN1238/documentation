Workflow 可以被视为可以接收消息的有状态 Web 服务。 Workflow可以有强大的信息
类似于端点的处理程序，结合 Workflow 的当前状态对传入消息做出反应。
Temporal 支持三种类型的消息：Signal、查询和 Update：

- 查询是读取请求。他们可以读取 Workflow 的当前状态，但不能阻止这样做。
- Signal 是异步写入请求。它们会导致正在运行的 Workflow 发生变化，但您无法等待任何响应
  或错误。
- Update 是同步、跟踪的写入请求。 Update 的发送者可以等待完成响应或
  失败时出错。

<a id="choosing-messages"></a>
## 作为 Workflow 作者，如何在 Signal、Update 和查询之间进行选择？ {#choosing-messages}

本节将帮助您编写接收消息的 Workflow。

<a id="for-write-requests"></a>
### 对于写请求 {#for-write-requests}

与 Signal 不同，Update 必须同步，并且必须等待运行 Workflow 的 Worker 确认
请求。

下表比较了何时使用 **Signals** 与 **Updates**。

| **需求类型**           | **当...时使用 Signal**                                                                     | **当...时使用 Update**                                                                                                                                                                    |
| ------------------------------ | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **异步通信** | 客户端希望在发送异步消息后快速继续。                      | 客户希望跟踪消息的完成情况。                                                                                                                                       |
| **结果处理**            | 客户可以接受“一劳永逸”的方式——不需要结果或例外。                    | 客户端需要结果或异常而不执行 Query。                                                                                                                             |
| **Worker 可用性**        | 客户端不依赖于 Worker 是否可用。                                         | 您希望在将 Update 接受到 Workflow 及其历史记录之前对其进行验证。                                                                                                     |
| **并发和吞吐量** | 您不想限制单个 Workflow 同时处理的消息数量。 | 每个 Workflow 不需要比 [云](/cloud/limits#per-workflow-execution-update-limits) 或 [自托管](/self-hosted-guide/defaults) 允许的限制更多的并发 Update。 |
| **延迟灵敏度**        | 由于客户并不期望得到结果，因此在使用 Signal 时，延迟通常并不重要。      | 客户需要低延迟的端到端操作，并愿意等待完成或验证。                                                                                      |

<a id="for-read-requests"></a>
### 对于读取请求 {#for-read-requests}

您通常想要执行 Query，因为：

- 查询是高效的——它们从不向 [Workflow Event History](/workflow-execution/event#event-history) 添加条目，
  而 Update 会（如果接受）。
- 查询可以对已完成的 Workflow 进行操作。

但是，由于查询无法阻塞，有时 Update 是最好的。当您的目标是阅读一次 Workflow 时
达到某种期望的状态，你有两种选择：

- 您可以使用查询定期轮询，直到 Workflow 准备就绪。
- 您可以将读取操作编写为 Update，这将为您提供更好的效率和延迟，尽管它会
  将条目写入 [Workflow Event History](/workflow-execution/event#event-history)。

<a id="for-readwrite-requests"></a>
### 用于读/写请求 {#for-readwrite-requests}

使用 Update 进行同步读/写请求。如果您的请求必须是异步的，请考虑发送 Signal
然后使用 Query 进行轮询。
