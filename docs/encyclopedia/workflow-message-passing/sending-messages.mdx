<a id="sending-updates"></a>
### 发送 Updates {#sending-updates}

:::note

要在 v1.25.0 之前的版本中使用 Workflow Update 功能，必须手动启用该功能。

将 [frontend.enableUpdateWorkflow执行](https://github.com/temporalio/temporal/blob/main/common/dynamicconfig/constants.go) 和 [frontend.enableUpdateWorkflowExecutionAsyncAccepted](https://github.com/temporalio/temporal/blob/main/common/dynamicconfig/constants.go) 动态配置值设置为 `true`。

例如，对于 Temporal CLI，运行以下命令：

```command
temporal server start-dev --dynamic-config-value frontend.enableUpdateWorkflowExecution=true
temporal server start-dev --dynamic-config-value frontend.enableUpdateWorkflowExecutionAsyncAccepted=true
```

:::

Update 可以从 Temporal CLIent 或 Temporal CLI 发送到 Workflow Execution。此调用是同步的，并将调用相应的 Update 处理程序。如果您希望发出异步请求，则应该使用 Signals。

在大多数语言（Go 除外）中，您可以调用 `executeUpdate` 来完成 Update 并获取其结果。

或者，要启动 Update，您可以调用 `startUpdate` 并传入 Workflow Update 阶段作为参数。对于等待什么，您有两种选择：

- 已接受 - 等待联系 Worker，这可确保 Update 持续存在。请参阅 [Update 验证器](/handling-messages#update-validators) 了解更多信息。
- 已完成 - 等待处理程序完成并返回结果。 （这相当于 `executeUpdate`。）

开始调用将为您提供一个句柄，您可以使用它来跟踪 Update，确定它是否被接受，并最终获取其结果或错误。

如果您想从 Workflow 中将 Update 发送到另一个 Workflow（例如 Child Workflow），您应该在 Activity 中执行此操作，并照常使用 Temporal CLIent。

Workflow Execution 运行期间可能发生的 Update 总数以及 Workflow Execution 可能具有的并发进行中的 Update 数量也有限制。
使用 [Update 验证器](/handling-messages#update-validators) 和 [Update ID](/handling-messages#exactly-once-message-processing) 可以使 [云](/cloud/limits#per-workflow-execution-update-limits) 和 [自托管](/self-hosted-guide/defaults) 保持在系统限制范围内。

<a id="update-with-start"></a>
#### Update-带启动 {#update-with-start}

:::tip

对于开源服务器用户，建议使用Temporal Server版本[Temporal Server 版本 1.28](https://github.com/temporalio/temporal/releases/tag/v1.28.0)。

:::

Update-with-Start 发送 Update 请求，并在必要时启动 Workflow。
必须指定 [`WorkflowIDConflictPolicy`](https://docs.temporal.io/workflow-execution/workflowid-runid#workflow-id-conflict-policy)。
Workflow ID 和 Update ID 可以用作幂等键，如下所示：

- 如果 Workflow 存在并且您提供了 Update ID，并且 Update 存在于最新的 Workflow 运行中，则 Update-With-Start 将附加到现有的 Update（无论 `WorkflowIDConflictPolicy` 是什么）
  - 如果 Workflow 已关闭，则仅当 Update 已完成时才会附加。
- 否则，它像往常一样使用[`WorkflowIDConflictPolicy`](https://docs.temporal.io/workflow-execution/workflowid-runid#workflow-id-conflict-policy)和[`WorkflowIDReusePolicy`](https://docs.temporal.io/workflow-execution/workflowid-runid#workflow-id-reuse-policy)来确定是否启动Workflow，然后立即启动新的Update。

Update-With-Start 非常适合延迟敏感的用例：

- **延迟初始化** -
  Update-With-Start 允许您在一次往返中将它们一起发送，而不是单独进行 Start Workflow 和 Update Workflow 调用。
  例如，可以使用 Update-With-Start 对购物车进行建模。
  Update 允许您在购物车中添加和删除商品。
  Update-With-Start 让客户开始购物，无论购物车已经存在还是刚刚开始购物。
  它确保在应用任何更改购物车内项目状态的 Update 之前，由 Workflow Execution 建模的购物车存在。
  对于此模式，将 `WorkflowIDConflictPolicy` 设置为 `USE_EXISTING`。
- **提前返回** -
  使用 Update-With-Start，您可以开始新的 Workflow Execution 并同步接收响应，同时 Workflow Execution 继续运行直至完成。
  例如，您可以使用 Update-With-Start 为付款流程建模。
  这允许您将支付验证结果同步发送回客户端，而交易 Workflow 在后台继续进行。
  如果您想断言它不会重用现有的 Workflow，请将 `WorkflowIDConflictPolicy` 设置为 `FAIL` 并为此模式使用唯一的 Update ID。

:::caution

与 Signal-with-Start 不同 - Update-With-Start _不是_原子的。
例如，如果 Update 无法交付，因为没有可用的正在运行的 Worker，则新的 Workflow Execution 仍将启动。
SDK 将重试 Update-With-Start 请求，但不能保证 Update 会成功。

:::

您还可以发送内置的“Stack Trace Query”进行调试。

<a id="stack-trace-query"></a>
#### 堆栈跟踪 Query {#stack-trace-query}

在许多 SDK 中，Temporal CLIent 公开预定义的 `__stack_trace` Query，它返回该 Workflow Execution 拥有的所有线程的调用堆栈。
这是在生产中对 Workflow Execution 进行故障排除的好方法。
例如，如果 Workflow Execution 停留在某个状态的时间超过预期时间，您可以发送 `__stack_trace` Query 以返回当前调用堆栈。
`__stack_trace` Query 名称不需要在 Workflow 代码中进行特殊处理。

:::note

堆栈跟踪查询仅适用于运行 Workflow Execution。

:::
