:::tip SUPPORT, STABILITY, and DEPENDENCY INFO

[Temporal Cloud](/evaluate/development-production-features/release-stages#general-availability) 和 [自托管部署](/cloud/nexus) 的 Temporal Nexus 现在为 [一般可用](/production-deployment/self-hosted-guide/nexus)。

:::

构建和部署 [Nexus 服务](/nexus/services) 有两种常见模式： 
- **[搭配图案](#collocated-pattern)**：在与现有 Workflow 和活动相同的 Worker 上运行。默认使用。
- **[路由器模式](#router-queue-pattern)**：将 Nexus 路由与 Workflow Execution 分开。 “路由器”Task Queue 上的专用 Nexus Worker 将操作路由到其他 Task Queue 上的 Workflow。 
当您需要独立扩展、每个 Worker 队列不同的 IAM 权限，或者想要添加 Nexus 而不修改现有 Worker 时，请使用。

## 并置模式（默认） {#collocated-pattern}

**并置模式**在与底层 Workflow 相同的 Worker 和相同 Task Queue 上运行 Nexus 操作处理程序。
这是默认且最简单的部署。

Nexus 端点的目标与底层 Workflow 使用的 Task Queue 相同。
单个 Worker 注册 Nexus 服务和 Workflow 类型，因此一切都一起运行。

<a id="why-start-here"></a>
### 为什么从这里开始 {#why-start-here}

- **最简单的设置：** 一台 Worker，一台 Task Queue，一次部署。没有额外的基础设施。
- **Eager Workflow Start：** 当处理程序在同一 Worker 中启动 Workflow 时，您可以使用 [渴望Workflow开始](/develop/worker-performance#eager-workflow-start) 在本地执行第一个 Workflow 任务，而无需额外调用 Temporal Server - 同时仍然记录持久状态。如果进程崩溃，Workflow 将在另一个 Worker 上恢复。
- **干净的外观：** 运营充当稳定的合同。您可以更改底层实现（今天为 Signal，明天为 Workflow），而不会影响调用者。

<a id="when-to-use-this-pattern"></a>
### 何时使用此模式 {#when-to-use-this-pattern}

- Nexus 入门。
- 同一团队拥有 Nexus 服务和基础 Workflow。
- 您不需要将 Nexus 路由与 Workflow Execution 分开缩放。
- 您正在设置一个简单的测试环境

默认情况下使用此模式，除非您有充分的理由使用下面的路由器队列模式

<a id="router-queue-pattern"></a>
## 路由器队列模式 {#router-queue-pattern}

**路由器队列模式**将 Nexus 路由与 Workflow Execution 分开。 “路由器”Task Queue 上的专用 Nexus Worker 将操作路由到同一 Namespace 中其他 Task Queue 上的 Workflow。

<a id="when-to-use-this-pattern"></a>
### 何时使用此模式 {#when-to-use-this-pattern}

- **单独缩放：** 独立于 Workflow Execution 缩放 Nexus 路由。
- **专用路由层：** 单个 Nexus Worker 将请求路由到不同 Task Queue 上的多个 Workflow 类型。
- **不同的 IAM 权限：** 不同 Task Queue 后面的 Worker 队列可能对不同的底层资源具有不同的 IAM 权限。
- **避免修改现有的 Worker：** 将路由器 Worker 添加到 Namespace，而不更改任何现有的 Worker 或 Workflow。

<a id="how-it-works"></a>
### 它是如何工作的 {#how-it-works}

1. 注册一个 Nexus Worker 来轮询专用“路由器”Task Queue。
2. 配置 Nexus 端点的目标 Task Queue 以指向此路由器 Task Queue。
3. 在每个 Nexus 操作处理程序中，在 Workflow 启动选项中指定不同的目标 Task Queue。
4. 现有的Worker继续轮询自己的Task Queue并执行路由器启动的Workflow。

<a id="production-usage"></a>
### 生产使用 {#production-usage}

由运行自助服务平台的组织在生产中使用，其中中央网关将请求路由到特定于域的 Namespace 和 Task Queue。
路由器 Worker 是轻量级的 - 它只处理路由逻辑。
