## 什么是多集群复制？ {#multi-cluster-replication}

多集群复制是一种将 Workflow Execution 从主动集群异步复制到其他被动集群的功能，用于备份和状态重建。
必要时，为了获得更高的可用性，集群操作员可以故障转移到任何备份集群。

Temporal 的多集群复制功能被认为是**实验性**，不受正常 [版本控制和支持政策](/temporal-service/temporal-server#versions-and-support) 的约束。

Temporal 自动将 Start、Signal 和 Query 请求转发到活动集群。
此功能必须通过每个 [全球 Namespace](/global-namespace) 的动态配置标志来启用。

启用该功能后，任务将发送到与 Namespace 匹配的父 Task Queue 分区（如果存在）。

所有 Visibility API 均可用于活动和备用集群。
这使得 [Temporal 用户界面](https://docs.temporal.io/web-ui) 能够与 Global Namespace 无缝工作。
即使全局 Namespace 处于待机模式，直接对 Temporal Visibility API 进行 API 调用的应用程序也可以继续工作。
但是，当从备用集群 Query 调整 Workflow Execution 状态时，他们可能会因复制延迟而出现滞后。

<a id="namespace-versions"></a>
## Namespace 版本 {#namespace-versions}

_version_ 是多集群复制中的一个概念，描述每个 Namespace 事件的时间顺序。

通过多集群复制，所有 Namespace 更改事件和 Workflow Execution 历史事件都会异步复制，以实现高吞吐量。
这意味着跨集群的数据**不**强一致。
为了保证Namespace数据和Workflow Execution数据达到最终一致性（特别是在故障转移期间存在数据冲突时），引入了**版本**并附加到Namespace。
在 Namespace 中生成的所有 Workflow Execution 历史记录条目也将附带附加到该 Namespace 的版本。

所有参与的集群都预先配置了唯一的初始版本和共享版本增量：

- `initial version < shared version increment`

当对 Namespace 从一个集群执行故障转移到另一个集群时，附加到 Namespace 的版本将按以下规则更改：

- 对于 `version % (shared version increment) == (active cluster's initial version)` 之后的所有版本，找到具有 `version >= old version in Namespace` 的最小版本

当存在数据冲突时，将进行比较，并且具有最高版本的Workflow Execution历史记录条目将被认为是事实来源。

当集群尝试更改 Workflow Execution 历史记录时，将检查版本。
仅当满足以下条件时，集群才能改变 Workflow Execution 历史记录：

- Namespace 中的版本属于该集群，即
  `(version in Namespace) % (shared version increment) == (this cluster's initial version)`
- 此 Workflow Execution 历史记录的最后一个条目（事件）的版本等于或小于 Namespace 中的版本，即
  `(last event's version) <= (version in Namespace)`

- 共享版本增量：10

## 冲突解决 {#conflict-resolution}

当 Workflow Execution 历史出现分歧时，将应用适当的冲突解决方案。

在多集群复制中，Workflow Execution 历史事件被建模为树，如 [版本历史](#version-history) 中的第二个示例所示。

Workflow Execution 分歧的历史将有多个历史分支。
在所有历史分支中，版本最高的历史分支被认为是 `current branch`，而 Workflow Execution 的可变状态是当前分支的摘要。
每当 Workflow Execution 历史分支之间发生切换时，就会发生 Workflow Execution 可变状态的完全重建。

Temporal 多集群复制依赖于跨集群的事件异步复制，因此在故障转移的情况下，由于复制任务滞后，可能会将 Activity 任务再次分派到新​​的活动集群。
这也意味着每当新集群进行故障转移后 Workflow Execution 变为 Updated 时，该执行的任何先前的复制任务都无法应用。
这会导致 Workflow Execution 在先前活动集群中取得的一些进度丢失。
在此类冲突解决期间，Temporal 会在丢弃复制任务之前将任何外部事件（如 Signal）重新注入到新的 Event History 中。
尽管某些进度可能会在故障转移期间回滚，但 Temporal 可以保证 Workflow Execution 不会陷入困境，并将继续取得进展。

Activity 执行完成不会跨集群转发。
根据配置，任何未完成的活动最终都会超时。
您的应用程序应该具有重试逻辑，以便在故障转移到新集群后重试 Activity 并再次分派到 Worker。
处理此问题类似于处理由 Worker 重新启动导致的 Activity 任务超时。

<a id="zombie-workflows"></a>
## 僵尸 Workflow {#zombie-workflows}

有一个现有合约，对于任何 Namespace 和 Workflow Id 组合，最多可以有一个运行 (Namespace + Workflow Id + Run Id) 打开/执行。

多集群复制旨在使所有参与集群中的 Workflow Execution 历史记录尽可能保持最新。

由于多集群复制的性质（例如，Workflow Execution 历史事件是异步复制的），不同的运行（相同的 Namespace 和 Workflow Id）可能会在不同时间到达目标集群，有时是无序的，如下所示：

```
+-----------+        +----------------+        +-----------+
| Cluster A |        | Network Layer  |        | Cluster B |
+-----------+        +----------------+        +-----------+
      |                       |                        |
      | Run 1 Replication     |                        |
      |---------------------> |                        |
      |                       |                        |
      | Run 2 Replication     |                        |
      |---------------------> |                        |
      |                       |                        |
      |                       | Run 2 Replication      |
      |                       |--------------------->  |
      |                       |                        |
      |                       | Run 1 Replication      |
      |                       |--------------------->  |
      |                       |                        |
```

由于 Run 2 首先出现在 Cluster B 中，因此由于规则 `at most one Run open`（见上文），Run 1 无法复制为“可运行”，因此引入了“僵尸”Workflow Execution 状态。
“僵尸”状态是指 Workflow Execution 不能被集群主动突变的状态（假设相应的 Namespace 在该集群中处于活动状态）。僵尸 Workflow Execution 只能由复制任务更改。

运行 1 的复制方式与运行 2 类似，但运行 1 的执行在运行 1 完成之前将变成“僵尸”。

<a id="workflow-task-processing"></a>
## Workflow 任务处理 {#workflow-task-processing}

在多集群复制的上下文中，Workflow Execution 的可变状态是跟踪所有待处理任务的实体。
在引入多集群复制之前，Workflow Execution 历史条目（事件）来自单个分支，并且 Temporal Server 只会将新条目（事件）追加到 Workflow Execution 历史记录中。

引入多集群复制后，一个Workflow Execution可能可以有多个Workflow Execution历史分支。
根据一个历史分支生成的任务可能会因在冲突解决期间切换历史分支而变得无效。

例子：

T = 0：根据Event Id：4，版本：2生成任务A

```
| -------- | ------------- |
| Events   |
| -------- | ------------- |
| Event ID | Event Version |
| -------- | ------------- |
| 1        | 1             |
| 2        | 1             |
| 3        | 2             |
| -------- | ------------- |
|          |
|          |
| -------- | ------------- |
| Event ID | Event Version |
| -------- | ------------- |
| 4        | 2             | <-- task A belongs to this event |
| -------- | ------------- |
```

T = 1：冲突解决发生，Workflow Execution 的可变状态被重建，历史事件 ID：4，版本：3 被写入持久化

```
| --------- | ------------- |
| Events    |
| --------- | ------------- |
| Event ID  | Event Version |
| --------  | ------------- |
| 1         | 1             |
| 2         | 1             |
| 3         | 2             |
| --------  | ------------- |
|           |
| --------- | ------------- |
|           |                
| --------  | ------------- |                                  | -------- | ------------- |
| Event ID  | Event Version |                                  | Event ID | Event Version |
| --------  | ------------- |                                  | -------- | ------------- |
| 4         | 2             | <-- task A belongs to this event | 4        | 3             | <-- current branch / mutable state
| --------  | ------------- |                                  | -------- | ------------- |
```

T = 2：任务 A 已加载。

此时，由于重建了Workflow Execution的可变状态（冲突解决），任务A不再相关（任务A对应的Event属于非当前分支）。
任务处理逻辑将根据相应的 Workflow Execution 的可变状态验证任务的事件 ID 和版本，然后丢弃任务 A。
