<a id="cluster-configuration"></a>
## Temporal 服务配置是什么？ {#cluster-configuration}

Temporal 服务配置是使用 YAML 定义的自托管 Temporal 服务的设置和配置详细信息。
设置自托管 Temporal 服务时，您必须定义 Temporal 服务配置。

有关使用 Temporal Cloud 的详细信息，请参阅 [Temporal Cloud 文档](/cloud)。

Temporal 服务配置由两种类型的配置组成：[静态配置](#static-configuration) 和 [动态配置](#dynamic-configuration)。

<a id="static-configuration"></a>
### 静态配置 {#static-configuration}

静态配置包含如何设置 Temporal 服务的详细信息。
静态配置仅读取一次并用于在启动时配置服务节点。
根据您想要部署自托管 Temporal 服务的方式，您的静态配置必须包含设置详细信息：

- Temporal 服务 — 前端、历史、匹配、Worker
- Temporal 服务的会员端口
- 持久性（包括历史分片计数）、Visibility、档案存储设置。
- TLS、身份验证、授权
- 服务器日志级别
- 指标
- Temporal 服务元数据
- 动态配置客户端

静态配置值无法在运行时更改。
某些值（例如指标配置或服务器日志级别）可以在静态配置中更改，但需要重新启动 Temporal 服务才能使更改生效。

有关静态配置密钥的详细信息，请参阅[Temporal 服务配置参考](/references/configuration)。

有关静态配置示例，请参阅 [https://github.com/temporalio/temporal/tree/master/config](https://github.com/temporalio/temporal/tree/master/config)。

<a id="dynamic-configuration"></a>
### 动态配置 {#dynamic-configuration}

动态配置包含您可以在 Temporal 服务设置中使用 Update 的配置密钥，而无需重新启动服务器进程。

Temporal 提供的所有动态配置密钥都具有 Temporal 服务使用的默认值。
您可以通过为 YAML 文件中的键设置不同的值并设置 [动态配置客户端](/references/configuration#dynamicconfigclient) 来轮询该文件中的 Update 来覆盖默认值。
为 Temporal 服务设置动态配置是可选的。

设置某些配置键 Update 的覆盖会立即覆盖 Temporal 服务配置。
但是，对于启动时检查的配置字段（例如线程池大小），您必须重新启动服务器才能使更改生效。

使用动态配置密钥来微调您自行部署的 Temporal 服务设置。

有关动态配置键的详细信息，请参阅[动态配置参考](/references/dynamic-configuration)。

有关动态配置示例，请参阅 [https://github.com/temporalio/temporal/tree/master/config/dynamicconfig](https://github.com/temporalio/temporal/tree/master/config/dynamicconfig)。

<a id="temporal-cluster-security-configuration"></a>
## Temporal 服务安全配置是什么？ {#temporal-cluster-security-configuration}

通过加密您的网络通信并设置 API 调用的身份验证和授权协议来保护您的 Temporal 服务（自托管和 Temporal Cloud）。

有关设置 Temporal 服务安全性的详细信息，请参阅 [Temporal 平台安全功能](/security)。

<a id="mtls-encryption"></a>
### mTLS 加密 {#mtls-encryption}

Temporal 支持相互传输层安全 (mTLS)，以加密 Temporal 服务内的服务之间或应用程序进程与 Temporal 服务之间的网络流量。

在自承载 Temporal 服务上，在 [Temporal 服务配置](/references/configuration#tls) 的 `tls` 部分中配置 mTLS。
mTLS 配置是 [静态配置](#static-configuration) 属性。

然后，您可以使用 [`WithConfig`](/references/server-options#withconfig) 或 [`WithConfigLoader`](/references/server-options#withconfigloader) 服务器选项通过此配置启动 Temporal 服务。

mTLS 配置包括两个部分，用于分隔 Temporal 服务内的通信以及从应用程序对 Temporal 服务进行的客户端调用。

- `internode`：用于加密 Temporal 服务内节点之间通信的配置。
- `frontend`：用于加密前端服务的公共端点的配置。

分别为 `internode` 和 `frontend` 设置 mTLS 可以让您使用不同的证书和设置来加密每个流量部分。

<a id="using-certificates-for-client-connections"></a>
### 使用证书进行客户端连接 {#using-certificates-for-client-connections}

使用 CA 证书对 Temporal 服务的客户端连接进行身份验证。

在 Temporal Cloud 上，您可以 [在 Temporal Cloud 设置中设置 CA 证书](/cloud/certificates) 并在客户端调用中使用最终实体证书。

在自承载 Temporal 服务上，您可以使用 Temporal 服务配置中的 `clientCAFiles` 或 `clientCAData` 属性以及 [`requireClientAuth`](/references/configuration#tls) 属性来限制对 Temporal 服务端点的访问。
这些属性可以在 [mTLS配置](/references/configuration#tls) 的 `internode` 和 `frontend` 部分中指定。
有关详细信息，请参阅 [tls配置参考](/references/configuration#tls)。

<a id="server-name-specification"></a>
### 服务器名称规范 {#server-name-specification}

在自托管 Temporal 服务上，您可以在 mTLS 配置的 `client` 部分中指定 `serverName` 以防止欺骗和 [中间人攻击](https://en.wikipedia.org/wiki/Man-in-the-middle_attack)。

输入 `serverName` 的值可启用已建立的连接以对端点进行身份验证。
这可确保向任何连接的客户端提供的服务器证书在其 CN 属性中具有指定的服务器名称。

此测量可用于 `internode` 和 `frontend` 端点。

有关 mTLS 配置的更多信息，请参阅 [tls配置参考](/references/configuration#tls)。

<a id="authentication-and-authorization"></a>
### 身份验证和授权 {#authentication-and-authorization}

{/* commenting this very generic explanation out. Can include it back in if everyone feels strongly.
**身份验证**是验证想要访问您的应用程序的用户实际上是您想要访问它的用户的过程。
**授权**是对 Temporal 服务或应用程序上的用户有权访问的应用程序和数据的验证。 */}

Temporal 提供身份验证接口，可设置该接口来限制对数据的访问。
这些协议涉及三个领域：服务器、客户端连接和用户。

Temporal 提供两个插件接口，用于 API 调用的身份验证和授权。

- [`ClaimMapper`](/self-hosted-guide/security#claim-mapper)
- [`Authorizer`](/self-hosted-guide/security#authorizer-plugin)

这两个插件的逻辑都可以定制以适应各种用例。
提供插件后，前端服务会在运行请求的操作之前调用其实现。

<a id="monitoring-and-observation"></a>
## Temporal 服务可观察性是什么？ {#monitoring-and-observation}

您可以使用自托管 Temporal 服务或 Temporal Cloud 发出的指标来监控和观察性能。

Temporal 默认以 Prometheus 支持的格式发出指标。
可以使用任何支持相同格式的度量软件。
目前，我们使用以下 Prometheus 和 Grafana 版本进行测试：

- **普罗米修斯 >= v2.0**
- **Grafana >= v2.5**

Temporal Cloud 通过 Prometheus HTTP API 端点发出指标，该端点可以直接用作 Grafana 中的 Prometheus 数据源或 Query 并将云指标导出到任何可观测平台。

有关云指标和设置的详细信息，请参阅以下内容：

- [Temporal Cloud 指标参考](/cloud/metrics/)
- [设置具有 Temporal Cloud 可观察性的 Grafana 以查看指标](/cloud/metrics/prometheus-grafana#grafana-data-source-configuration)

在自托管的 Temporal 服务上，在 Temporal 服务配置中公开 Prometheus 端点，并将 Prometheus 配置为从端点中抓取指标。
然后，您可以设置可观察平台（例如 Grafana）以使用 Prometheus 作为数据源。

有关自托管 Temporal 服务指标和设置的详细信息，请参阅以下内容：

- [Temporal 服务OSS指标参考](/references/cluster-metrics)
- [设置 Prometheus 和 Grafana 以查看 SDK 和自托管 Temporal 服务指标](/self-hosted-guide/monitoring)
