## What is a Search Attribute? {#search-attribute}

Search Attribute 是 [列表过滤器](/list-filter) 中使用的索引字段，用于过滤元数据中包含 Search Attribute 的 [Workflow Executions](/workflow-execution) 列表。

每个Search Attribute 都是包含在Workflow Execution 的Visibility 信息中的键值对元数据对象。
此信息可在 Visibility 商店中找到。

:::caution Do not use sensitive data or PII in Search Attributes

请勿在 Search Attribute **名称或值**中包含敏感数据、秘密或个人身份信息 (PII)。
Search Attribute 值未加密地存储在 Visibility 存储中，并且不由自定义 [有效负载编解码器](/payload-codec#payload-codec) 处理。
Temporal Server 必须能够以纯文本形式读取这些值以支持过滤和排序，因此在不破坏搜索功能的情况下不可能进行加密。

属性名称在 Namespace 配置、Query 表达式和 Temporal UI 中也可见。
在名称或值中使用敏感数据可能会导致任何具有 Namespace 访问权限的人面临暴露风险，并且可能违反 GDPR、HIPAA 或 SOC 2 等数据保护法规。

:::

Temporal 提供一些 [默认 Search Attribute](/search-attribute#default-search-attribute)，例如 `ExecutionStatus`，即 Workflow Execution 的当前状态。
您还可以在 Visibility 存储中创建 [定制Search Attribute](/search-attribute#custom-search-attribute) 密钥，并在启动 Workflow Execution 或在 Workflow 代码中分配值。

使用 [Continue-As-New](/workflow-execution/continue-as-new) 或 [Temporal 计划任务](/cron-job) 时，默认情况下，Search Attribute 键将转移到新的 Workflow 运行。
Search Attribute 值仅与 Workflow 一样长。

Search Attribute 对于搜索目的或需要基于集合的结果集的任务最有效。
对于需要获取有关 Workflow Execution 信息的业务逻辑，请考虑以下操作之一：

- 将状态存储在局部变量中并使用 Query 公开它。
- 通过活动将状态存储在外部数据存储中，并直接从存储中获取状态。

如果您的业务逻辑需要高吞吐量或低延迟，请通过“活动”存储和获取数据。
由于 Workflow 的状态更改和 Activity 更新 Visibility 存储之间的时间流逝，您可能会遇到延迟。

### 默认 Search Attributes {#default-search-attribute}

Temporal 服务具有一组可用的默认 Search Attribute。
默认 Search Attribute 在任何 Namespace 中全局设置。
这些 Search Attribute 是在创建初始索引时创建的。

| 姓名                               | 类型         | 定义                                                                                                                                                                                                   |
| ---------------------------------- | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 批处理用户                        | 关键词      | 由在 `TemporalBatcher` Namespace 分区中运行的内部批处理程序 Workflow 使用，指示启动批处理操作的用户。                                                                    |
| 二进制校验和                    | 关键词列表 | 运行 Workflow Execution 的 Worker 的二进制 ID 列表。自服务器版本 1.21 起已弃用，取而代之的是 `BuildIds` Search Attribute。                                                             |
| 构建ID                           | 关键词列表 | 已处理 Workflow Execution 的 Worker 构建 ID 列表，格式为 `versioned:{BuildId}` 或 `unversioned:{BuildId}`，或哨兵 `unversioned` 值。从服务器版本 1.21 开始可用。 |
| 关闭时间                          | 日期时间     | Workflow Execution 完成的时间。                                                                                                                                                          |
| 执行持续时间                  | INT          | 运行 Workflow Execution 所需的时间（以纳秒为单位）。仅适用于封闭式 Workflow。                                                                                                         |
| 执行状态                    | 关键词      | Workflow Execution 的当前状态。                                                                                                                                                                 |
| 执行时间                      | 日期时间     | Workflow Execution 实际开始运行的时间；大多数情况下与 `StartTime` 相同，但 Cron Workflow 和重试的 Workflow 不同。                                                 |
| 历史长度                      | INT          | Workflow Execution历史上的事件数量。仅适用于封闭式 Workflow。                                                                                                              |
| 历史大小字节                   | 长的         | Event History 的尺寸。                                                                                                                                                                               |
| 运行ID                              | 关键词      | 标识当前 Workflow Execution 运行。                                                                                                                                                               |
| 开始时间                          | 日期时间     | Workflow Execution 开始的时间。                                                                                                                                                            |
| 状态转换计数               | INT          | Workflow Execution 保持其状态的次数。仅适用于封闭式 Workflow。                                                                                                    |
| 任务队列                          | 关键词      | Task Queue 由 Workflow Execution 使用。                                                                                                                                                                       |
| Temporal更改版本              | 关键词列表 | 如果启用 GetVersion API，则存储更改/版本对。                                                                                                                                                |
| Temporal报告的问题           | 关键词列表 | 存储有关 Workflow 任务失败的信息。格式为 `category=<category> cause=<cause>`。
| TemporalScheduled开始时间         | 日期时间     | 根据 Schedule 规格，Workflow 是 Schedule 启动的时间。可以手动触发。设置在 Schedule 上。                                                                                 |
| TemporalScheduledById              | 关键词      | 启动 Workflow 的 Schedule 的 ID。                                                                                                                                                            |
| TemporalSchedule已暂停             | 布尔值      | 指示Schedule是否已暂停。设置在 Schedule 上。                                                                                                                                            |
| TemporalWorker部署           | 关键词      | 指示关联的 Worker 部署的名称。                                                                                                                                                      |
| TemporalWorker部署版本    | 关键词      | 指示关联的 Worker 部署的版本字符串，格式为 `<deployment name>:<build id>`。                                                                                              |
| TemporalWorkflow版本控制行为 | 关键词      | 指示关联的 Worker 版本控制行为（“固定”、“自动升级”，如果不使用 Worker 版本控制则为 null）。                                                                                      |
| WorkflowId                         | 关键词      | 标识 Workflow Execution。                                                                                                                                                                           |
| Workflow型                       | 关键词      | Workflow 的型号。                                                                                                                                                                                        |

- 所有默认的 Search Attribute 均保留且只读。
  您无法创建具有相同名称的自定义名称或更改现有名称。

- Search Attribute 在系统中未加密。
  请勿使用敏感数据或 PII 作为 Search Attribute 名称或值。

- 要在列表过滤器中使用带有 `Temporal` 前缀的默认 Search Attribute，您可以使用它们的无前缀别名。
  详细信息请参阅 [Search Attribute 别名](./list-filter.mdx#search-attribute-aliasing)。

- ExecutionStatus 值对应于 Workflow Execution 状态：正在运行、已完成、失败、已取消、已终止、ContinuedAsNew、TimedOut。

- StartTime、CloseTime 和 ExecutionTime 存储为日期，但受使用以纳秒为单位的 EpochTime 或 [RFC3339Nano格式](https://pkg.go.dev/time#pkg-constants) 中的字符串（例如“2006-01-02T15:04:05.999999999Z07:00”）的查询支持。

- ExecutionDuration 以纳秒为单位存储，但受使用纳秒、[Golang 持续时间格式](https://pkg.go.dev/time#ParseDuration) 或“hh:mm:ss”格式的整数的查询支持。

- CloseTime、HistoryLength、StateTransitionCount 和 ExecutionDuration 仅存在于关闭的 Workflow Execution 中。

- 在重试和 cron 用例中，ExecutionTime 可能与 StartTime 不同。

在以下条件下，您可以在列表过滤器中使用默认的 Search Attribute，例如在 Temporal Web UI 中或使用 `Temporal Workflow list` 命令：

- 如果没有高级 Visibility，您只能将 `=` 运算符与列表过滤器中的单个默认 Search Attribute 一起使用。
  例如：`Temporal Workflow list --Query "ExecutionStatus = 'Completed'"` 或 `Temporal Workflow list --Query "WorkflowType = 'YourWorkflow'"`。
- 使用高级 Visibility，您可以在列表过滤器中组合默认的 Search Attribute，以获得特定 Workflow Execution 的列表。
  例如：`Temporal Workflow list --Query "WorkflowType = 'main.YourWorkflowDefinition' and ExecutionStatus != 'Running' and (StartTime > '2022-06-07T16:46:34.236-08:00' or CloseTime < '2022-06-08T16:46:34-08:00')"`

### 自定义 Search Attributes {#custom-search-attribute}

您可以使用与您的业务需求相关的唯一键名称 [创建自定义 Search Attribute](/self-hosted-guide/visibility#create-custom-search-attributes)。

在以下条件下，在列表过滤器中使用自定义 Search Attribute，例如在 Temporal Web UI 中或使用 `Temporal Workflow list` 命令：

- 如果没有高级 Visibility，您无法在列表过滤器中使用自定义 Search Attribute。
- 使用高级 Visibility，您可以创建多个自定义 Search Attribute 并将它们与列表过滤器结合使用以获得特定 Workflow Execution 的列表。
  例如：`Temporal Workflow list --Query "WorkflowType = 'main.YourWorkflowDefinition' and YourCustomSA = 'YourCustomSAValue' and (StartTime > '2022-06-07T16:46:34.236-08:00' or CloseTime < '2022-06-08T16:46:34-08:00')"`
  - 对于 Temporal Server v1.19 及更早版本，您必须 [集成Elasticsearch](/self-hosted-guide/visibility#elasticsearch) 才能将自定义 Search Attribute 与列表过滤器一起使用。
  - 对于 Temporal Server v1.20 及更高版本，除了 Elasticsearch 之外，自定义 Search Attribute 功能还可在 MySQL（v8.0.17 或更高版本）、PostgreSQL（v12 及更高版本）和 SQLite（v3.31.0 及更高版本）上使用。

如果您使用 Elasticsearch 作为 Visibility 存储，则您的自定义 Search Attribute 全局适用，并且可以跨 Namespace 使用。
但是，如果将任何 [支持的 SQL 数据库](/self-hosted-guide/visibility) 与 Temporal Server v1.20 及更高版本一起使用，则您的自定义 Search Attribute 会与特定的 Namespace 关联，并且可用于该 Namespace 中的 Workflow Execution。

有关您可以创建的自定义 Search Attribute 的数量和大小的限制，请参阅 [自定义 Search Attribute 限制](/search-attribute#custom-search-attribute-limits)。

#### 支持的类型 {#supported-types}

自定义 Search Attribute 必须是以下类型之一：

- 布尔
- 日期时间
- 双人间
- 智力
- 关键字
- 关键词列表
- 文字

笔记：

- **Double** 由 `scaled_float` Elasticsearch 类型支持，比例因子为 10000（4 个十进制数字）。
- **日期时间** 在 Elasticsearch 6 中由毫秒精度的 `date` 类型和 Elasticsearch 7 中的纳秒精度的 `date_nanos` 类型支持。
- **Int** 是 64 位整数（`long` Elasticsearch 类型）。
- **关键字**和**文本**类型是取自 Elasticsearch 的概念。 **文本**中的每个单词都被视为可搜索的关键字。
  对于 UUID，这可能会出现问题，因为 Elasticsearch 会单独索引 UUID 的每个部分。
  要将整个字符串视为可搜索关键字，请使用 **Keyword** 类型。
  例如，如果密钥 `ProductId` 的值为 `2dd29ab7-2dd8-4668-83e0-89cae261cfb1`：
  - 作为 **关键字**，它只能与 `ProductId = "2dd29ab7-2dd8-4668-83e0-89cae261cfb1` 匹配。
  - 作为**文本**，它将与 `ProductId = 2dd8` 匹配，这可能会导致不需要的匹配。
- 对于 Temporal Server v1.19 及更早版本，**Keyword** 类型可以存储值列表。
- 对于 Temporal Server v1.20 及更高版本，**关键字** 类型仅支持单个值。
  要存储值列表，请使用 **KeywordList**。
- **Text** 类型不能在“Order By”子句中使用。

### 用法 {#usage}

Visibility 商店中提供的 Search Attribute 可以与 Workflow Execution 一起使用以获取 Temporal 服务。
要实际获得使用 [列表过滤器](/list-filter) 的结果，必须将 Search Attribute 作为元数据添加到 Workflow Execution。

- 要在 Visibility 商店中创建自定义 Search Attribute，请参阅 [创建自定义 Search Attribute](/self-hosted-guide/visibility#create-custom-search-attributes)。
- 要从 Visibility 存储中删除自定义 Search Attribute，请参阅 [删除自定义 Search Attribute](/self-hosted-guide/visibility#remove-custom-search-attributes)。
  Temporal Cloud 不支持删除自定义 Search Attribute。
- 要重命名 Temporal Cloud 上的自定义 Search Attribute，请参阅 [`tcld Namespace search-attributes rename`](/cloud/tcld/namespace/#rename)。

使用 Workflow，您可以执行以下操作：

- 在 Workflow 中设置 Search Attribute 的值
- Update 从 Workflow 代码中为 Search Attribute 设置的值
- 从 Workflow 代码中删除为 Search Attribute 设置的值
