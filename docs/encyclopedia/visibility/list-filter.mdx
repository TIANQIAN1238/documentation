<a id="list-filter"></a>
## 什么是列表过滤器？ {#list-filter}

[Visibility](/temporal-service/visibility) 列表 API 要求您提供列表过滤器作为类似 SQL 的字符串参数。

列表过滤器包括 [Search Attribute](/search-attribute) 名称、Search Attribute 值和 [运营商](#supported-operators)，以便它可以从 Visibility 存储中检索 Workflow Execution 的过滤列表。

列表过滤器 [Search Attribute](/search-attribute) 名称区分大小写。
每个列表过滤器都有一个 [Namespace](/namespaces) 范围。

使用时间范围的列表滤波器在 [弹性搜索](/self-hosted-guide/visibility#elasticsearch) 上提供 1 ns 的分辨率，在 [SQL数据库](/self-hosted-guide/visibility) 上提供 1 µs 的分辨率。

<a id="supported-operators"></a>
### 支持的运算符 {#supported-operators}

列表过滤器支持以下运算符：

- **`=, !=, >, >=, <, <=`**
- **`AND, OR, ()`**
- **`BETWEEN ... AND`**
- **`IN`**
- **STARTS_WITH**

:::note

Temporal Cloud 目前不支持 **ORDER BY** 运算符。

默认排序为：`ClosedTime DESC NULL FIRST`、`StartTime DESC`。 {/* `RunID DESC` 取决于使用哪个 Visibility 存储。 */}

`Text` 类型的自定义 Search Attribute 不能在 **ORDER BY** 子句中使用。

:::

<a id="partial-string-match"></a>
### 部分字符串匹配 {#partial-string-match}

当 Search Attribute 的类型为 [文本](#text) 与 [关键词](#keyword) 时，部分字符串匹配有不同的选项。

＃＃＃＃ 文本

`Text` 类型的 Search Attribute 是与 `=` 运算符匹配的 [分解成单词](https://www.elastic.co/guide/en/elasticsearch/reference/current/analysis-standard-tokenizer.html)。

例如，如果您有一个名为 `Description` 的自定义 `Text` Search Attribute，且具有以下任一值 -

```
my-business-id-foobar
my business id foobar
```

—则以下列表过滤器匹配 —

```
Description = 'foobar'
```

——但部分单词不会：

```
// Doesn't match
Description = 'foo'
```

<a id="text"></a>
#### 关键字 {#text}

对于 `Keyword` 类型的 Search Attribute（如 `WorkflowId`），使用 STARTS_WITH 作为前缀，使用 BETWEEN 作为后缀来执行部分字符串匹配。

- `WorkflowId STARTS_WITH "order-"` 与带有“order-”前缀的 Workflow Id 匹配，无论以下文本如何。

  ```
  order-
  order-1234
  order-abracadabra
  order-~~~abracadabra
  ```

- `WorkflowId BETWEEN "order-" AND "order-~"` 匹配 `order-` 之后字符且 ASCII 值低于 `~`（126，最高值可打印字符）的 Workflow Id，如下所示：

  ```
  order-
  order-1234
  order-abracadabra
  ```

它与 `order-~~` 不匹配。

:::note Filter Composition Quick Reference

**作品**

- 数据类型：
  - 带单引号或双引号的字符串文字
  - 数字（整数和浮点）
  - 布尔值
- 比较：`=`、`!=`、`>`、`>=`、`<`、`<=`
- 表达式/运算符：
  - `IN array`
  - `BETWEEN value AND value`
  - `STARTS_WITH string`
  - `IS NULL`、`IS NOT NULL`
  - `expr AND expr`、`expr OR expr`、`( expr )`
- 阵列：`( comma-separated-values )`

**请注意**

- 如果属性包含不在其中的字符，则用反引号括起来
  `[a-zA-Z0-9]`。
- `STARTS_WITH` 仅适用于关键字 Search Attribute。

:::

<a id="keyword"></a>
### 高效的 API 使用 {#keyword}

如果高级列表筛选器 API 检索大量 Workflow Execution（超过 10,000 个），则响应时间可能会更长。

从 Temporal Server v1.20 开始，您可以使用 `CountWorkflow` API 来高效计算 [Workflow Executions](/workflow-execution) 的数量。

要使用 `ListWorkflow` API 对结果进行分页，请使用页面令牌检索下一页。
继续，直到页面令牌变为 `null`/`nil`。

<a id="efficient-api-usage"></a>
#### 列表过滤器示例 {#efficient-api-usage}

以下是使用 [Temporal CLI](/cli/workflow#list) 设置列表过滤器的示例：

```
WorkflowType = "main.YourWorkflowDefinition" and ExecutionStatus != "Running" and (StartTime > "2021-06-07T16:46:34.236-08:00" or CloseTime > "2021-06-07T16:46:34-08:00")
```

当您使用前面的示例时，您会收到满足以下条件的 Workflow 列表：

- Workflow 类型为 `main.YourWorkflowDefinition`。
- Workflow 未处于运行状态。
- Workflow 在“2021-06-07T16:46:34.236-08:00”之后启动或在“2021-06-07T16:46:34-08:00”之后关闭。

以下是列表过滤器的其他示例。

```sql
WorkflowId = '<workflow-id>'
```

```sql
WorkflowId = '<workflow-id>' or WorkflowId = '<another-workflow-id>'
```

```sql
WorkflowId IN ('<workflow-id>', '<another-workflow-id>')
```

```sql
WorkflowId = '<workflow-id>' and ExecutionStatus = 'Running'
```

```sql
WorkflowId = '<workflow-id>' or ExecutionStatus = 'Running'
```

```sql
WorkflowId = '<workflow-id>' and StartTime > '2021-08-22T15:04:05+00:00'
```

```sql
ExecutionTime between '2021-08-22T15:04:05+00:00' and '2021-08-28T15:04:05+00:00'
```

```sql
ExecutionTime < '2021-08-28T15:04:05+00:00' or ExecutionTime > '2021-08-22T15:04:05+00:00'
```

```sql
WorkflowType STARTS_WITH '<workflow-type-prefix>'
```

<a id="list-filter-examples"></a>
### Search Attribute 别名 {#list-filter-examples}

Temporal 在大多数 [默认 Search Attribute](./search-attributes.mdx#default-search-attribute) 上添加 `Temporal` 前缀，以避免与自定义 Search Attribute 发生命名冲突。
为了更轻松地在列表过滤器中引用默认 Search Attribute，Temporal 支持别名，这样您就可以使用默认 Search Attribute 的无前缀名称。但是，如果您选择定义与默认 Search Attribute 的无前缀别名同名的自定义 Search Attribute，则您的自定义 Search Attribute 将覆盖该别名。

:::info Server Version Requirement

Search Attribute 别名需要 Temporal Server 版本 1.30 及更高版本。

:::

例如，默认的 Search Attribute `TemporalWorkflowVersioningBehavior` 的别名为 `WorkflowVersioningBehavior`。如果您尚未定义名为 `WorkflowVersioningBehavior` 的自定义 Search Attribute，则可以在列表过滤器中使用任一名称，并且两者都引用相同的 Search Attribute。

```sql
-- Using the original attribute name
WorkflowVersioningBehavior = 'pinned'

-- Using the Temporal-prefixed alias (equivalent)
TemporalWorkflowVersioningBehavior = 'pinned'
```

<a id="search-attribute-aliasing"></a>
#### 使用自定义 Search Attribute 进行别名解析 {#search-attribute-aliasing}

在列表过滤器中解析 Search Attribute 时，Temporal Server 按以下顺序检查匹配项：

1.当前Namespace中定义的自定义Search Attribute

2. 默认Search Attribute

这意味着，如果您定义与默认 Search Attribute 的别名同名的自定义 Search Attribute，则非 `Temporal` 前缀名称将引用您的自定义属性。您仍然可以使用 `Temporal` 前缀使用默认的 Search Attribute 进行搜索。

例如，如果您有一个名为 `SchedulePaused` 的自定义 Search Attribute，则使用以下 Search Attribute 的列表过滤器将返回不同的结果：

```sql
-- If you have a custom Search Attribute named 'SchedulePaused'
-- This will use your custom attribute, not the default Search Attribute
SchedulePaused = true

-- The original system attribute still works by using the Temporal prefix
TemporalSchedulePaused = true
```

`SchedulePaused` 将引用您的自定义 Search Attribute，而 `TemporalSchedulePaused` 将引用默认的 Search Attribute。
