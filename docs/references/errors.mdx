此参考列出了可能的 [Workflow 任务](/tasks#workflow-task) 错误以及解决方法。

> 对于其他类型的错误，请参阅 [Temporal 故障](https://docs.temporal.io/kb/failures)。

以下每个错误都对应一个 [Workflow任务失败原因](https://api-docs.temporal.io/#temporal.api.enums.v1.WorkflowTaskFailedCause)，它出现在 `Workflow_task_failed_event_attributes` 下的 [活动](/workflow-execution/event#event) 中。

<a id="bad-cancel-timer-attributes"></a>
## 错误取消 Timer 属性 {#bad-cancel-timer-attributes}

此错误表明 [Workflow 任务](/tasks#workflow-task) 在尝试取消 Timer 时失败。

{/* TODO add Timer term definition and link to it */}

检查您的 Timer 属性是否缺少 Timer Id 值。
添加有效的 Timer Id 并重新部署代码。

<a id="bad-cancel-workflow-execution-attributes"></a>
## 错误取消 Workflow Execution 属性 {#bad-cancel-workflow-execution-attributes}

由于未设置 [取消Workflow执行](/tasks#workflow-task) 属性，[Workflow 任务](/references/commands#cancelworkflowexecution) 失败。

重置所有缺失的属性并重新部署 Workflow 任务。

<a id="bad-complete-workflow-execution-attributes"></a>
## 错误的完整 Workflow Execution 属性 {#bad-complete-workflow-execution-attributes}

此错误表明 [Workflow 任务](/tasks#workflow-task) 由于未设置 [完成Workflow执行](/references/commands#completeworkflowexecution) 上的属性而失败。

重置任何缺失的属性。
如果负载超过大小限制，请调整负载的大小。

<a id="bad-continue-as-new-attributes"></a>
## 错误继续作为新属性 {#bad-continue-as-new-attributes}

此错误表明 [Workflow 任务](/tasks#workflow-task) 无法验证 [继续新的](/references/commands#continueasnewworkflowexecution) 属性。
该属性可能未设置或无效。

重置任何缺失的属性。
如果有效负载或备忘录超出大小限制，请调整输入大小。

检查 [Workflow](/workflows) 是否在取消别名密钥后验证 Search Attribute。

<a id="bad-fail-workflow-execution-attributes"></a>
## 错误失败 Workflow Execution 属性 {#bad-fail-workflow-execution-attributes}

此错误表明 [Workflow 任务](/tasks#workflow-task) 由于未设置 [失败Workflow执行](/references/commands#failworkflowexecution) 属性而失败。

如果遇到此错误，请确保设置了 `StartToCloseTimeout` 或 `ScheduleToCloseTimeout`。
重新启动[Workflow](/workers)和[Activity](/workflows)注册到的[Worker](/activities)。

<a id="bad-modify-workflow-properties-attributes"></a>
## 错误修改 Workflow 属性 属性 {#bad-modify-workflow-properties-attributes}

此错误表示 [Workflow 任务](/tasks#workflow-task) 无法验证更新插入备忘录或负载中的属性的属性。
这些属性要么未设置，要么超出大小限制。

重置所有未设置和空的属性。
调整 [备忘录](/workflow-execution#memo) 或有效负载的大小以适应系统的限制。

## 坏记录标记属性 {#bad-record-marker-attributes}

此错误表示 [Workflow 任务](/tasks#workflow-task) 由于 [标记](/references/events#markerrecorded) 名称未设置或不正确而失败。

输入有效的标记名称并重新部署任务。

<a id="bad-request-cancel-activity-attributes"></a>
## 错误请求取消 Activity 属性 {#bad-request-cancel-activity-attributes}

此错误表明可能未设置 [请求取消Activity](/references/commands#requestcancelactivitytask) 的属性，或者表明历史生成器状态无效。

Update 到 [Temporal SDK](/encyclopedia/temporal-sdks) 的最新版本。
在重试 [Workflow 任务](/tasks#workflow-task) 之前重置所有未设置的属性。

如果您仍然看到此错误，请检查 [不确定性原因](/workflow-definition#non-deterministic-change) 的代码。

<a id="bad-request-cancel-external-workflow-execution"></a>
## 错误请求取消外部 Workflow Execution 属性 {#bad-request-cancel-external-workflow-execution}

此错误表明 [Workflow 任务](/tasks#workflow-task) 在尝试取消外部 Workflow 时失败。
未设置或无效的属性可能会导致发生这种情况。

重置任何缺失的属性，例如 Workflow Id 或运行 Id。
调整任何超出长度限制的字段。

如果 [Child Workflow](/child-workflows) 设置为 `Start` 和 `RequestCancel`，请删除这些属性之一。
Child Workflow 无法在同一 Workflow 任务中执行这两个操作。

<a id="bad-schedule-activity-attributes"></a>
## 错误的 Schedule Activity 属性 {#bad-schedule-activity-attributes}

此错误表示 [`ScheduleActivityTask`](/references/commands#scheduleactivitytask) 或 [`CompleteWorkflowExecution`](/references/commands#completeworkflowexecution) 的属性未设置或无效。

重置任何未设置或空的属性。
调整接收到的有效负载的大小以保持在给定的大小限制内。

这可能会导致 Workflow 任务继续重试但没有成功。

确保在重试任务之前定义所有属性。
调整有效负载的大小以适应系统的大小限制。

<a id="cause-unhandled-update"></a>
## 原因未处理的 Update {#cause-unhandled-update}

当 Temporal Server 接收到 Workflow Update，而 Worker 上正在处理的 Workflow 任务生成将导致 Workflow 转换为关闭状态的命令时，会发生 `UnhandledUpdate`。

Temporal 拒绝 Workflow 任务完成，以保证 Update 最终由 Workflow 代码处理，并倒回 Workflow，以便它可以处理挂起的 Update。

当 Workflow 频繁接收 Update 时，可能会发生此错误。

<a id="grpc-message-too-large"></a>
## gRPC 消息太大 {#grpc-message-too-large}

当 Workflow 任务响应超过 gRPC 消息大小限制 4 MB 时，会发生此错误。
Workflow Execution 会自动终止，因为这是不可恢复的错误。

当 Workflow Schedule 在单个 Workflow 任务中包含太多活动、Child Workflow 或命令时，或者当 Workflow 返回较大结果时，通常会发生这种情况。

要解决此错误，请修复 Workflow 代码并启动新的 Workflow Execution。
将工作分成较小的批次，减少 Workflow 返回的大小，将 Continue-As-New 用于长时间运行的 Workflow，或使用自定义负载编解码器压缩大型负载。

有关详细的解决策略，请参阅 [BlobSizeLimitError 故障排除指南](/troubleshooting/blob-size-limit-error)。

## 超出待处理活动限制 {#pending-activities-limit-exceeded}

[Workflow](/workflows) 已达到待处理 [活动](/activities) 的容量。
因此，[Workflow 任务](/tasks#workflow-task) 未能阻止另一个 Activity 的创建。

在重新部署代码之前，让 Workflow 完成所有当前活动。

<a id="pending-child-workflows-limit-exceeded"></a>
## 待处理的 Child Workflows 超出限制 {#pending-child-workflows-limit-exceeded}

此错误表明 [Workflow](/workflows) 已达到待处理 [Child Workflows](/child-workflows) 的容量。
因此，[Workflow 任务](/tasks#workflow-task) 未能阻止添加额外的 Child Workflow。

重新部署此任务之前，请等待系统完成所有当前正在运行的 Child Workflow。

<a id="pending-nexus-operations-limit-exceeded"></a>
## 待处理的 Nexus 操作超出限制 {#pending-nexus-operations-limit-exceeded}

Workflow 已达到待处理 Nexus 操作的容量。因此，Workflow 任务无法阻止创建另一个 Nexus 操作。

在重试该任务之前，让 Workflow 完成任何当前的 Nexus 操作。

详细信息请参见 [根据 Workflow Nexus 操作限制](/cloud/limits#per-workflow-nexus-operation-limits)。

## 超出待处理请求取消限制 {#pending-request-cancel-limit-exceeded}

此错误表明 [Workflow 任务](/tasks#workflow-task) 在尝试添加更多取消请求后失败。
[Workflow](/workflows) 已达到取消其他 Workflow 的待处理请求的容量，并且无法接受更多请求。

如果您看到此错误，请在重试该任务之前给系统一些时间来处理待处理的请求。

<a id="pending-signals-limit-exceeded"></a>
## 待处理的 Signals 超出限制 {#pending-signals-limit-exceeded}

Workflow 已达到待处理 Signal 的容量。
因此，尝试将更多 [Signals](/tasks#workflow-task) 添加到外部 Workflow 后，[Workflow 任务](/sending-messages#sending-signals) 失败。

等待 Signal 由 Workflow 处理，然后重试该任务。

## 未处理的命令 {#unhandled-command}

此错误表示自上次 [Workflow 任务](/references/events) 启动以来有新的可用 [活动](/tasks#workflow-task)。
Workflow 任务失败，因为 [Workflow](/workflows) 尝试在不处理新事件的情况下关闭自身。

当 Workflow 收到大量 [Signals](/sending-messages#sending-signals) 时，可能会发生 `UnhandledCommand`。
如果 Workflow 没有足够的时间来处理这些 Signal，则 RetryWorkflow 任务是 Scheduled 来处理这些新事件。

为了防止此错误，请使用 ReceiveAsync 函数排空 Signal 通道。

如果您继续看到此错误，请检查日志中是否有失败的 Workflow 任务。
Workflow 可能已被不同的 [Worker](/workers#worker) 拾取。
