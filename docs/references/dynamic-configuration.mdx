Temporal 集群提供 [动态配置](/temporal-service/configuration#dynamic-configuration) 密钥，您可以将其应用于正在运行的集群，而无需重新启动服务。

创建集群配置时，动态配置键将设置为默认值。
您可以在测试集群设置时覆盖这些值，以根据工作负载要求获得最佳性能。

有关动态配置键的完整列表，请参阅 [https://github.com/temporalio/temporal/blob/main/common/dynamicconfig/constants.go](https://github.com/temporalio/temporal/blob/main/common/dynamicconfig/constants.go)。
确保检查服务器发行说明以了解这些键和值的任何更改。

对于动态配置键的默认值，请检查以下链接：

- [前端服务](https://github.com/temporalio/temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/frontend/service.go#L176)
- [历史服务](https://github.com/temporalio/temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/history/configs/config.go#L309)
- [配对服务](https://github.com/temporalio/temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/matching/config.go#L125)
- [Worker 服务](https://github.com/temporalio/temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/worker/service.go#L193)

设置动态配置是可选的。
仅当您需要覆盖默认值以在 Temporal 集群上获得更好的性能时才更改这些值。
另外，请确保在将这些更改投入生产之前测试您的更改。

＃＃ 格式

要覆盖默认动态配置值，请为要在 YAML 配置文件中更改的动态配置键指定自定义值和约束。
创建动态配置文件时使用以下格式。

```yaml
testGetBoolPropertyKey:
  - value: false
  - value: true
    constraints:
      namespace: 'your-namespace'
  - value: false
    constraints:
      namespace: 'your-other-namespace'
testGetDurationPropertyKey:
  - value: '1m'
    constraints:
      namespace: 'your-namespace'
      taskQueueName: 'longIdleTimeTaskqueue'
testGetFloat64PropertyKey:
  - value: 12.0
    constraints:
      namespace: 'your-namespace'
testGetMapPropertyKey:
  - value:
      key1: 1
      key2: 'value 2'
      key3:
        - false
        - key4: true
          key5: 2.0
```

<a id="format"></a>
### 约束 {#format}

您可以对某些动态配置键定义约束，以设置应用于 Namespace 或 Task Queue 级别的特定值。
不对动态配置键定义约束会在集群中设置值。

- 要无限制地设置配置键的全局值，请使用以下命令：

  ```yaml
  frontend.globalNamespaceRPS: # Total per-Namespace RPC rate limit applied across the Cluster.
    - value: 5000
  ```

- 对于可在 Namespace 级别自定义的键，除了全局适用于所有 Namespace 的一个默认值之外，您还可以为不同的 Namespace 指定多个值。
  要在 Namespace 级别设置值，请使用 `Namespace`（字符串），如以下示例所示。

  ```yaml
  frontend.persistenceNamespaceMaxQPS: # Rate limit on the number of queries the Frontend sends to the Persistence store.
    - constraints: {} # Sets default value that applies to all Namespaces
      value: 2000 # The default value for this key is 0.
    - constraints: { namespace: 'namespace1' } # Sets limit on number of queries that can be sent from "namespace1" Namespace to the Persistence store.
      value: 4000
    - constraints: { namespace: 'namespace2' }
      value: 1000
  ```

- 对于可以在 Task Queue 级别自定义的键，除了 Namespace 之外，您还可以指定 Task Queue 名称和任务类型。
  要在 Task Queue 级别设置值，请将 `taskQueueName`（字符串）与 `taskType`（可选；支持的值：`Workflow` 和 `Activity`）结合使用。

例如，如果您的 Workflow Execution 每秒创建大量 Workflow 和 Activity 任务，则可以向 Task Queue 添加更多分区（默认为 4）以处理任务的高吞吐量。
  为此，请将以下内容添加到动态配置文件中。
  请注意，如果更改分区数量，则必须为 Task Queue 上的读取和写入操作设置相同的计数。

  ```yaml
  matching.numTaskqueueReadPartitions: # Number of Task Queue partitions for read operations.
    - constraints: { namespace: 'namespace1', taskQueueName: 'tq' } # Applies to the "tq" Task Queue for both Workflows and Activities.
      value: 8 # The default value for this key is 4. Task Queues that need to support high traffic require higher number of partitions. Set these values in accordance to your poller count.
    - constraints: {
          namespace: 'namespace1',
          taskQueueName: 'other-tq',
          taskType: 'Activity',
        } # Applies to the "other_tq" Task Queue for Activities specifically.
      value: 20
    - constraints: { namespace: 'namespace2' } # Applies to all task queues in "namespace2".
      value: 10
    - constraints: {} # Applies to all other task queues in "namespace1" and all other Namespaces.
      value: 16
  matching.numTaskqueueWritePartitions: # Number of Task Queue partitions for write operations.
    - constraints: { namespace: 'namespace1', taskQueueName: 'tq' } # Applies to the "tq" Task Queue for both Workflows and Activities.
      value: 8 # The default value for this key is 4. Task Queues that need to support high traffic require higher number of partitions. Set these values in accordance to your poller count.
    - constraints: {
          namespace: 'namespace1',
          taskQueueName: 'other-tq',
          taskType: 'Activity',
        } # Applies to the "other_tq" Task Queue for Activities specifically.
      value: 20
    - constraints: { namespace: 'namespace2' } # Applies to all task queues in "namespace2".
      value: 10
    - constraints: {} # Applies to all other task queues in "namespace1" and all other Namespaces.
      value: 16
  ```

{/* Note that the values set with most constraints take priority over values that are set with fewer constraints, regardless of the order in which they are set in the dynamic configuration key. */}

有关如何设置动态配置的更多示例，请参阅 [样本服务器](https://github.com/temporalio/samples-server/blob/main/tls/config/dynamicconfig/development.yaml)。

<a id="constraints"></a>
## 常用动态配置键 {#constraints}

下表列出了可用于对 Temporal 集群的速率限制请求的常用动态配置键。

设置动态配置键是可选的。
如果您选择为 Temporal 集群设置 Update 这些值，请确保您配置了足够的资源来处理负载。

此处列出的所有值均适用于 Temporal Server v1.21。
检查 [服务器发行说明](https://github.com/temporalio/temporal/releases) 以验证升级版本时是否存在任何潜在的重大更改。

<a id="commonly-used-dynamic-configuration-keys"></a>
### 服务级别 RPS 限制 {#commonly-used-dynamic-configuration-keys}

每秒请求数 (RPS) 动态配置键设置向集群中的每个服务发出请求的速率。

扩展服务时，调整 RPS 以测试您的工作负载并设置可接受的配置基准。
超过这些限制会导致 `ResourceExhaustedError`。

| 动态配置键              | 类型 | 描述                                                                                                                                                                                                                                          | 默认值 |
| -------------------------------------- | ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- |
| 前端                               |      |                                                                                                                                                                                                                                                      |               |
| `frontend.rps`                         | INT  | 每个前端服务主机接受的请求的速率限制（请求/秒）。                                                                                                                                                                    | 2400          |
| `frontend.NamespaceRPS`                | INT  | 前端服务上每个 Namespace 接受的请求的速率限制（请求/秒）。                                                                                                                                                        | 2400          |
| `frontend.NamespaceCount`              | INT  | 每个前端服务主机上每个 Namespace 的并发 Task Queue 轮询数量的限制。                                                                                                                                                          | 1200          |
| `frontend.globalNamespaceRPS`          | INT  | 根据 Namespace 接受的请求的速率限制（请求/秒），跨集群应用。该限制均匀分布在可用的前端服务实例中。如果设置了此项，它将覆盖每个实例的限制 (`frontend.NamespaceRPS`)。 | 0             |
| `internal-frontend.globalNamespaceRPS` | INT  | 跨集群应用的每个内部前端服务主机上接受的请求的速率限制（请求/秒）。                                                                                                                                | 0             |
| 历史                                |      |                                                                                                                                                                                                                                                      |               |
| `history.rps`                          | INT  | 每个历史服务主机接受的请求的速率限制（请求/秒）。                                                                                                                                                                     | 3000          |
| 匹配                               |      |                                                                                                                                                                                                                                                      |               |
| `matching.rps`                         | INT  | 每个匹配服务主机接受的请求的速率限制（请求/秒）。                                                                                                                                                                    | 1200          |
| `matching.numTaskqueueReadPartitions`  | INT  | Task Queue 的读取分区数。必须使用 `matching.numTaskqueueWritePartitions` 进行设置。                                                                                                                                                 | 4             |
| `matching.numTaskqueueWritePartitions` | INT  | Task Queue 的写入分区数。                                                                                                                                                                                                         | 4             |

### 持久性存储的 QPS 限制 {#service-level-rps-limits}

每秒查询数 (QPS) 动态配置键设置服务每秒可以向持久性存储进行的最大查询数。

持久性存储速率限制是同步评估的。
根据您的数据库容量和工作负载调整这些键。
如果对持久性存储进行的查询数量超过动态配置值，您将看到任务出现延迟和超时。

| 历史                                   |      |                                                                                                                                                                                                                                                         |               |
| `history.persistenceMaxQPS`               | INT  | 历史记录主机每秒可以发送到持久性存储的最大查询数。                                                                                                                                                           | 9000          |
| `history.persistenceNamespaceMaxQPS`      | INT  | 历史主机可以发送到持久性存储的每个 Namespace 每秒的最大查询数。 <br /> 如果为此配置设置的值小于或等于 0，则将应用为 `history.persistenceMaxQPS` 设置的值。     | 0             |
| 匹配                                  |      |                                                                                                                                                                                                                                                         |               |
| `matching.persistenceMaxQPS`              | INT  | 匹配服务主机每秒可以发送到持久性存储的最大查询数。                                                                                                                                                  | 9000          |

| Worker                                    |      |                                                                                                                                                                                                                                                         |               |

<a id="size-limit-settings"></a>
### 辅助 Visibility 设置 {#size-limit-settings}

辅助 Visibility 配置密钥可在 Temporal 集群上启用双 Visibility。
这在迁移 Visibility 数据库或创建备份 Visibility 存储时非常有用。

| 动态配置键                  | 类型    | 描述                                                                                                                                                                                                                                                                                                                                                                                                          | 默认值 |
| ------------------------------------------ | ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- |
| `system.enableReadFromSecondaryVisibility` | 布尔值 | 启用从 [二级Visibility商店](/dual-visibility) 读取，并且可以根据 Namespace 进行设置。允许的值为 `true` 或 `false`。                                                                                                                                                                                                                                                                         | `false`       |
| `system.secondaryVisibilityWritingMode`    |         | 允许将 Visibility 数据写入辅助 Visibility 存储，并且可以按 Namespace 进行设置。将此值设置为 `on` 将禁用对主 Visibility 存储的写入操作。允许的值：<br /> `off`：仅允许写入主 Visibility 存储。 <br /> `on`：仅允许写入辅助 Visibility 存储。<br /> `dual`：允许写入主要和辅助 Visibility 存储。 | `off`         |

<a id="secondary-visibility-settings"></a>
### 服务器版本检查设置 {#secondary-visibility-settings}

Temporal Server 报告服务器版本及其连接的 SDK 版本，以确定 Web UI 是否应显示一条横幅，表明有新版本可供安装。可以通过定义以下值或将 `Temporal_VERSION_CHECK_DISABLED` 环境变量设置为 `1` 来禁用此功能。

| 动态配置键           | 类型    | 描述                                                                                                                       | 默认值 |
| ----------------------------------- | ------- | --------------------------------------------------------------------------------------------------------------------------------- | ------------- |
| `frontend.enableServerVersionCheck` | 布尔值 | 使Temporal Server能够报告有关当前服务器和SDK的版本信息。允许的值为 `true` 或 `false`。 | `true`        |
