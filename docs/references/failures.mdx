故障是 Temporal 系统中发生的各种类型错误的表示。

故障有不同类型，每种故障在 SDK 中都有不同的类型，在 Protobuf 消息（用于与 Temporal 服务通信并出现在 [Event History](/workflow-execution/event#event-history) 中）中具有不同的信息。

<a id="temporal-failure"></a>
## Temporal 失败 {#temporal-failure}

大多数 SDK 都有一个其他 Failures 扩展的基类：

- 打字稿：[Temporal失败](https://typescript.temporal.io/api/classes/common.TemporalFailure)
- 爪哇：[Temporal失败](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/failure/temporalFailure.html)
- Python：[失败错误](https://python.temporal.io/temporalio.exceptions.FailureError.html)
- PHP：[Temporal失败](https://php.temporal.io/classes/temporal-Exception-Failure-TemporalFailure.html)

基础 [失败原消息](https://api-docs.temporal.io/#temporal.api.failure.v1.Failure) 具有以下字段：

- `string message`
- `string stack_trace`
- `string source`：此故障源自的 SDK（例如，`"TypeScriptSDK"`）。在某些 SDK 中，此字段用于将调用堆栈重新水化为异常对象。
- `Failure cause`：此故障原因的 `Failure` 消息（如果适用）。
- `Payload encoded_attributes`：使用 [故障转换器](/failure-converter) 时包含编码的 `message` 和 `stack_trace` 字段。

<a id="application-failure"></a>
## 应用程序失败 {#application-failure}

Workflow、Activity 和 Nexus 操作代码使用应用程序故障来传达发生的特定于应用程序的故障。
这是由用户代码创建和抛出的唯一类型的 Temporal 故障。

- 打字稿：[应用失败](https://typescript.temporal.io/api/classes/common.ApplicationFailure)
- 爪哇：[应用失败](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/failure/ApplicationFailure.html)
- 转到：[应用程序错误](https://pkg.go.dev/go.temporal.io/sdk/temporal#ApplicationError)
- 蟒蛇：[应用程序错误](https://python.temporal.io/temporalio.exceptions.ApplicationError.html)
- PHP：[应用失败](https://php.temporal.io/classes/temporal-Exception-Failure-ApplicationFailure.html)
- 原型：[应用程序失败信息](https://api-docs.temporal.io/#temporal.api.failure.v1.ApplicationFailureInfo) 和 [失败](https://api-docs.temporal.io/#temporal.api.failure.v1.Failure)

<a id="errors-in-workflows"></a>
### Workflow 中的错误 {#errors-in-workflows}

Workflow 中的错误可能导致 **Workflow 任务失败**（将重试该任务）或 **Workflow Execution 失败**（Workflow 被标记为失败）。

只有 Workflow 异常即 Temporal Failures 才会导致 Workflow Execution 失败；所有其他异常都会导致 Workflow 任务失败并重试（在 Go 中，从 Workflow 返回的任何错误都会导致 Workflow Execution 失败，并且恐慌会导致 Workflow 任务失败）。
大多数类型的 Temporal 故障都是由 Temporal 服务引发的，例如取消 Workflow 时的 [取消失败](#cancelled-failure) 或 Activity 失败时的 [Activity 故障](#activity-failure)。
相反，您可以通过在 Workflow Definition 代码中抛出应用程序失败（返回 Go 中的任何错误）来显式使 Workflow Execution 失败。

<a id="workflow-task-failures"></a>
#### Workflow 任务失败 {#workflow-task-failures}

**Workflow 任务失败** 是无法处理 Workflow 任务的意外情况。
这可能是由 Workflow 代码中引发的非 Temporal 异常（Go 中的恐慌）触发的。
任何不扩展 Temporal 的 `FailureError` 异常的异常都被视为 Workflow 任务失败。
这些类型的失败将导致 Workflow 任务重试，直到
Workflow Execution 超时，默认无限制。

<a id="workflow-execution-failures"></a>
#### Workflow Execution 失败 {#workflow-execution-failures}

`ApplicationError`（`FailureError` 的扩展）可以在 Workflow 中引发，以使 Workflow Execution 失败。
Workflow Execution 失败会将 Workflow Execution 置于“失败”状态，并且不会再尝试执行此执行。
如果您要创建自定义异常，则需要扩展 [`ApplicationError`](https://docs.temporal.io/references/failures#application-failure) 类（[`FailureError`](https://docs.temporal.io/references/failures#temporal-failure) 的子类）。

<a id="errors-in-activities"></a>
### 活动中的错误 {#errors-in-activities}

在活动中，您可以抛出应用程序失败或另一个错误来使 Activity 任务失败。
在后一种情况下，错误将转换为应用程序故障。
转换期间，会设置以下应用程序失败字段：

- `type` 设置为错误的类型名称。
- `message` 设置为错误消息。
- `non_retryable` 设置为 false。
- `details` 未设置。
- `cause` 是从错误的 `cause` 属性转换而来的故障。
- `next_retry_delay` 未设置。
- 复制调用堆栈。

当 [Activity 执行](/activity-execution) 失败时，最后一个 Activity 任务的应用程序失败是 [Activity失败](#activity-failure) 的 `cause` 字段。
这个ActivityFailure是由Workflow调用Activity抛出的，可以在Workflow Definition中处理。

<a id="errors-in-nexus-operations"></a>
### Nexus 操作中的错误 {#errors-in-nexus-operations}

Nexus 操作可能会处于已完成、失败、取消和超时状态。

在底层，Nexus 操作机制将操作的生命周期分解为一个或多个 StartOperation 请求和完成回调，并在这些请求失败并出现可重试错误时自动重试这些请求。

Workflow 指定的 Schedule 关闭超时由调用者的机器强制执行，并且是操作转换到超时状态的唯一方法。

当操作处理程序返回同步响应或错误时，或者当异步操作（例如，由 Workflow 支持的操作）最终达到最终状态时，操作可能会以其他三种状态结束。

Nexus 操作处理程序可以返回可重试或不可重试的错误，以指示调用者的 Nexus 机器是否重试给定的请求。
在将响应发送给调用者之前超时的请求将自动重试。

默认情况下，错误被认为是可重试的，除非下面指定：

- 不可重试的应用程序失败
- 不成功的操作错误，可以将操作解决为失败或取消
- [处理程序错误](https://github.com/nexus-rpc/api/blob/main/SPEC.md#predefined-handler-errors) 具有以下类型：`BAD_REQUEST`、`UNAUTHENTICATED`、`UNAUTHORIZED`、`NOT_FOUND` 和 `RESOURCE_EXHAUSTED`

<a id="nexus-operation-task-failures"></a>
#### Nexus 操作任务失败 {#nexus-operation-task-failures}

Nexus 操作任务失败是指无法在处理程序中处理 Nexus 操作任务的意外情况。
这可以通过在 Nexus 处理程序代码中引发未知错误来触发。
这些类型的故障将导致重试 Nexus 操作任务。

<a id="nexus-operation-execution-failures"></a>
#### Nexus 操作执行失败 {#nexus-operation-execution-failures}

Nexus 操作处理程序可能会引发不可重试的应用程序故障，从而导致整个 Nexus 操作执行失败。
Nexus 操作执行失败会将 Nexus 操作执行置于“失败”状态，并且不会再尝试完成 Nexus 操作。

<a id="propagation-of-workflow-errors"></a>
#### Workflow 错误的传播 {#propagation-of-workflow-errors}

从 Nexus NewWorkflowRunOperation 处理程序创建的 Workflow 引发的应用程序错误将作为不可重试的错误自动传播到调用方，并导致 Nexus 操作执行失败。

<a id="using-failures-in-a-nexus-handler"></a>
#### 在 Nexus 处理程序中使用失败 {#using-failures-in-a-nexus-handler}

在 Nexus 操作处理程序中，您可以引发应用程序故障、Nexus 错误或其他错误，以使单个 Nexus 操作任务失败或使整个 Nexus 操作执行失败。

未知错误将转换为可重试的应用程序故障。转换期间，应用程序失败时设置以下字段：

- `non_retryable` 设置为 false。
- `type` 设置为错误的类型名称。
- `message` 设置为错误消息。

<a id="retryable-failures"></a>
#### 可重试的失败 {#retryable-failures}

可重试的 Nexus 操作 任务失败（例如未知错误）可使用内置 Retry Policy 自动重试。
当 Nexus 任务失败时，调用者 Workflow 会记录挂起的 Nexus 操作的事件尝试失败，并设置以下字段：

- `state` 设置为新状态，例如 BackingOff。
- `attempt` 设置为递增计数。
- 当重试 Nexus 任务时设置 `next_attempt_Schedule_time`。
- `last_attempt_failure` 设置有以下字段：
  - `message` 设置为错误消息。
  - `failure_info` 设置为应用程序失败。

例如，Nexus 处理程序中抛出的未知错误将显示为：

```
temporal workflow describe -w my-workflow-id
...
Pending Nexus Operations: 1

  Endpoint                 myendpoint
  Service                  my-hello-service
  Operation                echo
  OperationToken
  State                    BackingOff
  Attempt                  6
  ScheduleToCloseTimeout   0s
  NextAttemptScheduleTime  20 seconds from now
  LastAttemptCompleteTime  11 seconds ago
  LastAttemptFailure       {"message":"unexpected response status: "500 Internal Server Error": internal error","applicationFailureInfo":{}}
```

<a id="non-retryable"></a>
### 不可重试 {#non-retryable}

当 Activity 或 Workflow 引发应用程序故障时，故障的 `type` 字段将与 Retry Policy 的 [不可重试的错误](/encyclopedia/retry-policies#non-retryable-errors) 列表进行匹配，以确定是否重试 Activity 或 Workflow。
活动和 Workflow 还可以通过将应用程序故障的 `non_retryable` 标志设置为 `true` 来避免重试。

当 Nexus 操作处理程序引发应用程序故障时，默认情况下会使用无法自定义的内置 Retry Policy 重试。
Nexus 操作处理程序可以通过将应用程序故障的 `non_retryable` 标志设置为 true 来避免重试。
当从 Nexus 处理程序返回不可重试的错误时，整个 Nexus 操作执行失败，并且错误作为 Nexus 操作失败返回到调用者的 Workflow Execution。

<a id="activity-next-retry-delay"></a>
### 设置下次重试延迟 {#activity-next-retry-delay}

通过为给定的应用程序故障设置“下次重试延迟”，您可以告诉服务器在再次尝试 Activity 或 Workflow 之前等待该时间。
这将覆盖 Retry Policy 为您的特定异常计算的任何内容。

爪哇：[下次重试延迟](/develop/java/activities/timeouts#activity-next-retry-delay)
打字稿：[下一次重试延迟](/develop/typescript/activities/timeouts#activity-next-retry-delay)
PHP：[下次重试延迟](/develop/php/activities/timeouts#activity-next-retry-delay)

<a id="nexus-errors"></a>
### Nexus 错误 {#nexus-errors}

<a id="default-mapping"></a>
#### 默认映射 {#default-mapping}

默认情况下，从 Nexus 操作处理程序抛出的应用程序故障将根据 `non_retryable` 的设置映射到以下基础 Nexus 故障：

| `non_retryable` | Nexus 错误                | HTTP 状态码          |
| :-------------- | :------------------------- | :------------------------ |
| 假（默认） | HandlerErrorType内部   | 500 内部服务器错误 |
| 真的            | 操作失败错误 | 424 依赖失败     |

<a id="use-nexus-errors-directly"></a>
#### 直接使用Nexus错误 {#use-nexus-errors-directly}

为了改进外部 Nexus 调用者的语义和映射到 HTTP 状态代码（如果支持），我们建议 Nexus 操作处理程序直接抛出 Nexus 错误，其中包括下面的列表以及相关的重试语义。

例如，Nexus Go SDK 提供

- `Nexus.HandlerError(Nexus.HandlerErrorType, msg)`
- `Nexus.UnsuccessfulOperationError{state, failure}`

<a id="retryable-nexus-errors"></a>
#### 可重试的 Nexus 错误 {#retryable-nexus-errors}

| Nexus 错误类型                  | `non_retryable` |
| :-------------------------------- | :-------------- |
| HandlerErrorTypeResourceExhausted | 错误的           |
| HandlerErrorType内部          | 错误的           |
| 处理程序错误类型未实现    | 错误的           |
| 处理程序错误类型不可用       | 错误的           |

<a id="non-retryable-nexus-errors"></a>
#### 不可重试的 Nexus 错误 {#non-retryable-nexus-errors}

| Nexus 错误类型                | `non_retryable` |
| :------------------------------ | :-------------- |
| 处理程序错误类型错误请求      | 真的            |
| 处理程序错误类型未经身份验证 | 真的            |
| 处理程序错误类型未经授权    | 真的            |
| 未找到处理程序错误类型        | 真的            |
| 操作失败错误      | 真的            |

<a id="cancelled-failure"></a>
## 取消失败 {#cancelled-failure}

当请求 Workflow、Activity 或 Nexus 操作的 [消除](/activity-execution#cancellation) 时，SDK 以特定于语言的方式向用户表示取消。
例如，在 TypeScript 中，在某些情况下，Workflow API 函数直接抛出取消的故障，而在其他情况下，取消的故障包装在不同的故障中。
为了检查这两种类型的情况，TypeScript 有 [正在取消](https://typescript.temporal.io/api/namespaces/workflow#iscancellation) 帮助器。

当成功取消 Workflow、Activity 或 Nexus 操作时，取消的故障是 Activity 故障、Nexus 操作故障或“Workflow 失败”错误的 `cause` 字段。

- 爪哇：[取消失败](https://www.javadoc.io/doc/io.temporal/temporal-sdk/latest/io/temporal/failure/CanceledFailure.html)
- 转到：[取消错误](https://pkg.go.dev/go.temporal.io/sdk/temporal#CanceledError)
- 蟒蛇：[取消错误](https://python.temporal.io/temporalio.exceptions.CancelledError.html)
- PHP：[取消失败](https://php.temporal.io/classes/temporal-Exception-Failure-CanceledFailure.html)

<a id="nexus-operation-failure"></a>
## Nexus 操作失败 {#nexus-operation-failure}

当 Nexus 操作失败时，Nexus 操作失败将传递到 Workflow Execution。
它包含有关故障和 Nexus 操作执行的信息；例如，Nexus 操作名称和 Nexus 操作令牌。
失败的原因在消息和原因中（通常是应用程序错误或取消错误）。

- 执行：NexusOperationError
- 原型：NexusOperationFailureInfo

Nexus操作失败包含以下字段：

- 端点设置为端点的名称。
- 服务设置为服务的名称。
- 操作设置为操作的名称。
- 如果这是一个异步操作，则设置Operation_token，它可用于执行其他操作，例如取消操作。
- Scheduled_event_id 设置为 Scheduled 操作的调用者事件 ID。
- 消息设置为通用的不成功错误消息。
- 原因设置为具有以下字段的基础应用程序故障：
  - 不可重试设置为 true。
  - 类型设置为错误的类型名称。
  - 消息设置为错误消息。
- Nexus_error_code 设置为基础 Nexus 错误代码。
