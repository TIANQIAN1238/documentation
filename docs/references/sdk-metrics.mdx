:::info SDK metrics

此页面上的信息与 [Temporal SDK](/encyclopedia/temporal-sdks) 相关。

有关 [Temporal Cloud](/cloud/metrics/) 发出的指标，请参阅 [云指标](/cloud/overview)。

有关 [OSS集群](/references/cluster-metrics) 发出的指标，请参阅 [集群指标](/temporal-service)。

某些 SDK 发出的指标可能超出此 SDK 指标参考中列出的范围。
只有此指标参考中包含的指标才具有保证的、定义的行为。
其他指标被认为是已弃用、不一致或实验性的。

:::

Temporal SDK 会从 Temporal CLIent 使用情况和 Worker 进程发出一组指标。

- [如何使用 Go SDK 发出指标](/develop/go/platform/observability#metrics)
- [如何使用 Java SDK 发出指标](/develop/java/platform/observability#metrics)
- [如何使用 Python SDK 发出指标](/develop/python/platform/observability#metrics)
- [如何使用 TypeScript SDK 发出指标](/develop/typescript/platform/observability#metrics)
- [如何使用 .NET SDK 发出指标](/develop/dotnet/platform/observability#metrics)
- [如何使用 Ruby SDK 发出指标](/develop/ruby/platform/observability#metrics)
- [如何根据指标调整 Worker 性能](/develop/worker-performance)

所有指标在导出到其配置的目标之前都以 `Temporal_` 为前缀。
（此参考文献的部分内容已删除该前缀。）
目前，某些指标特定于某些 SDK。

TypeScript、Python、.NET 和 Ruby SDK 指标在 Core SDK 中定义。

PHP 和 Go 指标在 Go SDK 中定义。

Java 指标在 Java SDK 中定义。
指标在以下位置定义。

- [Java SDK 客户端指标](https://github.com/temporalio/sdk-java/blob/master/temporal-serviceclient/src/main/java/io/temporal/serviceclient/MetricsType.java)

:::note Metric units across SDKs

指标的测量单位可能会根据报告的 SDK 的不同而有所不同：

**基于核心的 SDK：** 直方图类型的指标默认以_毫秒_为单位进行测量。
对于使用 [核心SDK](/glossary#core-sdk) 的 SDK，可以将其自定义为使用秒数。
Core SDK 是多个 Temporal SDK 使用的共享通用核心库，包括 TypeScript、Python 和 .NET。

**Java 和 Go SDKs：** 直方图类型的指标以_秒_为单位测量。

:::

每个指标可能附加有以下键的某种组合：

- `task-queue`：Task Queue，Worker 实体正在轮询
- `Namespace`：Namespace Worker 绑定
- `poller_type`：以下之一：
  - `Workflow_task`
  - `Activity_task`
  - `Nexus_task`（仅限 Go 和 Java）
  - `sticky_Workflow_task`
- `Worker_type`：以下之一：
  - `ActivityWorker`
  - `WorkflowWorker`
  - `LocalActivityWorker`（仅限 Go 和 Java）
  - `NexusWorker`（仅限 Go 和 Java）
- `Activity_type`：与指标关联的 Activity 函数的名称
- `Workflow_type`：与指标关联的 Workflow 函数的名称
- `operation`：RPC方法名称；适用于与 Temporal CLIent gRPC 请求相关的指标

某些键可能并非在每个 SDK 中都可用，并且直方图指标在每个 SDK 中可能具有不同的存储桶。

### Activity_execution_cancelled

Activity 执行被取消。

- 类型：柜台
- 适用于：Java
- 标签：`Activity_type`、`Namespace`、`task_queue`

### Activity_execution_failed

Activity 执行失败。
这不包括 Go 和 Java SDK 中的本地 Activity 故障（请参阅 [local_Activity_execution_failed](#local_activity_execution_failed)）。

- 类型：柜台
- 适用于：Core、Go、Java
- 标签：`Activity_type`、`Namespace`、`task_queue`

### Activity_execution_latency

完成 Activity 执行的时间，从生成 Activity 任务到语言 SDK 响应完成（失败或成功）的时间。

- 类型：直方图
- 适用于：Core、Go、Java
- 标签：`Activity_type`、`Namespace`、`task_queue`

### Activity_poll_no_task

Activity Worker 对 Activity 任务的轮询超时，并且没有可从 Task Queue 中选择的 Activity 任务。

- 类型：柜台
- 适用于：Core、Go、Java
- 标签：`Namespace`、`task_queue`

### Activity_Schedule_to_start_latency

Activity 任务的 Schedule-To-Start 时间（以秒为单位）。
当生成 Activity 执行时，可以设置 [Schedule-启动超时](/encyclopedia/detecting-activity-failures#schedule-to-start-timeout)。
此指标对于确保及时从队列中处理 Activity 任务非常有用。一些 SDK 可能包括
`Activity_type` 标签，但指标不应因类型而异，因为它不会影响任务拉取的速率
从队列中。

- 类型：直方图
- 适用于：Core、Go、Java
- 标签：`Namespace`、`task_queue`

### Activity_succeed_endtoend_latency

成功完成 Activity 执行从 Scheduled 到完成的总延迟。
异步 Activity 完成不会记录此指标。

- 类型：直方图
- 适用于：Core、Go、Java
- 标签：`Activity_type`、`Namespace`、`task_queue`

### Activity_task_error

Activity 任务处理或执行期间发生内部错误或紧急情况。

- 类型：柜台
- 适用于：去，
- 标签：`Activity_type`、`Namespace`、`task_queue`、`Workflow_type`

### Nexus_poll_no_task

对 Nexus 任务的 Nexus Worker 轮询超时，并且没有可从 Task Queue 中选择的 Nexus 任务。

- 适用于：Go、Java
- 标签：`Namespace`、`task_queue`

### Nexus_task_Schedule_to_start_latency

Nexus 任务的 Schedule-To-Start 时间（以秒为单位）。 Schedule时间是从相应请求发生时开始的
当 SDK 开始处理任务时，点击前端服务。

该时间受到处理此请求时提供给前端的 `Request-Timeout` 标头的限制。

- 类型：直方图
- 适用于：Core、Go、Java
- 标签：`Namespace`、`task_queue`

### Nexus_task_execution_failed

处理 Nexus 任务导致错误。这包括从用户处理程序返回的任何错误和意外的错误
SDK 中的内部错误。

- 类型：柜台
- 适用于：Go、Java
- 标签：`Namespace`、`task_queue`、`Nexus_service`、`Nexus_operation`、`failure_reason`

`failure_reason` 标签的有效值：

- `internal_sdk_error`：处理 Nexus 任务时，SDK 中出现意外内部错误。表示一个
  SDK 中的错误。
- `handler_error_{TYPE}`：用户处理程序代码返回了预定义错误，如 [Nexus规格](https://github.com/nexus-rpc/api/blob/main/SPEC.md#predefined-handler-errors) 中所指定。
  如果处理程序返回意外错误，则 TYPE 将设置为 `INTERNAL`。
- `timeout`：用户处理程序代码在请求超时内未返回。
- `operation_failed`：用户处理程序代码指示操作失败。在 Go 中，这映射到
  `UnsuccessfulOperationError` 与 `failed` 状态。
- `operation_canceled`：用户处理程序代码指示操作已完成并已取消。在 Go 中，这个映射
  到具有 `canceled` 状态的 `UnsuccessfulOperationError`。
