## 什么是 Cloud Namespace 名称？ {#temporal-cloud-namespace-name}

Cloud Namespace 名称是客户为 Temporal Cloud 中的 [Namespace](/namespaces) 提供的名称。每个 Namespace 名称（例如 `accounting-production`）在客户账户范围内是唯一的，在 Namespace 创建后无法更改。

每个 Namespace 名称必须符合以下规则：

- Namespace 名称必须包含至少 2 个字符且不超过 39 个字符。
- Namespace 名称必须以字母开头，以字母或数字结尾，且只能包含字母、数字和连字符（-）。
- 每个连字符（-）前后必须紧跟字母或数字，不允许连续使用连字符。
- Namespace 名称中的所有字母必须为小写。

## 什么是 Temporal Cloud Account ID？ {#temporal-cloud-account-id}

Temporal Cloud Account ID 是 Temporal Technologies 分配的唯一客户标识符。每个 ID 是一个简短的字母数字字符串，例如 `f45a2`，至少 5 个字符长。此账户标识符在每个客户使用 Temporal Cloud 期间保持不变。

有时你可能需要知道你的客户 Account ID。访问账户的 Namespace 是获取此信息的简便方法。每个 Temporal Namespace 使用 Account ID 后缀。这是在任何 Temporal Cloud Namespace 名称中句点后面的字母数字字符串。

你可以从 [Temporal Cloud](https://cloud.temporal.io) Web UI 或通过命令行界面（CLI）使用 `tcld` 工具获取 Account ID。请按照以下步骤操作。

## 什么是 Cloud Namespace ID？ {#temporal-cloud-namespace-id}

Cloud Namespace ID 是 Temporal Cloud 中 [Namespace](/namespaces) 的全局唯一标识符。Namespace ID 由以下各部分连接而成：

1. [Namespace 名称](#temporal-cloud-namespace-name)
1. 句点（.）
1. Namespace 所属的 [Account ID](#temporal-cloud-account-id)

例如，对于 Account ID `123de` 和 Namespace 名称 `accounting-production`，Namespace ID 为 `accounting-production.123de`。

## 什么是 Cloud gRPC Endpoint？ {#temporal-cloud-grpc-endpoint}

Temporal Client 通过 gRPC 协议发送和接收消息，在应用程序代码与 Temporal Server 之间进行通信。gRPC 是一个具有低延迟和高性能特点的远程过程调用框架，为 Temporal 提供了高效的、与语言无关的通信框架。

每个 Temporal Namespace 使用 gRPC 端点进行通信。迁移到 Temporal Cloud 时，需要将代码中的 gRPC 端点从当前托管环境（无论是自托管还是本地托管）切换到 Temporal Cloud。

每个 Cloud Namespace 的详情页面上都有 gRPC 端点。按照以下步骤查找：

1. 登录 [cloud.temporal.io](https://cloud.temporal.io/namespaces) 上的账户。
2. 从左侧垂直导航栏导航到 Namespace 列表页面。
3. 点击 Namespace 名称，选择并打开要获取端点的 Namespace 页面。
4. 在 Namespace 详情页面，点击页面右上角的"连接"按钮。
5. 点击 gRPC 地址旁边的复制图标将其复制到剪贴板。

有关不同 gRPC 端点类型以及如何访问这些端点的更多信息，请参阅[如何在 Temporal Cloud 中访问 Namespace](/cloud/namespaces/#access-namespaces)。

## 如何在 Temporal Cloud 中创建 Namespace {#create-a-namespace}

:::info

创建 [Namespace](/namespaces) 的用户会自动获得该 Namespace 的 [Namespace Admin](/cloud/manage-access/roles-and-permissions#namespace-level-permissions) 权限。

要创建 Namespace，用户必须具有 Developer、Account Owner 或 Global Admin 账户级别的[角色](/cloud/manage-access/roles-and-permissions#account-level-roles)。

:::

:::tip

默认情况下，每个账户从 10 个 Namespace 开始。随着你开始创建 Namespace，此数量会自动增加。对于大规模需求，请提交[支持工单](/cloud/support#support-ticket)。

:::

### 创建 Namespace 所需的信息

要在 Temporal Cloud 中创建 Namespace，需要收集以下信息：

- [Namespace 名称](/cloud/namespaces#temporal-cloud-namespace-name)、区域和云提供商
- 已关闭 [Workflow Execution](/workflow-execution) 的 [Event History](/workflow-execution/event#event-history) 的[保留期](/temporal-service/temporal-server#retention-period)
- 如果使用 mTLS 身份验证，则需要 Namespace 的 [CA 证书](/cloud/certificates#certificate-requirements)
- [Codec Server 端点](/production-deployment/data-encryption#set-your-codec-server-endpoints-with-web-ui-and-cli)，用于向 Namespace 中 Workflow Execution 的 Event History 中的用户显示解码后的 Payload。详情请参阅[保护数据安全](/production-deployment/data-encryption)
- 每位用户的[权限](/cloud/manage-access/roles-and-permissions#namespace-level-permissions)

## Namespace 最佳实践有哪些？ {#best-practices}

本节提供了跨用例、服务、应用程序或域组织 [Namespace](/namespaces) 的一般指导。Temporal Cloud 提供 Namespace 即服务，因此 Namespace 就是端点。客户不仅应考虑 Namespace 命名约定，还应考虑如何使用 Namespace 作为边界来分组或隔离工作负载。

每个团队可以拥有自己的 Namespace，以提高模块化程度、安全性、调试能力和故障隔离。Namespace 可以控制可能耗尽速率限制的行为不当 Worker 的影响范围。敏感的 Workflow 状态（PCI 数据）可以通过每个 Namespace 的权限和独立的加密密钥进行保护。

不同 Namespace 中的 Temporal 应用程序可以通过 [Nexus](/cloud/nexus) 连接，为其他人提供清晰的服务契约，并具备内置的 [Nexus 访问控制](/cloud/nexus/security)。Nexus 支持跨团队、跨域、多区域和多云的使用场景。

### 约束与限制

在考虑适当的 Namespace 配置之前，应了解 [Temporal Cloud 限制](/cloud/limits) 页面上列出的约束。

有关在限制范围内管理 Namespace 的建议，请参阅 [Namespace 最佳实践](/best-practices/managing-namespace) 页面。

### 一般指导

Namespace 配置需要一定的考量。以下是一些需要考虑的一般性准则。

- Namespace 通常按用例定义。一个用例可以涵盖广泛的 Workflow 类型以及几乎无限规模的并发 [Workflow Execution](/workflow-execution)。
- Namespace 可以沿着服务、应用程序、域甚至子域等边界进一步划分。
- Namespace 应用于降低关键任务应用程序的"影响范围"。
- 生产和开发等环境通常有隔离要求。我们建议每个环境拥有自己的 Namespace。
- Namespace 应用于降低关键任务应用程序的"影响范围"。
- 使用 [Nexus](/nexus) 通过清晰的服务契约在团队、域和 Namespace 边界之间通信，而不是直接共享 Temporal 原语。Nexus 使每个团队能够拥有自己的 Namespace，通过 Nexus Endpoint 共享功能，同时提高安全性、故障排查能力和故障隔离。

### 示例

以下是一些组织 Namespace 的思路。

#### 示例 1：按用例和环境划分 Namespace

对于不存在多个服务和团队或域边界的简单配置，我们建议每个用例和环境组合使用一个 Namespace。

示例命名约定：

<pre>&lt;use-case>_&lt;environment></pre>

#### 示例 2：按用例、服务和环境划分 Namespace

当属于同一用例的多个服务通过 API（HTTP/gRPC）与外部 Temporal 通信时，我们建议每个用例、服务和环境组合使用一个 Namespace。

示例命名约定：

<pre>&lt;use-case>_&lt;service>_&lt;environment></pre>

#### 示例 3：按用例、域和环境划分 Namespace

当属于同一用例的多个服务需要通过 [Signal](/sending-messages#sending-signals) 或启动 [Child Workflow](/child-workflows) 相互通信时，我们建议每个用例、域和环境组合使用一个 Namespace。但在这种情况下，必须通过为每个 Workflow ID 添加服务特定的前缀来确保 Workflow ID 的唯一性。每个 Task Queue 的名称也必须是唯一的。如果涉及多个团队，域也可以代表团队边界。

示例命名约定：

<pre>&lt;use-case>_&lt;domain>_&lt;environment></pre>

示例 workflowId 约定：

<pre>&lt;service-string>_&lt;workflow-id></pre>

## 如何在 Temporal Cloud 中访问 Namespace {#access-namespaces}

{/* How to access a Namespace in Temporal Cloud */}

Temporal Cloud 支持使用 [API 密钥](/cloud/api-keys)_或_ [mTLS](/cloud/certificates) 对 Namespace 进行身份验证。要将 Namespace 从一种身份验证方法迁移到另一种，或在同一 Namespace 上同时使用 API 密钥和 mTLS 身份验证，请联系[支持团队](https://docs.temporal.io/cloud/support#support-ticket)。

:::info

在**同一** Namespace 上同时使用 **API 密钥**和 **mTLS** 身份验证处于[预发布阶段](/evaluate/development-production-features/release-stages)，不支持[高可用功能](/cloud/high-availability)，也不支持使用 API 密钥对 Namespace Endpoint 进行身份验证。

:::

连接到 Namespace 需要适用于该 Namespace 的特定端点。Temporal Cloud 中访问 Namespace 有两种 gRPC 端点类型：Namespace 端点和区域端点。

- **Namespace 端点**（`<namespace>.<account>.tmprl.cloud:7233`）——**推荐**
    - 此端点对每个 Namespace 唯一。无论 Namespace 使用哪个区域，它始终连接到该 Namespace。
    - 使用 Namespace 端点的 Temporal Client 无需知道 Namespace 所在的区域。
    - 对于具有[高可用](/cloud/high-availability)特性的 Namespace，Namespace 端点会自动将流量导向活跃区域，因此 Worker 和 Client 在故障切换期间无需更改端点。
- **区域端点**（`<region>.<cloud_provider>.api.temporal.io:7233`）
    - Temporal Cloud 在每个云区域只有一个区域端点。同一区域端点可以访问在该区域活跃的任何 Namespace（或在该区域有[副本](/cloud/high-availability)的 Namespace）。
    - Temporal Client 可以使用区域端点确保对 Namespace 的连接始终发生在该区域内。这在需要明确控制哪个区域处理请求的高级[高可用](/cloud/high-availability)配置中很有用。
    - 使用 mTLS 进行身份验证时，Temporal Client 必须在请求中将 `server_name` 属性设置为 Namespace 端点的值。这告知 Client 在 TLS 握手期间期望不同的 SNI 标头，因为向区域端点发出的请求会被重定向到特定 Namespace。

### 使用 API 密钥或 mTLS 配置 Temporal Client

要使用 API 密钥连接 [Temporal CLI](/cli)、[Client SDK](/develop)、[tcld](/cloud/tcld)、[Cloud Ops API](/ops) 和 [Terraform](/cloud/terraform-provider)，请参阅[使用 API 密钥进行身份验证](/cloud/api-keys#using-apikeys)。

要使用 mTLS 连接 [Temporal CLI](/cli) 和 [Client SDK](/develop)，请参阅[配置 Client 使用客户端证书](/cloud/certificates#configure-clients-to-use-client-certificates)。

### 访问 Temporal Web UI

访问 Temporal Web UI 请使用以下格式的 HTTPS 端点：`https://cloud.temporal.io/namespaces/<namespace>.<account>`。例如：`https://cloud.temporal.io/namespaces/accounting-production.f45a2`。

### 通过加密和私有连接访问 Namespace

为确保数据安全，所有进出 Namespace 的流量均使用 TLS 1.3 加密。

为增强保护：
- 为 Namespace 设置[私有连接](/cloud/connectivity#private-network-connectivity-for-namespaces)。
- 在自有网络架构中，为 Client 和 Worker 发出的出站网络请求设置允许列表，包含 Namespace 所在云提供商区域的 IP 地址范围：
  - [AWS IP 地址范围](https://docs.aws.amazon.com/vpc/latest/userguide/aws-ip-ranges.html)
  - [GCP IP 地址范围](https://cloud.google.com/compute/docs/faq#find_ip_range)

## 如何在 Temporal Cloud 中删除 Namespace {#delete-a-namespace}

:::info

要删除 Namespace，用户必须具有该 Namespace 的 Namespace Admin [权限](/cloud/manage-access/roles-and-permissions#namespace-level-permissions)。

:::

### 使用 Temporal Cloud UI 删除 Namespace

1. 前往 Temporal Cloud UI 并登录。
1. 在窗口左侧，选择 **Namespaces**。
1. 在 **Namespaces** 页面，选择一个 Namespace 名称。
1. 在 Namespace 页面，选择窗口右上角的 **Edit**。
1. 在 **Edit** Namespace 页面，选择窗口右上角的 **Delete Namespace**。
1. 在 **Delete Namespace** 对话框中，输入 `DELETE` 以确认删除该 Namespace。
1. 选择 **Delete**。

删除 Temporal Cloud Namespace 后，Temporal Service 会立即移除该 Namespace 的 Workflow Execution 和 Task Queue。删除 Namespace 前，请确保所有 Workflow 已完成、取消或终止。Namespace 删除是永久性的。

已关闭的 Workflow 历史记录将保留在 Temporal 存储中，直到用户定义的保留期届满。此期限反映 Workflow Execution 关闭时生效的策略。

如有进一步问题，请联系[支持团队](https://docs.temporal.io/cloud/support#support-ticket)。

### 使用 tcld 删除 Namespace

有关详细信息，请参阅 [tcld namespace delete](/cloud/tcld/namespace/#delete) 命令参考。

### Namespace 删除保护 {#delete-protection}

为防止意外删除 Namespace，Temporal Cloud 提供了保护功能。为生产环境 Namespace 启用删除保护后，可确保关键数据不会被意外删除。

请按照以下步骤操作：

- 访问 Temporal Cloud 上的 [Namespace 页面](https://cloud.temporal.io/namespaces)。
- 打开 Namespace 详情页面。
- 选择 Edit 按钮。
- 向下滚动到 Security 并点击展开按钮（向下箭头）。
- 启用 **Deletion Protection**

<CaptionedImage
  src="/img/cloud/namespace/deletion-protection.png"
  title="Deletion Protection is enabled by toggling the switch"
/>

要使用 [`tcld`](/cloud/tcld) 启用或禁用此功能，请使用以下命令。将值设置为 `true` 以启用，设置为 `false` 以禁用：

```
tcld namespace lifecycle set \
    --namespace <namespace_id.account_id> \
```

## 如何为 Temporal Cloud 中的 Namespace 添加标签 {#tag-a-namespace}

标签是可附加到 Temporal Cloud 中 Namespace 的键值元数据对，帮助运营人员更轻松地组织、追踪和管理 Namespace。

### 标签结构和限制

- 每个 Namespace 最多可有 10 个标签
- 对于给定的 Namespace，每个键必须唯一（例如，同一 Namespace 不能同时拥有 `team:foo` 和 `team:bar` 标签）
- 键和值的长度必须为 1-63 个字符
- 允许的字符：小写字母（`a-z`）、数字（`0-9`）、句点（`.`）、下划线（`_`）、连字符（`-`）和 @ 符号（`@`）
- 标签不是安全存储机制，不应存储个人身份信息（PII）或受保护健康信息（PHI）
- 标签不会改变被标记资源的行为
- 每个账户的唯一标签键数量有 1000 个的软限制

### 权限

- 只有[**账户管理员**和**账户所有者**](/cloud/manage-access/roles-and-permissions#account-level-roles)才能创建和编辑标签
- 所有有权访问 Namespace 的用户都可以查看其标签

### tcld

有关详细信息，请参阅 [tcld namespace tags](/cloud/tcld/namespace/#tags) 命令参考。

### Terraform

有关详细信息，请参阅 [Terraform provider](https://github.com/temporalio/terraform-provider-temporalcloud/blob/main/docs/resources/namespace_tags.md)。

### Web UI

标签可以通过 Temporal Cloud Web 界面查看和管理。查看 Namespace 时，你会看到显示的标签，如果你有适当的权限，可以添加、编辑或删除标签。
