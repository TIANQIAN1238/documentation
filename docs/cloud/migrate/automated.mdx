<a id="process-overview"></a>
## 流程概述 {#process-overview}

自动迁移旨在提供一种零停机、安全的迁移到 Temporal Cloud 的方法。本指南
概述了将 Workflow 从自托管 Temporal Server 转换为托管在其中的 Namespace 的当前流程
Temporal Cloud。

<a id="namespace-migration-schedule"></a>
### Namespace 迁移 Schedule {#namespace-migration-schedule}

规划迁移时，强烈建议按照从最不重要到最重要的 Namespace 的顺序进行迁移。
理想情况下，该项目将从指定为“测试”的 Namespace 开始，其中停机是可接受的结果
测试过程。在此基础上，根据潜在影响最小的顺序确定迁移的优先级。

![Temporal自动化迁移组件](/img/cloud/migration/ns_migration_schedule.png)

<a id="project-phases"></a>
### 项目阶段 {#project-phases}

迁移过程分为几个阶段，其中一部分涉及与 Temporal 协调以创建
必要的云端资源。

迁移涉及以下阶段：

1. 准备 - 初始准备涉及从自托管集群收集和评估关键数据点。
   收集的数据将由 Temporal 进行评估，以确保迁移资格。
2. 设置 - 验证资格后，将修改自托管集群的配置以支持
   迁移。所有云端组件都将被配置并且自托管
   将部署 [S2S代理](https://hub.docker.com/r/temporalio/s2s-proxy/tags)。
3. 测试 - 一旦所有必需的组件就位，将使用测试 Namespace 来验证迁移过程。
4. 启动 - 成功测试过程结束后，将开始生产 Namespace 的迁移。
5. 完成 - 将 Namespace 转移到 Temporal Cloud 并进行验证后，迁移将完成。

请在规划迁移之前查看 [附加说明](#additional-notes) 部分。

<a id="phase-1-prepare"></a>
## 第一阶段：准备 {#phase-1-prepare}

在准备迁移到 Temporal Cloud 时，必须收集以下数据点并通过以下方式提供给 Temporal：
支持票。

- Temporal Cloud 帐户列表。如果您是 Temporal Cloud 的新手，那么最常见的建议是创建一个
  包含您的 Namespace 的单个帐户。
- 每个集群的目标 Temporal Cloud 提供商/区域。请参阅 [当前区域](/cloud/regions) 列表。
- 每个集群的服务器配置。
- 对于每个 Namespace：
  - Namespace名称及其翻译的云端名称。由于 Temporal Cloud 中的 Namespace 遵循
    [具体格式](/cloud/namespaces#temporal-cloud-namespace-name)，现有的 Namespace 必须转换为新的
    格式。
  - Namespace 尺寸指标。这些用于为迁移正确调整云端资源的大小。
  - 自定义 Search Attribute（如果有）。 Temporal 必须确保您的自定义 Search Attribute，特别是在使用时
    基于 SQL 的 Visibility 存储，与 Temporal Cloud 兼容。

<a id="capture-cluster-configurations"></a>
### 捕获集群配置 {#capture-cluster-configurations}

Temporal 必须检查您的自托管服务器配置以确保兼容性。适用于 Temporal 的现代版本
OSS，对每个集群运行以下命令：

```
temporal operator cluster describe --address <frontend:7233> --output json
```

对于服务器版本 1.28.1 及更早版本，请使用以下替代方法之一：

```
tctl --address <frontend:7233> admin cluster describe
```

或者

```
grpcurl -v -plaintext <your temporal address and port> temporal.server.api.adminservice.v1.AdminService.DescribeCluster
```

<a id="capture-namespace-metrics"></a>
### 捕获 Namespace 指标 {#capture-namespace-metrics}

充分了解生产 Namespace 的当前使用模式至关重要，因为此数据用于
云侧资源规划。对于每个 Namespace，收集以下指标：

- 打开/关闭 Workflow 的总数
- 使用的总存储空间
- 当前的保留政策。请注意，这可能与
  [默认保留策略](/cloud/limits#default-retention-period) 中的 Temporal Cloud。
- 峰值 [每秒动作](/glossary#actions-per-second-aps) (APS)。参见 参考资料 部分
  [以下](#how-to-gather-self-hosted-metrics) 有关收集这些指标的说明。
- 如果您使用自定义 Search Attribute：
  - 提供您的 Namespace 的_CustomSearchAttributeAliases_（请参阅下面的命令）。
  - 提供使用的 VisibilityStore（例如 Elasticsearch 或 SQL）

如果需要，请使用以下命令捕获 _CustomSearchAttributeAliases_：

```
temporal operator namespace describe -n <namespace>
```

<a id="prepare-mtls-certificates"></a>
### 准备mTLS证书 {#prepare-mtls-certificates}

mTLS 用于保护 [S2S代理](#s2s-proxy-configuration) 通信通道的安全。按照概述的流程生成新证书
[这里](/cloud/certificates#issue-certificates)。您需要提供这些证书作为支持请求的一部分。


<a id="report-collected-data"></a>
### 报告收集的数据 {#report-collected-data}
通过 [支持票](/cloud/support#ticketing) 将所有收集到的数据和 mTLS 证书提供给 Temporal。

使用以下模板来收集数据。

```
Temporal Cloud Account:

<for each cluster>
Cluster Name:
Target Cloud/Region:
Type of Visibility Store:
Cluster Configuration:

Namespaces:
   <for each Namespace>
   Namespace Name:
   Translated Name:
   Metrics:
   Custom Search Attributes:
   <end>
<end>
```

<a id="phase-2-setup"></a>
## 第 2 阶段：设置 {#phase-2-setup}

迁移获得批准后，下一步是准备自托管集群和 Temporal Cloud
迁移的资源。

:::warning

仅当您的请求已获得 Temporal 批准后才能继续。

:::

<a id="prepare-temporal-cloud-namespace"></a>
### 准备 Temporal Cloud Namespace {#prepare-temporal-cloud-namespace}

对于每个要迁移的Namespace，在Temporal Cloud中创建一个空的Namespace作为迁移目标。自从
迁移无法继续到已在使用的 Namespace，Namespace 应保持为空（无 Workflow）。申请
任何所需的自定义 Search Attribute。

请务必根据需要调整 Namespace 的 [速率限制](/cloud/capacity-modes) 并创建任何自定义搜索
所需的属性。

如果您是 Temporal Cloud 的新手，请考虑您所需的 Namespace 连接路径。您可以通过以下方式连接
公共互联网或通过 [私人连接](/cloud/connectivity)。

<a id="s2s-proxy-configuration"></a>
### S2S代理配置 {#s2s-proxy-configuration}

[S2S代理](https://github.com/temporalio/s2s-proxy)需要云侧入站端点。继续部署
仅在收到来自 Temporal 的端点后。

该代理通过安全的 2 路隧道提供 API 转发至 Temporal Cloud。自托管代理将启动
与其云端对应方的出站连接 (TCP 8233) 并建立 2 向隧道。如果有防火墙
在路径内，确保它们允许此出站连接。

![Temporal自动化迁移组件](/img/cloud/migration/auto-migration-components.png)

使用以下过程来部署代理：

1. 从以下位置获取最新的Docker镜像
   [Temporalio/s2s-proxy 存储库](https://hub.docker.com/r/temporalio/s2s-proxy/tags)。
2. 收集上一步中生成的 mTLS 证书。
3. 部署 3 个 s2s-proxy 副本（至少 4 个 CPU 和 512mb 内存）。对于 Kubernetes 用户，请使用此
   [舵图示例](https://github.com/temporalio/s2s-proxy/blob/main/charts/s2s-proxy/README.md)作为参考。
   查看[例子](https://github.com/temporalio/s2s-proxy/blob/main/charts/s2s-proxy/example.yaml)配置文件
   作为参考。请注意，可以部署超过 1 个副本。如果部署了 1 个以上的副本，您必须
   Update Temporal 与所使用的副本数量，以便可以在云端配置相同的数量。
4. 使用以下命令测试访问。它应该显示迁移服务器的信息。

```
temporal operator cluster describe --address {the-outbound-external-address-of-your-proxy}
```

验证连接后，通知 Temporal，以便可以从云端代理验证连接。

S2S 代理上有多个可用指标（prometheus 端点：_proxy-pod-ip_:9090/metrics）。这些是
有助于监控代理的整体健康状况。特别是，度量

`Temporal_s2s_proxy_mux_connection_active`

将监视与云端代理的连接。


<a id="modify-cluster-configuration"></a>
### 修改集群配置 {#modify-cluster-configuration}

您的自托管集群的 [动态配置](/references/dynamic-configuration) 必须修改为
方便迁移。

:::warning

与 Temporal 密切协调此过程的步骤。

:::

调整最大连接保持活动时间以匹配云中的设置。

```yaml
frontend.keepAliveMaxConnectionAge:
  - value: '2h'
```

如果您之前已启用 [全球 Namespace](/global-namespace)，则故障转移版本增量必须获得
Temporal。这是**关键**，因为该值只能设置一次并且不能更改。

如果您从未启用全局 Namespace，那么您可以启用它并将初始故障转移版本设置为唯一值
（1 到 100 之间）。通过更新 _clusterMetadata_ 和 _dcRedirectionPolicy_ 来启用。完成后，重新启动所有
Temporal 服务（前端、历史、匹配、Worker），从前端开始。

```yaml
dcRedirectionPolicy:
  policy: 'all-apis-forwarding'

clusterMetadata:
enableGlobalNamespace: true # add this
failoverVersionIncrement: CHANGEME # usually 1000000, but coordinate with Temporal. must match cloud-side migration server
masterClusterName: _NO_CHANGE_
currentClusterName: _NO_CHANGE_
clusterInformation:
  _NO_CHANGE_:
    enabled: true
    initialFailoverVersion: CHANGEME # use unique number between 1 and 100 for each server
    rpcName: _NO_CHANGE_
    rpcAddress: _NO_CHANGE_
```

所有服务重新启动后，使用以下命令验证配置：

```
temporal operator cluster describe
```

预期输出示例如下：

```yaml
"failoverVersionIncrement": "1000000",
"initialFailoverVersion": "the number you picked"
"isGlobalNamespaceEnabled": true
```

<a id="phase-3-testing"></a>
## 第三阶段：测试 {#phase-3-testing}

应使用新创建的 Namespace 或被视为非生产的 Namespace 继续进行测试。在
无论哪种情况，都应使用测试 Workflow。混合使用已完成的、现用的和新的 Workflow 是理想的选择
测试期间。

测试使用以下过程：

1. 创建或确定非生产 Namespace，它可以在出现问题时容忍数据丢失。
2. 创建目标云端Namespace，并将Namespace定义添加到S2S代理配置中。
3. 针对 Namespace 运行测试 Workflow。
4. 对 Namespace 执行完整的端到端迁移（请参阅完整过程的剩余阶段）。

如果自托管部署中的所有数据都迁移到云端，则认为测试成功。

<a id="phase-4-initiate"></a>
## 第 4 阶段：启动 {#phase-4-initiate}

以下部分概述了启动迁移的过程。

<a id="migration-start"></a>
### 迁移开始 {#migration-start}

Temporal 将启动迁移。在此过程中，自托管 Namespace 保持活动状态，而云
Namespace 变为被动。 Workflow 从自托管 Namespace 复制到云 Namespace。曾经云
Namespace已与自托管Namespace完全同步，迁移已准备好移交。

下面的[命令](https://pkg.go.dev/github.com/temporalio/tcld#readme-start-a-migration)用于启动
迁移：

```
tcld migration start --endpoint-id <endpoint-id> --source-namespace <source-namespace> --target-namespace <target-namespace>
```

＃＃＃ 监视器

在初始同步期间，监控整个过程以确保取得进展非常重要。而Temporal
将从云端进行监控，也可以使用以下方式从自托管端监控进度
_replication_stream_stuck_ 指标。

以下 [命令](https://pkg.go.dev/github.com/temporalio/tcld#readme-get-a-migration) 也可用于
监控迁移进度：

```
tcld migration get --id <migration-id>
```

<a id="monitor"></a>
### 切换到云端 {#monitor}

同步过程完成后，Temporal 将翻转自托管和云 Namespace 的角色。此时，
云变为主动，自托管 Namespace 变为被动。然后将 Workflow 从云端复制到
自托管服务器。

以下[命令](https://pkg.go.dev/github.com/temporalio/tcld#readme-perform-handover-during-a-migration)是
用于触发切换：

```
tcld migration handover --id <migration-id> --to-replica-id <to-replica-id>
```

使用该命令时，在移交至Temporal Cloud时将`<to-replica-id>`替换为`cloud`。更换
返回自托管设置时，`<to-replica-id>` 和 `on-prem`。

<a id="handover-to-cloud"></a>
## 第 5 阶段：最终确定 {#handover-to-cloud}

<a id="phase-5-finalize"></a>
### 最终验证 {#phase-5-finalize}

在完成迁移之前使用以下清单：

- 确认Worker可以访问Namespace。通过公共互联网或 [私人连接](/cloud/connectivity)。
- 了解如何在 Temporal Cloud 上访问 Namespace 的指标。
- 监控一般 Workflow 指标（Schedule 启动延迟、启动与完成率、同步匹配率等）。
- 了解 [容量管理](/cloud/capacity-modes) 在 Temporal Cloud 中的工作原理。
- 计划 Worker 调整会话 - Temporal Cloud 与 Temporal Cloud 之间的性能变化自托管集群，可以
  导致意想不到的症状和优化。
- 了解如何联系您的 Temporal 解决方案架构师 (SA) 和客户主管 (AE) 寻求帮助。

<a id="final-validation"></a>
### 确认完成 {#final-validation}

一旦 Namespace 传输到云端并经过验证，迁移就会完成。注意这一步
是最终的，可能无法撤消。执行后，Workflow 从云 Namespace 复制到自托管服务器
已停止。

下面的[命令](https://pkg.go.dev/github.com/temporalio/tcld#readme-confirm-a-migration)用于确认
迁移：

```
tcld migration confirm --id <migration-id>
```

或 [中止](https://pkg.go.dev/github.com/temporalio/tcld#readme-abort-a-migration) 并回滚更改，无需
如果需要，影响您的 Workflow：

```
tcld migration abort --id <migration-id>
```

<a id="confirm-complete"></a>
### 将客户端转移到云端 {#confirm-complete}

有两种选项可将 Temporal CLIents 切换到云。

<a id="transfer-clients-to-cloud"></a>
#### 选项 1（推荐） {#transfer-clients-to-cloud}

部署两组 Temporal CLIent：一组指向您的 Temporal Server，一组指向云 Namespace 端点。这个
是推荐的选项，因为您的 Workflow 在移交期间将继续取得进展，即使您的云
Temporal CLIent 无法访问云（例如由于配置错误）。流程如下：

1. 将基于云的 Temporal CLIent 定向到云 Namespace 端点。最初，这些客户端将连接并
   发送 Poll 请求但不会接收任何任务。
2. 开始迁移。您的自托管 Namespace 处于活动状态，而您的云 Namespace 处于被动（或备用）状态。你的云
   Temporal CLIents 可以开始接收任务，但从云客户端到云 Namespace 的所有请求都将
   自动从云端转发到您的自托管服务器。
3、将Namespace交给云端。您的云 Namespace 变为主动，而您的自托管 Namespace 变为被动。
   来自自托管 Temporal CLIents 的所有请求都会自动从您的服务器转发到云。
4. 完成迁移。您的自托管 Temporal CLIents 将不再从您的服务器接收任何任务，从而允许您
   来阻止这些客户。

<a id="option-1-recommended"></a>
#### 选项 2 {#option-1-recommended}

部署一组 Temporal CLIent 并在迁移期间切换 Namespace 端点。使用此选项，如果您的 Worker
如果在切换过程中配置错误，则 Workflow 可能会停止前进。重要的是
确保所有 Worker 保持与云的连接以避免这种情况。流程如下：

1. 开始迁移。
2. 将您的 Temporal CLIents 切换为指向云 Namespace 端点。来自您的 Temporal CLIents 的请求将
   自动从云端转发到您的服务器。或者，您可以将 Temporal CLIents 切换到云端
   Namespace 切换后的端点。
3、将Namespace交给云端。来自您的 Temporal CLIents 的请求现在将由云提供服务，并且不会被
   转发到您的服务器。
4. 确认迁移完成。

<a id="option-2"></a>
## 附加说明 {#option-2}

<a id="additional-notes"></a>
### 限制 {#additional-notes}

以下是已知的限制。

- 需要 OSS 服务器版本 1.22 或更高版本。请参阅
  [升级](/self-hosted-guide/upgrade-server#upgrade-server) 程序根据需要。
- 历史分片计数必须是 2 的倍数。
- 如果您有多个自托管服务器，并且它们都配置有相同的集群名称（默认为 Temporal
  使用“active”作为集群名称），由于集群的原因，它们无法同时连接到单个迁移服务器
  名称冲突。有 2 个可用选项：
  1. 使用单个迁移服务器一次迁移一台服务器。
  2. 如果需要迁移所有服务器，请创建多个迁移服务器（每个自托管服务器一个）
     同时。
- 如果您在自托管设置中使用多集群复制并且之前曾对 Namespace 进行过故障转移，则
  这可能会影响您自动迁移的资格。具体来说，每当 Global Namespace 之前已
  启用以下限制适用：
  1.初始故障转移版本必须小于或等于1,000,000
  2. 故障转移版本增量必须是 1,000,000 的除数
- OSS通过以下方式支持跨Namespace命令（例如，父子、SignalExternal、CancelExternal）
  `system.enableCrossNamespaceCommands`配置。此配置在 Temporal Cloud 上被禁用。的
  必须禁用 `system.enableCrossNamespaceCommands` 配置，并且必须禁用使用跨 Namespace 调用的代码
  Updated 或在迁移之前删除。

###版本1.22.x-1.23.x的特殊动态配置

Temporal 版本 1.22.x 和 1.23.x 包括对基于流的复制的支持，但默认情况下处于禁用状态。自从
在这些版本中，基于流的复制已被验证为比基于轮询的复制更可靠。
1.22 和 1.23 中仍保留默认值。

当准备在这些版本上进行 S2C 迁移时，请配置以下动态设置以启用基于流的
复制：

```yaml
history.enableReplicationStream:
  - value: true
```

启用此配置将需要重新启动历史记录 Pod。

<a id="limitations"></a>
## 常见问题 {#limitations}

**使用自定义 Search Attribute 为什么很重要？**

自定义 Search Attribute 必须映射到 Temporal Cloud 中的 Namespace。此过程因类型而异
用于 Visibility 的数据存储。

**默认迁移哪些 Workflow？**

打开的 Workflow 始终会迁移。您可以指定已关闭的 Workflow 的日期范围。限制时间范围
关闭 Workflow 将大大减少迁移 Namespace 所需的时间。您的云端Namespace必须
在开始迁移之前配置您所需的保留期。

**我的 Workflow 的保留期很长。这与 Temporal Cloud 兼容吗？**

有时，自托管 [保留期限](/temporal-service/temporal-server#retention-period) 会超出
是 Temporal Cloud 中的 [支持](/cloud/limits#default-retention-period)。在这些情况下，建议使用
[档案的](/temporal-service/archival) 用于存储无法迁移的关闭的 Workflow。一般来说，归档是
建议在较长的保留期内使用，因为额外的数据会给系统的持久层带来压力。

**我在自托管 Temporal 集群中使用有效负载加密。云支持吗？**

是的。如果有效负载已通过 Data Converter 在您的自托管服务器中为 [加密的](/payload-codec#encryption)，则 
它们在迁移期间和之后将保持加密状态。


**我想在迁移过程中启用有效负载加密。支持吗？**

目前不支持在迁移过程中启用负载加密。如果需要加密的话最好
方法是在迁移之前启用它。

<a id="special-dynamic-configuration-for-version-122x-123x"></a>
## 如何收集自托管指标 {#special-dynamic-configuration-for-version-122x-123x}

Temporal Server（版本 > 1.17）提供 [行动](/cloud/pricing#action) 指标。您可以使用 Grafana 来捕获和
显示该指标。使用这个[样本](https://github.com/temporalio/dashboards/blob/master/server/server-general.json)
仪表板作为参考。要确定每秒操作数 (APS)，您可以使用以下 promql 查询。

`sum (rate(action{service_name="frontend"}[1m]))`

`sum (rate(action{service_name="frontend"}[1m]))  by (exported_Namespace)`

第一个 Query 将计算总 APS。第二个将根据 Namespace 隔离指标。

对于 Datadog，以下小部件将计算每秒的操作：

```
{
    "title": "Average Actions per Second",
    "type": "query_value",
    "requests": [
        {
            "formulas": [
                {
                    "formula": "query1"
                }
            ],
            "queries": [
                {
                    "name": "query1",
                    "data_source": "metrics",
                    "query": "sum:io.temporal.server.action.count{$server-name}.as_rate()",
                    "aggregator": "avg"
                }
            ],
            "response_format": "scalar"
        }
    ],
    "autoscale": true,
    "precision": 2
}
```
