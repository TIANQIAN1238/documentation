Temporal 应用程序开发指南的测试部分描述了促进 Workflow 和集成测试的框架。

在 Temporal 的上下文中，您可以创建以下类型的自动化测试：

- **端到端：** 运行 Temporal Server 和 Worker 及其所有 Workflow 和活动；从客户端启动 Workflow 并与之交互。
- **集成：** 端到端和单元测试之间的任何内容。
  - 使用模拟上下文和其他 SDK 导入（通常是网络请求）运行活动。
  - 通过模拟活动运行 Worker，并使用客户端启动 Workflow。
  - 使用模拟 SDK 导入运行 Workflow。
- **单元：** 运行一段 Workflow 或 Activity 代码（函数或方法）并模拟它调用的任何代码。

我们通常建议将大部分测试编写为集成测试。

由于测试服务器支持跳过时间，因此请使用测试服务器与 Worker 进行端到端测试和集成测试。

## 测试活动 {#test-activities}

Activity 可以使用模拟 Activity 环境进行测试，该环境提供了一种模拟 Activity 上下文、监听心跳和取消 Activity 的方法。
此行为允许您通过直接调用 Activity 来单独测试 Activity，而无需创建 Worker 来运行 Activity。

## 测试 Workflows {#test-Workflows}

### 如何模拟活动 {#mock-activities}

在对 Workflow 进行单元测试时模拟 Activity 调用。

当使用 Worker 集成测试 Workflow 时，您可以通过向 Worker 提供模拟 Activity 实现来模拟活动。

**RoadRunner 配置**

要在 PHP 中模拟 Activity，请使用 [RoadRunner 键值存储](https://github.com/spiral/roadrunner-kv) 并将以下行添加到“tests/.rr.test.yaml”文件中。

```yaml
# tests/.rr.test.yaml
kv:
  test:
    driver: memory
    config:
      interval: 10
```

如果您希望能够模拟活动，请使用“Temporal\Testing”Namespace 中的“WorkerFactory”
在你的 PHP Worker 中：

```php
// worker.test.php
use Temporal\Testing\WorkerFactory;

$factory = WorkerFactory::create();
$worker = $factory->newWorker();

$worker->registerWorkflowTypes(MyWorkflow::class);
$worker->registerActivity(MyActivity::class);
$factory->run();
```

然后，在模拟 Activity 的测试中，使用“ActivityMocker”类。

假设我们有以下 Activity：

```php
#[ActivityInterface(prefix: "SimpleActivity.")]
interface SimpleActivityInterface
{
    #[ActivityMethod('doSomething')]
    public function doSomething(string $input): string;
```

要在测试中模拟它，您可以这样做：

```php
final class SimpleWorkflowTestCase extends TestCase
{
    private WorkflowClient $workflowClient;
    private ActivityMocker $activityMocks;

    protected function setUp(): void
    {
        $this->workflowClient = new WorkflowClient(ServiceClient::create('localhost:7233'));
        $this->activityMocks = new ActivityMocker();

        parent::setUp();
    }

    protected function tearDown(): void
    {
        $this->activityMocks->clear();
        parent::tearDown();
    }

    public function testWorkflowReturnsUpperCasedInput(): void
    {
        $this->activityMocks->expectCompletion('SimpleActivity.doSomething', 'world');
        $workflow = $this->workflowClient->newWorkflowStub(SimpleWorkflow::class);
        $run = $this->workflowClient->start($workflow, 'hello');
        $this->assertSame('world', $run->getResult('string'));
    }
}
```

在前面的测试用例中，我们执行以下操作：

1. 在测试的`setUp()`方法中实例化`ActivityMocker`。
2. 在`tearDown()`中每次测试后清除缓存。
3. 模拟 Activity 调用以返回字符串“world”。

要模拟失败，请使用“expectFailure()”方法：

```php
$this->activityMocks->expectFailure('SimpleActivity.echo', new \LogicException('something went wrong'));
```

### 如何跳过时间{#skip-time}

一些长期运行的 Workflow 可以持续数月甚至数年。
实施测试框架允许您的 Workflow 代码跳过时间并在几秒钟内完成测试，而不是 Workflow 指定的时间。

例如，如果您的 Workflow 睡眠了一天，或者 Activity 发生故障且重试间隔较长，则无需等待整个睡眠周期的长度来测试睡眠功能是否起作用。
相反，通过及时向前跳跃并及时完成测试来测试睡眠后发生的逻辑。

大多数 SDK 中包含的测试框架是 Temporal Server 的内存中实现，支持跳过时间。
时间是“TestWorkflowEnvironment”实例的全局属性：跳过时间（自动或手动）适用于所有当前正在运行的测试。
如果不同的测试需要不同的时间行为，请按系列运行测试或使用测试服务器的单独实例运行测试。
例如，您可以并行运行所有具有自动时间跳过功能的测试，然后串行运行所有具有手动时间跳过功能的测试，然后并行运行所有不带时间跳过功能的测试。

#### 设置时间跳过 {#setting-up}

在您选择的 SDK 中设置跳时测试框架。

1. 在 `tests` 文件夹中，创建包含以下内容的 `bootstrap.php`：

```php
declare(strict_types=1);

require __DIR__ . '/../vendor/autoload.php';

use Temporal\Testing\Environment;

$environment = Environment::create();
$environment->start();
register_shutdown_function(fn () => $environment->stop());
```

如果您不想在所有测试中运行测试服务器，则可以添加一个条件以仅在存在“RUN_Temporal_TEST_SERVER”环境变量时启动测试：

```php
if (getenv('RUN_TEMPORAL_TEST_SERVER') !== false) {
    $environment = Environment::create();
    $environment->start('./rr serve -c .rr.silent.yaml --workflow-id tests');
    register_shutdown_function(fn() => $environment->stop());
}
```

2. 将 `bootstrap.php` 和 `Temporal_ADDRESS` 环境变量添加到 `phpunit.xml`：

```xml
<phpunit xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:noNamespaceSchemaLocation="https://schema.phpunit.de/9.3/phpunit.xsd"
         bootstrap="tests/bootstrap.php"
>
    <php>
        <env name="TEMPORAL_ADDRESS" value="127.0.0.1:7233" />
    </php>
</phpunit>
```

3. 将测试服务器可执行文件添加到 `.gitignore`：

```gitignore
temporal-test-server
```

## 如何重播 Workflow Execution {#replay}

回放可重现 Workflow Execution 的准确状态。
您可以从 Event History 的开头重播 Workflow。

仅当 [Workflow Definition](/Workflow-definition) 从确定性角度来看与提供的历史记录兼容时，重播才会成功。

当您测试 Workflow Definition 的更改时，我们建议执行以下操作作为 CI 检查的一部分：

1. 确定被测 Worker 代码将针对哪些 Workflow Type 或 Task Queue（或两者）。
2. 使用 SDK 客户端以编程方式或通过 Temporal CLI，从每个 Task Queue 下载最近打开和关闭的 Workflow 的一组代表性事件历史记录。
3. 通过重播运行事件历史记录。
4. 如果重播期间遇到任何错误，则 CI 失败。

以下是获取和重播事件历史记录的示例：

### 如何更新插入 XPROT004Xs {#upsert-search-attributes}

在以下示例中，事件历史记录从 Temporal 获取，然后重播。
如果 Workflow 是非确定性的，则会抛出“NonDeterministicWorkflowException”。
请注意，这需要启用 [高级 Visibility](/Visibility#advanced-Visibility)。

```php
/**
 * We assume you already have a WorkflowClient and WorkflowReplayer in scope.
 * @var \Temporal\Client\WorkflowClientInterface $workflowClient
 * @var \Temporal\Testing\Replay\WorkflowReplayer $replayer
 */

// Find all workflow executions of type "MyWorkflow" and task queue "MyTaskQueue".
$executions = $workflowClient->listWorkflowExecutions(
    "WorkflowType='MyWorkflow' AND TaskQueue='MyTaskQueue'"
);

// Replay each workflow execution.
foreach ($executions as $executionInfo) {
    try {
        $replayer->replayFromServer(
            workflowType: $executionInfo->type->name,
            execution: $executionInfo->execution,
        );
    } catch (\Temporal\Testing\Replay\Exception\ReplayerException $e) {
        // Handle a replay error.
    }
}
```

在下一个示例中，从 JSON 文件加载 Event History，重播事件的最大数量限制为 42。

```php
$replayer->replayFromJSON(
    workflowType: 'MyWorkflow',
    path: 'history.json',
    lastEventId: 42,  // optional
);
```

您可以使用 PHP 下载 Workflow 历史记录，然后从存储的历史对象中重播它：

```php
$history = $this->workflowClient->getWorkflowHistory(
    execution: $run->getExecution(),
)->getHistory();

(new WorkflowReplayer())->replayHistory($history);
```
