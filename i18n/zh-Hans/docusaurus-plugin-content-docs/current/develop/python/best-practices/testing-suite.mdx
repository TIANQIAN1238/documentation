我们通常建议将大部分测试编写为集成测试。

由于测试服务器支持跳过时间，因此请使用测试服务器与 Worker 进行端到端测试和集成测试。

## 测试框架 {#test-frameworks}

一些 SDK 提供流行测试框架、运行器或库的支持或示例。

用于 Temporal SDK 的 Python 测试的一种推荐框架是 [pytest](https://docs.pytest.org/)，它可以帮助固定装置建立和拆除测试环境，提供有用的测试发现，并使编写参数化测试变得容易。

## 测试活动 {#test-activities}

Activity 可以使用模拟 Activity 环境进行测试，该环境提供了一种模拟 Activity 上下文、监听心跳和取消 Activity 的方法。
此行为允许您通过直接调用 Activity 来单独测试 Activity，而无需创建 Worker 来运行 Activity。

### 运行 Activity {#run-an-Activity}

如果 Activity 引用其上下文，则您需要在单独测试时模拟该上下文。

要在测试中运行 Activity，请使用 [`ActivityEnvironment`](https://python.Temporal.io/Temporalio.testing.ActivityEnvironment.html) 类。

此类允许您在 Activity 上下文中运行任何可调用程序。
使用它来测试代码在各种条件下的行为。

### 聆听心跳 {#listen-toheartbeats}

当 Activity 发送心跳时，请确保您可以在测试代码中看到心跳，以便您可以验证它们。

要在 Activity 中测试 Heartbeat，请使用[`ActivityEnvironment`](https://python.Temporal.io/Temporalio.testing.ActivityEnvironment.html) 类。
此属性设置一个自定义函数，每次在 Activity 中调用“Activity.heartbeat()”函数时都会调用该函数。

```python
@activity.defn
async def activity_with_heartbeats(param: str):
    activity.heartbeat(f"param: {param}")
    activity.heartbeat("second heartbeat")

env = ActivityEnvironment()
heartbeats = []
# Set the `on_heartbeat` property to a callback function that will be called for each Heartbeat sent by the Activity.
env.on_heartbeat = lambda *args: heartbeats.append(args[0])
# Use the run method to start the Activity, passing in the function that contains the Heartbeats and any necessary parameters.
await env.run(activity_with_heartbeats, "test")
# Verify that the expected Heartbeats are received by the callback function.
assert heartbeats == ["param: test", "second heartbeat"]
```

## 测试 Workflows {#test-Workflows}

### 如何模拟活动 {#mock-activities}

在对 Workflow 进行单元测试时模拟 Activity 调用。

当使用 Worker 集成测试 Workflow 时，您可以通过向 Worker 提供模拟 Activity 实现来模拟活动。

为 Worker 提供模拟 Activity 实现。

```python
import uuid
from temporalio.client import Client
from temporalio.worker import Worker

# Import your Activity Definition and real implementation
from hello.hello_activity import (
    ComposeGreetingInput,
    GreetingWorkflow,
    compose_greeting,
)

# Define your mocked Activity implementation
@activity.defn(name="compose_greeting")
async def compose_greeting_mocked(input: ComposeGreetingInput) -> str:
    return f"{input.greeting}, {input.name} from mocked activity!"

async def test_mock_activity(client: Client):
    task_queue_name = str(uuid.uuid4())
    # Provide the mocked Activity implementation to the Worker
    async with Worker(
        client,
        task_queue=task_queue_name,
        workflows=[GreetingWorkflow],
        activities=[compose_greeting_mocked],
    ):
        # Execute your Workflow as usual
        assert "Hello, World from mocked activity!" == await client.execute_workflow(
            GreetingWorkflow.run,
            "World",
            id=str(uuid.uuid4()),
            task_queue=task_queue_name,
        )
```

模拟的 Activity 实现应具有与真实实现相同的签名（包括输入和输出类型）和相同的名称。
当 Workflow 调用 Activity 时，它会调用模拟实现而不是真实实现，从而允许您隔离测试 Workflow。

### 如何跳过时间{#skip-time}

一些长期运行的 Workflow 可以持续数月甚至数年。
实施测试框架允许您的 Workflow 代码跳过时间并在几秒钟内完成测试，而不是 Workflow 指定的时间。

例如，如果您的 Workflow 睡眠了一天，或者 Activity 发生故障且重试间隔较长，则无需等待整个睡眠周期的长度来测试睡眠功能是否起作用。
相反，通过及时向前跳跃并及时完成测试来测试睡眠后发生的逻辑。

大多数 SDK 中包含的测试框架是 Temporal Server 的内存中实现，支持跳过时间。
时间是“TestWorkflowEnvironment”实例的全局属性：跳过时间（自动或手动）适用于所有当前正在运行的测试。
如果不同的测试需要不同的时间行为，请按系列运行测试或使用测试服务器的单独实例运行测试。
例如，您可以并行运行所有具有自动时间跳过功能的测试，然后串行运行所有具有手动时间跳过功能的测试，然后并行运行所有不带时间跳过功能的测试。

#### 自动跳过时间 {#automatic-method}

您可以在您选择的 SDK 中自动跳过时间。
启动一个测试服务器进程，根据需要跳过时间。
例如，在跳时模式下，Timer（包括睡眠和条件超时）会快进，除非活动正在运行。

使用 start_time_skipping() 方法启动测试服务器进程并自动跳过时间。

使用 [`start_local()`](https://python.Temporal.io/Temporalio.testing.WorkflowEnvironment.html#start_local) 方法获得完整的本地 Temporal Server。

对现有 Temporal Server 使用 [`from_client()`](https://python.Temporal.io/Temporalio.testing.WorkflowEnvironment.html#from_client) 方法。

#### 手动跳过时间 {#manual-method}

在您选择的 SDK 中手动跳过时间。

要实现时间跳过，请使用 [`start_time_skipping()`](https://python.Temporal.io/Temporalio.testing.WorkflowEnvironment.html#start_time_skipping) 静态方法。

```python
from temporalio.testing import WorkflowEnvironment

async def test_manual_time_skipping():
    async with await WorkflowEnvironment.start_time_skipping() as env:
        # Your code here
        # You can use the env.sleep(seconds) method to manually advance time
        await env.sleep(3) # This will advance time by 3 seconds
        # Your code here
```

### 在 Workflow 中断言 {#assert-in-Workflow}

`assert` 方法在 Python 和 TypeScript 中可用。

有关 Python 中的断言语句的信息，请参阅 Python 语言参考中的 [`assert`](https://docs.python.org/3/reference/simple_stmts.html#the-assert-statement)。

## 如何重播 Workflow Execution {#replay}

回放可重现 Workflow Execution 的准确状态。
您可以从 Event History 的开头重播 Workflow。

仅当 [Workflow Definition](/Workflow-definition) 从确定性角度来看与提供的历史记录兼容时，重播才会成功。

当您测试 Workflow Definition 的更改时，我们建议执行以下操作作为 CI 检查的一部分：

1. 确定被测 Worker 代码将针对哪些 Workflow Type 或 Task Queue（或两者）。
2. 使用 SDK 客户端以编程方式或通过 Temporal CLI，从每个 Task Queue 下载最近打开和关闭的 Workflow 的一组代表性事件历史记录。
3. 通过重播运行事件历史记录。
4. 如果重播期间遇到任何错误，则 CI 失败。

以下是获取和重播事件历史记录的示例：

要重播 Workflow Executions，请使用 [`replay_Workflows`](https://python.Temporal.io/Temporalio.Worker.Replayer.html#replay_Workflows) 或[`replay_Workflow`](https://python.Temporal.io/Temporalio.Worker.Replayer.html#replay_Workflow) 方法，传递一个或多个事件历史记录作为参数。

在以下示例中（从服务器 v1.18 开始，需要启用 Advanced Visibility），从服务器下载事件历史记录，然后重播。
如果任何重放失败，代码都会引发异常。

```python
workflows = client.list_workflows(f"TaskQueue=foo and StartTime > '2022-01-01T12:00:00'")
histories = workflows.map_histories()
replayer = Replayer(
    workflows=[MyWorkflowA, MyWorkflowB, MyWorkflowC]
)
await replayer.replay_workflows(histories)
```

在下一个示例中，从 JSON 字符串加载单个历史记录：

```python
replayer = Replayer(workflows=[YourWorkflow])
await replayer.replay_workflow(WorkflowHistory.from_json(history_json_str))
```

在这两个示例中，如果 Event History 是不确定的，则会引发错误。
您可以通过将 `fail_fast` 选项设置为 `false` 来选择等待，直到使用 `replay_Workflows` 重播所有历史记录。

:::note

如果 Workflow 历史记录由 [Temporal Web UI](/web-ui) 或通过 [Temporal CLI](/cli) 导出，您可以通过“json.load()”函数将 JSON 文件历史记录对象作为 JSON 字符串或 Python 字典传递，该函数接受一个文件对象并返回 JSON 对象。

:::tip
当直接从服务器获取事件历史记录或导出事件历史记录时，请注意数据可以是 Protobuf 编码（“字节”）。然而，“Replayer”通常使用解码的历史记录（如“dict”）。

如果您在重放期间遇到与“dict”与“bytes”不匹配相关的“TypeError”异常，请确保您的 Event History 在传递到重放器之前已正确解码。
:::
