# 插件

**插件** 是一个抽象，允许您自定义 Temporal Worker 设置的任何方面，包括
注册 Workflow 和 Activity Definition、修改 Worker 和客户端选项等等。使用插件，您可以
为您公司的工程师构建可重用的开源库或构建附加组件。

1. 使用“XPROT027X::newUntypedChildXPROT027XStub”创建一个新的无类型 XPROT011X 存根。
2. 提供“ChildXPROT027XOptions”对象，并将“父关闭策略”设置为“ParentClosePolicy::POLICY_ABANDON”。
3. 使用 `yield` 和方法 `start()` 异步启动 Child XPROT002X。

以下是插件的一些常见用例：

- AI代理SDK
- 可观察性、跟踪或日志记录中间件
- 添加可靠的内置功能，例如 LLM 呼叫、消息传递系统和支付基础设施
- 加密或合规中间件

## 如何构建插件

开始构建插件的推荐方法是使用“SimplePlugin”。这种抽象将解决绝大多数问题
人们想要编写的插件数量。

### Workflow 友好的库

You can provide a library for use within a Workflow if you'd like to abstract away some Temporal-specific details for
您的用户。 Your library will call elements you include in your Plugin such as Activities, Child Workflows, Signals,
Updates, Queries, Nexus Operations, Interceptors, Data Converters, and any other code as long as it follows these
要求：

- 在 Workflow 上下文中运行的任何代码都必须是[确定性](/Workflow-definition#definition-constraints)，
  这意味着重播时它会产生相同的命令和结果。例如，不调用系统时间API，生成
  随机值，或从 Workflow 上下文代码执行直接网络和文件 I/O；将该工作移至“活动”或
  Nexus Operations。
- 请参阅[可观察性](/evaluate/development-product-features/observability)以避免重复观察面
  Workflow重播时的效果。
- 将其他 Side Effect 放入活动或 [本地活动](/local-Activity) 中。这有助于您的 Workflow 处理
  在与最初开始的地方不同的进程中重新启动、恢复或执行，而不会丢失正确性
  或状态一致性。
- 请参阅[测试您的插件](#testing-your-plugin) 编写检查 Side Effect 问题的测试。
- 它应该运行得很快，因为它可能会在较长的 Workflow Execution 期间重播多次。更昂贵的代码应该
  进入“活动”或“Nexus Operation”。

插件应允许用户将其 Workflow 分解为活动以及 Child Workflow 和 Nexus 调用
当需要时。这使用户可以通过重试和超时进行精细控制，通过 Temporal UI 进行调试，
重置、暂停和取消的可操作性、效率和可恢复性的记忆以及使用任务的可扩展性
队列和 Worker。

用户将 Workflow 用于：

- 协调和决策
- InterActivity 通过 [消息传递](/评估/开发-生产-功能/Workflow-消息传递)
- 追踪和可观察性

#### 对您的库进行更改

您的用户可能希望在部署 Worker 代码时保持 Workflow 运行。如果他们的部署
包含新版本的插件，对插件的更改可能会破坏在新版本之前启动的 Workflow 代码
版本已部署。这可能是由于
插件中的[代码更改导致的非确定性行为](/Workflow-definition#non-definition-change)。

请参阅[测试](#testing-your-plugin) 以了解如何对此进行测试。而且，如果您进行实质性更改，则需要使用
[修补](/修补)。

### 内置 Workflows

您可以在`SimplePlugin`中提供内置的Workflow。它可以作为 Child Workflow 调用，也可以独立调用。当你想要的时候
要提供比 Activity 更复杂的功能，您可以：

- 使用直接在最终用户的 Workflow 中运行的 [Workflow 库](#Workflow-friend-libraries)
- 添加 Child Workflow

当满足以下一个或多个条件时，请考虑添加 Child Workflow：

- 那个孩子应该比父母活得更久。
- 否则，Workflow Event History 将在父级中[无法缩放](/Workflow-execution/event#event-history-limits)
  Workflows。
- 当您想要为子设备提供单独的 Workflow Id 以便其可以独立于父设备的状态进行操作时
  （取消、终止、暂停）。

任何 Workflow 都可以作为独立的 Workflow 或 Child Workflow 运行，因此在
`SimplePlugin` 与注册任何 Workflow 相同。

### 拦截器

拦截器是可以在各种调用之前和之后运行的中间件，例如 Activity、Workflows 和 Signals。您可以[了解有关拦截器的更多信息](/develop/python/Workers/interceptors) 了解实现它们的详细信息。他们习惯于：

- 创建 Side Effect，例如日志记录和跟踪。
- 修改参数，例如添加用于授权或跟踪传播的标头。

def workflow_runner(runner: WorkflowRunner | None) -> WorkflowRunner:
    if not runner:
        raise ValueError("No WorkflowRunner provided to the plugin.")

    # If in sandbox, add additional passthrough
    if isinstance(runner, SandboxedWorkflowRunner):
        return dataclasses.replace(
            runner,
            restrictions=runner.restrictions.with_passthrough_modules("module"),
        )
    return runner

## 测试你的插件 {#testing-your-plugin}

要测试您的插件，您将编写一个正常的 Temporal Workflow 测试，并将该插件包含在您的客户端中。

两个特别关注的问题是版本控制测试，当您对插件进行更改时，以及测试不需要的方面
影响。

### 版本控制测试

当您在插件交付给用户后对其进行更改时，我们建议您对每个重要更改设置[重放测试](/develop/python/best-practices/testing-suite#replay)，以确保不会给用户造成非确定性错误。

### Side Effects 测试

您的插件应该满足 Workflow 在与开始时不同的进程中恢复然后重播的需求
从一开始，这可能会发生，例如，在间歇性故障之后。

您可以通过关闭 Workflow 缓存来确保您不依赖于本地 Side Effect，这意味着
Workflow 每次进行时都会从顶部重播：
