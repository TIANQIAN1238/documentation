由于 Temporal 中的 Workflow Execution 可以运行很长时间（有时是几个月甚至几年），因此通常需要对 Workflow Definition 进行更改，即使特定的 Workflow Execution 正在运行也是如此。

Temporal Platform 要求 Workflow 代码是[确定性](/Workflow-definition#definition-constraints)。
如果您对 Workflow 代码进行的更改会导致重播时出现不确定行为，则您需要使用我们的版本控制方法之一来优雅地对正在运行的 Workflow 进行 Update。
通过版本控制，您可以修改 Workflow Definition，以便新的执行使用 Updated 代码，而现有的执行继续运行原始版本。
您可以使用两种主要的版本控制方法：

- [Worker 版本控制](/生产部署/Worker-部署/Worker-版本控制)。 Worker 版本控制功能允许您标记 Worker 并以编程方式在版本化部署中推出它们，以便旧的 Worker 可以运行旧的代码路径，而新的 Worker 可以运行新的代码路径。
- [使用补丁进行版本控制](#patching)。此方法的工作原理是向与特定修订版相关的代码添加分支。它将代码更改应用于新的 Workflow Execution，同时避免对正在进行的 Workflow Execution 进行破坏性更改。

:::danger
2025 年之前对实验性 Worker 版本控制方法的支持将于 2026 年 3 月从 Temporal Server 中删除。请参阅[最新 Worker 版本控制文档](/Worker-versioning) 以获取指导。如果需要，您仍然可以参考 [Worker Versioning Legacy](/develop/java/Worker-versioning-legacy) 文档。
:::

## Worker 版本控制

Temporal 的 [Worker 版本控制](/生产部署/Worker-deployments/Worker-versioning) 功能允许您标记 Worker 并以编程方式在部署版本中推出它们，以便旧的 Worker 可以运行旧的代码路径，新的 Worker 可以运行新的代码路径。这样，您就可以将 Workflow 固定到特定版本，从而无需修补。

## 使用补丁进行版本控制 {#patching}

### 使用 GetVersion 进行修补

补丁在 Workflow 中定义特定更改的逻辑分支，类似于功能标志。
它将代码更改应用于新的 Workflow Execution，同时避免对正在进行的 Workflow Execution 进行破坏性更改。
当您想要进行可能影响现有 Workflow Execution 的实质性代码更改时，请创建补丁。请注意，无需修补 [固定 Workflows](/Worker-版本控制)。

考虑以下 Workflow Definition：

```java
public void processFile(Arguments args) {
    String localName = null;
    String processedName = null;
    try {
        localName = activities.download(args.getSourceBucketName(), args.getSourceFilename());
        processedName = activities.processFile(localName);
        activities.upload(args.getTargetBucketName(), args.getTargetFilename(), processedName);
    } finally {
        if (localName != null) { // File was downloaded.
            activities.deleteLocalFile(localName);
        }
        if (processedName != null) { // File was processed.
            activities.deleteLocalFile(processedName);
        }
    }
}
```

想象一下，您想要通过添加另一个 Activity 来计算文件校验和来修改此 Workflow。
如果现有的 Workflow Execution 由 Workflow 代码的原始版本启动，其中没有“calculateChecksum()”Activity，然后在添加了该 Activity 的新 Worker 上恢复运行，则服务器端 Event History 将不同步。
这将导致 Workflow 因不确定性错误而失败。

要解决此问题，您可以使用“Workflow.GetVersion()”来修补您的 Workflow：

```java
public void processFile(Arguments args) {
    String localName = null;
    String processedName = null;
    try {
        localName = activities.download(args.getSourceBucketName(), args.getSourceFilename());
        processedName = activities.processFile(localName);
        int version = Workflow.getVersion("checksumAdded", Workflow.DEFAULT_VERSION, 1);
        if (version == Workflow.DEFAULT_VERSION) {
            activities.upload(args.getTargetBucketName(), args.getTargetFilename(), processedName);
        } else {
            long checksum = activities.calculateChecksum(processedName);
            activities.uploadWithChecksum(
                args.getTargetBucketName(), args.getTargetFilename(), processedName, checksum);
        }
    } finally {
        if (localName != null) { // File was downloaded.
            activities.deleteLocalFile(localName);
        }
        if (processedName != null) { // File was processed.
            activities.deleteLocalFile(processedName);
        }
    }
}
```

当为新的 Workflow Execution 运行“Workflow.GetVersion()”时，它会在 Event History 中记录一个标记，以便所有未来对此更改 id 的“GetVersion”调用（示例中为“checksumAdded”）在此 Workflow Execution 上将始终返回给定的版本号，在示例中为“1”。

在版本 1 之前的所有 Workflow Execution 保留保留后，您可以删除该版本的代码。

```java
public void processFile(Arguments args) {
    String localName = null;
    String processedName = null;
    try {
        localName = activities.download(args.getSourceBucketName(), args.getSourceFilename());
        processedName = activities.processFile(localName);
        // getVersion call is left here to ensure that any attempt to replay history
        // for a different version fails. It can be removed later when there is no possibility
        // of this happening.
        Workflow.getVersion("checksumAdded", 1, 1);
        long checksum = activities.calculateChecksum(processedName);
        activities.uploadWithChecksum(
            args.getTargetBucketName(), args.getTargetFilename(), processedName, checksum);
    } finally {
        if (localName != null) { // File was downloaded.
            activities.deleteLocalFile(localName);
        }
        if (processedName != null) { // File was processed.
            activities.deleteLocalFile(processedName);
        }
    }
}
```

### 在 Event History 中添加对版本化 Workflow Visibility 的支持

当您调用“getVersion”时，SDK 会在历史记录中记录一个“UpsertWorkflowSearchAttribute”事件。
这会添加一个名为“TemporalChangeVersion”的 [Search Attribute](/search-attribute#default-search-attribute)，允许您根据版本过滤 Workflow。

#### 启用自动更新插入 (Java SDK v1.30.0+)

从 Java SDK v1.30.0 开始，您可以通过在“WorkflowImplementationOptions”中设置“enableUpsertVersionSearchAttributes”来启用自动“TemporalChangeVersion”更新插入：

```java
WorkflowImplementationOptions options = WorkflowImplementationOptions.newBuilder()
    .setEnableUpsertVersionSearchAttributes(true)
    .build();

worker.registerWorkflowImplementationTypes(options, YourWorkflowImpl.class);
```

启用后，每次调用“Workflow.getVersion”都会自动更新插入“TemporalChangeVersion”Search Attribute，其中包含“<Change ID>-<Version>”条目的关键字列表。

:::note

此选项向后兼容，并且可以在运行的 Workflow 上安全启用。
但是，一旦启用，您将无法回滚到不支持此选项的 SDK 版本。

:::

#### 手动更新插入

如果您使用的是 v1.30.0 之前的 Java SDK 版本，或者更喜欢自己控制 upsert，则可以手动设置该属性。

```java
import io.temporal.common.SearchAttributeKey;

public static final SearchAttributeKey<List<String>> TEMPORAL_CHANGE_VERSION =
    SearchAttributeKey.forKeywordList("TemporalChangeVersion");
```

当您调用“getVersion”时设置此属性：

```java
int version = Workflow.getVersion("MovedThankYouAfterLoop", Workflow.DEFAULT_VERSION, 1);

if (version != Workflow.DEFAULT_VERSION) {
  Workflow.upsertTypedSearchAttributes(TEMPORAL_CHANGE_VERSION
      .valueSet(Arrays.asList("MovedThankYouAfterLoop-" + version)));
}
```

对于多个“getVersion”调用，收集所有版本更改并设置一次属性：

```java
List<String> list = new ArrayList<String>();
int versionOne = Workflow.getVersion("versionOne", Workflow.DEFAULT_VERSION, 1);
int versionTwo = Workflow.getVersion("versionTwo", Workflow.DEFAULT_VERSION, 1);
if (versionOne != Workflow.DEFAULT_VERSION) {
   list.add("versionOne-" + versionOne);
}
if (versionTwo != Workflow.DEFAULT_VERSION) {
   list.add("versionTwo-" + versionTwo);
}
Workflow.upsertTypedSearchAttributes(TEMPORAL_CHANGE_VERSION.valueSet(list));
```

修补允许您对当前运行的 Workflow 进行更改。
它是一种引入兼容更改而不引入非确定性错误的强大方法。

### Workflow 转换

要了解修补为何有用，演示剪切整个 Workflow 会很有帮助。

由于不兼容的更改仅影响相同类型的打开的 Workflow Execution，因此您可以通过在进行更改时创建全新的 Workflow 来避免确定性错误。
为此，您可以复制 Workflow Definition 函数，为其指定不同的名称，然后将这两个名称注册到您的 Worker。

例如，您可以将“PizzaWorkflow”复制为“PizzaWorkflowV2”：

```java
import io.temporal.workflow.WorkflowInterface;
import io.temporal.workflow.WorkflowMethod;

@WorkflowInterface
public interface PizzaWorkflow {

  @WorkflowMethod
  public OrderConfirmation pizzaWorkflow(PizzaOrder order);
}

public class PizzaWorkflowImpl{

  @Override
  public OrderConfirmation pizzaWorkflow(PizzaOrder order){
      // implementation code omitted for this example
  }
}

@WorkflowInterface
public interface PizzaWorkflowV2 {

  @WorkflowMethod
  public OrderConfirmation pizzaWorkflow(PizzaOrder order);
}

public class PizzaWorkflowImplV2 implements PizzaWorkflowV2{

  @Override
  public OrderConfirmation pizzaWorkflow(PizzaOrder order){
      // implementation code omitted for this example
  }
}
```

有必要创建一个单独的接口，因为一个 Workflow 接口只能有一个 Workflow 方法。

然后，您需要 Update、Worker 配置以及任何其他标识符字符串来注册两个 Workflow Type：

```java
worker.registerWorkflowImplementationTypes(PizzaWorkflowImpl.class);
worker.registerWorkflowImplementationTypes(PizzaWorkflowImplV2.class);
```

此方法的缺点是，它需要您复制代码并向 Update 复制用于启动 Workflow 的任何命令。
随着时间的推移，这可能变得不切实际。
此方法也不提供对任何仍在运行的 Workflow 进行版本控制的方法——它本质上只是一个转换，与修补不同。
