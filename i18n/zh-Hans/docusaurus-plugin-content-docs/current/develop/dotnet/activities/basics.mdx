## 开发 Activity {#develop-Activity}

Workflow 所做的主要事情之一是协调活动的执行。
Activity 是一种正常的方法执行，旨在执行单个、明确定义的操作（短期或长期运行），例如 Query 数据库、调用第三方 API 或对媒体文件进行转码。
Activity 可以与 Temporal Platform 之外的世界交互，或使用 Temporal CLIent 与 Temporal 服务交互。
为了使 Workflow 能够执行 Activity，我们必须定义 [Activity Definition](/Activity-definition)。

您可以使用方法中“Temporalio.Activities”Namespace 中的“[Activity]”属性来开发 Activity Definition。
要将方法注册为具有自定义名称的 Activity，请使用属性参数，例如 `[Activity("your-Activity")]`。
否则，Activity 名称是非限定方法名称（如果方法是异步的，则没有“Async”后缀）。

活动可以是异步的或同步的。

```csharp
using Temporalio.Activities;

public class MyActivities
{
    // Activities can be async and/or static too. We just demonstrate instance methods since many
    // use them that way.
    [Activity]
    public string MyActivity(MyActivityParams input) =>
        $"{input.Greeting}, {input.Name}!";
}
```

[Activity Definition](/Activity-definition) 可以支持的参数总数没有明确的限制。
但是，最终编码到 gRPC 消息有效负载中的数据的总大小是有限制的。

单个参数的最大大小限制为 2 MB。
gRPC 消息的总大小（包括所有参数）限制为最大 4 MB。

另外，请记住，所有有效负载数据都记录在 [Workflow Execution Event History](/Workflow-execution/event#event-history) 中，大型事件历史记录可能会影响 Worker 性能。
这是因为整个 Event History 可以通过 [Workflow Task](/tasks#Workflow-task) 传输到 Worker 进程。

某些 SDK 要求您传递上下文对象，而其他 SDK 则不需要。
当涉及到您的应用程序数据（即序列化并编码到有效负载中的数据）时，我们建议您使用单个对象作为包装传递给活动的应用程序数据的参数。
这样您就可以更改传递到 Activity 的数据，而不会破坏方法签名。

Activity 参数是具有“[Activity]”属性的方法的方法参数。
这些可以是 Temporal 可以转换的任何数据类型，包括记录。
从技术上讲，这可以是多个参数，但 Temporal 强烈鼓励使用包含所有输入字段的单个参数。
