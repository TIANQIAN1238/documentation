如果发生事故或中断，Temporal 会自动<ToolTipTerm term="fail over" src="failover" /> 将您的 Namespace 从主服务器转移到副本服务器。
这使得 Workflow Execution 能够以最小的中断或数据丢失继续运行。
您还可以根据您的情况监控或测试来[手动启动故障转移](/cloud/high-availability/failovers)。

将控制权从副本返回到主副本称为 <ToolTipTerm term="failback" />。
Temporal 管理的故障转移后，一旦 Temporal 运行正常，就会自动故障恢复到原始区域。
有关自动和手动故障恢复选项的详细信息，请参阅[通过故障恢复返回到主数据库](#failbacks)。

## 故障转移

有时，Namespace 可能会因意外事件而暂时不可用。
Temporal Cloud 使用定期运行状况检查来检测这些问题。

### 健康检查

Temporal Cloud 监控错误率、延迟和基础设施问题，例如请求超时。
如果发现指标超出允许阈值的不健康状况，Temporal 会自动将主服务器切换到副本服务器。
在大多数情况下，副本不受该问题的影响。
此过程称为故障转移。

### 自动故障转移

故障转移可防止数据丢失和应用程序中断。
现有的 Workflow 继续运行，随着事件得到解决，新的 Workflow 开始运行。

Temporal Cloud 自动处理故障转移，确保连续性，无需手动干预。
事件解决后，Temporal Cloud 会自动执行[故障恢复](#failbacks)，将 Workflow Execution 处理转移回原始区域。

<CaptionedImage src="/img/cloud/high-availability/failover.png" title="On failover, the replica becomes active and the Namespace endpoint directs access to it." />

为了更好地控制故障转移过程，您可以[禁用自动故障转移](/cloud/high-availability/failovers#disabling-Temporal-initerated)。

:::tip

您可以使用 Web UI、tcld CLI 或 Cloud Ops API 手动触发故障转移来测试具有高可用性功能的 Namespace 的故障转移。
Terraform 提供程序不支持触发故障转移。
在大多数情况下，我们建议您让 Temporal 为您处理故障转移。

故障转移后，请注意以下几点：

- 使用多区域 Namespace 时，您的 CNAME 可能会更改。
  例如，它可能从 aws-us-west-1.region.tmprl.cloud 切换到 aws-us-east-1.region.tmprl.cloud。
  此更改不会影响同区域的 Namespace。

- 您的 Namespace 端点_不会改变_。
  如果故障转移前为“my_Namespace.my_account.tmprl.cloud:7233”，则故障转移后将为“my_Namespace.my_account.tmprl.cloud:7233”。

:::

### 故障转移过程 {#failover-process}

Temporal的自动故障转移过程的工作原理如下：

- 在正常操作期间，主服务器将操作和元数据异步复制到其副本，使它们保持同步。
- 如果主数据库不可用，Temporal 会通过运行状况检查检测问题。
  它使用其可用的[故障转移方案](#scenarios)之一自动切换到副本。
- 副本接管活动角色并成为主副本。
  运营继续进行，干扰最小。
- 当原始主设备恢复时，角色可以切换回（默认情况下故障恢复）或保持原样，具体取决于您的 Namespace 设置。
  具有故障转移和故障恢复功能的自动角色切换可最大限度地减少停机时间，从而实现一致的可用性。

:::info

Namespace 故障转移（Update 是 Namespace 记录中的“活动区域”字段）是元数据 Update。
此 Update 通过 Namespace 元数据机制进行复制。

:::

## 故障转移场景 {#scenarios}

Temporal Cloud 故障转移机制支持多种执行 Namespace 故障转移的模式。
这些模式包括优雅的故障转移（“切换”）、强制故障转移和混合模式。
混合模式是 Temporal Cloud 的默认 Namespace 行为。

### 优雅的故障转移（切换）{#graceful-failover}

在此模式下，Temporal Cloud 完全处理并耗尽复制任务。
Temporal Cloud 在故障转移之前暂停到 Namespace 的流量。
优雅的故障转移可以防止进度丢失并避免数据冲突。

在正常故障转移期间，Namespace 可能会经历短暂的不可用状态。 
此持续时间可由调用者限制，默认为 10 秒。 
如果系统在此超时内无法达到一致状态，则故障转移尝试将中止，Namespace 将恢复到之前的状态。
在此不可用期间：

- 现有的 Workflow 停止进度。
- Temporal Cloud 返回“服务不可用错误”。
  Temporal SDK 可重试此错误。
- 不会发生状态转换，并且不会分派任务。
- 像 start/Signal Workflow 这样的用户请求被拒绝。
- 切换期间操作暂停。

这种模式更注重一致性而不是可用性。

### 强制故障转移 {#forced-failover}

在此模式下，Namespace 会立即在副本中激活。
由于复制滞后而未复制的事件在到达新的活动 Namespace 时会经历冲突解决。

此模式优先考虑_可用性_而不是一致性。

### 混合故障转移模式 {#hybrid-failover}

虽然为了一致性而首选优雅的故障转移，但它们并不总是实用。
Temporal Cloud 的混合故障转移模式（默认模式）将初始优雅故障转移尝试限制为 10 秒或更短。

在此期间：

- 现有的 Workflow 停止进度。
- Temporal Cloud 返回“服务不可用错误”，由 SDK 重试。

如果优雅的方法无法解决问题，Temporal Cloud 会自动切换到强制故障转移。

该策略平衡了_一致性_和_可用性_要求。

### 场景总结

|故障转移场景|特点 |
| ---------------------------- | ------------------------------------------------------- |
|优雅的故障转移（切换）|倾向于_一致性_而不是可用性。                 |
|强制故障转移 |优先考虑_可用性_而不是一致性。            |
|混合故障转移模式|平衡_一致性_和_可用性_要求。 |

## 网络分区

在任何时候，只有主数据库或副本数据库处于活动状态。
唯一的例外发生在[网络分区](https://en.wikipedia.org/wiki/Network_partition)，即网络分裂成单独的子网时。
如果发生这种情况，您可以将副本提升为活动状态。
**警告：** 这会暂时使两个区域都处于活动状态。
解决网络分区并恢复隔离域/区域之间的通信后，冲突解决算法将确定主节点或副本是否保持活动状态。

:::tip

在传统的主动/主动复制中，多个节点同时服务请求并接受写入，从而确保强同步数据一致性。
相比之下，对于具有高可用性功能的 Temporal Cloud Namespace，只有主服务器在任何给定时间接受请求和写入。
Workflow 历史事件首先写入主服务器，然后异步复制到副本，确保副本保持同步。

:::

## 冲突解决 {#conflict-resolution}

具有副本的 Namespace 依赖于异步事件复制。
由于 <ToolTipTerm term="replication lag" />，对主数据库所做的 Update 可能不会立即反映在副本中，特别是在故障转移期间。
如果发生非正常故障转移，复制滞后可能会导致 Workflow 进度暂时受挫。

当 Retry Policy 的[最大尝试次数](https://docs.Temporal.io/retry-policies#maximum-attempts) 设置为 0 时，未复制的 Namespace 可以配置为为活动执行提供_at-most-once_语义。
高可用性 Namespace 为活动的执行提供“至少一次”语义。
已完成的活动可能会在新活动的 Namespace 中重新分派，从而导致重复执行。

当故障转移后新活动副本中的 Workflow Execution 为 Updated 时，无法直接应用故障转移后到达的先前活动 Namespace 的事件。
至此，Temporal Cloud 已经分叉了 Workflow 历史。

故障转移后，Temporal Cloud 创建新的执行分支历史记录，并开始其 <ToolTipTerm term="conflict resolution"/> 进程。
Temporal 服务确保 Workflow 历史记录保持有效，并且可在故障转移后或冲突解决后由 SDK 重播。
此功能对于确保 Workflow Execution 继续前进而不丢失进度以及保持复制的一致性至关重要，即使在导致复制中断的事件期间也是如此。


## 执行手动故障转移 {#triggering-failovers}

对于某些用户来说，Temporal 的自动运行状况检查和故障转移无法提供足够的细微差别和控制。
因此，您可以根据自己的自定义警报手动触发故障转移并用于测试目的。
本节解释了如何以及之后会发生什么。

:::warning Check Your Replication Lag

在启动故障转移之前，请务必检查 <ToolTipTerm term="replication lag" />。
当存在明显的复制滞后时，强制故障转移更有可能回滚 Workflow 进度。

:::

### 触发故障转移 {#manual-failovers}

您可以使用 Temporal Cloud Web UI、tcld CLI 或 Cloud Ops API 手动触发故障转移，具体取决于您的偏好和设置。

:::info Terraform not supported

[Temporal Cloud Terraform 提供程序](https://registry.terraform.io/providers/Temporalio/Temporalcloud/latest) 不支持触​​发故障转移。
您必须使用 Web UI、tcld CLI 或 Cloud Ops API。

:::

以下说明概述了每种方法的步骤：

<Tabs>

<TabItem value="webui" label="Web UI">

1. 在Temporal Cloud Web UI上访问[Namespace页面](https://cloud.Temporal.io/Namespaces)。
1. 导航到 Namespace 详细信息页面，然后从菜单中选择 **触发故障转移** 选项。
1. 确认您的操作。
   确认后，Temporal 启动故障转移。

</TabItem>

<TabItem value="tcldcli" label="tcld">

要手动触发故障转移，请在终端中运行以下命令：

```
tcld namespace failover \
    --namespace <namespace_id>.<account_id> \
    --region <target_region>
```

“<target_region>”必须是 Namespace 具有准备进行故障转移的副本（副本状态为“已激活”）的区域的名称（例如：“us-east-1”）。

如果使用带有“--api-key”标志的 API 密钥身份验证，则必须将其直接添加到 tcld 命令之后和“Namespace 故障转移”之前。

</TabItem>

<TabItem value="ops-api" label="Cloud Ops API">

您可以使用 [Cloud Ops API](/ops) 以编程方式触发故障转移。
该 API 可通过 HTTP 和 gRPC 获取。

**使用 HTTP**

向 [`FailoverNamespaceRegion`](https://saas-api.tmprl.cloud/docs/httpapi.html#tag/high-availability/POST/cloud/Namespaces/{Namespace}/failover-region) 端点发送 POST 请求：

```
POST https://saas-api.tmprl.cloud/cloud/namespaces/<namespace>/failover-region
```

请求正文：

```json
{
  "region": "<target_region>",
  "asyncOperationId": "<optional_async_operation_id>"
}
```

- `region`（必需）：要故障转移到的区域的[区域代码](/cloud/regions)。必须是 Namespace 具有处于“已激活”副本状态的副本的区域，表示该副本已准备好进行故障转移。示例：`aws-us-east-1`
- `asyncOperationId`（可选）：用于跟踪异步操作的用户定义 ID。如果没有设置，服务器会分配一个。

**使用gRPC**

将 FailoverNamespaceRegion 与 RPC 结合使用[`FailoverNamespaceRegionRequest`](https://buf.build/Temporalio/cloud-api/docs/main:Temporal.api.cloud.cloudservice.v1#Temporal.api.cloud.cloudservice.v1.FailoverNamespaceRegionRequest):

```protobuf
message FailoverNamespaceRegionRequest {
    // The namespace to failover.
    string namespace = 1;
    // The id of the region to failover to.
    // Must be a region that the namespace is currently available in.
    string region = 2;
    // The id to use for this async operation - optional.
    string async_operation_id = 3;
}
```

这两种方法都会返回一个 [`FailoverNamespaceRegionResponse`](https://buf.build/Temporalio/cloud-api/docs/main:Temporal.api.cloud.cloudservice.v1#Temporal.api.cloud.cloudservice.v1.FailoverNamespaceRegionResponse)，其中包含可用于跟踪故障转移状态的异步操作。

</TabItem>

</Tabs>

一旦故障转移异步操作成功返回，Namespace 将进行故障转移。
Temporal 管理故障转移 Workflow 的重试。
在极少数情况下，内部错误导致故障转移无法完成，Temporal 待命团队会自动传呼进行干预并强制故障转移完成。

Temporal 将主服务器故障转移到副本服务器。
有关如何以及何时发生故障恢复的详细信息，请参阅[通过故障恢复返回到主数据库](#failbacks)。

### 故障转移后事件信息 {#info}

任何故障转移后，无论是由您还是由 Temporal 触发，[Temporal Cloud Web UI](https://cloud.Temporal.io/Namespaces)（在 Namespace 详细信息页面上）和审核日志中都会出现一个事件。
故障转移的审核日志条目使用“操作”：“FailoverNamespace”事件。
故障转移后，副本将变为活动状态，接管隔离域或区域。

您无需实时监控Temporal Cloud的故障转移响应。
每当发生故障转移事件时，Temporal Cloud [通过电子邮件通知您](/cloud/notifications#admin-notifications)

### 通过故障恢复返回到主数据库 {#failbacks}

在 Temporal 管理的（自动）故障转移后，Temporal Cloud 在运行正常后会自动故障恢复到原始区域。
请关注 [Temporal 的状态页面](https://status.Temporal.io) 了解 Update 的原始区域运行状况。

#### Temporal 管理的故障转移后

当Temporal因断电而触发自动故障转移时，一旦区域恢复，Temporal也会触发自动故障恢复到原始区域。
您无需采取任何行动。

如果您更喜欢自己管理故障恢复，您有两种选择：

- **选择退出自动故障恢复（手动管理故障恢复）：**
  在 Namespace 上[禁用 Temporal 管理的故障转移](#disabling-Temporal-initiated)。
  当您准备好故障恢复到原始区域时，[触发故障转移](#manual-failovers)到该区域，然后重新启用 Temporal 托管的故障转移。

- **永久留在新区域（“故障转发”）：**
  [触发故障转移](#manual-failovers) 到已处于活动状态的区域。
  这告诉 Temporal，只要新区域健康，您就希望将其视为主要区域。
  Temporal 管理的自动故障转移保持启用状态，因此，如果新区域出现中断，Temporal 仍会为您提供保护。

#### 用户触发的故障转移后

如果您在中断期间自行触发故障转移（而不是依赖 Temporal 管理的故障转移），Temporal 将_不会_自动为您进行故障回复。
当原始区域运行正常时，您必须[触发故障转移](#manual-failovers)回到原始区域。
监控 [Temporal 的状态页面](https://status.Temporal.io) Update 的区域运行状况。

自动故障恢复仅在 Temporal 管理的（自动）故障转移后可用。

#### 如何检查您的Namespace是否会自动故障恢复

如果您不确定 Namespace 是否会自动故障恢复，请检查 Namespace 详细信息页面上的 Temporal Cloud Web UI 中的故障转移列表：

- 如果最近的故障转移是 **Temporal 触发的**，则当原始区域运行状况良好时，Temporal 将自动故障恢复到 Namespace。
- 如果最近的故障转移是**用户触发的**，则 Namespace 将_不会_自动故障恢复。您必须自己触发故障回复。

## 禁用 Temporal 启动的故障转移 {#disabling-Temporal-initied}

当您将副本添加到 Namespace 时，如果发生事件或中断，Temporal Cloud 会自动将 Namespace 故障转移到其副本。
_这是推荐的默认选项。_

但是，如果您希望禁用 Temporal 启动的故障转移并处理您自己的故障转移，则可以按照以下说明进行操作：

<Tabs>

<TabItem value="webui" label="Web UI">

1. 导航至 Temporal Cloud 中的 Namespace 详细信息页面。
1. 选择“禁用 Temporal 启动的故障转移”选项。

</TabItem>

<TabItem value="tcldcli" label="tcld">

要禁用 Temporal 启动的故障转移，请在终端中运行以下命令：

```
tcld namespace update-high-availability \
    --namespace <namespace_id>.<account_id> \
    --disable-auto-failover=true
```

如果使用带有“--api-key”标志的 API 密钥身份验证，则必须将其直接添加在 tcld 命令之后和“Namespace Update-high-availability”之前

</TabItem>

</Tabs>

Temporal Cloud 禁用其运行状况检查启动的故障转移。
要恢复默认行为，请取消选择 WebUI 中的选项或在 CLI 命令中将“true”更改为“false”。

## 最佳实践：Worker 和故障转移 {#Worker}

启用 Namespace 的高可用性不需要特定的 Worker 配置。
该过程对于 Worker 来说是不可见的。
当 Namespace 故障转移到副本时，Temporal 协调的 DNS 重定向可确保您现有的 Worker 继续轮询 Namespace 而不会中断。

当 Namespace 故障转移到不同区域中的副本时，Worker 将进行跨区域通信。

- 如果您的应用程序无法容忍这种延迟，请在副本所在区域部署第二组 Worker，或选择同一区域中的副本：
- 在区域完全中断的情况下，原始区域中的 Worker 可能会与原始 Namespace 一起发生故障。
  为了让 Workflow 在此级别的中断期间保持移动，请将第二组 Worker 部署到次要区域。

:::tip

Temporal Cloud 强制最大连接生命周期为 5 分钟。
这为您的 Worker 提供了重新解析 DNS 的机会。

:::

## 最佳实践：Scheduled 故障转移测试 {#testing}

微服务和外部依赖项有时会失败。
测试故障转移可确保您的应用程序能够有效地处理这些故障。
Temporal 建议对生产中的关键任务应用程序进行定期和定期的故障转移测试。
通过在非紧急情况下进行测试，您可以验证您的应用程序是否能够继续运行，即使部分基础设施出现故障也是如此。

<DiscoverableDisclosure label="Why test?">

:::tip Safety First

如果这是您第一次执行故障转移测试，请使用特定于测试的 Namespace 和应用程序运行它。
这可以帮助您在将其应用到生产环境之前获得操作经验。
练习运行有助于确保流程在生产中的实际事件中顺利运行。

:::

故障转移测试（也称为“<ToolTipTerm term=”触发测试”/>）”可以：

- **验证复制部署**：
  在多区域设置中，故障转移测试可确保您的应用程序可以在主要区域发生中断时从另一个区域运行。
  在标准设置中，故障转移测试改为使用隔离域。
  这可以在关键任务部署中保持高可用性。
  手动测试确认故障转移机制按预期工作，以便您的系统有效处理事件。

- **评估复制滞后**：
  在多区域部署中，监控区域之间的复制延迟至关重要。
  在启动故障转移之前检查延迟以避免回滚 Workflow 进度。
  当使用隔离域时，这一点不太重要，因为故障转移通常是瞬时的。
  手动测试可以帮助您练习这一关键步骤并了解其影响。

- **评估恢复时间**：
  手动测试可帮助您测量实际恢复时间。
  您可以检查它是否满足 [高可用性 Namespace SLA](/cloud/sla) 中所述的 20 分钟或更短的预期恢复时间目标 (RTO)。

- **识别潜在问题**：
  故障转移测试可以发现正常操作期间不可见的问题。
  这包括诸如[积压和容量规划](https://Temporal.io/blog/Workers-in-Production#testing-failure-paths-2438)以及外部依赖项在故障转移事件期间的行为方式等问题。

- **验证无故障编程**：
  Temporal 使用“故障忽略编程”模型，您的应用程序不需要显式处理多种类型的故障。
  测试故障转移可确保此模型在您的应用程序中按预期工作。

- **操作准备**：
  定期测试可以让您的团队熟悉故障转移过程，提高他们处理实际事件发生时的能力。

定期测试故障转移可确保基于 Temporal 的应用程序保持弹性和可靠性，即使基础设施发生故障也是如此。
