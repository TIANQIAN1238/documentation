当云中断中断 Namespace 时，Temporal Cloud 会采取措施维持 Namespace 的可用性和数据持久性。从中断中恢复所需的时间称为“恢复时间”。丢失的数据量（事件历史记录）称为“恢复点”。耐用的系统应该具有较短的恢复时间和恢复点。

为了帮助用户规划在云中断期间保持关键 Workflow 可用，Temporal Cloud 发布了每种中断的恢复时间和恢复点目标。这些目标称为恢复时间目标 (RTO) 和恢复点目标 (RPO)。这些目标是对 Temporal Cloud 的[服务级别协议 (SLA)](/cloud/sla) 的补充。

为了实现最低的 RPO 和 RTO，Temporal Cloud 提供[高可用性](/云/高可用性) 功能，使 Workflow 保持运行并最大限度地减少停机时间。当 Namespace 上启用高可用性时，用户选择一个区域来放置“副本”，该副本将在发生故障时接管。副本的位置决定了所使用的复制类型以及可以处理的中断类型。多区域复制是指活动副本和副本位于同一云上的不同区域（例如，AWS us-east-1 和 AWS us-west-2）。多云复制是指活动副本和副本位于不同的云（例如 AWS 和 GCP）中。同区域复制是指主服务器和副本服务器位于同一区域。 Temporal 始终将活动副本和副本放置在不同的[单元](/cloud/overview#cell-based-infrastruct) 中。

随着 Workflow 在活动区域​​中的进展，历史事件将异步复制到副本。
由于复制是异步的，因此高可用性不会影响活动区域中 Workflow Execution 的延迟或吞吐量。
如果活动区域或单元发生中断，Temporal Cloud 将故障转移到副本，以便现有 Workflow Execution 将继续运行，并可以启动新的 Workflow Execution。

Temporal Cloud 的恢复点目标和恢复时间目标取决于中断类型以及 Namespace 启用的[高可用性](/cloud/high-availability) 功能。 Temporal Cloud 只能在有能力缓解中断的情况下设置 RPO 和 RTO。因此，以下 RPO 和 RTO 适用于具有相应复制类型并已启用 Temporal 启动的故障转移（默认情况下启用）的 Namespace。

1. **可用区中断**：
    1._适用的Namespace：_所有Namespace
    2. _目标：_ 零 RPO 和接近零 RTO
    3._更多详细信息：_从历史上看，这些是云中最常见的中断类型。 Temporal Cloud 跨三个可用区复制每个 Namespace。单个可用区的故障由 Temporal Cloud 在后台自动处理，不会造成数据丢失，并且最终用户几乎不会观察到停机时间。
2. **小区中断**： 
    1._适用的Namespace：_具有同区域复制、多区域复制或多云复制功能的Namespace
    2. _目标：_ 1 分钟 RPO 和 20 分钟 RTO
    3. _更多详细信息：_ Temporal Cloud 在[单元架构](/cloud/sla) 上运行。每个单元都包含托管 Namespace 所需的软件和服务。虽然不太可能，但单元有可能由于未捕获的软件错误或子组件故障（例如底层数据库中断）而经历中断。
3. **区域停电**： 
    1._适用的Namespace：_具有多区域复制或多云复制的Namespace
    2. _目标：_ 1 分钟 RPO 和 20 分钟 RTO
    3. _更多详细信息：_ 在[极少数情况下](https://Temporal.io/blog/how-devs-kept-running-during-the-aws-us-east-1-oct-20-2025)，云提供商内的整个区域都会降级。由于 Namespace 依赖于云提供商的基础设施，因此 Temporal Cloud 也不能免受这些中断的影响。
4. **全云中断**： 
    1._适用的Namespace：_具有多云复制功能的Namespace
    2. _目标：_ 1 分钟 RPO 和 20 分钟 RTO
    3._更多详细信息：_整个云提供商的大部分或所有区域都出现中断。由于云提供商努力保持云区域的解耦，因此这些中断是最罕见的。尽管如此，它们过去[已经发生过](https://status.cloud.google.com/incidents/ow5i3PPK96RduMcb1SsW)。

笔记：

- 上述目标仅适用于已启用 Temporal 启动的故障转移（默认情况下启用）的 Namespace。 Temporal 启动的故障转移由 Temporal 的工具和/或待命工程师启动，无需用户操作。即使启用了 Temporal 启动的故障转移，用户也始终可以在 Namespace 上启动故障转移。在中断期间，用户启动的故障转移不会取消或意外逆转 Temporal 启动的故障转移。

:::note

Temporal Cloud 指标有助于监控生产部署。
本文档涵盖了监控 Temporal Cloud 的最佳实践。

:::

- 上述目标适用于计划外的云中断。它们不适用于健康期间用户发起的故障转移（例如，灾难恢复演练）。阅读[触发故障转移](/cloud/high-availability/failovers)，了解 Namespace 故障转移在健康期间应如何执行。

- 一旦云中断解决，Temporal 的值班工程师将努力恢复不受高可用性保护的 Namespace 的服务。即使云提供商恢复了底层服务，云中断也会对 Temporal 的系统和应用程序产生持续影响。因此，当底层服务恢复时，受影响的 Namespace 可能无法立即可用。受影响的 Namespace 的中断持续时间可能比云提供商的中断持续时间更长。

- 所有 Namespace 每 4 小时备份一次。如果中断导致不受高可用性保护的 Namespace 上的数据丢失，则 Temporal 将使用备份来恢复尽可能多的数据。


## 最小化恢复点

Temporal 在工具和流程方面投入了大量精力，可最大程度地减少恢复点并实现 Temporal 启动的故障转移的 RPO，包括：

- 一流的[数据复制技术](https://youtu.be/mULBvv83dYM?si=RDeWb3gVsEtgGM4z&t=334)，使副本保持最新状态。

- 针对每个 Temporal Cloud Namespace 的复制延迟进行监控、警报和内部 SLO。

但是，Namespace 上的用户操作可能会影响恢复点。例如，吞吐量突然飙升至比 Namespace 之前所见的高得多，可能会造成一段复制滞后，其中副本落后于活动副本。

Temporal 为每个 Namespace 提供[复制滞后](/cloud/metrics/openmetrics/metrics-reference#Temporal_cloud_v1_replication_lag_p99) 指标。该指标近似于 Namespace 在给定时刻发生最坏情况故障时所能达到的恢复点。

:::note

Temporal 建议监控复制延迟，并在延迟过高（例如超过 1 分钟）时发出警报。

:::

## 最小化恢复时间

Temporal 在工具和流程方面投入了大量精力，可最大限度地缩短恢复时间并实现 Temporal 启动的故障转移的 RTO，包括：

- 历史事件被_异步_复制。这确保了即使副本区域出现网络故障或中断，Namespace 仍然可以在活动区域​​中运行 Workflow。

- 自动检测断电。我们有广泛的内部警报来检测 Namespace 的中断，并不断改进该系统。

- 经过实战测试的 Temporal Workflow，可快速执行给定区域中所有 Temporal Cloud Namespace 的故障转移。

- 专家工程师全天候 (24/7) 随时待命监控 Temporal Cloud Namespace，并随时准备在发生停电时提供帮助。

为了实现尽可能短的恢复时间，Temporal 建议您：

- 在 Namespace 上启用 Temporal 启动的故障转移（默认）
- 投资一个流程来检测中断并触发手动故障转移。

即使启用了 Temporal 启动的故障转移，用户也可以在 Namespace 上触发手动故障转移。将手动故障转移过程与 Temporal 启动的故障转移相结合有几个好处：

- 您可以检测 Temporal 无法检测到的中断。在云中，区域中断不会同等影响所有服务。即使您的 Worker 或其他云基础设施中断，Temporal 及其依赖的服务也可能不会受到中断的影响。如果您[监控关键路径中的服务](https://sre.google/sre-book/monitoring-distributed-systems/)并对异常错误率发出警报，您可能会在 Temporal Cloud 之前发现中断。

- 您可以按特定顺序对故障转移进行排序。您的云基础设施可能不仅仅包含 Temporal Namespace：Temporal Worker、计算池、数据存储和其他云服务。如果您手动进行故障转移，则可以选择这些部分切换到副本区域的顺序。然后，您可以通过故障转移演练来测试该排序，并确保其顺利执行，不存在数据一致性问题或瓶颈。

- 您可以比 Temporal 更积极地主动进行故障转移。虽然 20 分钟的 RTO 对于大多数用例来说应该足够了，但有些人可能会努力达到更低的 RTO。对于高频交易、拍卖或热门体育赛事等工作负载，错误时间的中断可能会导致每分钟巨大的收入损失。您可以采用比 Temporal 更积极地进行故障转移的姿势。例如，您可以在第一次出现可能出现中断的迹象时触发手动故障转移，然后再了解是否存在真正的区域中断。

- 即使您有强大的工具来检测中断并触发故障转移，启用 Temporal 启动的故障转移也可以提供“安全网”，以防您的自动化错过中断。如果我们检测到 Namespace 可能很快就会中断，例如，由于滚动故障影响了其他 Namespace，但尚未影响您的 Namespace，它还为 Temporal 提供了先发制人的余地，可以先发制人地对您的 Namespace 进行故障转移。


## 了解 Temporal 的 RTO 与 SLA

Temporal 具有恢复时间目标 (RTO) 和服务级别协议 (SLA)。它们具有互补的目的并适用于不同的情况。

|方面| RTO |服务水平协议 |
|----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
|它是什么？                       |中断中断 Namespace 的总时间的目标或高优先级目标。                                                                                                                                                                            |一项合同协议，规定服务错误率的上限，并产生财务影响。                                                                                                                                                                                                                                                                                                                                                                      |
|它是如何测量的？               |实现的恢复时间以每次中断的分钟数来衡量。                                                                                                                                                                                              |实现的服务错误率以每月错误率来衡量。                                                                                                                                                                                                                                                                                                                                                                                              |
|计算是如何进行的？ |给定中断中实现的恢复时间是指从 Namespace 开始中断到 Namespace 恢复到全部功能（在故障转移到健康区域后或中断缓解后）之间的总时间。 | Temporal 测量 Temporal Cloud 失败的请求百分比，并应用[公式](/cloud/sla) 来获取当月的最终百分比。                                                                                                                                                                                                                                                                                                                       |
|部分降级算不算？    |大多数中断都包含__部分降级__的时期，其中一定比例的 Namespace 操作失败，而其余部分正常完成。当它们破坏 Namespace 时，部分降级的时间段将计入恢复时间的计算中。                       |部分降级仅部分计入服务错误率计算。错误率 10% 的 5 分钟窗口的计数少于错误率 100% 的 5 分钟窗口。                                                                                                                                                                                                                                                                                      |
|什么被排除？                 |对于部分降级，Namespace 的中断取决于 Temporal 的专家判断，但良好的经验法则是服务错误率 >=10%。                                                                                                |我们排除了 Temporal 无法控制的中断，以减轻影响，例如，影响 Namespace 的底层云提供商基础设施故障，而未启用高可用性和 Temporal 启动的故障转移。如果 Namespace 具有相关的高可用性功能并启用了 Temporal 启动的故障转移，则 Temporal 可以采取措施缓解中断，并且通常不计入 SLA。 [SLA 页面](/cloud/sla) 上的完全排除。 |

以下示例说明了区域中断中不同类型的 RTO 和 SLA 计算。这些假设的 Namespace 基于 [真实世界中断] 中的实际 Temporal Cloud 性能(https://Temporal.io/blog/how-devs-kept-running-during-the-aws-us-east-1-oct-20-2025)。

假设区域“middle-earth-1”从 UTC 10:00:00 开始经历了级联故障，导致各种实例和机器随着时间的推移出现故障。 Temporal 的自动故障转移针对所有 Namespace 触发，并于 10:15:00 UTC 完成。

- Namespace 0 位于该区域，但其小区并未受到中断的影响。唯一的停机时间是在故障转移操作期间的几秒钟。它的恢复时间接近于零，并且其服务错误率可以忽略不计。  优雅的故障转移成功，Namespace 的恢复点为 0。

- Namespace 1_A 位于该区域，其单元经历了部分降级，导致 10% 的请求在前 5 分钟内失败，25% 在第二个 5 分钟内失败，50% 在第三个 5 分钟内失败。由于10:00:00到10:15:00影响较大，因此恢复时间为15分钟。如果该月没有其他服务错误，则该月的服务错误率为：( (1 - 10%) + (1 - 25%) + (1 - 50%) + 8925 * 100% ) / 8928 = 99.990%。 （注意：一个月 31 天中有 8928 个 5 分钟时段。） 平滑故障转移成功，并且此 Namespace 实现了 0 的恢复点。

- Namespace 1_B 与 Namespace 2_A 位于同一单元中，因此它也经历了部分降级，导致 10% 的请求失败。然而，其所有者通过自己的工具检测到中断，并决定在 10:05:00 手动进行故障转移。 Namespace 的恢复时间为 5 分钟，服务错误率为 ( 1 * (1 - 10%) + 8927 * 100% ) / 8928 = 99.998%。优雅的故障转移成功，Namespace 的恢复点为 0。

- Namespace 2_A 位于该区域，并且其单元在中断开始时完全网络分区，导致 100% 的请求失败。由于10:00:00到10:15:00影响较大，因此恢复时间为15分钟。如果该月没有其他服务错误，则该月的服务错误率为：( 3 * (1 - 100%) + 8928 * 100% ) / 8640 每月 5 分钟时段 = 99.97%。由于Namespace是网络分区的，所以平滑故障切换没有成功，采用了强制故障切换。达到的恢复点等于网络分区时的复制延迟，即几秒钟。

- Namespace 2_B 位于该区域并且完全网络分区，导致 100% 的请求失败。然而，其所有者通过自己的工具检测到中断，并决定在 10:05:00 手动进行故障转移。 Namespace 的恢复时间为 5 分钟，服务错误率为 ( 1 个 5 分钟周期 * (1 - 100%) + 8639 个 5 分钟周期 * 100% ) / 每月 8640 个 5 分钟周期 = 99.99%。由于Namespace是网络分区的，所以平滑故障切换没有成功，采用了强制故障切换。达到的恢复点等于网络分区时的复制延迟，即几秒钟。

上述所有 Namespace 均位于受影响区域，并超过了 1 分钟 RPO。但他们的恢复时间和服务错误率各不相同。

- 请注意 Namespace 1_A 和 Namespace 2_A 如何自动进行故障转移，并具有**相同的恢复时间但不同的服务错误率**。请注意 Namespace 2_B 和 Namespace 1_A 如何恰好具有**相同的服务错误率但不同的恢复时间**。这说明了即使在相同的中断情况下，RTO 和 SLA 也存在差异。两者都是 Temporal Cloud 用户衡量 Namespace 可用性的宝贵工具。

- 请注意手动故障转移的 Namespace（Namespace 1_B 和 Namespace 2_B）如何实现比自动故障转移的 Namespace（Namespace 1_A 和 Namespace 2_A）更短的恢复时间。这说明了**主动、积极的手动故障转移如何能够比自动故障转移实现更好的恢复时间**。
