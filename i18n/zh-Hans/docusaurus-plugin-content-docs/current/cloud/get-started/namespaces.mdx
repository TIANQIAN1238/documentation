## 云 Namespace 名称是什么？ {#Temporal-云-Namespace-名称}

云 Namespace 名称是客户为 Temporal Cloud 中的 [Namespace](/Namespaces) 提供的名称。每个 Namespace
名称（例如“accounting-product”）在客户帐户范围内是唯一的。之后就无法更改
Namespace 已配置。

每个 Namespace 名称必须符合以下规则：

- Namespace 名称必须包含至少 2 个字符且不超过 39 个字符。
- Namespace 名称必须以字母开头，以字母或数字结尾，并且仅包含字母、数字和
  连字符 (-) 字符。
- 每个连字符 (-) 必须紧接在 _and_ 之前，后跟字母或数字；连续的连字符是
  不允许。
- Namespace 名称中的所有字母必须为小写。

## Temporal Cloud 帐户 ID 是什么？ {#Temporal-云帐户-id}

Temporal Cloud 帐户 ID 是 Temporal Technologies 分配的唯一客户标识符。每个Id都是一个短字符串
数字和字母，如“f45a2”，至少五个字符长。该帐户标识符将在整个过程中保留
每个客户使用 Temporal Cloud 的时间。

有时您可能需要知道您的客户帐户 ID。访问帐户的 Namespaces 提供了一种简单的方法
捕获此信息。每个 Temporal Namespace 使用帐户 ID 后缀。这是字母数字字符串

## 什么是云 Namespace Id？ {#Temporal-cloud-Namespace-id}

Cloud Namespace Id 是 Temporal Cloud 中 [Namespace](/Namespaces) 的全局唯一标识符。 Namespace ID 是
由以下内容连接而成：

1. [Namespace名称](#Temporal-cloud-Namespace-name)
1. 句号 (.)
1. Namespace所属的【账户ID】(#Temporal-cloud-account-id)

例如，对于账户 ID“123de”和 Namespace 名称“accounting-product”，Namespace ID 为
`会计生产.123de`。

## 什么是云 gRPC 端点？ {#Temporal-云-gRPC-端点}

Temporal CLIents 通过通过发送和接收消息在应用程序代码和 Temporal Server 之间进行通信
gRPC 协议。 gRPC是一个远程过程调用框架，具有低延迟和高性能的特点。 gRPC 提供
Temporal 具有高效、与语言无关的通信框架。

每个 Temporal Namespace 使用 gRPC 端点进行通信。迁移到 Temporal Cloud 时，您需要切换
代码中的 gRPC 端点从当前托管（无论是自托管还是本地托管）到 Temporal Cloud。

gRPC 端点显示在每个 Cloud Namespace 的详细信息页面上。请按照以下步骤查找它：

1. 登录您在 [cloud.Temporal.io](https://cloud.Temporal.io/Namespaces) 上的帐户。
2. 从左侧垂直导航导航至Namespace列表页面。
3. 点击或单击 Namespace 名称以选择并打开要检索其端点的 Namespace 的页面。
4. 在Namespace详情页面，点击页面右上角的“连接”按钮。
5. 单击 gRPC 地址旁边的复制图标将其复制到剪贴板。

有关详细信息，请参阅[如何在 Temporal Cloud 中访问 Namespace](/cloud/Namespaces/#access-Namespaces)
不同的 gRPC 端点类型以及如何访问它们。

## Namespace 最佳实践有哪些？ {#最佳实践}

本节提供跨用例、服务、组织 [Namespaces](/Namespaces) 的一般指南
应用程序或域。 Temporal Cloud 提供 Namespace 即服务，因此 Namespace 是端点。客户
不仅应考虑 Namespace 命名约定，还应考虑如何使用 Namespace 来分组或隔离工作负载
一个边界。

每个团队都可以拥有自己的 Namespace，以改进模块化、安全性、调试和故障隔离。 Namespaces
包含可能耗尽速率限制的行为不当 Worker 的爆炸半径。敏感的Workflow状态（PCI数据）可
使用每个 Namespace 权限进行保护，并使用单独的加密密钥进行加密。

Temporal 不同 Namespace 中的应用程序可以通过公开干净的服务与 [Nexus](/cloud/Nexus) 连接
供其他人与内置 [Nexus 访问控制](/cloud/Nexus/security) 一起使用的合同。 Nexus支持跨团队，
跨域、多区域和多云用例。

### 约束和限制

在考虑适当的 Namespace 配置之前，您应该了解以下限制 
在 [Temporal Cloud 限制](/cloud/limits) 页面上列出。

有关在限制范围内管理 Namespace 的建议，请参阅 [Namespace 最佳实践](/best-practices/managing-Namespace) 页面。

### 一般指导

Namespace 配置需要一些考虑。以下是一些需要考虑的一般准则。

- Namespace 通常根据用例进行定义。一个用例可以涵盖广泛的 Workflow Type 和几乎
  并发 [Workflow Executions](/Workflow-执行) 的无限规模。 
- Namespace 可以沿着其他边界进行分割，例如服务、应用程序、域甚至子域。
- Namespace 应用于减少关键任务应用的“爆炸半径”。
- 生产、开发等环境通常有隔离的要求。我们建议每个
  环境有自己的Namespace。
- Namespace 应用于减少关键任务应用的“爆炸半径”。
- 使用 [Nexus](/Nexus) 通过干净的服务契约跨团队、域和 Namespace 边界进行通信，而不是直接共享 Temporal 原语。
  Nexus 使每个团队都可以拥有自己的 Namespace，以提高安全性、故障排除和故障隔离，同时通过 Nexus 端点共享功能。

## 如何访问 Temporal Cloud 中的 Namespace {#access-Namespaces}

{/* 如何在 Temporal Cloud 中访问 Namespace */}

Temporal Cloud 支持使用 [API 密钥](/cloud/api-keys) 对 Namespace 进行身份验证_或_
[mTLS](/云/证书)。将 Namespace 从一种身份验证方法迁移到另一种身份验证方法，或使用两种 API
密钥和mTLS认证在同一个Namespace上，请联系
[支持](https://docs.Temporal.io/cloud/support#support-ticket)。

:::info

在**相同** Namespace 上使用 **同时** API 密钥和 mTLS 身份验证
[预发布](/evaluate/development-product-features/release-stages) 并且不支持
[高可用性功能](/cloud/high-availability) 或使用 API 密钥对 Namespace 端点进行身份验证。

:::

连接到 Namespace 需要适用于给定 Namespace 的特定端点。
有两种类型的 gRPC 端点用于访问 Temporal Cloud 中的 Namespace：Namespace 端点和区域端点。

- **Namespace 端点** (`<Namespace>.<account>.tmprl.cloud:7233`) — **推荐**
    - 该端点对于每个 Namespace 都是唯一的。无论 Namespace 使用哪个区域，它始终连接到 Namespace。
    - 使用 Namespace 端点的 Temporal CLIent 不必知道 Namespace 位于哪个区域。
    - 对于具有[高可用性](/cloud/high-availability) 的 Namespace，Namespace 端点会自动将流量定向到活动区域，因此 Worker 和客户端无需在故障转移期间更改端点。
- 区域端点（`<region>.<cloud_provider>.api.Temporal.io:7233`）
    - Temporal Cloud 对于每个云区域只有一个区域端点。同一区域端点可以访问该区域中活动的任何 Namespace（或在该区域具有[副本](/cloud/high-availability)）。
    - Temporal CLIent 可以使用区域端点来确保与 Namespace 的连接始终发生在该区域内。这在您想要明确控制哪个区域处理请求的高级[高可用性](/cloud/high-availability) 设置中非常有用。
    - 使用 mTLS 进行身份验证时，Temporal CLIent 必须在其对 Namespace 端点值的请求中将“server_name”属性设置为“<Namespace 端点值>”。这告诉客户端在 TLS 握手期间期望不同的 SNI 标头，因为对区域端点的请求被重定向到特定的 Namespace。

### 使用 API 密钥或 mTLS 配置 Temporal CLIent

要使用 API 密钥连接 [Temporal CLI](/cli)、[Client SDK](/develop)、[tcld](/cloud/tcld)，
  [Cloud Ops API](/ops) 和 [Terraform](/cloud/terraform-provider)，请参阅
  [使用 API 密钥进行身份验证](/cloud/api-keys#using-apikeys)。

要使用 mTLS 连接 [Temporal CLI](/cli) 和 [Client SDK](/develop)，请参阅
  [配置客户端以使用客户端证书](/cloud/certificates#configure-clients-to-use-client-certificates)。

### 访问 Temporal Web UI

要访问 Temporal Web UI，请使用以下形式的 HTTPS 端点：
`https://cloud.Temporal.io/Namespaces/<Namespace>.<帐户>`。例如：
`https://cloud.Temporal.io/Namespaces/accounting-product.f45a2`。

### 通过加密和专用连接访问 Namespace

无论您是自托管 XPROT026X 还是使用 XPROT009X，您都可以管理代码的运行时。
要将 XPROT027X 迁移到 XPROT009X，您需要更改客户端连接代码中的一些参数，例如更新 Namespace 和 XPROT042X 端点。

为了增强保护：
- 设置到 Namespace 的[专用连接](/cloud/connectivity#private-network-connectivity-for-Namespaces)。
- 在您自己的网络架构中，为来自您的客户端和 Worker 的传出网络请求设置允许列表，其中包含 IP 地址范围
  您的 Namespace 所在的云提供商区域：
  - [AWS IP 地址范围](https://docs.aws.amazon.com/vpc/latest/userguide/aws-ip-ranges.html)
  - [GCP IP 地址范围](https://cloud.google.com/compute/docs/faq#find_ip_range)

要管理证书过滤器，请使用 [tcld Namespace 证书过滤器](/cloud/tcld/Namespace/#certificate-filters)
命令。有关更多信息，请参阅
[如何在 Temporal Cloud 中管理证书过滤器](/cloud/certificates#manage-certificate-filters)。

## 如何删除 Temporal Cloud 中的 Namespace {#delete-a-Namespace}

:::info

要删除 Namespace，用户必须拥有 Namespace 管理员 [权限](/cloud/manage-access/roles-and-permissions#Namespace-level-permissions)
Namespace。

:::

### 使用 Temporal Cloud UI 删除 Namespace

1. 进入Temporal Cloud UI并登录。
1. 在窗口左侧，选择 **Namespaces**。
1. 在 **Namespaces** 页面上，选择 Namespace 名称。
1. 在Namespace页面上，选择窗口右上方的“**编辑**”。
1. 在 **编辑** Namespace 页面上，选择窗口右上角的 **删除 Namespace**。
1. 在 **删除 Namespace** 对话框中，键入“DELETE”以确认删除该 Namespace。
1. 选择**删除**。

删除 Temporal Cloud Namespace 后，Temporal 服务会立即删除 Namespace 的 Workflow Execution
和 Task Queue。在移除 Namespace 之前，请确保所有 Workflow 均已完成、取消或终止。的
Namespace 的移除是永久性的。

关闭的 Workflow 历史记录保留在 Temporal 存储中，直到用户定义的保留期到期。这个时期
反映 Workflow Execution 关闭时生效的策略。

如有其他问题或疑虑，请联系[支持](https://docs.Temporal.io/cloud/support#support-ticket)。

### 使用 tcld 删除 Namespace

有关详细信息，请参阅 [tcld Namespace 删除](/cloud/tcld/Namespace/#delete) 命令参考。

### Namespace 删除保护{#delete-protection}

为了防止意外删除 Namespace，Temporal Cloud 提供了保护功能。当您启用删除时
Namespace 保护您的生产环境，确保关键数据不会被无意删除。

请按照下列步骤操作：

- 访问 Temporal Cloud 上的 [Namespaces 页面](https://cloud.Temporal.io/Namespaces)。
- 打开您的 Namespace 详细信息页面。
- 选择编辑按钮。
- 向下滚动到安全性并单击披露按钮（向下的插入符号）。
- 启用**删除保护**

<CaptionedImage
src="/img/cloud/Namespace/deletion-protection.png"
  title="通过切换开关启用删除保护"
/>

要使用 [`tcld`](/cloud/tcld) 启用或禁用此功能，请使用以下命令。将值设置为“true”即可
启用或“false”禁用：

```
tcld namespace lifecycle set \
    --namespace <namespace_id.account_id> \
    --enable-delete-protection <Boolean>
```

## 如何在 Temporal Cloud 中标记 Namespace {#tag-a-Namespace}

标签是键值元数据对，可以附加到 Temporal Cloud 中的 Namespace，以帮助操作员组织、
更轻松地跟踪和管理 Namespace。

### 标签结构和限制

- 每个Namespace最多可以有10个标签
- 对于给定的 Namespace，每个密钥必须是唯一的（例如，Namespace 不能同时具有 `team:foo` 和 `team:bar` 标签）
- 键和值的长度必须为 1-63 个字符
- 允许的字符：小写字母 (`a-z`)、数字 (`0-9`)、句点 (`.`)、下划线 (`_`)、连字符 (`-`) 和
  在符号 (`@`) 处
- 标签不是安全存储机制，不应存储 PII 或 PHI
- 标签不会改变标记资源的行为
- 每个帐户有 1000 个唯一标签键的软限制

### 权限

- 只有[**账户管理员**和**账户所有者**](/cloud/manage-access/roles-and-permissions#account-level-roles)可以创建和编辑标签
- 所有有权访问 Namespace 的用户都可以查看其标签

### tcld

有关详细信息，请参阅 [tcld Namespace 标签](/cloud/tcld/Namespace/#tags) 命令参考。
