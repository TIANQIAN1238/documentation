随着组织扩大 Temporal 采用规模，用例变得更加复杂，并且可能很难找到所需的所有相关信息。
团队内的部落知识被孤立，导致 Workflow 设计、错误处理和测试模式不一致。
这种碎片化导致支持负担增加，并且新开​​发人员的入职速度变慢。
您还可能无意中引入安全漏洞或合规性漏洞。
为了防止这些问题，您可以建立一个内部 Temporal 知识中心来解决多个团队时出现的常见问题
独立采用Temporal。

本指南介绍了知识中心的内容、如何传达知识以及如何保持其有用性。
要引导您的知识中心，请使用
[Temporal Platform Hub](https://go.Temporal.io/platform-hub) 模板作为起点。

## 什么属于您的知识中心

尽管 Temporal 本身具有[完整的文档](https://docs.Temporal.io/)，但并非所有内容都适用于您的组织或团队的用例。
知识中心将文档提炼为您的团队需要的特定信息。
组织内容的一种方法是根据开发人员所处的阶段。
以下示例大纲显示了要包含的部分。

＃＃＃ 评价

评估部分可帮助开发人员了解 Temporal 是什么以及它是否适合他们的问题。
本部分的目标是增加知识中心流量并降低支持问题率。
本节中包括以下项目：

- **失败是正常的**：Temporal 旨在应对失败和问题，但*您的应用程序逻辑*也必须如此。
- **部分故障通常比完全故障更难处理**：“大部分工作”的系统暴露出最多的缺陷。
- **恢复路径值得与稳定状态一样多的测试**：分析恢复应用程序行为与分析失败行为一样多。
- **在破坏之前建立可观察性**：确保指标、日志和 XPROT023X 工具在注入故障之前就位。
- **测试是一个持续的过程**：测试永远不会结束。测试是一种实践。

＃＃＃ 建造

构建部分面向刚刚开始使用 Temporal 的开发人员。
本节的目标是为开发人员提供从零到运行 Workflow 的单一路径。
本节中包括以下项目：

- **入门**引导开发人员完成 30 分钟的快速入门，涵盖环境设置、入门模板、
  并在本地和 Temporal Cloud 上运行第一个 Workflow。使用现有的 Temporal [快速入门](/quickstarts)，或根据您自己的用例构建一个。
- **学习路径**提供从基础到高级主题的自定进度课程，按人物和链接定制
  前往【Temporal的免费培训】(https://learn.Temporal.io/)。

＃＃＃ 船

船舶部分提供了开发人员从本地原型到生产所需的架构标准和护栏
部署。
本节的目标是提供开发人员缩短生产时间所需的参考材料。
本节中包括以下项目：

- **架构和标准**记录 Namespace 约定、连接要求和 Worker 部署
  每个团队都遵循的标准。
- **成本指南**解释了计费操作、存储层和成本节约技巧，以便开发人员构建具有成本效益的产品
  Workflows。
- **共同责任**定义了跨 IAM 的平台和应用程序团队之间的所有权矩阵，
  基础设施、开发、部署、可观察性和运营。
- **设计模式** 策划 Workflow 模式，其中包含开发人员可以采用的描述和代码示例链接
  直接。

＃＃＃ 操作

操作部分为开发人员提供了在发生事件时进行自助服务并找到答案的工具，而无需上报给平台团队。
本节的目标是减少需要支持的问题数量。
本节中包括以下项目：

- **故障排除和升级**涵盖可观察性工具、常见问题的操作手册、升级路径、SLA、
  以及警报定义示例。
- **支持和常见问题解答**记录您的支持级别、票证提交流程、Temporal 帐户联系人、专家主导
  会话类型和常见问题。

## 衡量知识中心的成功

创建知识中心后，建立指标来衡量其对组织的有效性。
下表显示了组织用来衡量其知识中心影响的示例指标，
以及现实的前后目标：

|公制|知识中心之前 |目标|它告诉你什么 |
| :--- | :--- | :--- | :--- |
| **首次使用 Workflow 的时间** |几天到几周（开发人员将分散的资源拼凑起来）|不到 30 分钟（开发人员遵循单一入门指南）|测量登机摩擦。初次使用 Workflow Signal 的时间很短，您的入门指南非常有效。 |
| **Workflow 投入生产的时间** |几周到几个月（由于不清楚的 Namespace 配置和部署流程而受阻）|不到 2 周（开发人员遵循记录的自助服务配置）|衡量发展与交付价值之间的差距。很长一段时间表明缺少文档和自动化机会。 |
| **支持提问率** |每周通过 Slack 向平台团队提出 20-30 多个问题 |每周少于 5 次 |衡量自助服务解决方案。下降趋势表明开发人员正在知识中心寻找答案，而不是询问平台团队。 |
| **知识中心流量** | N/A（不存在集中资源）|每月页面浏览量稳定或增长 |确定开发人员依赖哪些内容以及仍存在差距。页面流量下降可能表明该页面已过时；高流量和高跳出率可能表明该页面没有回答问题。 |

这些指标创建了一个反馈循环：测量、识别差距、改进内容，然后再次测量。

## 哪些内容不属于您的知识中心

知识中心并不是[Temporal官方文档](https://docs.Temporal.io/)的镜像。
避免重复 Temporal 已维护的 SDK API 参考、概念解释或发行说明。
当内容发生变化时，您的副本就会变得混乱而不是清晰。
相反，链接到官方文档并保留您的知识中心以用于特定于组织的决策、约定、
Temporal 的公共文档未涵盖的操作过程和操作过程。

## 如何维护和交流您的知识中心

如果信息变得陈旧，或者开发人员不知道它的存在，那么拥有彻底、完整的知识中心就没有用处。

### 分配所有权

指定平台团队或开发人员体验团队作为所有者。
该团队负责初始内容创建、持续维护并审查来自
应用团队。

### 让它被发现

开发者找不到知识中心就等于没有知识中心。

- 注册一个重定向到知识中心的短 URL（例如“go/Temporal”）。
- 将链接固定在 Temporal 相关的通信渠道（Slack、Microsoft Teams）中。
- 在 Slack 中回答问题时，回复相关知识中心页面的链接，而不是重新解释
  内联。
  这养成了首先检查集线器的习惯。

### 定期审查指标

定期跟踪[衡量知识中心的成功](#measuring-success-of-your-knowledge-hub) 中的指标，以确定什么是
工作以及仍然存在差距的地方。

捕获通过 Slack 或票证到达平台团队的每个问题，作为新内容的候选者。

通过拉取请求模板等轻量级流程征求应用程序团队的贡献。

### 保持内容最新

- **按节奏审核**：至少每季度审核一次每一页。
  为每个页面分配评论所有者和日期，以便查看过时情况。
- **将 Update 与事件联系起来**：每当您的组织更改其 Temporal 架构时，Update 就是知识中心，
  Updates 其部署工具，或修改其共享责任模型。
- **积极修剪**：删除或存档不再适用的内容。
  过时的文档比没有文档更糟糕，因为开发人员遵循它并会得到意想不到的结果。
