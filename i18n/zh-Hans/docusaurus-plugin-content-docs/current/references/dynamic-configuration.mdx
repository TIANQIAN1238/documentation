Temporal 集群提供[动态配置](/Temporal-service/configuration#dynamic-configuration) 密钥，您可以将 Update 的密钥应用到正在运行的集群，而无需重新启动服务。

创建集群配置时，动态配置键将设置为默认值。
您可以在测试集群设置时覆盖这些值，以根据工作负载要求获得最佳性能。

有关动态配置键的完整列表，请参阅 [https://github.com/Temporalio/Temporal/blob/main/common/dynamicconfig/constants.go](https://github.com/Temporalio/Temporal/blob/main/common/dynamicconfig/constants.go)。
确保检查服务器发行说明以了解这些键和值的任何更改。

对于动态配置键的默认值，请检查以下链接：

- [前端服务](https://github.com/Temporalio/Temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/frontend/service.go#L176)
- [历史服务](https://github.com/Temporalio/Temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/history/configs/config.go#L309)
- [匹配服务](https://github.com/Temporalio/Temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/matching/config.go#L125)
- [Worker服务](https://github.com/Temporalio/Temporal/blob/5783e781504d8ffac59f9848b830868f3139b980/service/Worker/service.go#L193)

设置动态配置是可选的。
仅当您需要覆盖默认值以在 Temporal 集群上实现更好的性能时才更改这些值。
另外，请确保在将这些更改投入生产之前测试您的更改。

有关详细信息，请参阅[CLI 参考](/cli/)。

要覆盖默认动态配置值，请为要在 YAML 配置文件中更改的动态配置键指定自定义值和约束。
创建动态配置文件时使用以下格式。

```yaml
testGetBoolPropertyKey:
  - value: false
  - value: true
    constraints:
      namespace: 'your-namespace'
  - value: false
    constraints:
      namespace: 'your-other-namespace'
testGetDurationPropertyKey:
  - value: '1m'
    constraints:
      namespace: 'your-namespace'
      taskQueueName: 'longIdleTimeTaskqueue'
testGetFloat64PropertyKey:
  - value: 12.0
    constraints:
      namespace: 'your-namespace'
testGetMapPropertyKey:
  - value:
      key1: 1
      key2: 'value 2'
      key3:
        - false
        - key4: true
          key5: 2.0
```

### 约束

您可以对某些动态配置键定义约束，以设置适用于 Namespace 或 Task Queue 级别的特定值。
不对动态配置键定义约束会在集群中设置值。

- 要无限制地设置配置键的全局值，请使用以下命令：

  ```yaml
  frontend.globalNamespaceRPS: # Total per-Namespace RPC rate limit applied across the Cluster.
    - value: 5000
  ```

- 对于可在 Namespace 级别自定义的键，除了全局适用于所有 Namespace 的一个默认值之外，您还可以为不同的 Namespace 指定多个值。
  要在 Namespace 级别设置值，请使用“Namespace”（字符串），如下例所示。

  ```yaml
  frontend.persistenceNamespaceMaxQPS: # Rate limit on the number of queries the Frontend sends to the Persistence store.
    - constraints: {} # Sets default value that applies to all Namespaces
      value: 2000 # The default value for this key is 0.
    - constraints: { namespace: 'namespace1' } # Sets limit on number of queries that can be sent from "namespace1" Namespace to the Persistence store.
      value: 4000
    - constraints: { namespace: 'namespace2' }
      value: 1000
  ```

- 对于可以在 Task Queue 级别自定义的键，除了 Namespace 之外，您还可以指定 Task Queue 名称和任务类型。
  要在 Task Queue 级别设置值，请将“taskQueueName”（字符串）与“taskType”（可选；支持的值：“Workflow”和“Activity”）结合使用。

例如，如果您的 Workflow Execution 每秒创建大量 Workflow 和 Activity 任务，您可以向 Task Queue 添加更多分区（默认为 4）以处理任务的高吞吐量。
  为此，请将以下内容添加到动态配置文件中。
  请注意，如果更改分区数量，则必须为 Task Queue 上的读取和写入操作设置相同的计数。

  ```yaml
  matching.numTaskqueueReadPartitions: # Number of Task Queue partitions for read operations.
    - constraints: { namespace: 'namespace1', taskQueueName: 'tq' } # Applies to the "tq" Task Queue for both Workflows and Activities.
      value: 8 # The default value for this key is 4. Task Queues that need to support high traffic require higher number of partitions. Set these values in accordance to your poller count.
    - constraints: {
          namespace: 'namespace1',
          taskQueueName: 'other-tq',
          taskType: 'Activity',
        } # Applies to the "other_tq" Task Queue for Activities specifically.
      value: 20
    - constraints: { namespace: 'namespace2' } # Applies to all task queues in "namespace2".
      value: 10
    - constraints: {} # Applies to all other task queues in "namespace1" and all other Namespaces.
      value: 16
  matching.numTaskqueueWritePartitions: # Number of Task Queue partitions for write operations.
    - constraints: { namespace: 'namespace1', taskQueueName: 'tq' } # Applies to the "tq" Task Queue for both Workflows and Activities.
      value: 8 # The default value for this key is 4. Task Queues that need to support high traffic require higher number of partitions. Set these values in accordance to your poller count.
    - constraints: {
          namespace: 'namespace1',
          taskQueueName: 'other-tq',
          taskType: 'Activity',
        } # Applies to the "other_tq" Task Queue for Activities specifically.
      value: 20
    - constraints: { namespace: 'namespace2' } # Applies to all task queues in "namespace2".
      value: 10
    - constraints: {} # Applies to all other task queues in "namespace1" and all other Namespaces.
      value: 16
  ```

{/* 请注意，设置最多约束的值优先于设置较少约束的值，无论它们在动态配置键中设置的顺序如何。 */}

有关如何设置动态配置的更多示例，请参阅[samples-server](https://github.com/Temporalio/samples-server/blob/main/tls/config/dynamicconfig/development.yaml)。

## 常用动态配置键

下表列出了可用于对 Temporal 集群的速率限制请求的常用动态配置键。

设置动态配置键是可选的。
如果您选择为 Temporal 集群使用 Update 这些值，请确保您配置了足够的资源来处理负载。

此处列出的所有值均适用于 Temporal Server v1.21。
检查[服务器发行说明](https://github.com/Temporalio/Temporal/releases)，以验证升级版本时是否存在任何潜在的重大更改。

### 服务级别 RPS 限制

每秒请求数 (RPS) 动态配置键设置向集群中的每个服务发出请求的速率。

扩展服务时，调整 RPS 以测试您的工作负载并设置可接受的配置基准。
超过这些限制会导致“ResourceExhaustedError”。

|动态配置键|类型 |描述 |默认值|
| -------------------------------------- | ---- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------- |
|前端 |      |                                                                                                                                                                                                                                                      |               |
| `前端.rps` |国际 |每个前端服务主机接受的请求的速率限制（请求/秒）。                                                                                                                                                                    | 2400 | 2400
| `前端.NamespaceRPS` |国际 |前端服务上每个 Namespace 接受的请求的速率限制（请求/秒）。                                                                                                                                                        | 2400 | 2400
| `前端.NamespaceCount` |国际 |每个前端服务主机上每个 Namespace 的并发 Task Queue 轮询数量限制。                                                                                                                                                          | 1200 | 1200
| `frontend.globalNamespaceRPS` |国际 |每个 Namespace 接受的请求的速率限制（请求/秒），跨集群应用。该限制均匀分布在可用的前端服务实例中。如果设置了此项，它将覆盖每个实例的限制（`frontend.NamespaceRPS`）。 | 0 |
| `内部前端.globalNamespaceRPS` |国际 |跨集群应用的每个内部前端服务主机上接受的请求的速率限制（请求/秒）。                                                                                                                                | 0 |
|历史|      |                                                                                                                                                                                                                                                      |               |
| `历史.rps` |国际 |每个历史服务主机接受的请求的速率限制（请求/秒）。                                                                                                                                                                     | 3000 | 3000
|匹配|      |                                                                                                                                                                                                                                                      |               |
| `匹配.rps` |国际 |每个匹配服务主机接受的请求的速率限制（请求/秒）。                                                                                                                                                                    | 1200 | 1200
| `matching.numTaskqueueReadPartitions` |国际 | Task Queue 的读取分区数。必须使用“matching.numTaskqueueWritePartitions”进行设置。                                                                                                                                                 | 4 |
| `matching.numTaskqueueWritePartitions` |国际 | Task Queue 的写入分区数。                                                                                                                                                                                                         | 4 |

持久性存储速率限制是同步评估的。
根据您的数据库容量和工作负载调整这些键。
如果对持久性存储进行的查询数量超过动态配置值，您将看到任务出现延迟和超时。

|动态配置键 |类型 |描述 |默认值|

|历史|      |                                                                                                                                                                                                                                                         |               |

| `history.persistenceNamespaceMaxQPS` |国际 |历史主机可以发送到持久性存储的每个 Namespace 每秒的最大查询数。 <br /> 如果为此配置设置的值小于或等于 0，则将应用为“history.persistenceMaxQPS”设置的值。     | 0 |
|匹配|      |                                                                                                                                                                                                                                                         |               |
| `matching.persistenceMaxQPS` |国际 |匹配服务主机每秒可以发送到持久性存储的最大查询数。                                                                                                                                                  | 9000 |

| Worker |      |                                                                                                                                                                                                                                                         |               |

### Activity 和 Workflow 默认策略设置

您可以使用以下动态配置键在集群级别定义 Activity 和 Workflow [重试策略](/encyclopedia/retry-policies) 的默认值。

|动态配置键 |类型 |描述 |默认值|
| ------------------------------------------------ | -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- |
| `history.defaultActivityRetryPolicy` | Map（键值对元素）|当代码中未为 Activity 显式设置时，Activity Retry Policy 的服务器配置。    | [Retry Policy 的默认值](/encyclopedia/retry-policies#default-values-for-retry-policy) |
| `history.defaultWorkflowRetryPolicy` | Map（键值对元素）| Retry Policy 用于未设置字段，其中用户已设置显式“RetryPolicy”，但未指定所有字段。 | [Retry Policy 的默认值](/encyclopedia/retry-policies#default-values-for-retry-policy) |

### Visibility 辅助设置

辅助 Visibility 配置密钥可在 Temporal 集群上启用双 Visibility。
这在迁移 Visibility 数据库或创建备份 Visibility 存储时非常有用。

|动态配置键|类型 |描述 |默认值|
| ------------------------------------------------------ | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------- |
| `system.enableReadFromSecondaryVisibility` |布尔 |启用从[辅助 Visibility 存储](/dual-Visibility) 读取，并且可以针对每个 Namespace 进行设置。允许的值为“true”或“false”。                                                                                                                                                                                                                                                                         | `假` |
| `system.secondaryVisibilityWritingMode` |         |允许将 Visibility 数据写入辅助 Visibility 存储，并且可以针对每个 Namespace 进行设置。将此值设置为“on”将禁用对主 Visibility 存储的写入操作。允许的值：<br /> `off`：仅允许写入主 Visibility 存储。 <br /> `on`：仅允许写入辅助 Visibility 存储。<br /> `dual`：允许写入主要和辅助 Visibility 存储。 | `关闭` |

### 服务器版本检查设置

Temporal Server 报告服务器版本及其连接的 SDK 版本，以确定 Web UI 是否应显示一条横幅，表明有新版本可供安装。可以通过定义以下值或将“Temporal_VERSION_CHECK_DISABLED”环境变量设置为“1”来禁用此功能。

|动态配置键 |类型 |描述 |默认值|
| ----------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- |
| `frontend.enableServerVersionCheck` |布尔 |使Temporal Server能够报告有关当前服务器和SDK的版本信息。允许的值为“true”或“false”。 | `真实` |

### Nexus 设置

与Nexus管理相关的设置。

有关这些配置的更多详细说明，请参阅 [in-repo Nexus 架构文档](https://github.com/Temporalio/Temporal/blob/6cb70c0785fd7ba4574ed91d772fe17dff4981b4/docs/architecture/Nexus.md)。

|动态配置键|类型 |描述 |默认值|
| ------------------------------------------------------ | -------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------- |
| `system.enableNexus` |布尔 |启用 Nexus 功能。在 1.31.0 中删除（Nexus 始终启用）。                                                                                                                                                                            | `true`（自 1.27 起）|
| `component.Nexusoperations.useSystemCallbackURL` |字符串|控制是否使用 Temporal://system 作为服务器生成的回调 URL。 （服务器 1.30 或更高版本）| `假` |
| `component.Nexusoperations.callback.endpoint.template` |字符串|定义用于构造 Nexus 回调 URL 的 URL 模板。 （仅实验性外部端点目标功能或早于 1.30 的服务器需要。）取消设置 |
| `component.callbacks.allowedAddresses` |对象|定义服务器将接受的回调 URL 模式的安全允许列表；用于限制可以调用哪些回调端点。 （仅实验性外部端点目标功能或早于 1.30 的服务器需要。）取消设置 |
